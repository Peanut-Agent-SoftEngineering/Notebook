# CMU_15-445_p09

## 一、B+树的核心设计决策与优化

> 主题：B+树节点大小、分裂合并策略、变长键处理及并发控制等核心设计决策的权衡与优化。

### 1. 节点大小选择 (Node Size Selection)

- **核心概念**：B+树的节点大小**并非固定不变**，而是需要根据**存储设备的速度**动态权衡。目标是**最大化顺序I/O**，每次磁盘访问带回最多的键值。
- **关键术语**：**顺序I/O**、**存储层级**（HDD/SSD/内存）
- **设备与推荐节点大小**：
    - **慢速旋转磁盘 (HDD)**：~**1 MB**
        - *原因*：磁头寻道时间极长（~10ms），必须单次I/O带回海量数据，最大化吞吐。
    - **现代SSD (NAND/ NVMe)**：~**10 KB**
    - **内存 (In-Memory / DRAM)**：~**512 Bytes**
        - *原因*：对齐CPU缓存行（Cache Line），避免撑爆**L3缓存**（容量有限，约8-32MB）。若节点过大，缓存命中率暴跌。
- **底层逻辑**：**设备延迟越高，单次I/O成本越贵，节点越大**；反之则越小，追求缓存亲和性。
- **工作负载决定最优大小**：
    - **点查询（Point Lookup）为主**：需要频繁从根节点遍历到叶子节点并取出单一数据。此时**较小的节点大小**更优，因为能减少不必要的I/O，加快随机查找速度。
    - **范围扫描（Range Scan）为主**：需要在叶子节点链表中连续扫描大量数据。此时**较大的节点大小**更优，能最大化顺序I/O的带宽利用率。
- **工程建议**：现代数据库（如Bustub）为简化实现，常默认使用 **4 KB** 的节点/页面大小，这是一个兼顾多种场景的折中值。

### 2. 分裂与合并策略 (Split & Merge Policies)

- **核心概念**：何时触发节点分裂/合并，并非只有“满则分，半则合”一种策略，存在多种**启发性算法**。
- **关键术语**：**分裂阈值**、**合并阈值**、**再平衡**
- **常见策略**：
    - **即时分裂**：节点填满立即分裂（延迟最低，但空间利用率低）。
    - **延迟分裂**：仅在插入失败（如父节点已满）时才分裂（减少分裂次数，提高空间利用率）。
    - **提前合并**：当节点占用率低于某个阈值（如40%）时触发合并，减少树的高度。
    - **懒惰合并**：节点空到一定地步才合并（减少合并波动，但可能浪费空间）。
- **延迟合并的决策逻辑**：
    - **判断条件**：并不是所有低于半满的情况都需要立刻合并。
        - **如果节点为空：** 必须立即合并，没有保留价值。
        - **如果节点略低于半满：** 可以暂时容忍，等待后续插入将其“填回”半满状态。
    - **决策依据**：
        - **树的高度（Height）**：较高的树意味着合并操作的代价更大，可容忍更宽松的延迟阈值。
        - **硬件特性**：不同硬件（HDD vs. SSD vs. 内存）下合并操作的代价不同。
- **算法建议**：参考经典书籍 **《Modern B-Tree Techniques》** (Goetz Graefe, 2010)，被誉为B+树领域的“圣经”。

### 3. 页面大小对并发的影响 (Concurrency & Page Size)

- **核心概念**：在多线程B+树中，**节点大小直接影响锁竞争粒度**。
- **关键术语**：**锁粒度** (Lock Granularity)、**热点** (Hot Spot)
- **权衡**：
    - **大节点**：持有锁时间长，阻塞更多线程，**并发性能差**。
    - **小节点**：锁持有时间短，但节点链接操作更多，**锁开销增加**。
- **设计原则**：对于**多线程环境**，节点大小需额外考虑**争用（Contention）** 因素，不能仅凭存储设备决定。

### 4. 变量数据布局 (Variable-Length Key Handling)

- **核心概念**：B+树中的键可能是**变长**（如字符串、varchar），如何高效存储？
- **关键术语**：**键压缩** (Key Compression)、**前缀截断** (Prefix Truncation)
- **常见实现方式**：
    - **直接存储**：节点内保存完整键（浪费空间，比较开销大）。
    - **前缀压缩**：相邻键共享公共前缀，只保存差异化后缀（极大减少节点存储键数，提升扇出，降低树高）。
    - **后缀截断：** 在内部节点（非叶子节点）中，只存储足以区分左右子树的最小前缀（例如“cat”和“catalog”只需存“cat”以及分隔符“alog”的逻辑）。
- **技巧**：对字符串索引，**前缀压缩**是B+树内存占比下降50%-70%的常见优化手段。

#### 四种主要处理方式

##### 1. 存储指针（Record ID/Pointer）
- **核心概念**：节点内**不存储键的实际字节**，而是存储一个指向完整元组（Tuple）的**指针（Record ID）**
- **工作机制**：比较时通过指针去获取真实键值
- **适用场景**：
  - ✅ 内存系统：特别是内存受限环境，如**T树（T-tree）**
  - ❌ 磁盘系统：每次比较都需要页表查找（Page Table Fetch），导致大量I/O
- **优势**：避免键在树中重复存储，节省空间
- **劣势**：额外一次间接寻址，增加访问延迟

##### 2. 变长节点（Variable-Length Nodes）
- **核心概念**：节点大小随存储的键长度动态调整
- **主要问题**：
  - **碎片化**：磁盘或内存中频繁产生碎片
  - **回收难题**：节点释放后需要处理空洞（Hole Plugging）
  - **管理复杂**：增加实现难度和运行时开销

##### 3. 填充（Padding）
- **核心概念**：将所有键**填充到统一长度**
- **关键点**：通常在索引构建时定义最大键长度
- **优势**：保持节点结构一致，简化管理
- **劣势**：可能浪费空间（短键时）

##### 4. 其他方法（简述）
- **前缀压缩（Prefix Compression）**：只存储键之间的差异部分
- **键拆分为定长片段**：适用于特定模式的键

#### 选择策略的影响因素

| 因素 | 说明 |
|------|------|
| **硬件环境** | 内存 vs 磁盘、缓存行大小 |
| **线程访问模式** | 读写比例、并发度 |
| **键分布特征** | 键长度变化范围、前缀重复性 |
| **性能要求** | 空间效率 vs 访问延迟 |

#### 关键公式（如果采用指针方案）
- 空间节省：$S_{saved} = N \times (L_{avg} - P_{size})$
  - 其中 $N$ = 键数量，$L_{avg}$ = 平均键长度，$P_{size}$ = 指针大小
- 访问代价：每次比较增加 $O(1)$ 间接寻址

> **本质权衡**：在内存系统中，用**一次间接访问**换取**空间节省**；在磁盘系统中，则决不可接受这种额外的I/O开销。

### 5. 节点内查找（Node Lookup）

#### 线性扫描（Linear Search）

**核心概念**：在到达某个节点后，通过**逐一遍历键数组**查找目标键。

- **适用性：** 无论节点是根节点、内部节点还是叶子节点，查找逻辑相同
- **排序无关：** 无论键数组是否排序，线性搜索都能工作
- **性能瓶颈：** 如果节点尺寸很大、键数量多，线性扫描**速度缓慢**

#### SIMD向量化加速

**核心概念**：利用现代CPU的**单指令多数据（Single Instruction, Multiple Data）** 能力，同时比较多个键值。

- **基本原理**：不同于传统CPU逐条处理单个数据的模式（例如一次加法只加两个数），向量化指令允许**将多个数据装入特殊宽寄存器**，然后**一条指令同时对所有元素执行同一操作**。
- **关键术语**：
  - **SIMD (Single Instruction Multiple Data)**：单指令多数据流
  - **向量寄存器 (Vector Register)**：如Intel SSE的 **128位寄存器**
  - **打包操作 (Packed Operation)**：同时对寄存器内所有元素执行操作
- **效果**：显著提升节点内的线性搜索吞吐量

#### 二分查找（Binary Search）

**核心机制**
- **前提条件**：数据必须**预排序**
- **操作流程**：
  1. 从数组**中间位置**开始比较
  2. 若目标值大于当前值，则向右半区跳转
  3. 若目标值小于当前值，则向左半区跳转
  4. 重复直到找到目标或区间为空
- **时间复杂度**：O(log N)，比线性搜索O(N)快得多
- **适用场景**：数据量大且频繁搜索时

#### 插值查找（Interpolation Search）

**核心原理**
- **利用数据分布知识**：如果知道键值的**分布特性**（如均匀分布），可以**猜测目标值可能在的位置**
- **查找方法**：类似于查字典时，不直接翻到中间，而是根据字母顺序估算位置
- **算法公式**（线性插值）：
```
位置 = low + ( (target - arr[low]) * (high - low) ) / (arr[high] - arr[low])
```
- **性能与限制**：
  - **最佳情况**：数据均匀分布时，可达 **O(log log N)**，比二分查找更快
  - **最坏情况**：数据分布不均匀时，退化为 O(N)
  - **实际使用**：较少通用，但对特定场景（如游戏引擎中均匀分布ID的查找）极快
- **严格的使用条件**：
  - **必须是整数键**（浮点数需额外处理，不推荐）
  - **必须已排序**
  - **必须无空缺（无间隙）**：即键值连续递增
  - 典型适用场景：**序列化主键**（如自增ID）

#### 三种搜索对比总结

| 方法 | 前提条件 | 速度特征 | 最佳应用场景 |
|------|---------|---------|------------|
| **线性搜索（SIMD）** | 无需排序 | 硬件加速后快于朴素线性 | 小数据集、需原地查找 |
| **二分查找** | 必须预排序 | O(log N) 稳定 | 大规模有序数据 |
| **插值查找** | 排序+已知分布 | 均匀分布时O(log log N) | 数值均匀分布且有大量查找需求 |

### 6. 专属于B+树的优化技术

#### 指针游走（Pointer Wastlay）
- **核心概念**：优化节点间指针遍历效率
- **应用场景**：减少存储开销，提升查询速度

#### 批量插入（Bulk Insert）
- **核心概念**：一次性插入大量数据而非逐条插入
- **优化原理**：
  - 减少树结构调整次数
  - 降低节点分裂开销
  - 提升整体写入性能
- **核心步骤**：
  1. **预排序所有键值**
  2. **按排序顺序排列叶子节点**
  3. **自底向上构建索引**（从叶子层向上搭建索引结构）
- **优化细节**：
  - 示例中在每个叶子节点**预留一个额外空间**
  - 某些系统会**压缩至100%填充**以最小化空间占用
  - 这种优化被称为**索引重建/优化**（Re-optimize on Index）

#### 缓冲更新（Buffered Updates）
- **核心概念**：延迟应用修改操作，累积后再批量处理
- **代表性实现**：**Bε树（B Epsilon Tree）** 或 **分形树（Fractal Tree）**
- **工作流程**：
  1. 修改操作不立即应用到树结构
  2. 将日志条目存入树中某个位置
  3. 积累足够多修改后，一次性压缩并应用变更
- **实际应用**：TokuTec（被Percona收购）拥有分形树索引专利，用于MySQL引擎

#### 前缀压缩（Prefix Compression）
- **核心思想**：多个键共享相同前缀时，只存储一次前缀
- **优化效果**：减少存储空间，提升缓存利用率

#### 去重（Deduplication）
- **核心概念**：相同键出现多次时，仅存储一次
- **应用场景**：减少冗余存储，提升查询效率

#### 后缀截断（Suffix Truncation）
- **核心概念**：在**内部节点**的指引项中，无需存储完整的键，只需存储**最小必要前缀**
- **工作原理**：
  - 仅保留足以**区分左右子树**的最小前缀
  - 例如：只需知道键值前缀能否判断"走左还是走右"
- **关键限制**：
  - 若两个字符串都以"ABC"开头，则需要**扩展**截断前缀或重新处理
- **目的**：进一步**减少内部节点的存储开销**

#### 指针交换（Pointer Swizzling）

**核心概念**：将页面ID替换为**真正的内存指针**，绕开 Buffer Pool Manager 的页表查找，实现从 Page ID 到内存地址的 **“直接跳转”**。

- **工作原理**：
    - 将 B+ 树节点中原来存储的 **Page ID**，替换为直接指向该页面在 Buffer Pool 中 **内存地址的指针**。
    - 这样，当遍历 B+ 树时，就可以直接通过指针跳转到目标节点，无需任何查询。

- **关键前提**：**页面必须被钉住在内存（Pinned）中**。
    - 只有当一个页面被钉住时，它才**保证不会被换出（Swapped Out）到磁盘**。这是指针混淆能够安全的绝对前提。
    - 如果页面可以被换出，那么内存指针会变成野指针，导致系统崩溃。

- **优势与性能提升**：
    - **消除查找开销**：无需经过缓冲池的页表查找
    - **直接内存访问**：直接从指针跳转到数据位置
    - **显著加速**：B+树遍历速度大幅提升

- **页面内的存储机制**：
    - **预留额外空间**：在页面头部或子节点指针区域，预先**分配一点额外空间**
    - **标记Swizzle状态**：当指针被Swizzle后，**反转一个bit或标记一个标志位**，表示该指针不再是一个普通的页标识符
    - **需要维护的簿记信息**：系统必须追踪哪些指针已经被Swizzle

- **磁盘持久化与失效处理**：
    - **换出到磁盘时的处理**：
      - 当页面被写回磁盘时，必须**使所有Swizzle指针失效**（invalidate）
      - 原因：Swizzle指针指向的是内存地址，当页面重新读取时，这些内存地址可能已经无效
      - **关键操作**：页面被重新读入内存时，确保**清除所有Swizzle地址的缓存**

- **传统 B+ 树访问（无指针混淆）的时间代价估算**：
$$
\text{Total Cost} = \sum_{\text{each node traversal}} \left( \text{Latch}_{\text{B+ Tree node}} + \text{Hash Lookup}_{\text{Page Table}} + \text{Latch}_{\text{Page Table}} \right)
$$

- **指针混淆优化后的时间代价**：
$$
\text{Total Cost} = \sum_{\text{each node traversal}} \left( \text{Latch}_{\text{B+ Tree node}} \right)
$$

### 总结：B+树设计决策矩阵

| 决策维度 | 主要考量因素 | 典型值/策略 | 核心目标 |
| :--- | :--- | :--- | :--- |
| **节点大小** | 存储设备速度 | HDD: 1MB, SSD: 10KB, Mem: 512B | 最大化顺序I/O/缓存命中 |
| **分裂/合并** | 空间利用率 vs 操作频率 | 懒惰合并 vs 即时分裂 | 减少树结构变动开销 |
| **锁粒度** | 并发线程数 | 小节点 + 乐观锁 / 大节点 + 范围锁 | 减少锁争用 |
| **变长键** | 键长度分布 | 前缀压缩 / 后缀截断 | 提高节点扇出，降低树高 |

> **工程建议**：在游戏引擎或实时渲染系统中，若在内存中实现B+树（如用于GPU剔除数据排序），**首选512B节点大小**，并配合**前缀压缩**。对于持久化场景（引擎资源管理），则按存储介质选节点大小。

## 二、B+树并发控制

> 主题：B+树并发访问的锁存机制、锁存爬行协议、乐观与悲观策略及死锁处理。

### 1. 并发控制协议基础

在数据库系统中，为了让多线程安全、高效地共享数据结构，需要一套规则来协调它们的操作。这套规则就是 **并发控制协议**。

#### 核心思想：协调与隔离

- **核心功能**：确保多个线程同时操作共享数据结构时，**不产生数据损坏**，并且每个线程的操作结果都**符合预期**。
- **前提假设**：大部分系统（如 VoltDB、Hekaton、Redis）内部是**单线程引擎**，它们天然避免了多线程竞争。但真正的多线程系统必须依赖并发控制协议。

#### 两大正确性标准

1.  **逻辑正确性**
    - **定义**：关注**数据内容的可见性**。即一个线程执行的操作（如插入键“Foo”）后，另一个线程是否能立即读到这个结果。
    - **作用**：保证数据库的逻辑语义（如事务的隔离性）不被破坏。

2.  **物理正确性**
    - **定义**：关注**数据结构内部表示的完整性**。即指针、页、节点等底层引用是否有效、安全。
    - **作用**：保护数据结构的**物理结构**（如B+树的节点、链表指针），防止程序崩溃或读到错误数据。

#### 本课程关注的重点

- **主要关注点**：**物理正确性**。
- **原因**：这是所有并发数据结构的基础和前提。如果底层指针都指向无效区域，逻辑正确性无从谈起。

### 2. 闩锁（Latch）基础

#### 核心概念：闩锁与锁的区别

- **逻辑正确性**：事务并发控制，确保"写入后能读取正确结果"——这是**锁（Lock）**的职责
- **物理正确性**：数据结构操作不出错，如不因野指针崩溃——这是**闩锁（Latch）**的职责

##### 锁（Lock）的特征
- **高层次逻辑保护**：保护事务/查询不被其他并发事务干扰
- **持有时间长**：可能在整个事务生命周期内持有
- **有死锁检测与处理**：系统具备回滚机制

##### 闩锁（Latch）的特征
- **低层次物理保护**：保护数据结构的临界区（如页、节点）
- **持有时间极短**：进入节点→执行操作→释放闩锁
- **无回滚机制**：若无法获取闩锁，直接中止并重试整个操作
- **核心目标**：防止线程间互相干扰导致数据结构损坏

#### 模式兼容性矩阵

| 线程1 \ 线程2 | 读模式 | 写模式 |
|--------------|--------|--------|
| **读模式** | ✅ 兼容 | ❌ 不兼容 |
| **写模式** | ❌ 不兼容 | ❌ 不兼容 |

**关键规则**：只要有任何一个线程持有写模式，无论其他线程想要什么模式，**都不能同时共享该闩锁**

#### 实现方案

##### 阻塞式OS互斥锁（Blocking OS Mutex）
- **核心思路**：使用标准互斥锁实现，但**只有一种模式**——独占锁
- **简单直观**，但功能有限（无法支持读模式共享）

##### futex（Fast User-space Mutex）
- **核心思想**：先在用户空间进行自旋锁尝试，无需陷入内核
- **两级架构**：
  - **用户空间自旋层**：快速获取，无需系统调用
  - **内核互斥层**：自旋失败后的重量级回退方案
- **实践建议**：
  - 课程学习阶段：可以使用标准互斥锁
  - **生产环境**：必须避免使用标准互斥锁

##### 读写锁（Reader-Writer Latch）
- **基本概念**：提供两种模式：
  - **读模式（Read Mode）**：多个线程可以同时持有
  - **写模式（Write Mode）**：独占访问
- **内部维护的关键计数器**：
  - 持有锁的线程数（读模式）
  - 等待锁的线程数
- **实践建议**：课程中使用标准库的 **pthread_rwlock_t** 即可

#### 读写锁的公平性机制

**核心概念**：关键在于**公平性算法**决定等待队列的行为。

- **重要行为**：当一个写者在等待时，后续的读请求也会被**阻塞**，即使读与读之间本可并行。这是为了**防止写者饿死**（writer starvation）
- **确保在任意时刻，**等待时间最长的写者**能够尽快获得锁

#### 读写锁的元数据开销

| 类型 | 元数据大小 | 说明 |
|------|-----------|------|
| **futex锁** | 单字节/32位 | 极简，可嵌入数据结构页 |
| **自旋锁（用户空间实现）** | 单字节/32位 | 极简，适合高性能场景 |
| **读写锁** | 队列+计数器 | **元数据大**，但换来了更好的并发性能 |

### 3. 哈希表加锁（Hash Table Latching）

#### 线性探测哈希表（Linear Probing Hash Table）

**特点**：最简单的哈希表实现，不需要处理复杂的并发，适合作为**加锁入门示例**

**关键设计模式**：
1. **每个槽位（slot）或桶（bucket）**独立加锁
2. 使用**写锁**进行插入/删除操作
3. 使用**读锁**进行查找操作
4. 线性探测时，需要**链式加锁**（逐步释放前面的锁，获取后面的锁）

#### 页级锁存 vs 槽级锁存

| 特性 | 页级锁存 | 槽级锁存 |
|------|---------|---------|
| **锁粒度** | 整个物理页 | 单个逻辑槽位 |
| **查找操作** | 必须获取页的读取锁 | 无需获取任何锁（检查自由状态） |
| **插入操作** | 必须获取页的写入锁（阻塞所有读取） | 仅对目标空闲槽位获取写入锁 |
| **跨页扫描** | 释放前一页的锁，获取下一页的锁 | 无需释放/获取页锁，直接移动槽位 |
| **并发性能** | 低（读写互斥） | 高（读写可并行，仅槽位级冲突） |
| **实现复杂度** | 简单（页级管理） | 更复杂（需管理每个槽位的状态和锁） |

### 4. B+树并发访问中的两大问题

- **写写冲突**：多个线程试图同时写入同一个节点。  
  **解决方案**：使用互斥锁存（如写锁存）保护节点，确保同一时间只有一个线程写入。
- **指针失效**：一个线程正在遍历索引层，而另一个线程在更低层进行结构性修改（如分裂、合并），导致上层线程跟随的指针指向无效位置。  
  **核心挑战**：需要确保遍历线程看到的指针始终有效。

### 5. 闩锁爬行（Latch Crabbing）协议

#### 核心思想与操作流程

- **本质**：一种**细粒度锁控制协议**，允许多线程**同时遍历和修改** B-Tree，同时保证**物理一致性**

- **基本操作流程**：
    1. **从根节点出发**，首先获取**父节点**上的闩锁
    2. 然后获取**要前往的子节点**上的闩锁
    3. 确认**父节点安全**后，**释放父节点的闩锁**
    4. 重复这个模式，**逐层向下**，直到到达叶节点

- **“安全”的定义**（取决于操作类型）：
    - **插入操作**：节点**未满**——即使下方发生分裂，父节点也能**容纳上推的新 key**
    - **删除操作**：节点**不会过度稀疏**——即使下方发生合并，父节点也不需要进行合并操作

#### 读操作时的锁存释放策略

- **核心策略**：**沿途释放锁存**（Release Latch Behind）
- 从根节点开始，逐层向下获取**读锁存**（Read Latch）
- 每获取下一层节点的锁存后，立即释放父节点的锁存
- 因为读操作不会修改树结构，所以**父节点锁存在子节点获取后即可安全释放**

#### 写操作时的锁存爬行（Latch Crabbing）

- **核心策略**：**锁存爬行**（就像螃蟹横着走一样，持有一个锁存的同时获取下一个）
- 获取父节点**独占锁存**（Exclusive Latch）后，在向下移动时**不能立即释放**
- **关键判断**：只有当确认当前节点**不会因为子节点的修改而触发分裂或合并**时，才能释放父节点锁存

#### 安全节点判定规则

- **对于删除操作**：节点key数 > 最小值（保证删除后不会触发合并）
- **对于插入操作**：节点key数 < 最大值（保证插入后不会触发分裂）
- 只有遇到**安全节点**时，才能释放父节点锁存

#### 锁释放决策矩阵

| 操作类型 | 何时可以释放父节点锁 | 必须持有的锁 |
|---------|-------------------|------------|
| **简单插入**（无分裂） | 立即释放 | 仅当前节点 |
| **插入导致分裂** | 直到确认分裂不会向上传播 | 从分裂点到根路径上的节点 |
| **删除导致合并** | 直到确认合并不会向上传播 | 从合并点到根路径上的节点 |

### 6. 乐观锁存与悲观锁存

#### 核心问题：根节点成为性能瓶颈

- **关键概念**：在传统的 B+ 树操作中，**每次访问都从根节点开始加写锁（Write Latch）**。
- **问题所在**：无论插入、删除还是读取操作，每次都必须获取根节点的锁存器。随着并发量增加，**根节点会成为巨大的争用瓶颈**。

#### 乐观锁存优化策略

**核心思想：**“假设罕见，快速通行”

- **基本假设**：B+ 树中引发 **分裂（Split）** 或 **合并（Merge）** 的操作是**罕见**的。
- **流程概述**：
    1.  从根节点到叶子节点，**一路只加读锁（Read Latch）**。
    2.  到达叶子节点后，**升级为写锁（Write Latch）**。
    3.  判断假设是否正确：当前叶子节点是否安全（无需分裂或合并）？
    4.  如果安全：直接执行修改操作，完成。
    5.  **如果不安全：放弃当前操作，从头开始重新遍历，但这次使用悲观的写锁策略**。

#### 关键优势

- **无需回滚**：因为在发现假设错误时，**尚未对数据做任何实际修改**，所以只需重新开始，**无需撤销（Undo）任何更改**。

#### 乐观锁的劣势：高竞争下的开销

- **高竞争问题**：当多个线程同时尝试修改数据结构时，乐观假设经常失败，导致**频繁的重启**和**重复遍历**，反而增加了开销。
- **权衡**：实际工程中，大多数工作负载下乐观锁仍是最优选择，但需根据具体场景评估。

### 7. 死锁处理策略

#### 核心设计哲学：**放弃死锁检测与预防，依靠程序员编程纪律**

- **关键决策**：基于锁存器的设计天生不支持死锁检测或预防机制
- **根本原因**：
    - 锁存器（latch）是轻量级同步原语，不具备死锁检测的复杂性
    - 数据库系统内核属于**高敏、高性能领域**，需要高水平程序员保证代码质量

#### 主动终止策略（Suicide Strategy）

**核心机制**：当线程发现自己可能导致或已经卷入死锁时，**主动释放持有的所有锁存器并重启操作**

- **关键参数**：超时时间（timeout）决定线程“自杀”的激进程度

#### 自适应超时优化

- **工作负载感知**：扫描过程中经过的节点越多，说明已做的工作量越大，此时可**增加超时时间**，减少激进自杀的概率
- **饥饿与浪费循环控制**：
    - **重启计数法**：记录当前操作已重启的次数
    - **退避休眠**：重启次数越多，等待时间越长
    - 公式化描述：$\text{sleep\_time} = f(\text{restart\_count}, \text{work\_done})$

#### 优先选择策略：读锁自杀优于写锁

- **基本原则**：持有**读锁的线程应主动自杀**而非写锁持有者
  - 读锁操作通常**计算量更小**，已做工作量少
  - 写锁操作通常涉及更复杂的数据修改

### 8. 并发扫描与Sibling指针

#### 核心问题：叶子节点扫描与并发控制

- **传统B+树**没有**sibling指针**，扫描叶子节点时只能“自顶向下”重复查找，导致两阶段操作：
  - **第一阶段**：从根节点遍历到叶子节点末尾，发现已到边界
  - **第二阶段**：重新从根节点遍历到下一个叶子节点
- **并发困境**：两次遍历之间，其他线程可能插入新节点，导致“乐观路径”失效

#### B-link树：引入Sibling指针

- **改进方案**：在每个叶子节点存储指向**右侧兄弟节点**的指针
- **扫描优化**：到达叶子节点末尾后，直接通过sibling指针跳转到下一个节点，无需重新从根遍历
- **并发安全**：
  - **右锁（right latch）** 保证物理正确性：持锁期间无人修改，避免数据丢失
  - **无需线程间通信**：不传递消息，避免性能损耗

#### 锁存器管理（关键）

- 当你从一个叶节点（Node C）跳转到其兄弟节点（Node B）时，**不能在跳转前释放 Node C 的锁存器**。
- **原因**：防止其他线程在跳转过程中修改 Node C 的兄弟指针，导致你跳转到一个已经失效的位置。
- **协议**：**“到达下一个节点之前，不得释放当前节点的锁存器”**。
    1.  持有 Node C 的读锁。
    2.  通过兄弟指针尝试获取 Node B 的读锁。
    3.  **成功获取 Node B 的读锁后**，再释放 Node C 的读锁。

### 9. 并发数据结构的选择：无锁 vs. 有锁

- **核心概念：无锁数据结构（Lock-Free Data Structures）并非万能灵药**
  - 像 **Rust** 这类强调内存安全的新语言，并不能解决所有复杂问题（如锁竞争、逻辑正确性）。
  - 无锁数据结构（如 **跳表（Skip List）**）在高竞争（high contention）场景下，性能往往不如精心实现的有锁数据结构。
  - **关键结论**：在大多数情况下，**一个写好的有锁数据结构（well-written locked data structure）在速度和正确性上都能碾压无锁版本**。

#### 高竞争下的性能瓶颈

- **核心原因**：在高竞争下，无锁数据结构需要大量 **自旋（spinning）** 操作，导致性能急剧下降。
- **具体例子**：**Bw树**（一种无锁B树变体）虽然是无锁的，但需要额外的查找表和复杂的“父指针交换”机制来维护一致性，远不如一个精心实现的有锁B树（B+Tree）高效。

#### 实际选择法则

- **有锁数据结构（Locked Data Structures）**：在绝大多数情况下，使用好的锁（如读写锁、分段锁）实现的哈希表、B树等数据结构，性能足够且正确性易于保证。
- **无锁数据结构**：仅在极端低延迟、不可阻塞的特定场景下才值得考虑，且实现复杂度极高。

### 10. 总结性权衡表

| 策略 | 适用场景 | 优点 | 缺点 |
|------|----------|------|------|
| **乐观锁（读锁为主）** | 读多写少、节点分裂概率低 | 并发度高，读操作几乎无阻塞 | 写冲突时需回退重试，增加开销 |
| **悲观锁（写锁为主）** | 写密集、节点分裂频繁 | 避免回退重试，确保一致性 | 锁竞争激烈，降低读并发 |
| **混合策略（先乐观后悲观）** | 通用场景（课程推荐） | 平衡性能与正确性 | 需处理“重试窗口”竞争条件 |
