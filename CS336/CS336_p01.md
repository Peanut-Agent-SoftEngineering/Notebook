## 课程核心：从零构建语言模型的必要性

### **核心理念：亲手构建，才能真正理解**

- **课程定位：** CS336，覆盖**从数据到模型**的完整流程，包括**数据系统**和**建模**。
- **关键信念：** 理解技术必须**亲手构建**，从零开始走通全过程，而非仅停留在API调用层面。
- **教学方式：** 强调实践，学生将完整经历数据准备、系统搭建、模型训练等环节。

### **开设此课程的深层动机：应对“抽象层危机”**

#### **背景：研究人员与底层技术渐行渐远**
- **8年前：** 研究者自己实现并训练AI模型。
- **6年前：** 至少会使用BERT等开源模型，**下载并微调**。
- **现在（授课时）：** 大量研究者仅**调用专有模型API**，停留在**提示工程**层面。

#### **问题：抽象层带来便利，但也存在“漏洞（漏洞）”**
- 如同编程语言或操作系统，**抽象层**（如API调用）虽让更多人能做事，但**掩盖了底层真实机制**。
- 这种“理解不透彻”的状态对**基础研究**构成障碍。

#### **为什么坚持“从零构建”？**
- **基础研究的需要：** 真正推动技术前沿，需要能够**撕毁堆栈**，协同设计**数据、系统、模型**多个层面。
- **工业化的现实：** 大型模型（如GPT-4，约1.8万亿参数）的**训练成本极高**（传言达1亿美元，集群投入超500亿美元），且**公开细节极少**（受竞争和安全限制）。唯有亲手实践，才能摆脱“黑箱”依赖。

## 第 2 部分

### 规模化差异：小模型 vs. 大模型的关键行为变化

*   **核心概念：小规模实验的局限性**
    *   课程中构建的是 **小型语言模型**，但现实世界（如GPT-4）是 **前沿大模型**。两者在行为上存在本质差异，不能简单将小模型的经验直接套用到大模型上。
    *   **关键术语：** **涌现行为 (Emergent Behavior)**、**规模效应 (Scaling Effects)**。

*   **现象一：计算资源消耗 (FLOPs) 的比例漂移**
    *   **核心差异：** 在小模型中，**注意力层 (Attention)** 和 **多层感知机层 (MLP)** 消耗的 **FLOPs（浮点运算数）** 大致相当。但在 **千亿参数级** 模型（如175B）中，**MLP层的FLOPs占据绝对主导地位**。
    *   **工程师洞察：** 如果你一直专注于优化小模型的注意力机制，当模型规模放大时，这些优化的收益会被稀释。因为瓶颈已经转移到了MLP层。
    *   **方法论：** 这种差异可以通过**草稿纸上的数学推导**预先估算出来，无需实际运行代码。

*   **现象二：涌现行为的不可见性**
    *   **核心概念：** 某些能力（如**上下文学习 (In-Context Learning)**）在模型达到一定规模前，性能表现近乎随机。一旦规模超过某个阈值（例如更高的训练FLOPs），准确率会**突然跃升**。
    *   **陷阱：** 如果实验只停留在小规模，你会错误地得出结论，认为模型不具备这些能力。实际上，它们是**规模扩展后才会涌现**的属性。

### 本课程能传授的知识分类

尽管小模型有局限性，但课程旨在提供三类有价值的知识，而非单纯复刻大模型：

1.  **机制原理 (Mechanisms) - 可直接传授**
    *   **内容：** 实现Transformer、高效利用GPU进行模型并行等**基础要素**。
    *   **价值：** 这是发动机的制造手册，是硬技能。

2.  **思维方式 (Mindset) - 核心培养目标**
    *   **内容：** **压榨硬件极限**的极致追求，以及**认真对待扩展性**的思维方式。
    *   **关键洞察：** 推动AI发展的关键并非全新的架构，而是由OpenAI开创的**扩展性思维**。所有技术组件早已存在，但“如何规模性地思考”才是变革的引擎。

3.  **直觉 (Intuition) - 部分传授**
    *   **内容：** 哪些**数据**和**建模决策**能产生优质模型。
    *   **局限：** 这部分只能**部分传授**，因为在小规模上有效的架构或数据集，在大规模下可能失效。
    *   **价值：** 即使只能掌握“两点五”的知识，性价比依然极高，因为它让你具备了做出正确方向性判断的直觉。

### 研究直觉的现实：实验结果决定一切

*   **核心观点：** 许多成功的架构选择并非源于理论推导，而是**实验后结果说话**。
*   **典型案例：SwiGLU 激活函数**
    *   它是一种在课程中会讲到的**非线性激活函数**。
    *   它被广泛采用的原因是 **“效果相当不错”** ，而非因为一篇论文提供了完美的理论解释。研究者诚实地承认：“我们无法解释”，但实验证明了它的价值。

## 对“苦涩的教训”的正确理解：规模化算法是关键

### 常见误解：规模是唯一的救世主

*   **流行观点**：“苦涩的教训”意味着**规模（scale）才是关键**，算法不重要，只要不断投入资金扩大模型和数据，就能成功。
*   **反驳**：这种看法**与事实相去甚远**。单纯的资源堆砌无法解决效率问题。

### 正确思维：规模化算法（Scalable Algorithms）的决定性作用

*   **核心公式**：模型最终准确度 ≈ **投入资源数量 × 实际效率**。
*   **关键洞察**：在**大规模投入下，效率比任何时候都重要**。
    *   当花费数亿美元时，**无法承受低效带来的浪费**（如失败的集群任务、调试成本）。
    *   开源方案通常比现有方案更高效，意味着效率仍有巨大提升空间。

### 算法效率的量化证据

*   **效率本质**：效率是**硬件与算法的结合**。
*   **算法效率提升（2012-2019）**：一篇2020年开源论文显示，在训练ImageNet达到特定精度所需的时间里，**算法效率提升了44倍**。
*   **显著意义**：这一**算法效率的改进速度已超过摩尔定律**（硬件提升速度）。
    *   若没有这种算法效率改进，成本将增加44倍。
    *   语言模型领域亦有类似结论。

### 研究人员的核心目标

*   **正确的思维模式**：在**给定的计算预算和数据预算下，构建最佳的模型**。
    *   无论规模大小，这本质上是**资源利用率与精度的优化问题**。
    *   筹集更多资源固然能获得更好模型，但**研究人员的目标是提升算法效率**（最大化效率）。

## 语言模型的历史回顾与关键转折

### 早期形式：n-gram 模型

*   **起源**：语言模型起源于**克劳德·香农**将语言模型视为**估算英语熵值**的方法。
*   **应用**：在自然语言处理中，作为机器翻译、语音识别等大型系统的一部分。
*   **一个里程碑（2017年左右）**：谷歌正在训练**非常大的n-gram模型**（五元语法），覆盖**两万亿个标记（tokens）**。
    *   **对比**：这一标记数量**远超GPT-3**（直到约两年前才达到该量级）。
    *   **关键局限**：n-gram模型**未能展现当今语言模型所具有的“有趣现象”**（如涌现能力）。

### 深度学习革命：关键要素逐步到位

*   **首个神经语言模型**：**约书亚·本吉奥（Yoshua Bengio）团队**（2003年）。
*   **序列建模**：**伊利亚（Ilya Sutskever）与谷歌团队**开发了有效建模序列的方法。
*   **优化器**：**原子优化器（Atom Optimizer）**（十年前多数人的选择）。
*   **注意力机制**：在机器翻译背景下开发出**张力机制**，最终演变为著名的**注意力机制**。
*   **核心论文**：这一切最终导致了 **“注意力就是一切”（Attention Is All You Need）**——即Transformer论文的诞生。

### 基础单元详解：实现一个基本完整流程

**目标**：在基础单元中，你需要实现一个**基本完整的流程**，涵盖以下三个关键组件。

#### 1. 分词器（Tokenizer）

- **核心概念**：**分词器** 的作用是将 **字符串转换为整数序列**。
- **工作机制**：
    - 将文本字符串分割成一个个**片段**（Subwords）。
    - 将每个片段**映射**为一个唯一的整数 ID。
- **关键约束**：输入的整数序列必须**保持固定维度**，才能输入到模型中进行处理。
- **重点算法**：**BPE（Byte Pair Encoding）编码**。
    - 是一种相对**简单**且至今仍被**广泛使用**的 Tokenizer 算法。
    - 它通过学习合并最频繁的字节对来构建词汇表，是当前主流方法变体的基础。

## 第 7 部分

### 分词与模型架构基石：从BPE到Transformer的改进

#### 1. 分词策略：BPE与免分词前沿
- **核心概念**：在将文本输入模型前，必须先将其**转换为整数序列**。主流方法为**BPE（Byte Pair Encoding）分词器**。
- **BPE特点**：相对简单，被广泛采用（几乎所有前沿模型的基础）。
- **前沿探索**：存在**无需分词的免分词方法**，直接从**原始字节**处理，设计特定架构。虽有潜力，但**尚未扩展到前沿规模**，当前仍以BPE为主流。

#### 2. 模型架构的演进：从原始Transformer到现代变体
- **起点**：**原始Transformer模型**是几乎所有前沿模型的基础。
- **核心不变**：核心架构（注意力机制 + MLP层 + 层归一化）自2017年以来基本保持，但**大量小改进的累积**产生了显著的性能差异。

- **关键改进领域及变体**：

| 组件 | 原始形式 | 现代改进与技术术语 | 核心思想 |
| :--- | :--- | :--- | :--- |
| **位置编码** | 波状绝对位置嵌入（如Sinusoidal） | **旋转位置嵌入（RoPE）** | 更有效地让模型感知相对位置关系。 |
| **归一化** | 层归一化（LayerNorm） | **RMS归一化（RMSNorm）** | 更简单的归一化方法，计算效率更高。 |
| **归一化位置** | 层输出之后归一化（Post-Norm） | 归一化位置改变（Pre-Norm等） | 将归一化放在残差连接之前，训练更稳定。 |
| **MLP层** | 标准密集MLP | **专家混合（Mixture of Experts, MoE）** | 用多个稀疏的“专家”网络替换单个密集MLP，扩展参数容量而不显著增加计算量。 |
| **注意力机制** | 标准全注意力（O(n²)复杂度） | 滑动窗口注意力、线性注意力、GQA、MLA | 旨在**防止二次爆炸**的复杂度，提高长序列处理能力。 |
| **整体架构** | Transformer | **状态空间模型（如Hyena）**、混合模型 | 完全替代注意力机制，或用其他操作（如卷积）与Transformer混合，结合不同架构的优势。 |

#### 3. 训练策略：优化器与关键超参数
- **核心优化器**：**Adam**及其变体（如AdamW）仍是绝对主流，经过多年验证与改进。
- **新兴优化器**：**Muon**、**Soap**等近期优化器已展现出潜力，值得关注。
- **关键微调因素（细节决定成败）**：
    - **学习率调度**（Learning Rate Schedule）
    - **批量大小**（Batch Size）
    - **正则化**（Regularization）策略
    - **超参数**（Hyperparameters）的调优

- **重要提示**：**一个调优良好的模型与一个普通模型之间，可能存在数量级的性能差异**，这比架构选择本身更重要。

#### 4. 实践环节：实现一个简易的“Transformer From Scratch”
- **任务核心**：实现完整的GPT式技术栈，但**限制你使用PyTorch的高阶封装**（如`torch.nn.Transformer`），只允许使用基础函数。
- **必须实现**：
    1.  **BPE分词器**（Tokenizer）
    2.  **Transformer模型**（简化版，包括注意力、MLP等）
    3.  **交叉熵损失函数**（Cross-Entropy Loss）
    4.  **Adam优化器**（Adam Optimizer）
    5.  **训练循环**（Training Loop）
- **数据集与评测**：
    - 提供**开放网络文本数据集**（OpenWebText）。
    - 目标是**最小化验证集上的困惑度（Perplexity）**。
- **竞赛机制**：
    - 90分钟H100 GPU的训练时间作为基准。
    - 设有排行榜（Leaderboard），以此检验模型与训练的质量。

## 第 8 部分

### GPU 性能优化：计算、并行与推理

本单元聚焦于充分利用硬件性能的三个核心组件：**内核 (Kernels)**、**并行性 (Parallelism)** 和 **推理 (Inference)**。

#### 1. 内核：最大化单 GPU 计算效率

**核心挑战：** 数据传输是瓶颈，计算是工厂，数据是仓库。如何组织计算以减少数据搬运，最大化 GPU 利用率？

- **GPU 硬件结构概览**：
    - **计算核心阵列**：GPU 芯片由大量执行浮点运算的单元组成。
    - **内存层级**：
        - **片外内存**：主存，容量大但访问延迟高。
        - **片上缓存**：L2 和 L1 缓存，访问速度更快。
    - **核心原则**：计算必须在芯片上进行，数据可能存储在片外。**组织计算的本质是管理数据从内存到计算单元的流动**。

- **关键优化技术：融合 (Fusion) 与分块 (Tiling)**
    - **目标**：减少数据传输次数和距离。
    - **应用场景**：矩阵乘法等计算密集型操作。
    - **实现手段**：将多个连续操作合并（融合），或将数据分块加载到片上缓存再计算（分块），以避免反复读取片外内存。
    - **工具**：使用 **Triton**（OpenAI 开发）来编写高效的 GPU 内核，它提供了高层次的抽象来构建这些优化操作。

#### 2. 并行性：多 GPU 规模的部署策略

**核心挑战：** 当模型和数据规模超过单 GPU 时，如何在多个 GPU（GPU-to-GPU 连接，如 NVLink）及 GPU 与 CPU 节点（通过交换机连接）之间，高效地拆分任务并管理数据移动。**核心理念与单 GPU 一致：最小化数据移动。** 只不过这里的“数据移动”指的是跨 GPU 或跨节点的通信。

- **并行策略类型**：
    - **数据并行 (Data Parallelism)**：每个 GPU 持有完整的模型副本，处理不同的数据批次。梯度在 GPU 间同步。适合数据量大、模型能放入单卡的情况。
    - **张量并行 (Tensor Parallelism)**：将单个张量（如权重矩阵）切分到多个 GPU 上，每个 GPU 负责计算一部分。适合单层模型过大无法放入单卡。
    - （其他策略如 Pipeline Parallelism 等会在后续内容展开）

#### 3. 推理：部署与持续运行的成本核心

**核心挑战：** 推理成本随用户数量线性增长，正超越训练成本成为主要开销。**目标是降低每次推理的计算和内存开销。**

- **为什么推理如此重要？**
    - **使用场景**：文本生成、强化学习、测试时计算、模型评估等。
    - **成本特性**：训练是**一次性**巨额成本，推理是**持续线性增长**的成本（用户越多，成本越高）。

- **推理的两个关键阶段**：
    - **预填充 (Prefill / Pre-filling)**：并行处理输入的 prompt，生成第一个 token。
    - **解码 (Decoding)**：**自回归**地逐个生成后续 token。这是推理中主要的时间和计算瓶颈，因为每个新 token 的生成都依赖之前的所有 token，难以完全并行化。

## 第 9 部分

### 推理效率与计算权衡

#### 1. 推理阶段的两大核心阶段

*   **预填充（Prefill）阶段**
    *   **核心概念：** 一次性处理整个**输入提示（Prompt）**，计算所有中间**激活值（Activations）**。
    *   **关键特性：** **天然并行**。所有 Token 在输入时已知，因此可以像训练时一样，利用 GPU 的并行计算能力一次性处理。
    *   **瓶颈：** 主要受限于**计算能力**（Compute-bound）。

*   **解码（Decoding）阶段**
    *   **核心概念：** **自回归（Autoregressive）** 逐 Token 生成过程。当前 Token 生成依赖于之前的所有 Token。
    *   **关键特性：** **天然串行**。难以充分利用 GPU，导致效率低下。
    *   **瓶颈：** 主要受限于**内存带宽**（Memory-bound）。因为每一步都需要频繁地在内存和计算单元之间移动模型权重和中间数据。

#### 2. 加速推理的三大策略

*   **策略一：使用更便宜的模型（Cheaper Models）**
    *   **核心思路：** 直接用一个计算成本更低、规模更小的模型替代大模型。
    *   **代价：** 通常会牺牲一部分生成质量或准确性。

*   **策略二：推测解码（Speculative Decoding）**
    *   **核心思路：** 引入“双模型”机制，实现“以小博大”。
        1.  **草稿生成：** 使用一个**低成本草稿模型**快速预测生成多个候选 Token。
        *   **验证与接受：** 使用**原始完整模型**对这些候选 Token 进行**并行评分**（Parallel Scoring）。
        *   **核心概念：** 如果草稿模型的预测结果符合某种“优质标准”（例如，与完整模型的评分一致），则完整模型可以一次性接受所有草稿 Token，从而变相实现“并行生成”。
    *   **收益：** 在保证最终生成质量（与原始模型无异）的前提下，大幅提升解码速度。

*   **策略三：系统级优化（System-level Optimizations）**
    *   **数据并行：** 最直观的并行方式，将模型副本分布在多个 GPU 上，各自处理不同数据批次。
    *   **模型并行（如 SDP 等复杂方法）：** 将单个模型切分到多个 GPU 上。课程会讲解完整概念，但实现复杂，建议先学习再尝试。

#### 3. 关键实践：基准测试与分析（Benchmarking & Analysis）

*   **核心观点：** 相较于单纯实现功能，**养成基准测试和分析的习惯**是更重要的能力。
*   **价值：** 不了解“我的实现效果如何？”以及“瓶颈在哪里？”，就好比“盲人摸象”，无法进行有效的优化和性能提升。

---

### Scaling Laws：最佳模型规模与数据量匹配

#### 1. 核心问题：给定计算预算，如何分配？

*   **根本矛盾：**
    *   **大模型 + 少量数据**：模型能力可能无法被充分挖掘（欠训练）。
    *   **小模型 + 大量数据**：数据无法被模型完全吸收（模型容量不足）。
*   **目标：** 寻找对于特定计算预算（以 FLOPs 衡量），**模型参数量**与**训练数据量**之间的最优平衡点。

#### 2. Chinchilla 最优（Chinchilla Optimal）

*   **核心概念：** 指由 DeepMind 的 Chinchilla 论文等研究提出的、关于**最优计算分配**的指导原则。
*   **规律发现：**
    *   对于给定的 FLOPs 预算，可以调整模型参数量，并测量模型性能。
    *   将不同计算层级下的**最优参数量**绘制成图，会呈现出一条惊人的**线性关系**。

#### 3. 非常实用的经验法则

*   **核心公式/法则：**
    > 对于特定参数量的模型 $N$，其最优训练 Token 数量 $D$ 应满足：
    > $$D \approx 20 \times N$$

*   **例子：** 一个 **14亿（1.4B）** 参数的模型，应该用大约 **280亿（28B）** 个 Token 进行训练。
*   **局限性：** 该法则主要关注**训练成本**，**并未考虑推理阶段的计算成本**。在实际应用中，可能需要根据推理服务的部署成本进行微调。

## 第 10 部分

## 神经网络训练与数据工程

### 训练API与超参数实验

- **核心概念：将训练过程抽象为一个“训练API”**
    - 你可以通过这个API指定模型架构、超参数（如批量大小）等配置。
    - 每次查询返回一个**损失值**作为反馈，你的决策直接影响模型训练效果。
    - 任务本质：在**浮点运算预算（FLOPS budget）** 限制下，通过实验收集数据，拟合**缩放定律（Scaling Laws）**，并预测在更大规模下应选择的超参数与模型规模。

- **关键任务约束：**
    - **计算资源有限**：类似于前沿实验室的真实场景，一旦FLOPS预算耗尽，实验即终止。
    - **优先级规划**：必须谨慎决定要运行哪些实验，目标是**最小化最终损失**，在预算内找到最优配置。
    - **排行榜机制**：任务会有一个排行榜，核心指标是在给定浮点预算下，损失值最小化。

### 数据：模型行为的决定性因素

- **核心观点：数据决定了模型的行为**
    - 模型的行为**主要由数据决定**（并非全部，但极其关键）。
    - 训练数据的构成直接决定了模型的能力：多语言数据 -> 多语言能力；代码数据 -> 代码能力。
    - **数据来源多样化**：典型数据集由大量不同来源的片段混合而成。
        - **通用网络爬虫数据**（例如 Common Crawl）
        - **邮件交换数据**
        - **维基百科**
        - **GitHub 及其他精选来源**

- **数据整理（Data Curation）的重要性**
    - **常见误区**：认为模型训练直接使用“互联网数据”。实际上，数据不会从天而降，必须通过**主动获取和整理**。
    - **数据质量差异巨大**：以Common Crawl为例，随机抽取的文档可能渲染异常，结构混乱，不能直接用于训练。
    - **脏数据示例**：随机抽样Common Crawl文档可能显示为乱码、无意义字符或格式错误。

### 评估体系：如何衡量模型效果

- **标准指标：困惑度（Perplexity）**
    - 类似于模型的“标准化测试”，用于量化模型对语言的预测能力。
    - 困惑度越低，说明模型对给定数据序列的预测越准确。

- **评估中的决策点：**
    - **指令遵循能力**：如何评估模型生成符合指令语句的效果？这涉及许多主观和策略性决策。
    - **推理增强技术的影响**：在测试时使用**思维链（Chain-of-Thought）** 或集成方法，如何影响评估结果？
    - **系统级评估**：现代语言模型常被嵌入更复杂的系统中，评估不能仅停留在模型本身，需要考虑**整个系统的综合表现**。

## 第 11 部分

### 对齐（Alignment）：让基础模型变得有用

#### 核心概念：从“预测下一个词”到“理解并执行指令”
- **基础模型**：经过海量数据（如Common Crawl）训练后，模型能够准确预测下一个token。但这**不等于**它能遵循人类指令。
- **对齐（Alignment）**：将基础模型改造为**有用、安全、可交互**的系统。其核心目标包括：
  - **指令遵循**：正确执行用户给出的提示（如“用项目符号列出要点”）。
  - **风格控制**：根据要求调整输出长度、语气（机智、俏皮、简洁等）。
  - **安全性**：模型必须能拒绝回答有害或危险的请求。

#### 数据来源与处理：对齐的前提
- **原始数据的质量问题**：
  - **Common Crawl**虽然规模巨大，但包含大量**垃圾网站、非目标语言、噪声内容**。
  - 原始数据是**非纯文本**的（HTML、PDF、代码目录），必须经过**显式转换**（如HTML→文本），这是一个**有损过程**，需要平衡内容保留与结构去除。
- **数据清洗的关键步骤**：
  - **分类器训练**：用于识别和过滤掉低质量、有害或无关内容。
  - **去重（Deduplication）**：移除重复数据，减少冗余，提升训练效率。
- **前沿模型的现实**：如今顶级模型需要**购买数据**，因为公开可抓取的数据质量已无法满足顶尖表现需求。

#### 对齐的两阶段方法

##### 1. 有监督微调（Supervised Fine-Tuning, SFT）
- **目的**：让基础模型学会“用户-助手”对话模式。
- **操作**：收集一组**提示-响应对**（用户提问，期望的助手回答），然后进行标准的监督学习。
- **效果**：只需**少量示例（如一至两千个）** 即可使模型具备基本的指令跟随能力。示例越多，效果越好，但论文表明少量微调已足够。

##### 2. （隐含阶段：强化学习与人类反馈，RLHF）
- 虽然本段未详细展开，但通常**有监督微调之后**会进入**基于人类反馈的强化学习（RLHF）** 阶段，以进一步优化模型在偏好、安全性和长尾指令上的表现。

#### 关键公式/算法
- 本段未涉及具体公式，但可提炼出关键概念：
  - **损失函数**：在SFT阶段，通常使用**交叉熵损失**，优化模型对正确响应的预测概率。
  - **Token预算（Token Budget）**：在作业场景中，给定一个总Token数量，需要**在预算内最小化困惑度（Perplexity）**，这是对数据选择策略的直接量化考核。

#### 实践要点（来自作业背景）
- **作业四**提供一个原始Common Crawl数据集，学生需完成：
  1. 训练分类器进行数据过滤。
  2. 执行去重。
  3. 在给定Token预算下，通过选择最有效的数据组合来**最小化最终模型的困惑度**。

---

### 总结：对齐的关键脉络图
1. **基础模型潜力巨大**（能预测token），但**原始**。
2. **数据清洗是基石**（从HTML到文本，去噪，去重）。
3. **有监督微调是第一步**（少量示例即可“教会”指令遵循）。
4. **对齐的最终目标是“有用”**：能正确理解指令、控制风格、并保证安全。

## 第 12 部分

### 从监督微调到基于反馈的学习：核心脉络与关键算法

#### 1. 监督微调（SFT）的局限性
- **核心概念**：SFT 阶段虽然能产出不错的模型，但其性能提升严重依赖**高质量、大规模的人工标注数据**。
- **关键瓶颈**：获取更多 SFT 数据的成本极高——需要人工坐下来逐条标注指令和答案，即 **“标注疲劳”**。
- **转向动机**：为了用更轻量的标注形式（如偏好、验证器），让算法承担更多工作，从而扩展模型能力。

#### 2. 两种主要的“轻量标注”模式
- **偏好数据**：一种更经济的标注形式。
    - **数据形式**：给定一个提示，由模型生成 **A、B 两个响应**，然后让标注者（或用户）直接判断哪个更好。
    - **示例**：对“训练语言模型的最佳方法是什么？”这个问题，A 答案优于 B 答案。
    - **本质**：不需要完美的答案，只需要**相对排序**，标注成本极低。

- **验证器数据**：一种更硬核的自动评估方式。
    - **验证器类型**：
        - **形式验证器**：在数学或代码领域，可以通过程序（如编译器、定理证明器）自动判断答案对错。
        - **学习型验证器**：训练另一个语言模型作为裁判，来评估响应的质量（但本身仍存在评估准确性问题）。

#### 3. 关键算法与强化学习
- **背景**：基于反馈的学习本质上是 **强化学习（RL）** 在语言模型中的应用。目标是根据某种奖励信号（偏好或验证器分数）来优化模型参数。

- **PPO（近端策略优化）**：
    - 最早应用于指令调优模型的 RL 算法之一，但训练复杂，需要**引入价值函数（Value Function）**，计算开销大。

- **DPO（直接偏好优化）**：
    - **核心思想**：**简化 PPO**。不再需要显式的奖励模型和强化学习循环，直接从偏好数据中推导出策略梯度的解析式。
    - **优势**：数据形式更简单（只需要 A/B 偏好对），训练更稳定、更高效，且效果出色。
    - **重要性**：证明了“从偏好数据中学习”可以绕过复杂的 RL 系统。

- **GRPO（群体相对偏好优化）**：
    - **核心思想**：**进一步简化 PPO**。
    - **关键改进**：移除 PPO 中的 **价值函数（Value Function）**，通过从一组模型输出（群体）中计算相对优势来替代。
    - **优势**：效率更高，减少内存占用，非常适合在有限硬件上训练。

### 分词 (Tokenization) 基础与设计动机

#### 核心概念与动机
- **核心目的**：将原始文本（通常是 **Unicode 字符串**）转换为**整数集合**，每个整数代表一个 **分词 (Token)**。这是模型处理文本的输入输出接口。
- **效率驱动**：现代模型（如 Transformer）的架构计算效率低下，因此必须通过**分词**来提升模型处理文本的效率。这本质上是出于**训练效率**和**计算资源优化**的考虑。
- **设计哲学**：分词器的设计决策反映了当前阶段的计算资源约束。例如：
  - **单轮训练**：在有限计算资源下，倾向于用更少的计算资源确定超参数，而不是在单个数据点上花大量时间。
  - **数据限制**：前沿实验室正面临数据限制，这可能导致设计决策改变。例如，如果有更多计算资源，可能会采用**不同的架构**（而非Transformer），或者进行**更聪明的多轮训练**。

#### 应用场景与模型规模
- **狭窄应用场景**：可以使用**更小的基础模型**，通过**对齐 (Alignment)** 或**微调**即可表现良好，资源投入集中在对齐上。
- **广泛应用场景**：无法替代训练**大型模型**，必须投入大量资源进行预训练。

#### 分词器的核心功能
- **编码 (Encode)**：将字符串（文本）转换为整数列表。
- **解码 (Decode)**：将整数列表恢复为原始字符串。
- **词汇表大小**：表示**分词可能取值的数量**，即整数的取值范围。它决定了模型能处理的离散符号数量。

#### 分词器实际示例
- **可视化工具**：可以通过在线网站查看不同分词器如何分解文本。输入如 `"Hello"` 或 `"你好"`，它会显示：
  - **整数列表**：分词器输出的整数序列。
  - **片段分解**：原始字符串被分解为多个片段（子词或字符）。
- **重要细节**：在当代分词器中，**空格是分词的一部分**，这与传统 **NLP (自然语言处理)** 不同。传统方法通常将空格作为分隔符忽略，而现代分词器保留空格，使其成为一个独立的 Token 或嵌入在其他 Token 中。

## 第 14 部分

### 分词器（Tokenization）深入解析

#### 核心概念：从字符串到索引的映射

*   **基本工作流程**：分词器将原始字符串分解为多个片段（Token），然后将每个片段映射为一个整数索引。这个过程必须**可逆**，即能够从索引序列完美还原原始字符串。
*   **关键观察：空格的处理**：
    *   **空格是分词的一部分**，不会被忽略。这与传统自然语言处理（NLP）不同，后者通常先去除空格。
    *   **惯例：空格通常位于Token的前面**。例如，` hello` 和 `hello` 是两个完全不同的Token。这种设计是BPE（字节对编码）过程的产物，而不是故意为之，但已成为事实标准。
    *   **原因**：预分词器会将空格置于Token起始位置，这是算法内置的“约定”。理论上放在末尾也可以，但放在开头更合理。
*   **数字的分词**：数字是从左到右依次处理的，**不会按千位或语义分组**。例如，`12345` 可能会被切分成 `12`、`34`、`5` 这样的片段，而不是按数位意义分组。

#### 示例：GPT-4o分词器

*   **观察指标：压缩率**：
    *   **定义**：**压缩率 = 字节数 / Token数**，即每个Token平均代表多少字节的数据。
    *   **GPT-4o的压缩率**：大约是 **1.6**，意味着每个Token平均携带1.6字节的信息。
*   **操作流程**：
    1.  **预分词**：输入字符串，先进行预处理（例如，将空格前置）。
    2.  **BPE分词**：对每个预处理后的部分应用字节对编码算法，得到索引。
    3.  **验证**：解码索引序列，确保能还原原始字符串，验证双向转换的正确性。

### 两种原始分词方法的比较

#### 1. 基于字符的分词（Character-based Tokenization）

*   **原理**：将每个Unicode字符映射为其码点（Code Point, 如 `a → 97`, 地球emoji → 127757）。
*   **核心公式**：
    *   字符串 $S = \{c_1, c_2, \dots, c_n\}$，其中 $c_i$ 是Unicode字符。
    *   Token序列 $T = \{code\_point(c_1), code\_point(c_2), \dots, code\_point(c_n)\}$。
*   **问题**：
    *   **低效压缩**：**压缩率接近 1:1**（因为一个字符可能由多个字节组成，实际为 ~1.5字节/Token），远不如GPT-4o的1.6。
    *   **词汇表浪费**：词汇表大小为所有可能Unicode码点的数量（超过百万），其中**罕见字符占用了大量槽位**，而常见字符（如字母）利用率低。这是对词汇表预算的低效使用。
    *   **码点数值过大**：某些字符串码点数值巨大，处理不便。

#### 2. 基于字节的分词（Byte-based Tokenization）

*   **原理**：将Unicode字符串表示为**字节序列**（例如UTF-8编码），然后逐字节进行分词。
*   **优势**：
    *   词汇表大小固定在256（0-255），非常紧凑。
    *   能处理任何Unicode字符串，因为所有字符串都能转换为字节序列。
*   **问题**：
    *   压缩率依然较低，因为**单字节能表达的信息有限**。
    *   无法利用语言中的高频子词模式（如常见前缀、后缀）。

---

### 核心原理总结

| 方法 | 分词单元 | 词汇表大小 | 压缩率（字节/Token） | 核心瓶颈 |
| :--- | :--- | :--- | :--- | :--- |
| **基于字符** | Unicode码点 | ~143,000+ | ~1.5 (很低) | 词汇表庞大且浪费，罕见字符占据大量槽位 |
| **基于字节** | 单字节 (0-255) | 256 | ~1.0 (很低) | 无法捕获语言子词模式，信息密度低 |
| **BPE (如GPT-4o)** | 高频子词 | 可控 (如50k) | **~1.6** (较优) | 平衡词汇表大小与信息密度，捕获常见模式 |

**关键洞察**：分词器的核心设计目标是**在有限的词汇表预算下，最大化信息压缩率**。基于字符和基于字节的方法虽然简单，但压缩率低且效率不足。BPE通过迭代合并高频子词，在两者之间取得了良好平衡。

## 第 15 部分

### 分词技术的演进与挑战：从字节到BPE

#### 1. 基于字节的分词 (Byte-level Tokenization)

*   **核心概念**：将 **Unicode字符串** 编码为 **UTF-8字节流**，然后以固定大小的 **字节块**（例如，每2个字节）作为 **Token**。
*   **工作流程**：
    1.  将文本转换为UTF-8字节序列（每个字符1-4个字节）。
    2.  将连续的字节分组，形成Token。
*   **“理想”的优点**：
    *   **词汇表极小**：因为任何字节的值都在0-255之间，词汇表大小被严格限制在256（或根据分组大小，如65536）。
    *   **无未知词**：任何文本都可以用字节表示，不会遇到 `<unk>` 问题。
    *   **优雅且通用**。
*   **致命缺陷：序列过长，效率灾难**
    *   **压缩比极差**：**每Token只包含1个字节**，压缩比为1。
    *   **注意力机制二次方复杂性**：序列长度 $L$ 导致注意力计算复杂度为 $O(L^2)$。序列/字节数极长，会使模型计算成本变得**不可接受**。

#### 2. 基于单词的分词 (Word-based Tokenization): 解决问题的错误直觉

*   **核心直觉**：不同Token应能表示不同长度的文本（一个单词 vs 一个字母）。这是一种“适应性”方案。
*   **工作流程**：使用 **正则表达式预分词**，将文本按空格、标点等分割成独立的单词。每个单词分配一个唯一的整数ID。
*   **严重问题：词汇表无界与未知词**
    *   **词汇表动态膨胀**：现实世界中的词汇量（专有名词、拼写错误、新造词）几乎是无限的，无法预先穷举。
    *   **难以处理的 `<unk>` 标记**：任何在训练集中未出现过的单词，都会在推理时变成 **`<unk>`** 标记，导致信息丢失和困惑度计算错误。

> **总结**：基于单词的方法抓住了“可变粒度”的正确直觉，但“词汇表无界”的缺陷使其不实用。

#### 3. BPE (Byte Pair Encoding): 最佳实践方案

*   **核心概念**：一种结合了“字节级”的通用性与“单词级”的适应性的 **数据压缩算法**，后被引入NLP领域，成为现代大语言模型（如GPT-2）的**事实标准分词方案**。

*   **历史渊源**：
    1.  **起源**：1994年由Philip Gage发明，用于通用数据压缩。
    2.  **NLP引入**：通过神经机器翻译论文，替代了痛苦的基于单词的分词。
    3.  **LLM时代**：通过GPT-2正式进入语言建模时代。

*   **核心优势**：
    1.  **Bidirectional**：算法设计上具有双向（向前/向后）的确定性，避免了歧义。
    2.  **无特殊符号问题**：不像某些方案需要处理特殊情况。
    3.  **效率与适应性**：通过合并，高频词成为单个Token（长序列，高压缩比），低频字词/字符作为单个或少数Token（短序列），完美平衡了序列长度与词汇表大小。

*   **关键算法步骤（简化为理解）**：
    1.  将所有文本转换为 **Unicode** 字节序列。
    2.  **统计频率**：统计所有相邻字符/字节对的频率。
    3.  **合并**：将最频繁出现的一对字符合并成一个新的“词块”（Token）。
    4.  **迭代**：重复步骤2-3，直到达到预定义的**词汇表大小**（例如32,768个Token）。

## 第 16 部分

### 核心思想：从“预设规则”到“数据驱动”的分词

*   **核心概念**：**BPE (Byte Pair Encoding，字节对编码)** 分词器。
*   **关键转变**：抛弃了使用预设规则（如按空格、按字符）进行分词的静态方法。**核心思想是让分词器直接在原始文本数据上训练**，自动学习出最优的“单词”划分。
*   **基本洞察**：在原始文本中，**频繁出现的字符序列**（如“th”、“ing”或整个单词“the”）应该被合并成一个单一的 **Token**，从而提高编码效率。而**罕见序列**则会被拆分成更多的 Token。
*   **预处理细节**：为了计算效率，GPT-2 论文中先使用一个基于单词的分词器将文本粗略分割成“段落”，然后再对每个“段落”独立运行 BPE 算法。这是课程中你将实现的方法。

### BPE 算法详解：一个“合并”的迭代过程

BPE 算法非常直观，本质上是一个**反复合并最频繁出现的相邻 Token 对**的循环。

1.  **第一步：初始化**
    *   **输入**：原始文本字符串。
    *   **转换**：将字符串切分为**字节序列**（Byte Sequence），每个字节对应一个 0-255 的整数 ID。
    *   **初始词汇表**：这些 0-255 的整数 ID 构成了初始的词汇表。向量表只是一个从 Token ID 映射回其代表字节序列的索引。

2.  **迭代合并（核心循环）**
    *   **统计**：扫描当前的字节序列，统计所有**相邻字节对**（Token Pair）的出现次数。
    *   **选择**：找出出现次数**最高**的字节对。如果存在平局，可以选择任意一个（例如按数值大小选择）。
    *   **合并**：为这个新的字节对分配一个新的 Token ID（从 256 开始，依次递增）。
    *   **更新词汇表**：将新的 Token ID 及其对应的字节序列加入到词汇表中。
    *   **应用合并**：遍历整个字节序列，将所有出现的目标字节对其替换为新的 Token ID。

3.  **终止条件**
    *   重复步骤 2，直到达到预设的词汇表大小或预设的合并次数。

### 算法示例：以字符串为例

**初始序列 (字节 ID)**： `t(116) h(104) e(101) c(99) a(97) t(116) h(104) e(101) h(104) a(97) t(116)`

1.  **第一次合并**：
    *   统计字节对次数：`(116,104)` 出现 2 次，`(104,101)` 出现 2 次，等。
    *   假设 `(116,104)` 出现最多，选择它。
    *   创建新 Token：**`256`**，代表序列 `[116, 104]`。
    *   应用合并：序列变为 `256 (116,104) e(101) c(99) a(97) 256 (116,104) e(101) h(104) a(97) t(116)`。
    *   词汇表大小从 256 扩展到 257。

2.  **第二次合并**：
    *   **统计**：现在 `(256, 101)` 出现 2 次，`(104, 97)` 出现 1 次，等。
    *   **选择**：`(256, 101)`。
    *   **创建新 Token**：**`257`**，代表序列 `[256, 101]`（即 `[116, 104, 101]`，对应 “the”）。
    *   **应用合并**：序列变为 `257, c(99), a(97), 257, h(104), a(97), t(116)`。
    *   **关键效果**：**序列的长度（Token 数量）在缩小**，因为高频模式被压缩了。词汇表则变得更丰富。

3.  **第三次合并**：
    *   选择 `(257，99)`？不，实际更可能选择 `(257, h)`。假设选择 `(257, 104)`。
    *   创建新 Token：**`258`**，代表 `[257, 104]`。
    *   序列进一步被压缩。

### 编码与解码过程

*   **编码 (Encode)**：
    1.  将输入字符串转换为字节序列 (0-255)。
    2.  按训练时**合并的顺序**，逐条重放合并操作。
    3.  每次扫描序列，找到可以应用当前合并模式的 Token 对并替换。
    4.  最终得到一个由 Token ID 组成的整数序列。

*   **解码 (Decode)**：
    1.  获取 Token ID 序列。
    2.  遍历每个 Token ID，从**词汇表**中查找其对应的字节序列。
    3.  将所有字节序列拼接起来。
    4.  将字节序列转换回原始字符串。

### 公式与算法总结

*   **BPE 训练 (Training)**:
    *   初始状态：$S_0 = \text{ByteSequence}(text)$，词汇表 $V_0 = \{0, 1, ..., 255\}$，合并列表 $M = []$。
    *   对于 $i = 1$ 到 $N$:
        1.  统计 $S_{i-1}$ 中所有相邻对 $(a, b)$ 的频率 $f(a, b)$。
        2.  选择最高频对：$(a^*, b^*) = \arg\max f(a, b)$。
        3.  创建新 Token ID: $t_{new} = \text{len}(V_{i-1})$。
        4.  更新词汇表：$V_i = V_{i-1} \cup \{t_{new} \mapsto [a^*, b^*]\}$。
        5.  更新合并列表：$M.\text{append}((a^*, b^*, t_{new}))$。
        6.  更新序列：$S_i = \text{Replace}(S_{i-1}, (a^*, b^*) \rightarrow t_{new})$。

*   **BPE 编码 (Encoding)**:
    *   输入：字符串 $text$，合并列表 $M$。
    *   $S = \text{ByteSequence}(text)$。
    *   对于 $M$ 中的每个合并操作 $(a， b \rightarrow t)$:
        *   $S = \text{Replace}(S, (a, b) \rightarrow t)$。
    *   输出：Token ID 序列 $S$。

*   **BPE 解码 (Decoding)**:
    *   输入：Token ID 序列 $S$，词汇表 $V$。
    *   $bytes = []$。
    *   对于 $S$ 中的每个 token $id$:
        *   $bytes.\text{extend}(V[id])$。
    *   输出：$\text{StringFromBytes}(bytes)$。

## 分词器总结：从字符串到索引的映射

### 核心概念
- **分词器 (Tokenizer)** 的核心任务是建立 **字符串 (文本)** 与 **整数序列 (索引)** 之间的 **双向映射**
- 核心流程：输入字符串 → 转换为索引序列 → 按顺序重放合并过程 → 获取最终索引
- **验证**：通过重放合并操作，确保编码解码过程的正确性

---

### 传统分词方法的局限性

#### 字符级分词
- **简单但低效**
- 每个字符独立编码，序列长度巨大
- 语义信息丢失严重（"cat" 和 "c"+"a"+"t" 无关联）

#### 单词级分词
- **词汇表爆炸**：英文约 17 万个常用词，加上变体、专有名词等会更大
- **OOV (Out-of-Vocabulary) 问题**：新词、拼写错误无法处理
- **存储和计算开销大**

### BPE (Byte Pair Encoding) 算法详解

#### **算法本质**
- **1994 年提出的古老算法**，至今仍是主流分词方案的核心
- **基于语料库统计信息** 做出合理的合并决策
- **动态分配词汇表** 来表示字符序列

#### 核心原理
1. **初始化**：将文本拆分为字符级（或字节级）序列
2. **统计所有相邻字符对 (Pair) 的出现频率**
3. **合并频率最高的字符对**，形成新的 "子词 (Subword)"
4. **将新的子词添加进词汇表**
5. **重复步骤 2-4**，直到达到预设的词汇表大小或停止条件

#### 优势
- **在字符级和单词级之间取得平衡**
- **能处理未见过的词**：通过子词组合来表示新词
- **词汇表大小可控**（通常在 32k-100k 之间）

#### 关键操作示例（伪代码）
```python
# 假设初始词汇表: {'h','e','l','o','w','r','d'}
# 统计高频对: 'h','e' 出现最多
# 合并后: 词汇表增加 "he"，文本中 "h e" 替换为 "he"
# 继续统计: "he", "l" 出现最多
# 合并后: 词汇表增加 "hel"
# ...
```

### 关键公式与算法总结

**BPE 合并策略**：
\[
\text{Merge}(a, b) = \arg\max_{(a,b)} \text{Count}(a, b)
\]
- 其中 \(a, b\) 为相邻子词对
- 每次选择频率最高的对进行合并

**词汇表更新公式**：
\[
V_{t+1} = V_t \cup \{ \text{new\_token} \}
\]
- \(V_t\) 为当前词汇表
- \(\text{new\_token} = \text{concat}(a, b)\) 为合并后的新子词
