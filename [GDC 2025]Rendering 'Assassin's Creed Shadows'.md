# [GDC 2025] Rendering 'Assassin's Creed Shadows'

> 主题：深入解析《刺客信条：影》的渲染技术全貌，涵盖从引擎架构演进、LOD与GPU驱动管线、全局光照解算、光线追踪集成、天气与季节系统，到跨平台性能管理的完整工程实践。

## 一、Anvil引擎的演变与项目背景

### 1.1 引擎起源与碎片化挑战
- **诞生时间**：随《刺客信条1》于2007年发布，实际开发始于2004年。
- **核心定位**：专为**大型系统性开放世界**、**远距离渲染**及**系统性玩法**设计的引擎。
- **历史路线**：最初为统一引擎，随后分叉出服务于《彩虹六号：围攻》的 **Blacksmiths** 分支与《幽灵行动》系列的 **Silix** 分支。
- **核心痛点**：育碧内部曾引擎林立，且每个引擎衍生出多个并行分支，导致技术栈严重碎片化，阻碍了技术共享与协作效率。

### 1.2 转向统一平台：单一代码库策略
- **解决方案**：转向以 **Anvil** 作为**共享引擎**，覆盖多个游戏、品类与品牌。
- **开发模式**：采用 **单一代码库（Monorepo）** 模式，全球技术与制作团队共同协作。

### 1.3 《刺客信条：影》的次世代渲染目标
- **项目基准**：作为纯次世代（Gen5）作品，将**可伸缩性（Scalability）** 推至全新高度。
- **游戏规模**：
    - 开放世界尺寸达 **16km × 16km**。
    - 同时支持实时**昼夜循环**、**大规模破坏**、**虚拟几何**、**系统性天气**与完整的**四季更替**。
- **硬件策略**：仅面向当代主机，提供**性能、平衡、质量**三档模式。质量模式下优先采用依托虚拟几何的光追方案。

## 二、渲染技术与几何管线

### 2.1 LOD分级与超远距离渲染
- **LOD分级系统**：
    - **短距离主网格**：承载大部分资产。
    - **长距离网格**：仅保留大型资产或关键地标。
    - **加载网格**：在实体层级指定所属网格，距离由 **LOD选择器** 驱动。
    - **伪造实体网格**：包含减面与聚合资产，最远覆盖 **4 公里**。
    - **点云渲染器**：用于树木等的公告板方案，本质为**轻量级四边形渲染器**，最远渲染至 **8 公里**。
    - **超远距离地形**：超过8公里后仅保留**地形远景（Terrain Vista）**。
- **LOD递进顺序**：纯地形 → LOD网格 → 伪造实体 → 点云。

### 2.2 GPU驱动管线的演进：从AC Unity到Database系统
- **起源**：自2014年《刺客信条：大革命》（AC Unity）引入。
- **早期方案（DX11）**：
    - 使用 **Multi-Draw Indirect** 完成 **GPU集群剔除** 与 **间接绘制调用**。
    - 支持实例、集群、三角形级别的**精细剔除**。
    - **主要局限**：不支持 **Bindless 纹理**，只能做到**逐材质**的批处理。
- **进阶方案（DX12）**：
    - **完全Bindless** 资源访问。
    - 批处理粒度从“材质”提升到**“着色器”**，实现 **逐Shader** 批处理。
    - **运行时动态合批**，设计目标为每帧处理数百万实例。
- **Database 系统**：
    - 一个 **CPU 与 GPU 共享的数据容器**，采用**数据导向设计**，是存放全场景完整描述数据的**唯一真实来源**。
    - 支持在不同表之间建立**关系（Relations）**，模拟C++风格的指针与引用。
    - 利用自研编译器 **SIG（Shader Input Groups）** 自动生成着色器绑定代码与数据访问接口，确保C++与HLSL中的一致性。

### 2.3 场景组织与剔除管线
- **核心树（Core Tree）**：整个场景以树形结构组织，由**实体组（Entity Groups）** 和**叶节点（Leaf Nodes）** 构成层次结构。
- **剔除管线**：
    - **CPU 粗粒度剔除**：对叶节点进行初步筛选。
    - **GPU 全Pass视锥剔除**：对所有渲染Pass的视锥体进行剔除，并执行LOD选择。
    - **GPU 逐Pass剔除**：执行特定Pass的视锥剔除与遮挡剔除。
    - **运行时可选剔除**：在最终提交前支持 **Cluster 剔除** 和 **角度剔除**。

### 2.4 微多边形几何管线
- **技术基础**：基于Nanite思想的离散集群网格管线演进。
- **核心架构**：
    - 网格由具备**连续层次细节**的集群层级结构构建。
    - 根据投影大小动态选择**硬件光栅化（Mesh Shader）** 或**软件光栅化**。
    - 固定集群规模为 **128 个三角形**。
- **应用范围**：用于 **G-buffer** 和**阴影光栅化**，目前仅支持**静态不透明几何**（如城市建筑）。
- **软件光栅优化**：根据三角形配置交换XY坐标，以改善分支一致性与工作负载均衡。

### 2.5 渲染管线分工
- **微多边形管线**：负责城市建筑等大规模静态不透明几何。典型城镇场景约 **28,000** 个实例，**3,400 万** 三角形，其中 **90%** 的三角形通过软件光栅化完成。
- **GPU 实例渲染器**：负责大规模植被等 **Alpha-Test** 内容。同一城镇场景中处理约 **9,000** 个实例，近 **200 万** 三角形。
- **场景互换**：在森林场景中，GPU实例渲染器承担主要压力，微多边形管线占比较低。

## 三、全局光照（GI）方案

### 3.1 从烘焙体积到稀疏探针的演进
- **早期方案瓶颈**：
    - AC Unity 采用 **均匀的mipmap辐射体积烘焙**，不支持动态时间。
    - 若将均匀方案用于16×16公里世界并加入四季变化，数据量将膨胀至 **2 TB**，烘焙时间需 **600 天** 以上。
- **核心优化策略**：
    - **转向CPU烘焙**：解决显存墙问题，获得确定性和更好的并行分发效率。
    - **密度图与非均匀分布**：美术通过 **密度图** 仅在关键区域保留高GI分辨率。
    - **稀疏探针与几何剔除**：采用**稀疏八叉树结构**，仅在包含几何表面处细分探针，并剔除物体内部的无效探针。
- **最终成果**：相比全密度方案，**平均仅保留约 10% 的探针**，使数据量和烘焙时间变得实际可用。

### 3.2 GI数据的存储与表示
- **时间变化**：存储 **11 个关键帧**（1个局部光源，10个太阳位置）以支持昼夜循环。
- **颜色空间**：采用 **YCoCg** 格式，Y通道存储**方向性亮度**，Co、Cg以色度形式存储。
- **季节近似处理**：Lumen（Y）在所有季节中共享一份；仅色度值按季节分离存储。春/夏视为相同季节。

### 3.3 运行时GI：级联体积与探针系统
- **级联3D体积**：烘焙的稀疏体素数据在运行时解压缩并注入**级联3D体积**，通过跨级联混合和GI区块LOD混合实现连续细节层次。
- **光照探针系统**：
    - 约 **10,000 个探针** 组成的5级联，每帧动态更新约 **1,000 个**。
    - 利用**室内体积**为探针打上分类标签，采样时只混合分类一致的探针。
    - 探针数据被卷积为**近场辐射缓存**（高频）和**远场辐射缓存**（稳定后备）。

### 3.4 镜面反射方案
按优先级采用多层级混合策略：
1.  **屏幕空间反射**
2.  **可重光照的GBFR Q-map**：预烘焙的局部反射代理，运行时读取反照率、法线、深度并重新光照。最多支持3种季节变体，游戏中共 **16,000 张**。
3.  **动态Q-map**：在玩家位置实时生成，使用大幅度减面的代理实体以降低开销。

## 四、光线追踪技术

### 4.1 混合光线追踪GI流水线
- **抽象层设计**：在名为 **Fusion** 的硬件抽象层之上构建统一API，同时支持硬件/软件光线追踪与Inline/非Inline管线，切换仅需极小工作量。
- **两步流程（逐像素）**：
    1.  **屏幕空间光线**：优先执行以降低开销。
    2.  **世界空间光线**：屏幕空间未命中时，延续T值在BVH中继续追踪。
- **数据流**：命中结果存入 **光线追踪G缓冲区**，在“命中照明”阶段进行滤波与重新照明。若像素无命中，则回退采样**探针缓存**获取间接光照。
- **额外叠加RT-AO**：用于补偿追踪分辨率限制、光追世界与光栅化世界的差异，恢复高频遮蔽细节。

### 4.2 加速结构与优化
- **全局LOD偏移**：控制BVH中实例的细节层级。
- **纹理降级方案**：支持无纹理或平均颜色模式。
- **透明度缩放技术**：通过缩放三角形使其在相交计算中更稀疏，避免昂贵的Any Hit调用，性能比Any Hit快 **30%**。
- **内联光线追踪**：采用统一着色器处理追踪材质，简化资源制作流程。

### 4.3 性能开销
- **光线追踪探针**：多数主机上约 **1 毫秒**。
- **逐像素漫反射光追**：约 **4-5 毫秒**，总耗时（含探针）约 **5-6 毫秒**，分摊到图形队列和异步计算队列中执行。
- **镜面反射光追**：成本高昂，以**半分辨率**运行以在PS5 Pro和高端PC上达到性能目标。

### 4.4 降噪方案
- 最终使用 **MSD** 特化各向异性滤波器，核心采用**循环模糊**方法。
- 融合了**材质遮罩**以更好地处理去遮挡问题。
- 支持 **球谐降噪**，在YCoCg空间中编码球谐系数以保留光照方向信息。

## 五、天气与季节系统

### 5.1 系统化框架
- 由 **Atmos（大气模拟）**、**流体模拟** 和 **动画图** 协同驱动。
- **Amiens 图系统**：一个**完全数据驱动**的天气逻辑核心，由技术美术掌控所有表现行为。

### 5.2 雨水：湿滑表面系统
- **核心原理**：在G-Buffer中存储湿度遮罩，根据湿度与水坑级别，**暗化基色**并**降低粗糙度**。

### 5.3 降雪与动态积雪
- **动态雪积累**：
    - 通过冷/暖区域驱动，对材质反照率向白色插值，并调整粗糙度。
    - 对于植被，随雪级别升高**降低透射率与散射**。
    - 利用**微可见度**和**法线方向阈值**避免不合理积雪（如垂直表面不积雪）。
- **完整生命周期**：固体雪 → 湿润度/积水 → 融化变干，整个过程由 Ampere 图驱动。

### 5.4 室内外天气遮挡
- 复用GI系统的**室内体积数据**，光栅化到**室内深度图**，计算遮挡因子以防止雨雪错误出现在建筑内部。
- 针对雨滴方向渲染环境深度图，以剔除被遮挡的粒子和涟漪。

## 六、跨平台与性能管理

### 6.1 平台管理器
- **定位**：引擎“一等公民”，基于平台和上下文实施**数据驱动的性能设置**。
- **三类策略**：
    - **配置文件**：如性能/质量模式，支持热切换与需要重启的冷启动。
    - **空中上下文**：由玩法事件触发，如拍照模式中提升毛发渲染质量。
    - **修改器**：数据驱动，可笔刷绘制或由3D体积触发，用于解决局部性能瓶颈。

### 6.2 资源追踪
- **高层级视图**：追踪资源生命周期，定位帧内内存峰值点。
- **低层级工具**：检查分配器，排查隐性内存浪费。
- **性能回归遥测**：游戏内埋点计数器，监控GPU性能回归和动态分辨率缩放因子。

### 6.3 macOS/iOS 适配经验
- **核心挑战**：对从M1到M4的性能跨度进行画质层级划分。
- **关键决策**：在低端Mac上采用软件光线追踪作为代替方案。

## 七、总结与未来方向

- **项目里程碑**：首款搭载 **Monoray** 与 **Shader Engine** 的《刺客信条》作品，使用了扩展性最强的Anvil引擎版本。
- **待改进领域**：
    - **微多边形**：希望更广泛地使用，拓展到更多管线（如Alpha Test植被）。
    - **全局光照**：计划让 **红帽合成** 更加普及以简化开发管线。
    - **QA复杂性**：动态昼夜、季节与高度碎片化的超分技术使测试难度极高，需重新思考复杂度管理。
    - **天气与季节系统**：图形化原型制作便捷，但测试与复现问题依然困难。
- **架构演进**：明确希望推进**帧模块化与客制化**，向**数据驱动的帧图调度**方向探索，但对此变革仍持谨慎态度。
