# CMU_15-445_p14

## 一、查询优化概述

> 主题：查询优化是数据库系统中最复杂、最核心的部分，其目标是为SQL查询找到正确且代价最低的执行计划。

### 查询优化器的核心目标

查询优化器是数据库系统中负责将用户提交的SQL查询转换为**正确**且**代价最低**的执行计划的核心模块。其本质是**黑魔法**，其难度也被认为是数据库系统中最高、最复杂的部分之一。

*   **核心任务：** 为一个给定的SQL查询，找到一个**正确的**且**代价最低**的执行计划。

*   **正确性 (Correctness) 的绝对性：**
    *   正确性是优化的**硬约束**和**绝对前提**。
    *   一个执行速度极快但结果错误的计划是毫无价值的。

*   **代价 (Cost) 的相对性：**
    *   **“代价”** 是一个**相对**的概念，其具体含义可以根据不同的应用场景和需求进行调整。
    *   **默认场景：** 在本课程及大多数通用场景下，**代价**通常指 **执行时间**。
    *   **其他可能场景：** 网络开销、资源消耗、能耗等。

### 为什么查询优化是数据库系统中最困难的部分？

- **核心概念**：**查询优化器**的质量是区分商业数据库系统（如Oracle、SQL Server、DB2、Teradata、Snowflake）与开源系统（如MySQL、PostgreSQL、SQLite）的关键因素。
- **投入巨大**：商业公司花费数十年和数百万美元，只为让查询优化器能够智能处理各种复杂查询情况。
- **人才稀缺**：数据库公司招聘时，最抢手的人才就是擅长查询优化的工程师。

> **关键认知**：没有一个查询优化器能真正找到“最优”计划，因为这是**NP完全问题**。简单查询（如 `SELECT * FROM table WHERE id = value`）容易优化，但一旦涉及**表连接（JOIN）**，问题就变得极其复杂。

### 核心挑战：NP完全问题与“足够好”的策略

- **最优不可达**：由于问题的NP完全性质，优化器无法保证对所有查询都找到最优执行计划。它的真实目标是**在有限时间内找到一个足够好的执行计划**。
- **搜索空间爆炸**：随着查询中表连接数量的增加，可能的执行计划数量呈指数级增长，无法穷举。

> **核心算法思路**：必须采用**启发式方法**和**剪枝技术**来大幅缩小搜索空间，在优化质量和优化耗时之间取得平衡。

### 查询优化的整体流程与定位

在数据库执行一个SQL查询的完整流程中，查询优化是承上启下的关键环节：

1.  **SQL查询输入：** 用户或应用程序提交一个SQL语句。
2.  **语法分析与解析：** 将SQL文本解析成数据库系统能理解的**逻辑计划（Logical Plan）**。
3.  **查询优化/规划 (Query Optimization/Planning)：** 此步骤接收逻辑计划，生成多个可能的**物理计划（Physical Plan）**，并根据**代价模型**评估和选择最优的一个。
4.  **执行：** 数据库引擎按照选定的最优物理计划执行并返回结果。

### 关键术语与底层假设

*   **逻辑计划 (Logical Plan)：** 一个抽象的、与具体执行算法和数据布局无关的查询描述，定义了需要哪些操作（如`Join`, `Filter`, `Scan`）。
*   **物理计划 (Physical Plan)：** 逻辑计划的具体实现版本，指定了每个操作具体使用的算法（如 `Hash Join` vs `Merge Join`, `Index Scan` vs `Table Scan`）和数据访问路径。
*   **查询优化器 vs. 程序员手动优化：** 数据库厂商相信，由精心设计的查询优化器来决定最优计划，通常要优于由普通程序员手动编写或依赖提示（hints）进行优化的方式。

## 二、SQL查询执行流水线

> 主题：SQL查询从文本到执行计划的完整转换过程，包括解析、绑定、逻辑计划生成与重写。

### 查询执行全流程概览

```
SQL字符串 → [SQL重写器] → [解析器] → [绑定器] → [逻辑计划生成器] → [逻辑计划重写器] → ... → 物理计划
```

#### 阶段一：SQL重写器（可选，罕见）
- **作用**：在SQL字符串层面进行**正则表达式**替换，将其改写为另一条SQL字符串。
- **典型场景**：例如，将表名 `Foo` 重命名为 `Foo1`，以匹配系统内部特殊处理逻辑。
- **注意**：**大多数系统不执行此步骤**。

#### 阶段二：解析器
- **输入**：原始SQL字符串。
- **输出**：**抽象语法树**，由一系列**Token**（如`SELECT`、`FROM`、`WHERE`、表名、列名）组成。
- **本质**：将文本字符串转换为结构化语法表示。

#### 阶段三：绑定器
- **输入**：抽象语法树（Token流）。
- **输出**：带有**内部数据库对象引用**的语法树。
- **核心工作**：**“解析语义、校验存在性”**。
    - 去**目录**中查找：Token中的表名是否真实存在？列名是否属于该表？操作符是否合法？
    - 获取对象的内部ID、类型信息、元数据。
    - **常见错误抛出点**：如果此处报错，通常是“表不存在（table not found）”或“列不存在（column doesn’t exist）”。

#### 阶段四：逻辑计划生成器
- **输入**：绑定后的语法树。
- **输出**：**逻辑计划**——一个包含逻辑算子的树结构。
- **本质**：将带有对象信息的语法树**转换为数据库引擎能理解的执行意图**。

#### 阶段五：逻辑计划重写器（可选，但常见）
- **作用**：直接在**逻辑计划树**上进行优化，不涉及物理实现细节。
- **常见操作**：谓词下推、投影消除、视图展开、子查询去关联化。
- **注意**：虽然标为可选，**大部分生产系统都会进行此步骤**。例如，PostgreSQL会做逻辑优化，而**Bustub（CMU教学数据库）则不会**。

### 关键术语速查表

| 英文术语 | 中文对应 | 本质 | 是否必备 |
| :--- | :--- | :--- | :--- |
| **SQL Rewriter** | SQL重写器 | 字符串级别的正则替换 | 极少数系统 |
| **Parser** | 解析器 | SQL字符串 → 抽象语法树 | 必备 |
| **Binder** | 绑定器 | 语法树 + 目录 → 带对象信息的树 | 必备 |
| **Logical Plan** | 逻辑计划 | 面向语义的算子树（做什么） | 必备 |
| **Tree Rewriter / Optimizer** | 逻辑计划重写器/优化器 | 逻辑算子级别的重写优化 | 大部分系统 |
| **Logical Operator** | 逻辑算子 | 抽象操作（如Scan, Join, Filter） | — |
| **Physical Operator** | 物理算子 | 具体算法（如HashJoin, SeqScan） | — |

## 三、逻辑优化：启发式与基于规则的优化

> 主题：通过模式匹配规则对逻辑计划进行确定性重写，包括谓词拆分、下推、连接替换和投影下推等核心技术。

### 核心概念：基于规则的优化

**基于规则的优化器 (Rule-Based Optimizer)** 无需代价模型，直接使用**硬编码的规则 (Hard-coded Rules)** 或**模式匹配 (Pattern Matching)** 来重写查询。其本质是一种**启发式 (Heuristic)** 方法，执行那些**总是有益的**、确定的变换。

- **核心任务**：将逻辑计划通过模式匹配规则（类似访问者模式）转换为新的逻辑计划。通过遍历查询计划，识别当前算子、子节点和父节点，重写树结构。
- **核心概念**：**规则是确定性的**，一旦匹配，即“肯定正确”，无需成本比较。这用于**引导成本优化**，避免盲目穷举，快速排除低效计划。

### 谓词拆分与下推

#### 核心目标：尽早过滤数据，减少中间结果集大小

##### 第一步：拆分合取谓词 (Conjunctive Predicate Splitting)
- **核心思想**：将 `WHERE` 子句中的多个 `AND` 条件**拆解为独立的谓词**，每个谓词对应一个独立的 `Filter` 节点。
  - 原始状态：`σ(条件1 AND 条件2 AND 条件3)(R)`
  - 拆分后：`σ(条件1)(σ(条件2)(σ(条件3)(R)))`（逻辑等价）
- **为什么这么做？**
  - 拆分为独立节点后，我们可以**自由移动、排序、重组**这些 Filter 节点，为后续优化（如下推）铺平道路。
- **注意：析取谓词 (OR) 的处理**
  - 对于 `OR` 条件，不能简单拆分。通常需要**复制表达式子树**，用 `UNION`（集合并）来等价转换，处理更复杂。

##### 第二步：谓词下推 (Predicate Pushdown)
- **核心定义**：将 **Filter 节点向数据源方向（查询树底部）移动**，直到**谓词所需的所有属性（列）都已可用**的最早位置。
- **为什么这么做？**
  - 原始计划：先对所有表做全表扫描 -> 再做大笛卡尔积 -> 最后执行所有 Filter。
  - **问题**：笛卡尔积会生成巨大中间结果，极其耗费内存与计算。
  - **解决**：通过下推 Filter，**在表扫描后立即滤除无关行**，显著减少后续操作的输入规模。

#### 下推规则与示例
- **规则**：对于一个 Filter 谓词，如果它只引用**单个表的列**（如 `album.name = 'Andy's OG mix'`），那么它可以被直接下推到该表扫描节点的**正下方**。
- **示例**：
  - 原始位置：`σ(album.name = X)` 在顶层（所有 Join 之后）。
  - 下推后位置：`σ(album.name = X)` 放在 `Scan(Album)` 操作之后，`Join 之前`。
- **效果**：大幅减少后续 Join 操作需要处理的行数。

### 笛卡尔积替换为内连接

- **核心概念**：将原本的笛卡尔积操作（全交叉）替换为**内连接 (INNER JOIN)**，并直接利用连接谓词作为 `ON` 子句。
- **操作步骤**：
  1. 找到笛卡尔积操作上方的连接谓词（例如 `artist.id = peers.artist_id`）。
  2. 用**内连接**替换笛卡尔积。
  3. 将连接谓词作为**内连接的 `ON` 子句**。
- **优化效果**：避免生成大量无意义的中间结果。

### 投影下推 (Projection Pushdown)

- **核心概念**：在查询执行树中，尽早移除所有**冗余属性（列）**，只保留最终输出或后续操作真正需要的列。
- **核心概念：尽早丢弃不需要的属性**
- **目的**：减少参与后续操作（如Join、聚合）的**元组宽度**，从而降低内存带宽和计算开销
- **关键洞察**：这不是必须的优化，但在**物化模型 (Materialization Model)** 下效果显著
- **适用场景**：假设系统采用**行存储**或**延迟物化**，即每个操作符之间会复制整个元组（即使某些列后续不会被使用）。
- **具体做法**：
  - 分析最终输出需要哪些列。
  - 从**树的最底层开始**，在每个操作符（扫描、连接）后插入投影，只保留后续需要的列。

#### 潜在陷阱：过于激进的投影下推
- **核心问题**：投影下推虽然一般能减少数据传输，但**过于激进**可能导致：
  - **增加内存使用**：因为需要额外的投影操作符来剥离列。
  - **增加执行时间**：频繁的列裁剪会引入额外的处理开销。
- **权衡原则**：需要**平衡**减少数据传输量和增加计算/内存开销之间的关系。

#### 逻辑算子 vs 物理算子
- **逻辑算子**：只定义“做什么”（如Join、Projection），不规定“怎么做”
- **物理算子**：实现具体执行策略（如Hash Join、Nested Loop Join）
- **关键区别**：投影下推在逻辑层表现为独立的Projection算子，但在**物理层**可与Scan算子合并
    - 例如：扫描表时**仅读取需要的列**，而不是先读取整行再做投影

#### 与谓词下推 (Predicate Pushdown) 的比较
- **谓词下推更重要**：减少参与计算的行数，效果通常比投影下推更显著（行数减少 > 行宽减少）
- **两者可协同工作**：先通过谓词下推过滤行，再通过投影下推减少每行的列数

### 示例：逻辑计划优化四步法

以查询“找出所有出现在我的混音带上的艺术家”为例（基于Spotify数据库模型：艺术家-专辑多对多关系，`appears`表为关联表）。

1. **步骤1：拆分复杂谓词**
   - 将涉及多个表连接的查询拆解为多个单表过滤。
   - 结果：**谓词下推（Predicate Pushdown）**，尽早减少数据量。

2. **步骤2：选择最优连接顺序**
   - 评估不同连接顺序的成本。
   - **核心思想**：选择**基数最小（cardinality）**的中间结果作为后续连接的驱动表，减少开销。

3. **步骤3：应用代数等价变换**
   - 使用关系代数规则（如**连接结合律、交换律**）重新排列算子位置。
   - 规则示例：**投影消除（Projection Elimination）**，只保留查询所需的列。

4. **步骤4：合并重复或冗余算子**
   - 检测并移除**重复的谓词**。
   - 合并**相邻的过滤或投影操作**为单一算子。

### 表达式树的优化

**核心概念**：除了逻辑操作符的规则应用，WHERE、ON、HAVING等子句本身也是独立的**表达式树**，同样需要优化。

#### 表达式优化的实现机制
- **条件分支法**：使用大量的 if-then-else 语句来匹配特定表达式模式
- **模式匹配规则引擎**：声明式规则定义，寻找特定模式并应用优化

#### 典型优化示例：移除无用谓词

**核心概念**：识别并简化**始终为真或始终为假**的表达式。

- **示例：** `WHERE 1 = 0`
- 这是一个**恒假**的条件，优化器应将其重写为 `FALSE` 常量。
- 重写为 `FALSE` 后，优化器可以进一步识别：
  - 对于**连接操作**：可直接转化为 **空结果集**
  - 对于**选择操作**：**无数据返回**
  - 可以完全**跳过**后续的查询执行

#### 谓词优化：常量折叠与空值处理

- **核心概念：** 在查询优化阶段，数据库引擎会尝试对 `WHERE` 子句中的谓词进行**静态评估**，判断其能否在**执行前**就确定为永假（`FALSE`）。
- 如果能确定，就可以**跳过全表扫描**，直接返回空结果集。

**示例 1：`1 = 0` 的优化**
- **优化行为**：大多数数据库（如 PostgreSQL、MySQL）会将其识别为**永假表达式**。
- **实现细节**：引擎在优化器中内置了**常量折叠（Constant Folding）** 规则。

**示例 2：`NOW() IS NULL` 的优化**
- **PostgreSQL**：能够优化。它识别出 `NOW()` 是**内置函数**，且返回值**不可能为 NULL**。
- **SQLite**：**不优化**，因为SQLite的优化器**缺乏函数元数据**。

**示例 3：`RANDOM() IS NULL` 的优化差异**
- **MySQL**：能够优化（`RANDOM()` 从不返回 NULL）。
- **PostgreSQL**：**不能优化**（优化器**不做此假设**）。

#### 重要概念：**One-Time Filter（一次性过滤器）**
- 指在**查询计划的最顶层**（甚至**在扫描之前**）应用的一个过滤器。
- 当优化器能确定谓词恒为假时，它会插入一个**单次求值**的过滤器，直接返回零行，**阻止**后续的扫描操作。

#### 关键结论：优化器保守性的权衡
- **理想情况**：优化器应对所有内置函数的行为有**完备的元数据**。
- **现实情况**：许多系统（包括 PostgreSQL）**只对部分简单表达式**做折叠。
- **原因**：维护函数元数据增加了优化器的复杂度。保守是更安全的选择。

## 四、嵌套子查询优化

> 主题：将嵌套子查询去关联化，重写为等价的连接操作或提取独立执行，以提升执行效率。

### 核心优化思想：将嵌套查询“扁平化”

- **核心概念**：数据库查询优化器的首要任务是将**嵌套子查询**分解为更高效的执行计划
- **基本原理**：
  - 将嵌套查询拆分成独立的 **查询块**，逐个处理
  - 对于**相关子查询**（内层查询依赖外层），尝试重写为 **JOIN** 操作
  - 对于**非相关子查询**（内层查询独立），可以**单独执行一次**并缓存结果

### 子查询去关联化：将相关子查询重写为联接

#### 核心概念：**相关子查询 → 等价联接转换**

- **问题场景**：查询语句中包含一个**相关子查询**，即内层查询引用了外层查询的列。
- **关键观察**：这种 `WHERE` 子句中的相关性条件，本质上就是一个**等值连接**的语义。

#### 算法/操作机制
1. **规则匹配**：优化器检测到嵌套查询中的 `WHERE` 子句引用了外层的列，且条件是等值判断。
2. **提取连接条件**：将等值谓词从子查询中"拉出"，作为外层查询的连接条件。
3. **结构化重写**：将原本的两个查询块合并为一个单一的 **JOIN 操作**，消除嵌套。
4. **后续优化**：重写后，整个查询计划被视为一个**子树**，可以继续应用常规的优化。

### 非相关子查询优化：标量值提升 (Scalar Value Promotion)

#### 优化策略
- **识别条件**：内层子查询**不包含**对外层查询的引用（即非相关）
- **执行方式**：
  1. **先单独执行**内层子查询一次
  2. 将结果作为**标量常量**注入到外层查询中
  3. 基于已知常量值重新优化外层查询计划
- **性能优势**：避免为外层每行重复执行内层查询（**逐行执行**是最糟糕的策略）

#### 系统实现差异
- **MySQL**：两阶段优化，先执行非相关子查询，获取实际值，再带已知值重新进入优化器。
- **其他系统**：预编译语句中，优化器只能**猜测**子查询结果，生成通用计划。

#### 解关联的限制与注意事项

| 子查询类型 | 是否可提取 | 推荐优化方式 |
|-----------|-----------|-------------|
| **非相关子查询** | ✅ 可以 | 单独执行一次，注入标量值 |
| **相关子查询** | ❌ 不可直接提取 | 尝试重写为 JOIN |

- **技术警示**：**相关子查询**不能直接提取标量值（因为它依赖外层变量），更优策略是**转换为 JOIN** 而非提取常量。

## 五、基于代价的优化

> 主题：通过代价模型估算不同执行计划的成本，结合统计信息进行搜索，选择最优物理计划。

### 基于规则的优化 vs 基于代价的优化

| 特征 | 基于规则 (Rule-Based) | 基于代价 (Cost-Based) |
| :--- | :--- | :--- |
| **核心逻辑** | 硬编码规则，模式匹配 | 代价模型，枚举搜索 |
| **灵感来源** | 常识/经验启发 | 数据统计与数学模型 |
| **能否计算“最优”** | 不能，只保证“不错” | 能，找到全局/局部最优 |
| **适用场景** | 简单、确定的转换 | 复杂查询，需要精细调优 |
| **成熟度** | 早期优化器 | 现代数据库引擎 |

> **工程师视角**: 开发一个 **CBO** 时，你不仅要实现枚举算法（动态规划、贪心等），还需要构建一个**高度准确的代价模型**，这通常依赖于**直方图 (Histogram)**、**采样 (Sampling)** 等统计信息。在实际引擎中，通常会**结合使用**：先用基于规则的启发式做快速重写，再用基于代价的搜索做精细选择。

### 代价模型 (Cost Model)

- **核心概念**：数据库系统使用**内部度量标准**评估不同查询计划的执行代价，而非真实的挂钟时间。
- **特征**：
  - 是**系统内部数值**，不可跨数据库（如PostgreSQL vs MySQL）比较。
  - 目标不是找到最优计划，而是通过**启发式方法**找到"足够好"的计划。
- **深层理解**：成本模型需要处理连接操作带来的**计划组合爆炸**问题，所以必须依靠假设和启发式来缩小搜索空间。

#### 代价模型的构成要素

代价模型可以包含多种**物理成本指标**，不同的硬件和系统环境会影响其权重：

| 成本类型 | 具体内容 |
| :--- | :--- |
| **CPU 消耗** | 处理数据所需的计算量 |
| **磁盘 I/O** | 数据读/写操作次数 |
| **缓冲池命中率** | 所需数据是否已在内存（缓冲池）中 |
| **内存用量** | 执行过程中占用的内存大小 |
| **网络开销** | 分布式系统中节点间传递的消息数量 |

#### 查询代价的三种类型

1. **物理代价 (Physical Costs)**：与实际硬件资源消耗相关的代价（I/O、内存、CPU）。
2. **逻辑代价 (Logical Costs)**：**与物理实现无关**的固有代价，由操作符本身决定的输出元组数量。
3. **算法代价 (Algorithmic Costs)**：基于**大O复杂度**的代价估算，学术界常用，但**业界实践中少见**。

#### PostgreSQL的代价模型实践

- **物理代价+逻辑代价组合**：主要结合物理代价和逻辑代价进行估算。
- **魔法常数权重 (Magic Constant Factors)**：管理员可配置的预设常数权重。

**默认权重设定：**
| 访问类型 | 相对速度比例 | 说明 |
|---------|------------|------|
| **内存访问** | **基准值** (最快) | 假设比磁盘快 **400倍** |
| **顺序I/O** | 比随机I/O快 **4倍** | 顺序扫描效率更高 |
| **随机I/O** | 最慢 | 每次随机读取代价最高 |

#### 成本常数的设置问题

- **默认值与调优困境**：默认值被设计为“足够好”（good enough）的底线，**推荐做法**：需要通过“试错法”（trial and error）手动调整。
- **商业数据库系统的自动化方案**：像 **DB2、Oracle** 等商业系统，在首次启动时会自动运行**微型基准测试（Micro Benchmarks）**，根据测试结果**推导出适合当前硬件的常数因子**。

### 统计信息与选择性估算

#### 核心概念：**选择性（Selectivity）**
- **定义**：谓词 `p` 的选择性是表中**满足该谓词条件的元组比例**。
- **公式**：
  \[
  \text{selectivity}(p) = \frac{\text{满足谓词 } p \text{ 的元组数}}{\text{表中的总元组数}}
  \]

#### 代价优化器的三大基础假设

为了简化选择性和代价计算，优化器通常会做出以下近似假设：

1. **均匀分布假设（Uniform Distribution）**：假定数据在直方图桶内是均匀分布的。
2. **独立性假设（Independence）**：不同列之间的谓词条件相互独立，互不影响。
3. **包含性假设（Containment）**：对于连接查询，较小表的值集是较大表值集的子集。

#### 三大核心假设概述

- **均匀数据假设（Uniformity Assumption）**：所有值出现的频率相同
  - **现实问题**：真实数据往往存在**数据倾斜（Data Skew）**。
  - **改进方案**：维护**热值列表（Heavy Hitter List）**，只记录出现频次最高的前10个值及其精确计数。
- **谓词独立性假设（Independence Assumption）**：不同谓词之间相互独立
  - **公式表达**：`Selectivity(cond1 AND cond2) = Selectivity(cond1) × Selectivity(cond2)`
  - **典型反例**：`WHERE make = 'Honda' AND model = 'Accord'`，make和model高度相关。
- **包含原则（Inclusion Principle）**：内表每一行都能在外部表中找到匹配
  - **适用条件**：当存在**外键约束（Foreign Key）**时，此假设成立。

#### 谓词选择性的误算：**相关性灾难**

- **核心概念：** 在估计复合谓词的选择性时，如果假设各列**完全独立**，会导致严重误算。
- **解决方案：** **多列统计信息 / 联合统计信息**
  - **关键术语：** **Correlated Columns**（相关列）
  - 高级数据库允许你**定义**哪些列是相关的，并为它们的**组合**维护统计信息。

#### 统计信息的自动维护与挑战

- **核心问题：** 不是“要不要维护”，而是**“多久维护一次”**。
- **自动性：**
  - **单列统计信息：** 系统**自动**维护。
  - **多列（相关性）统计信息：** 需要用户**主动指定**哪些列组合需要维护联合统计信息。
- **维护成本：** 重新计算统计信息本质上就是**运行一次查询**（扫描大量数据）。
- **最佳实践：**
  - **批量加载后立即刷新：** 如果通过定时任务大量写入数据，应在完成后立刻手动执行 `ANALYZE`。
  - **可配置刷新策略：** 通过参数设置统计信息刷新的**激进程度**。

#### 数据统计的三大存储方式

##### 1. 直方图

- **等宽（Equi-width）直方图**
  - **核心概念**：将数据的值域范围分成若干个 **宽度相同** 的桶，每个桶内的数据数量（高度）可能不同。
  - **缺点**：假设桶内数据分布均匀，但实际分布可能很不均匀。
- **等深（Equi-depth）直方图**
  - **核心概念**：每个桶的 **高度**（数据出现次数）尽可能相等，而 **宽度**（值域范围）则动态变化。
  - **优点**：对数据密集区域自动分配更窄的桶，对稀疏区域分配更宽的桶，能更好地适应数据倾斜。

##### 2. 概率草图

- **本质**：一种**概率性数据结构**，通过哈希和位操作来估算**基数**或**频率**。
- **优势**：能大幅压缩存储空间。
- **典型代表**：HyperLogLog（估算基数）、Count-Min Sketch（估算频率）。

##### 3. 采样方法（Sampling）

- **基本思路**：在运行时分析过程中，**维护一个内部辅助表**，该表包含原表的**元组子集**，用于**估算谓词的选择性**。
- **工作流程**：
  1. 对采样表**直接运行谓词**。
  2. 假设采样结果**能代表整个数据集**。
- **实际实现情况**：**SQL Server** 实现了此方法，**大多数系统**未采用。

### 优化搜索策略

#### 核心思想：规则+代价混合策略

大多数数据库系统采用 **规则优化与代价搜索相结合** 的方法。

- **先执行规则优化**：清理查询计划，移除那些**已知无效或冗余**的操作。
- **再应用代价搜索**：基于代价模型，寻找最优的执行路径。
- **典型应用场景**：代价搜索最常用于 **连接顺序优化**。
- **全量代价搜索系统**：如 **SQL Server** 和 **CockroachDB**，它们会对**逻辑变换和物理变换**进行全盘代价搜索。

#### 单表访问路径选择

在查询优化中，**访问路径选择** 是规划搜索的第一步。核心目标是**基于代价模型选择最优的访问方法**，并决定谓词的应用顺序。

- **搜索空间**：所有可能的查询执行计划的集合。由于枚举所有计划是 **NP-完全** 的，必须采用启发式搜索。
- **时间预算**：优化器允许的最大搜索时间。原则是**搜索开销不能超过查询执行带来的收益**。
- **Sergeable Query（可搜索参数化查询）**：指能够直接利用索引的查询。

**单表访问的决策流程：**
1. **枚举访问方法**：考虑所有可能的表访问方式（全表扫描、索引扫描等）。
2. **应用代价模型**：估算每种方法的具体成本。**索引扫描不一定总是最优**，具体取决于`WHERE`子句的**选择性**。
3. **谓词应用顺序**：当存在多个谓词时，可以**重新排序**它们以最大化过滤效率。

#### 多表连接：物理优化策略

##### 自底向上 (Bottom-Up) vs. 自顶向下 (Top-Down)

| 策略 | 核心思想 | 常见系统 |
| :--- | :--- | :--- |
| **自底向上 (Bottom-Up)** | 分阶段处理：先通过规则进行逻辑重写，再用动态规划探索连接顺序。 | PostgreSQL, DB2 |
| **自顶向下 (Top-Down)** | 在一个统一的框架内**同时**完成所有转换（规则重写和顺序搜索）。 | SQL Server, CockroachDB |

##### 自底向上优化：动态规划与 System R 算法

这是成本优化器中最经典的实现方法，源自1970年代的IBM **System R** 项目。

- **核心机制：动态规划 (Dynamic Programming)**
  - **本质**：一种**分治**思想。
  - **工作流程**：
    1. **起点**：从最原始的、未经优化的单表关系开始。
    2. **迭代构建**：首先计算所有**两表连接**的最佳计划，然后基于此计算**三表连接**的最佳计划，如此类推。
    3. **输出**：通过不断将新表与已优化的子计划连接，最终得到所有表连接后的最佳物理计划。

##### 左深连接树的假设

**核心概念：** 1970年代起，传统优化器强制假设只构建**左深连接树**。

- **树类型定义**：
  - **左深树（Left-Deep Tree）**：每次连接只将一个新表加入左侧分支，形成单侧线性增长。
  - **右深树（Right-Deep Tree）**：与左深树对称，被视为等价被丢弃。
  - **浓密树（Bushy Tree）**：允许两侧同时连接子结果。
- **设计动机**：1970年代计算机极慢，需要大幅缩小搜索空间。
- **关键公式**（连接顺序枚举复杂度）：
  - 全排列枚举：$O(n!)$ 种连接顺序
  - 左深树剪枝后：$O(n \cdot 2^{n-1})$ 种

##### 自顶向下查询优化方法

这是SQL Server等商业数据库采用的**核心优化策略**。

- **核心思想**：从查询的最终目标出发，自上而下地探索所有可能的执行计划，并利用**分支定界（Branch and Bound）** 搜索来剪除成本过高的分支。
- **关键术语**：
  - **分支定界（Branch and Bound）**：一种**剪枝**策略。一旦发现当前子计划的**累积成本**已经超过了目前已知的**最佳计划成本**，则立即停止探索该分支。
  - **属性（Property）**：对数据分布或排序状态的要求。
  - **实施规则（Enforcement Rules）**：当子计划输出的**属性**不满足父操作的需求时，优化器会自动插入**排序（Sort）** 等操作来强制其满足要求。

**工作流程详解：**
1. **起始点（根节点）：** 定义目标和属性（如要求最终数据按某列排序）。
2. **向下探索：** 枚举逻辑和物理计划（如考虑不同的连接算法和扫描方式）。
3. **核心机制：分支定界剪枝**
   - 如果当前子路径的成本仍小于当前最佳成本，则**允许继续向下探索**。
   - 一旦累计成本超过了最佳成本，**立即停止**对该分支的探索。
4. **属性满足与实施规则：** 如果子操作无法自然满足属性，优化器会通过**实施规则**在子计划上方**自动插入一个排序（Sort）操作**。

#### 物理属性与强制规则

- **核心概念**：查询计划中，每个算子处理数据时，数据会带有特定的**物理属性**（Physical Properties），例如 **排序顺序（Sort Order）**、**分布方式（Distribution）**。
- **强制规则**：在查询计划的根节点或中间节点，可能需要数据满足特定的物理属性。
- **作用**：在进行**自顶向下（Top-Down）** 的搜索时，优化器会根据这些规则来裁剪（Prune）不满足要求的候选算子。

#### 动态规划的核心公式

**状态定义**：
- $C(S)$ = 连接集合 $S$ 中所有表的最小代价
- $T_i$ = 第 $i$ 个表

**递推关系**：
$$C(S) = \min_{i \in S} \left[ C(S \setminus \{i\}) + \text{join\_cost}(T_i, \text{result of } S \setminus \{i\}) \right]$$

**初始条件**：
$$C(\{i\}) = \text{access\_cost}(T_i)$$

**搜索复杂度**：$O(3^n)$ 种可能的连接树 → 左深树假设降为 $O(n \cdot 2^{n-1})$

### 优化器实现方式

**优化器的实现可以基于硬编码规则或规则引擎**

- **硬编码（if-else 方式）**：如 PostgreSQL 和 MySQL，优化逻辑直接写成一系列 if-then-else 语句。
- **规则引擎（模式匹配方式）**：如 CockroachDB、SQL Server，采用模式匹配引擎，维护一个规则目录，通过匹配查询计划树来应用转换规则。
  - 规则具有 **幂等性（Idempotent）**：多次应用同一规则结果不变。
  - 规则可能具有 **可交换性（Commutative）**：不同顺序应用规则得到相同结果。
  - 可以为规则指定 **优先级（Priorities）**：控制规则触发的先后顺序。
