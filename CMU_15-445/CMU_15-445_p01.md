## 四、数据库核心概念：从文件系统到DBMS

> 主题：数据库与数据库系统的区分、文件系统作为数据库的缺陷、DBMS的核心价值

### 数据库 vs 数据库系统：关键区分

- **数据库（Database）**：**存储在某处的数据集合**
  - 这是你的业务数据本身
  - 代表现实世界中的某个实体（如学生、课程、专辑）
  - 示例：CMU的学生注册数据
- **数据库系统（DBMS - Database Management System）**：**管理数据库的软件**
  - 这是你常说的 MySQL、PostgreSQL、MongoDB、Oracle
  - 数据库系统负责**存储、查询、维护**数据库数据
  - 示例：Oracle、Snowflake

> 🔑 **核心洞察**：无论你是游戏引擎工程师还是渲染工程师，**职业生涯中始终会遇到某种数据库**——小到Excel，大到PB级系统。理解数据库系统内部的运作机制（数据存储、查询执行）比仅仅会用更重要。

### 简易数据库设计实例：数字音乐商店

#### 场景设定
- 目标：模拟 iTunes / Spotify / Bandcamp 的简单数据库
- 核心数据实体：**艺术家（Artists）** 和 **专辑（Albums）**

#### 最简存储方案设计

**数据模型**：只包含两个核心表（文件）

| 艺术家表 | 专辑表 |
|---------|--------|
| 艺术家ID | 专辑ID |
| 艺术家名称 | 专辑名称 |
| (其他基本信息) | 艺术家ID（外键）|
| | 发行年份 |

#### 一个过于简单的系统实现（Strawman System）

这是讲师构建的“稻草人系统”——一个**极其简陋但能工作的存储方案**，用于后续论证为什么需要成熟的数据库系统：

```python
# 伪代码：最原始的内存数据存储
artists = []   # 列表存储艺术家
albums = []    # 列表存储专辑

# 存储艺术家
def add_artist(name):
    artists.append({"id": len(artists), "name": name})

# 存储专辑
def add_album(title, artist_id, year):
    albums.append({
        "id": len(albums),
        "title": title,
        "artist_id": artist_id,
        "year": year
    })
```

> ⚠️ **关键问题**：这种简单方案虽然能存储数据，但会面临**性能、并发、持久化、查询能力**等一系列问题，这正是本课程要深入探讨的内容。

### 文件系统作为数据库的五大缺陷

| 问题维度 | 具体表现 | 数据库解决方案 |
|---------|---------|-------------|
| **元数据** | 结构信息硬编码在应用中，列名无标签，必须硬编码字段偏移量 | Schema 定义与数据分离 |
| **查询效率** | 每次查询都要全量扫描（O(n)时间复杂度） | 索引（O(log n) / O(1)）、查询优化器 |
| **安全性** | 解析漏洞（代码注入风险）、类型混用（无类型强制机制） | 类型系统、参数化查询 |
| **完整性** | 任意修改、无约束（无访问控制、无验证规则） | 约束（PRIMARY KEY, FOREIGN KEY, CHECK）|
| **关系建模** | 多对多无法轻松表示、删除时产生孤立数据 | 关系表、规范化设计、级联操作 |

### 从文件系统到数据库的核心演进

| 维度 | 文件系统方案的问题 | 数据库提供的解决 |
|------|-------------------|-----------------|
| **关系维护** | 应用代码手动级联删除 | 外键约束 + 级联操作 |
| **查询效率** | 线性扫描 O(n) | 索引 O(log n) / O(1) |
| **多语言适配** | 每语言重写解析逻辑 | 统一SQL接口 |
| **并发控制** | 文件级粗粒度锁 | 行级锁 + MVCC |
| **崩溃恢复** | 无保障，可能写坏数据 | WAL日志 + 回滚/重做 |
| **分布式扩展** | 无法实现 | 分片 + 复制 + 分布式事务 |

### DBMS 的核心定位

- **定义**：**数据库管理系统（DBMS）** 是一种软件，向应用代码暴露 API
- **功能**：允许你在数据库中 存储、分析、操作 数据
- **核心理念**：**"永远不要重新发明轮子"**
  - 如果创业开发 Web 应用或 SaaS 服务，你希望花时间在构建产品特性上
  - 确保数据安全、崩溃恢复等基础能力 不是 产品的差异化卖点
  - 用户默认你的服务不会丢失数据——这是基本假设

### 使用 DBMS 的压倒性优势

- **节省时间**：无需手动管理文件存储的安全性、完整性和恢复逻辑
- **可靠性**：DBMS 经过广泛测试和部署，远胜于临时用 JavaScript 手写的存储方案
- **标准做法**：几乎总是应该使用数据库系统

## 五、数据模型

> 主题：数据模型的定义、关系型数据模型的核心地位、区块链不是数据模型

### 数据模型 (Data Model) 的核心概念

- **核心思想：** **数据模型** (Data Model) 是描述**数据的外观和结构**的蓝图。它定义了“一个实体在数据库中长什么样”，而不仅仅是属性是整数还是浮点数。
- **模式 (Schema)：** 这是根据**数据模型**的规则，实际告诉数据库系统“我们想要这个数据库具体长什么样”的描述。
- **本质：** 数据是抽象的，模式是具体的实现。

### 关系型数据模型 (Relational Data Model)

- **地位：** 这是本课程**绝对核心**的内容，也是讲师认为**最重要**的数据模型。
- **广泛应用：** 绝大多数你常见的数据库（如 PostgreSQL, MySQL, SQLite, Oracle, SQL Server）都是**关系型数据库**。
- **历史与现状：** 它并非最新的技术，但却是工业界最主流、最可靠的选择。
- **对比**（新潮 vs. 老派）：
  - **NoSQL (非关系型):** 这是一个非科学的、**松散的分类**。通常指文档/对象模型（如 MongoDB）。
  - **旧时代模型 (Obsolete):** 如层次模型 (Hierarchical)、网络模型，大多已过时，但仍存在于银行等老牌系统中 (如 IBM 的 IMS，1960年代构建)。**通常建议：不要轻易去维护这些老系统的公司工作。**
  - **现代特定模型:**
    - **数组/矩阵/向量模型：** 用于处理机器学习和卫星/医学影像数据。
    - **键值模型 (Key-Value):** 会在本课程后续提及。

### 区块链 (Blockchain) 为何未列入？

- **重要澄清：** 讲师被问到“为什么区块链/Web3不在列表中？”
- **回答：** **区块链本身不是一个数据模型。** 它是一个更上层、更复杂的系统架构或平台，其底层可能使用了某种数据模型（如键值对），但数据模型的定义层次更低。列表中的是**数据模型**，而不是**数据系统或应用平台**。
- **核心概念：区块链是分布式账本实现，而非数据模型本身**
  - 区块链底层机制是**分布式账本**（distributed ledger），它需要上层数据结构（如关系型、键值对、文档型）来解释存储的字节
  - 教授观点（可讨论）：**区块链唯一真正有用的场景是比特币**，其他多数应用对拜占庭容错分布式账本的需求是过度设计，甚至可能是骗局

### 关键术语与总结

| 术语 | 解释 | 重要性/应用 |
| :--- | :--- | :--- |
| **数据模型** (Data Model) | 数据结构的抽象蓝图 | 定义了数据的“形状” |
| **模式** (Schema) | 数据模型的具体化，告诉系统如何构建 | 数据库的实际配置 |
| **关系型** (Relational) | 以表、行、列组织的模型 | **关系型数据库的基石** |
| **NoSQL** | 非SQL的松散分类 | 常用于文档/对象存储 |
| **旧时代模型** | 层次/网络模型 (1960-70s) | 过时，仅存于旧系统 |
| **键值模型** (Key-Value) | 简单的键值对存储 | 会在课程中提及 |
| **数组/矩阵模型** | 基于向量、矩阵 | 用于机器学习/科学计算 |
| **区块链** | 并非数据模型，而是一种系统/协议 | 不属于数据模型分类 |

## 六、关系模型的诞生与核心思想

> 主题：早期数据库系统的痛点、Ted Codd的突破、关系模型的三大基石

### 早期数据库系统的痛点：物理与逻辑的**强耦合**

#### **IMS（Information Management System）**：IBM为阿波罗登月计划构建的数据库系统
- 基于 **COBOL** 语言
- **核心问题**：数据库的**逻辑结构**（你如何思考数据）与**物理实现**（数据在磁盘上如何存储）**紧密绑定**

#### 具体表现：你必须**预先决定**数据存储结构
- 选择存储方式（例如：**哈希表** vs **树结构**）
- **不同选择暴露不同API**：
  - 选择哈希表 → 只能使用哈希相关的操作接口
  - 选择树结构 → 才能使用范围查询等树相关操作

#### **致命后果**：当你改变主意时（例如从哈希表改为树结构）：
1. **必须导出全部数据**，重新以新的数据结构加载
2. **必须修改所有应用程序代码**，因为API已经改变

### 关键人物与突破性洞察

- **数学家 Ted Codd**（刚刚在宾夕法尼亚大学获得博士学位）
  - 加入IBM纽约研究所
  - 观察到IBM数据库开发者们**重复造轮子**：
    - 每次需求变更（如：添加新列、新属性），都要重写大量数据库代码
    - 每个人都在做同样的修改，却**互不沟通**
  - **核心洞察**：存在一种**更好的方式**

### 关系型模型（Relational Model）的诞生

- **时间线与文献**：
  - **1969年**：首次提出关系型模型（技术报告）
  - **1970年**：在**ACM通讯**发表后续论文（成为里程碑）

- **核心思想**（数学上的抽象）：
  - 提出一种**数学化的抽象**方式来**表示数据库**以及**与数据库交互**
  - **逻辑层与物理层分离**：
    - 用户只关心**逻辑数据模型**（表、行、列）
    - 不再需要关心物理存储是哈希表还是树
    - 物理实现的改变**不再影响应用程序代码**

### 三大基石（三个核心原则）

#### **1. 数据结构：只使用“关系”**
- **关系（Relation）** = 一张**表**（Table），由行（元组）和列（属性）组成。
- **存储对象**：**简单数据**（原子值，不可再分），而非复杂嵌套结构。
- **核心突破**：你只需定义“有哪些属性（列）”，无需关心底层如何组织数据指针或索引。

#### **2. 物理存储：完全交由实现决定**
- **过去**：你必须说“我的数据存成B树”或“用哈希表”。
- **现在**：只需声明“这是我的关系/表”，数据库系统（DBMS）自动选择最佳存储方案（行存、列存、分区等）。
- **关键后果**：**数据独立性**——应用程序与物理存储解耦，即使底层实现改变，SQL查询无需修改。

#### **3. 查询方式：声明式语言取代过程式代码**
- **过去**：写COBOL/FORTRAN循环，一步步告诉数据库“先打开文件，再逐条比较，找到后写缓存...”。
- **现在**：使用**高级声明式语言**（如SQL），只告诉系统“我要什么”（what），系统自动决定“怎么做”（how）。
- **实现机制**：DBMS包含**查询优化器**，将SQL翻译成物理执行计划（如选择索引、连接算法）。

### 重要注解：现实中的“关系模型”并非教条

- **纯化者的理想**：Codd定义的严格数学关系（如：每个元组唯一、属性顺序无关等）**在商业数据库中从未完全实现**。
- **相互借鉴**：
  - 关系数据库（PostgreSQL、MySQL）借用了NoSQL的JSON列存储、稀疏数据支持。
  - NoSQL系统（如MongoDB）借鉴了关系模型的声明式查询语言和二级索引。
- **核心启示**：关系模型的核心贡献是**抽象层**，而非具体实现细节。

### 关键术语对照表

| 关系模型术语 | 现代SQL等价物 | 含义 |
|------------|-------------|------|
| 关系（Relation） | 表（Table） | 数据的有序/无序集合 |
| 元组（Tuple） | 行（Row） | 单条记录 |
| 属性（Attribute） | 列（Column） | 记录的某个字段 |
| 声明式语言 | SQL（结构化查询语言） | 只描述结果，不描述过程 |
| 物理数据独立性 | - | 应用代码与存储引擎解耦 |

## 七、关系模型核心概念：主键、外键与关系

> 主题：关系模型的三部分、主键与外键的定义与作用、多对多关系的处理

### 关系模型（Relational Model）核心概念

#### **1. 关系模型概述**
- **核心定义**：关系模型由三部分组成：
  - **结构（Structure）**：定义数据库中的关系和属性的外观
  - **完整性约束（Integrity Constraints）**：定义允许存储在数据库中的数据规则
  - **操作API（Manipulation API）**：提供读写数据、执行查询并生成结果的接口

#### **2. 关系（Relation）与表（Table）**
- **核心概念**：关系是一个**无序集合**，包含一组**关系属性**，用于表示现实世界中的实体
- **术语澄清**：
  - **关系（Relation）** 不是指表之间的关系，而是指**属性之间的关系**
  - 在本课程中，**关系** 和 **表** 可以互换使用，没有实际区别

#### **3. 元组（Tuple）与行（Row）**
- **核心概念**：元组是一组属性值的**集合体现**，代表一个具体的实体实例
- **术语合并**：**元组、行、记录** 在本课程中含义相同，可互换使用

#### **4. 值的原子性（Atomic Values）**
- **原始关系模型要求**：元组中的值必须是**标量（Scalar）** 或**原子（Atomic）** 的
  - 例如：单个整数、单个浮点数、单个日期、单个字符串等
- **现代系统的放宽**：
  - 现代数据库允许违反这一规则，例如：
    - 属性值可以是**整数数组**
    - 属性值可以是 **JSON对象**
- **关键术语**：**NULL值** 是被允许的，但可通过约束定义某属性不允许为NULL；但从数学/模型角度，NULL始终是可能的

### 主键 (Primary Key)

- **核心定义**：主键是由**一个或多个属性**组成的集合，用于唯一标识表中的一条特定记录（元组）
- **作用**：提供**精确识别**单个记录的方法
- **数学表示**：设关系R有属性集A，主键PK是A的子集，满足：
  - **唯一性**：对于任意两个不同的元组t1和t2，$t_1[PK] \neq t_2[PK]$
  - **极小性**：PK的任何真子集都不能唯一标识元组
- **选择原则**：优先使用**天然唯一**的列（如身份证号、用户ID）。如果业务数据（如名字）可能重复，应引入**人工ID**。
- **实现方式**：
  - **标准SQL**：使用 `SEQUENCE`（序列）自动生成唯一数字。
  - **MySQL**：使用 `AUTO_INCREMENT`（自增字段）。
  - **自定生成**：UUID（通用唯一标识符）。
  - **隐式主键**：某些数据库系统（如SQLite）如果没有显式定义主键，会自动生成一个隐藏的 `rowid` 列作为主键。

### 外键 (Foreign Key)

- **核心概念**：外键是一种**引用机制**，用于**关联**不同数据表（关系）之间的数据，定义**数据之间的映射关系**（例如：哪个艺术家属于哪个专辑）。
- **核心术语**：`FOREIGN KEY`，`REFERENCES`，`关联`，`数据依赖`，`参照完整性`。
- **工作原理**：在一个表（子表/引用表）中，某一列（外键列）的值必须与另一个表（父表/被引用表）的主键列中的值匹配。这确保了数据间的**逻辑一致性**。
- **关键示例与问题**：
  - **场景**：有“艺术家表”（artist）和“专辑表”（album）。我们需要知道专辑是哪个艺术家发行的。
  - **简单映射**：如果专辑只有一个艺术家，可以在专辑表中添加一个 `artist_id` 列，将其设为外键，引用艺术家表中的 `artist_id` 主键。这样专辑表就通过 `artist_id` 与艺术家表建立了关联。
  - **多对多关系问题**：一个“混音带”（mixtape）可能包含**多个艺术家**。此时，简单的 `artist_id` 单列无法直接存储。这意味着需要引入**第三张关联表**（连接表/交叉表）来处理多对多关系。

### 多对多关系与交叉引用表

#### 核心概念：处理多对多关系的映射问题

在关系型数据库中，**无法简单地在单个表中使用数组** 来存储一对多的关系（例如：一张专辑对应多个艺术家）。

- **问题场景**：  
  如果一张专辑（Album）有多个艺术家（Artist），直接在 Album 表中设置一个 `artist_id` 字段是行不通的，因为一个字段只能存储一个值，无法表示“多个艺术家”。
- **数组方案的局限性**：  
  虽然理论上可以用数组存储多个 ID，但：
  - 许多数据库不支持整数数组类型。
  - 无法声明数组元素为外键（Foreign Key）。
  - 无法强制约束“同一艺术家在一张专辑中只能出现一次”的唯一性。

#### 核心解决方案：创建交叉引用表（Cross-Reference Table）

**交叉引用表** 是关系数据库中处理 **多对多（Many-to-Many）** 关系的标准实现方式。

- **结构**：  
  新建一张独立的中间表，包含两个外键：
  - `artist_id`：引用 Artist 表的 ID。
  - `album_id`：引用 Album 表的 ID。
  
  **示例**：  
  ```  
  Table: Album_Artist  
  - artist_id (FK → Artist.id)  
  - album_id (FK → Album.id)  
  ```

- **作用**：  
  通过这张中间表，可以在不同实体之间定义不同的**基数关系（Cardinality）**：
  - 一对多（One-to-Many）：一个 Artist 可以有多个 Album。
  - 多对一（Many-to-One）：多个 Artist 可以贡献到同一个 Album。
  - 多对多（Many-to-Many）：通过中间表实现 Artist 与 Album 的任意关联。

- **约束行为**：当向 `AlbumArtist` 表插入一行（例如 `artist_id=333, album_id=444`）时，数据库会自动检查 `Artists` 表中是否存在 `id=333` 的记录，以及 `Albums` 表中是否存在 `id=444` 的记录。如果不存在，则**拒绝插入**。
- **核心结论**：这个机制**确保了数据完整性**，防止出现“指向不存在记录”的无效引用（即**孤行（Orphan Rows）**）。

### 总结与关键公式

**关系模式定义**：
$$R(A_1, A_2, ..., A_n)$$

- $R$：关系名称（表名）
- $A_i$：第i个属性（列名）

**元组表示**：
$$t = (v_1, v_2, ..., v_n)$$
- $v_i$：对应属性$A_i$的标量值（或NULL）

**主键约束**：
$$PK \subseteq \{A_1, A_2, ..., A_n\}, \quad PK \neq \emptyset$$
- 对于任意 $t_i, t_j \in R$，若 $i \neq j$，则 $t_i[PK] \neq t_j[PK]$

## 八、数据操作语言：过程式与声明式

> 主题：DDL与DML的区分、过程式方法与声明式方法的对比

### 数据定义与操作语言区分

- **核心概念：** 数据库操作分为**数据定义语言（DDL，Data Definition Language）** 和**数据操作语言（DML，Data Manipulation Language）**。
- **数据定义语言（DDL）：** 负责**定义数据库结构**（Schema），例如创建表、定义主键、外键等。它是数据库的“骨架”。
- **数据操作语言（DML）：** 负责**实际的数据操作与查询**，是数据库的“血肉”。这是交互的核心。

### 数据操作的两大类方法

- **核心概念：** 执行数据库操作（如查询、更新）时，有两种根本不同的策略。
- **1. 过程式方法 （Procedural Method）**
  - **特点：** 开发者需要编写代码，**精确地、一步步地**定义数据库执行操作的**具体策略和步骤顺序**。
  - **类比：** 像是在写详细的食谱，必须告诉厨师“先切葱，再热油，然后放入葱...”。
  - **基础：** 基于**集合（Sets）** 和**包（Bags，即允许重复元素的集合）** 的理论。

- **2. 非过程式/声明式方法 （Non-Procedural / Declarative Method）**
  - **特点：** 开发者只需要**声明“想要什么”**，而不需要指定“如何获取”。数据库系统自身负责决定最优的执行策略。
  - **类比：** 像是告诉厨师“我要一盘鱼香肉丝”，厨师自己会决定切菜、炒菜、调味的顺序。
  - **典型代表：** **SQL（结构化查询语言）** 的核心思想。你只需要写 `SELECT ... FROM ... WHERE ...`，数据库负责找出最快的方式。

## 九、关系代数：查询执行的核心构建块

> 主题：关系代数的七种基本运算符、选择与投影操作

### 核心概念：数据处理的两大方法

- **过程性方法（Procedural Method）**：明确告诉系统需要执行的具体算法和步骤
  - 代表：**关系代数（Relational Algebra）**
  - 基于**集合（Sets）**或**袋子（Bags）**的数学基础

- **声明性方法（Declarative Method）**：只描述想要的结果，系统自行决定实现方式
  - 代表：**关系演算（Relational Calculus）**
  - **实际应用**：**SQL**就是声明性语言的典型代表
  - 注意：关系演算在**实际数据库引擎开发中几乎用不到**，只有在**查询优化器**（Query Optimizer）开发中才可能需要

### 关系代数：查询执行引擎的基石

#### **七种基本运算符（Seven Fundamental Operators）**

由Codd提出的七种**基本关系运算**，它们构成了**执行引擎**处理**单语句查询**的基础构建块

#### 核心设计原则

- **输入输出规则**：每个运算符接收**一个或两个关系**作为输入，**输出始终是一个关系**
- **链式组合**：可以像链式调用一样将运算符串联起来，回答复杂的高级查询问题
- **数学基础**：理论基于**集合（Sets）**（无重复元素），但实际数据库实现基于**袋子（Bags）**（允许重复值）

### 选择操作（Select / Restrict）

- **核心概念**：从一个输入关系中，**筛选出满足特定条件的元组（行）子集**。相当于 SQL 中的 `WHERE` 子句。
- **关键术语**：
  - **选择谓词（Predicate）**：使用一阶谓词逻辑（如 `=`、`>`、`<`、`IN` 等）定义元组必须满足的条件。
  - **布尔逻辑组合**：支持通过 **合取（AND，即 conjunctions）** 和 **析取（OR，即 disjunctions）** 组合多个谓词，实现复杂过滤。
  - **历史命名**：原论文中称为 **Restrict**，但现代教材和工业界普遍称为 **Select**。
- **操作示例**：
  - 输入关系 `R`，包含属性 `A_ID` 和 `B_ID`。
  - **示例1**：选择所有 `A_ID = 'a2'` 的元组。
  - **示例2**：选择同时满足 `A_ID = 'a2'` **且** `B_ID > 1` 的元组（AND 组合）。
- **SQL 类比**：`SELECT * FROM R WHERE A_ID = 'a2' AND B_ID > 1;`
- **谓词能力扩展**：
  - ✅ **跨列比较**：允许比较不同列，如 `WHERE A_ID > B_ID`。
  - ✅ **子查询/集合**：理论上允许，如 `WHERE A_ID > (SELECT ... FROM ...)` 或 `WHERE A_ID IN (...)`。

### 投影操作（Projection）

- **核心概念**：从一个输入关系中，**移除某些列（属性），仅保留指定的列**，并自动消除结果中可能出现的重复行。
- **关键术语**：
  - **属性子集（Attribute Subset）**：明确指定输出关系需要包含哪些列（属性列表）。
  - **去重（Duplicate Elimination）**：由于投影丢弃了某些列，可能导致多行结果完全相同，关系模型要求结果集合中不允许重复元组，因此会自动去重。
- **操作目的**：缩减数据维度，只关注对当前查询有用的属性。
- **SQL 类比**：`SELECT DISTINCT A_ID, B_ID FROM R;`（注意：标准 SQL 的 `SELECT` 默认不自动去重，需显式使用 `DISTINCT` 才等价于关系代数的投影语义）。
- **与选择操作的关系**：选择是**行过滤**，投影是**列裁剪**，两者常组合使用（先 `WHERE` 筛选行，再 `SELECT` 指定列）。

### 投影运算的扩展

- **核心概念**：投影运算是**从关系中移除不需要的属性**，并允许**重新排列和计算属性值**。
- **关键术语**：希腊字母 **π (小写 Pi)** 作为投影运算符。其底层逻辑与SQL中的 **SELECT 子句** 直接对应（即“SELECT后面跟的内容”）。
- **操作细节**：
  - **输出是**：一个属性经过筛选、排序和数学/字符串函数操作后的新关系。
  - **计算范围**：投影**逐元组 (tuple by tuple)** 处理输入关系，即对每一行独立应用变换。
  - **包含函数**：任何对列（属性）的内置操作，如**数学运算（减100）**、**字符串填充、空格处理等**，都属于投影的范畴。
  - **聚合函数例外**：像 SUM, COUNT, AVG 等聚合函数不属于投影，那是另一个独立的概念（聚合运算）。
- **示例说明**：
  - 操作：`π (vid - 100, bid)` 表示先从关系中选取 `vid` 和 `bid`，然后对 `vid` 值减去100，再颠倒属性的输出顺序。
- **与SQL的类比**：
  - 关系代数：`π (属性表达式列表)(关系名)`
  - SQL：`SELECT 属性表达式列表 FROM 表名`
- **重要纪律（与SQL的关键差异）**：
  - **无序性**：投影结果**不保证顺序**。关系代数是基于集合（Set）的，而集合天生无序。
  - **无重复**：关系代数**不允许重复元组**。结果集会自动去重（因为集合的元素是唯一的）。而SQL默认允许重复，如需去重需加 `DISTINCT`。

## 十、关系代数：集合运算与连接操作

> 主题：并集、交集、差集运算、笛卡尔积与连接操作

### 并集 (Union) 运算

- **核心概念**：**从集合论中借用的基本运算**，将两个具有**完全相同属性集**的关系合并成一个新关系。
- **关键术语**：运算符是 **∪ (并集)**，表示将两个关系的所有元组“拼接”到一起。
- **操作约束**：
  - 关系代数要求：被并集的两个关系必须**属性数量、类型和顺序完全一致**（即**并相容性**）。
  - 如果两关系都有属性 `aid` 和 `bid`，则并集的结果就是将所有行上下堆叠。
- **结果特性**：
  - **单位化**：结果仍然是**无序的**，且**不允许重复**。
  - **输出顺序**：示例中展示的“第一个关系在上，第二个在下”只是用于演示，实际结果顺序不确定。
- **与SQL的类比**：
  - 关系代数：`关系A ∪ 关系B`
  - SQL：`SELECT * FROM A UNION [ALL]`
  - **关键区别**：
    - 关系代数的 `∪` 对应 SQL 的 **`UNION`**（默认去重）。
    - SQL 的 **`UNION ALL`** 则是**允许有重复**的并集，这等价于“包含重复”的集合操作。关系代数本身不支持产生重复，因此 `UNION ALL` 在关系代数中没有直接等价物。

### 交运算 (Intersection)

- **核心概念**：取**同时出现在两个关系**中的元组。
- **关键术语**：`INTERSECT` (SQL), `∩` (关系代数)
- **SQL 语法**：  
  ```sql
  SELECT * FROM R
  INTERSECT
  SELECT * FROM S;
  ```
- **重复处理**：
  - 若 R 和 S 本身无重复元组，则交的结果中每个元组**只出现一次**（因为集合无重复）。
  - 若 R 或 S 内部有重复（如 R 中有两个相同的 (a,3)），则交的结果**保留该重复次数**的匹配逻辑（但关系代数仍按集合论处理，SQL 需显式用 `INTERSECT ALL` 保留重复）。

### 差运算 (
