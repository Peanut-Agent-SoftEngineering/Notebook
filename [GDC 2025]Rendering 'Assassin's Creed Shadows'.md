演讲者与游戏背景

- **演讲者**：Nicolas Lopez，来自 Enville 渲染团队（推测为 Ubisoft 旗下工作室）。
- 本场演讲发布于《A Sanskrit Shadows》（推测即《刺客信条：影》）发售当日。
- 演讲仅聚焦于该作部分渲染技术，大量内容因时长限制未能涵盖。

Envil 引擎发展脉络

- **Envil 引擎**最初为《刺客信条 1》（2007）而生，实际研发始于 2004 年。
- 设计目标：
    - **大型系统性开放世界**
    - **长距离渲染**
    - **系统性玩法**
- 除《刺客信条》系列外，还用于《Gustricon》《For Honor》《RnB Usage》等多种类型的作品（涵盖竞技 FPS、射击、SS 等），应用场景复杂。

引擎分支与碎片化现状

- 随着时间推移，Envil 衍生了大量**分支（fork）**：
    - **C-meter**：《刺客信条》专用引擎。
    - **Blacksmiths**：《RnB Usage》等。
    - **Silix**：《Gustricon》开放世界。
- 更宏观地，Ubisoft 内部同时存在多个引擎：
    - Envil、Stodrop、Dunia、Voyager、UB、Art 等。
- 每个引擎自身也有多个分支，例如 C-meter 有三条分支，Dunia 同样多条。
- **关键矛盾**：引擎与分支数量过多，导致技术资源分散、维护成本高、共享困难，最终促使 Ubisoft 改变工作方式（后续内容可能涉及统一或整合）。

技术要点提炼

- **引擎承载的渲染挑战**：大规模开放世界、远距离可见性、多种游戏类型的渲染需求共存。
- **组织级痛点**：多引擎/多分支带来的**碎片化**问题，推动引擎架构演进。

（本次内容为演讲开场与引擎历史背景，尚未涉及具体渲染技术，但为后续深入 Envil 的渲染支柱和性能方案奠定了背景认知。）

**引擎统一化与Monorepo策略**

*   **历史痛点：引擎与分支过多**
    *   **内部引擎数量众多**：Envil、Stodrop、Dunia、Voyager、UB、Art 等多个自有引擎并存。
    *   **分支泛滥**：单个引擎（如 Anvil 的前身或相关项目）内部存在 3 个以上的分支，导致维护成本极高。
    *   **核心矛盾**：大型系统开发周期以**年**为单位，代码寿命可长达**十年**。多引擎、多分支的重复劳动严重拖累竞争力，使得“重写大型系统”的机会极度稀缺。

*   **当前方案：统一的 Envil 与单一代码库**
    *   **共享引擎**：全公司范围内，跨游戏、跨品类、跨品牌共同使用 **Envyl（Anvil 引擎的进化形态）**。
    *   **Monorepo**：所有代码集中托管在**单一代码库**中。
    *   **全球协作**：技术团队与制作团队分布世界各地，在同一代码基础上协同工作。

**次世代标杆：《刺客信条：影》的技术定位**

*   **继承与超越**
    *   **基准作品**：《刺客信条：英灵殿》是上一代开放世界 RPG 的标杆，但为**跨世代**作品。
    *   **代际跳跃**：《影》是系列首款**纯次世代（Gen 5）** 游戏，目标是将**可伸缩性**推向新高度。
*   **英灵殿的技术特征与局限**
    *   画质与性能模式下的画面表现**极为相似**，可伸缩性有限。
    *   可伸缩性的主要驱动因素仅仅是**原生分辨率**。
    *   在 Xbox Series X 上以 4K 原生运行时，GPU 占用仅略高于 **20ms（对应 30fps）**，性能余量充足。
*   **《影》的开放世界复杂度**
    *   **基础规格**：**16x16 公里**的开放世界，附带完整的**昼夜循环**。
    *   **核心系统**：
        *   **大规模破坏**
        *   **Retressing**（重镶嵌/动态地形改造，与破坏系统相关）
        *   **虚拟几何体**
        *   **系统化天气与季节**（包含冬季雪景等动态变化）
    *   **平台策略**：仅登陆 **Gen 5 主机**与 PC，支持三种现代主机的渲染模式：
        *   **性能模式**、**均衡模式**、**画质模式**，对应不同的目标分辨率。
        *   开发早期即决定：**画质模式**下优先保证 **Retressing** 的视觉效果。

**渲染架构关键点**

*   **远景渲染**：Envyl 引擎从一开始就为**长距离渲染**构建，以支撑庞大的开放世界视野。
*   **技术协同**：所有子系统（阴影、破坏、虚拟几何、天气季节）必须在一个统一的引擎框架内高效协作，这正是从多引擎混乱走向单一代码库后，技术突破得以实现的基础。

### 渲染架构与细节层次 (LOD) 系统

- **性能模式选择**：开发早期即确定在**画质模式（Quality Modes）**下优先采用**重细分（Tessellation）**技术。

- **引擎核心设计**：
    - **Envil 引擎**专为**远距离渲染**设计，从底层构建。
    - 世界被划分为**单元格（Cells）**，并原生支持**多用户编辑**。
    - 数据组织为**流式网格层（Streaming Grid Layers）**，网格定义完全**数据驱动**，在不同游戏中可灵活配置。

- **细节层次与网格划分（LO-Degrades & Grids）**：
    - **短距离主网格（Short-Range Main Grid）**：包含大部分主要资产。
    - **短距离道具网格**：存放靠近玩家时会消失的小道具，对整体渲染和玩法影响甚微。
    - **长距离网格（Long-Range Grid）**：保留大型资产或兴趣点（Points of Interest），以支持更远的可见距离。
    - **加载网格（Loading Grids）** 由**LOD选择器（LOD Selectors）** 驱动，两者必须匹配，否则会导致物体异常消失。

- **超远距离渲染层**：
    1.  **代理实体网格（Fake Entity Grid）**：包含**减面资产**和**聚合资产**，覆盖范围最远可达**4公里**。
    2.  **点云（Point Clouds / Mass Imposter Rendering）**：大规模冒名顶替体渲染系统，渲染距离扩展至**8公里**。
    3.  **远景地形（Terrain Vista）**：8公里以外，仅显示烘焙了大量细节的裸地形。

- **可视化演示顺序**：
    - 裸地形 → 叠加LOD网格（短/主/长距离） → 叠加代理实体与点云。
    - **点云渲染器**实质是一个**超快速的大规模四边形渲染器（Mass Quad Renderer）**，用于渲染最远8公里的树木。通过**淡入淡出（Fading）** 与近景LOD网格平滑过渡。

### GPU驱动管线（GPU-Driven Pipeline）演进

- **技术基石**：GPU驱动管线自《刺客信条：大革命》（2014）起便成为 Envil 引擎的基石，经历多次迭代。
- **演进路线**：
    - **Unity / 大革命阶段**：引入**GPU集群剔除（GPU Cluster Culling）** 与**间接绘制调用（Indirect Draw Calls）**。
    - **Valhalla / 英灵殿阶段**：进一步推进**大规模实例化（Mass Instancing）** 与**批处理（Batching）**，并新增支持（原文在此截断）。

### GPU 驱动渲染管线的演进：从《刺客信条：大革命》到《英灵殿》

#### 1. 《刺客信条：大革命》（2014）中的“批次渲染器”
- **定位**：基于 **DX11 类 API** 的 GPU 驱动管线初代方案。
- **核心目标**：降低昂贵的 **DX11 绘制调用** 开销。
- **关键技术**：
    - 利用 **多重间接绘制** 实现 GPU 端剔除与渲染合批网格。
    - 引入 **精细剔除**，覆盖 **实例、集群（Cluster）和三角形** 层级。
- **局限性**：
    - **CPU 端仍承担较重任务**，管线并非完全 GPU 驱动。
    - **不支持无绑定纹理**，限制了合批规模。
    - 仅能实现 **“按材质”合批**。
    - **不支持相同计算**，导致网格渲染延迟，产生显著额外成本。

#### 2. 《英灵殿》中的“GPU 实例渲染器”
- **定位**：针对 **DX12 类 API** 设计的全新管线，简记为 **GPU IR**。
- **关键进步**：
    - **完全无绑定（Fully Bindless）**，消除纹理绑定限制。
    - 合批粒度从 **“按材质”** 升级为 **“按着色器”**。
    - 在 **运行时动态合批**，可处理每帧 **数百万实例** 的剔除。
- **架构迁移**：更多处理步骤从 **CPU 移至 GPU**，实现高度 GPU 驱动化。
- **核心技术依赖**：名为 **Database** 的共享数据容器。

#### 3. Database：CPU 与 GPU 间的共享数据层
- **本质定义**：
    - 一个 **可被 CPU 与 GPU 共享的数据结构容器**。
    - 遵循 **数据导向设计**，内建 **引用机制**。
    - 可视为 **“超级结构化缓冲区”**。
- **实现方式**：
    - 基于内部着色器编译器 **SIG（Shader Input Groups）** 生成绑定代码与工具函数。
    - **扩展 SIG 语言**，使其能够 **声明“表”**。
    - 通过 **Database 类比**（如行、对象引用）来组织场景数据。
- **代码生成与访问**：
    - 示例：在 **SIG 格式** 顶部声明表。
    - 自动生成 **C++ 和 HLSL** 中的访问接口（如 getters、setters）。
    - 开发者在 C++ 和着色器中看到 **完全相同的编号与操作**，无需手写绑定代码。
- **核心特性：关系**
    - **关系** 用于将 **不同表相互链接**，构建复杂场景图。
    - 例如：一个 “行” 就是一个指向某对象的引用（类似指针）。
- **在 GPU 管线中的角色**：
    - 用于 **共享完整的场景描述**，保证 CPU 与 GPU 在视角、剔除、绘制等各阶段使用一致的数据。

### 数据库关系与场景数据管理

#### 关系表的核心抽象
- **关系**：连接不同表之间的桥梁。
    - **一对一关系**：相当于数据库中的**指针**。
    - **一对多关系**：模拟一个带有大小的指针（例如，指向多个子元素的指针），允许一个实体关联多个目标。
- **目标**：实现类似C++风格的完整描述，用于表达**网格关系**与**属性**。
    - 同一个表可以拥有在**CPU端**和**GPU端**的不同实例。
    - 数据需要在实例间高效传播（通常是 CPU → GPU）。

#### 数据同步模式
为了控制数据从CPU传播到GPU的方式，系统支持三种模式：
1.  **完全拷贝模式**
    - 最直接的方式：直接将CPU端的整个数据拷贝到GPU端的**字节地址缓冲区**中。
2.  **脏数据更新模式**
    - 在**行级别**或**页级别**维护一个**脏标记掩码**。
    - 仅将标记为“脏”的行或页复制到目标表，实现增量更新。
3.  **大型表的稀疏驻留模式**
    - 针对极大型表，为避免CPU内存饱和，CPU端仅保留**脏行**。
    - 这些脏行在被刷新到GPU之前短暂驻留在CPU端，随后被清除。

#### GPU端场景描述与CalMesh细节
- **GPU场景表示**：与CPU端高度相似但经过针对性优化，结构与结构缓冲区类似。
- **CalMesh的LOD结构**：
    - 每个CalMesh最多支持 **5个LOD级别**。
    - 每个LOD拥有两个切换距离：
        - 用于**菜单视图**的距离。
        - 用于**阴影**的距离。
    - 支持对**阴影LOD距离**进行自定义。
- **几何体与PSO的关联**：
    - 在**子网格**级别通过**批处理哈希**进行绑定。
    - 设计保证每个PSO仅对应唯一批次，便于重复且易于管理。

#### 场景组织与剔除
- **声明方式**：所有上述结构均通过**SIG中的表**来声明，表之间使用**行**与**范围关系**相互链接。
- **世界结构**：
    - 采用**四叉树**进行全局组织。
    - 为高效剔除，依赖由**实体组**和**叶节点**构成的层次结构。
    - 叶节点是具体实例（如Cal实例）的载体。

### 叶节点与实例分组

- **核心概念：世界组织为剔除树（Culling Tree）**
    - 高性能剔除依赖于**实体组（Entity Groups）** 和 **叶节点（Leaf Nodes）** 构成的层次结构。
    - **叶节点本质上是一组实例的集合**。在引擎内部应将其重命名以更清晰地表达此概念。
- **分组策略与目的**
    - 将**空间上特别接近的实例**聚合到同一个叶节点中。
    - **两大收益**：
        1.  **最小化包围体（Bounding Volumes）**，减少重叠和误判。
        2.  提升 **CPU 端粗粒度剔除（Coarse Culling）** 的效率，因为决策以组为单位。

### 数据表（Tables）的生命周期与抽象

- **核心概念：类似 C++ 的 GPU 数据管理接口**
    - 目标是在保持**最大缓存效率（Cache Efficiency）** 的同时，提供友好的编程界面。
- **生命周期四步走**：
    1.  **分离声明**：明确区分 **CPU 表** 和 **GPU 表**。
    2.  **数据填充**：向表中创建条目（如剔除网格表中的一个 Mesh 条目），并设置各类属性。
    3.  **同步触发**：显式触发更新，将 CPU 端数据变更同步至 GPU（这是之前所述**复制策略**的实现）。
    4.  **GPU 端使用**：在着色器中自由获取并利用该数据。
- **高级特性：超级结构化缓冲区（Super Structured Buffer）**
    - 可通过注解（Annotations）自主决定数据布局：
        - **SoA**（Structure of Arrays，数组结构）
        - **AoS**（Array of Structures，结构数组）
        - 甚至进行**混合搭配**。
    - 这本质上是一个比传统结构化缓冲区更灵活、更强大的数据库抽象。

### 完整的 GPU 与硬件管线概览

整个渲染管线被清晰地解构为多个阶段，以实现极致的并行与剔除效率。

- **CPU 端：粗粒度剔除**
    - 执行 **CPU 粗粒度剔除（CPU Coarse Culling）**。
    - 淘汰整个**叶节点**（即实例组），为 GPU 端减轻负载。

- **GPU 端：精细与分步剔除**
    - **两列核心阶段**：**帧剔除（Frame Culling）** 和 **逐通道剔除（Per-Pass Culling）**。

    1.  **帧剔除阶段**
        - **功能**：对所有通过 CPU 剔除的实例，并行**测试所有渲染通道的视锥体**。
        - **附加逻辑**：在此阶段同时完成大量 **选择（Selection）和混合（Blending）逻辑**，意味着它不只是做布尔可见性判断。

    2.  **逐通道剔除阶段**
        - **功能**：执行**特定于通道的视锥体与遮挡剔除**。
        - **示例**：对于太阳阴影，会为每个级联贴图执行**视锥体剔除**。
        - **数据准备**：准备实例所需数据，指明**在统一缓冲区何处**获取**几何、材质和常量数据**，供后续顶点/像素着色器使用。
        - **关键机制**：维护一种**描述符/索引形式**，实现间接、统一的数据访问。

    3.  **运行时可选剔除**
        - 在最终进行真正的渲染调用前，支持可选开启的 **簇剔除（Cluster Culling）** 和 **三角形剔除（Triangle Culling）**。

## Micro-Polygon 几何管线

### 整体定位
- 与 **GPU 实例化渲染器** 并行存在的另一套渲染通路，专门服务于**静态不透明几何体**（如城市建筑）。
- 基于 Brian Karis 在 2021 年提出的思想，属于**离散集群网格管线**的演进。
- 目标：**以约一半的显存占用**提供与传统 GPU 管线相同的几何细节，并支持连续 LOD 与高效裁剪。

### 核心数据组织与 LOD
- 网格被构建为 **集群层级（Cluster Hierarchy）**，每个 **集群由 128 个三角形** 组成（相比 AC Unity 等老方案的 64 个更大粒度）。
- **连续 LOD**：集群之间形成递进的简化关系，不再依赖离散的 LOD 级别切换。
- **按需流式加载**：单独根据可见性对集群进行流式传输，降低常驻内存压力。
- 描述内存布局与访问路径的**元数据**被单独维护，用于材质、常量等数据的定位。

### 光栅化决策：硬件 vs 软件
- 根据三角形在屏幕上的**覆盖像素大小**动态选择：
  - **大三角形** → 硬件光栅化（借助 Mesh Shader）。
  - **小三角形（微多边形）** → **软件光栅化**，以规避硬件对小三角形的效率瓶颈。
- 软件光栅化基于**扫描线算法**，默认按行扫描。
- **关键优化**：根据三角形是偏“水平”还是“垂直”属性，**交换 X/Y 轴**后再扫描，使扫描方向始终与三角形长轴对齐，**提升分支一致性（Branch Coherence）和工作负载平衡**。

### 渲染流程：双阶段缓冲
- 采用**可见性缓冲（Visibility Buffer）+ G-Buffer 两遍分级绘制系统**：
  1. 首遍写入可见性缓冲（仅记录实例/集群 ID 等标识）。
  2. 第二遍根据可见性缓冲结果，按材质着色生成 G-Buffer（适合微多边形/G-Buffer 和阴影图栅格化）。
- 这种设计允许在着色前进行高效的**集群级与角度级剔除**。

### 与其他技术的相似点
- **Metis 库**：用于集群划分（与 Nanite 共享同一分区工具）。
- **网格简化**：仍基于 **SimpliGon** 生成集群组简化（并非完全无降阶的虚拟几何）。
- **绑定架构**：全管线 **Bindless（无绑定）**，与 GPU 实例化渲染器一致。
- **用户自定义 Shader**：提供手动 API，支持在 Shader Graph 中插入定制代码。

### 运行时剔除能力
- 在最终光栅化前，支持可选的：
  - **集群级剔除**（Cluster Culling）
  - **角度剔除**（Angle Culling，基于朝向与可见性）

### 场景分工与数据统计
- **Micro-Polygon 管线**：主攻**城市静态建筑**，在 Q2 场景中约 **28,000 个实例，3,400 万三角形**，其中 **90% 通过软件光栅化**（具体比例受几何分布、渲染分辨率影响）。
- **GPU 实例化渲染器**：专用于**重度 Alpha Test 植被**（如树木），同场景约 **9,000 个实例，近 200 万三角形**。
- 场景适配反转：在**森林场景**中，Micro-Polygon 的使用量极少，植被全由 GPU 实例化器接管。

### 关键术语速览
- **Cluster Hierarchy / 连续 LOD**
- **硬件光栅化 vs 软件光栅化（扫描线）**
- **可见性缓冲（Visibility Buffer）与 G-Buffer 两级结构**
- **Bindless（无绑定资源）**
- **Metis 分区、SimpliGon 简化**
- **分支一致性（Branch Coherence）**：通过坐标轴对调优化软件栅格化
- **集群剔除 / 角度剔除**

### 场景渲染：实例化与微多边形取舍
- **森林场景**以大量树木为主，约**9,000个实例**，跨全Pass渲染，总面数接近**200万三角形**。
- 反向场景下（如城市微多边形），**GPU实例化渲染器（GPU Instance Renderer）**承担主要压力。
- 剔除前场景实例达**300万**，最终渲染**3万**；剔除前三角形达**15亿**，最终渲染**700万**。
- 在此类密度下，当前无需在微多边形中强制支持**Alpha Test**，但已将其列为未来工作。

### 全局光照（GI）技术演进：从烘焙到光追的全光谱管线
**核心理念：** 构建一条**可扩展的GI管线光谱**，覆盖从完全烘焙到完全光追的漫反射与镜面反射，让玩家根据硬件自行选择，同时保障60 FPS目标。

#### 1. 《刺客信条：大革命》——体积化世界GI的起点
- 实现**体积化开放世界全局光照（Volumetric Open World GI）**。
- 基于GPU使用**光线束（Ray Bundles）**进行烘焙，细节可参考《God of War》中Josh Opson的分享。
- 采用**均匀Mipmap辐照度体积（Uniform Mipmap Radiance Volumes）**覆盖世界，通过Mipmap实现远距离的细节层级。
- 初期**不支持动态昼夜变化**，该特性在后续《刺客信条：枭雄》中加入。

#### 2. 《刺客信条：起源》——面向超大世界的稀疏化
- 将GI改造为**稀疏体积结构（Sparse Volumetric GI）**，以适配**16×16公里**的超大开放世界。
- 在保持体积光照质量的同时，显著降低存储与计算开销。

#### 3. 《刺客信条：影》——动态季节与全光追GI
- 引入**动态季节系统**，并要求GI对其做出响应。
- 全新开发**光线追踪全局光照（Ray-Traced GI）管线**：
    - 支持**硬件光追与软件计算着色器光追**两种模式。
    - 提供从**完全烘焙（漫反射+镜面）到完全光追（漫反射+镜面）**的全部中间选项。
- **特殊需求——隐匿处（Hideout）**：全动态的“SIM CT”建造系统，必须依赖光追GI，否则无间接光照。
- 整体设计从项目之初即瞄准**可伸缩性**，避免为强行搭载光追而牺牲其他模块性能，最终获得社区积极反馈。

**AC 系列 GI 烘焙：从均匀探针到稀疏分布**

**背景与规模矛盾**
- **前作技术回顾**
    - **《AC Unity》**：使用**均匀 Mipmap** 覆盖辐射度体积（Radiance Volumes），Mipmap 作远距离 LOD。
    - 不支持动态昼夜，采用 **4 套固定静态环境光** 伪造变化。

    - 若将 Unity/Syndicate 方案**线性外推**至 16×16 km 开放世界：

**节省存储与计算的三大核心策略**

1.  **转向 CPU 烘焙后端**
    - **驱动因素：**
        - GPU 烘焙占用大量 **显存（VRAM）**，易触发“**内存断崖**”（Memory Cliff）。
        - 消除构建农场中**不同 GPU / 驱动的差异**，确保烘焙**确定性**（Deterministic）。
        - 减少本地机器工作量，调度更快。
    - **本质优势**：**输入确定** → 明确“需烘焙什么”，避免无效计算。

2.  **密度图与差异化分辨率**
    - **核心思想**：**探针密度无需全局均匀**，这是节省空间的关键。
    - **实现方式**：使用 **密度图（Density Map）** 绘制不同区域的 GI 分辨率。
    - **对比**：《AC Unity》全局为 **50 cm 探针间距**。起源中仅**高细节区域（如城市）** 保持此分辨率，其余区域大幅降低密度。
    - **结果**：高分辨率区域面积远超旧作，但数据总量仍然过大，需进一步压缩。

3.  **稀疏探针分配与剔除**
    - **分配逻辑**：以**层级网格**形式进行**稀疏分布**。仅在**包含表面的体素**中细分放置探针。
    - **剔除逻辑**：直接丢弃**位于几何体内部的探针**（无光照贡献），进一步节省空间。
    - **压缩效果**：相比均匀满布方案，最终平均**仅保留约 10% 的探针**。

### 稀疏探针分布与存储策略

**核心思路**：通过**自适应稀疏分布**大幅减少探针数量，并利用**分离的色彩空间存储**支持动态时间与季节变化。

-   **三级稀疏探针分布**
    -   探针以**三级稀疏结构**（类似稀疏八叉树）组织。
    -   **细分条件**：只有包含几何体表面的体素才会被进一步细分。
    -   **内部剔除**：完全位于几何体内部的探针会被直接丢弃，进一步节省空间。
    -   **存储效率**：相比均匀分布，最终平均只保留约 **10%** 的探针数量。

-   **时间变化支持（Time of Day）**
    -   **关键帧机制**：共存储 **11** 个关键帧：1 个用于局部光源，10 个对应太阳的不同方位。
    -   **数据驱动设计**：该系统为纯数据驱动，可根据实际需求灵活扩展关键帧数量。
    -   **分离式色彩存储**：太阳光、局部光和天光的数据采用 **YCoCg 颜色空间** 分开存储。
        -   **亮度（Y通道）**：即 Lumen 值，**具有方向性**，以高精度色度计标准存储。
        -   **色度（Co、Cg通道）**：作为标量值存储。

-   **季节变化近似方案**
    -   **核心目标**：在必须多次烘焙 GI 的前提下，将季节数据的额外存储开销降至最低。
    -   **假设一（季节合并）**：假定春季与夏季在 GI 插值上**为同一季节**。
        -   依据：两者的烘焙光照数据（LBDO）几乎完全相同，视觉差异主要来自资产、雾气与天气的变化。
    -   **假设二（亮度恒定）**：假定 **亮度（Lumen）在所有季节中保持不变**，仅存储一次；而为每个季节单独存储**色度（Chroma）值**。
    -   **误差处理**：对于因几何体跨季节变动（如植被）而产生亮度-色度错配的物体，可由技术美术**标记为问题对象并在烘焙中跳过**。实践中问题极少。

### 运行时集成与级联体积

**目标**：将稀疏的离线数据在运行时重建为高效、连续的查询结构。

-   **稀疏体素注入**
    -   运行时，烘焙好的稀疏体素经过**解压缩**步骤变为稠密数据。
    -   解压后数据被**插入到级联式 3D 体积**（Cascaded 3D Volumes）中，这些级联在运行时是均匀采样结构。

-   **无缝细节层次混合**
    -   **跨级联混合**：在相邻级联边界执行混合，确保过渡平滑。
    -   **原位 GI 块 LOD 混合**：当新的 GI 块加载时，在块内直接进行 LOD 混合，**彻底消除因 LOD 切换导致的视觉突变（Popping）**。

-   **方案对比**
    -   **左侧（旧方案，如 AC Unity）**：采用**离散 GI 网格**，通过 Mipmap 随视距进行离散流式加载，类似 2D 纹理流式传输，虽有不错效果但细节层次呈跳跃式。
    -   **右侧（新方案）**：采用**稀疏 GI 块**插入**级联 GI 体积**，构建出**高度连续的细节层次**，实现了更平滑的视觉效果。

### 级联 GI 体积与细节层次

- **核心思路**：使用**级联 GI 体积**来构建更为连续的细节层次（Continuous Level of Detail）。
- **数据规模对比**：
    - 若采用类似 Unity 的实现方案，全局光照数据将膨胀至 **2 TB**。
    - 最终方案通过结合**密度图（Density Map）** 与**稀疏网格（Sparse Grid）**，将数据量大幅压缩至约 **9 GB**。
- **渲染质量**：
    - 方案追求的是**视觉上合理可信（Plausible and Believable）** 的近似结果，而非物理上的绝对精确。
    - 在极小的数据占用下，最终效果令人满意。

### 镜面反射的多级策略

镜面反射的求解按优先级依次采用以下三种技术：

1.  **屏幕空间反射（Screen Space Reflections）**
2.  **可重光照的 GBFR Q-maps**（一种预计算的数据结构）
3.  **动态 Q-map**：在玩家位置生成，充当**区域性 Q-map**（Regional Q-map），作为全量烘焙的补充。

### GBFR Q-map 技术细节

- **基础数据**：Q-map 存储了 **Albedo（反照率）、Normal（法线）、Depth（深度）**，并在运行时动态读取以实现重光照。
- **阴影烘焙与时间混合**：
    - 在 8 位纹理中存储 **8 个关键帧（Key Frames）**，每个比特对应一个关键帧的阴影状态。
    - 运行时根据当前**时间（Time of Day）**选择最邻近的两个关键帧，并对它们进行**混合（Blend）**。
    - 这是一个效果惊人且巧妙的技巧，由于 Q-map 分辨率不高，即使无滤波也难以察觉瑕疵。

### 季节变化的实现：变体状态

- **机制**：引入**变体状态（Variation States）**，实现数据驱动的、基于游戏逻辑的实体级变化，主要用于季节更替。
- **存储与规模**：
    - 每个 Q-map 通常存储 **3 种变体**（春/夏视为同一状态以减少数据量）。
    - 游戏中共有 **16,000 个 Q-map**（不含变体数量），多数都包含季节变体，仅部分室内场景除外。

### 动态 Q-map 优化

- **设计目标**：以极低成本渲染**代理实体（Fake Entities）**。
- **实现手段**：
    - 使用**极度减面、聚合的细节层次（Decimated, Aggregated LODs）**。
    - 采用更低分辨率的阴影。
- **效果对比**：同一位置下，动态 Q-map（右）与本地全量 Q-map（左）的渲染结果**非常接近**，能在节省大量性能的同时保持视觉连贯性。

#### 回调接口：实现内联与非内联的平滑切换

### 混合光线追踪全局光照管线

#### 整体流程：两阶段混合方案
- **核心思路**：将**逐像素光线追踪**与**探针级联**结合，高效计算间接光照。
    - **第一阶段**：逐像素追踪，产生**首个弹射**光照。
    - **第二阶段**：利用类似 **DDGI** 的动态漫反射全局光照探针级联，获取**后续弹射**光照。

#### 光线追踪管线详解（逐像素追踪）
- **混合追踪方式**，优先使用屏幕空间加速，硬件光追作为后备。
    1.  **屏幕空间光线追踪**（Screen Space Ray Tracing）
        - 先在屏幕空间追踪光线，命中则直接存入 **Hit Buffer**。
        - **作用**：大幅降低对昂贵硬件光追的依赖，是性能优化的关键。
    2.  **世界空间光线追踪**（World Space Ray Tracing）
        - 屏幕空间未命中时，**保持相同的光线参数 T 值**，回退到硬件加速的 BVH 世界空间追踪。
        - 命中结果同样存入 Hit Buffer。
- **命中重光照与未命中处理**：
    - 所有命中进入**去滤镜/重光照 Pass** 进行着色。
    - 若某像素**始终未命中**，触发 **Miss 事件**：直接**采样探针缓存**（Probe Cache）获取首个弹射的辐照度。
- **降噪与最终合成**：
    - 光线追踪结果经过降噪。
    - 可叠加**艺术驱动的 RT-AO**（光线追踪环境光遮蔽）和**高光项**。
    - 配合**极细微的屏幕空间 AO（SSAO）** 补充厘米级细节。

#### DDGI 探针级联系统
- **结构**：由 **5 层级联** 组成，共约 **10,000 个探针**。
- **更新策略**：每帧更新约 **1,000 个探针**，分摊计算成本。
- **作用**：为首个弹射之后的**后续弹射**提供高效的光照查询。

#### 为何额外添加 RT-AO？
尽管全局光照计算本身应包含环境光遮蔽，但额外添加 RT-AO 有以下关键原因：
- **频率不匹配**：
    - 逐像素光线使用**完美像素精度**，而探针为**较低分辨率**，会丢失部分 AO 细节。
- **追踪分辨率限制**：
    - 屏幕空间光线常在大多平台上以**四分之一分辨率**追踪，丢失高频细节。
- **光追世界与光栅化世界存在差异**：
    - 出于性能，**光追世界不包含植被（如草）**，导致物体与地面间的接触阴影缺失。
    - 最终叠加的 RT-AO 能有效**将房屋、树木等大型物体“压”回地面**。
- **混合细节补充**：
    - 极细微的 SSAO 项弥补追踪分辨率不足带来的厘米级细节损失。

# 基于不透明度缩放的高效光线追踪与加速结构

## 1. 加速结构与 LOD 管理

- **全局 LOD 偏移** 统一控制 BVH 中几何体的细节层级。
- **微多边形近似**：基于 **LOD 误差度量** 输出代理几何，无需依赖 Mega Geometry，为当前最佳实践（引用 NVIDIA 分享）。
- **纹理降级策略**：
  - 低端平台使用 **无绑定纹理** 或 **平均颜色**（如金属表面仅取平均基色）。
  - 支持按 **尺寸** 或 **类型** 自动测量与选择。
- **可覆盖性**：所有策略均支持 **按资产** 或 **按材质** 局部调整，以解决特殊需求。

## 2. 可见性测试的难题：Any Hit 的性能陷阱

在茂密、风格化植被场景中，传统半透明阴影存在两难：

- **精确路径**：使用 **Any Hit Shader** 评估每个交点 → **开销极大**，重叠命中过多，性能崩溃。
- **限制命中数**：过度遮挡，画面发黑 → 仅对植被做补偿被视为 **Hack**，效果不理想。

## 3. 核心解法：平均不透明度驱动的三角形缩放

- **原理**：依据材质 **平均不透明度** 整体缩放三角形尺寸。高不透明度区域保留更多几何，低不透明区域收缩，从而在 **_Closest Hit_** 模式下近似半透明可见性。
- **性能提升**：比传统 Any Hit 追踪 **快约 30%**。
- **视觉质量保障**：
  - **漫反射 GI** 与参考图高度吻合。
  - **分工机制**：
    - **屏幕空间** 解析 **最近命中** → 获得像素级精确遮挡（细节完美）。
    - **世界空间** 追踪缩放后几何 → 提供多层光照一致性（不依赖屏幕信息）。
  - 形成 **“两全其美”** 的混合方案。
- **镜面反射兼容性**：即使对于树叶等离散三角形集，缩放后几何仍能提供良好的 **近似镜面反射追踪**。

## 4. 细节补充与着色管线

- **SSO 微调项**：在四分之一分辨率光追基础上，叠加一项 **极微妙的屏幕空间遮蔽（SSO）** 以恢复厘米级细节，显著提升树冠、叶片等区域的视觉精度。
- **内联光追 + Uber Shader**：
  - 采用 **内联光线追踪** 实现材质评估。
  - 基于 **Uber Shader** 统一处理光追材质，避免 Capcom 方案中需用 JSON 文件重映射阴影的复杂元数据维护流程，降低工程风险。

- **统一材质表示**：所有材质存储于**材质表**中，通过命中几何体ID直接索引，简化了重追踪（retrace）时的材质获取。

- **季节变换支持**：在材质表中引入**材质版本**概念，以应对落叶等场景变化，避免产生过多无效遮挡，并利用**查找表（LUT）** 驱动叶片颜色变化。

- **静态积雪**：预先**烘焙到地形Vista数据**中，因此漫反射光线追踪可直接获取积雪信息，无需额外开销。

- **典型场景规模**：以Xbox Series X上的城市场景为例，约产生**2,000个BLAS**，对应**30,000个实例**，BVH总体数据量约**320 MB**（PC平台因内容更多会更大）。

#### 软件光线追踪栈（低端平台）
- **设计目标**：定位为**低端硬件**（如GTX 1070级别PC）的备选方案，未在主机上发布。
- **结构特点**：采用**三层BVH**结构，并基于**空间分区**实现局部更新——仅当单元格加载新物体或物体移动时才更新相关部分，减少构建开销。

### 光探针与辐射缓存

- **光探针的多系统复用**
  - 光探针不仅用于全局光照，也被应用于**延迟天气**及**音效**等系统
  - 参与**探针分类**：在采样时刻与采样位置，查找周围**8个最近探针**
  - 仅对**具有相同分类标记**的探针进行加权采样，避免跨类别泄漏

- **双层缓存结构**
  - **近场辐射缓存**：类似 **DDGI** 的卷积流程，负责高频细节
  - **远场辐射缓存**：作为**后备回退**，提供顶级稳定性
  - 信号稳定策略：在困难区域**追踪较短射线**，当不自信时回退至**顶级稳定的辐射缓存**

- **精度与稳定性的权衡**
  - 场景示例：极窄的开口与大量间接光照精英（高挑战性）
  - 开启后结果略**损失精度**，但有效抑制闪烁，本质是**完美精度与时间稳定性之间**的权衡

### 半透明表面的探针采样

- **随机完全路径追踪**
  - 对半透明表面执行**随机完全路径追踪**
  - 当光线击中**半透明表面**（例如日式纸门 `Sojitter`）时，在**表面双侧**采样光探针

- **双向混合**
  - 依据材质的**半透明因子**，对双侧探针光照进行混合，以近似次表面散射效果

### 全向集群光照结构

- **与常规集群光照的对比**
  - 基础仍是 **Cluster Lighting**（现代引擎标配）
  - 核心差异：常规结构基于视锥体划分（形态类似**青蛙细胞** `Frog Cells`），而全向结构将集群体**映射到以相机为中心的均匀球面网格**上

- **两级层次剔除**
  - **第一级：粗集群**，包含 $16 \times 16 \times 16$ 个子集群，总计 **260k 个细集群**
  - **剔除流程**：先执行**粗集群剔除**，再在子集群级别执行**细集群剔除**，最大化光照提交效率

### 镜面反射光线追踪

- **最后一刻的特性**
  - 游戏延期后，作为**紧急冲刺**特性实现并成功交付
  - 设计上大致遵循**光线追踪漫反射GI**的现有管线
  - 目标平台：**PS5 Pro** 和 **PC**

- **开发痛点**
  - 开发相对容易，因为大量复用了漫反射GI的已有工作
  - 主要困难来自**BVH质量问题**：这些问题在低频漫反射GI下不可见，但在高频镜面反射及**降噪**阶段完全暴露

### 三套降噪器方案

- **降噪器矩阵**
  - **经典时域降噪器**：属于 **SVGF** 算法家族
  - **NVIDIA Energy Reblur**：能量保持的递归模糊方案
  - **Sondrop MSD**：**最终交付方案**，一种特殊的**各项异性滤波器**，基于**递归模糊**思路实现，完全如同模糊操作

### 光线追踪全局光照去噪方案

我们采用的去噪方案与 **SVGF 家族** 算法类似，同时融合了来自 NVIDIA 的 **能量重分布（Energy Redistribution）** 和来自 Sondrop 的 **MSD** 技术，并最终以该融合版本发布。

- **核心滤波器设计**
    - 基于**循环模糊（Recurrent Blur）** 的时空滤波器，行为与标准模糊类似。
    - **材质遮罩（Material Masking）**：针对角色、植被和动态物体应用特定的启发式规则，专门处理**去遮挡（Disocclusion）** 等挑战性场景。
    - **球谐函数（Spherical Harmonics, SH）去噪**：
        - 信号存储于 **YCoCg** 颜色空间，其中 **Y 通道承载球谐函数数据**。
        - 对比效果：原始光线追踪漫反射输出 → 标准去噪 → 启用 SH 去噪后，最终光照与画面的真实感和稳定性有显著提升（尤其在整体大场景的全局光照中）。

### 性能数据与光线追踪策略

- **探针与逐像素追踪成本**
    - **光线追踪探针**：在大多数主机上开销约 **1 ms**（PS5 Pro 拥有更优的硬件，性能更强）。
    - **逐像素光线追踪场景**：开销约 **4~5 ms**（PS5 Pro 除外）。
    - **漫反射总预算**：合计在 **5~6 ms** 之间，分摊至图形队列与异步计算队列（ACQ）。

- **镜面反射处理**
    - 成本较高，因此采用**半分辨率**渲染（宽度减半、高度不变）。
    - 初始计划以半分辨率交付，因效果明显更佳，且符合预算。
    - **PS5 Pro** 保持了目标分辨率，因为整体光线追踪开销更低，且漫反射成本被显著分摊。

### 天气与季节系统

采用**系统性方法**实现天气与季节变化，核心组件为 **Atmos 模拟**，并通过内部图形系统进行控制。

- **Atmos 大气模拟**
    - 实时模拟并传播大气物理量，如**水汽（Vapor）、温度（Temperature）、湿度（Humidity）**。
    - 利用玩家周围的**低分辨率体素数据**进行风场碰撞与流动模拟。
    - 这些物理量直接驱动体积系统：**体积云的形成与消散**、风与降雨等。

- **渲染与实体概念**
    - 雨、雪等效果在**延迟渲染（Deferred）** 管线中完成。
    - 引入 **多状态实体（Multi-state Entities）** 概念，使场景元素能随天气/季节平滑切换状态。

动态天气与湿滑/积雪材质系统

### 1. 天气数据驱动与逻辑编排

*   **环境模拟数据**
    *   运行时围绕玩家采集**低分辨率体素数据**，计算温度、湿度、风场等物理量。
    *   这些量被传入**体积云、风、雨**等大气系统，实时控制云的生成、消散及风力降雨。
*   **逻辑编排系统**
    *   过往作品使用的是**曲线驱动的环境管理器**，逻辑有限。
    *   本作升级为**全数据驱动的 Atmospheric Graph 系统（Amines Graph）**。
    *   它从引擎和大气模块接收输入，负责调度整个天气影响的渲染逻辑。
    *   **核心驱动者**：技术美术，通过图表完全掌控一切行为。

### 2. 延迟雨渲染与湿润系统

*   **技术基础**
    *   整套雨渲染基于 **Sebastian Lagarde 的 “Red Block Post” 系列技术**。
*   **湿润与积水机制**
    *   计算场景的**湿润度 (Wetness)** 和 **积水等级 (Puddle Level)**。
    *   湿润效果通过修改材质实现：
        *   **暗化基础色**（使潮湿表面更暗）。
        *   **降低粗糙度**（形成水面般的反光）。
    *   **数据存储**：在 G-Buffer 中预留一个**单精度浮点通道**，存储**湿润度遮罩**。
    *   **特殊排除**：角色等动态网格使用独立的**角色动态材质层系统**，不参与此通用湿润流程。
*   **渐变流程**：场景在干燥、湿润、积水状态之间平滑过渡。

### 3. 雪系统架构

整个雪系统分为**静态深度雪**与**动态积雪**两大部分。

#### A. 深度雪 (Deep Snow)
*   **实现方式**：利用**数据驱动的印章工具 (Data-Driven Stamper)**，将捕捉的图案和纹理直接烙印在场景中。
*   **应用对象**：可对地形或任意**高度图**进行永久性变形，塑造深厚的雪堆痕迹。

#### B. 动态积雪 (Dynamic Accumulation)
*   **原理相似性**：与延迟雨系统高度一致，通过定义输入量驱动雪材质的渐变。
*   **区域控制**：引入由技术美术绘制的 **冷/暖区域概念**，结合**静态/动态雪掩膜**精确控制雪的落点与累积。
*   **材质修改管道**：
    1.  **反照率混合**：根据当前**雪量等级**，线性插值表面颜色，向**雪的白色反照率**靠拢。
    2.  **粗糙度统一**：同样依据雪量等级，将表面粗糙度调整为雪的粗糙度。
    3.  **透射与镜面衰减**（主要针对植被）：降低树叶的**透射率（Transmittance）** 和**镜面反射（Specular）**，模拟被雪覆盖后散射增加、高光减弱的效果。雪量越大，衰减越强。
    4.  **微观可见性遮罩**：利用 **Micro-Visibility** 技术剔除掉那些被遮挡、本该无雪的区域。
    5.  **法线方向阈值**：基于**表面法线朝向**对积雪进行阈值切除，实现**垂直表面无雪堆积**的自然法则（如竖直墙壁、树干侧面）。
*   **消融过程**：
    *   当暴风雪消散后，雪会按累积过程的**逆顺序缓慢消退**，最终完全转化为水/消失，恢复材质原有属性。

这些系统共同构成了一套完整、动态且高度可控的天气与表面交互方案。

### 动态雪积累与融化
- **雪积累规则**：雪不会堆积在垂直表面。
- **完整生命周期**：
    - 雪暴持续时，雪动态累积。
    - 雪暴消散后，雪会**反向衰减**，依次转化为**湿润状态、水坑**，最终融化并干燥。
    - 整个过程由 **Ampere 图（Ampere‘s graph）** 完全驱动逻辑。
- **设计要点**：状态转换完全内聚在一个图形化逻辑系统中，便于调整。

### 室内区域的天气遮挡
- **问题**：需要防止雪和雨错误地出现在室内。
- **复用室内体积**：直接使用为大型 GI（全局光照）系统生成的室内体积数据。
- **实现方式**：
    - 将这些体积**光栅化到一张室内深度图**中。
    - 播放天气效果时，采样该深度图计算出**遮挡因子**，剔除室内部分。
    - 该方法非常廉价，因为体积本质只是平面。
- **雨粒子遮挡**：
    - 对于雨粒子及其涟漪，额外**沿雨的方向**渲染一张周围环境的深度图。
    - 利用该深度图进行遮挡，这使得雨粒子能够“穿透”窗户等结构进入室内（即不遮挡窗口区域）。

### 性能与可扩展性管理：平台管理器
- **背景**：由于在不同性能和画质档位下会交付不同的 GI 系统，帧组成差异巨大。
- **核心组件：Platform Manager（平台管理器）**
    - **数据驱动**的性能设置，可按**平台**和**上下文**进行配置。
    - 主要用于引擎和图形系统，具备自动生成的 UI 且支持**实时编辑**。
- **配置层次**：
    - **Profile（配置文件）**：对应性能模式、画质模式等，部分设置只需切换世界，其余需**重启可执行程序**（Profile Boot Settings）。
    - **Air Context（空气上下文）**：由游戏玩法触发（如进入菜单、照片模式），用于为特定场景微调功能与性能（例如在照片模式中提升头发渲染质量）。
    - **Modifiers（修饰器）**：由数据或 3D 体积触发，可手动绘制放置，用于解决**局部性能问题**或实现特定需求（如在洞穴内部动态禁用某些世界系统）。
- **价值**：
    - 在引擎中是**一等公民**，极大简化了特性的简化和性能分析（Profiling）。
    - 方便与硬件供应商迭代，可在编辑器中**模拟主机设置**（虽有限制，但足以提供大致预期）。

### 跨平台开发与低端设备适配

*   **苹果生态移植挑战**
    *   主要困难并非性能绝对差异，而是**开发环境与工作流**的切换，例如需要直接在 **Xcode** 中工作。
    *   团队中曾有过 Mac 开发经验的成员起到了关键作用。
*   **低端设备（M1 等）优化策略**
    *   利用**平台管理器 (Platform Manager)** 动态决策各平台启用的特性。
    *   在低端 Mac 上，可能采用了**软件光栅化 (Software Rasterization)** 来保证运行，且只会裁剪那些“不会破坏游戏体验”的效果。
    *   强调伸缩性设计的价值：同一套代码库通过动态配置适配从 M1 到 M4 的大跨度硬件。

### 虚拟几何体 (Nanite) 与植被 Alpha Test

*   **当前局限与策略**
    *   **现状**：目前 Nanite **不直接支持 Alpha Test（如树叶镂空）**。
    *   **原因与底气**：没有急于攻坚此项，是因为团队已经拥有一套**超高效的 GPU 驱动 Alpha Test 渲染管线**。
    *   **现有方案表现**：即使不使用 Nanite 处理植被，现有管线依然可以**无压力渲染数百万三角形**（如演示中的大型森林）。
    *   **长远规划**：先集中精力“把不透明几何做对”，之后会专门投入时间攻克 Nanite 对 Alpha Test 的支持，因为这是一个公认的难点，需要谨慎处理。

### 帧预算与多模式渲染调度

*   **不同渲染模式的帧时间差异**
    *   核心矛盾在于 **RTGI（光线追踪全局光照）开启与关闭的帧之间**性能差异巨大，分别对应性能模式（16ms/60fps）和质量模式（33ms/30fps）。
*   **GPU 调度难题**
    *   初期尽量让不同模式的 **GPU 调度 (GPU Scheduling)** 保持接近，以避免出现模式专有的隐性图形 Bug。
    *   坦言这种统一调度在长期看几乎不可能维持，未来必然要为不同帧率模式**定制作业调度**。
    *   在多款游戏同一套引擎（“mono repo”）的语境下，更难统一调度。
*   **未来演进方向**
    *   正在尝试引入**数据驱动的帧图调度系统 (Data-driven Frame Graph Scheduling)**，以更灵活地应对多帧率、多硬件的调度需求。

### 数据驱动的帧图调度挑战与定制化需求

- **核心挑战：在统一报告中集成不同游戏上下文**
    - 需要在同一渲染报告（**mono report**）中处理**多款游戏**的渲染逻辑
    - 游戏间的差异性导致**难以用静态、统一的帧图结构**覆盖所有情况
    - 直接硬编码或手动管理多个游戏路径会使系统复杂度急剧上升

- **解决思路：走向更高程度的定制化与数据驱动**
    - **数据驱动的帧图调度**
        - 将帧图（Frame Graph）的构建与调度逻辑**外置为数据配置**
        - 运行时根据游戏类型、场景特征等**动态决定渲染 Pass 的依赖与执行顺序**
        - 避免为不同游戏维护多套固定的硬编码图结构
    - **增强的定制化能力**
        - 提供更灵活的**帧图描述机制**，允许按游戏需求添加、删除或重排 Pass
        - 支持**条件式节点与依赖**，根据不同上下文激活不同渲染路径

- **工程实践中的谨慎推进**
    - 虽然数据驱动方案更具扩展性，但团队对在该项目中**大规模推广持谨慎态度**
    - 可能考虑到：新引入的调度系统带来的**验证成本**、**对已有稳定管线的风险**，以及**性能开销**的不确定性
    - 体现了在游戏开发中，**技术激进性与项目风险之间的平衡考量**
