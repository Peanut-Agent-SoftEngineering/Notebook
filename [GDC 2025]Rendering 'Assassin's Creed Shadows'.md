## 第 2 部分

### 引擎架构演进：从多分支到统一仓库

在育碧内部曾存在大量并行维护的游戏引擎（如Anvil、Stodrop、Dunia、Voyager、UB•Art等），且每个引擎又有多个分支，这导致：

- **开发力量分散**：相同功能在不同引擎中被重复实现，资源浪费严重。
- **系统迭代周期长**：大型渲染/引擎系统从开发到成熟需要数年，代码使用寿命可达十年以上，因此“推倒重来”的机会极其有限。
- **竞争力下降**：多引擎多分支的模式最终被认定为难以持续，反而不如集中力量打磨单一共享引擎。

**当前策略**：

- 以 **Anvil引擎** 作为跨游戏、跨类型、跨品牌的统一共享引擎。
- 采用 **单一仓库（Monorepo）** 模式，所有技术团队与产品团队在世界各地协同开发同一份代码库。

## 第 3 部分

### 层级化数据结构与流式网格

Envil 引擎为了支持**长距离渲染（Long-Range Rendering）**，从底层设计了一套基于网格的数据组织方式。

-   **全局网格分区**：
    -   世界被划分为**单元格（Cells）**，天然适应多用户协作编辑。
    -   数据被组织成**流式网格层（Streaming Grid Layers）**，且所有网格的定义完全由**数据驱动**。不同项目（如《AC Shadows》与其他游戏）的网格配置可以完全不同。
-   **核心层级定义（LOD 网格层级）**：
    -   **短距 / 主网格（Short-Range / Main Grid）**：承载绝大多数常规资产。短距网格里的小道具可以在离玩家较近时就消失，对整体渲染和玩法影响小。
    -   **长距网格（Long-Range Grid）**：主要用于大型地标或兴趣点，让它们在更远距离上保持可见。
-   **加载约束与匹配**：
    -   加载网格可以在**实体级别**指定。
    -   具体距离由**LOD 选择器（LOD Selectors）**决定，但必须与加载网格的设置匹配，否则物体会在不正确的距离上消失。

## 第 4 部分

从《刺客信条：大革命》到《英灵殿》，育碧的GPU驱动管线经历了从**部分GPU驱动**到**全GPU驱动**的演进，并为此设计了一套名为**Database**的共享数据系统。

### 1. 初代GPU驱动管线：AC Unity的Batch Renderer

- **定位与目标**：基于 **DX11** 的GPU驱动渲染器，核心目标是减少昂贵的DX11绘制调用。
- **核心能力**：
    - 利用 **Multi-Draw Indirect** 对**Cluster Mesh**进行剔除与渲染。
    - 引入了**实例簇（Instance Cluster）**和**三角形（Triangle）**级别的精细剔除。
- **主要局限**：
    - 仍有大量工作保留在**CPU**端执行，并非完全GPU驱动。
    - **不支持Bindless纹理**，导致可批处理的范围受限，仅能做到**基于材质（Per-Material）的批处理**。
    - 无法统一处理某些**计算（Compute）**操作，导致渲染延迟和额外开销。

### 2. 次代GPU驱动管线：GPU Instance Renderer (GPU IR)

- **设计目标**：完全面向 **DX12** 类API，实现**全GPU驱动**。
- **核心突破**：
    - **完全Bindless（Fully Bindless）**：解除了纹理限制。
    - **基于着色器批处理（Per-Shader Batching）**：不再按材质，而是按着色器进行批处理，进一步合并状态切换。
    - **运行时批处理**：在GPU上实时构建批处理，能够处理每帧**数百万个实例的剔除**。
    - **负载全面GPU化**：大量原本在CPU端的步骤被迁移至GPU执行。

### 3. 核心支撑：Database系统

为了让CPU与GPU高效共享全场景描述，设计了独立的**Database**系统。

- **本质定义**：
    - 一个可在CPU与GPU之间共享的**高级结构化缓冲区（Super-Structured Buffer）**。
    - 遵循**数据导向设计（Data-Oriented Design）**，并内置**引用机制**（类似于指针概念，例如一个对象ID就等价于指向该对象的指针）。
- **实现方式——Shader Input Groups (SIG)**：
    - 基于内部编译器 **SIG** 生成Shader绑定和上层函数。
    - 扩展了SIG语言，使其能够直接**声明数据表（Tables）**，自动打通C++与HLSL。
    - **自动化接口**：声明表后，相关的 **getter/setter** 全部自动生成，确保两端代码访问的一致性。

- **关键特性：关系（Relations）**：
    - Database支持在不同表格之间建立**关系（Relations）**，将表与表链接在一起，形成完整的场景数据图。

## 第 5 部分

### 数据库驱动的场景表示与层级结构

#### 1. 数据库风格的数据管理（SIG 框架）
这部分介绍了如何在 CPU 和 GPU 之间以类似数据库的“表”和“关系”来组织渲染数据，实现高效、自动化的同步。

- **结构化数据表**
    - 在 C++ 和 HLSL 中使用**相同的数据结构编号**。
    - 所有访问接口（`getters`/`setters`）均为**自动生成**，显著降低了手动编码量。
    - 这提供了类似直接操作结构体的便利，但底层是表格化存储。

- **核心概念：关系（Relations）**
    - **一对一（One-to-One）**：等同于数据库中的**指针**，精确链接两个表项。
    - **一对多（One-to-Many）**：模拟一个“带计数”的指针，允许一个条目指向多个子条目。
    - **目标**：实现一套**贴近 C++ 语言习惯的网格关系和属性描述**，让 CPU 和 GPU 能够共享同一套逻辑，只是存储实例不同。
        ![截图](Rendering_%27Assassin%27s_Creed_Shadows%27_frames/chunk_05_13m07s-15m59s/frame_main_000_001307.jpg)

- **多模式数据同步（CPU ↔ GPU）**
    为了高效、按需地将数据从 CPU 传送到 GPU，SIG 支持三种同步策略：

    1.  **全量复制（Copy Mode）**
        - 最简模式，直接将整份 CPU 数据拷贝到 GPU 的字节地址缓冲区（Byte Address Buffer）中。
    2.  **脏行/脏页更新（Dirty Row/Dirty Page Update）**
        - 在 CPU 端维护一个**行级别或页级别的脏掩码（Dirty Mask）**。
        - 只将标记为“脏”的数据行或数据页复制到目标表，适合部分更新。
    3.  **CPU 暂存模式**
        - 针对**超大规模表**，为避免占用全部 CPU 内存，CPU 端**仅持有脏行**。
        - 数据一旦被冲刷（Flush）到 GPU，即可从 CPU 侧移除。

#### 2. GPU 端场景描述（CalMesh 结构）
场景在 GPU 上被组织成一个与 CPU 逻辑高度相似，但经过优化的紧凑结构，专注于几何与材质的关联。

- **CalMesh 核心构成**
    - 一个 CalMesh 最多包含 **5 个 LOD 级别**。
    - 每个 LOD 定义了**两组距离**：
        - 一组用于**主视图（Menu View）** 的可见性判断。
        - 另一组用于**阴影（Shadows）** 的 LOD 计算，支持独立调优。
    - **网格与 PSO 的绑定**
        - 通过**子网格（Sub-mesh）** 级别上的一个**批次哈希（Batch Hash）** 来关联几何体与管线状态对象（PSO）。
        - 原则：**保证每个 PSO 只对应一个批次**，这使得批次易于复制和管理。

- **声明方式**
    - 在 SIG 中通过“表”来声明上述所有数据，形式上非常贴近结构化缓冲区（Structured Buffers）。
    - 各表之间通过**行（Row）和范围（Range）关系**进行链接，构成了完整的场景图描述。

#### 3. 世界空间组织与剔除
在更宏观的层面，整个游戏世界被组织为层级树结构，以支持高性能的可见性剔除。

- **核心四叉树（Core Quad-tree）**
    - 整个世界的空间组织基于一颗**核心四叉树**。
- **剔除性能依赖**
    - 高效剔除依赖于一个由 **实体组（Entity Groups）** 和 **叶节点（Leaf Nodes）** 构成的**层级结构**。
    - 叶节点是执行具体绘制调用和关联 `CalInstance` 的最小单元。
    - 层级的结构使得引擎能够从顶层到叶子快速过滤掉不可见的节点组。

## 第 6 部分

### 叶节点与实例组织

- **世界组织为核心树**：整个场景以核心树（core tree）组织，以实现高效的剔除性能。
- **层级结构依赖**：剔除的高效性依赖于由**实体组（entity groups）**与**叶节点（leaf nodes）**构成的层级结构。
- **叶节点的本质**：
  - 它是**一组实例的集合**。
  - 将空间上邻近的实例聚集到同一个叶节点中，可以**最小化包围体（bounding volumes）**，使CPU端的粗粒度剔除更高效。
  - 因为该步骤完全在CPU上执行，所以减少包围体重叠、提高剔除精度非常关键。

![截图](Rendering_%27Assassin%27s_Creed_Shadows%27_frames/chunk_06_15m41s-18m43s/frame_main_000_001541.jpg)

### 表格数据的生命周期与GPU管线

- **核心思想**：采用类C++的接口管理表格，同时保持极高的缓存效率与数据访问模式。
- **生命周期阶段**：
  1. **声明**：分别为CPU和GPU声明独立的表格。
  2. **填充**：在CPU侧创建条目（如向`Calmesh`表格中添加网格条目），并设置各项属性。
  3. **触发更新**：调用更新接口，将修改同步至GPU（底层实现基于之前描述的复制策略）。
  4. **GPU端使用**：在着色器中直接获取并随意使用这些数据。
- **数据布局灵活性**：
  - 可通过注解指定表格的内存布局：**数组结构体（SoA）**、**结构体数组（AoS）**，甚至是两者混合。
  - 这本质上是作者希望在DX中拥有的**超级结构化缓冲（Super Structured Buffer）**。

### 完整的GPU与中央硬件管线概览

- **建议参考**：推荐阅读三篇相关文章以获取全部细节，但后续仍会描述与引擎强相关的独特步骤（如维护阶段）。

![截图](Rendering_%27Assassin%27s_Creed_Shadows%27_frames/chunk_06_15m41s-18m43s/frame_main_003_001842.jpg)

#### CPU端：粗粒度剔除与维护
- **CPU粗剔除（Course Culling）**：
  - 对叶节点进行剔除，即**对整个实例组做可见性判断**。
  - 此步骤在CPU上完成，大幅减少后续GPU需要处理的实例数量。

#### GPU端：多阶段细化剔除
- **阶段一：帧剔除（Frame Culling）**
  - 输入为通过CPU剔除的实例。
  - 针对**所有渲染Pass的视锥体**进行统一剔除。
  - 同时执行大量的**选择与混合逻辑**。

- **阶段二：每Pass剔除（Per-Pass Culling）**
  - 执行**Pass特定的视锥体剔除**与**耗尽剔除（depletion culling）**。
  - 举例：对于太阳阴影，**按级联（cascade）进行反视锥体剔除**（确保处于视锥体外但投射阴影的物体保留）。
  - **准备实例数据**：为顶点/像素着色器生成访问几何和材质数据所需的**描述符**。由于使用统一缓冲处理顶点、材质与常量，需要维护一种映射，以知晓数据访问地址。

- **运行时可选剔除**
  - 在最终执行`Draw`调用之前，可选的**簇剔除（Cluster Culling）**与**角度剔除（Angle Culling）**会被执行，进一步过滤。
  - 至此，结合之前描述的GPU实例渲染器，整个剔除流水线构成了完整的、从CPU到GPU的多层次可见性系统。

### 微多边形几何管线（Micro Polygon Geometry Pipeline）
- **来源与定位**：基于 Brian Karis 2021 年的分享，是**离散集群网格管线（Discrete Cluster Mesh Pipeline）**的演进版本。
- **核心数据组织**：网格由**集群层级结构（Cluster Hierarchy）**组成，支持**连续 LOD**。单个集群包含 **128 个三角形**（相较 AC Unity 等老管线常用的 64 个三角形更大）。
- **流式加载与光栅化**：根据可见性**按集群流式传输**。三角形根据屏幕尺寸**动态选择硬件光栅化或软件光栅化**。该管线用于 **G-Buffer** 和**阴影光栅化**。
- **内存优势**：同等几何体量下，内存占用约为传统 GPU 管线的一半。
![截图](Rendering_%27Assassin%27s_Creed_Shadows%27_frames/chunk_07_18m25s-21m52s/frame_main_000_001825.jpg)
- **技术共性**：
  - 使用 **Metis 库**进行集群划分。
  - 根据三角形大小在 **Mesh Shader** 与**软件光栅化**间切换。
  - 采用 **Two-Pass 分层缓冲**体系：先从可视缓冲（Visibility Buffer）渲染材质信息，再写入 G-Buffer。
- **关键差异**：
  - 简化算法仍依赖 **SimpliGon** 生成集群组 LOD。
  - 支持**用户自定义着色器**：通过手动 API 在 Shader Graph 中嵌入自定义代码。
  - 完全**Bindless 资源访问**（与 GPU Instance 渲染器一致）。
- **软件光栅化优化**：针对扫描线算法，**动态交换 X/Y 坐标**。当三角形呈“垂直”走向时，翻转坐标轴以**提升分支一致性（Branch Coherency）和负载均衡**。

### 规模爆炸：当GI遇到超大型开放世界

当项目转向更RPG化的超大开放世界（如《起源》）时，简单的技术缩放完全不可行：

- 若将Unity/枭雄的方案直接扩展到**16×16 km²**的世界，烘焙数据可能达到**500 GB**。
- 再叠加四季变化后，接近**2 TB**的GI数据量与**超过600天**的烘焙时间。
- 即便不考虑算力，**蓝光光盘的容量也完全无法容纳**。

### 对策一：转向确定性CPU烘焙

放弃GPU烘焙，改用CPU后端，原因如下：

- **VRAM限制**：GPU烘焙对显存要求极高，容易触发“内存悬崖”。
- **确定性**：驱动和GPU差异更少，输入确定，能精确判断需要烘焙与不需要烘焙的内容，减少不必要的计算。
- **分发效率**：由于在本地机器上执行的工作更少，烘焙任务分发更快。

### 对策二：密度图与稀疏探针放置

**烘焙质量取决于探针密度，但密度不必全局均匀**。

- 使用**密度图**手动绘制GI分辨率分布，确保**关键区域（如城市）拥有足够细节**。
![截图](Rendering_%27Assassin%27s_Creed_Shadows%27_frames/chunk_09_24m14s-27m05s/frame_main_003_002430.jpg)
- 示例：AC Unity 全局以**50 cm间距**均匀排布探针；而新方案中，只有地图中的城市（蓝色区域）达到该分辨率，但即便这些区域也已超过前作的整体面积。
![截图](Rendering_%27Assassin%27s_Creed_Shadows%27_frames/chunk_09_24m14s-27m05s/frame_aux_005_002450.jpg)
- 尽管只有小部分区域保持了高密度，**数据量仍然过大**。

### 对策三：稀疏树结构与几何剔除

进一步压缩数据：

- 探针按**稀疏八叉树（sparse octree）**分布，**仅当节点包含表面时才触发细分**。
- 自动**丢弃位于几何体内部的探针**。
- 最终探针数量约为均匀密铺方案的**10%**。
![截图](Rendering_%27Assassin%27s_Creed_Shadows%27_frames/chunk_09_24m14s-27m05s/frame_main_007_002602.jpg)

这种混合策略（CPU确定性烘焙 + 密度图分级 + 稀疏树裁剪）为超大世界的实时GI铺平了道路。

## 第 10 部分

### 简介：从均匀分布到稀疏分布的优化

- **核心问题**：全局光照（GI）探针若均匀放置，数量巨大，需要 **稀疏化** 以节省内存和计算。
- 解决方案采用 **稀疏树形结构**：
    - 按三维空间进行稀疏划分。
    - **仅包含几何体表面的区域** 才进行进一步细分。
    - **剔除几何体内部的探针**，大幅减少数量。
- 效果：
    - **平均仅需均匀分布下约 10% 的探针数量**，显著节省空间。

### 时间与季节的动态光照支持

- **时间系统（Time of Day）**：
    - **11 个关键帧**：其中 1 个用于本地光源，10 个用于太阳的不同位置。
    - 系统为数据驱动，可根据需求扩充关键帧。
    - **光照分离存储**：太阳光、本地光、天空光分别处理。
- **色彩空间与编码**：
    - 使用 **YCoCg 格式** 存储。
    - **Y（亮度/ Lumen）**：具备方向性，按高精度光度存储。
    - **Cg/Co（色度）**：按标量存储。
- **季节变化（Seasons）的近似策略**：
    - 目标：**最小化烘焙体积的额外开销**，即使需要多次烘焙。
    - **假设一**：春/夏视为同一季节进行插值。因 **LOD（细节层次）版本的资产几乎相同**，季节变化更多来自资产变更、雾效和天气。
    - **假设二**：**亮度（Lumen）在所有季节保持不变**，仅存储一次；而**色度（Chroma）按季节分别存储**。
    - 近似合理性及限制：
        - 存在植被随季节剧烈变化时，上下层几何体改变可能导致亮度与色度不匹配。
        - **解决方案**：技术美术可标记并跳过问题几何体。实践中问题不多。

### 运行时级联体积与细节层次

- **运行时的体素注入**：
    - 烘焙好的 **稀疏体素数据** 首先经 **解压缩步骤**，使其非稀疏化。
    - 然后注入 **级联的 3D 体积（Cascaded 3D Volumes）** 中。
    - 这些级联体积在运行时是 **均匀采样** 的。
- **跨级联混合与 LOD 无缝过渡**：
    - 执行 **跨级联混合（Cross-cascade blending）** 和 **GI 块 LOD 混合（In-place GI block LOD blending）**，在加载新 LOD 时避免突兀的视觉跳变。
    - 示意图对比：
        - 左图：传统的离散 GI 网格（如 AC Unity 中），中低模随视距流式离散加载。
        - 右图：本方案采用稀疏 GI 块插入级联体积，创造出 **更连续的细节层次（Continuous level of detail）**。
    ![截图](Rendering_%27Assassin%27s_Creed_Shadows%27_frames/chunk_10_26m50s-29m59s/frame_main_000_002650.jpg)
    ![截图](Rendering_%27Assassin%27s_Creed_Shadows%27_frames/chunk_10_26m50s-29m59s/frame_aux_003_002848.jpg)
- **类比 2D 结构流式加载**：该过程与基于距离的 2D 结构流式加载逻辑一致，只是应用于 3D 体积。

## 第 11 部分

### 级联全局光照体积与稀疏网格

- **级联GI体积**：为了获得更连续的细节层次（LOD），使用级联（cascaded）的GI体积结构。

1.  **屏幕空间反射（SSR）**

## 第 13 部分

### 混合光线追踪 GI 管道：集成屏幕空间、世界空间与 DDGI 探针

**核心架构**：混合 GI 系统将**屏幕空间光线追踪**、**硬件加速世界空间光线追踪**与**基于探针的 DDGI 级联**组合在一起，分两步完成间接光照计算。

- **屏幕空间追踪优先**：在屏幕空间进行光线步进，这是一层廉价的“加速”手段，避免直接使用昂贵的硬件光线。若命中，结果存入 Hit Buffer。
- **世界空间回退**：屏幕空间未命中的光线，以相同的距离（T 值）继续在世界空间借助硬件 BVH 进行追踪，结果同样进入 Hit Buffer。
- **命中数据存储**：所有追踪命中以 **PBR 属性集** 的形式存储在 **RT G-Buffer** 中，并在后续的 **去光斑 Pass（Defiltering/Relighting Pass）** 中进行重光照。

- **首次反弹与后续反弹**：
  - 若某像素完全没有追踪命中，触发 **Miss 事件**，转去采样 **探针缓存**，由此获得首次间接反弹。
  - 后续的间接反弹则通过采样 **DDGI 级联**（共 5 级、约 10,000 个探针，每帧更新约 1,000 个）来补全。

从流程上看：
```
屏幕空间射线 → [命中？] → Hit Buffer
                     ↓ 否
              世界空间射线(BVH) → Hit Buffer
                          ↓
              去光斑/重光照 Pass ← 探针缓存(Miss时)
                          ↓
                    DDGI 级联采样(后续反弹)
                          ↓
               降噪 + 可选 RT-AO + 镜面反射 → 最终图像
```

![截图](Rendering_%27Assassin%27s_Creed_Shadows%27_frames/chunk_13_35m15s-38m25s/frame_main_000_003515.jpg)

### 为什么在光照积分之后还要单独添加 RT-AO？

理论上，光线追踪的间接光照已包含环境光遮蔽信息，但实际管线中额外叠加 **RT-AO** 是出于以下艺术与技术考量：

1.  **频率差异与探针精度损失**  
    光线追踪使用精确的逐像素光线，而 DDGI 探针分辨率有限，尤其在 **AO 细节** 上会被探针插值平滑掉。

2.  **追踪分辨率折中**  
    多数平台上屏幕空间射线以 **四分之一分辨率** 追踪以提高性能，导致高频几何细节（如细缝、凹角）丢失。

3.  **追踪世界与光栅化世界的不一致**  
    出于性能原因，**跟踪世界可能不包含某些高频元素**（例如草地），导致直接追踪结果缺少这些物体本应产生的遮蔽效果。

4.  **SSO 的补充作用**  
    系统仍然评估极细微的 **屏幕空间遮蔽（SSO）**，用以补偿四分之一分辨率追踪所丢失的 **厘米级细节**。

最终添加的 RT-AO 能够显著加强大型物体（房屋、树木等）与地面之间的落地感与接触感，使画面更加扎实。

![截图](Rendering_%27Assassin%27s_Creed_Shadows%27_frames/chunk_13_35m15s-38m25s/frame_main_004_003713.jpg)

### 加速结构与微多边形细节追踪策略

**核心挑战**
在光线追踪中，使用四分之一分辨率光线追踪时，需要补充细微的屏幕空间环境光遮蔽（SSO）项来恢复厘米级表面细节，这对植被（如树冠、叶片）效果提升显著。
![截图](Rendering_%27Assassin%27s_Creed_Shadows%27_frames/chunk_14_38m04s-41m25s/frame_main_000_003804.jpg)

**加速结构构建原则**
- **双模式支持**：
  - **无绑定纹理/平均颜色**：针对极低端平台或金属纹理等，通过资源尺寸或类型来度量。
  - **微多边形处理**：基于**细节层次误差度量（LOD Error Metric）** 来决定BVH中放置的几何体密度。
- **技术定位**：这是当前（无Mega Geometry等技术下）生成合适代理几何体的最佳方式。
- **灵活覆盖**：所有规则可按资产或材质单独调整，以解决局部细节或性能问题。

**阴影追踪的陷阱与解决方案**
- **“Any Hit”着色器的两难**：
  - 追求**精确结果**时，大量重叠区域导致成本极高，在风格化森林场景中成为性能瓶颈。
  - 试图**限制命中数**时，又会产生过度遮蔽的结果，针对植被的所谓补偿更像一种不够优雅的Hack。
![截图](Rendering_%27Assassin%27s_Creed_Shadows%27_frames/chunk_14_38m04s-41m25s/frame_aux_003_003834.jpg)
- **缩放三角形策略**：根据**平均不透明度**来缩放几何三角形。
  - **性能**：比“Any Hit”方案快约**30%**。
  - **质量**：漫反射全局光照（GI）与参考图极为接近。
  - **原理**：最近命中由屏幕空间精确解析，提供像素级完美；光线追踪则提供整体的多次散射一致性，实现“双世界最优”。
  - **附加收益**：即便对于叶片这样的非连通三角形集合，也是高光重追踪的良好近似。
![截图](Rendering_%27Assassin%27s_Creed_Shadows%27_frames/chunk_14_38m04s-41m25s/frame_aux_004_003919.jpg)
![截图](Rendering_%27Assassin%27s_Creed_Shadows%27_frames/chunk_14_38m04s-41m25s/frame_aux_005_003931.jpg)

**材质与着色集成**
- 采用**内联光线追踪**，并依靠**Uber Shader**方法处理追踪材质，避免了像Capcom那样使用JSON文件重映射材质参数的复杂工作流。

## 第 16 部分

### 光照探针分类与缓存稳定性

- **探针分类与加权采样**
  - 在采样位置，寻找**周围8个探针**。
  - 探针已根据场景材质、几何等特征**进行分类**，仅与采样点**分类相同**的探针才参与加权。
  - 加权混合采样的方式与 DDGI 内辐亮度缓存思想一致。

- **双层辐亮度缓存架构**
  - **近场辐亮度缓存**（near-radiance cache）：直接由光探针卷积得到，精度较高。
  - **顶层辐亮度缓存**（radiance cache）：作为**回退**（fallback）使用，底层数据更稳定。
  - **信号稳定策略**：主动追踪**较短的光线**，当短射线未能获取可靠信息时，**回退至顶层辐亮度缓存**，以换取整体稳定性。

- **精度与稳定性的权衡**
  - 复杂场景下（大量间接光照、极窄开口）对比明显：关闭该机制时噪声严重，开启后结果**略微损失精确度**，但**大幅提升稳定性**。
  - 这是对完美精度与稳定性的经典工程折衷。

![截图](Rendering_%27Assassin%27s_Creed_Shadows%27_frames/chunk_16_43m46s-46m53s/frame_main_000_004346.jpg)

### 半透明表面的双边探针采样

- **半透明材质处理**
  - 对半透明表面执行**随机完整评估**。
  - 当光线（如 GI 射线）击中半透明表面（例如障子纸——日本纸门）时，**在表面两侧分别采样探针**。
  - 最终结果按照**半透明因子**对两侧采样值进行混合。

### 全向聚类光源结构（Omnidirectional Cluster）

- **与传统 Cluster Lighting 的区别**
  - 基于常规的 Cluster Lighting 代码构建。
  - 核心改进：将聚类体**映射到相机周围的均匀网格**（而非视锥体分块），形成全向覆盖。
  - 视频对比展示了从 **Frog Cells** 到 **Omnidirectional Structure** 的变化。

- **两级层次与剔除**
  - 结构分为 **I-clusters** 与 **clusters** 两级。
  - 每个 I-cluster 包含 **16×16×16 个子 cluster**，总计约 **26 万个 cluster**。
  - 剔除流程：先执行**粗粒度 cluster 剔除**，再在 cluster 级别执行**细粒度剔除**，大幅减少光照计算量。

### 镜面光线追踪的开发与挑战

- **后期紧急追加的特性**
  - 原本主要支持漫反射光线追踪，**镜面反射光线追踪**（Specular RT）是在项目延期后**最后关头**加入的。
  - 设计思路基本沿用**漫反射 GI 的光追方案**。
  - 目标平台为 **PS5 Pro 与 PC** 等高端硬件。

- **遇到的问题**
  - 开发相对轻松，因为大量复用了扩散 GI 的工作。
  - 主要难点来自 **BVH 质量问题**——在低频的漫反射 GI 中不易察觉，但在高频的镜面反射与降噪过程中暴露无遗。

### 降噪器方案

- **引擎内置三种降噪器**
  - **真实降噪器**（True Denoiser）：算法家族类似 **SVGF**。
  - **NVIDIA Energy Reblur**：用于能量重分布与模糊。

### Defhard 湿润与积水渲染（基于 G-Buffer 遮罩）

> 本实现参考了 **Sebastian Lagarde 的 Red Block Post 系列**。

#### 原理与步骤

-   **湿润度控制**：维持**全局湿润度（Wetness）** 和**积水等级（Puddle Level）** 两个参数。
-   **材质修改**：

    -   **降低粗糙度**：湿表面变光滑，粗糙度随湿润度下降。
-   **G-Buffer 遮罩**：
    -   在 G-Buffer 中存储单精度浮点**湿润遮罩（Wetness Mask）**，精确标记哪些像素参与湿润系统。
    -   **排除对象**：角色等动态物体使用独立的角色动态层系统，不受此全局湿润影响（图中红色区域）。

### 动态积雪系统

#### 深层积雪（Deep Snow）与地形变形

-   **实现手段**：使用**数据驱动的 Stomper**（压制器），可以**印章式（Stamp）** 应用捕获的纹理与形状。

![截图](Rendering_%27Assassin%27s_Creed_Shadows%27_frames/chunk_18_49m26s-52m33s/frame_main_010_005112.jpg)

#### 动态表面积雪（Defhard Snow）

> 原理类似 **Defhard 湿润系统**，通过修改材质输入实现。

-   **区域控制**：引入由技美绘制的**冷/暖区域遮罩**（静态 + 动态），决定雪的累积区。
-   **材质参数的动态插值**（根据当前雪量等级 `SnowLevel`）：
    1.  **反照率（Albedo）**：向雪的反照率（通常为白色）进行 `Lerp`。
    2.  **粗糙度（Roughness）**：向雪的指定粗糙度插值。
    3.  **植被半透明与散射**：雪量越高，越大幅降低植被的透光度（Transmittance）和次表面散射。
-   **逐像素遮罩约束**：
    -   **微可见性（Micro Visibility）**：利用高频细节遮罩剔除细小缝隙，避免不自然的积雪。
    -   **法线方向阈值**：根据像素法线方向剔除垂直表面（雪不积累在绝对垂直面上）。公式可表达为：
        $$Mask_{snow} = Mask_{micro} \cdot \mathbb{1}(N_z > Threshold)$$
        仅法线 Z 分量（指向上方）大于阈值的区域才接受积雪。
-   **生命周期**：
    -   暴风雪时，雪量逐渐累积，上述效果按正向执行。
    -   暴风雪消散后，雪量进入**消融状态**，反向 `Lerp` 回原始材质参数，最终化为水。

## 第 19 部分

动态积雪的定向积累与全局融化系统
定向遮挡与积累逻辑
核心概念：积雪不会在垂直表面（如墙壁、树干侧面）累积，而是基于“自上而下”的雪量高度图进行定向积累。

![截图](Rendering_%27Assassin%27s_Creed_Shadows%27_frames/chunk_19_52m17s-55m22s/frame_main_000_005217.jpg)

关键术语：高度图、定向积累、遮罩排除

工作流程：

系统维护一张表示当前积雪厚度的高度图。

利用环境光遮蔽或定向深度图，检测表面朝向：只有接近水平的面才允许积雪增加，垂直面保持无雪。

随着暴风雪持续，高度图数值增加；当暴风雪停止，系统自动反转逻辑，进入衰退阶段。

![截图](Rendering_%27Assassin%27s_Creed_Shadows%27_frames/chunk_19_52m17s-55m22s/frame_aux_001_005221.jpg)

生命周期与材料转换
这是一个完全由 Ampere’s Graph 驱动的状态机，管理雪的全部生命周期。

核心流程：

积累阶段：雪量增加，覆盖表面。

衰退阶段：雪量逐渐减少，露出底层材质。

湿润阶段：积雪融化，表面材质参数切换为“潮湿/水坑”状态。

最终干燥：水分完全蒸发，恢复原始干燥材质。

室内外空间的遮挡与排除系统
室内体积的复用与遮挡
核心概念：利用已有的室内体积数据生成深度图，作为天气效果的遮挡掩码，以极低成本排除室内的积雪和雨水。

![截图](Rendering_%27Assassin%27s_Creed_Shadows%27_frames/chunk_19_52m17s-55m22s/frame_aux_005_005240.jpg)

关键技术：

数据来源：与烘焙全局光照和光线追踪全局光照相同的室内体积盒子。

遮挡因子计算：将室内体积光栅化为一张室内深度图。由于体积多为简单的平面构成，该操作成本极低。

播放天气效果时，查询深度图以获取遮挡因子，确保室内区域完全不受积雪和雨水影响。

动态粒子的深度遮挡
核心概念：针对运动的雨水粒子，沿雨滴下落方向实时渲染一张环境深度图，用于精确遮挡。

遮挡原理：

渲染一张与雨水方向对齐的动态深度图，捕捉场景几何。

将深度图用于粒子系统：当雨粒子穿透开启的窗户时，能正确落入室内；而被墙壁或屋顶遮挡时，粒子被剔除。

同时，该深度图也用于遮挡雨水飞溅效果和涟漪，使它们只在受雨淋的表面上生成。

![截图](Rendering_%27Assassin%27s_Creed_Shadows%27_frames/chunk_19_52m17s-55m22s/frame_aux_006_005244.jpg)

数据驱动的性能配置系统
核心概念：平台管理器是一个“一等公民”系统，用于管理不同配置下的性能和图形特性，使帧结构可以完全不同。

关键术语：配置文件、上下文、修饰器

### 一、macOS / iOS 平台移植经验

- **硬件跨度巨大**：M1 到 M4 芯片之间性能差异显著，低端设备（low-end）成为主要挑战。
- **核心取舍策略**：
  - 在不破坏游戏可玩性的前提下，**妥协部分特性**以换取运行能力。
  - 并非粗暴降质，而是动态判断哪些效果可以在低端平台上舍弃。
- **开发环境挑战**：
  - 主要难点在于**直接使用 Xcode 工作和适应 Mac 生态**，这与主机及 PC 开发流程差异较大。
  - 拥有熟悉该环境的团队（“以前就在该工作室工作过”）起到很大帮助。
- **关键技术决策**：
  - 在**低端 Mac 上使用了软件光栅化（Software Rasterization）**，作为 Nanite 的降级方案，这与主讲人之前提到的平台管理器动态配置有关。
  - 平台管理器（Platform Manager）高度动态，有时甚至**不确定最终交付给某一平台的具体技术组合**，需要根据测试灵活调整。

### 二、Nanite 与植被（Alpha Test）处理

- **当前状态**：Nanite 尚**不支持 Alpha Test（透明度测试）** 几何体。
- **不急于实现的原因**：
  - 团队已有一套**极为高效的 GPU 驱动管线（GPU-driven pipeline）** 专门处理 Alpha Test 植被。
  - 该管线能够**毫无压力地渲染数百万三角形**（如现有的大规模森林场景），视觉效果足够出色。
- **开发策略**：
  - 采取“先啃硬骨头”的思路：**先将不透明几何体（Opaque Geometry）做到极致**，再回头攻克 Alpha Test 在 Nanite 中的压缩与光栅化难题。
  - 内部清楚这是难点，**宁愿多花时间也要做对**，避免仓促引入导致质量下降。

### 三、多帧率目标下的帧调度挑战

- **双模式帧时间预算**：
  - **性能模式**：16ms（60fps）
  - **画质模式**：33ms（30fps）
- **核心矛盾**：
  - 有无 RTGI（光线追踪全局光照）的帧在 GPU 工作负载上**差异巨大**。
  - 两种模式本质上是两种完全不同的帧结构，试图维持**相同的 GPU 调度逻辑**极其困难。
- **开发痛点与方向**：
  - 强行统一调度会诱发**模式特有的 Bug**，成为重大开发负担。
  - 最终不得不走向**为每种模式定制调度（tailored scheduling）** 的方案。
  - 正在探索**数据驱动的帧图调度（Data-Driven Frame Graph Scheduling）**，以灵活应对复杂度。
  - 额外挑战：代码处于**单一代码仓库（Monorepo）** 中，需要同时服务于多款不同游戏，进一步要求更高的调度定制化程度。

### 数据驱动帧图调度与多游戏上下文挑战

- **痛点：单一报告（Mono Report）的限制**
    - 在相同上下文中运行不同游戏时，传统的**单一报告**（Mono Report）工作流难以满足多游戏、多场景的调度需求。
    - 游戏之间的差异性导致渲染管线需要更高的灵活性和可定制性。

- **解决方案：数据驱动帧图调度**
    - 引入**数据驱动帧图**（Data-Driven Frame Graph）来管理渲染管线的调度。
    - 通过数据配置而非硬编码，动态调整不同游戏下的渲染任务编排，实现更高程度的**定制化**。

- **风险与权衡**
    - 团队对在该游戏中大规模推进此方案 **“有点害怕”** （scared），暗示新技术在大型项目中可能引入未知风险或稳定性隐患。
    - 最终是否全量部署，在发言中未明确结论，可能仍在探索或保守应用阶段。
