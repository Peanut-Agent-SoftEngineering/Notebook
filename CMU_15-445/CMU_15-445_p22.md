## 一、分布式数据库中的事务处理

> 主题：本讲介绍分布式数据库在OLTP场景下的事务处理机制，涵盖分布式事务的核心概念、共识协议、复制策略以及CAP定理等关键内容。

### 核心主题：OLTP 与 OLAP 的区分

- **OLTP（在线事务处理）**：聚焦于高并发、低延迟的短事务，例如银行转账、电商订单创建。面向**前端应用**，与外部世界实时交互。
- **OLAP（在线分析处理）**：聚焦于复杂查询、大规模数据聚合，用于决策支持，例如年度销售报表。通常为**只读操作**，查询模式**不可重复**、数据量大。
- 本讲重点讲解 **分布式数据库在 OLTP 场景下的事务处理机制**。

### 分布式数据库的基础架构回顾

- **三种主流架构**：
  - **共享内存**：所有节点共享同一块物理内存，虽简化了数据访问，但扩展性受限，成本高。
  - **共享磁盘**：各节点拥有独立内存，但共享持久化存储（如 SAN 或云存储）。现代云原生数据库多采用此架构，平衡了扩展性与数据一致性。
  - **无共享**：每个节点拥有独立的 CPU、内存和磁盘，数据通过水平分区（Sharding）分布。这是传统分布式数据库的典型设计，强调**完全可扩展性**。
- **关键操作：分区/分片**：
  - **水平分区**：将数据表按某个键（如用户ID）拆分成多个子集，分布到不同节点。
  - 目的是将负载分散，避免单点瓶颈。

### 事务协调的核心问题

- **定义**：分布式环境下，一个事务可能涉及多个节点上的数据修改。如何确保所有节点上的操作要么全部成功提交，要么全部回滚，是协调的核心。
- **两种协调模式**：
  - **集中式协调**：引入一个中央协调器（或中间件）来负责所有事务的提交决策。优点：逻辑简单，易于实现全局一致性；缺点：单点故障风险，协调器可能成为性能瓶颈。
  - **去中心化协调**：节点之间通过协议（如 Paxos、Raft）自行协商决定事务的提交状态。优点：高可用、无单点；缺点：实现复杂，网络开销大。

### 从单节点到分布式：提交决策的根本变化

- **单节点提交**：简单直接，只需运行 **2PL（两阶段锁）** 或 **OCC（乐观并发控制）** 的本地验证即可决定事务是否提交。
- **分布式提交**：事务涉及多个节点上的数据，核心挑战变为——**如何让所有参与节点就“事务可以安全提交”达成共识**。
- **核心概念**：分布式提交协议是解决“多节点原子性”问题的关键，确保事务在所有节点上要么全部提交，要么全部中止。

### 核心协议体系：提交协议 vs. 共识协议

- **提交协议（Commit Protocols）**：主要用于数据库系统中，决定一个分布式事务是否可以提交。
- **共识协议（Consensus Protocols）**：如 **Paxos** 和 **Raft**，属于更通用的 **复制状态机协议**，可用于维护分布式系统中的各种状态（不仅仅是事务提交）。
- **关键区分**：本讲座聚焦于**如何在数据库系统中应用这些协议**，而非深入探讨共识协议本身的全部细节。Paxos/Raft 可作为构建分布式数据库的底层工具。

### 分布式数据库的实用路径：从复制开始

- **LCL2 复制数据库**：通常是最推荐的起步方案。当你需要扩展系统时，**先做数据复制**（Replication）通常是第一步。
- **注意**：复制数据库本质上仍属于分布式数据库范畴，但相比完全分片的分布式架构，其复杂度更低，适合大多数场景。

### 一致性理论基础：CAP 与 PACELC

- **CAP 定理**：分布式系统在 **一致性（Consistency）**、**可用性（Availability）**、**分区容忍性（Partition Tolerance）** 三者中最多只能同时满足两个。
- **PACELC 定理**：是 CAP 的进阶扩展，进一步讨论了在网络分区（P）与正常运行时（E）之间的权衡，即在分区时权衡一致性与可用性，在无分区时权衡延迟与一致性。

### 终极案例：Google Spanner

- **定位**：融合了本讲座及本学期涉及的**多项核心技术的集大成者**。
- **意义**：Spanner 展示了如何将 **分布式事务、复制协议、一致性模型** 等所有组件整合，构建出一个真正可用的全球分布式数据库系统。
- **关注点**：学习 Spanner 如何在实际工程中解决各种分布式难题，是理解这些协议价值的绝佳案例。

### 工程师视角的总结与建议

- **核心思维**：面对任何宣称的“分布式数据库”产品，你需要具备**批判性分析能力**——已知这些协议的原理和陷阱，能够判断系统设计是否正确、是否存在隐藏的缺陷。
- **实际建议**：除非有特殊需求（如比特币级别的不可篡改），绝大多数场景无需极端复杂的共识协议。**从复制开始，逐步演进**是更务实的工程路径。
- **学习路径**：理解提交协议 → 掌握复制原理 → 了解共识协议基础 → 结合 CAP/PACELC 分析系统设计 → 研究 Spanner 等成熟案例进行融会贯通。

## 二、分布式系统中的共识问题

> 主题：分布式系统的核心挑战在于让一组独立的节点就某一件事达成一致，这表现为选主和提交决策两个关键问题，两者都需要共识。

### 理想与现实：共识的两种场景

- **理想情况（Happy Path）**：
  - 所有节点都存活。
  - 网络畅通无阻，消息即时送达，不丢包。
  - 这种情况下达成共识非常简单，因为所有信息都是确定和即时的。

- **残酷的现实（Real World）**：
  - **节点永久性故障（Hard Crash）**：机器掉电、硬盘损坏，重启后失去所有内存状态，像一张白纸。这是最严重的失败。
  - **节点临时性故障（Ephemeral Failure）**：这是更常见、也更棘手的情况，系统表现得不稳定。
    - **网络分区（Network Partition）**：交换机重启或网线被拔，导致部分节点之间的通信中断。消息可能部分送达，部分丢失。
    - **消息延迟（Message Delay）**：消息没有丢，但很久才到。
    - **GC暂停（Garbage Collection Pause）**：如果系统运行在Java或Scala运行时，垃圾回收器可能导致进程“假死”几秒钟。对于其他节点而言，这个被暂停的节点看起来就像**永久宕机**了，即使它实际上还在运行。这会造成严重的**误判**。

### 核心难题：如何处理“部分同意”？

在上述非理想情况下，共识变得极其困难：

- **问题**：我们可能**不想等待所有节点**都同意。例如，为了性能或可用性，我们希望只要多数节点同意就提交事务。
- **挑战**：如果 **一部分节点同意（Yes）**，而 **另一部分节点不同意或失联（No/Timeout）**，系统该如何处理？
  - 对于不同意的节点，我们是强制它们接受，还是回滚整个操作？
  - **关键点**：在共享一切（Shared Everything）的架构中，我们不需要担心这些问题，因为所有节点访问的是同一份数据和状态。而分布式系统必须面对这些决策。

### 关键假设：节点行为可控

- **核心假设**：分布式数据库系统中所有节点都是**行为良好**且**受我们控制的**。
- **行为良好的含义**：节点不会出现恶意行为（例如，一个随机互联网用户下载了数据库软件并试图参与你的事务）。这在商业场景（如银行）中是合理的——你不会让别人运行你的代码。
- **因此**：如果所有节点一致同意提交一个事务，后续不会出现某个节点反悔声称“我们没提交过”。
- **例外情况**：可能发生硬件故障或日志损坏（这些问题通过重做日志处理）。但人为恶意不在当前讨论范围内。

### 拜占庭容错 vs. 非拜占庭场景

- **拜占庭容错协议**：当你不信任其他节点时，需要采用这种协议。区块链（如比特币）构建于此之上。
- **授课者观点**：除比特币等极少数应用外，绝大多数应用不需要拜占庭容错。它**难度更高、速度更慢**。
- **本课程范围**：假定节点可信，因此**不涉及拜占庭容错**。

## 三、两阶段提交（2PC）协议

> 主题：两阶段提交是分布式数据库中最常用的提交协议，通过准备和提交两个阶段确保事务的原子性，但存在阻塞和单点故障等问题。

### 角色定义

- **协调者节点（Coordinator）**：应用服务器请求的节点，负责协调整个提交流程。
- **参与者节点（Participant）**：协调者需要与之协调的其他节点。

### 两阶段流程

**第一阶段：准备（Prepare Phase / Voting Phase）**

1. 应用服务器向协调者发送“我要提交这个事务”的请求。
2. 协调者向所有参与者发送 **准备请求**（Prepare Request），询问“是否可以安全提交该事务？”
3. 每个参与者执行事务（但不提交），根据自身状态（如资源锁、约束检查）独立做出决策，回复协调者 **“我可以提交”**（Yes）或 **“我不能提交”**（No）。

**第二阶段：提交/中止阶段（Commit Phase / Decision Phase）**

- **成功路径**：
  - **条件**：协调者**必须收到所有参与者的肯定（Yes）回应**。
  - **操作**：协调者广播 **“提交”（Commit）** 命令。
  - **结果**：所有参与者正式提交事务，并回复确认（Acknowledgment）。协调者收到所有确认后，对外宣布事务提交成功。
- **失败路径**：
  - **条件**：**任意一个参与者返回否定（No）回应**，或超时无响应。
  - **操作**：协调者立即对外宣布事务中止，并广播 **“中止”（Abort）** 命令。
  - **原因**：可能是完整性约束冲突、与另一事务冲突等，参与者无需解释原因。

#### 核心公式/算法描述

```
1. Coordinator收到"Commit Transaction X"
2. Coordinator发送"Prepare(X)" -> 所有Participant
3. For each Participant:
     - 如果成功执行X: 回复"Vote_Yes"
     - 否则: 回复"Vote_No"
4. Coordinator检查所有回复:
     - 如果全部是"Vote_Yes": 
         Coordinator发送"Commit(X)" -> 所有Participant
         所有Participant执行Commit并回复"Ack"
     - 如果有"Vote_No":
         Coordinator发送"Abort(X)" -> 所有Participant
         所有Participant回滚事务
```

### 关键约束与缺陷

#### 1. 全票通过机制
- **核心要求**：**必须等待所有（100%）参与者回应**，才能做出最终决策。
- **对比 Paxos**：Paxos 等更现代的一致性算法仅需 **多数派（Quorum / Majority）** 同意即可推进，容错性更强。

#### 2. 活性问题（Liveness Issue）
- **核心问题**：当某个参与者无响应或响应延迟时，协调者**必须阻塞等待**，直到超时或收到回复。
- **后果**：在等待期间，协调者**无法处理任何新的事务提交**，系统进入僵局状态，影响整体吞吐量。

#### 3. 单点故障风险
- 协调者是整个流程的中心，其崩溃会导致所有参与者处于不确定状态（已准备但未提交/中止），需要额外的恢复机制（如超时+日志）来处理。

### 协议简化流程图

```
成功路径：
协调者 --> 广播 "Prepare" --> 参与者们
参与者们 --> 全部回复 "Yes" --> 协调者
协调者 --> 广播 "Commit" --> 参与者们
参与者们 --> 全部回复 "Ack" --> 协调者 (事务成功)

失败路径：
协调者 --> 广播 "Prepare" --> 参与者们
某参与者 --> 回复 "No" --> 协调者
协调者 --> 立即中止 --> 广播 "Abort" --> 参与者们 (事务失败)
```

### 通俗类比

可以把两阶段提交想象成一个**集体跳伞**：

- **阶段一（准备）**：机长（协调者）问所有跳伞员（参与者）：“准备好跳了吗？” 每个跳伞员必须回答“准备好了”。
- **阶段二（提交）**：如果所有人都回答“准备好了”，机长下令“跳！”；只要有一人说“没准备好”或没人回答，机长就喊“取消！”，所有人都不跳。

**核心代价**：只要一个人掉链子，整个团队就得等着，这就是它“慢”和“不灵活”的根本原因。

## 四、两阶段提交的崩溃恢复与日志机制

> 主题：日志是崩溃恢复的关键，节点通过预写日志记录状态，在崩溃后通过日志和协调者-参与者之间的交互确保最终一致性。

### 核心概念：日志是崩溃恢复的“记忆线”

- **每个节点（协调者与参与者）** 必须维护一个 **预写日志（Write-Ahead Log, WAL）**。
- 日志内容：记录所有 **发送/接收的消息**（来源、目标、内容）以及 **本节点状态**（如已投票、已提交）。
- **目的**：当节点崩溃重启后，通过查看日志判断自己在事务中的角色和进度，从而决定下一步动作（提交、中止或询问）。

### 参与者崩溃后的恢复流程

1. **重启后检查日志**：
   - 若发现自己处于 **已准备（Prepared）状态**（即已向协调者发送“同意提交”投票），则无法单方面决定事务结果。
2. **主动联系协调者**：
   - 向协调者发送恢复请求：“事务T123，我曾在准备阶段崩溃，请告知当前结果。”
3. **协调者响应**：
   - 若协调者 **未收到** 参与者的“同意”消息 → 事务 **自动中止（abort）**（因为未获得全部同意）。
   - 若协调者 **已收到** 参与者“同意”消息，但后续“提交”指令未送达 → 协调者告知参与者“需提交（commit）”。
4. **未进入准备状态的情况**：
   - 若参与者从未发送过“同意”投票（即日志中无准备记录），则直接 **安全地将本地事务回滚（abort）**。

### 协调者崩溃后的恢复流程

- 协调者重启后，若发现事务 **处于“提交中”阶段**（已向参与者发送过“提交”指令但未确认全部完成），则需：
  - **重发“提交”指令** 给所有参与者，确保它们完成本地提交。
  - 这是一种 **强制修复（go fix it）** 动作，保证事务最终一致性。

### 关键公式/算法要点

- **日志状态机**：  
  `本地状态` ∈ { `未投票`, `已准备` (Prepared), `已提交` (Committed), `已中止` (Aborted) }
- **恢复决策逻辑**（伪代码）：
  ```
  if (role == 参与者) {
      if (日志中存在Prepared记录) {
          向协调者发送恢复查询 (transaction_id, current_state);
          根据协调者回复执行commit或abort;
      } else {
          本地直接abort;  // 从未承诺过要提交
      }
  }
  if (role == 协调者) {
      if (事务处于“提交中”阶段) {
          重发commit消息给所有参与者;
      }
  }
  ```

### 为什么必须记录日志？

- **网络不可靠**：参与者发出的“同意投票”可能丢失，协调者发出的“提交指令”可能丢失。
- **节点不可靠**：崩溃后内存状态清零，只有日志能恢复进度。
- **防止数据不一致**：通过日志和协调者-参与者之间的二次交互，确保所有节点最终对事务结果达成一致（提交或中止）。

## 五、两阶段提交中的崩溃与超时处理

> 主题：协调者和参与者的崩溃处理机制，以及网络分区导致的脑裂问题，是2PC协议需要解决的关键挑战。

### 协调者崩溃后的处理机制

**核心概念：** 当协调者在 **提交阶段（Commit Phase）** 崩溃时，参与者需要自主决定事务的最终状态。

- **默认处理策略：** 参与者等待超时（Timeout）后，默认将事务标记为 **已中止（Aborted）**
- **选举新协调者：** 需要从剩余节点中 **选举（Elect）** 一个新的协调者
- **职责移交：** 新协调者负责向所有参与者广播“中止”指令，并确保所有节点统一执行 **回滚（Rollback）**
- **协议不完整性：** 等待超时触发期间存在 **竞态条件（Race Condition）**，这是该协议的一个固有问题

### 参与者崩溃后的处理机制

**核心概念：** 对于参与者的崩溃，处理方式取决于崩溃发生的时间点。

- **准备阶段崩溃：** 如果参与者尚未收到提交消息（即在**Prepare阶段**），系统默认该事务为**中止状态**
- **心跳检测：** 通过 **心跳机制（Heartbeat）** 检测节点存活状态，长时间无响应则视为已死亡
- **通信超时：** 使用**定时器（Timeout）** 作为基础检测手段

### 网络分区带来的问题

**核心挑战：** 当出现**网络分区（Network Partition）** 时，节点本身可能正常运行但无法互相通信，这会导致严重问题。

- **误判问题：** 节点A认为节点B已死亡，同时节点B也认为节点A已死亡
- **双主冲突：** 双方都会尝试**争夺控制权（Take Control）**，造成系统状态不一致
- **分脑问题：** 这是典型的 **脑裂（Split-Brain）** 现象，需要专门的**故障处理机制（Handling Mechanism）** 来解决

## 六、两阶段提交的性能瓶颈与优化策略

> 主题：2PC协议的性能瓶颈主要来自网络通信开销，通过提前准备投票和提前确认等优化策略可以减少延迟。

### 性能瓶颈

**核心概念：** 2PC协议的代价主要来自网络通信开销，特别是在广域网（WAN）环境下。

#### 通信开销分析

- **基本通信次数：** 每笔事务需要 **两轮往返（Two Round Trips）** 通信
- **地理延迟极限：**
  - 地球周长对应的光速延迟约为 **300毫秒**
  - 这是物理层面的**下限（Lower Bound）**，实际延迟只会更高
- **性能影响：** 若总是等待网络响应，**每秒只能提交约10笔事务**（10 TPS）

#### 性能公式

```
每笔事务延迟 = 2 × 网络往返时间（RTT）
吞吐量上限 ≤ 1 / （2 × RTT）
```

### 优化一：提前准备投票（Early Prepare Voting）

- **核心概念**：将**查询**（Query）与**预投票**（Prepare Vote）**合并发送**
- **工作流程**：
  1. 应用程序/中间件向节点发送查询请求
  2. 同时附带消息：“我稍后会对此事务发起提交，你先准备”
  3. 节点执行查询、返回结果，并**提前给出prepare阶段的投票**（同意/拒绝）
  4. 后续实际进入2PC prepare阶段时，该节点**无需再次响应**，节省一轮通信
- **关键术语**：**Piggyback（搭便车）**——在现有消息中携带额外控制信息
- **适用场景**：**极其有限**，仅在以下情况可用：
  - 应用程序能**确定这是最后一个查询**
  - 常见于**存储过程（Stored Procedure）** 或**RPC**系统
  - 传统API（JDBC/ODBC）**缺乏“last query”命令**，因此不适用

### 优化二：Prepare后提前确认（Early Acknowledgment after Prepare）

- **核心概念**：在**收到所有prepare阶段的投票后**，立即通知应用程序事务已提交，**无需等待第二阶段（commit/abort）完成**
- **工作流程**：
  1. 协调者向所有参与者发送prepare消息
  2. 收到所有参与者的**同意投票**后
  3. **立即**告知应用程序：“事务已提交”
  4. 后台才发送第二阶段commit消息给参与者（异步并行执行）
- **关键特性**：
  - 这是**2PC协议中最常见的优化**，几乎所有工业实现都采用
  - 减少应用程序感知的**提交延迟**（从2轮RTT降至1轮RTT）
- **时序对比**：
  - **无优化**：App → prepare → 投票 → commit → ack → 通知App（2轮网络延迟）
  - **有优化**：App → prepare → 投票 → **立即通知App**（1轮网络延迟）

### 两种优化对比总结

| 优化方案 | 减少的交互轮数 | 通用性 | 典型应用 |
|---------|--------------|-------|---------|
| Early Prepare Voting | 减少prepare阶段的请求 | **低**（需确定性最后查询） | 存储过程、RPC |
| Early Acknowledgment | 减少commit阶段的等待 | **高**（几乎通用） | 所有2PC系统默认实现 |

- **核心公式**：优化后的2PC延迟 ≈ **1.5 RTT**（原2PC为2 RTT，依赖网络拓扑）
- **工业实践**：几乎所有数据库分布式事务文档都会提及Early Acknowledgment，而Early Prepare Voting是**小众优化**

### “逻辑提交”与性能权衡

#### 核心概念：提前告知应用“已提交”

- **核心思想**：在2PC中，协调者**无需等待所有参与者完成第二阶段（Commit/Abort）响应**，即可向应用宣告“事务已提交”。
- **关键时机**：当协调者收到**所有参与者对“准备阶段”（Prepare Phase）的确认（ACK）** 后，**立即**通知应用事务已提交。
- **实现依据**：协调者此时已将“所有参与者同意提交”的信息**写入日志并刷入磁盘**。即使后续协调者崩溃，重启时也能通过日志恢复，并继续完成第二阶段的提交操作。

#### 逻辑提交 vs. 物理提交

- **逻辑上已提交**：协调者收到所有ACK后，从**全局一致性**角度看，事务已确定要提交，因为所有参与者都已承诺会提交。
- **物理上尚未提交**：对于**参与者节点**而言：
  - 它们**尚未收到“正式提交”指令**。
  - 它们**不知道事务是否真的已提交**（协调者可能在此时崩溃）。
  - 该事务的修改在参与者节点上**尚未持久化**，处于“**未决状态**”。

#### 关键问题与权衡

1. **读取一致性问题**：
   - 如果在“逻辑提交”后、参与者节点收到正式提交指令前，有外部请求读取该事务的修改，可能需要**阻塞（stall）** 该读取操作，等待参与者节点确认提交状态。

2. **协调者仍是单点瓶颈**：
   - 所有事务的提交过程**必须经过协调者**，这本质上并未解决分布式数据库的单点故障问题（**瓶颈**依然存在）。
   - **性能瓶颈主要不在于CPU**：协调者的CPU处理能力通常不是瓶颈。
   - **真正的瓶颈是网络往返延迟**（Network Round Trips）：2PC需要等待多次网络消息（Prepare Phase的广播与ACK收集），**网络延迟**才是关键开销。

3. **优化方向**：
   - **不同事务可由不同节点作为协调者**，分散提交压力，避免所有事务都集中在一个节点上。

## 七、Paxos 共识协议

> 主题：Paxos是首个被证明正确的共识协议，通过多数派同意机制解决分布式系统中的一致性问题，但实现复杂且存在活锁问题。

### 核心背景与历史

- **Paxos 的诞生**：由 **Leslie Lamport** 提出，最初是为了试图证明“在异步网络中不存在既安全又具备弹性的协议”，结果却意外地设计出了 Paxos 本身——一个可以达成共识的协议。
- **原始论文**：《The Part-Time Parliament》（约 1991-92 年），风格极其奇特。
  - 以虚构的希腊岛屿 Paxos 和古代部落的石板故事来隐喻协议。
  - **阅读体验**：更像是一种文学消遣或好奇心驱动，**直接学习协议的效率不高**。
- **后续论文演化**：
  - **《Paxos Made Simple》**（Leslie Lamport 后续编写）：依然公认**难读**。
  - **《Paxos Made Live》**（Google，约 2003-04 年）：被广泛认为是**最容易理解的 Paxos 实践讲解**，真正让许多人“顿悟”的点。
  - **Raft 协议**（Stanford 设计）：**设计初衷**就是成为一个类似 Paxos 但**更容易被人类理解**的共识协议。但即便如此，阅读难度依然不低。

### 核心概念：Leader 与日志同步

- **领导者（Leader）选举**：在 Paxos 中，**任何人都可以成为 Leader**，只要通过选举过程。
- **关键难点**：成为 Leader 后，必须执行最关键的一步——**更新自己的日志**。
  - 目标：确保所有节点的日志**重新回到同步状态**。
  - 这本质上是 **Paxos 中最麻烦的边界情况（Corner Cases）处理**的一部分。
- **视角声明**：本讨论专注于**交易（Transactions）上下文**中的 Paxos 应用，刻意忽略 Paxos 中诸多细节（如不同的提交协议、各种边界条件）。

### 学习路线建议

1. **阅读顺序**（从易到难）：
  - **第一选择**：Google 的 **《Paxos Made Live》**（最推荐，实践性强）。
  - **第二选择**：学习 **Raft 协议**（设计上更贴近人类思维）。
  - **最后**：如果对理解设计原委感兴趣，可以阅读 Lamport 的原始或简化论文（了解理论根基）。
2. **重要警告**：
  - **“魔鬼在细节中”**：Paxos 的实现遍布棘手的细节和边界情况。
  - **不推荐**从 Lamport 原版论文入手，容易因叙事方式而理解困难。

### Paxos 协议的核心角色与术语

- **参与者重命名**：与基础共识算法（如 Multi-Paxos 的简化版本）不同，Paxos 引入了三类角色：
  - **Proposer（提议者）**：负责发起提案（Proposal）。相当于基础共识中的“协调者（Coordinator）”。
  - **Acceptor（接受者）**：负责投票和存储已选定的值。相当于基础共识中的“参与者（Participant）”。
  - **Learner（学习者）**：学习已被选定（Chosen）的最终值，但不参与投票。用于复制状态或故障恢复。
- **核心区别**：
  - **Proposer** 只负责“提议”，不负责最终决策。
  - **Learner** 是被动接收者，用于观测最终一致性结果（Raft 中称为 “Follower” 的变体，但功能不同）。
  - **Acceptor** 是核心投票节点。

### Paxos 的通信模型

#### 第一阶段：提议（Prepare）

- **关键步骤**：
  - **Proposer** 向所有 **Acceptor** 发送一个 **Commit Request**（实质上是一个“准备请求” Prepare Request）。
  - 请求内容为：“**我提议一个值（Value），请问你们是否愿意接受（Prepare）**”。
- **核心特性**：
  - **不保证即时回复**：Acceptor 可能拒绝（比如已经接受了更高编号的提议），也可能等待。
  - **编号机制**：每个提议附带一个唯一且递增的全局 **Proposal Number**（提案编号）。编号越高，优先级越高。
- **算法本质**：通过两阶段（Prepare → Accept）实现“多数派同意”才能选定值。

#### 第二阶段：接受（Accept）

- 提议者收到多数派的Promise后，发送Accept请求（包含提案值 `v`）。
- 如果接受者未对更高编号的提案做出Promise，则**接受（Accept）**该提案，并广播给学习器（Learners）。

#### 冲突与重试

- 若另一个提议者突然提出编号 `n+1` 的提案，所有接受者一旦看到 `n+1`，会**立即拒绝**编号 `≤ n` 的后续提案。
- **解决方案**：失败方必须提出更高编号的提案（如 `n+2`），重复Prepare-Accept循环。

### 核心公式与规则

- **单调递增编号**：`Proposal_N_{new} > Proposal_N_{old}`，否则被拒绝。
- **接受者行为**：一旦接受编号 `N` 的提案，则承诺忽略任何编号 `< N` 的后续请求。
- **多数派原则**：提案必须获得**法定多数（Quorum）**的接受者同意才能提交。

### 两阶段提交与 Paxos 的关系

- **核心发现**：Leslie Lamport（Paxos 发明者）和 Jim Gray（两阶段提交早期采用者）的研究表明
- **关键结论**：**两阶段提交是 Paxos 的退化情况（degenerative case）**
  - 如果在 Paxos 中要求所有节点都同意提交事务，本质上就等同于两阶段提交
- 这意味着两阶段提交在容错性上比 Paxos 更严格（要求更高的一致性）

### 公式化总结

- 两阶段提交：$Commit \iff \forall n \in Nodes: agree(n)$
- Paxos：$Commit \iff \exists majority \subseteq Nodes: agree(majority)$
- 关系：$2PC = Paxos$ 当 $|majority| = |Nodes|$

## 八、Multi-Paxos：解决 Paxos 的活锁问题

> 主题：Multi-Paxos通过引入领导者机制解决基础Paxos的活锁问题，使系统能够连续提交多个事务。

### 核心问题：Paxos 的活锁（Liveness Issue）

- **问题现象**：在基础 Paxos 协议中，当有多个提议者（Proposer）同时竞争时，系统可能陷入**活锁**状态。
  - 提议者A提出 `n`，提议者B提出 `n+1`。
  - 提议者A看到 `n+1` 后，提出 `n+2`。
  - 提议者B看到 `n+2` 后，提出 `n+3`。
  - 如此循环往复，提案编号不断增大，但**没有任何一个提案最终被提交（Commit）**。
- **根本原因**：没有一种机制来协调多个提议者，导致他们互相覆盖、互相阻塞。
- **关键术语**：**Liveness Issue (活锁问题)**，**Thrashing (系统颠簸/抖动)**。

### 解决方案：Multi-Paxos

- **核心思想**：引入**领导者（Leader）** 或**唯一提议者（Single Proposer）** 的概念。
- **工作原理**：
  1. **选举一个领导者**：通过某种机制（如租约、选举算法）选出一个唯一的提议者。
  2. **领导者的特权**：这个领导者被授权可以**连续地提议并提交多个事务（Transactions）**，而无需担心其他提议者的干扰。
  3. **串行化提案**：在领导者的任期内，所有提案都由它一个节点发出，从根本上消除了多个提议者互相竞争导致的活锁。
- **关键术语**：**Multi-Paxos**，**Leader (领导者)**，**Single Proposer (唯一提议者)**。

### 领导者机制：租约（Lease）

- **实现方式**：选举出来的领导者拥有一段**租约（Lease）**。
  - 在租约有效期内，它就是唯一被认可的提议者。
  - 其他节点在此期间不会尝试发起新的提案。
- **租约时长**：一个较短的时间段，例如 `2秒` 或 `10秒`。
  - 时间太短可能导致频繁选举。
  - 时间太长可能导致领导者故障时系统不可用时间过长。
- **核心优势**：通过“快照”的方式，将多节点竞争的“**并发问题**”转化为单节点串行的“**顺序问题**”，极大地简化了协议并保证了进度（Progress）。
- **关键术语**：**Lease (租约)**，**Anointed Leader (指定的领导者)**。

### 核心公式/算法点

- **活锁的数学模型**：`Proposer A: n -> Proposer B: n+1 -> Proposer A: n+2 -> Proposer B: n+3 -> ...` 无终止循环。
- **Multi-Paxos 的简化模型**：`Leader L` 在租约期内，连续提出 `Tx1, Tx2, Tx3,...`，所有接受者只需针对这个领导者的提案进行**二阶段提交（Prepare-Accept）**，且通常可以跳过大部分 Prepare 阶段。

### 工程实践要点

- **节点角色**：提议者和接受者（Acceptor）**可以是同一组节点**。不需要为提议者单独设立一批节点。
  - 大多数系统实现中，节点既可以是提议者（被选为 Leader 时），也可以是接受者。
- **处理“已看到”的提案**：
  - 即使接受者已经接受了领导者的提案（如 `n`），它仍然可以**看到**另一个节点发来的更高编号的提案（如 `n+1`）。
  - **关键**：看到并**拒绝**`n+1` 以维护 Leader 的权威，或者告知 Leader 有冲突，但**这不影响**已完成的 `n` 的提交。`n` 可以立即返回结果，`n+1` 的干扰被 Leader 机制屏蔽。

## 九、分布式共识中的状态同步与一致性

> 主题：在分布式共识中，节点状态不同步、读取一致性保证、网络分区处理以及完整性约束等问题需要系统化的解决方案。

### 节点状态不同步问题

- **核心场景**：分布式系统中，当某些节点（如节点3）未提交一笔交易，而其他节点已提交时，该节点会处于**局部过时状态**。
- **核心术语：全局一致性与事务一致性**
  - 事务的提交意味着系统状态向前推进（state moved forward）。
  - 节点3因缺少最新提交的事务，**不再同步（out of sync）**，无法提供与全局一致的数据。
- **关键结论**：在节点恢复同步之前，**不能从该节点读取数据**，否则将破坏一致性。

### 节点同步机制：日志回放（Log Replay）

- **如何使过时节点更新？**
  - 过时节点需向其他存活节点主动请求缺失的**预写日志（Write-Ahead Log, WAL）条目**。
  - 核心步骤：`“嘿伙计们，我错过了那笔交易，看起来是个好东西，请把日志条目给我，让我恢复正确状态。”`
- **核心公式/过程**：
  ```
  过时节点 → 请求缺失的WAL条目 → 从Leader/协调节点获取 → 按序回放日志 → 状态追上最新提交点
  ```

### 读取一致性保证：Lease与协调者

- **如何确保读操作不读到过时数据？**
  - 引入**租约（Lease）** 机制，明确当前的最新版本号。
  - 所有读请求必须经过**协调节点（Proposal Leader）**，询问“我该读取哪个版本的数据？”
- **核心目的**：通过中心化查询最新版本，避免从过时节点读数据。

### 网络分区与超时（防止Split Brain）

- **场景**：当某节点（如节点3）因网络问题无法与多数派（Majority）通信时，必须等待**超时（Timeout）** 后才能选举新Leader。
- **核心原则**：
  - 如果不等待超时，多个分区可能各自选举Leader，导致**脑裂（Split Brain）**，破坏一致性。
  - 超时前，所有节点保持现有状态，禁止单方面推进。

### 完整性约束（Integrity Constraint）与分布式提交模式的抉择

- **问题本质**：如果节点3在提交时发现本地完整性约束被违反（如唯一性约束、外键约束），该事务是否应该被全局拒绝？
- **不同情况分析**：
  - **如果系统是副本复制（Replicas）**：所有副本应拥有相同的完整性约束，约束违反会在所有节点同时出现，因此单节点约束检查足以代表全局。
  - **如果是分片（Sharded）**：可能存在**局部性完整性约束**（例如某分片的本地规则）。此时依赖单一节点的本地约束检查**不足以保证全局正确性**。
- **解决方案：两阶段提交（Two-Phase Commit, 2PC）**
  - 在分片场景下，必须使用2PC确保所有参与节点“全体
