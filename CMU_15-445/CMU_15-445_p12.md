### 一、编程实践建议与调试策略

#### 1. **防御性编程（Defensive Programming）**

- **核心概念**：在代码中广泛插入**断言（asserts）** 和**不变量检查（invariant checks）**，尤其是在并发数据结构（如B+树）中。
- **关键术语**：**assert**、**invariant check**、**null pointer trap**
- **目的**：在错误发生的**即时位置**捕获问题，而不是等到5个函数调用后才因段错误（segfault）而崩溃。
- **最佳实践**：确保在数据结构从一个状态转换到下一个状态时，传递的数据始终是正确的。

#### 2. **性能分析工具**

- **核心概念**：使用**性能分析器（profiler）** 找出代码中的热点（hot spots）。
- **推荐工具**：
  - **Callgrind**（Valgrind套件的一部分）——轻量级，适合本课程项目
  - **perf**（Linux性能工具）——功能更强大，但可能“大材小用”
- **关键术语**：**profiler**、**hot spot**、**bottleneck**
- **行动指南**：运行分析器后，问自己：“我是否在做什么蠢事？如何才能加速这段代码？”

### 三、处理模型（Processing Models）

#### 1. **核心问题**

> 数据库系统如何遍历查询计划树（query plan tree）并执行各个算子（operators）？

#### 2. **三种主要模型**

| 模型 | 核心思想 | 优点 | 缺点 |
|------|----------|------|------|
| **迭代器模型（Iterator Model）** | 每个算子实现 `open()` → `next()` → `close()` 接口，**一次一行**（volcano/pipeline模型） | 简单通用，支持流水线 | 函数调用开销大 |
| **物化模型（Materialization Model）** | 每个算子**一次性输出所有结果**（materialized） | 适合OLTP，减少调度开销 | 内存压力大，不适合大结果集 |
| **向量化/批处理模型（Vectorized/Batch Model）** | 每次`next()`返回**一批元组**（batch/vector），如1024行 | 减少函数调用，利用SIMD指令，缓存友好 | 实现复杂度较高 |

- **关键术语**：**Iterator (volcano) model**、**Materialization model**、**Vectorization model**
- **实际应用**：
  - **OLTP**（事务型）：通常使用**物化模型**
  - **OLAP**（分析型）：通常使用**向量化模型**（如MonetDB、ClickHouse）

#### 3. **公式/算法要点**

```
迭代器模型执行流程：
Loop: result ← child.next()  →  process result  →  emit to parent

物化模型执行流程：
results ← child.output()  →  process entire result set  →  emit materialized output

向量化模型执行流程：
Loop: batch ← child.next_batch(N)  →  process batch (vectorized)  →  emit batch
```

---

### 四、访问方法（Access Methods）

#### 1. **核心问题**

> 数据库如何从表中获取数据给查询算子？

#### 2. **顺序扫描（Sequential Scan / Full Table Scan）**

- **核心概念**：**页级遍历**，通过缓冲池管理器（buffer pool manager）顺序读取数据页
- **关键术语**：**pages**、**buffer pool**、**I/O cost**
- **适用场景**：
  - 表中大部分数据需要被访问
  - 表很小（小于内存）
  - 查询没有合适索引
- **优化变体**：
  - **Zone Map / 数据块统计信息**：跳过不包含目标数据的块
  - **Late Materialization**：仅在最后才组装完整元组（对宽表有效）
  - **并行顺序扫描**：多线程同时扫描不同段

#### 3. **索引扫描（Index Scan）**

- **核心概念**：通过索引结构（B+树、哈希索引）**直接定位**满足条件的元组
- **关键术语**：**index lookup**、**B+树 traversal**、**index-only scan**
- **适用场景**：
  - 查询条件具有高选择性（selectivity）
  - 需要范围查询（range query）或排序输出
- **子类型**：
  - **Index Scan**：通过索引找到元组ID，再回表（fetch）获取完整数据
  - **Index-Only Scan**：索引本身包含查询所需的所有列，**无需回表**，性能最佳

#### 4. **选择策略**

- **核心原则**：基于**代价估计（cost estimation）** 和**统计信息（statistics）**
- **关键术语**：**selectivity**、**cardinality**、**cost model**
- **公式**：
  ```
  顺序扫描代价 ≈ 表中的页数（pages）
  索引扫描代价 ≈ 索引遍历代价 + 回表页数（pages fetched）
  当 选择性较低（selectivity high）→ 顺序扫描更优
  当 选择性较高（selectivity low）→ 索引扫描更优
  ```

---

### 五、总结与重点提炼

- **防御性编程**：用`assert`和`invariant checks`预防性捕获bug
- **性能分析**：使用`Callgrind`或`perf`找出热点，针对性优化
- **处理模型**：
  - **迭代器模型** = 逐行流水线 → 简单但函数调用开销大
  - **物化模型** = 一次性全结果 → 适合OLTP，内存压力高
  - **向量化模型** = 批量处理 → 适合OLAP，缓存/SIMD友好
- **访问方法**：
  - **顺序扫描** = 全表扫 → 低选择性场景
  - **索引扫描** = 索引查找 → 高选择性场景
  - **Index-Only Scan** = 无需回表 → 最佳性能
- **核心公式**：
  - 扫描代价 ≈ **页面数量**
  - 索引扫描代价 ≈ **索引深度 + 回表页面数**

## 查询处理模型（Processing Models）

### **核心概念**
数据库系统的查询处理模型定义了**查询计划树的遍历方式**、**算子间的数据传递形式**以及**执行方向**（自顶向下或自底向上）。

### **迭代器模型（Iterator Model）**
- **别称**：火山模型（Volcano Model）、管道模型（Pipeline Model）
- **历史**：由B+树论文作者Graefe在80年代末90年代初的学术项目“Volcano”中系统化提出，虽非首创，但首次明确定义了该模型。
- **核心机制**：
  - 每个算子实现 `Next()` 接口，**每次调用返回一条元组（Tuple）** 或空标记（表示结束）。
  - 查询计划树**自顶向下**驱动：父算子调用子算子的 `Next()` 请求数据。
  - 数据流是**按需拉取（Pull-based）** 的，一次一条元组。
- **特点**：
  - 最直观、最易实现，是大多数非数据仓库型数据库系统的默认选择。
  - **最符合人类对代码执行的直觉**。
- **局限性**：对于数据仓库/分析型负载，可能不是最优选择，因为逐元组调用开销较大（函数调用次数多、CPU缓存不友好）。

### **物化模型（Materialization Model）**
- 与迭代器模型不同，物化模型让每个算子通过**一次调用**返回**所有结果**（即物化完整的输出）。
- 适用于**OLAP（在线分析处理）** 场景，减少算子间的函数调用，但需要更多内存来存储临时结果。

### **向量化/批处理模型（Vectorized/Batch Model）**
- 迭代器模型和物化模型的折中方案。
- 算子每次调用 `Next()` 返回**一批元组**（例如1024条数据）。
- **优势**：
  - 显著减少函数调用次数（相比迭代器模型）。
  - 提升**CPU缓存局部性**（一次处理连续内存中的多个数据）。
  - 易于利用**SIMD（单指令多数据）** 指令集进行并行处理。
- 现代分析型数据库（如ClickHouse、DuckDB）广泛采用。

## 存取方法（Access Methods）

### **顺序扫描（Sequential Scan）**
- **核心思想**：直接遍历表的整个数据页，逐条检查元组是否符合条件。
- **适用场景**：当没有索引、查询条件选择性低（需访问大量元组）或表很小时。
- **优势**：简单、磁盘I/O顺序性好（预读取友好）。
- **劣势**：对全表逐字节扫描，数据量大时性能差。

### **索引扫描（Index Scan）**
- **核心思想**：通过索引（如B+树、哈希索引）快速定位满足条件的元组，避免全表扫描。
- **适用场景**：查询条件**选择性强**（只需要少量元组），且存在对应索引。
- **关键术语**：
  - **主键索引扫描（Primary Key Scan）**：通过索引获取元组位置，直接随机访问数据页（可能带来大量随机I/O）。
  - **二级索引扫描（Secondary Index Scan）**：先通过二级索引找到元组ID，再回表查询完整数据（可能导致**回表（Lookup）** 开销）。
  - **索引覆盖扫描（Index-Only Scan）**：所需字段全在索引中，无需回表，性能最佳。

## 修改数据的查询（Queries That Modify Data）

### **核心问题**：当查询包含INSERT、UPDATE、DELETE时，数据库需要处理**簿记（Bookkeeping）** 工作。

- **并发控制**：确保修改操作的原子性和隔离性（如加锁、MVCC多版本并发控制）。
- **索引维护**：更新或删除元组时，必须同步更新相关索引（例如B+树的插入/删除操作）。
- **约束检查**：主键唯一性、外键完整性、CHECK约束等。
- **日志记录**：为了崩溃恢复，写入WAL（预写式日志）。
- **触发器和约束级联**：外键的ON DELETE CASCADE等逻辑。

## 表达式与条件子句（Expressions and Where/Having Clauses）

### **表达式求值**
- 查询中的 `WHERE`、`HAVING`、`JOIN ON` 等条件会被解析为**表达式树**。
- 数据库需要高效求值：
  - **逐行解释执行**：遍历表达式树，对每行数据求值（简单但慢）。
  - **JIT编译（即时编译）**：将表达式编译为目标机器码执行，大幅提升性能（现代数据库如PostgreSQL 12+、ClickHouse支持）。
- **常见优化**：
  - **短路求值（Short-Circuit Evaluation）**：逻辑 `AND` 或 `OR` 中，一旦条件确定立即停止计算。
  - **谓词下推（Predicate Pushdown）**：将过滤条件尽量下推到表扫描或索引扫描阶段，提前减少数据量。

# 迭代器模型（Iterator Model）与流水线处理

## **核心概念：迭代器模型**
- **基本思想**：查询计划中的每个物理运算符（如Join实现、Scan操作符）都提供一个 **`next()` 函数**接口
- **每次调用** `next()` 时，操作符返回两种结果之一：
  - **返回一个元组**（Tuple）：可以是列的子集，也可以是记录ID（Record ID）
  - **返回空指针**（Null Pointer）：表示没有更多元组需要处理，告知调用者停止调用
- **关键机制**：父节点操作符调用子节点的 `next()`，从查询计划底部向上拉取数据，直到生成最终输出返回给客户端

## **流水线模型（Pipeline Model）解释**
- **核心目标**：理想情况下，希望**尽可能地将单个元组沿着流水线向上推送**，直到无法继续处理，然后再返回底部获取下一个元组
- **工作流程**：
  1. 从顶部根操作符开始，调用其 `next()` 函数
  2. 根操作符递归调用子操作的 `next()` 函数
  3. 数据从底部向上流式传输

## **实际示例分析**
### 查询语句：
```
SELECT * FROM R, S WHERE R.id = S.sid AND S.value > 100
```

### **物理操作符实现特点**
- **叶子节点**（Scan操作符）：直接遍历目标表中的元组
- **内部节点**（如Join、Filter）：通过 **`for` 循环** 调用子节点的 `next()` 获取数据

### **执行顺序**
1. **根操作符**（Projection投影）调用 `next()`
2. 逐层向下递归调用子操作的 `next()` 函数
3. 数据从底部叶子节点开始，沿着查询计划树向上流动

## **关键术语与公式**
- **`next()` 函数**：迭代器模型的核心接口，返回值类型可表示为：
  ```
  Tuple next() {
      if (hasMoreTuples) return tuple;
      else return NULL;
  }
  ```
- **元组（Tuple）**：可以是完整行、列子集或记录ID
- **空指针（Null Pointer）**：流结束信号，控制迭代终止

## **重要说明**
- **发明者**：虽然没有完全发明此概念，但他是**首次系统化编码和解释**迭代器模型的实际含义
- **并行化技术**：发明了使迭代器模型能够并行化的技术（将在下节课讨论）

## 第 5 部分

## 基于拉取模型（Pull Model）的查询执行与流水线

### 1. 核心概念：**拉取模型（Pull Model）与迭代器（Iterator）模式**

- **基本机制**：每个查询操作符都实现一个 `next()` 函数，其父操作符通过调用子操作符的 `next()` 来“拉取”数据。
- **数据流**：根操作符（通常是投影 `Projection`）调用其子操作符的 `next()`，子操作符再调用其子操作符的 `next()`，形成**自上而下的调用链**。
- **关键性质**：每次 `next()` 调用只**返回一个元组（Tuple）**，即“一次一元组”的处理模式。

### 2. 查询执行流程示例（以哈希连接 Hash Join 为例）

#### 执行顺序与状态机

- **阶段一：构建（Build Phase）**
  - 连接操作符（`Join`）被父节点调用 `next()`
  - 首先**迭代遍历左侧子节点**，反复调用其 `next()`，逐元组读取数据
  - 每读取一个元组，就将其**插入哈希表（Hash Table）**中
  - 直到左侧子节点所有元组读取完毕，哈希表构建完成

- **阶段二：探查（Probe Phase）**
  - 连接操作符开始**迭代遍历右侧子节点**，反复调用其 `next()`
  - 右侧数据流经过**过滤操作符（Filter）**：过滤操作符调用其子节点的 `next()`，对返回的元组进行**谓词评估（Predicate Evaluation）**，只将符合条件的元组向上传递
  - 过滤后的元组到达连接操作符，与已构建的哈希表进行**探查匹配**
  - 若匹配成功，则拼接并向上传递；若不匹配，则继续读取下一个元组

#### 操作符的层次调用关系

```
Projection (根操作符)
  └─ Join (Hash Join) —— 流水线中断点（Pipeline Breaker）
      ├─ 左子节点: Table Scan（构建阶段：读取所有数据构建哈希表）
      └─ 右子节点: Filter（探查阶段：逐元组读取并过滤）
          └─ 子节点: Table Scan（逐元组扫描）
```

### 3. 流水线（Pipeline）与流水线中断点（Pipeline Breaker）

#### 核心概念：**操作符的高度可组合性（Composability）**

- 只要所有操作符都实现了 `next()` 接口，它们就可以**自由组合成任意查询计划**
- 一个操作符**不关心其子节点具体如何实现**——它可以是另一个连接、索引扫描或表扫描

#### **流水线中断点（Pipeline Breaker）**

- **定义**：一个操作符在其父节点调用 `next()` 时，**无法立即返回一个元组**，而是必须先处理完其子节点的全部数据
- **哈希连接（Hash Join）是典型的流水线中断点**：
  - 因为必须先**完全构建左侧哈希表**才能开始探查阶段
  - 这意味着在构建阶段，它会在一个循环中反复调用左侧子节点的 `next()`，直到处理完所有元组

#### 流水线中断的影响

- 中断点之前的所有操作符可以形成一个**流水线**：数据在操作符之间逐元组传递，无需物化中间结果
- 中断点之后的流水线只能从探查阶段开始形成
- **性能影响**：中断点会打破 CPU 缓存的局部性，并可能导致需要物化中间结果

### 4. 关键算法与公式

#### 哈希连接构建阶段

```python
# 伪代码：构建阶段
while tuple = left_child.next():
    hash_table.insert(tuple.hash_key, tuple)
```

#### 哈希连接探查阶段

```python
# 伪代码：探查阶段
while tuple = right_child.next():
    if filter.evaluate(tuple):  # 过滤操作
        matches = hash_table.probe(tuple.hash_key)
        if matches:
            for match in matches:
                return join_tuples(tuple, match)  # 单次调用返回一个元组
```

## 第 6 部分

### 查询执行中的流水线与流水线中断

#### 核心概念：**流水线执行模型**

- **流水线的定义**：在数据库查询执行中，**流水线（Pipeline）** 是一种高效的处理方式，它允许一个元组（Tuple）从树的底层一路向上处理，**无需将中间结果写回磁盘**。
- **工作流程**：
    1.  **构建阶段**：首先，系统需要遍历左侧（通常是构建哈希表的一侧）的所有元组，以构建哈希表（例如，对于哈希连接）。这一步是**阻塞的**，必须看到全部数据。
    2.  **探测阶段**：一旦哈希表构建完成，当从树的另一侧调用 `next()` 时，就会形成一个流水线。处理流程是：从底部节点（例如，表扫描）获取一个元组。
        - 该元组通过**谓词评估**（Predicate Evaluation）。
        - 如果评估为真，则向上传递到下一个算子（例如，哈希探测）。
        - 哈希探测成功（找到匹配），则继续向上传递到投影（Projection）算子。
        - 最后输出结果。

- **流水线的优势**：
    - **避免磁盘I/O**：这是最大的优势。每个元组在被获取后，可以在内存中完成所有必要的计算，**一次性输出**，完全不需要将中间结果写回磁盘或临时文件。这对于磁盘密集型系统来说，性能提升巨大。
    - **工程实现简单**：流水线模型将复杂的查询分解为串行步骤。
    - **输出控制极其简单**：例如，如果查询带有 `LIMIT 10` 子句，则顶层的循环只需在计数达到10时停止调用 `next()`，查询立即结束。无需处理海量数据再截断。

#### 关键术语：**流水线中断符（Pipeline Breakers）**

- **定义**：有些运算符是**阻塞（Blocking）** 的，它们必须等待所有子节点发射完所有元组后才能开始工作。这些算子被称为**流水线中断符**。
- **常见的流水线中断符**：
    - **连接（Joins）**：特别是哈希连接，必须先构建完整个哈希表（处理所有左侧输入）。
    - **子查询（Sub-queries）**：某些子查询需要完全执行完毕。
    - **排序（Sorting）**：这是最典型的例子。**无法对数据进行全局排序，直到看到所有数据**。
- **影响**：流水线中断符会破坏流水线的连续性，迫使系统在这一点上物化（Materialize）中间结果。

#### 流水线的优缺点（讨论）

- **优点**：
    - 极致的内存效率。
    - 输出响应速度快（尤其是带 `LIMIT` 时）。
    - 降低了查询执行器的实现复杂度。
- **潜在缺点**：
    - **巨大的函数调用开销**：如果所有数据都在内存中，那么每个元组在流水线中经过每个算子时都会产生一次函数调用（`next()`）。这会导致大量的CPU开销，成为纯内存场景下的性能瓶颈。

#### 总结

流水线模型是现代数据库查询执行的核心。它的精髓在于**“一个元组完成所有工作”**，通过减少磁盘I/O来大幅提升性能。工程师需要识别并谨慎处理**流水线中断符**（如排序、连接），因为它们会打断高效的流水线流程。同时，也要注意在内存处理场景下，过度的函数调用可能成为新的瓶颈。

## 第 7 部分

### 单次单元组处理模型的局限性与架构权衡

#### **核心概念：火山模型（Iterator Model）的“拉取”范式**
- **核心机制**：每个查询计划节点通过 `next()` 接口向上层返回**单个元组**，形成“拉取”数据流。
- **关键问题**：这种设计导致**高度碎片化**的函数调用链，每个元组都可能触发大量函数调用。
  - **代价**：如果数据驻留内存，**函数调用开销**仍不可忽视；若涉及磁盘I/O，虽然“跳转”成本相对可控，但**磁盘访问本身始终是瓶颈**。
- **优化局限**：无法对**下层节点**的执行行为进行全局优化。
  - **实例**：`ORDER BY ... LIMIT` 操作必须**拉取所有子节点数据**，进行完整排序后，才能应用限制。
  - **本质**：这是**执行模型固有特性**，与优化器的查询计划生成无关。

#### **执行计划与优化器的职责分离**
- **假设前提**：当前讨论基于**给定优化后的执行计划**，不涉及优化器如何生成该计划。
- **翻译流程**：SQL字符串 → (优化器) → 最优计划树 → (执行器) → 各节点函数实现（如PostgreSQL的Project节点）。
- **优化器核心任务**（期中后覆盖）：
  - 确定**连接顺序**（内表 vs 外表）
  - 选择**连接算法**
  - 下推谓词（如 `WHERE` 条件至Join之前）
  - 对比关系代数中谓词位于顶层的默认位置
- **教学挑战**：数据库系统各部分**紧密耦合**，必须先掌握连接算法才能讨论优化器，形成依赖关系。

#### **该模型的实际瓶颈**
- **单次元组处理**：每次获取一个元组后，可能立即执行昂贵操作（如聚合、函数计算）。
- **错过批量优化机会**：如果能在同一时刻处理**多个元组**，就可以通过**向量化**或**批处理**来“碰撞”数据，大幅降低指令级开销和函数调用成本。
- **结论**：尽管函数调用代价是可接受的代价，但**单次单元组**的细粒度处理方式才是火山模型性能扩展的**真正天花板**。

## 第 8 部分

### 物化模型（Materialization Model）

#### **核心概念：自底向上，一次性产出全部结果**
- 与**迭代器模型**（自顶向下，`Next()` 逐条拉取）完全相反。
- **物化模型**不再依赖 `Next()` 函数，而是要求每个算子实现一个**一次性生成全部输出**的方法（如 `Output()` 或 `Materialize()`）。
- 执行流程是**自底向上（Bottom-Up）**：**子算子**完整执行，**将所有结果物化**（存为临时数据），然后**整体传递给父算子**。

#### **工作原理**
1.  **底层算子**（如扫描表）一次性读取所有数据，生成完整的结果集。
2.  将整个结果集**整体返回**（而非逐条返回）给上层算子。
3.  **上层算子**拿到全部数据后，完整执行自己的逻辑，再将其全部结果传递给更上层。
4.  重复直到根节点，最终得到完整查询结果。

#### **与迭代器模型的关键对比**

| 特性 | 迭代器模型（Iterator Model） | 物化模型（Materialization Model） |
| :--- | :--- | :--- |
| **执行方向** | 自顶向下（Top-Down） | 自底向上（Bottom-Up） |
| **数据传输** | 逐条/逐批（通过 `Next()`） | 一次性传输完整结果集 |
| **内存压力** | 较低（流水线化，边生成边消费） | **极高**（需要内存容纳完整中间结果） |
| **适用场景** | OLTP、内存不充裕、数据量大 | 数据量小、或需要批量优化 |

#### **关键挑战与讨论**
- **内存溢出问题**：如果某个算子的输出结果巨大（超过内存），物化模型会直接崩溃。此时必须采用 **混合散列连接** 或 **递归分区** 等**磁盘溢出策略**。
- **与查询计划的结合**：大多数数据库系统并不会在运行时动态判断是否需要溢出磁盘，而是**在查询规划阶段就确定好执行策略**，生成一个**固定的查询计划**。一旦确定采用物化模型，整个计划就“固化”下来。

#### **一句话总结**
> 物化模型是**直观但“笨重”** 的方式：**每个算子一次性算完所有结果，再整体传给上级**。它简化了编程逻辑（人类最容易理解），但内存消耗巨大，因此曾是20年前的主流实现，如今不再普遍。

## 第 9 部分

### 物化执行模型 (Materialization Execution Model)

#### 核心概念

**物化模型** 的核心思想是 **“一次性输出所有结果”**。每个算子会完全阻塞，直到其子算子将**全部数据**作为一个完整的缓冲区或列表返回。这与流水线模型（逐行或逐批推送）截然相反。

#### 执行流程（以 Join 为例 Top-Down）

1.  **顶向下调用**：根节点调用其子节点的 `output` 函数。
2.  **阻塞等待**：该调用是**阻塞调用**。当前算子会完全挂起，直到子节点返回**所有元组**。
3.  **构建哈希表**：接收到左子节点的全部数据后，算子可以一次性构建完整的哈希表。
4.  **处理右子树**：然后调用右子节点的 `output` 函数，同样等待其返回全部元组。
5.  **结果汇总**：将右子节点的结果与哈希表匹配后，将完整结果向上传递。

#### 关键技术：内联 (Inlining) 与谓词下推

由于数据一次性集成，直接传递整个表（如十亿行）是极其低效的。因此，**物化模型强制进行内联优化**：

*   将树**顶层的算子（如过滤谓词）嵌入到底层扫描算子中**。
*   例如：在扫描时，**当数据页从内存调入后，立即应用过滤谓词**，只将符合条件的元组放入输出缓冲区。
*   这避免了将大量无关数据一路推送到上层后再丢弃。

#### 数据传递单元

*   **完整元组**：传递整行包含所有列的数据。
*   **单列 + 书签**：也可以仅传递单列数据，但必须配合额外的**记账机制**，用于追踪当前传递的是原始表的哪一部分（如行ID）。

#### 核心优缺点

*   **优点**：
    *   简化了算子的实现（无需管理复杂的中间状态）。
    *   便于进行全局优化（内联）。
*   **缺点**：
    *   **内存爆炸**：所有中间结果必须完全物化到内存中，对于大表极不友好。
    *   **延迟高**：必须等到整个子树完成后才能开始后续处理。

## 第 10 部分

### 查询引擎中的迭代器模型与物化策略优化

#### 核心概念：流水线式物化 vs. 一次性物化
- **关键术语**：**内联谓词求值 (Inline Predicate Evaluation)**、**迭代器模型 (Iterator Model)**、**生成器模式 (Generator Pattern)**
- **核心思想**：在经典的迭代器模型（火山模型）中，查询执行是一行一行（或一批一批）拉取的。然而，早期某些系统（如VoltDB、RavenDB）采用了一种**“完全物化再过滤”**的策略：先将操作符所有输出结果生成并存储在内存中（类似Python的`yield`一次返回全部），然后再应用过滤谓词。
- **优缺点权衡**：
    - **优点**：简化了内存访问模式，所有数据都在缓冲区，方便后续批量操作。
    - **缺点**：当处理**大表**时，如果操作符的输出结果过大，则会导致**内存抖动 (Thrashing)**——生成所有输出时一次性占满内存，甚至触发OS交换（swap）。

#### 优化动机：为何特定系统选择完全物化？
- **场景**：**操作型工作负载 (Operational Workloads)**，如OLTP系统。
- **核心瓶颈**：对于索引扫描（Index Scan），每次只返回**一条记录**（如`WHERE id = Andy`），函数调用（`next()`）的开销远大于磁盘I/O。在**内存数据库**中（如VoltDB），无磁盘等待，因此**函数调用开销成为主要矛盾**。
- **解决方案**：跳过“按需拉取-过滤”的常规步骤（图中步骤4），直接**将数据拉入输出缓冲区再统一过滤**。这实质上是**用内存带宽换取函数调用开销**。

#### 具体系统案例分析

| 系统 | 类型 | 采用策略 | 核心原因 |
| :--- | :--- | :--- | :--- |
| **VoltDB / H-Store** | OLTP内存数据库 | 完全物化 + 内联谓词 | 索引扫描每次只返回一条记录，消除函数调用开销（内存中无磁盘延迟，函数调用成为瓶颈） |
| **RavenDB** | 文档数据库 | 完全物化 | 同样是操作型负载，每次获取少量文档，减少调度开销 |
| **MonetDB** | OLAP列式存储系统 (荷兰CWI) | 逆向优化（依赖MMAP + 尽量下推扫描） | **注意点**：MonetDB是OLAP系统，情况特殊。它依赖**内存映射文件 (MMAP)** 来管理数据。它并未完全采用“生成全部再过滤”，而是为了**减少物化开销**，尽量将谓词下推至扫描阶段，利用OS的换页机制（swap）处理大表溢出。其论文核心目标是“减少中间物化”，与VoltDB的动机相反。 |
| **DuckDB** | OLAP系统 | 源自MonetDB分支 (MonetDB Light)，但做了彻底重写 | 继承了MonetDB的列式思想和物化策略，但进行了现代化优化 |

#### 核心公式与算法暗示
- **开销模型对比**：
    - **传统迭代器**：总开销 ≈ **每行函数调用开销** × 行数 + 过滤开销
    - **完全物化方法**：总开销 ≈ **一次批量物化开销**（含内存拷贝）+ 过滤开销 + **可能的交换开销**
- **关键判断条件**：
    - 当 `每行函数调用开销 × 行数` > `批量物化开销 + P(交换) × 交换代价` 时，完全物化更优。
    - 大多数OLTP系统的行数（单次查询）极小，使得完全物化总是更优。

#### 总结性洞察
- **没有银弹**：查询执行策略的选择取决于**工作负载特性**（OLTP vs. OLAP）和**存储层级**（内存 vs. 磁盘）。
- **代价的转换**：在内存数据库中，CPU的**函数调用代价**取代了磁盘I/O成为首要优化目标；在磁盘数据库中，则需优先考虑减少随机I/O。
- **设计反思**：现代分析型数据库（如DuckDB）的演进，体现了对“**分批物化**”和“**向量化执行**”的回归，即在保持大块数据吞吐的同时，避免一次性完全物化的内存压力。

## 第 11 部分

# 向量化模型：迭代模型与物化模型之间的平衡点

## 核心概念：三种处理模型的对比

- **迭代模型（Iterator Model / Volcano Model）**：每次`next()`调用返回**一个元组（Tuple）**
  - 优势：内存占用极低，控制简单
  - 劣势：无法利用现代CPU的向量化指令（SIMD）

- **物化模型（Materialization Model）**：一次`next()`调用返回**所有元组**
  - 对于**OLAP查询产生大量中间结果**时，会导致**磁盘溢出（spill to disk）**
  - 虽然MapD（现OmniSci）等系统采用此方式并声称通过OS换页机制工作良好，但**对于OLAP场景并不推荐**

- **向量化模型（Vectorized Model）**：每次`next()`调用返回**一批（batch）元组**
  - 是迭代模型和物化模型之间的**折中方案**
  - 批处理大小取决于硬件和查询特征

## 向量化模型的关键设计

### 批处理尺寸（Batch Size）的确定

- **SIMD寄存器对齐**：并非要求整个batch与寄存器大小严格对齐
  - 实际要求的是：在处理数据块时，数据块的**对齐方式**应适应SIMD指令的寄存器宽度
  - 但**batch本身可以远大于单个SIMD寄存器**

### 核心优势：SIMD向量化操作

- 在操作符的控制循环中，当调用子节点的`next()`获取一个batch后：
  - 内部循环可以**同时处理多个元组的值**
  - 通过**向量化指令（SIMD）**并行执行操作，大幅提升吞吐量

### 实现架构

```
传统迭代模型：
Next() → [1个元组] → 处理 → Next() → [1个元组] → ...

向量化模型：
Next() → [一批元组] → for each in batch: SIMD处理 → Next() → [一批元组] → ...
```

## 实际应用建议

- **OLAP场景**：向量化模型是当前主流选择（如ClickHouse、DuckDB等）
  - 避免物化模型的磁盘溢出问题
  - 同时获得比迭代模型更高的计算效率
- **OLTP场景**：迭代模型仍适用，因为每次只处理单条记录
- **批处理大小调优**：需要根据具体硬件（CPU缓存行、SIMD寄存器宽度）和查询模式进行调整

## 第 12 部分

### 3.3 向量化处理模型 (Vectorized Processing Model)

#### 3.3.1 核心概念与动机
- **核心概念**：在**单次函数调用**（如 `next()`）中，**批量处理**多个元组（Batch of Tuples），而非每次都只处理一个。
- **技术实现**：利用现代CPU的 **SIMD（单指令多数据流）** 指令集，对寄存器大小的数据块进行并行操作。
- **关键术语**：**批处理大小 (Batch Size)**、**SIMD寄存器**、**向量化执行**。
- **设计哲学**：这是一个 **“中庸”** 模型，介于一次处理一个元组（迭代器模型）和一次处理所有元组（物化模型）之间。通过平衡函数调用开销与缓存局部性，达到最佳性能。

> **注意**：批处理的大小（一个Batch内包含的元组数量）可以大于单个SIMD寄存器的容量。系统会根据具体的硬件架构（寄存器宽度）和查询特性，动态调整最优的批处理大小。

#### 3.3.2 经典实现与历史渊源
- **起源系统**：该模型由 **VectorWise** 系统（原MonetDB/X100的分支）首先发明，并得到广泛推广。
- **业界影响**：
  - VectorWise后被Actian公司收购。
  - VectorWise的联合创始人后来成为 **Snowflake** 的联合创始人。因此，**Snowflake的核心执行引擎也基于向量化模型**。
  - 现代几乎所有专门用于分析型负载的系统（OLAP）都支持该模型。
- **老旧系统的适应性**：
  例如 **DB2 (1983)**、**Oracle (1978)** 等极老的关系型数据库，虽最初设计为基于行的迭代器模型，但它们都开发了**专门的、可插拔的列式存储引擎**，这些引擎内部采用了向量化模型来处理分析型查询。

#### 3.3.3 代码层面的直观理解（Python生成器类比）
- 将每个查询操作符（如Scan、Filter、Join）视为一个**Python生成器**。
- 操作符内部的 `next()` 函数负责处理一个**批次**的数据。
- 在批次内部，循环执行核心计算逻辑，这个循环可以被**编译器或运行时自动向量化**（利用SIMD）。

```python
# 伪代码示意
def vectorized_next(self, output_batch):
    # 内部循环处理一个batch内的所有元组
    for i in range(self.current_batch_size):
        # 对output_batch[i]执行并行计算操作
        output_batch.data[i] = self.process(self.input_batch.data[i])
    # 如果批次已满（或需要），则yield（输出）出去
    if output_batch.is_full():
        yield output_batch
```

### 3.4 三种处理模型的适用场景总结

| 模型 | 最佳适用场景 | 核心优势 | 主要缺点 |
| :--- | :--- | :--- | :--- |
| **迭代器模型 (Iterator / Volcano)** | **通用型、TP（事务处理）** 系统 | 实现简单，内存占用极低（流水线处理） | 函数调用开销巨大，不利于SIMD优化 |
| **物化模型 (Materialization)** | **非常特定的OLTP** 系统或需要频繁随机访问的场景 | 无函数调用开销，适合处理大量输入输出不对称的算子 | 内存带宽消耗极大，缓存不友好 |
| **向量化模型 (Vectorized)** | **分析型（OLAP）** 系统（内存或磁盘） | **函数调用开销低 + SIMD友好 + 缓存友好** 的最佳平衡 | 实现复杂度较高，批处理大小需精细调优 |

### 3.5 补充讨论：用户如何选择数据库？以及为何存在如此多种类？

#### 用户选择数据库的考量因素
1.  **负载类型 (Workload)**:
    - **OLTP**：关注高并发、低延迟的短查询（如：银行交易、电商下单）。通常选择基于**迭代器模型**的系统（如MySQL、PostgreSQL）。
    - **OLAP**：关注扫描大量数据进行聚合分析的长查询（如：年度财务报表、用户行为分析）。通常选择基于**向量化模型**的系统（如Snowflake、ClickHouse）。
    - **HTAP**：需要兼顾两者。需要混合引擎或特殊架构。

2.  **数据模型与结构化**:
    - 关系型 (SQL) vs. 文档型 (NoSQL) vs. 图型 (Graph) 等。不同的引擎针对不同的数据模型做了极致的优化。
3.  **部署方式与成本**:
    - 自建 (On-Premise) vs. 云原生 (Cloud-Native)。云数据库往往提供了更灵活的弹性伸缩和按需付费能力。
4.  **生态与社区**:
    - 工具链、驱动、迁移成本、人才储备等。例如Oracle的商业支持，PostgreSQL的丰富扩展。
5.  **特定功能需求**:
    - 全文搜索、地理空间支持、时序数据特殊优化、严格的ACID保证等。

#### 数据库种类繁多的根本原因
- **没有银弹 (No Silver Bullet)**：没有一种数据库能在**所有维度**（一致性、可用性、分区容错性、性能、成本、易用性）上同时做到最优。不同的架构设计（如三种执行模型）本身就是对特定场景的深度**权衡**。
- **硬件演进驱动**：从单核到多核，从HDD到SSD到NVM，从普通内存到HBM，每一次硬件革命都催生了为充分利用其特性而设计的新数据库。
- **数据量与价值的变化**：过去的数据以结构化为主，规模较小；今天的数据是海量、多模态的。处理如此巨大的数据，需要针对特定数据模型和查询模式进行专门优化。
- **开发与维护成本**：构建一个“通用且完美的数据库”的工程复杂度极高，而打造一个“在某个特定领域做到极致”的专用数据库，其研发周期和风险都更低。因此，我们看到大量的垂直领域数据库（如时序数据库TSDB、图数据库Neo4j）。

        *   **向量化优势**：使用SIMD，一次性从内存中加载一批数据到**专用寄存器 (specialized registers)**，用一条指令完成多个比较操作（如一次性判断8个值是否等于某个县）。
        *   **性能提升**：大幅减少了指令次数和内存调用，对于**长扫描、单列操作**的**分析查询 (Analytical Queries)** 极其高效（免去了索引依赖）。

## 第 14 部分

# 数据处理模型：向量化与硬件加速

## 核心概念：两种数据处理范式的对比

### 迭代模型（Iterator Model）
- **核心思想**：逐条处理数据元组，一次获取一个元组，执行条件判断，决定是否输出
- **操作流程**：
  1. 从数据源逐个获取元组
  2. 对每个元组运行 WHERE 子句 / IF 子句进行条件检查
  3. 满足条件则产生输出
- **局限性**：处理效率受限于单条处理的串行性

### 向量化模型（Vectorized Model）
- **核心思想**：**同时获取多个数据元组（如4个），利用SIMD指令集并行处理**
- **操作流程**：
  1. 一次性加载四个元组到SIMD寄存器
  2. 使用**单条指令**同时执行比较操作
  3. 一次性获得所有结果
- **关键优势**：
  - **单指令多数据（SIMD）**：一条指令同时处理多个数据元素
  - 减少指令执行总数，提高吞吐量
  - 充分利用CPU的向量处理能力

### 重要澄清
> "处理模型的高层概念与具体实现方式无关——无论是向量化、硬件加速器与否，核心思想保持不变"

也就是说，**向量化模型本质上是迭代模型的性能优化版本**，而非全新的处理范式。

---

## SIMD实现的技术细节

### 关键公式
对于一个包含N个元素的向量化操作：
$$ \text{速度提升} \approx \frac{\text{标量指令数}}{\text{SIMD指令数}} \times \frac{\text{数据宽度}}{\text{寄存器宽度}} $$

### 实际实现方式
- 高层概念上使用 `for` 循环描述逻辑
- **实际实现**需使用 **SIMD指令集**（如SSE、AVX）来实现向量化
- 编译器自动向量化 vs 手动编写底层SIMD代码

## GPU加速数据库：深入讨论

### 现有实现案例
- 存在基于 **CUDA** 的GPU加速数据库
- 例如：英国公司 **Bright Light** 使用CUDA实现数据库加速
- 其他数据库可以通过工具查询哪些数据库支持GPU加速

### 为什么GPU数据库可能是个"坏主意"？

核心痛点在于 **数据传输瓶颈**：

1. **数据位置问题**：
   - 数据通常存储在 **磁盘** 或 **系统内存** 中
   - 需要将数据从磁盘提取到系统内存
   - 再通过PCIe总线传输到GPU显存

2. **数据传输路径**：
   - 磁盘 → 系统内存 → GPU显存（向下传输）
   - GPU处理完 → 系统内存（结果回传）
   
3. **性能分析**：
   ```python
   数据传输延迟 = 磁盘读取时间 + 内存传输时间 + PCIe传输时间
   CPU方案：磁盘读取 + 内存处理
   GPU方案：磁盘读取 + 内存传输 + PCIe下行 + GPU处理 + PCIe上行
   ```

### 关键见解
> **一个优化良好的基于CPU的生成/执行引擎，往往能超越GPU方案**

原因：
- **无需支付数据传输成本**
- CPU架构在处理内存中数据时更加高效
- **新型硬件技术**（如CXL互连）正在改善传输瓶颈，但尚未根本改变现状

## 实践要点总结

### 选择指南
| 场景 | 推荐方案 | 原因 |
|------|---------|------|
| 数据已在CPU内存中 | 向量化CPU处理 | 零传输成本，SIMD加速 |
| 数据在磁盘 | CPU处理（经过优化） | 传输成本往往抵消GPU优势 |
| 数据已在GPU内存 | GPU处理 | 无传输开销 |
| 计算密集型过滤 | GPU可考虑 | 但需评估传输成本 |

### 行业现状观察
- GPU数据库确实存在并得到应用
- 是否有价值取决于具体使用场景和数据位置
- 学术界和工业界对此仍有争议

#### 核心背景：GPU/FPGA加速数据库的“炒作”与现实

-   **核心概念**：历史上，许多初创公司试图用 **GPU**（图形处理器）或 **FPGA**（现场可编程门阵列）来加速数据库查询，但这些尝试大多以失败告终。

-   **Swarm 64（FPGA加速器）**：

-   **BlazingDB（后更名为BlazingSQL）**：

    -   **意义**：这与其说是加速数据库，不如说是**为AI/ML工作流提供数据预处理管道**，这个逻辑在当时是通的。
    -   **结局**：公司关闭，核心团队（与Apache Arrow、Python Pandas、Nvidia RAPIDS的创始人）合并，组成了新公司 **Voltron Data**。
    -   **新兴技术**：**Apache Arrow** 项目是其中的核心组件，它是一种跨平台的内存列式数据格式。

### 数据流的计算方向

#### 两种主流架构

-   **核心概念**：数据处理可以沿着**两个方向**在系统中流动：
    1.  **自顶向下：** 大部分系统采用此方式，查询在高层分解并向下推动到存储引擎。
    2.  **自底向上：** 另一种设计范式。

-   **重要性**：对于理解课程内容，具体方向不是核心关注点，但需要知晓存在这两种不同的设计思路。

-   **关键比喻：管道（Pipeline）**：
    -   可以将这想象成一个**流水线**。
    -   如果采用**自底向上**的设计，你必须**非常仔细地按内存对齐方式来组织数据**，以确保数据在向上传递时能高效处理。
    -   **核心要点**：内存对齐和布局（Memory Alignment）是高效数据处理的基础，无论采用哪种方向，都需要精心设计。

## 第 16 部分

## 查询执行：访问方法与扫描

### 一、查询计划的两种构建方式

**核心概念：自顶向下（Top-Down） vs. 自底向上（Bottom-Up）**

- **自底向上（Bottom-Up）构建**
    - 像**流水线（Pipeline）**方法，从底层节点开始构建
    - **优势**：可以精细控制**内存对齐（Memory Alignment）**，将数据放入特定的CPU寄存器
    - **性能**：比简单在内存中处理能获得**更好的性能**
    - **代表系统**：德国的 **Hyper**（前身Tablo bot）采用此方式
- **自顶向下（Top-Down）构建**（主流）
    - 大多数数据库系统采用的方式
    - 从查询计划树的根节点开始向下构建

### 二、访问方法（Access Methods）详解

**核心概念：访问方法是查询计划中**叶子节点**的操作方式，关系代数中并未定义，属于**实现层**的内容**

- **本质**：无论底层数据来自哪里，由于**操作符是可组合的（Composable）**，上层系统无需关心数据来源
- **数据来源类型**：
    - **真实表**：通过`CREATE TABLE`创建的物理表
    - **中间查询结果**：子查询或嵌套查询产生的临时数据
    - **物化视图（Materialized View）**：预计算查询并存储在临时空间的结果

#### 三种基本选择

| 访问方法 | 描述 | 示例系统命名 |
|---|---|---|
| **表扫描（Table Scan）** | 遍历所有页面 | Oracle: Heap Scan, Partition Range Scan |
| **索引扫描（Index Scan）** | 通过索引定位数据 | Oracle: Index Scan, Row ID Scan |
| **多索引扫描（Multiple Index Scan）** | 同时使用多个索引 | — |

### 三、顺序扫描（Sequential Scan / Selection Scan）

**核心概念：最基础的访问方法，直接遍历表的**每一页

- **工作流程**
    1. **迭代所有页**：遍历表中的每个页面
    2. **通过缓冲池访问**：不直接操作磁盘，而是请求缓冲池（Buffer Pool）
    3. **缓冲池职责**：从磁盘读取页面到内存帧（Frame），返回指向该内存位置的指针
    4. **处理数据**：在扫描操作符内部的循环中执行处理逻辑
        - 评估谓词（Predicate Evaluation）
        - 计算聚合（Aggregation Computation）
        - 其他自定义操作

- **代码逻辑示意**
    ```plaintext
    for each page in table:
        page_ptr = buffer_pool.get_page(page_id)  // 通过缓冲池获取页面指针
        for each tuple in page:
            if evaluate_predicate(tuple):          // 评估过滤条件
                produce_result(tuple)              // 向父操作符输出结果
    ```

- **关键特点**
    - 不关心数据在磁盘上的物理组织形式
    - 适用于**全表扫描**或**选择性较低**的查询场景
    - 通过**缓冲池（Buffer Pool）** 机制实现磁盘与内存的解耦

## 第 17 部分

## 扫描操作符的内部机制与优化

### 核心概念：游标（Cursor）与迭代器模型

- **迭代器模型（Iterator Model）**：这是一个请求驱动的数据流模式。父操作符通过调用 `Next()` 来向下层（子）操作符索取一个元组。这是一种**拉取（Pull-based）** 模式。
- **游标（Cursor）**：是扫描操作符内部的**本地状态**。它记录了当前扫描的进度位置，以便在多次 `Next()` 调用之间**恢复执行上下文**。

> **游标的核心职责：** 维护“我上次看到哪里了？”
> - `lastPageID`：最后访问的页面编号
> - `lastSlotID`：该页面上的最后一个槽位编号（如果页面内包含多个元组）

- **执行流程**：
  - 父操作符调用 `Next()` 请求数据。
  - 扫描操作符内的 `for` 循环继续处理。
  - 内部执行：评估谓词（Predicate）、计算聚合（Aggregation）等操作。
  - 生成一个结果元组后，使用 `yield`（类似Python生成器）将其**向上传递**给父操作符。
  - 下一次 `Next()` 调用时，直接从游标记录的断点位置继续执行。

### 顺序扫描（Sequential Scan）的本质与局限

顺序扫描操作符是**数据库系统中最基础、也是最差的保底执行方案**。

- **适用场景**：
  - 没有可用的索引。
  - 没有预计算数据（如物化视图）。
  - 必须直接读取表的原始页面来生成结果。

- **性能瓶颈（为什么它是“最坏的选择”）**：
  - **磁盘I/O缓慢**：从磁盘随机读取大量页面是最大的开销。
  - **污染缓冲池**：扫描大量页面会“淹没”（Flood）缓冲池，破坏缓存命中率（Buffer Pool Trashing），影响后续查询。

### 扫描加速技术（已覆盖内容）

大部分优化技术已在课程前期内容（如缓冲池管理）中讨论过：

- **预取（Prefetching）**：在扫描当前页面的同时，**提前预取**后续页面。确保当 `Next()` 请求下一页时，该页已在内存中，避免等待磁盘。
- **缓冲池旁路（Buffer Pool Bypass）**：为当前查询分配一个**私有临时内存空间**。加载页面到这里处理，完成后立即丢弃，**不经过缓冲池**，从而避免破坏LRU（最近最少使用）替换策略。
- **并行查询（Parallel Queries）**：利用多个线程/进程同时扫描表的不同部分（将在下一节课程中详细讲解）。
- **堆聚簇（Heap Clustering） / 索引聚簇（Index Clustering）**：如果数据在物理存储上已经**按照某个索引键预排序**，扫描时会更加有序和高效。
- **延迟物化（Late Materialization）**：在扫描过程中，推迟将原始字节转换为完整元组（以及列数据）的操作，只在必要时才进行物化，减少数据复制开销。

## 第 18 部分

### 数据跳过与近似查询

#### 核心思想：**跳过不必要的数据**，减少 I/O，加速查询

- **目标**：避免读取全部数据，直接生成查询结果。
- **分类**：分为 **有损（Lossy）** 和 **无损（Lossless）** 两种策略。

---

### 有损近似查询（Lossy Approximate Query）

#### 核心概念：**以轻微误差换取显著性能提升**

- **基本逻辑**：不扫描所有数据，而是基于数据**样本**或**预计算的概要结构**来近似计算结果。
- **类比**：类似 **MP3/MP4 压缩**，人耳/人眼无法察觉的微小损失可以接受。

#### 应用场景：**聚合查询（Aggregation）**

- 适用于 **count, min, max, sum, avg** 等操作。
- 典型例子：网站日访问量约 100 万，抽样后估算相差几千，结果可接受。

#### 实现方式

- **内部数据结构**：系统维护专门的数据结构来快速给出近似值。
    - **HyperLogLog**：用于近似计数（distinct count）。
    - **Sketch** / **Bloom Filter** 的变体（如 **Balloon Filter**）。
- **查询语法**：部分系统提供特殊关键字，例如：
    - `APPROXIMATE COUNT`（Snowflake, BigQuery, Databricks, Oracle）

#### 专用系统：**专为近似查询设计**

- **BlinkDB**（UC Berkeley 学术系统）
- **ComputeDB**（原 SnappyData，后被收购）
- **特点**：用户可指定统计误差范围（如“误差在 5% 内”），系统自动选择采样策略以保证精度。

---

### 关键公式与算法（无特定公式，但涉及重要概念）

- **近似计数**：使用 **HyperLogLog** 算法，空间复杂度 O(log log N)，精度可控。
- **近似分位数**：使用 **T-Digest** 或 **GK Sketch**。
- **采样误差**：通常基于**中心极限定理**和**置信区间**计算，公式：
  \[
  \text{误差范围} \approx z \cdot \sqrt{\frac{p(1-p)}{n}}
  \]
  其中 \(z\) 为置信度系数，\(p\) 为样本比例，\(n\) 为样本大小。

---

### 对比与总结

| 特性 | 有损近似 | 无损精确 |
|------|----------|----------|
| **准确性** | 可能偏差，但统计可控 | 完全正确 |
| **性能** | 极快，跳过大量数据 | 慢，必须扫描全量 |
| **适用场景** | 实时大屏、趋势分析、成本敏感 | 财务对账、需精确审计 |
| **例子** | HyperLogLog, BlinkDB | 传统 B-Tree 索引扫描 |

#### 工程提示
- **适用阈值**：当数据量极大（如 > 1 亿行）且误差容忍度 > 1% 时，近似查询**显著优于**全量扫描。
- **组合使用**：现代引擎常将**无损数据跳过**（如 Zone Map、Min-Max 索引）与**有损近似**结合，在精确查询时跳过无关分区，在雷达图/仪表盘场景下启用近似。

## 第 19 部分

# 近似查询与Zone Maps：数据跳过技术

## 核心概念：**近似查询（Approximate Queries）**

- **基本思想**：允许数据库在**一定统计误差范围内**返回查询的近似答案
- **关键优势**：可以**跳过大量数据**，无需扫描全部记录
- 用户明确接受误差范围，系统自动计算近似结果

---

## Zone Maps：最主流的无损跳过技术

### 核心定义
**Zone Maps** 是一种**预计算元数据**技术，用于加速查询执行：

- **工作原理**：对数据块（一个页面、一个列、一组页面）**预计算聚合统计信息**
- **查询优化**：`WHERE` 子句首先检查 Zone Map，判断是否可能存在匹配数据
- **跳过机制**：如果 Zone Map 显示**不可能有匹配结果**，直接跳过整个数据块

### 历史背景与术语
- 最早由 **Netezza**（2000年代基于FPGA的硬件加速数据库）发明
- IBM收购Netezza后关闭（因为当时FPGA不被市场接受）
- 当前几乎所有数据库都在使用类似机制，只是叫法不同
- 在 **Parquet**、**ORC** 等列式存储格式中称为 **元数据维护（Metadata Maintenance）**

---

## Zone Maps 的工作原理

### 1. **预计算聚合（Pre-computed Aggregations）**
对于每个数据块，计算以下**标准聚合**：
- **最小值（Min）**
- **最大值（Max）**
- **平均值（Average）**
- **总和（Sum）**
- **计数（Count）**

### 2. **实际执行流程**

以示例数据为例：
```
原始数据块（5个条目）：[100, 200, 300, 400, 500]
Zone Map预计算：min=100, max=500, avg=300, sum=1500, count=5
```

**查询示例**：
```sql
SELECT * FROM table WHERE value > 600
```

**执行步骤**：
1. 访问 **Zone Map** 元数据，而非原始数据
2. 检查条件：`value > 600`
3. 对比 Zone Map：`max = 500`
4. **逻辑判断**：`max(500) < 600` → **不可能存在满足条件的数据**
5. **直接跳过**整个数据块，无需扫描

### 3. **性能优势**
- Zone Map **体积极小**，远小于原始数据块
- 避免 **全表顺序扫描（Sequential Scan）**
- 特别适合**云数据库**场景：元数据获取成本极低

---

## 关键公式与算法

### 数据跳过决策逻辑

对于查询条件 `value OP threshold`，跳过条件为：

**当 OP 为 `>` 时**：
$$
\text{如果 } \max(\text{block}) \leq \text{threshold} \rightarrow \text{跳过}
$$

**当 OP 为 `<` 时**：
$$
\text{如果 } \min(\text{block}) \geq \text{threshold} \rightarrow \text{跳过}
$$

**当 OP 为 `=` 时**：
$$
\text{如果 } \max(\text{block}) < \text{threshold} \text{ 或 } \min(\text{block}) > \text{threshold} \rightarrow \text{跳过}
$$

### 数据块粒度
- 在 **Parquet** 等列式存储中：**10MB ~ 100MB** 为一个数据块
- Zone Map 为此数据块维护一组聚合信息

---

## 实际应用场景

### 1. **列式数据库（Column Store）**
- 每个列独立存储，Zone Map 对每列分别维护
- 查询时检查相关列的 Zone Map

### 2. **云数据库系统**
- 数据与元数据分离存储
- Zone Map 可快速获取（网络开销极小）
- 大幅减少数据传输量

### 3. **大数据文件格式**
- **Parquet**：自动维护行组（Row Group）级别的统计信息
- **ORC**：维护条纹（Stripe）级别的统计信息
- 查询引擎自动利用这些元数据进行数据跳过

## 第 20 部分

# 数据库查询优化：区域映射与索引扫描

## 区域映射（Zone Map）的核心概念

**区域映射**是一种**预计算的数据块摘要信息**，用于快速判断数据块是否包含满足查询条件的记录。

- **工作原理**：为每个数据块或数据分区预先计算并存储该区域内数据的最小值和最大值
- **核心优势**：通过检查元数据（区域映射）而非扫描实际数据，实现**早期数据过滤**
- **应用场景**：广泛应用在云数据库系统中（如Snowflake）

## 区域映射的优缺点权衡

### 块大小与选择性（Selectivity）

**关键权衡**：数据块大小直接影响区域映射的过滤效率

- **大块数据问题**：如果区域映射覆盖的数据量过大（如10亿条记录），查询条件（如`value > 600`）可能**覆盖整个块的范围**，导致区域映射失效，仍需**全表扫描**
- **小块数据优势**：将数据分成更小的块，每个区域映射可以更精确地过滤，**提高选择性**

### Snowflake的优化策略

Snowflake采用**预排序（Pre-sorting）** 机制：

- 在数据写入前对列数据进行排序
- 排序后的数据使得区域映射的**最小/最大值范围更紧密**
- 大幅提升区域映射的**过滤效率**

## 索引扫描（Index Scan）

### 查询优化器（Query Optimizer）的角色

**查询优化器/计划器**（也被称为编译器）是数据库系统的核心组件：

1. **输入**：SQL查询语句
2. **处理过程**：将SQL转换为**物理执行计划**
3. **决策依据**：在生成物理计划时，自动决定使用哪个索引

### 索引选择的决策因素

| 决策条件 | 说明 |
|---------|------|
| **属性匹配** | 索引必须包含查询的WHERE子句中涉及的列 |
| 其他复杂因素 | 将在讲座14中详细讨论 |

**示例**：
- 查询条件：`WHERE colA = ? AND colB = ?`
- 索引定义：基于`colC`的索引 → **该索引对查询无效**
- 正确选择：需要基于`colA`或`colB`的索引

### 索引扫描类型

1. **单索引扫描（Single Index Scan）**：使用一个索引完成查询
2. **多索引扫描（Multi-Index Scan）**：结合多个索引，取交集或并集结果

## 重要公式与算法

### 区域映射过滤条件

对于查询条件 `value > X`：

$$ \text{需要扫描该块} \iff X < \text{zone\_map.max} $$

即：如果查询边界不超出区域映射的最大值，则**可能需要扫描整个块**

### 选择性度量

$$ \text{选择性} = \frac{\text{满足条件的记录数}}{\text{总记录数}} $$

- **高选择性**（< 10%）：索引扫描效率高
- **低选择性**（> 50%）：全表扫描可能更优

## 实践要点

1. **区域映射设计原则**：根据数据分布特征选择合适的数据块大小
2. **预排序策略**：对排序键建立区域映射，可显著提升过滤效率
3. **查询优化器的智能化**：自动选择最优索引，开发者无需手动指定
4. **性能监控**：定期评估区域映射的过滤比，调整分区策略

## 第 21 部分

### 查询优化中的索引选择：选择性分析

#### 核心概念：**索引可用性**与**选择性**

- **索引过滤条件匹配**：查询的 `WHERE` 子句必须包含索引的列才能生效。例如，索引建立在 `C` 列上，但查询条件是 `A` 和 `B`，则该索引**毫无用处**。
- **操作类型兼容性**：索引数据结构决定其支持的查询操作。
  - **哈希索引**：只支持 `=` 等值查询，**无法处理范围查询**（如 `>`, `<`）。
  - **B-Tree 索引**：支持等值和范围查询。
  - 数据库**查询优化器**内部集成了关于索引结构和操作能力的元信息，自动决策。

#### 选择最优索引：**选择性最大化**

- **选择性定义**：索引能过滤掉的数据比例。**选择性越高，越能快速丢弃无用数据，减少查询成本**。
- **实际场景推演**：假设一张学生表(`students`)，包含 `100` 条元组，有两个索引：`age` 和 `department`。
  - **查询语句**：`SELECT * FROM students WHERE age < 30 AND department = 'CS' AND country = 'USA'`
  - **场景一**：
    - `age < 30` 匹配 **99** 人
    - `department = 'CS'` 匹配 **2** 人
    - **选择**：使用 `department` 索引，因为它**选择性更高**（2% vs 99%），能更快丢弃数据。
  - **场景二**（反转数据分布）：
    - `age < 30` 匹配 **2** 人
    - `department = 'CS'` 匹配 **99** 人
    - **选择**：使用 `age` 索引，因为它的**选择性更高**。
- **优化器决策流程**：面对多个备选索引，优化器会：
  1.  检查 `WHERE` 子句的列组合。
  2.  评估每个索引对查询的支持能力（操作类型兼容性）。
  3.  基于统计信息（如列的直方图、基数）计算**每个索引的选择性**。
  4.  **选择选择性最高（丢弃数据最快）的索引**。

#### 索引扫描执行机制

- **工作流程**：与顺序扫描类似，但数据源不同。
  - **迭代器模式**：数据库构建一个**索引遍历迭代器**。
  - **调用 `Next()`**：每次调用从索引中返回下一条匹配的元组（记录ID或实际数据）。
  - **数据上抛**：将结果传递给父运算符（如投影、聚合）。
- **两点说明**：
  - **顺序扫描**：直接遍历整个表。
  - **索引扫描**：通过索引结构快速定位到符合条件的数据起始位置，**只遍历可能匹配的数据页或记录**，避免全表扫描。

#### 当多个索引同时可行时

- 如果场景变得复杂，两个索引的**潜在选择性**都很好（例如 `age < 30` 和 `department = 'CS'` 各自都只能过滤掉一半数据），优化器需要更复杂的策略。
- **可能方案**：
  - **使用单个最优索引**：根据统计信息评估哪个更能减少实际读取的页面数量（IO成本）。
  - **位图扫描**：分别用两个索引获取位图，然后进行**位与（AND）** 操作，精确找出同时满足两个条件的行。这适用于**多个可索引列且每个列的选择性都不极端**的情况。

## 多索引扫描 (Multi-Index Scan)

### 核心概念

- **多索引扫描**是一种查询优化策略，当**单个索引**的选择性不够好，但**多个索引的组合**可以更高效地缩小结果集时使用。
- 核心思想：分别利用多个索引进行扫描，得到各自匹配的**记录ID集合**，然后通过**集合运算**（求交/并）合并这些ID，最后再去检索实际数据行。

### 不同数据库的实现名称

- **PostgreSQL**：称为 **Bitmap Scan**，因为索引扫描的输出是一个位图，然后进行位图的交集/并集操作。
- **MySQL**：称为 **Index Merge** 操作。
- **Oracle / DB2 / SQL Server**：也支持类似的多索引扫描功能。

### 工作流程

1. **确定要使用的索引**：分析WHERE子句，找出所有可能用到的索引。
2. **分别扫描每个索引**：对每个索引执行其对应的索引扫描部分，得到**匹配的行ID集合**。
3. **合并结果集**：
   - 如果WHERE子句是**AND（合取）**，则对记录ID集合进行**交集（Intersection）**。
   - 如果WHERE子句是**OR（析取）**，则对记录ID集合进行**并集（Union）**。
4. **检索原始元组**：根据合并后的记录ID去实际数据表中取回完整行。
5. **额外过滤**：如果WHERE子句中还有**无法被索引覆盖的条件**，则在取回行后进行额外的谓词评估（Predicate Evaluation），输出最终结果。

### 具体示例

假设查询条件为：
```sql
SELECT * FROM students 
WHERE age < 30 
  AND department = 'CS' 
  AND country = 'US';
```

**执行步骤：**

1. **扫描年龄索引**：从 `age` 索引中获取所有 `age < 30` 的记录ID集合。
2. **扫描系别索引**：从 `department` 索引中获取所有 `department = 'CS'` 的记录ID集合。
3. **取交集**：对上述两个ID集合进行**求交**，得到同时满足年龄和系别条件的记录ID。
4. **检索数据并额外过滤**：根据交集后的记录ID去取回完整行数据，再对 `country = 'US'` 进行过滤（因为国家条件可能不在上述索引中）。

### 类比理解

想象你有两本书的目录，一本按**年龄**分类，一本按**系别**分类。你想找“**年龄小于30且是CS系**”的人：
- 在年龄目录里标出所有小于30的人（第一组页码）。
- 在系别目录里标出所有CS系的人（第二组页码）。
- **翻回目录页，找出两组的共同页码**（求交集）。
- 最后拿着这些页码去正文页看具体内容，并手动筛选出“来自美国”的记录。

### 关键要点

- **索引条件部分匹配**：不是所有查询条件都能被索引覆盖，多索引扫描允许每个索引只负责一部分条件。
- **集合运算效率**：合并操作（交集/并集）通常通过位图运算在高性能系统中实现，非常快。
- **适用场景**：当**单一索引的选择性很差**（会导致大量磁盘访问）且**多索引互补**时，此方法能显著提升性能。

## 第 23 部分

### 多索引扫描 (Multi-Index Scan) 与位图扫描 (Bitmap Scan)

#### 核心概念与工作流程

*   **核心思想**：当查询的 `WHERE` 子句中包含多个条件（例如 `department = X AND country = Y`），并且这些条件各自有独立的索引时，数据库不会只使用一个索引，而是**分别利用多个索引进行探查，再将结果进行集合运算（交集或并集）**。
*   **流程分解**：
    1.  **获取记录ID集合**：对每个条件（如 `department = X`），使用对应的索引进行扫描，得到一组**记录ID（Record IDs）**。
    2.  **集合运算**：由于条件是“与”（AND），需要对所有获取到的记录ID集合求**交集**。
        *   **关键实现细节**：在 PostgreSQL 中，这个记录ID集合不是简单的列表，而是一个**位图（Bitmap）**。位图中的每一位对应表中可能存在的每一个元组（Tuple）。
    3.  **获取元组**：对求交集后得到的记录ID集合进行物理数据获取（Fetch），将实际的元组从数据页中读出。
    4.  **过滤**：在对元组进行进一步处理（如传给上层操作符）之前，可以再次检查一些无法直接用索引加速的**附加谓词**（如 `country = Z`，如果 `country` 没有索引）。
    5.  **向上传递**：最终通过所有检查的元组被传递给父操作符。

#### 为什么需要这种机制？

*   **源于查询计划的可组合性**：现代数据库将查询计划分解为独立的操作符（Operators），**每个操作符只关心从子操作符接收元组，并向父操作符输出元组**。
*   **对上层透明**：上层的操作符（如Hash Join、Sort）**完全不需要关心**下层是如何产生元组的。它可以是简单的全表扫描，也可以是上述复杂的多索引扫描+交集逻辑。
*   **扩展了“一次一循环”模型**：看似简单的“一次一循环”（在一个循环里遍历表）可以被轻松扩展为“**多个循环+集合运算**”，只要最终能产生一个元组流即可。

#### PostgreSQL 中的位图扫描 (Bitmap Scan)

*   **重要澄清**：虽然 PostgreSQL 的执行计划中会出现“位图扫描”（Bitmap Scan），但这**不代表 PostgreSQL 支持位图索引**。
*   **实际含义**：PostgreSQL 的“位图扫描”本质上就是执行**上述多索引扫描（Multi-Index Scan）** 策略的实现名称。它使用位图作为中间数据结构来高效地完成多个索引结果的集合运算。

### 修改查询 (Modification Queries)

#### 核心概念

*   **包含的语句**：`INSERT`, `UPDATE`, `DELETE`, `TRUNCATE`。
*   **`TRUNCATE` 的特殊性**：`TRUNCATE` 逻辑上等同于删除表中所有元组，但**实现上通常不是逐行扫描删除**，而是更高效地**直接删除整个表再重新创建**，因为它避免了逐行扫描的开销。

#### 执行模型

*   **核心思想**：处理修改查询的操作符**依赖其子操作符来确定“修改什么”**。
*   **工作流程（以 `UPDATE ... WHERE ...` 为例）**：
    1.  子操作符（`Index Scan` 或 `Sequential Scan`）负责执行 `WHERE` 子句，进行谓词过滤。
    2.  子操作符定位到需要修改的元组。
    3.  子操作符将**记录ID**（或元组的某个唯一标识）**向上传递给**父操作符（`Update` 操作符）。
    4.  `Update` 操作符收到记录ID后，执行实际的修改操作（如更新数据页中的对应行）。

## 第 24 部分

## 更新与插入操作的执行模型

### **核心概念：DML操作（更新/删除/插入）的执行流程**

#### **更新（Update）与删除（Delete）操作**
- **更新操作符**会等待子操作符传递**记录ID（Record ID）**或部分数据
  - 子操作符（索引扫描或顺序扫描）执行自己的谓词过滤后，向上传递需要修改的记录ID
  - 更新操作符接收到ID后，执行实际的**数据修改**
  - 输出结果可以是：
    - 记录ID（返回"我删除了/插入了/更新了哪个记录"）
    - 完整元组（当使用 **RETURNING子句**时，返回修改后的完整元组对象）

#### **插入（Insert）操作的两种实现方式**
- **方式一：在插入操作符内物化完整元组**
  - 插入操作符负责组装元组的所有属性
  - 然后将组装好的元组插入目标表
- **方式二：直接使用子操作符传递的值（推荐）**
  - 插入操作符接收子查询传入的元组值
  - 直接插入到目标表
  - **优势**：可以直接实现 `SELECT INTO` 功能

#### **Upsert（更新或插入）操作**
- 逻辑与更新操作相同：如果记录存在则更新，不存在则插入

### **关键元数据管理**
- 操作符内部需要维护额外元数据：
  - **已访问过的元组**：记录已处理过的元组，避免重复处理
  - **已修改过的元组**：追踪已修改的元组，防止一个操作符内多次修改同一元组

### **重要问题场景：未追踪已修改元组导致的错误**

#### **示例：薪资更新查询**
```sql
-- 给所有薪资低于1100的员工加100元
UPDATE people 
SET salary = salary + 100 
WHERE salary < 1100
```

**潜在问题**：
- 如果不追踪已修改的元组，更新操作符可能会陷入**无限循环**或**重复更新**
- 例如：原本薪资为1050的员工，更新后变为1150
- 如果操作符未标记该元组已被修改，重新扫描时该元组可能再次符合条件（取决于实现细节），导致重复加薪

#### **解决方案**
- 所有DML操作符必须**维护已修改元组的记录**
- 确保每个元组在一个查询计划执行过程中**只被修改一次**

### **算法与实现要点**

**标准执行流程**：
1. 子操作符执行谓词过滤 → 2. 传递记录ID到DML操作符 → 3. DML操作符执行修改 → 4. 返回结果

**重要公式**：
- 更新后值 = 原始值 ± 增量（如：`salary = salary + 100`）
- 元组修改判定条件：`WHERE predicate` 确保只处理符合条件的元组

## 万圣节问题（Halloween Problem）——更新操作引发的扫描异常

### 核心问题：更新操作导致元组“复活”，被重复处理

- **场景描述**：数据库中有员工表（employees）和工资字段（salary）。执行一个更新查询：**给所有工资低于1100美元的人加薪100美元**。
- **执行流程**：
    1. **索引扫描**：更新操作符调用子节点，子节点执行索引扫描，查找所有 salary < 1100 的记录。
    2. **遍历叶子节点**：沿索引的叶子节点逐条扫描匹配的元组。
    3. **匹配第一条**：例如找到 Andy，工资为 999 美元，满足条件，元组被向上传递。
    4. **更新操作**：
        - **先删除**：从索引中删除该元组（因为索引基于 salary 字段，而 salary 即将改变）。
        - **更新值**：将 salary 从 999 修改为 1099。
        - **再插入**：将更新后的元组重新插入索引。
    5. **继续扫描**：索引扫描继续沿叶子节点前进。
    6. **问题出现**：由于更新后的元组（Andy, 1099）的物理位置可能在当前游标之后，**扫描会再次遇到这个刚刚更新过的元组**。
    7. **重复处理**：该元组仍然满足 salary < 1100 的条件（1099 < 1100），于是被再次匹配，导致 Andy 第二次被加薪。

### 根本原因：物理位置改变导致游标“回看”

- **关键矛盾**：
    - **扫描器**（Scan Operator）按物理顺序（叶子节点顺序）前进。
    - **更新操作**修改了索引键值，导致元组的**物理位置发生移动**（可能移动到当前扫描位置之后）。
    - 结果：**已处理的元组被插入到游标前方，导致扫描器再次遇到它**。
- **类比**：就像在一条路上依次检查每栋房子，当你修改了一栋房子的门牌号后，它被移到了你还没检查到的路段，于是你之后又会检查到它。

### 术语提炼

- **万圣节问题（Halloween Problem）**：数据库系统中，更新操作改变了元组的物理位置（索引或表本身），导致扫描操作重复处理该元组的异常现象。
- **别名**：也被称为“幻影更新”（Phantom Update）或“自我更新异常”。
- **Wikipedia**：该问题有专门的维基百科条目，是数据库理论中的经典问题。

### 后果与启示

- **语义错误**：从查询语义上讲，最终结果是**同一个元组被更新了多次**，违反了“每人只加薪一次”的预期。
- **本质**：这是一个**迭代器模型**与**原地更新**之间的矛盾。在数据库实现中，必须小心处理**更新操作与扫描操作之间的交互**，否则会导致逻辑错误。
- **解决方向**：常见的应对策略包括使用**快照隔离**、**查询重写**（如先收集所有匹配的元组ID再统一更新）、或采用**游标稳定性**机制。

## 第 26 部分

## 扫描与更新操作中的“万圣节问题”（Halloween Problem）

### 核心问题：扫描过程中更新同一张表导致的重处理

- **问题描述**：当执行一个 **UPDATE** 语句时，如果**扫描操作**（如顺序扫描）与**更新操作**作用于**同一张表**，可能会出现**更新后的元组再次被扫描到**，导致**重复处理**。
- **举例**：假设你在扫描一张员工表，并发出一条 `UPDATE employees SET salary = salary * 2` 的查询。扫描器从第一条记录开始，更新后该记录在物理上可能**向前移动**（比如索引结构变化或记录位移），导致扫描器再次“看到”这个已更新的元组，从而**再次给它加薪**，造成逻辑错误。

### 问题起源：IBM System R 的万圣节发现

- **背景**：IBM 研究人员在构建**第一个关系数据库 System R** 时，于某个**周五下班前**发现了这个棘手问题。
- **命名由来**：因为当时正值**万圣节**，团队成员决定先放下问题去聚会，周一再处理，因此该问题被戏称为“万圣节问题”（Halloween Problem）。

### 解决方案：在扫描操作中维护“已见过”的元组元数据

**核心思想：记录已处理的元组标识，防止重复更新。**

- **额外状态跟踪**：在扫描操作内部，必须维护一个**额外的数据结构**，记录**已经处理过的元组的标识符**。
- **标识符类型**：可以是**物理记录ID（RID）** 或**主键**。因为更新后元组可能在存储层移动（如记录移动导致 RID 变化），所以使用逻辑主键通常更可靠。
- **元组级别过滤**：扫描器在获取下一个元组时，检查该元组的标识是否已在“已处理”集合中。如果存在，则跳过，不执行更新。
- **事务内的特殊处理**：如果同一个事务中**运行了两次相同的 UPDATE 查询**，那么第二次运行时应该允许再次更新（因为两次查询本身是独立的操作）。但若**同一查询内部**出现重复扫描，则必须阻止。因此，需要区分“查询内”和“事务内”的元组可见性。

**关键区别**：
- **查询级跟踪**：只在当前查询执行期间记录已处理的元组，防止单次查询内的重复。
- **事务级跟踪**：记录整个事务内修改过的元组，可用于处理事务内多次执行相同查询的情况（此时重复更新是用户预期的行为）。

### 公式/算法要点

- **扫描与更新分离**：最佳实践是**先收集所有要更新的元组标识（RID/主键）**，然后再基于这些标识执行更新，彻底避免扫描与更新的交错。
- **位图或哈希表**：常用**哈希表**或**位图**来快速判断元组是否已被处理。

## 访问方法（Access Methods）总结

### 1. 顺序扫描（Sequential Scan）

- **默认选择**：当没有更优索引可用时，顺序扫描是数据库的**回退选项**。
- **加速手段**：
    - **并行化**：将表分片（分区）到多个线程/核心同时扫描，是最有效的加速方式。
    - **Zone Maps（区域映射）**：在存储层面（如数据页级别）维护每页的**最大值/最小值**等统计信息。扫描时可以跳过明显不包含目标数据的页，大幅减少 I/O。

### 2. 索引扫描（Index Scan）

- **单索引扫描**：使用单个索引快速定位目标数据行。
- **多索引扫描**：涉及多个索引时，分为两种策略：
    - **交集（Intersection）**：取多个索引结果集的**交集**（AND 操作）。
    - **并集（Union）**：取多个索引结果集的**并集**（OR 操作）。
- **位图扫描**：对每个索引返回的元组标识生成**位图**，然后通过位运算（AND/OR）快速合并结果，最后回表获取完整数据。

### 3. 插入操作的选择差异（问题补充）

- **Choice 1（直接插入）**：简单地在表中追加一条新记录，不涉及扫描，通常直接写入文件末尾。
- **Choice 2（通过扫描定位插入点）**：如果插入需要维护特定顺序（如 B+ 树索引的平衡）或需要检查重复键，则可能需要先扫描查找插入位置。此时若扫描与插入操作在同一事务内，也可能触发万圣节问题，需要**谨慎处理元组可见性**。

## 表达式求值

### 插入操作的具体实现

- **核心概念**：插入操作符的两种实现方式
  - **方式一**：使用独立的 **Materialize 操作符**，将输入参数列表转换为实际的元组（Tuple）
  - **方式二**：让 **Insert 操作符自身** 完成参数到元组的转换

- **关键选择**：虽然许多系统（如Postgres）采用方式一，但**方式二更符合关系代数原则**
  - 操作符应该**保持可组合性**
  - Insert 操作符不需要关心下层是 Materialize 还是 Scan 操作符
  - 只需接收下层传来的数据并执行插入

### 表达式树（Expression Tree）

- **核心概念**：SQL 查询中的各种子句（WHERE、HAVING、投影列表）都被表示为**表达式树**
  - **树结构分层**：
    - **外层**：查询计划树（Join、Scan 等操作符构成）
    - **内层**：每个操作符内包含多个**表达式树**，用于求值

- **工作示例**：连接操作 `R JOIN S ON R.id = S.sid AND S.value > 100`
  - 表达式树结构：
    ```
    AND
    ├── EQUALS
    │   ├── R.id
    │   └── S.sid
    └── GREATER_THAN
        ├── S.value
        └── 100
    ```

- **运行时求值过程**：
  1. 从顶层节点开始**遍历树**
  2. 遇到 AND 节点，先访问左子树
  3. 进入 EQUALS 节点，获取 R.id 和 S.sid 的值
  4. 执行比较操作，结果返回给 AND 节点
  5. 如果结果为 false，可**短路**，无需访问右子树
  6. 如果结果为 true，继续访问右子树执行 `S.value > 100` 的检查

- **关键优化**：**短路求值**
  - 在 AND 连接中，如果第一个条件为 false，直接跳过第二个条件的求值
  - 显著提升性能，避免不必要的计算

## 第 28 部分

## 查询表达式树的求值机制

### **核心概念：表达式树（Expression Tree）的动态求值**
数据库系统将 SQL 查询中的 `WHERE` 子句（比如 `s.value = $1 + 9`）解析为一棵 **表达式树（expression tree）**。查询执行时，针对每一行数据（tuple），引擎会 **“逆向遍历”（reverse traverse）** 这棵树，自顶向下、递归地求值。

---

### **求值过程详解（以 `s.value = $1 + 9` 为例）**

1.  **树结构**：
    *   **根节点**：比较操作符 `=`
    *   **左子节点**：属性访问 `s.value`
    *   **右子节点**：加法操作符 `+`
        *   **左子节点**：参数占位符 `$1`（表示第一个传入的参数）
        *   **右子节点**：常量节点 `9`

2.  **求值流程**：
    *   **从根开始**：执行 `=` 比较，先求左子节点值。
    *   **左子节点求值**：`Attribute(s.value)`
        *   利用当前执行上下文里的 **tuple 指针**，结合 **系统目录（catalog）** 查找 `s.value` 的偏移量（例如已知是第二个属性，整型），直接读取数据（例如 `1000`）。
    *   **右子节点求值（加法）**：
        *   **左子节点 `$1`**：从 **查询参数列表（query parameters）** 中取出第一个参数值（例如 `991`）。
        *   **右子节点 `9`**：直接读到常量 `9`。
        *   执行加法 `991 + 9 = 1000`。
    *   **回到根节点**：执行比较 `1000 = 1000`，结果为 `true`。

3.  **短路优化（short-circuiting）**：
    *   如果是 `AND` 表达式，若左子节点求值结果为 `false`，则 **直接返回 false**，不再遍历右子树（逻辑与的短路行为）。

---

### **为何性能低下？**

*   **核心瓶颈**：对于 **10 亿行数据（1 billion tuples）**，每行都要重复 **遍历整棵表达式树**。
*   **代价**：
    *   每次遍历都涉及 **多级函数调用**（节点跳转、上下文切换）。
    *   **频繁查找系统目录** 来定位字段偏移。
    *   **大量的内存随机访问**（树节点的指针跳转），导致 **CPU 缓存命中率低**。

---

### **预处理语句（Prepared Statement）的优化作用**

*   **机制**：通过 `PREPARE XXX AS SELECT ... WHERE s.value = $1` 创建命名查询句柄，后续可像调用函数一样传入参数执行。
*   **优化点**：虽然预处理能将查询计划缓存下来（避免重复解析），但其 **执行时的表达式树遍历开销仍然存在**。真正的性能飞跃需要 **“编译”（compilation）** 而非“解释”（interpretation），例如将表达式树 **编译为原生机器码**（如LLVM JIT），消除函数指针跳转和目录查询。

## 第 29 部分

## 表达式求值优化：从解释执行到即时编译

### 解释执行（Interpreted Evaluation）的瓶颈

**树遍历表达式求值**是最基本的实现方式，几乎所有数据库系统（PostgreSQL、MySQL等）最初都采用此方案。

- **工作原理**：为每个元组递归遍历表达式树
- **性能问题**：
  - **分支预测困难**：`if` 语句的走向不确定（有时为 true，有时为 false）
  - **CPU流水线效率低**：现代CPU偏好可预测的代码路径
  - **元组规模巨大时问题凸显**：处理10亿个元组时，每次都要执行完整的树遍历

**核心概念**：
- 操作符（如等号`==`）不知道下层如何产生数据，只关心输入类型正确
- 这种**模块化设计**导致每次都要重新解析整个树结构

---

### 即时编译（JIT Compilation / Codegen）的突破

**即时编译**是高端OLAP数据仓库和最新版PostgreSQL采用的优化方案。

**基本原理**：
```
表达式树 → 生成函数代码 → 编译器(GCC/Clang/LLVM) → 机器码(共享对象)
```

**执行过程**：
- 不再遍历表达式树
- 直接调用编译生成的函数
- 传入指向元组的指针，函数内部知道如何跳转到正确偏移量
- 整个求值逻辑被**内联**（inline）到查询计划执行中

**关键优势**：
- **彻底消除分支预测问题**：代码路径单一且确定
- **零树遍历开销**：直接执行机器码
- **编译优化**：利用现代编译器的优化能力（寄存器分配、指令重排等）

---

### 两种方案的对比

| 维度 | 解释执行 | 即时编译 |
|------|----------|----------|
| 执行方式 | 每次遍历表达式树 | 直接调用编译后的机器码 |
| 分支预测 | 不可预测，CPU管道频繁清空 | 代码路径确定，预测友好 |
| 适用场景 | 通用、小规模查询 | 大规模OLAP查询 |
| 系统支持 | PostgreSQL, MySQL等主流系统 | 高端数据仓库, PostgreSQL最新版 |

- 不仅编译表达式，而是**编译整个查询计划**
- 生成高度优化的机器码，包含整个查询管线的各阶段处理逻辑
- 在课程高级部分会详细讨论此技术

## 第 30 部分

### JIT 编译与查询表达式优化：深入理解与性能剖析

#### 核心概念：将表达式编译“内联”进查询计划
- **核心思想**：不仅仅是编译单个表达式，而是将整个表达式的计算逻辑**直接嵌入（inlined）** 到查询计划的执行流程中，实现**即时编译**。
- **性能优势**：通过消除传统解释执行中的函数调用开销、虚函数分派和分支预测失败，获得**极高的执行速度**。
- **关键术语**：**JIT Compilation（即时编译）**、**Code Generation（代码生成）**、**Inlining（内联）**。
- **高级扩展**：在更高级的课程中，会讨论如何对整个**查询计划树（Query Plan Tree）** 进行 JIT 编译，而不仅仅是表达式。这意味着整个查询执行流程（如扫描、过滤、聚合）都能被编译成机器码执行。

#### PostgreSQL 的 JIT 支持：行业实践与演示
- **现状**：PostgreSQL 已经原生支持 JIT 编译，这是一个非常强大的特性。
- **演示目标**：通过一个具体的性能对比实验，直观展示 JIT 带来的性能提升。

#### 实验设置与基线测试（无 JIT）
- **数据集**：一张包含 **5000 万个整数元组**的表。
- **测试查询**：`SELECT count(*) FROM fake_data WHERE (value * value > 100) AND (value % 2 = 0 OR value % 4 = 0)`
    - 包含：聚合函数 (`count`)、算术乘法、取模运算和逻辑组合。
- **前置条件**：使用 `pg_prewarm` 确保整个表的数据**已加载到缓冲池**，消除 I/O 影响，专注于 CPU 计算性能。
- **无 JIT 结果**：执行时间 **5.8 秒**。
    - 观察：查询计划显示为 `Seq Scan` 并应用 `Filter`，所有数据命中共享缓存。

#### 启用 JIT 后的性能对比
- **操作**：启用 PostgreSQL 的 JIT 功能，再次运行完全相同的查询。
- **有 JIT 结果**：执行时间 **3 秒**。
    - **性能提升**：相较于无 JIT 版本，**缩短了 2.8 秒**，加速比接近 **2 倍**。
- **PostgreSQL 的报告**：查询计划中会显示 `JIT: Inlined expressions`，并给出 JIT 各阶段的耗时分解（编译、优化、代码生成）。

#### 关键洞察：JIT 的成本模型与决策
- **并非总是划算**：JIT 编译本身有**固定开销**（例如演示中仅 **36 毫秒**）。PostgreSQL 内部维护一个**成本模型（Cost Model）**。
- **决策逻辑**：
    1.  模型预估待处理的数据量。
    2.  计算 JIT 编译的启动成本（如 36ms）。
    3.  对比总执行时间：如果编译成本占比较大（例如，查询只跑 1 秒，编译花 36ms，则收益有限），则**不会启用 JIT**。
- **底层原理**：SQL 和关系模型的优势之一，在于**声明式查询**使得系统有充分的**元数据（统计信息）** 来做出这种精密的优化决策（是否进行 JIT 编译）。

#### 总结与核心要点
- **JIT 编译是关系数据库的一种高级优化特性**，它通过将表达式编译成机器码并内联到执行计划中，显著加速 CPU 密集型计算。
- **性能提升取决于查询的复杂度和数据量**。对于需要大量计算（如复杂 `WHERE` 子句）且数据量大的查询，JIT 加速效果尤为明显。
- **数据库有自动的成本模型**来决定是否应用 JIT，避免了“杀鸡用牛刀”的情况。
- **公式/算法（无显式公式）**：核心算法是**将抽象语法树（AST）形式的表达式，动态编译为目标平台的机器码**，这涉及到中间表示（IR）、指令选择、寄存器分配等编译器后端技术。

## 第 31 部分

## 查询执行：自动优化与并行执行概述

### 1. 代价权衡：查询优化器自动决策
- **核心概念**：**代价模型**与**自动优化**
    - 查询优化器的核心职责是评估不同执行策略的代价（CPU、I/O、内存等）
    - 如果简单扫描只需几毫秒，但复杂查询却耗时一秒，这就是一个**糟糕的权衡**，优化器必须避免
- **SQL 与关系模型的强大之处**：
    - 用户无需了解底层实现细节（如索引类型、表达式编译）
    - 用户只需声明式地写 `SELECT COUNT(*) FROM table WHERE condition`
    - 数据库系统**自主决定**最优执行路径，包括访问方法和表达式实现方式
- **示例**：`LIMIT 1` 的场景
    - 即使加了 `LIMIT 1`，PostgreSQL 也自动判断是否跳过索引编译
    - 优化器会智能选择执行策略，用户完全不需要手动干预

### 2. 查询计划的多种执行方式
- **核心概念**：**单一查询计划，多种执行路线**
    - 同一个查询计划（如 `COUNT(*)`）可以有不同的物理执行方式
    - 支持索引的数据库**优先使用索引扫描**（如 PostgreSQL, MySQL InnoDB）
    - **无索引系统**（如 Vertica）依赖**可分段扫描**，但其数据是**预排序**的
- **用户视角**：完全透明
    - 编写 SQL 的人不需要知道底层是否用了索引、是否做了编译
    - 数据库系统自动处理所有执行策略细节

### 3. 表达式树 vs. 即时编译
- **核心概念**：**表达式树**的灵活性与性能瓶颈
    - 表达式树（Expression Tree）非常灵活，易于解释执行和优化
    - 但缺点：**解释执行慢**，每次都需要遍历树节点，函数调用开销大
- **现代优化**：**即时编译（JIT Compilation）**
    - 现代数据库（如 PostgreSQL, SQL Server）可以将表达式编译成机器码
    - 大幅减少解释执行的开销，显著提速
    - 对用户完全透明：写 SQL 时无需知道是否存在 JIT

### 4. 未来预告：并行查询与分布式数据库
- **核心概念**：**并行执行**与**分布式扩展**
- **并行查询执行**（下一节课内容）：
    - 将查询计划的各个部分在**多线程、多工作进程**上并行运行
    - 核心挑战：数据切分、并行聚合、同步开销
- **分布式数据库**（后续内容）：
    - 与并行执行本质相同，但节点分布在**不同机器**上
    - 额外挑战：**网络通信**、数据一致性、故障处理
    - 策略：**尽可能延迟分布式操作**，先本地处理后合并

### 5. 关键公式与算法提示
- **代价公式（概念性）**：
    - \( \text{Total Cost} = \text{CPU Cost} + \text{I/O Cost} + \text{Memory Cost} \)
    - 优化器的目标是最小化总代价
- **并行执行效率公式**：
    - \( \text{Speedup} = \frac{T_{\text{serial}}}{T_{\text{parallel}}} \)
    - 理想情况是线性加速，但实际受数据倾斜、同步开销等影响

### 总结核心要点
- **查询优化器的价值**：用户写声明式 SQL，数据库自动选择最优执行路径
- **执行方式多样性**：索引扫描、分段扫描、表达式树、JIT 编译——由系统决定
- **并行与分布式的递进**：先解决单机多线程并行，再扩展到多机分布式，且尽量延迟网络操作
- **用户完全无感知**：所有优化对编写 SQL 的用户透明，这是 SQL 和关系模型的巨大优势

## 数据库核心概念复习指南

### 关系模型基础
- **核心概念**：理解**关系代数（Relational Algebra）** 是操作关系数据的基础语言
- **关键术语**：
  - **关系代数运算符**：选择（Selection）、投影（Projection）、连接（Join）等
  - **符号理解**：必须能看懂并手动组合运算符表达式
  - **重复元组处理**：关系代数中的集合是无重复的，但SQL查询结果允许重复（需掌握 DISTINCT 的使用场景）
- **重点掌握**：运算符的语义、合成规则、重复元组的处理逻辑

### SQL 查询
- **核心概念**：考试不会要求你手写完整SQL（因为无法在不运行的情况下评分），但**会给出SQL代码要求你解释其功能**
- **关键知识点**：
  - **子句类型**：
    - **WHERE 子句**：过滤条件
    - **基本查询结构**：SELECT-FROM-WHERE 的核心范式
    - **输出控制**：ORDER BY（排序）、LIMIT（限行）、OFFSET（偏移）
  - **高级操作**：
    - **JOIN（连接）**：INNER JOIN、LEFT/RIGHT/FULL OUTER JOIN 的区别
    - **聚合（Aggregation）**：GROUP BY + HAVING 的配合使用
    - **CTE（公用表表达式）**：WITH 子句，用于分解复杂查询
- **重要提示**：**窗口函数（Window Functions）不考**，这是明确的考试范围信号

### 数据库存储引擎
- **页面管理策略**（Buffer Pool 替换算法）：
  - **LRU（最近最少使用）**：淘汰最久未访问的页面
  - **LRU-K**：记录最近K次访问时间，避免一次顺序扫描污染缓存
  - **Clock（时钟算法）**：近似LRU，用环形缓冲区+引用位实现
- **磁盘文件结构**：
  - **页目录（Page Directory）**：管理页面位置与空闲空间
  - **记录存储格式**：固定长度 vs 可变长度记录的元数据组织
- **页面内布局**（核心对比）：
  - **Slotted Pages（槽页）**：**记录可移动，槽数组固定**。优点：支持空闲空间合并；缺点：需要槽目录维护
  - **Log-Structured（日志结构）**：**只追加写入，不原地更新**。优点：写效率高；缺点：需要垃圾回收

### 哈希索引
- **静态哈希**：
  - **线性探测（Linear Probing）**：冲突时逐个向后找空位
  - **Robin Hood哈希**：**平衡探测距离**，长探测键与短探测键交换位置
  - **Cuckoo哈希**：**双哈希表+踢出机制**，冲突时踢出旧键到另一张表
- **动态哈希**（支持扩容）：
  - **可扩展哈希（Extendable Hashing）**：使用**全局深度（Global Depth）** 和**局部深度（Local Depth）** 管理桶分裂，通过目录（Directory）进行弹性扩展
  - **线性哈希（Linear Hashing）**：**渐进式分裂**，通过分裂指针逐桶增长，无需全局目录，支持溢出链（Overflow Chains）

### 重点公式与算法
- **Slotted Page 结构**：
  - 页头：`slot_count + free_space_pointer`
  - 槽数组：每个slot存`(record_id, offset, length)`，记录数据从页尾向前增长
- **可扩展哈希分裂条件**：
  - 当桶的**局部深度 < 全局深度**时，只需分裂该桶
  - 当**局部深度 == 全局深度**时，需要**目录翻倍** -> 全局深度+1
- **线性哈希分裂规则**：
  - 分裂指针 `split_ptr` 初始为0
  - 当负载因子超过阈值时，分裂 `split_ptr` 指向的桶，然后 `split_ptr++`
  - 查找时：先用 `hash(key) MOD (N * 2^level)`，若结果 < split_ptr，再用 `hash(key) MOD (N * 2^(level+1))`

- **不需要区分不同DBMS的实现差异**（如PostgreSQL vs MySQL 的具体做法）
- **重点考察通用概念**：算法的原理、优缺点、适用场景
- **SQL题型**：给定SQL代码，分析其功能（如：这是一种什么类型查询？输出中是否有重复行？JOIN类型是什么？）
