## 一、关系模型与声明式查询

> 主题：关系模型的核心思想是让用户声明“想要什么”，由数据库系统决定“如何实现”，实现用户意图与系统实现的解耦。

### 核心价值：用户指定“是什么”，而非“如何做”

*   **核心目标**：将 **用户意图** 与 **系统实现** 解耦。
*   **关键理念**：用户只需用高级语言（如 SQL）声明**想要什么答案**（*what*），数据库系统负责决定**如何计算**这个答案（*how*）。
*   **历史背景**：Ted Codd 提出的关系模型，是对当时低级、指令式数据库访问方式的革命性颠覆。它摆脱了用户需要编写类似“如何遍历文件、如何执行索引”等底层指令的束缚。

### 抽象与实现分离：数据库系统的黑盒魔力

*   **核心机制**：**查询计划**（Query Plan）的自动生成。
    *   用户的 SQL 声明被转化为一个内部的、高效的执行计划。
    *   这个计划的生成是极其复杂的，但完全对用户**隐藏**。
*   **优化范围**：系统在“如何做”的层面拥有巨大的自由度，这体现在多个层次：
    *   **存储层**：如何物理组织数据（如行存、列存、索引结构），这将在后续课程讨论。
    *   **算法层**：如何执行连接（Join），例如是使用嵌套循环连接（Nested Loop Join）还是哈希连接（Hash Join）。
    *   **执行策略**：如何并行化、如何利用缓存等。
*   **核心理念**：这种 **抽象化（Abstraction）** 正是 SQL 和关系数据库强大生命力的源泉。它将底层实现的复杂性封装起来，让用户专注于业务逻辑。

### 从声明式到执行：系统的巨大责任

*   **用户视角**：简单、声明式的编程模型。
*   **系统视角**：复杂的优化器需要承担巨大的责任，它必须：
    1.  解析 SQL 语句。
    2.  转换成逻辑计划（Logical Plan）。
    3.  应用等价变换规则（如谓词下推）。
    4.  基于统计信息（如基数、数据分布）估算执行代价。
    5.  从无数可能的计划中，搜索并选择**代价最低**的物理计划（Physical Plan）。

### SQL与关系代数的关系

*   **关系代数**：是一种**逻辑操作**的定义，描述了"做什么"（例如：选择、投影、连接等基本算子）
*   **SQL**：是一种**语法规范**，用于表达查询意图
*   **查询流程**：SQL查询 → 转换为**逻辑查询计划**（基于关系代数） → 转换为**物理查询计划**（可执行的操作）
*   **关键术语**：
    *   **逻辑查询计划**：基于关系代数算子的抽象操作序列
    *   **物理查询计划**：数据库实际执行的具体操作，包括扫描方式、索引使用等
    *   **Explain命令**：用于查看数据库生成的物理查询计划

### SQL 的数学基础——袋代数（Bag Algebra）

#### **三种基本数据集合类型**

| 集合类型 | 是否有序 | 是否允许重复 | 典型操作 |
|---------|---------|-------------|---------|
| **列表（List）** | ✅ 有 | ✅ 允许 | 可在末尾追加元素 |
| **集合（Set）** | ❌ 无 | ❌ 不允许重复 | 插入重复键会失败 |
| **袋（Bag）** | ❌ 无 | ✅ 允许重复 | 介于列表和集合之间 |

#### **为什么 SQL 基于袋代数而非集合代数？**

*   **核心原因**：**查询处理性能优化**
    *   在关系代数课上，我们使用的基础是**集合论**
    *   但在实际数据库系统中，如果**允许重复数据（即使用袋）**，许多查询算子和算法的执行效率会**大幅提升**
    *   后续课程中会详细看到这一点如何影响查询处理的具体实现

> **关键区别**：SQL 的底层数学基础是**袋代数（Bag Algebra）**，而不是之前学习的**集合代数（Set Algebra）**。数据库系统选择袋模型，是因为它能让查询处理变得更加高效。

## 二、SQL 语言的历史与标准

> 主题：SQL 起源于 IBM 的 System R 项目，其成功并非技术最优，而是得益于 IBM 的背书和 Oracle 的模仿，最终通过 ANSI/ISO 标准化成为行业标准。

### 核心语言与系统：SQL的前身

*   **SQUARE（Specifying Queries as Relational Expressions）**
    由IBM的两位研究者（SQL原作者）在发明SQL之前设计的语言。
    *   **核心特点**：基于关系表达式来指定查询。
    *   **致命缺陷**：**语法极其复杂且难以编写**，导致实用性差。

*   **SQL（Structured English Query Language）**
    作为SQUARE的“后继者”诞生，旨在简化查询表达。
    *   **命名由来**：原名为 **SEQUEL**（Structured English Query Language），但IBM因商标纠纷被另一家英国公司起诉（对方拥有名为“SQL”的制造语言），因此被迫缩写为SQL。
    *   **发音习惯**：至今许多人（尤其是早期从业者）仍将其读作 **“sequel”**，而非逐字母拼读，以保留原始名称。
    *   **语言双关**：同时期伯克利大学的 **Ingres** 系统拥有查询语言 **QUEL**（由Postgres发明者Mike Stonebreaker开发），而“SEQUEL”的命名是对“QUEL”的戏仿——意为“QUEL的续集”（sequel to QUEL）。

### 关键系统的商业演变

*   **System R（IBM）**
    SQL最初诞生于System R项目（1970年代），但IBM从未将System R直接商业化。
    ➔ **衍生版本**：IBM基于System R开发了多个商业分支：**System/38**、**SQL/DS**、以及最著名的 **DB2**。
    *   **DB2的现状**：至今仍在活跃维护，为IBM持续创造利润。

*   **Ingres（伯克利） vs. Oracle（商业模仿）**
    *   **Ingres**：使用QUEL语言，由Mike Stonebreaker（后创立Postgres）主导，但最终未能成为行业标准。
    *   **Oracle**：创始人Larry Ellison曾“电话窃听”IBM System R团队，直接模仿其错误码和设计细节。
    *   **历史讽刺**：Oracle之所以最终击败Ingres成为霸主，核心原因是**IBM选择了SQL**——由于IBM当时的市场统治力（“软件巨擘”地位），其采纳的SQL自动成为行业事实标准。Oracle完全复制IBM路线，而Ingres因未跟随SQL标准而被边缘化。

### SQL 历史与标准

*   **核心概念**：**SQL（结构化查询语言）** 并非Oracle原创，其真正崛起得益于 **IBM** 的 **DB2** 和 **SQL标准** 的建立。
*   **关键术语**：**ANSI标准（1986）**、**国际标准（1987）**、**SQL-92**。
*   **历史脉络**：
    *   Oracle之所以成为数据库巨头，是因为IBM推出SQL（DB2）并成为主流，Oracle顺势支持了IBM的语言，从而占据市场先机。
    *   1986年成为ANSI标准，1987年成为国际标准，名称从“SEQUEL”缩短为“SQL”。
*   **现代发展**：
    *   SQL并非过时语言，最新版本为 **SQL:2016**。
    *   核心结构（`SELECT`、`INSERT`、`UPDATE`、`DELETE`）不变，但不断加入新特性，例如 **JSON支持** —— 这原本是 **NoSQL** 系统的常见功能，表明SQL标准在吸收其他数据管理概念。
*   **SQL-92 最低要求**：
    *   一个数据库系统若声称支持SQL，至少必须实现 **SQL-92** 标准定义的功能，包括：`SELECT`、`INSERT`、`UPDATE`、`DELETE`、`CREATE TABLE`、`DROP TABLE`、**事务处理** 等。

### 标准实现的现实

*   **核心概念**：**没有数据库系统完全遵守SQL标准**。
*   **原因**：
    *   标准制定委员会成员来自各大数据库公司，他们各自 **推销自家特性** 并试图让对方接受，导致标准本身存在利益博弈。
    *   各数据库系统在具体实现时，常常 **偏离标准** 以突出自身特色或保留历史兼容性。
*   **实际影响**：
    *   例如“谁最遵守标准？”这个问题，业内公认 **PostgreSQL** 是最严格遵循SQL标准的数据库之一，但即便如此也存在偏差。
    *   演示中会发现不同数据库对同一SQL语句的解析和执行可能有差异。

## 三、SQL 语言核心组成

> 主题：SQL 标准由 DML、DDL、DCL 三大组件构成，分别负责数据操作、结构定义和权限控制。

### SQL 的三大组成部分

SQL 标准规范由三个主要语言组件构成，每个组件负责数据库系统的不同方面：

*   **DML（数据操纵语言）**
    *   **核心功能**：用于**读取（查询）** 和**修改（插入、更新、删除）** 数据库中的数据
    *   这是日常与数据库交互最常用的部分

*   **DDL（数据定义语言）**
    *   **核心功能**：定义数据的**结构和元数据**
    *   涵盖范围远超表定义，包括：
        *   **索引**（Indexes）
        *   **命名空间**（Namespaces）
        *   **触发器**（Triggers）
        *   **函数**（Functions）
        *   **视图**（Views）
        *   **完整性约束和引用策略**
        *   **值域约束**（确定表列可以取哪些值）

*   **DCL（数据控制语言）**
    *   **核心功能**：管理**安全权限**和访问控制
    *   例如：定义谁可以读取哪些数据

*   **事务（Transactions）**（重要补充概念）
    *   **核心概念**：允许将**一组操作原子性地执行**
    *   这是数据库系统中非常关键的特性，确保数据的一致性和完整性

> **标准遵从性**：虽然存在 SQL 标准，但不同数据库厂商（如 Oracle、PostgreSQL）的实现并不完全一致。Oracle 被认为是最遵从标准的，PostgreSQL 次之。每个数据库在语法的细节上都会有所不同。

## 四、聚合函数与分组操作

> 主题：聚合函数将多行数据压缩为单个标量值，GROUP BY 子句将数据分桶后独立应用聚合，HAVING 子句用于过滤聚合后的结果。

### 聚合函数（Aggregation Functions）

**核心概念：** 聚合是一种**将一组元组（Tuple Bag）压缩为单个标量值（Scalar Value）** 的函数。它接收多条记录，逐条处理，最终输出一个单一数值。

**SQL 92 标准五大聚合函数：**
*   **AVG** - 计算平均值
*   **MIN** - 求最小值
*   **MAX** - 求最大值
*   **SUM** - 求和
*   **COUNT** - 计数（统计元组数量）

**扩展功能：**
*   数据库系统可支持**标准差（Standard Deviation）**、**几何平均值（Geometric Mean）** 等非标准聚合
*   部分系统允许**用户自定义聚合函数（User-Defined Aggregation Functions）**

**实际示例：**
```sql
-- 统计拥有 @cs 登录名的学生数量
SELECT COUNT(*) FROM students WHERE login LIKE '%@cs';
```

### `COUNT` 函数的不同写法与含义

*   **`COUNT(attribute)`**：计算某列（属性）的非空值个数。例如 `COUNT(login)` 会统计 `login` 字段**有值**的行数。
*   **`COUNT(*)`**：统计结果集中**所有行**的数量，与具体列的值无关。这是 SQL 的标准写法，性能上通常等价于 `COUNT(attribute)`，因为数据库优化器会自动处理。
*   **`COUNT(1)`**：为每一行添加常量值 `1`，然后统计其总和。**结果与 `COUNT(*)` 完全相同**，但更强调“累加”的数学含义。

> **要点**：这三种写法在语义和结果上没有区别。`COUNT(*)` 是最通用、最推荐的做法，因为数据库系统（如 PostgreSQL, MySQL）会将其内部优化为 `COUNT(1)`，从而避免读取所有列的数据，提高性能。

### 多聚合与去重聚合

*   **多聚合查询（Multiple Aggregates）**
    可以在同一个 `SELECT` 子句中同时使用多个不同的聚合函数。

    *   **示例**：同时查询特定学生群体的平均 GPA 和学生总数。
        ```sql
        SELECT AVG(gpa), COUNT(*)
        FROM student
        WHERE login LIKE '%@cs';
        ```
        此查询会返回一个包含两列的单行结果：`avg_gpa` 和 `count`。

*   **去重聚合（DISTINCT Aggregation）**
    为 `COUNT`、`SUM`、`AVG` 等函数添加 `DISTINCT` 关键字，可以统计非重复值的数量。

    *   **示例**：统计 `login` 字段中，满足 `@cs` 条件的不同用户名（handle）数量。
        ```sql
        SELECT COUNT(DISTINCT login)
        FROM student
        WHERE login LIKE '%@cs';
        ```
        **关键**：`DISTINCT` 会先对 `login` 值进行去重，再执行聚合。

### 分组操作（GROUP BY）

**核心概念：** 分组操作将元组**按某些属性的值分成多个组**，然后对每个组**独立应用聚合函数**，为每个组生成一个结果值。

#### 分组操作的工作流程
1.  **拆分阶段**：根据指定属性值，将原始元组集合划分为多个子集
2.  **聚合阶段**：对每个子集分别应用聚合函数
3.  **输出阶段**：每个子集生成一行结果

#### 分组查询的基本语法
```sql
SELECT 分组列, 聚合函数(聚合列)
FROM 表名
WHERE 条件
GROUP BY 分组列;
```

**关键规则：**
*   **SELECT 子句**中只能包含：
    *   `GROUP BY` 中指定的分组列
    *   聚合函数（AVG, MIN, MAX, SUM, COUNT 等）
*   非分组列且非聚合的列**不能直接出现在 SELECT 中**

#### 核心要点

*   **必须映射**：`SELECT` 中的非聚合列列表，必须是 `GROUP BY` 中列列表的**子集**。
*   **多列分组**：`GROUP BY colA, colB` 会将 `(colA, colB)` 值完全相同的行分为一组。
*   **聚合函数**：`AVG`, `SUM`, `COUNT`, `MIN`, `MAX` 是标准聚合函数，它们分别作用于每个组内的列。

#### 对比：错误与正确写法

| 场景 | 写法 | 结果 |
| :--- | :--- | :--- |
| **错误** | `SELECT course_id, AVG(gpa), student_name FROM t GROUP BY course_id` | `student_name` 无意义（随机/报错），因为未在 `GROUP BY` 中指定且不是聚合列。 |
| **正确** | `SELECT course_id, AVG(gpa) FROM t GROUP BY course_id` | 清晰输出每门课程的 `course_id` 和平均 GPA。 |

### SQL查询执行逻辑：聚合、分组与过滤的深层机制

#### 核心概念：SQL查询的**执行顺序**与**作用域限制**

SQL并非简单地按照书写顺序（SELECT → FROM → WHERE → GROUP BY → HAVING）逐行执行。其实际逻辑执行顺序为：

1.  **FROM**：确定数据源（表）。
2.  **WHERE**：对**原始行**进行逐行过滤。
3.  **GROUP BY**：将过滤后的行分组。
4.  **HAVING**：对**分组后的结果**进行过滤（基于聚合值）。
5.  **SELECT**：选择并计算最终输出的列（包括聚合函数结果）。

这个顺序决定了哪些列可以在哪个子句中被引用，这是理解SQL限制的关键。

#### 关键规则一：`SELECT` 中的非聚合列必须在 `GROUP BY` 中

*   **问题场景**：当你使用 `GROUP BY` 时，如果 `SELECT` 中出现了**不参与聚合**的列（例如 `student_name`），而该列**没有**出现在 `GROUP BY` 子句中，这会产生冲突。
*   **根本原因**：
    *   分组后，每个组包含**多行**数据（例如，一个课程ID下有多个学生）。
    *   数据库无法唯一确定从该组的**哪一行**中取出这个非聚合列的值。
    *   **数据库优化器**会强制要求：所有非聚合列必须在 `GROUP BY` 中明确指定，以保证结果的确定性。
*   **示例**：
    ```sql
    -- 错误写法: student_name 不在 GROUP BY 中
    SELECT course_id, student_name, AVG(gpa) AS avg_gpa
    FROM enrollments
    GROUP BY course_id; -- 导致错误
    ```
    *   *解释*：`course_id` 分组后，每个 `course_id` 组内包含多个 `student_name`，数据库不知道该输出哪个 `student_name`。
*   **正确做法**：**要么**将 `student_name` 加入 `GROUP BY`，**要么**将其作为聚合函数的一部分（如 `MAX(student_name)`）。

#### 关键规则二：`WHERE` 子句**不能**引用聚合结果

*   **问题场景**：你想在 `WHERE` 中过滤**聚合后的结果**（例如，`AVG(gpa) > 3.9`）。
*   **根本原因**：
    *   `WHERE` 在 `GROUP BY` 和聚合**之前**执行。
    *   当处理到 `WHERE` 时，数据库逐行扫描原始数据，此时**尚未计算**任何分组的聚合值（如 `AVG(gpa)`）。
    *   因此，`WHERE` 子句中无法引用聚合函数或聚合列的别名（`AVG(gpa)` 或 `avg_gpa`）。
*   **错误示例**：
    ```sql
    -- 错误: WHERE 中不能使用聚合结果
    SELECT course_id, AVG(gpa) AS avg_gpa
    FROM enrollments
    WHERE AVG(gpa) > 3.9  -- 此时 avg_gpa 尚未计算
    GROUP BY course_id;
    ```
*   **解决方案**：使用 `HAVING` 子句。
    ```sql
    -- 正确: 使用 HAVING 过滤聚合结果
    SELECT course_id, AVG(gpa) AS avg_gpa
    FROM enrollments
    GROUP BY course_id
    HAVING AVG(gpa) > 3.9;
    ```

#### 关键规则三：`HAVING` 子句中的**重复计算与优化**

*   **常见困惑**：为什么在 `HAVING` 中必须**重新书写**聚合表达式（如 `AVG(gpa)`），而不能直接使用 `SELECT` 中定义的别名（如 `avg_gpa`）？
    *   **原因**：SQL的**执行顺序**决定了 `SELECT` 子句中的别名在 `HAVING` 之后才生效。`HAVING` 无法访问 `SELECT` 作用域内的别名。
    *   **优化器行为**：尽管你在 `HAVING` 和 `SELECT` 中写了两次 `AVG(gpa)`，但**现代数据库优化器非常智能**。它会识别出这是**同一个表达式**，并**只计算一次**。因此，无需担心性能损失。
*   **结论**：虽然看起来是重复代码，但这是SQL语法的必要约束，且底层执行是高效的。

### 聚合后的过滤：HAVING 子句

**核心概念**：在SQL中，**聚合操作后的过滤必须使用 HAVING 子句**，而不能放在 WHERE 子句中

*   **关键区别**：
    *   `WHERE`：在聚合前过滤行
    *   `HAVING`：在聚合后过滤组

*   **优化细节**：数据库系统通常能识别重复计算，例如两次计算 `AVG(gpa)`，系统会自动优化为只计算一次

**重要示例**：
```sql
-- 错误写法（聚合条件不能放在WHERE）
SELECT dept_name, AVG(gpa)
FROM student
WHERE AVG(gpa) > 3.0
GROUP BY dept_name;

-- 正确写法（使用HAVING）
SELECT dept_name, AVG(gpa)
FROM student
GROUP BY dept_name
HAVING AVG(gpa) > 3.0;
```

## 五、字符串函数与操作

> 主题：SQL 字符串操作包括 LIKE 模式匹配、正则表达式、大小写转换、拼接等，但各数据库在大小写敏感性和拼接语法上存在显著差异。

### 字符串函数与数据库方言差异化

#### 核心挑战

**关键概念**：SQL 标准（SQL-92）与实际数据库实现存在巨大差异，**"一次编写到处运行"在SQL中基本不现实**

*   **字符串标准规定**：
    *   字符串**大小写敏感**（case-sensitive）
    *   使用**单引号**引用字符串

*   **实际实现差异**：

| 数据库 | 大小写行为 | 标准符合度 |
|--------|------------|------------|
| PostgreSQL | ✅ 大小写敏感 | 严格遵循标准 |
| Oracle | ✅ 大小写敏感 | 严格遵循标准 |
| DB2 | ✅ 大小写敏感 | 严格遵循标准 |
| MySQL | ❌ 大小写不敏感 | 偏离标准 |
| SQLite | ❌ 大小写不敏感 | 偏离标准 |

#### 常见陷阱：大小写敏感性

**核心问题**：MySQL等数据库的字符串**默认大小写不敏感**，导致匹配行为与其他数据库不同

*   **示例场景**：
  ```sql
  -- 在PostgreSQL中（大小写敏感）：
  SELECT * FROM users WHERE name = 'john';
  -- 不会匹配 'John' 或 'JOHN'
  
  -- 在MySQL中（大小写不敏感）：
  SELECT * FROM users WHERE name = 'john';
  -- 会匹配 'john', 'John', 'JOHN' 等任意大小写组合
  ```

*   **解决方案**：使用 `UPPER()` 或 `LOWER()` 函数统一大小写
  ```sql
  -- 标准做法（跨数据库兼容）
  SELECT * FROM users 
  WHERE UPPER(name) = UPPER('john');
  ```

### LIKE 操作符：基础字符串模式匹配

*   **核心语法**：`WHERE column_name LIKE 'pattern'`
*   **通配符 (Wildcards)**：
    *   **`%`**：匹配 **零个或多个字符**（与 bash 中的 `*` 不同，SQL 用 `%`）
    *   **`_`**：匹配 **恰好一个字符**
    *   **示例**：
        *   `WHERE name LIKE 'J%'` → 以 J 开头的所有名字
        *   `WHERE name LIKE '_oe'` → 第一个字符任意，后跟 "oe"（如 Joe，Zoe）
*   **易混淆点**：
    *   SQL 的通配符是 `%`，而不是常见的 `*`，这是历史遗留习惯
    *   **字符串引号**：SQL 标准字符串用**单引号**（`'Hello'`），MySQL 允许双引号但其他数据库不兼容

### 正则表达式 (Regex) 支持

*   **标准差异**：
    *   SQL 标准定义了基本正则，但具体实现依赖数据库
    *   多数数据库支持 **Perl 兼容正则 (PCRE)**，但调用函数不同
*   **示例**：
    *   MySQL: `REGEXP`
    *   PostgreSQL: `~`
    *   SQLite: 需外部扩展
*   **注意**：商业数据库（Oracle, SQL Server）可能使用不同语法

### 字符串函数 (String Functions)

*   **标准 SQL 函数**（所有数据库基本都支持）：
    *   大小写转换：`UPPER()`， `LOWER()`
    *   修剪：`TRIM()`， `LTRIM()`， `RTRIM()`
    *   子串：`SUBSTRING(str, start, length)`（注意索引从 1 开始）
    *   长度：`LENGTH()`
*   **使用位置**：
    *   **SELECT 输出**：`SELECT UPPER(name) FROM users`
    *   **WHERE 条件**：`WHERE LOWER(city) = 'new york'`
    *   可在谓词任意一侧使用

### 字符串拼接的兼容性问题

**核心痛点**：不同数据库使用**完全不同的拼接语法**，这是跨平台移植中最常见的问题。

| 数据库系统 | 拼接操作符/函数 | 示例 |
| :--- | :--- | :--- |
| **SQL Server** | `+` | `'Hello' + ' ' + 'World'` |
| **Oracle** | `||` | `'Hello' \|\| ' ' \|\| 'World'` |
| **PostgreSQL** | `||` | `'Hello' \|\| ' ' \|\| 'World'` |
| **MySQL** | `CONCAT()` | `CONCAT('Hello', ' ', 'World')` |
| **SQLite** | `||` | `'Hello' \|\| ' ' \|\| 'World'` |

*   **开发者陷阱**：由于早期使用 MySQL（习惯 `CONCAT()`），移植到 SQL Server 时使用 `+` 会失败，反之亦然
*   **最佳实践**：如果代码需要跨数据库兼容，使用 **`CONCAT()` 函数**（多数现代数据库支持）或显式标注数据库类型

## 六、日期与时间处理

> 主题：日期和时间处理是 SQL 中最混乱、最易出错的部分，各数据库在函数名、语法和隐式行为上存在大量差异。

### 核心痛点

*   **SQL标准仅定义基本行为**，但具体实现、函数名称、关键字支持因数据库而异。
*   **看似简单的操作**（如获取当前时间、计算天数差）在不同系统中需采用完全不同写法。

### 获取当前日期/时间

*   **NOW() 函数**
    *   **PostgreSQL**：支持 `NOW()`，返回当前时间戳。
    *   **MySQL**：支持 `NOW()`，功能相同。
    *   **SQLite**：**无此函数**，会报错 `no function`。
    *   **SQL Server**：**无此函数**。

*   **CURRENT_TIMESTAMP 关键字/函数**
    *   **PostgreSQL**：**仅支持关键字**（`CURRENT_TIMESTAMP`），无同名函数。
    *   **MySQL**：**同时支持**函数与关键字。
    *   **SQLite**：**无函数**，但支持关键字。
    *   **SQL Server**：**无函数**，但支持关键字。

> **关键结论**：即使是“今天是什么日期”这样基础的操作，也无法用统一写法跨数据库执行。

### 计算天数差：示例与陷阱

**目标**：计算从今年第一天到今天的天数。

##### **步骤一：获取当前日期并转为日期类型**
- 所有数据库都支持 **字符串转日期** 操作：
  ```sql
  -- 通用写法
  CAST('2024-01-01' AS DATE)
  ```

##### **步骤二：寻找天数差函数**
- **PostgreSQL**：支持 `EXTRACT(DAY FROM ...)`，但只能提取**日期中的日部分**（例如1月2日返回2），无法直接计算差值。
- **跨数据库差异**：
  - **PostgreSQL**：需使用 `(CURRENT_DATE - '2024-01-01'::DATE)` 直接减法，返回整数天数。
  - **MySQL**：需使用 `DATEDIFF(CURRENT_DATE, '2024-01-01')`。
  - **SQLite**：需使用 `julianday(CURRENT_DATE) - julianday('2024-01-01')`。
  - **SQL Server**：需使用 `DATEDIFF(DAY, '2024-01-01', GETDATE())`。

> **核心教训**：即使基础类型相同，**日期运算的语法** 也完全不一致，每个系统都有自己的“方言”。

### MySQL 中计算日期差值的多种方法

#### 核心概念：**`UNIX_TIMESTAMP`**

*   **定义**：将日期时间转换为 **自1970年1月1日（Unix纪元）以来的秒数**。
*   **用途**：通过将日期转化为统一的数值（秒数），可以进行精确的数学运算（减法、除法），从而避开MySQL某些类型转换的陷阱。
*   **公式**：计算任意两个日期差值的天数公式为：
  \[
  \text{天数差值} = \frac{\text{日期A的Unix秒数} - \text{日期B的Unix秒数}}{60 \times 60 \times 24}
  \]
  其中 `60 \times 60 \times 24 = 86400` 是一天的总秒数。

#### 陷阱分析：直接相减导致“垃圾数字”

*   **现象**：直接对 `DATE` 或 `DATETIME` 类型的值进行减法运算，会得到一个 **看起来毫无意义的8XX整数**（例如 `801`、`728`），而不是期望的天数。
*   **原因**：MySQL在无法直接进行标准日期运算时，会执行 **隐式的数字编码**：将日期 `YYYY-MM-DD` 的月份和日部分直接组合成一个整数。
    *   例如，若计算 `'2024-09-02' - '2024-01-01'`，MySQL会将其视为数字 `20240902 - 20240101 = 801`。
    *   **详细分解**：
        *   结果的第一个数字（8）来自：当前月份减目标月份：**9 - 1 = 8**。
        *   结果的最后两位（01）来自：当前日子减目标日子：**2 - 1 = 1**。
    *   **结论**：这个结果 **不是** 天数，而是一个由月份和日期的差拼接而成的“伪整数”，**完全不能用于计算日期差**。

#### 推荐方案：`DATEDIFF()` 函数

*   **定义**：MySQL内置的 **原生日期差值函数**。
*   **语法**：`DATEDIFF(date1, date2)`
*   **行为**：直接返回 `date1` 和 `date2` 之间相差的 **天数**。它 **自动处理** 了月份和年份的跨越，结果是一个整数。
*   **优势**：
    *   **代码简洁**：一行代码即可完成，无需手动转换时间戳或进行除法。
    *   **语义清晰**：函数名明确表达了意图，代码可读性高。
    *   **准确可靠**：避免了隐式类型转换的陷阱，结果绝对正确。
*   **示例**：
    ```sql
    SELECT DATEDIFF('2024-09-02', '2024-01-01');
    -- 结果为: 245
    ```
*   **最佳实践**：在需要计算两个日期之间的天数时，**始终优先使用 `DATEDIFF()`**，而不是手动相减或使用复杂的Unix时间戳转换，除非有特定的性能或时区处理需求。

### 不同数据库系统中的日期差分操作

#### 核心问题概述
- 在不同数据库系统中，计算两个日期之间的天数差（例如计算从年初到当前日期的天数）**没有一个统一的标准方法**
- 每种数据库系统都有其独特的语法和函数，导致实现方式千差万别

#### MySQL 实现方式

**方法一：使用 `DATEDIFF` 函数**
- **核心语法**：`DATEDIFF(today_date, beginning_of_year)`
- **基本逻辑**：直接传入两个日期参数，函数返回天数差

**方法二：使用日期直接相减**
- MySQL 支持**日期类型的直接算术运算**
- 两个日期相减直接得到天数差值

#### SQLite 实现方式

**核心挑战：缺乏 `DATEDIFF` 函数**
- SQLite 没有内置的 `DATEDIFF` 函数
- 也无法直接对日期进行算术运算

**解决方案：儒略历转换**
- **核心概念**：将日期转换为**儒略历天数**（Julian Day Number）
- **操作步骤**：
  1. 使用 `julianday()` 函数将日期转换为儒略历天数
  2. 对两个儒略历天数进行相减
  3. 使用 `CAST ... AS INTEGER` 将结果转换为整数
- **示例代码结构**：
  ```sql
  CAST(julianday(today_date) - julianday(beginning_of_year) AS INTEGER)
  ```

#### SQL Server 实现方式

**方法一：使用 `CONVERT` 函数和日期相减**
- **背景说明**：SQL Server 没有独立的 `DATE` 类型，而是使用 `DATETIME` 类型
- **操作步骤**：
  1. 使用 `CONVERT` 函数将字符串转换为 `DATETIME` 类型
  2. 对两个 `DATETIME` 值进行相减
- **结果问题**：直接相减会返回一个不直观的日期值（如 `1900年9月3日`），**不是直接的天数差**

**方法二：使用 `DATEDIFF` 函数**
- **核心语法**：`DATEDIFF(day, beginning_of_year, today_date)`
- **重要陷阱**：**参数顺序影响结果的正负**
  - `DATEDIFF(day, today_date, beginning_of_year)` → 返回**负数**
  - `DATEDIFF(day, beginning_of_year, today_date)` → 返回**正数**
- **与 MySQL 的对比**：MySQL 中 `DATEDIFF` 的参数顺序是 `(today_date, beginning_of_year)`，与 SQL Server **恰好相反**

#### 关键教训与最佳实践

**数据库兼容性痛点**
- **同一操作（计算日期差）**在不同数据库系统中存在**多种不同的实现方式**：
  - MySQL：`DATEDIFF(today, beginning)` 或直接相减
  - SQLite：`CAST(julianday(today) - julianday(beginning) AS INTEGER)`
  - SQL Server：`DATEDIFF(day, beginning, today)` → **参数顺序与 MySQL 相反**

**实用建议**
- **跨数据库开发时**：务必查阅具体数据库的文档，验证函数参数顺序和返回值类型
- **代码审查重点**：
  - 检查 `DATEDIFF` 的参数顺序（MySQL vs SQL Server）
  - 确认日期类型的支持情况（SQL Server 无独立 DATE 类型）
  - 验证结果的正负是否符合预期

**核心公式总结**
**日期差 = 较晚日期 - 较早日期**（逻辑上的通用原则）

| 数据库 | 推荐实现方式 | 关键注意点 |
|--------|-------------|-----------|
| MySQL | `DATEDIFF(today, beginning)` | 日期类型可直接相减 |
| SQLite | `CAST(julianday(today) - julianday(beginning) AS INTEGER)` | 需要儒略历转换 |
| SQL Server | `DATEDIFF(day, beginning, today)` | **参数顺序与 MySQL 相反** |

## 七、查询输出控制

> 主题：通过 ORDER BY 排序、LIMIT/OFFSET 分页控制查询输出，输出重定向可将结果存储到表中。

### 排序控制：ORDER BY 子句

*   **核心概念**：由于**关系代数的 Bag 语义**，表中的元组（行）**没有固有顺序**，插入顺序也不保证能反映在查询结果中。若要保证输出顺序，必须显式使用 `ORDER BY`。
*   **关键术语**：**Bag 语义（Bag Semantics）**、**排序（Sort Order）**、**升序/降序（ASC/DESC）**
*   **排序方式**：
    *   **指定列名**：`ORDER BY grade ASC` – 按 `grade` 列升序排列。
    *   **默认排序**：若不指定，**默认是 `ASC`（升序）**。
    *   **指定偏移量**：`ORDER BY 2 DESC` – 按查询
