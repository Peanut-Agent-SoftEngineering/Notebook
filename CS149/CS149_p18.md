# CS149_p18

## 一、能效计算：从 Dennard Scaling 到专业化的必然性

> 主题：能量受限的计算时代下，能效成为新的性能瓶颈，专业化是提升能效的关键路径。

### 1. 核心驱动力：能效是新的性能瓶颈

*   **背景现状：** 我们正处在一个**能量受限**的计算时代。
*   **核心问题：** 为什么能效如此紧迫？
    *   **Dennard Scaling 的终结：** 过去，每一代新制程工艺能提供更多晶体管，但单个晶体管功耗降低，使得总功耗不变。**十年前，这个规律停止了**。
    *   **当前困境：** 现在，增加晶体管数量直接导致功耗上升。
    *   **根本公式：** **能量 = 功率 × 时间**
        *   在现代计算中，**功率 (Power)** 受限于散热和供电能力，几乎固定。
        *   因此，要提升**性能 (性能 = 1/时间)**，唯一途径是**降低每操作的能量 (Energy)**。
*   **关键结论：** 提升性能 = 提升能效。**能效是新时代提升性能的杠杆。**

### 2. 能量受限的广泛场景：无处不在的限制

*   **高性能计算 (HPC) 和数据中心：**
    *   **限制来源：** 电力供应和冷却系统。
    *   **成本压力：** 在大型设施中，为整个系统供电和制冷的成本巨大，甚至可能**超过硬件本身的采购成本**。
*   **移动设备：**
    *   **限制来源：**
        1.  **被动散热：** 没有风扇，只能依靠被动散热。
        2.  **电池容量：** 必须依靠有限的电池提供计算所需能量。
*   **核心观点：** 从云端到终端，整个计算领域都已被“能效墙”所困。

### 3. 解困之道：走向专业化

*   **核心机制：** 为了在固定的功率预算内获得更高的性能，必须**提高效率**。
*   **提高效率的关键步骤：** **专业化**。
*   **专业化的本质：** 去除计算中不直接用于“推动计算前进”的**额外开销**。
    *   *类比：* 通用CPU需要为各种指令集和分支预测等消耗能量，而专业化硬件（如GPU）只专注“数据并行”，剔除了这些“无关”的能量消耗。
*   **回顾：** 我们之前讨论的**异构性**和**数据并行计算**正是利用了这一原理。
    *   **GPU 的专业化：** 专注于数据并行计算，通过大规模简化控制逻辑，在同样的功耗下获得远超CPU的算力。

### 4. 核心公式梳理

| 概念 | 关系 | 解释 |
| :--- | :--- | :--- |
| **功率 (Power)** | **受限的常数** | 由供电和散热能力决定（TDP），无法轻易增加。 |
| **能量 (Energy)** | **能量 = 功率 × 时间** | 完成一个任务总消耗。 |
| **性能** | **1 / 时间** | 单位时间内完成的运算量。 |
| **能效** | **1 / 能量** | 单位能量能完成的运算量。 |
| **核心矛盾** | **功率固定 → 要提升性能 (缩短时间) → 必须降低每任务能量 → 必须提升能效** | |
| **解决方案** | **专业化** | 设计专用硬件，去除通用处理中不必要的开销，从而在固定功率下大幅缩短时间。 |

## 二、专门化 vs. 通用处理：性能提升的根源与代价

> 主题：通用CPU效率低下的原因，以及SIMD/GPU通过摊销指令开销提升效率的机制与局限。

### 1. 为什么通用CPU效率低下？

*   **根本矛盾**：执行一条指令时，**实际用于计算的能量占比极低**，仅约6%。
*   **能量浪费在何处**：
    *   **指令获取与解码**：读取指令、理解其意图。
    *   **依赖性检查**：检查当前指令是否依赖其他正在执行的指令。
    *   **资源冲突仲裁**：判断执行单元是否被占用，以及操作数是否可用。
    *   **数据移动**：将操作数从寄存器文件/缓存读取，或将结果写回。
    *   **控制与同步**：保持所有流水线阶段同步的时钟分配与电路开销。
*   **结论**：通用CPU的大部分能量和晶体管被用于**“指令管理”而非“数据计算”**。

### 2. 专门化（GPU/SIMD）如何解决效率问题？

*   **核心策略**：通过**“摊销”** 来降低每个数据元素上的“非计算”开销。
*   **具体机制**：**一次指令获取与解码作用于多个数据元素**。
    *   **类比**：原本需要“获取-解码-执行”N次，现在只需要一次指令处理，然后执行N个数据操作。
*   **关键度量**：**SIMD宽度**（一个指令能处理多少数据元素）。
    *   **理想情况**：如果SIMD宽度为8，一个指令就能完成8个数据操作，指令管理开销被**除以8**。

### 3. SIMD宽度的极限：为什么不做16、32或64？

*   **核心矛盾**：**数据并行度** vs. **硬件利用率**。
*   **现实困境**：你的程序可能**无法提供足够的数据并行性**来填满所有SIMD通道。
    *   **“空闲通道”就是浪费**：当一个SIMD指令只使用了8个通道中的3个时，剩余5个通道的计算单元完全闲置，但**指令获取、解码等开销仍然被完整支付**。
*   **结论**：
    *   **必须平衡**：不要盲目追求极端SIMD宽度。
    *   **实际限制**：取决于应用场景能否提供**稳定、足够宽**的数据并行性。
    *   **GPU的做法**：通过**大量线程并行**来隐藏数据并行度不足的问题。

### 4. 总结：专门化的收益与代价

| 维度 | 通用CPU | 专用GPU (利用SIMD) |
| :--- | :--- | :--- |
| **能量效率** | 低（大部分能量花在控制/指令上） | **高**（控制开销被大量数据操作摊销） |
| **峰值性能** | 中等 | **高**（宽SIMD单元） |
| **平均性能** | **稳定**（对所有算法公平） | **波动大**（严重依赖数据并行度） |
| **核心挑战** | 指令级并行性（ILP）提取 | **数据并行性（DLP）的填充** |

**一句话总结：**
专用化通过**摊销指令开销**获得了巨大的峰值性能提升，但这个提升能否转化为实际收益，**完全取决于应用程序能否提供足够的“数据并行性”来填满所有处理单元**。

## 三、专用计算架构：从通用到定制的效率跃迁

> 主题：从SIMD到ASIC的专用计算图谱，以及DSP作为折中方案的特点。

### 1. SIMD的局限性：能效提升的瓶颈

*   **核心概念**：即使对于**数据并行和适合SIMD**的应用，SIMD带来的能效提升也**有限**。
*   **关键术语**：**SIMD能量开销**
    *   在一项斯坦福的研究中，分析了一个CD播放器运行视频编码的能量消耗。
    *   结果显示，**SIMD相关能耗占比并不高**。
    *   **结论**：仅仅依赖CPU内的SIMD，无法在能效上实现巨大突破。

### 2. ASIC：能效与面积的极致

*   **核心概念**：针对**单一特定算法**设计专用硬件，能获得**千倍的硅面积效率提升**和**百倍的能源效率提升**。
*   **关键术语**：**ASIC**、**硅面积效率**、**能源效率**
*   **性能对比（40nm工艺， vs. Intel Core i7）**：
    *   **硅面积效率提升**：ASIC 比 CPU 高约 **1000倍**。
    *   **能源效率提升**：ASIC 比 CPU 高约 **100倍**。
*   **ASIC的致命缺点**：
    *   **灵活性为零**：只能运行**一种**算法。
    *   **开发周期长**：设计、流片需要 **18个月**。
    *   **沉没成本高**：除非算法极其重要，否则不值得投入。

### 3. DSP：一种折中方案

*   **核心概念**：DSP通过**增强通用CPU的指令集和地址模式**，专门为信号处理算法优化，从而在**效率与可编程性**之间取得平衡。
*   **关键术语**：**DSP**、**复杂地址模式**、**位反转寻址**
*   **具体优化**：
    *   **指令集**：加入能一步完成特定算法（如乘加MAC）的复杂指令。
    *   **地址模式**：加入支持FFT算法所需的**位反转寻址**等特殊模式。
*   **权衡结果**：
    *   **编程难度**：介于**ASIC（最难）** 和**通用CPU（最易）** 之间。
    *   **效率**：介于**ASIC（最高）** 和**通用CPU（最低）** 之间。

### 4. 专用计算图谱：效率与灵活性的三元组

| 架构类型 | 效率 (Energy/Area) | 灵活性 (Programmability) | 适用场景 | 开发周期 | 典型例子 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **通用CPU** | 低 | 高 | 任意通用任务 | 短 (软件) | Intel Core i7 |
| **SIMD/GPU** | 中 | 较高 | 数据并行、吞吐量型任务 | 较复杂 (并行编程) | AVX, CUDA |
| **DSP** | 高 | 低 | 固定的信号处理流水线 | 长 (低级汇编) | 音频解码、FFT |
| **ASIC** | 极高 | 无 | 唯一的、极其重要的算法 | 极长 (18个月) | ASIC矿机、FFT芯片 |

*   **核心公式**：**效率提升 = 灵活性牺牲**。

## 四、专用计算单元与可编程硬件的平衡

> 主题：分子动力学加速器（Anton）、机器学习加速器（TPU）和FPGA的设计权衡。

### 1. 分子动力学专用加速器：安东 (Anton)

*   **核心概念**：为了解决**分子动力学**（研究蛋白质折叠等）这一特定领域的问题，而专门设计的加速器。
*   **关键术语**：**分子动力学**、**N体模拟**
*   **核心优势**：通过**算法与硬件的协同设计**，安东专用加速器在性能和能效上远超通用CPU和GPU。
*   **行业内的张力**：在专用加速器（硬件方案）与“统计方法/机器学习”（软件方案）之间存在方法论上的竞争。

### 2. 机器学习专用加速器：张量处理单元 (TPU)

*   **核心概念**：为加速**机器学习**算法（特别是深度学习）而设计的专用处理器。其核心杀手锏是**高速、大规模的矩阵乘法**。
*   **关键术语**：**密集矩阵乘法**
*   **硬件特点**：为了适应快速演进的ML算法，大多数现代ML专用架构都具备**可编程性**，但核心计算核依然是矩阵乘法。
*   **核心公式 (以TPU为例)**：
  `\[
  \text{Output}_{(m,n)} = \sum_{k} \text{InputA}_{(m,k)} \times \text{InputB}_{(k,n)}
  \]`

### 3. 固定算法与可编程性的折中：FPGA

*   **核心矛盾**：设计针对**特定固定算法**的硬件（如ASIC）效率极高但灵活性差。是否有一种**中间立场**？
*   **解决方案**：**现场可编程门阵列 (FPGA)**。
*   **关键结构与原理**：
    *   **可配置逻辑块 (CLB)**：FPGA的基本单元。它本质上是一个**查找表 (LUT)**，用于实现布尔函数。
        *   **核心思想**：将布尔函数的**真值表**映射为查找表。
    *   **寄存器**：与组合逻辑块（LUT）结合，提供存储功能。
    *   **互连结构**：将大量的逻辑块排列成阵列，并通过可编程的连线将它们**连接成更复杂的逻辑电路**。
*   **硬宏模块**：为了提高**密度**并更有效地利用**硅面积**，现代 FPGA 引入了**硬宏模块**，如**密集内存**（Block RAM）和**DSP块**（专用乘法器）。

## 五、可编程计算领域的设计权衡：从 CPU 到 ASIC

> 主题：FPGA与云资源的结合，以及从CPU到ASIC的完整计算技术谱系。

### 1. FPGA 与云资源

*   **访问方式**：除了传统的实验室开发板，现在也可以通过**云服务**（如 Amazon AWS）访问高性能 FPGA 资源。
*   **连接技术**：
    *   与 **CPU** 通过 **PCIe** 接口连接。
    *   与 **其他 FPGA** 通过专用链路进行高速通信。
    *   与 **内存** 通过高速内存控制器连接。

### 2. 计算技术谱系：从 CPU 到 ASIC

*   **核心权衡**：在整个现代计算技术空间中，存在一个根本性的**能源效率**与**可编程性**之间的权衡。

| 技术路径 | 可编程性 (易用性) | 能源效率 (性能/瓦特) | 开发复杂度 |
| :--- | :--- | :--- | :--- |
| **通用 CPU** | **最高** (高级语言编程) | **最低** | 最低 |
| **GPU** | 中等 (CUDA 编程) | 较高 | 中等 |
| **DSP** | 较低 (汇编语言) | 高 | 较高 |
| **领域特定计算** | 领域优化 (如 ML: PyTorch) | 更高 | 领域特定 |
| **FPGA** | **硬件设计** (RTL) | 很高 | 非常高 |
| **ASIC** | **无** (固定功能) | **最高** | **极高 (成本高)** |

*   **设计决策建议**：
    *   **向左移动**：如果 CPU 就能满足性能需求，直接使用高级语言编程是最快且成本最低的方案。
    *   **向右移动**：需求更高性能时，则需在编程难度与开发成本上做出妥协。
    *   **FPGA 与 ASIC 的本质**：使用 FPGA 意味着你实际上成为了**硬件设计师**；而制造 ASIC 则是不计成本、追求极致效率的终极选择。

## 六、从通用计算到专用加速器：内存、计算与编程模型的演进

> 主题：GPU向TPU学习，DSP的适用场景，以及设计/编程加速器时需控制的关键要素。

### 1. 性能与效率的权衡：通用 vs 专用

*   **核心概念：** 虽然GPU是强大的通用并行处理器，但在特定领域（尤其是**机器学习**），专用硬件（如**TPU**）可能提供更高的**每瓦特性能**或**每美元性能**。
*   **核心洞察：**
    *   **“GPU在向TPU学习”**：最新的GPU集成了**Tensor Core**单元，这是GPU通用性向机器学习专用性的一次妥协与融合。
    *   **“Tensor Core很难编程”**：作为CUDA程序员，直接操控Tensor Core具有挑战性，其编程模型与传统的通用线程编程有显著差异。

### 2. 为什么DSP适合特定任务？

*   **核心概念：** **DSP** 拥有专门化的**地址机制**和**计算单元**，使其在处理**数字信号**（如音频、图像滤波）时比通用GPU更高效。
*   **核心洞察：** DSP的专用性在于其**内存访问模式**和**计算模式**与信号处理算法高度匹配。

### 3. 设计/编程加速器时需控制的“控制点”

*   **核心性能杠杆：内存系统**
    *   **核心洞察：** 任何硬件的性能改进，**绝大部分来自于内存系统的组织方式**。
    *   **关键任务：**
        1.  **分析你的应用**：识别出其中的**访问行为**和**局部性特征**。
        2.  **定制硬件内存**：根据应用特征，设计**缓存结构、内存带宽、数据路径**等。
        3.  **匹配计算**：将**专门化的计算单元**与优化后的内存系统结合起来。

### 4. 当前编程加速器的两种方式及其痛点

*   **传统方式：硬件描述语言 (HDL)**
    *   **方法：** 使用 **VHDL** 或 **Verilog** 在**寄存器传输级别 (RTL)** 进行设计。
    *   **痛点：** **极度痛苦、效率低下**。

*   **现代方式：高级综合 (HLS)**
    *   **方法：** 用 **C 程序** 编写算法，然后依赖 **编译器** 将其自动转化为硬件描述。
    *   **两个核心问题：**
        *   **问题一：C 程序不是硬件描述**：C 程序是为**通用顺序处理器**设计的，而硬件是**并行、并发**的。
        *   **问题二：Pragma 的本末倒置**：为了写出高性能的代码，你必须**深入了解底层硬件细节**，才能知道该加哪些 Pragma。这违背了 HLS 的初衷。

## 七、空间语言：面向性能程序员的硬件加速器设计

> 主题：空间语言的核心定位、关键设计原则和核心机制。

### 1. 核心定位：从高级综合到硬件感知

*   **传统高级综合（HLS）的困境**：试图用纯高级语言描述硬件，但为了获得**有价值和性能良好的结果**，必须**降级到硬件级别**。
*   **空间语言**：一种**特定于加速器设计的领域特定语言（DSL）**，专为**性能导向的程序员**设计，使其能够**直接指定硬件结构**。

### 2. 关键设计原则：程序员关心的核心

*   **并行性**：整个学期（即性能导向编程）的核心问题。
    *   **独立并行性**：将任务分配到**独立的计算单元**上执行。
    *   **依赖并行性**：处理相互依赖的并行任务。
        *   **流水线并行性**：类似于**CPU指令流水线**——每个阶段独立工作，但整体形成依赖关系。
*   **局部性**：需要管理**专门的内存**和**数据移动方式**。
    *   程序员必须能够**明确指定内存层次结构**，并控制其使用方式。

### 3. 核心机制：表达并行与局部性的契约

*   **并行设计模式**：语言内置了程序员熟悉的**并行模式**，如 **map**、**reduce**，这些模式可以**嵌套**。
*   **内存与局部性**：
    *   提供**内存模板**，允许程序员**显式控制内存层次结构**。
    *   支持通过**参数化**探索整个设计空间，将这些参数**暴露给编译器**。

## 八、空间语言核心概念与内存层次操作

> 主题：空间语言中的内存层次结构、显式数据移动指令和并行模式。

### 1. 内存层次结构与显式数据移动

*   **核心思想**：**空间语言**要求程序员**显式地在不同级别的内存层次之间移动数据**。
*   **芯片上内存 (SRAM)**：位于芯片内部，访问速度极快。
*   **芯片外内存 (DRAM)**：位于芯片外部，容量大但访问延迟高。
*   **专用内存结构**：
    *   **累加器**：用于执行归约操作的专用寄存器。
    *   **FIFO (队列)**：一种先进先出的队列结构，用于数据流缓冲。
    *   **线缓冲**：一种可以**横向滑动**的二维数组，常用于图像处理中实现滑动窗口卷积。
    *   **移位寄存器**：用于数据流的移位操作。

### 2. 显式数据移动指令：加载与存储

*   **核心概念**：程序员通过指令显式地将数据在不同内存层之间搬移。
*   **指令模式**：
    *   **加载**：将数据从 **DRAM** 传输到 **SRAM**。
    *   **存储**：将数据从 **SRAM** 写回 **DRAM**。

### 3. 不规则数据访问：聚集与散列

*   **聚集**：根据一个索引数组中的地址，从源存储中**随机读取**多个不连续的元素，然后将它们**组合**成一个**密集**的、连续的内存缓冲区。
*   **散列**：是聚集的反向操作，将密集缓冲区中的数据，根据索引数组写回源存储的不连续位置。

### 4. 数据流与控制模板

*   **核心概念**：数据流是提升效率的关键，允许数据以流水线方式持续在内存层次和处理单元间流动。
*   **控制模板**：
    *   **`excel` 块**：将程序划分为两个区域：**可加速部分**（在空间架构上运行）和**CPU部分**。
    *   **`excel*`**：用于指定一个 `excel` 块是**持续运行**还是**只运行一次**。

### 5. 并行模式与 DSL 嵌入

*   **`foreach` (映射 Map)**：对一个集合中的**每个元素**，**独立且并行地**执行代码块。
*   **`reduce` (归约 Reduce)**：将一个集合中的所有元素**组合**成一个单一结果。
*   **语言嵌入**：空间语言是**内嵌**在 **Scala** 语言中的领域特定语言 (DSL)。

## 九、内积计算加速器设计与实现

> 主题：使用空间语言设计内积计算加速器的具体步骤，包括数据分块和并行化策略。

### 1. 核心概念：Reduce 与 For-Each 的并行控制

*   设计时需指定多个**设计参数**：
    *   **并行度**：每个 `for-each` 和 `reduce` 操作可以运行多宽。
    *   **调度策略**：选择**流水线化**或**流式处理**。
    *   **缓冲区大小**：可配置范围，编译器会据此探索最佳实现。
*   **内存银行管理**：当需要**并行访问同一内存单元**时，编译器**自动处理**内存**复制**或**合理安排内存银行**。

### 2. 实例：构建空间加速器计算内积

*   **C 代码逻辑**：
  ```c
  int result = 0;
  for (int i = 0; i < N; i++) {
      result += vec_one[i] * vec_two[i];
  }
  ```

*   **加速器设计步骤**：
    1.  **数据移动基础：DMA**：使用 **DMA** 高效地在主内存与加速器之间传输数据。加速器内部定义**两个 SRAM 块**作为数据落脚点。
    2.  **循环结构改造**：从单层循环改为**双重嵌套循环**：
        *   **外层循环**：遍历所有**瓷砖（Tile）**。
        *   **内层循环**：在单个瓷砖内计算**部分内积**。

### 3. 数据分块（Tiling）的核心原理

*   **核心概念**：**数据分块**与**空间局部性**
    *   **避免“单元素”访问：** 直接从DRAM读取单个元素效率极低。
    *   **类比“超市购物”：** 去一次超市（DRAM访问）很“贵”，所以应该一次性批量采购（读取整个Tile），放进冰箱（片上缓存）。
    *   **显式数据移动：** 在加速器/DSP中，你可以**显式编程数据移动**。

*   **三步式Tile处理流程**：
    1.  **加载**：从 `vec_one` 和 `vec_two` 中加载一个Tile大小的数据到本地缓冲区。
    2.  **Tile内缩减**：对加载到本地的两个Tile中的元素进行**元素级乘法**，并将所有乘积累加成一个**局部和**。
    3.  **跨Tile缩减**：将所有Tile计算出的局部和累加起来，得到最终的点积结果。

### 4. 并行化策略

*   **并行性源一：流水线**：在“加载”、“计算”、“累加”三个步骤之间构建**流水线**。
*   **并行性源二：Tile间并行**：对**不同的Tile**并行执行三步流程。
*   **并行性源三：Tile内并行** — **核心优化点**
    *   **本质：** 对乘法与缩减进行并行化。
    *   **最佳方法：使用SIMD指令或向量单元。**
        *   **并行乘法：** 多条乘法指令同时执行。
        *   **并行缩减树：** 使用加法器树，将局部N个乘积并行累加。

## 十、空间程序员的优化三要素与流式执行模型

> 主题：空间程序员的优化三要素（并行化、数据局部性、流水线），以及流式执行模型的核心概念与优势。

### 1. 空间程序员的优化三要素

*   **减少树与硬件资源的权衡**
    *   **更宽的树**：意味着更高的并行性。
    *   **代价**：需要**更多硬件资源**。
    *   **控制点**：开发者可以主动控制树的宽度。

*   **控制 Tile 大小与数据局部性**
    *   **Tile 大小**决定了每次内存访问时获取的数据量。
    *   **优化目标**：通过调整 Tile 大小，优化数据在内存层次结构中的复用率。

*   **流水线化与重叠执行**
    *   **核心思想**：通过**流水线调度**，将外层循环的减少操作重叠执行。
    *   **关键条件**：需要**双缓冲**，即第一阶段在处理当前数据时，第二阶段必须能够处理来自前一次执行的数据。

### 2. 空间程序员的职责与编译器的分工

**空间程序员的责任：**
*   **算法表达**：使用 `for each` 和 `reduce` 等高级构造。
*   **内存层次管理**：**显式地**指定数据应该放在不同层级的内存中。
*   **数据移动**：负责显式的数据搬运。
*   **并行性与调度控制**：指定并行度、Tile大小和流水线调度策略。

**编译器的责任：**
*   **低级细节**：处理内存的银行分配和缓冲管理。
*   **生成配置**：针对特定目标硬件生成最优的低级配置代码。

### 3. 流式执行模型：核心概念与优势

*   **核心思想**：流式执行模型的核心在于**将计算拆分为多个独立的内核**，并通过**流水线**的方式让它们并行执行。
*   **机制**：每个内核的计算可以**重叠**，即当第一个内核在处理一个数据块时，第二个内核可以同时处理上一个数据块的结果。

*   **流式实现的主要优势**：
    *   **优势一：额外并行性与性能**：通过重叠和管道化不同输出块的计算，流式实现能挖掘出**传统融合内核无法获得的额外并行度**。
    *   **优势二：隐式融合，无需显式创建融合内核**：在流式模型中，你**不必**手动编写一个庞大的、耦合的**融合内核**。内核可以保持**独立实现**，只要使用 FIFO 或缓冲区进行连接，操作会自动融合执行。

### 4. 流式处理的时空语义与内存优化

*   **核心概念：生产者-消费者模式与FIFO管道**
    *   两个`for each`循环之间通过**FIFO**连接，生产者将计算结果压入FIFO，消费者从FIFO取出数据继续处理。

*   **流式处理 vs 朴素实现**

| 特性 | 朴素实现（物质化） | 流式处理 |
|------|-------------------|-----------|
| **内存占用** | 物质化整个 `N * N` 矩阵 | 仅需 **两个元素的FIFO** |
| **数据移动** | 所有数据同时存在并移动 | 数据在FIFO间流水线式移动 |
| **并行性** | 需等待整个矩阵生成 | 可**重叠**生产者和消费者的计算 |

*   **Flash Attention 优化**：通过**重新排列操作**、使用**运行总和**和**缩放**来进一步降低FIFO内存需求。
    *   **运行总和**：在流式处理过程中，维护部分结果的累计和。
    *   **缩放因子**：当新数据到达时，使用缩放因子调整已有累计值。

### 5. 流式执行模型与加速器设计精要

*   **核心执行模型：流式与并行**
    *   **流式执行模型**：一种非传统的执行方式，其核心思想是 **“数据流驱动计算”**。
    *   **优势**：**隐式并行**、**融合与填充**。

*   **加速器的核心优势：能源效率**
    *   **定量范围**：典型的改进幅度为 **100倍到1000倍**。
    *   **根本原因**：加速器是**为特定应用场景专门设计的**，砍掉了所有不必要的通用性开销。

*   **加速器设计的关键方法论**：
    1.  **深刻理解你的应用程序**：瓶颈在哪里？是**内存受限**还是**计算受限**？
    2.  **识别并利用并行性与局部性**：找出应用程序所展示的**特定类型的并行性**，设计**内存层次结构**确保最大的局部性。
    3.  **探索设计空间，进行权衡**：将不同的设计决策看作一个多维空间，寻找最优解。
