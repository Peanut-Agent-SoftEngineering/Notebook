# MIT_6.824_p14

## 一、FaRM 系统概述与设计哲学

> 主题：FaRM 是一个利用 RDMA 实现极速分布式事务的研究原型系统，与 Spanner 形成鲜明对比。

### 核心定位：Spanner 的对立面

- **设计哲学对比：**
    - **Spanner**：一个成熟的**部署系统**，核心目标是**地理分布**（Geo-replication），即在跨洲际（如东西海岸）的不同数据中心间复制数据。其创新在于利用**同步时钟**来优化**只读事务**。性能：读写事务延迟在 **10 到 100 毫秒**。
    - **FaRM**：一个**研究原型**，核心目标是**探索新硬件（RDMA）的极限性能**。
        - **假设**：所有副本都在**同一个数据中心**内。
        - **容错目标**：仅针对**单节点崩溃**或整个机房断电后的**恢复**，而非应对跨数据中心灾难。
        - **性能**：简单事务延迟可达 **58 微秒**（微秒 vs 毫秒，性能差距千倍以上）。

### 核心差异化设计：瓶颈与性能

- **核心概念：延迟与性能对比**
  - **FaRM** 处理一个简单事务仅需 **58 微秒**。
  - **Spanner** 处理同样的事务则需要大约 **10 毫秒**。
  - **关键结论**：FaRM 比 Spanner 快约 **100 倍**。
  - **根本原因**：两者针对的瓶颈完全不同。

- **核心概念：设计瓶颈定位**
  - **Spanner 的瓶颈**：**光速与网络延迟**。其设计主要担忧是跨数据中心的物理距离导致的延迟。
  - **FaRM 的瓶颈**：**服务器 CPU 时间**。通过将所有副本放置在**同一个数据中心**，FaRM 从架构上消除了光速和网络延迟的制约。

### 核心特性：面向极致性能的“精简”设计

*   **核心概念：** FaRM 的设计哲学是 **“极尽精简，只为速度”**。它不是通用的分布式数据库，而是针对特定场景（内存计算）进行深度优化的专用事务系统。

## 二、FaRM 系统架构与数据分布

> 主题：FaRM 采用单数据中心部署，通过分片和主备复制实现数据分布与容错。

### 核心架构设计：性能与持久性的权衡

#### 1. 单数据中心部署

- **核心概念：单数据中心部署**
  - FaRM 的所有副本都运行在 **一个数据中心内**。
  - 使用 **配置管理器 (Configuration Manager)** 来决定每个数据分片 (Shard) 的 Primary 和 Backup。
  - 依靠 **ZooKeeper** 来实现配置管理器，但这并非论文核心。

#### 2. 分片与主备复制

- **核心概念：分片与主备复制**
  - 数据按 **键 (Key)** 进行 **分片 (Sharding)**，分布在多个 Primary-Backup 对上。
  - 每个分片有一个 **Primary** 和一个或多个 **Backup** 副本。
  - **写操作**：必须同时更新 **Primary 和所有 Backup**。
  - **读操作**：**始终从 Primary 读取**。

### 容错机制与设计哲学

- **核心概念：故障容忍度**
  - **目标**：提供 **容错 (Fault Tolerance)** 而非强一致性共识（如 Paxos）。
  - **生存条件**：对于每个分片，只要 **至少有一个副本存活**，该分片就可用。
  - **系统级恢复**：即使整个数据中心发生断电，只要系统中**每个分片都至少有一个活着的副本**，系统就能恢复。

- **核心概念：F+1 副本策略**
  - **公式**：如果系统有 **F + 1 个副本**，则可以容忍 **最多 F 个副本故障**。
  - **关键区别**：这种容错不需要“多数派”（Majority）存活，仅需一个副本，代价是放弃了跨数据中心的地理分布能力。

### 托管内存与对象版本控制

#### 1. 区域（Region）与对象定位机制
*   **核心概念：** 全局物理内存被划分为多个**区域（Region）**，每个区域是数据复制的独立单元。
*   **配置管理器（Configuration Manager）：** 负责追踪每个区域由哪些服务器（主服务器与备份服务器）进行复制。客户端通过**区域编号（Region Number）** 在配置表中查询该区域当前的**主服务器（Primary）** 和**备份服务器（Backup）** 地址。
*   **对象寻址：** 客户端持有**区域编号**和该区域内的**内存地址（Straight Memory Address）**。它先通过区域编号选择主服务器，然后将地址交给 **RDMA 网卡（RNIC）**，指令为：“请读取此地址上的数据，以获取该对象”。

#### 2. 服务器内存布局与版本化对象
*   **核心架构：** 服务器内存中，每个被复制的区域包含大量**版本化对象（Versioned Objects）**。每个对象**同时只维护一个版本**（即最新版本）。

*   **对象头部结构（Header）：**
    *   **锁标志位（Lock Flag）：** 位于版本号的**最高位**（high bit）。用于并发控制。
    *   **版本号（Version Number）：** 除最高位外的其余位。每次对象被**写操作（Write）** 修改时，版本号递增。

*   **对象数据区：** 跟随在头部之后，存储实际的数据负载。

    > **示例：**
    > 对象布局 = `[锁标志(1bit) | 版本号(N bits)]` + `[对象数据]`

#### 3. 消息队列（Message Queues）与日志（Logs）
*   **核心用途：** 为了实现服务器间的高效通信（通过RDMA），每个服务器内存中都存放着**一对消息队列**（用于接收和发送）和**日志（Log）**，针对系统中的**每一个其他计算机**。
    *   **数量关系：** 如果一个集群有 `N` 个可运行事务的服务器，每个服务器内存中都有 `N-1` 个这样的日志结构（排除自身）。
*   **工作方式：** 当计算机 A 上的事务代码需要与计算机 B 通信时，它通过 RDMA **直接追加（append）** 数据到计算机 B 内存中为该计算机 A 预留的日志中。这避免了传统的 CPU 中断开销。

## 三、高性能的三大核心技巧

> 主题：FaRM 通过分片并行、全内存存储和 RDMA/内核旁路三大技术实现极致性能。

### 1. 分片并行（Sharding）

*   **目标**：这是论文实现“高事务性能”的首要手段。
*   **实现**：将数据分散到大量分片上（实验中多达**90个**或**45个**服务器）。
*   **原理**：只要不同分片上的操作是**相互独立**的，就能获得接近**90倍的自动加速**，因为所有操作可以在90台服务器上并行运行。

### 2. 全内存存储（All-in-RAM）

*   **核心**：所有数据必须**完全驻留在服务器的RAM**中，不使用磁盘持久化。
*   **目的**：RAM的读写速度远超磁盘，这是实现低延迟、高吞吐的关键。
*   **挑战**：RAM在断电后数据会丢失。
*   **解决方案**：采用**非易失性RAM（Non-Volatile RAM）** 方案来应对电力故障。
    *   *核心思想*：一种巧妙的技术，使得RAM中的数据在掉电后依然能恢复，从而保全性能优势，同时解决持久化问题。

### 3. RDMA 与内核旁路（Kernel Bypass）

*   **RDMA（远程直接内存访问）**：
    *   **定义**：一种智能网络接口卡（NIC）技术。
    *   **工作机制**：允许网卡直接读取或写入服务器的**内存**，而**无需中断服务器CPU的处理**。
*   **内核旁路（Kernel Bypass）**：
    *   **定义**：应用程序代码可以直接访问网络接口卡，**无需经过操作系统内核**。
    *   **优势**：消除了内核上下文切换和协议栈处理的巨大开销，显著降低网络延迟和CPU占用。

### 总结：系统性能配方

> 高性能 = **分片并行** + **全内存存储** + **RDMA/内核旁路**

## 四、非易失性 RAM (NVRAM) 方案

> 主题：FaRM 通过机柜级电池供电和快速 SSD 转储，将普通 RAM 模拟为非易失性内存，以消除持久化瓶颈。

### 为何使用RAM而非磁盘？

- **核心概念**：**所有数据更新都在RAM中完成**，绝不持久化到磁盘。这是实现极限性能的基础。
- **性能对比（延迟）**：
    - **写RAM**：约 **200 纳秒**
    - **写固态硬盘 (SSD)**：约 **100 微秒** (慢约500倍)
    - **写机械硬盘 (HDD)**：约 **10 毫秒** (慢约50,000倍)
- **结论**：*使用RAM进行写入带来了数个数量级的性能优势*，这是事务处理速度远超传统持久化方案（如Raft）的原因。

### 多副本在RAM中为何不足以保障持久性？

- **核心概念**：**故障独立性假设**。分布式系统通过多副本（F+1个副本可容忍F个故障）来保证数据安全。
- **致命问题**：**站点级故障 (Site-wide Failure)**。例如整个数据中心的断电，会导致所有服务器同时宕机，违背了“故障是独立的”这一基本假设。
- **结论**：*仅仅让数据在多个服务器的RAM中复制，无法应对数据中心级的灾难*。

### 解决方案：机柜级电池供电 (Battery-Powered Racks)

- **核心概念**：**物理层持久化方案**。通过为每个机柜配备大容量电池，将电源系统设计为电池不间断供电。
- **工作原理**：
    - 正常时，使用市电并为电池充电。
    - 断电时，**电池自动接管**，维持机柜内所有服务器继续运行。
- **时间窗口**：电池通常只能维持约 **10分钟** 的供电。
- **设计意图**：*这10分钟对于持久化整个RAM状态（例如，刷新到后备磁盘）已经足够，或者是为了等待发电机启动。它提供了一个短暂的“电源稳定窗口”来保证数据最终能安全写入，而非让机器无限期运行。*

### NVRAM 方案：通过电池与磁盘实现内存的非易失性

#### 核心目标：让普通RAM能在断电后“记住”数据

- **核心问题**：服务器使用普通的易失性RAM（Volatile RAM），一旦断电，所有数据都会丢失。
- **解决方案：** 通过一个**电池备份系统**和**快速固态硬盘（SSD）**的组合，模拟出非易失性RAM的效果。目标是在断电时，能将内存数据保存下来，供电恢复后使用。

#### 工作流程：两步走，断电时自动保存

1.  **电池紧急供电**：
    - 主电源故障时，**电池系统**立即接管，为所有服务器供电。
    - 但这个电池容量有限（如10分钟），无法支撑长时间运行。

2.  **内存数据快速转储**：
    - 电池开始供电的同时，向**所有服务器发送中断信号或消息**：“主电源故障，10分钟后电池耗尽，准备数据保存。”
    - 服务器软件收到警报后，立即执行**紧急流程**：
        - **停止所有处理**（停止渲染任务）。
        - **将完整的RAM内容拷贝到本地的固态硬盘（SSD）上**。这个过程很快，只需几分钟。
    - 数据拷贝完成后，服务器自动**关机**。

#### 恢复流程：重启后从磁盘恢复

- 数据中心电力恢复后，所有服务器启动。
- 在启动过程中，服务器会**读取之前保存在SSD上的内存镜像（Memory Image）**，并将其恢复到RAM中。
- 经过一些必要的恢复操作后，**服务器的持久状态完全没有丢失**。

#### 关键术语与公式化表达

- **核心概念**：**NVRAM（Non-Volatile RAM，非易失性RAM）**。本质上还是普通RAM，但通过**电池系统**和**SSD**执行`Store RAM to SSD`操作，达到了非易失性的效果。
- **算法/流程**：
    ```
    if (主电源故障 == true) {
        电池系统.启动();
        for each 服务器 in 农场 {
            服务器.发送警报(“保存RAM”)；
            服务器.停止处理所有任务()；
            SSD.写入(服务器.RAM)；
            服务器.关机()；
        }
        wait for 电力恢复；
        for each 服务器 in 农场 {
            if (服务器.SSD.存在内存镜像) {
                服务器.RAM = SSD.读取(内存镜像)；
                服务器.执行恢复流程()；
            }
        }
    }
    ```

#### 重要限制：此方案只对电力故障有效

- **局限性**：**NVRAM方案仅能抵御电力故障**。
- **无法解决的问题**：**硬件故障**（如内存条物理损坏）、**软件Bug导致的崩溃**等。
    - 这类故障会导致服务器直接重启，**RAM内容丢失**。
    - 由于故障发生时，没有电源失效事件来触发电池系统，所以**内存数据无法被保存到SSD**。
- **关键公式**：
    - **NVRAM 保护范围** = {电力故障}
    - **NVRAM 保护范围** ≠ {硬件故障, 软件崩溃, 操作系统错误}

#### 总结：双保险策略

正因为NVRAM方案的保护能力有限，所以农场才会**在NVRAM之外，还采用了“每个分片（shard）都有多个副本”的策略**。NVRAM处理电力故障，而副本处理其他类型的故障。这个NVRAM方案的核心价值在于，它**消除了持久化写入作为渲染管线瓶颈**的问题——因为RAM的读写速度远快于任何传统磁盘，系统可以在内存中频繁更新状态，而不用立刻写入慢速的磁盘，因为断电时数据能通过这套机制保全。

## 五、传统网络通信栈的瓶颈分析

> 主题：传统 RPC 架构中，系统调用、中断处理和多次数据拷贝是主要性能瓶颈，软件处理开销远大于物理网络传输速度。

### 传统RPC架构的性能瓶颈

#### 传统数据包处理流程

**核心概念**：传统远程过程调用（RPC）的数据包从应用程序到网络接口需要经过**用户态-内核态边界**和**多层协议栈**，造成显著性能开销。

- **处理路径**：应用程序（用户态）→ 系统调用（用户/内核边界）→ **Socket层** → 内核网络栈 → 网卡
- **主要瓶颈**：
  - **系统调用开销**：每次用户态与内核态切换都需要上下文切换，成本高昂
  - **多层协议处理**：TCP/IP协议栈在数据链路层、网络层、传输层的处理消耗大量CPU时间
  - **数据拷贝**：数据在用户缓冲区和内核缓冲区之间需要多次拷贝

### 一次RPC请求的完整路径

当应用程序发送一条消息（例如RPC请求）时，数据需要穿越**整个软件与硬件堆栈**，这是一个漫长且代价高昂的过程。

#### 发送端路径（Application → Wire）

1. **系统调用（System Call）**：
   - 应用程序通过**系统调用**进入内核态
   - 系统调用本身是**昂贵的操作**（上下文切换）

2. **Socket层 - 缓冲与拷贝**：
   - 内核中的 **Socket层** 负责**缓冲管理**
   - 关键代价：**数据拷贝**（从用户空间拷贝到内核缓冲区）

3. **TCP协议栈**：
   - 复杂的 **TCP协议栈** 处理大量工作：
     - **重传（Retransmission）**
     - **序列号（Sequence Numbers）**
     - **校验和（Checksums）**
     - **流量控制（Flow Control）**
   - 所有这些处理都需要**大量的CPU计算**

4. **网络接口卡驱动（NIC Driver）**：
   - 内核中的 **NIC驱动** 与硬件通信
   - 通过**寄存器**配置网卡

5. **DMA与队列**：
   - **直接内存访问（DMA）**：网卡直接从主机内存搬移数据包
   - 关键数据结构：**数据包队列**
     - **入站队列**（Rx Queue）：DMA写入内存，等待内核读取
     - **出站队列**（Tx Queue）：内核放入，网卡择机发送

6. **物理发送**：
   - 网卡将**比特流**通过**电缆**发送到网络

#### 接收端路径（Wire → Application）

接收端执行**逆向栈操作**，同样代价高昂：

1. **NIC硬件**接收比特流
2. **中断机制**：网卡通过**中断**通知内核
3. **驱动代码**运行，处理数据包
4. **TCP协议栈**处理数据包
5. **写入缓冲区**，等待应用程序读取
6. **应用程序**调用**系统调用**，将数据从内核缓冲区**拷贝到用户空间**

### 核心瓶颈总结

#### 三个主要性能杀手

| 瓶颈类型 | 具体代价 | 影响 |
|---------|---------|------|
| **系统调用** | 用户态/内核态切换 | 每次调用具有固定开销 |
| **中断处理** | CPU被强制打断 | 干扰指令流水线 |
| **数据拷贝** | 数据在内存中多次搬移 | 耗费内存带宽和CPU周期 |

#### 性能数据

- **传统架构RPC性能上限**：约 **几十万次/秒**（few hundred thousand RPC messages per second）
- **看似很多，实则不足**：这个数字**远低于**目标系统（如FARM）所需的性能
- **关键差距**：传统软件的RPC速度**远远低于**底层网络硬件的物理极限

### 网络硬件能力 vs 软件瓶颈

- **典型网络速率**：**10 Gbps（千兆位/秒）**
- **软件瓶颈**：生成**小消息**（数据库常用）的RPC软件，**极难**匹配硬件速度
- **根因**：软件栈的**处理开销**远大于物理传输延迟

### 关键术语与概念速查表

| 术语 | 解释 |
|------|------|
| **系统调用（System Call）** | 应用程序请求内核服务的机制，代价高昂 |
| **Socket层** | 内核中负责网络通信抽象和缓冲的中间层 |
| **TCP协议栈** | 提供可靠传输的复杂协议，包含重传/校验/流控 |
| **NIC（网络接口卡）** | 物理网络硬件，负责发送/接收比特流 |
| **DMA（直接内存访问）** | 硬件无需CPU参与，直接读写主机内存 |
| **数据包队列** | NIC用于DMA操作的入/出站环形缓冲区 |
| **中断（Interrupt）** | 硬件通知CPU的机制，会打断当前执行流 |

### 核心启示

> **经典网络通信栈中，软件处理开销（系统调用、中断、数据拷贝）是性能瓶颈的主要来源，而非物理网络的传输速度。**

这解释了为什么现代高性能系统（如FARM）必须**彻底重新设计**网络通信架构，绕过传统内核协议栈，才能达到**千万级甚至亿级**RPC吞吐量。

## 六、内核旁路（Kernel Bypass）与 RDMA

> 主题：内核旁路允许应用程序直接与 NIC 交互，RDMA 则进一步实现远程内存的直接读写，两者共同消除了传统网络栈的 CPU 开销。

### 内核旁路（Kernel Bypass）与高性能网络

#### 1. 传统网络栈的性能瓶颈

**核心概念**：传统的操作系统网络协议栈（如TCP/IP）是 **高吞吐、低延迟** 网络应用（如数据库）的主要性能瓶颈。

- **问题根源**：
  - 现代网卡（NIC）的带宽极高（例如 **10 Gbps**）。
  - 为了利用这样的带宽，软件需要能够 **每秒生成或消费数百万甚至数千万条小消息**。
  - 传统做法是：应用程序将数据向下传递到 **复杂的内核网络协议栈**。
  - 这个过程涉及 **系统调用、数据拷贝、中断** 和复杂的上下文切换，导致巨大的开销，使得软件很难达到网卡的吞吐能力。
  - 简言之：**硬件太快，传统软件架构跟不上**。

#### 2. 解决方案：内核旁路（Kernel Bypass）

**核心概念**：绕过操作系统内核，让应用程序 **直接且安全地** 与网络接口卡（NIC）交互，以消除内核层带来的性能损失。

**关键机制**：

- **直接硬件访问**：
  - 内核仅负责 **一次性的配置**（配置计算机的保护机制），授权应用程序直接访问NIC的寄存器。
  - 之后，应用程序可以直接通过 **读写NIC寄存器** 来命令它发送或接收数据包，无需每次操作都陷入内核。

- **直接内存访问（DMA）到应用空间**：
  - **接收**：NIC通过DMA技术，将网络数据直接写入 **应用程序的私有内存空间**。应用程序可以立即看到数据，内核完全不知情、无干预。
  - **发送**：应用程序在自身内存中创建 **发送队列（Queues）**。NIC可以直接通过DMA读取这些队列并发送到网络。

- **彻底消除内核介入**：
  - **无系统调用**：发送/接收数据不调用 `send()`/`recv()`。
  - **无中断**：数据到达时，内核不会触发中断来通知应用。
  - **结果**：所有与网络相关的内核代码都被绕过了。

> **核心好处**：应用程序和NIC之间建立了一条 **“直通”** 的数据通路，延迟极低，吞吐量极高。

#### 3. 实施方式与权衡

- **现代网卡支持**：此技术需要 **现代且功能强大的网卡** 的硬件支持，旧网卡无法实现。
- **常用工具包**：**DPDK（Data Plane Development Kit）** 是一个流行的开源工具包，可以很方便地实现内核旁路，构建高性能网络应用。
- **代价和挑战**：**用户态需要接管网络协议**。
  - 通常情况下，操作系统内核为你处理了复杂的逻辑，例如：**计算校验和、处理重传、流控** 等。
  - 当你使用内核旁路时，**应用程序必须自己实现或依赖用户态的协议栈库** 来完成这些工作。这增加了应用开发的复杂性。

#### 4. 对操作系统的影响（以FAQ形式解答）

- **疑问**：这是否需要运行一个修改过的操作系统？
- **解答**：**通常不需要**。
  - 这是一种 **基于标准操作系统** 但 **突破其默认功能** 的优化技术。
  - 在现代 **Linux** 中，已经有了对内核旁路技术的 **完整支持**（例如通过DPDK或VFIO）。
  - 对于其他系统（如示例中提到的Windows），也无需从根本上修改操作系统内核，它更多是网卡驱动和用户态库层面的协同工作。

### 绕过操作系统与远程直接内存访问（RDMA）

#### 一、内核绕过（Kernel Bypass）的现状

- **核心概念**
  - **内核绕过**是一种允许应用程序直接与硬件设备通信的技术，无需经过操作系统的内核层
  - 普通应用程序代码**不能直接操作设备**，因此需要**内核的授权和干预**

- **操作系统支持情况**
  - **Windows系统**：渲染农场运行在某种形式的Windows上（不确定是否修改了Windows内核）
  - **Linux系统**：已经提供了**完整的内核绕过支持**
  - **关键条件**：Linux内核已修改，允许**将硬件访问权限委托给应用程序**

- **硬件层面的配合**
  - **现代网卡（NIC）** 具备智能特性，知道如何与**多个不同的队列（queues）** 通信
  - 支持**多应用程序同时使用**，每个应用程序拥有**自己独立的队列集合**
  - **重要改动**：涉及多层次的修改（操作系统、驱动、硬件固件等）

#### 二、RDMA（远程直接内存访问）技术详解

- **核心概念**
  - **RDMA = Remote Direct Memory Access**（远程直接内存访问）
  - 是一种**特殊的网卡（NIC）** 支持的通信方式，**目前不是默认配置**，但已可商业购买

- **硬件要求**
  - 通信双方都需要**安装支持RDMA的专用网卡**
  - 通过**交换机**连接多个服务器，实现任意两台服务器之间的通信
  - RDMA网卡内部包含**硬件和固件**，负责执行内存读写操作

- **工作流程**
  1. **源主机应用程序**发送**特殊请求**到RDMA系统
  2. **RDMA网卡**接收请求后，**直接读取或写入**目标应用程序地址空间中的内存（通常是一个**缓存行（cache line）** 大小）
  3. **返回结果**到源应用程序的**接收队列**
  4. **关键特性**：目标主机上的CPU完全**不知晓**这次读/写操作

- **核心优势**

| 传统方式 | RDMA方式 |
|---------|---------|
| 需要CPU参与中断处理 | **CPU完全绕过** |
| 涉及多次数据拷贝 | **直接内存访问** |
| 造成CPU负载和延迟 | **零CPU开销** |
| 需要操作系统介入 | **完全在网卡固件中执行** |

- **性能特征**
  - **无中断（no interrupts）**：网卡自主完成所有操作
  - **低延迟**：跳过CPU和操作系统层
  - **高吞吐量**：直接硬件到硬件的数据传输

### RDMA 核心概念与工作机制

#### RDMA 的本质：CPU 零参与的内存读写
- **核心概念**：RDMA 允许一台计算机直接读写另一台计算机的**内存**，而**完全不需要目标机器的 CPU、操作系统或应用程序参与**。
- **关键术语**：
    - **One-sided RDMA (单侧RDMA)**：指应用程序像读写本地内存一样，直接读写远程机器的内存。这是 RDMA 最纯正、最高效的形态。
    - **Firmware-based Execution**：整个读写操作全部由**网卡 (NIC) 的固件**完成，不涉及任何中断、系统调用或内核上下文切换。
    - **Kernel Bypass (内核旁路)**：RDMA 通常与内核旁路结合使用。如果请求仍需经过内核，会**严重损失性能**。因此，几乎所有高性能 RDMA 实现都依赖内核旁路。

#### 性能优势：为何比传统 RPC 快得多
- **极低的开销**：
    - 传统 RPC 需要发送方发起系统调用、内核处理、网络传输，接收方内核中断、应用程序响应。RDMA 省去了所有这些步骤。
    - **操作简化**：应用只需对远程内存发起 `read` 或 `write`，网卡固件完成后直接返回结果，**无 CPU 负载**。
    - 对于简单的内存读写，RDMA 比 “带魔法内核旁路” 的 RPC 还要快得多。

#### 可靠性保证：RDMA 如何不牺牲 TCP 的优势
- **核心问题**：TCP 提供可靠、有序、无重复的数据传输。RDMA 如果只是 “裸读写”，会丢失这些关键性质。
- **解决方案**：
    - **NIC 内部的可靠序列协议**：RDMA 网卡之间运行自己的**可靠、有序、带确认 (ACK) 的私有协议**（类似于 TCP，但非 TCP）。它处理丢包重传、重复检测、顺序保证。
    - **确认机制**：发起 RDMA 操作的应用会收到明确的**成功/失败应答**，确保读写操作是可靠完成的。
- **前提条件**：该机制仅适用于**局域网 (Local Network)**。由于依赖极低延迟和短距离通信，RDMA 不适合跨数据中心部署。

#### FA RM 的混合使用策略
- **直接使用 One-sided RDMA**：在某些场景下，FA RM 直接发起 RDMA 读写，获得最低延迟。
- **将 RDMA 作为消息传递通道**：在其他场景下，FA RM 把 RDMA 当作一个**消息追加 (append) 工具**，用于实现类似 RPC 的消息协议，而不是单纯的内存读写。这保留了 RDMA 的高效性，同时适应更复杂的通信模式。

### 远程直接内存访问 (RDMA) 的核心原理与工程应用

#### 核心概念：RDMA 如何在高性能系统中运作

- **RDMA (Remote Direct Memory Access)**：一种允许一台计算机**直接读写另一台计算机内存**的技术，无需目标端CPU介入。
- **关键机制**：
  - **单边 (One-Sided) 操作**：客户端直接访问服务器内存，**绕过目标CPU**，实现极低延迟。
  - **消息队列 (Message Queue)**：FARM系统通过RDMA将消息**追加**到目标端的传入队列中，目标端通过**轮询 (Polling)** 检查队列是否有新消息**（无中断机制）**。
  - **日志追加 (Log Appending)**：FARM也利用RDMA将日志条目写入另一端服务器的**持久化内存**（非易失性内存）。

#### 性能数据与优势

- **吞吐量**：**每秒1000万次**小尺寸RDMA读/写操作（远快于TCP/IP的RPC）。
- **延迟**：简单RDMA读/写操作**约5微秒**（比本地内存访问慢，但比任何传统网络传输快得多）。
- **非易失性内存 (NVM)**：所有写入（消息队列、日志）**直接持久化到磁盘**，断电不丢失。

#### 工程愿景：完全基于RDMA构建系统

- **理想目标**：客户端通过**单边RDMA读写**直接操作数据库服务器内存中的数据，**完全绕过数据库服务器的CPU和软件栈**。
- **核心挑战**：能否在**仅使用单边RDMA**（不依赖服务器端软件解释消息）的情况下，实现**事务 (Transactions)** 的读写操作？

#### 关键术语与总结

| 术语 | 说明 |
| :--- | :--- |
| **One-Sided RDMA** | 无需目标端CPU参与的直接内存访问。 |
| **Polling (轮询)** | 目标端周期检查队列或内存区域，以发现新消息。 |
| **NVM (非易失性内存)** | 断电后数据不丢失的内存，用于持久化消息队列和日志。 |
| **5微秒延迟** | RDMA操作的低延迟标杆，远低于传统网络。 |

> **一句话总结**：RDMA通过**直接内存访问+轮询检查**，实现了**微秒级延迟、千万级吞吐**的高效通信。它的终极幻想是**完全绕过服务器CPU**，仅用RDMA实现数据库事务。

## 七、乐观并发控制（OCC）与 RDMA 的结合

> 主题：FaRM 采用乐观并发控制（OCC）来桥接单边 RDMA 与传统事务协议之间的矛盾，实现无锁读取和提交时验证。

### 乐观并发控制与RDMA的结合：FaRM的设计哲学

#### 核心矛盾：传统事务协议与单边RDMA的不兼容

- **单边RDMA（One-sided RDMA）** 的核心特征是 **服务器CPU不参与** 数据读写操作，客户端直接通过网卡访问服务器内存。
- **传统事务+复制+分片协议**（如两阶段提交、Spanner的多版本并发控制）都要求 **服务器软件主动参与**：
  - 两阶段提交中，服务器需要检查并 **设置锁**（locking）、判断记录是否被锁定。
  - Spanner中，服务器负责 **版本管理**，找到最新或已提交的版本。
  - 这些操作依赖服务器CPU的逻辑判断，而单边RDMA无法直接触发服务器上的软件逻辑。
- **结论**：直接使用纯单边RDMA来实现 **带复制和分片的事务系统** 是 **不直接可行的**。

#### FaRM的回答：用乐观并发控制（OCC）桥接矛盾

- **悲观方案（Pessimistic Concurrency Control）** 使用 **锁（locks）** ，要求服务器在读写前主动检查并加锁，这与单边RDMA的“无服务器参与”特性冲突。
- **乐观并发控制（Optimistic Concurrency Control, OCC）** 假设 **冲突很少发生**，允许客户端 **直接通过RDMA读取数据**，不提前加锁，在提交阶段才验证冲突。
- **FaRM的关键技巧**：
  - **使用单边RDMA读**：客户端可以直接从服务器内存中 **读取数据副本**，无需服务器介入。
  - **不使用单边RDMA写**：因为写操作需要协调、版本控制、提交判断等逻辑，FaRM **不通过单边RDMA直接修改数据**。
  - 通过OCC，FaRM **绕过了服务器必须参与锁管理和版本判断的限制**，让事务系统与单向RDMA共存。

#### 关键术语与公式（无具体公式，但需理解逻辑结构）

- **单边RDMA（One-sided RDMA）**：允许客户端绕过CPU直接读写远程内存，但无法触发服务器协议逻辑。
- **乐观并发控制（OCC, Optimistic Concurrency Control）**：一种无锁并发控制策略，事务执行阶段不做检查，提交时验证数据是否被修改过。
- **事务+复制+分片的冲突**：传统协议（如2PC、多版本）依赖服务器软件，而RDMA要求硬件直通，两者本质矛盾。
- **FaRM的设计取舍**：
  - **读：RDMA直接读**（高效、无服务器负担）
  - **写：不使用RDMA直接写**（需要服务器协调、版本管理、提交验证）

### 乐观并发控制（Optimistic Concurrency Control）与RDMA

#### 并发控制的两大类别

- **悲观方案 (Pessimistic Schemes)**：使用锁机制
  - **核心概念**：在读取或写入数据前，**必须获取锁**
  - 事务持有锁直到提交(commit)或中止(abort)
  - 当多个事务冲突时（例如同时写入相同数据，或一个读一个写），**必须阻塞等待锁释放**
  - **开销**：需要跟踪锁的拥有者、释放时机等状态信息

- **乐观方案 (Optimistic Schemes)**：不使用锁
  - **核心概念**：无需获取锁即可直接读取数据，写入在本地缓冲

#### 悲观锁与RDMA的冲突

- **关键问题**：传统RDMA（如FaRM使用的）**无法直接实现基于锁的悲观方案**
  - 为什么？因为必须有某个实体强制实施锁机制
  - 单边的RDMA读/写操作缺乏必要的**原子性原语**（如test-and-set）
  - **未来可能性**：若RDMA网卡支持更丰富的原子操作（如test-and-set），理论上可以实现基于锁的方案
  - **FaRM的选择**：不采用悲观锁，而是使用**乐观方案**

#### 乐观方案的工作流程

1. **无锁读取**：直接读取数据，无需检查是否被其他事务修改
2. **缓冲写入 (Buffered Writes)**：写入操作不直接修改共享数据，而是**在客户端本地缓冲**
3. **事务提交阶段**：
   - **验证阶段 (Validation Stage)**：检查事务执行期间是否有其他事务修改了已读取的数据
   - **核心验证逻辑**：判断读取的数据是否一致（serializability），即“在我读取时，是否有人同时在写入？”
   - **成功条件**：若验证通过（无冲突），则提交事务
   - **失败处理**：若验证失败（读取到“垃圾”数据），事务中止

#### 乐观方案的核心优势

- **避免锁的开销**：无需管理锁状态，特别适合**RDMA这种低延迟、单边操作**的场景
- **读取不阻塞**：读取操作永远不等待，适合读密集型工作负载
- **写入延迟**：写入操作延迟到验证阶段才实际生效

#### 乐观方案的潜在风险

- **验证失败可能频繁**：在**高冲突**工作负载下，事务中止率会升高
- **浪费计算资源**：事务可能做了大量工作后才发现数据不一致

### 乐观并发控制（Optimistic Concurrency Control, OCC）

#### 核心思想：先执行，后验证
- **与传统悲观锁不同**，OCC 不假设冲突会发生。它允许事务在无锁情况下自由读取数据，直到提交时才检查冲突。
- **关键流程**：
    1.  **读取阶段**：事务直接读取数据，不获取任何锁（利用**单边RDMA**直接读远端内存，速度极快）。
    2.  **验证阶段**：在提交时，检查**读集**（读取的所有数据）
