- **实践指南（对 Lab 2 及以后）：**
    - **不要追求花哨的 CPU 性能优化**。
    - **不要使用细粒度锁（Fine-Grained Locking）** 等技巧。
    - **策略：** 使用**大锁（Big Locks）** 来保护**大的临界区**。
    - **核心原则：** **代码必须易于推理（Easy to Reason About）**。性能是次要的。

## 二、Goroutines 与闭包

> 主题：Goroutine 与闭包的配合使用模式，以及循环中变量捕获的常见陷阱与正确做法。

### Goroutines 与闭包 (Closures)

**核心概念：闭包是 Goroutine 的黄金搭档**

- **闭包 (Closure)**：一个匿名函数，可以访问并修改其**定义处所在作用域**的变量
  - 例如：Goroutine 内部可以直接修改外层定义的 `a` 变量
  - 可以引用外层的 `WaitGroup` 等同步原语

**典型模式：匿名 Goroutine + 闭包**
```go
func main() {
    var a int
    var wg sync.WaitGroup
    
    wg.Add(1)
    go func() {
        defer wg.Done()
        a = 42  // 闭包捕获并修改外层变量
    }()
    
    wg.Wait()
    fmt.Println(a)  // 输出: 42
}
```

### 循环中的 Goroutine 与变量捕获陷阱

**核心问题：标识符捕获 (Identifier Capture)**

- **陷阱场景**：在 `for` 循环中启动多个 Goroutine
- **根本原因**：循环变量 `i` **共享同一内存地址**
- **危险行为**：所有 Goroutine 最终都读取到循环结束后的值

**错误示例 - 变量被外层循环修改**
```go
for i := 0; i < n; i++ {
    go func() {
        // ⚠️ 所有 Goroutine 可能都看到 i 的最终值
        fmt.Println(i)  // 可能全部打印 n
    }()
}
```

**正确做法：通过参数传递拷贝值**
```go
for i := 0; i < n; i++ {
    go func(serverID int) {
        // ✅ 每个 Goroutine 拥有自己的拷贝
        fmt.Println(serverID)
    }(i)  // i 的值被拷贝传入
}
```

### 关键要点总结

1. **闭包强大但危险**：变量捕获可能引入难以调试的竞态条件
2. **循环 + Goroutine**：总是通过函数参数传递循环变量副本

## 三、周期性任务与安全终止模式

> 主题：使用 `for + time.Sleep` 实现周期性任务，并通过共享变量和互斥锁实现 Goroutine 的安全终止。

### 基础周期任务模式

**核心概念**：使用 `for` + `time.Sleep()` 实现简单周期性工作

```go
for {
    // 执行周期性工作
    time.Sleep(间隔时间)
}
```

- **无需复杂机制**，足够实现基本周期性操作
- 适用于无限循环的周期任务场景

### 带安全终止的周期任务模式

**核心概念**：引入共享变量 + 互斥锁，实现协程可控安全退出

#### 问题背景
当需要创建诸如 Raft 协议中的心跳协程时，必须能够优雅地终止这些协程（例如调用 `.Kill()` 方法时），防止后台协程泄漏。

#### 终止机制实现

**共享变量终止标志 + 互斥锁保护**

1.  **定义共享退出标志和锁**
    ```go
    var done bool
    var mu sync.Mutex
    ```

2.  **工作协程中检查终止条件**
    ```go
    go func() {
        for {
            // 执行工作...
            
            mu.Lock()
            if done {
                mu.Unlock()
                return  // 优雅退出
            }
            mu.Unlock()
            
            time.Sleep(间隔)
        }
    }()
    ```

3.  **控制线程触发终止**
    ```go
    func main() {
        // ... 运行一段时间后
        mu.Lock()
        done = true
        mu.Unlock()
    }
    ```

#### 核心原理：内存可见性保证
- **`sync.Mutex` 保证 happens-before 关系**
- 写入线程: `done = true` → `mu.Unlock()`
- 读取线程: `mu.Lock()` → 必然观察到之前的 `done = true` 写入

**关键理解**：Go 的同步原语（锁、channel、条件变量）**是跨线程通信的唯一可靠手段**，确保一个线程的写入能被另一个线程正确观察到。

### 简化版实现：Raft 实例终止模式

**核心概念**：将终止标志封装在 Raft 实例内部，简化代码

```go
func (rf *Raft) ticker() {
    for !rf.killed() {  // 持续检查是否被终止
        // 执行周期性工作
        time.Sleep(间隔)
    }
}
```

- `rf.killed()` 内部实现类似上述 `done` + 锁 的模式
- 更简洁，适用于 Lab 实现场景
- 所有活跃协程都能统一通过该方法判断是否退出

### 关键总结

| 组件 | 作用 | 备注 |
|------|------|------|
| **`done` 标志** | 终止指示器 | 共享变量，必须同步访问 |
| **`sync.Mutex`** | 保证跨线程可见性 | 写后解锁，读前加锁确保观察到写入 |
| **`rf.killed()`** | Raft 实例内置终止检查 | Lab 推荐用法 |
| **`return` 退出协程** | 协程安全终止 | 防止后台协程泄漏 |

**最佳实践**：在 Raft 等分布式系统中，通过实例级别的终止标志统一管理所有工作协程的生命周期，避免协程泄漏和资源浪费。

## 四、互斥锁（Mutex）与并发安全

> 主题：互斥锁的核心机制、使用模式以及锁的真正目的——保护不变量。

### 核心问题：并发访问共享数据

- **核心概念**：当多个线程（或协程）**并发读写**同一块**共享数据**时，必须保证这些读写操作是**原子（Atomic）**的。
- **竞态条件（Race Condition）**：如果不加保护，线程会相互覆盖对方的写入，导致结果丢失。
    - **Example**：启动1000个协程，每个都执行 `counter++`。直觉上最终结果应为1000，但由于更新操作非原子，实际可能只输出 `947`。
    - **底层原因**：`counter++` 实际上不是一步操作，而是三步骤：**读取 counter -> 计算 +1 -> 写回 counter**。多线程同时读取时，会读到同一个旧值，然后各自加1写回，导致只有一次加1生效。

### 解决方案：锁（Lock）与临界区（Critical Section）

- **术语定义**：
    - **锁/互斥锁（Mutex）**：一种同步原语，用于**保证互斥（Mutual Exclusion）**。
    - **临界区（Critical Section）**：由锁保护的代码块，保证**同一时间只有一个线程**能执行这段代码。
- **工作流程**：
    1.  访问共享数据前，**先加锁（Lock）**。
    2.  执行数据修改操作（读/写）。
    3.  操作完成后，**必须解锁（Unlock）**。
- **效果**：所有临界区内的操作被**串行化（Serialized）**，一个接一个执行，从而确保数据一致性。

### 实用技巧：`defer` 解锁模式

- **痛点**：在复杂函数中，如果提早返回或忘记编写 `Unlock`，会导致**死锁**。
- **Go 语言模式**：使用 `defer` 关键字将 `Unlock` 调度到函数返回前执行。
    - **示例代码结构**：
      ```go
      func someRPCHandler() {
          lock.Lock()
          defer lock.Unlock()  // 无论函数如何退出，都会执行解锁
          // ... 操作共享数据（如 Raft 结构体） ...
      }
      ```
- **优势**：减少人为错误，确保解锁操作一定会被执行，是**RPC Handler** 等场景的**标准编程规范**。

### 锁的核心概念：保护不变式

- **核心思想**：**锁保护的是不变式（invariant）**，而不仅仅是保护共享数据。
- **正确操作模式**：
  - **获取锁（Lock）** → **执行可能暂时破坏不变式的操作** → **恢复不变式** → **释放锁（Unlock）**
- **关键原则**：在锁释放前，必须确保不变式恢复原状，这样**其他线程永远不会观察到"进行中"的不一致状态**。
- **典型错误示例**：在每次共享变量访问前后分别加锁解锁，导致其他线程有机会在操作中间观察到错误状态。

### 正确写法示范

```
Lock
    // 执行一系列操作，可能暂时破坏不变式
    // 例如：更新 total 的多个步骤
    // 确保在 Unlock 前恢复不变式
Unlock
```

### 两种等价视角理解锁

1. **规则视角**：访问共享数据时必须加锁。
2. **抽象视角**：锁保护不变式 → 获取锁 → 操纵数据（允许暂时破坏不变式）→ 恢复不变式 → 释放锁。

### 锁与不变量的正确关系

| 错误认知 | 正确认知 |
|----------|----------|
| 锁保护共享变量本身 | 锁保护**变量相关的逻辑不变量** |
| 加锁即保证数据安全 | 需要**整个修改序列**才会保障安全 |
| 最终结果正确即可 | **中间状态也必须对其他线程不可见** |
| 关注"访问原子性" | 关注"**事务原子性**"（transactional atomicity） |

### 实践建议

- **设计锁的范围**：以**不变量边界**而非单个变量为粒度。
- **识别不变量**：编写并发代码前，明确哪些逻辑约束必须被保护。
- **检查原子性**：如果一段代码会临时破坏不变量，**必须将其整体包裹在同一个锁的临界区内**。
- **避免"弱观察"**：审计或监控线程应假设数据可能在任意时刻处于**合法不变量状态**，而非运行中的临时状态。

## 五、Go 内存模型与 Mutex 的深层必要性

> 主题：从 Go 内存模型的角度解释为什么必须使用同步原语来保护共享变量，以及编译器优化可能导致的意外行为。

### 核心概念：Go 的内存模型（Memory Model）

- **"happens-before" 关系**：Go 编译器拥有很大的优化自由度。它不会严格按照你代码的"直觉"顺序来生成机器码。
- **编译器优化的反直觉行为**：
    - 例如在 `for` 循环中读取共享变量 `done`，编译器可能会将其**提升（Lift）** 到循环外部。它只读取一次，如果 `done` 是 `false`，就直接编译成一个**死循环（infinite loop）**。
    - **结果**：即使另一个线程（Goroutine）修改了 `done` 为 `true`，这个死循环也永远看不到这个修改，因而程序不会终止。
- **无同步原语的后果**：如果一个 Goroutine 中没有任何 `Mutex` 锁、`Channel` 发送/接收等**同步原语**，Go 环境**不保证**该 Goroutine 能观察到其他并发线程（Goroutine）所做的任何修改。
- **关键结论**：对于**共享变量**的跨线程读写，**必须**在读写前后持有锁。这是避免内存模型带来的晦涩陷阱的通用规则。

### Goroutine 生命周期与主线程

- **主 Goroutine 的特殊性**：`main()` 函数运行在一个主 Goroutine 中。
- **程序终止规则**：如果**主 Goroutine 退出**，整个程序会立即终止，所有其他还在运行的 Goroutine 都会被强制杀死。

### 为什么需要 `Mutex`（高级视角）

- **摆脱直觉依赖**：不要用"顺序执行"的直觉去理解并发代码。编译器优化和 CPU 乱序执行会导致不可预测的结果。
- **遵循通用规则**：**"在读写共享变量之前，先获取锁；写完后，解锁"**。这是保证正确观测到共享变量修改的最简单、最安全的方法。
- **示例错误代码分析**：
    - 主线程：写入 `done = true`
    - 工作线程：在 `for` 循环中没有锁，只是读 `done`
    - **错误结果**：工作线程可能永远看不到 `done` 的修改，陷入死循环。即使代码看起来"应该"终止。

### 总结与最佳实践

- **不要试图绕开锁**：即使代码看起来简单，没有锁的共享变量访问在多线程下是**未定义行为**（在Go的严格内存模型下）。
- **锁的作用**：锁不仅仅是保护临界区，它还充当了**内存屏障**的作用，确保了"happens-before"关系的正确建立。
- **死循环的潜在原因**：若你发现一个看似能退出的循环根本不退出，首先检查**共享变量的访问是否被正确的同步原语（如 Mutex）保护**。

## 六、条件变量（Condition Variable）

> 主题：条件变量的设计动机、使用模式以及如何优雅地避免忙等待和魔法常量问题。

### 核心问题：忙等待（Busy Waiting）

#### 错误实现模式
```cpp
// 错误：忙等待 - 浪费100% CPU核心
while (true) {
    lock();
    if (election_determined) {
        unlock();
        break;
    }
    unlock();  // 立刻解锁再重新加锁
}
```

#### **关键问题分析**
- **正确性**：代码逻辑正确，能正常完成选举同步
- **效率灾难**：在**紧密循环**中不断加锁/解锁，**占用100% CPU核心**
- **性能反噬**：高CPU占用可能导致程序整体变慢，反而**阻碍进度推进**

### 朴素解决方案：硬编码等待

#### 改进版本
```cpp
while (true) {
    lock();
    if (election_determined) {
        unlock();
        break;
    }
    unlock();
    wait(50);  // 等待50毫秒
}
```

#### **核心缺陷：魔法常量（Magic Constant）**
- **为什么是50ms？** 无法给出合理理由
- **问题本质**：硬编码等待时间是**代码设计不纯**的标志
- **缺乏通用性**：不同场景需要不同等待时间，无法预测最佳值

### 正确解决方案：条件变量（Condition Variable）

#### **设计哲学**
- **问题本质**：需要**等待某个条件变为真**
- **正确做法**：使用专门为此设计的并发原语——**条件变量**

#### **条件变量的核心机制**
- **等待**：线程进入休眠，不消耗CPU
- **零CPU浪费**：只在条件真正变化时被唤醒

#### **使用模式**
```cpp
// 等待端
lock(mutex);
while (!condition) {
    wait(condition_variable, mutex);  // 自动释放锁并休眠
}
// 条件满足，继续执行
unlock(mutex);

// 通知端
lock(mutex);
condition = true;
notify(condition_variable);  // 唤醒等待线程
unlock(mutex);
```

### 条件变量的工作模式

**核心模式：** 条件变量必须与**互斥锁（mutex）**配对使用。它负责协调：
+ **修改线程：** 执行修改共享数据的操作，**可能使条件变为真**
+ **等待线程：** 等待共享数据上的某个条件成立

#### 典型流程分解

1. **等待线程（main thread）**
   ```cpp
   lock(mutex);            // 1. 获取锁
   while (!condition) {    // 2. 检查条件（必须用while！）
       cond.wait(mutex);   // 3. 原子操作：释放锁 + 挂起线程 + 加入等待队列
   }
   // 4. 条件成立后，自动重新获取锁，然后继续执行
   unlock(mutex);
   ```
   - `cond.wait()` 的**原子性**：保证在释放锁和挂起线程之间没有竞争窗口。
   - **必须用 `while` 循环**而不是 `if`，因为线程可能被**虚假唤醒（spurious wakeup）**。

2. **修改线程（worker threads）**
   ```cpp
   lock(mutex);            // 1. 获取锁
   modify_shared_data();   // 2. 修改数据，可能使条件变为真
   cond.broadcast();       // 3. 唤醒所有等待该条件的线程（仍持有锁！）
   unlock(mutex);          // 4. 释放锁
   ```
   - `broadcast()` 必须在**持有锁**时调用
   - `broadcast()` 唤醒所有等待者，而不是仅一个（`signal()` 则唤醒一个）

### 与忙等待的对比

| 特性 | 忙等待 | 硬编码等待 | 条件变量 |
|------|--------|------------|----------|
| CPU占用 | 100% | 低但不可控 | 趋近于0 |
| 响应延迟 | 极低 | 取决于等待时间 | 即时 |
| 代码质量 | 差 | 差（魔法常量） | 优雅 |
| 设计意图 | 无 | 临时方案 | 专用原语 |

### 广播 vs. 信号量（Broadcast vs. Signal）

*   **核心术语：**
    *   **Signal (信号):** 唤醒**恰好一个**等待线程。适用于效率优化。
    *   **Broadcast (广播):** 唤醒**所有**等待线程。所有被唤醒的线程会竞争锁，然后重新检查条件。
*   **实践建议：**
    *   **始终使用 `Broadcast`**，除非你非常清楚 `Signal` 在特定场景下的效率优势。
    *   对于课程实验，使用 `Broadcast` 保证正确性即可，无需担心CPU微优化。

## 七、Channel：同步通信机制

> 主题：Go 语言 Channel 的核心概念、同步语义、死锁风险以及最佳应用场景。

### 核心概念：Channel 不是队列

Channel 是一个**同步**通信原语，**没有内部存储能力**（默认无缓冲）。

*   **关键行为：**
    *   **发送时无接收者 -> 阻塞**：发送操作会阻塞，直到另一个协程准备好接收数据。
    *   **接收时无发送者 -> 阻塞**：接收操作会阻塞，直到另一个协程准备好发送数据。
    *   **数据传递是同步的**：发送和接收操作在数据交换的那一刻**同时发生**。

```go
// 示例：无缓冲Channel的同步行为
ch := make(chan int)

go func() {
    time.Sleep(1 * time.Second) // 模拟延迟
    ch <- 42                    // 发送数据（直到main中接收才完成）
}()

val := <-ch // 接收数据（阻塞直到goroutine发送）
println(val) // 输出：42
```

### 理解要点

| 特性 | 说明 |
|------|------|
| **同步性** | 发送和接收是**原子配对**的，没有中间缓存 |
| **阻塞语义** | 发送/接收操作是**阻塞式**的，直到对方准备好 |
| **数据交换** | 数据直接从发送方**复制**到接收方，没有中间存储 |
| **适用场景** | 协程间**信号同步**、**数据传递**（非缓冲消息队列） |

### 与队列的关键区别

*   **传统队列**：生产者可以push数据后立即返回，数据暂存在队列中。消费者后续pull。
*   **无缓冲Channel**：生产者必须**等待**消费者"伸手来拿"，数据传递是**瞬时同步**的。
*   **类比：** 传统队列是**邮箱**（投递后离开），Channel是**握手**（必须双方同时接触）。

### 同步发送行为示例

```
// 示例1：无缓冲信道阻塞行为
ch := make(chan int)        // 创建无缓冲信道
go func() {
    time.Sleep(1 * time.Second)  // 等待1秒
    <-ch                         // 接收数据
}()
start := time.Now()
ch <- 1                         // 发送操作会阻塞1秒
elapsed := time.Since(start)   // elapsed ≈ 1秒
```

- **关键洞察**：发送操作 `ch <- 1` 会 **完全阻塞** 直到接收方 `<-ch` 执行
- **时间分析**：主 goroutine 在发送时被阻塞 **整整1秒**，这就是同步的本质

### 死锁与潜在错误

```go
// 示例2：单goroutine死锁
ch := make(chan int)
ch <- 1    // 阻塞！没有接收者
<-ch       // 永远不会执行到这里
```

- **死锁检测**：Go 运行时能检测到 **所有 goroutine 都休眠** 的状态
- **死锁结果**：程序会 **直接崩溃** 并报出死锁错误

```go
// 示例3：隐式死锁（更难检测）
go func() {
    for {}    // 忙等待，不干有用工作
}()
ch := make(chan int)
<-ch        // 永远阻塞，没有发送者
```

- **隐式错误**：当有其他 goroutine 在运行时，**死锁检测器不会触发**
- **实际表现**：程序看起来 **卡住挂起**，没有报错提示
- **调试难度**：这种错误难以追踪，因为看起来像是正常运行

### 核心规则总结

1.  **信道不能孤立使用**：发送和接收必须在 **不同 goroutine** 中同时进行
2.  **同步配对机制**：
    - 发送等待接收
    - 接收等待发送
    - 双方就绪后 **立即完成数据交换**
3.  **阻塞后果**：如果没有配对的操作，goroutine 将 **永远阻塞** 且无法恢复

### 缓冲通道（Buffered Channels）

- **核心概念**：缓冲通道拥有**内部存储空间（容量）**，发送操作在**缓冲区未满**时是**非阻塞**的。
- **运作机制**：
  - 缓冲区有空位时，发送方直接存入数据，无需等待接收方。
  - 当缓冲区**填满**后，行为退化为无缓冲通道——后续发送会阻塞，直到有接收方取走数据腾出空间。
- **重要警告**：**应避免使用缓冲通道**。它们很少能解决实际问题。
  - 当需要人为指定任意数字（如容量为1）来使代码工作，通常意味着设计存在问题。

### 通道的最佳应用场景

通道虽然强大，但只适用于**特定的小范围问题**，并非万能工具。

#### 场景一：生产者-消费者队列

- **经典模式**：多个Goroutine（生产者）并行计算结果，主Goroutine（消费者）持续收集并处理这些结果。
- **示例代码**：
  ```go
  ch := make(chan int)
  // 生成多个生产者
  for i := 0; i < 5; i++ {
      go doWork(ch)  // 生产者：计算并发送结果
  }
  // 主消费者：持续接收并打印
  for result := range ch {
      fmt.Println(result)
  }
  ```
- **特点**：主Goroutine通过`for range`持续接收，确保所有数据都被处理。

#### 场景二：替代Wait Group（等待所有任务完成）

- **核心价值**：通道可以实现等同于`WaitGroup`的功能——等待多个Goroutine全部结束。
- **实现方式**：
  1. 创建一个通道
  2. 启动已知数量的Goroutine
  3. 每个Goroutine完成后**发送信号**（如发送空结构体）
  4. 主Goroutine**接收对应次数**的信号，确保所有任务完成
- **关键点**：必须**精确知道**启动了多少个Goroutine，以便接收相应次数的信号。

### 总结：通道使用的黄金法则

1. **首选无缓冲通道**：同步、简单、确定性强
2. **避免盲目使用缓冲通道**：除非有非常明确的理由，否则不要用
3. **通道不是锁的替代品**：它们解决的是**通信**问题，而非**互斥**问题
4. **保持通道使用模式清晰**：生产者-消费者、信号通知等经典模式是最安全的选择

## 八、WaitGroup 与 Channel 的对比

> 主题：WaitGroup 和 Channel 作为同步原语各自的使用场景和选择依据。

### WaitGroup：轻量级的"任务计数器"

- **核心概念**：**`WaitGroup`** 是一种**计数信号量**，专门用于等待一组并发的 goroutine 全部完成。它的工作原理非常直观，通过三个方法协作：
    - **`Add(delta int)`**：增加内部计数器。通常在创建 goroutine 之前调用，用于指定需要等待的 goroutine 数量。
    - **`Done()`**：减少内部计数器。通常在每个 goroutine 的结尾调用（通常通过 `defer` 保证执行），标志该任务结束。
    - **`Wait()`**：阻塞当前 goroutine，直到内部计数器归零。

- **关键用法与注意事项**：
    - **避免竞态条件**：`Add()` 的调用**必须**在 `Wait()` 和 `Done()` 之前，且最好在启动 goroutine 之前完成。这确保了在 `Wait()` 检查计数器时，所有预期的计数都已注册。
    - **典型模式**：`Add` 在父 goroutine 中执行，`Done` 在子 goroutine 中执行，`Wait` 在父 goroutine 中阻塞等待。这避免了子 goroutine 的 `Done` 在父 goroutine 的 `Add` 之前执行的风险。

- **代码示例**：
    ```go
    var wg sync.WaitGroup
    
    for i := 0; i < 5; i++ {
        wg.Add(1) // 在启动前增加计数器
        go func() {
            defer wg.Done() // 确保任务完成时计数减一
            // 执行任务...
        }()
    }
    
    wg.Wait() // 阻塞，直到所有 5 个 goroutine 都调用 Done
    ```

### Channel vs. WaitGroup：何时选择？

- **Channel 实现 WaitGroup 功能**：
    - **原理**：通过一个**无缓冲 channel**，每个 goroutine 完成时发送一个信号（如 `struct{}{}`），主 goroutine 接收相同数量的信号。
    - **优缺点**：虽然可以实现相同的等待效果，但不如 `WaitGroup` 直观和轻量。Channel 本质上是用于**通信**和**数据传递**的，而非单纯的任务同步。

- **何时使用 Channel，何时使用 WaitGroup？**
    - **优先选择 `WaitGroup`**：当你的主要目标是 **等待一组 goroutine 全部完成**，而不关心它们的执行结果或中间通信时。这是最直接、最符合语义的选择。
    - **谨慎使用 Channel**：Channel 的设计初衷是 **goroutine 之间的通信**（如生产者-消费者模式）。虽然可以模拟同步，但作者建议**除非你经过深思熟虑，否则应避免使用 Buffer Channel（缓冲通道），甚至尽量避免使用 Channel**。
    - **推荐原则**：
        - **任务同步**：用 `WaitGroup`。
        - **数据传递和工作流**：用 `Channel`。例如，实现一个**生产者-消费者队列**，或用于在 goroutine 之间安全地传递计算结果。

- **关键总结**：
    - `WaitGroup` 是**专门为"等待完成"而设计**的原语。
    - Channel 是**为"数据通信"而设计**的原语。用 Channel 来实现 `WaitGroup` 的功能是可行的，但并不推荐，因为它增加了复杂性，且没有带来额外的好处。

## 九、并行 RPC 调用模式

> 主题：在 Raft 等分布式系统中，如何通过并行 RPC 调用和同步原语实现高效的并发通信。

### 并行 RPC 调用的实用模式

**应用场景：Raft 协议中的并行请求**

1. **请求投票 (RequestVote)** - 向所有 Follower 并行发送
2. **追加日志 (AppendEntries)** - Leader 并行通知所有 Follower

**为什么需要并行？**
- RPC 是**阻塞操作** (Blocking Operation)
- 串行调用会导致严重的延迟累积
- **关键指标**：`延迟 = max(所有节点延迟)` vs 串行的 `sum(所有节点延迟)`

**实现模板**
```go
for _, server := range peers {
    go func(srv *Server) {
        reply := srv.CallRPC(args)
        // 处理回复
        mu.Lock()
        results = append(results, reply)
        mu.Unlock()
    }(server)
}
```

### 同步原语：WaitGroup

**核心用途**：主 Goroutine 等待一组并发操作完成

- **`wg.Add(n)`**：设置计数器初始值
- **`wg.Done()`**：每个 Goroutine 完成时调用，计数器减 1
- **`wg.Wait()`**：阻塞直到计数器归零

**最佳实践**
- `defer wg.Done()` 确保即使在 panic 或 early return 时也能正确通知
- Add 操作应在启动 Goroutine **之前**完成

### 关键要点总结

1. **并行 > 串行**：RPC 密集型操作必须并行化
2. **同步原语**：优先使用 `WaitGroup` 而非 `time.Sleep` 等待

## 十、Raft 选举投票计数：并行与互斥锁

> 主题：Raft 选举中投票计数的正确实现，包括锁的使用、先请求后加锁的模式以及避免忙等待。

### 投票计数代码的核心逻辑

- **核心数据结构**：`count`（已获得的"是"票数）和 `finished`（已收到的总响应数）
- **核心目标**：并行向所有对等节点（peers）发送投票请求，并实时判断是否赢得选举（`count >= 多数`）或输掉选举（`finished == 总节点数`）
- **执行流程**：主 goroutine 循环等待条件满足，然后根据结果决定升为 Leader 或降为 Follower

### 正确实现：使用互斥锁（Mutex）保护共享变量

**关键模式**：**先发送请求（耗时操作），后加锁更新状态**

- **为什么先请求后加锁？** 网络请求（RPC）是慢速操作，如果在持有锁时发送请求，会严重阻塞其他 goroutine，降低并发性能
- **正确步骤**：
  1. 发起投票请求（**不持有锁**）
  2. 请求返回后，**获取锁**
  3. 在锁保护下更新 `count` 和 `finished`
  4. **释放锁**

**伪代码结构**：

```go
var mu sync.Mutex
var count, finished int

// 在每个并行 goroutine 中：
go func() {
    reply := requestVote(peer)  // 1. 先请求，不持锁
    mu.Lock()                   // 2. 加锁
    if reply.VoteGranted {
        count++                 // 3. 更新共享变量
    }
    finished++                  // 3. 更新共享变量
    mu.Unlock()                 // 4. 解锁
}()

// 在外部主循环：
mu.Lock()
for !(count >= 5 || finished == 10) {
    mu.Unlock()
    // 短暂休眠或等待条件变量（cond.Wait）
    mu.Lock()
}
mu.Unlock()
// 根据结果决定升Leader或降Follower
```

### 重要模式总结

- **先请求，后加锁更新**：减少锁的持有时间，提升并发效率
- **循环检查条件 + 加锁保护**：共享条件的判断必须在锁保护内完成
- **避免忙等待（busy waiting）**：简单的轮询检查会消耗 CPU，生产环境应使用**条件变量（sync.Cond）** 实现高效的等待/通知机制

### 核心公式与判断条件

- **赢得选举的判断**：$count \geq \lfloor \frac{N}{2} \rfloor + 1$，其中 $N$ 是集群总节点数
  - 示例中 N=10，多数票为 $5$ 票（包含自己的一票）
- **输掉选举的判断**：$finished == N$ 且 $count < 多数$

## 十一、Raft 实现中的常见并发 Bug

> 主题：Raft 实现中常见的并发 Bug 分析，包括竞态条件、死锁以及正确的修复方法。

### Bug 1: 选举期间的竞态条件 (Race Condition)

- **核心问题**: 在 `AttemptElection` 函数中，状态（state, currentTerm, votedFor）的修改与发起 RPC 调用之间缺乏足够的同步，导致在不同 goroutine 中同时读写共享状态。

- **代码示例 (有 Bug 的伪代码)**:

```go
type RaftNode struct {
    mu          sync.Mutex
    state       State     // follower, candidate, leader
    currentTerm int
    votedFor    int
}

func (rf *RaftNode) AttemptElection() {
    // 1. 修改状态 (可能未加锁或锁的粒度不够)
    rf.state = Candidate
    rf.currentTerm++
    rf.votedFor = rf.me

    // 2. 发起 RPC (在循环中启动多个 goroutine)
    for _, peer := range rf.peers {
        go rf.CallRequestVote(peer, rf.currentTerm) // 传递了 currentTerm
    }
}

func (rf *RaftNode) CallRequestVote(peer int, term int) {
    // 3. 构建请求参数 (此时 term 可能已过期)
    args := RequestVoteArgs{Term: term}

    // ... 发送 RPC ...
}
```

- **Bug 剖析**:
  1. **状态修改与 RPC 发送的时序**: 在 `AttemptElection` 中递增 `currentTerm` 后，立即启动多个 goroutine 去发送 `RequestVote` RPC。这些 goroutine 捕获的是 `rf.currentTerm` 的**当前副本**。
  2. **并发写入导致数据竞争**: 当多个节点同时尝试选举时，或者该节点在发送完部分 R
