### 项目2深度概念精析

*   **核心挑战：并发数据结构与内存管理**
    *   **背景：** 传统数据结构的操作（如插入、删除）在单线程下是顺序且安全的。但在**多线程渲染或游戏引擎物理/逻辑更新**等并发环境中，多个线程可能同时访问或修改同一数据结构。
    *   **核心问题：** **竞态条件**与**数据竞争**。例如，一个线程正在读取链表节点A的 `next` 指针，而另一个线程同时删除了节点A。这会导致**野指针**或**内存泄漏**。
    *   **本质难点（“内存中移动的东西”）：**
        *   不是简单的算法逻辑，而是必须**显式同步**所有对共享内存的访问。
        *   需要处理**无锁编程**或**精细粒度锁**，以避免死锁并保证数据一致性。
        *   需要考虑**内存模型**（如C++的 `std::memory_order`）来确保不同线程看到的变量修改顺序是一致的。
    *   **对游戏引擎/渲染工程师的启示：**
        *   这直接对应**实体组件系统（ECS）中的并行组件更新**、**渲染命令缓冲区（Command Buffer）的多线程构建**以及**资产流式加载**。理解如何安全地在不同线程间传递数据（如纹理、网格）是引擎性能优化的关键。
    *   **实践建议：**
        *   将项目2视为对**C++内存模型与并发原语**的实战考验。
        *   推荐阅读：*C++ Concurrency in Action* 中关于**无锁数据结构**和**原子操作**的章节。
        *   **核心原则：** **先保证正确性（无数据竞争），再谈性能优化。**

- **重点掌握**：运算符的语义、合成规则、重复元组的处理逻辑

#### SQL 查询
- **核心概念**：考试不会要求你手写完整SQL（因为无法在不运行的情况下评分），但**会给出SQL代码要求你解释其功能**
- **关键知识点**：
  - **子句类型**：
    - **WHERE 子句**：过滤条件
    - **基本查询结构**：SELECT-FROM-WHERE 的核心范式
    - **输出控制**：ORDER BY（排序）、LIMIT（限行）、OFFSET（偏移）
  - **高级操作**：
    - **JOIN（连接）**：INNER JOIN、LEFT/RIGHT/FULL OUTER JOIN 的区别
    - **聚合（Aggregation）**：GROUP BY + HAVING 的配合使用
    - **CTE（公用表表达式）**：WITH 子句，用于分解复杂查询
- **重要提示**：**窗口函数（Window Functions）不考**，这是明确的考试范围信号

#### 数据库存储引擎
- **页面管理策略**（Buffer Pool 替换算法）：
  - **LRU（最近最少使用）**：淘汰最久未访问的页面
  - **LRU-K**：记录最近K次访问时间，避免一次顺序扫描污染缓存
  - **Clock（时钟算法）**：近似LRU，用环形缓冲区+引用位实现
- **磁盘文件结构**：
  - **页目录（Page Directory）**：管理页面位置与空闲空间
  - **记录存储格式**：固定长度 vs 可变长度记录的元数据组织
- **页面内布局**（核心对比）：
  - **Slotted Pages（槽页）**：**记录可移动，槽数组固定**。优点：支持空闲空间合并；缺点：需要槽目录维护
  - **Log-Structured（日志结构）**：**只追加写入，不原地更新**。优点：写效率高；缺点：需要垃圾回收

#### 哈希索引
- **静态哈希**：
  - **线性探测（Linear Probing）**：冲突时逐个向后找空位
  - **Robin Hood哈希**：**平衡探测距离**，长探测键与短探测键交换位置
  - **Cuckoo哈希**：**双哈希表+踢出机制**，冲突时踢出旧键到另一张表
- **动态哈希**（支持扩容）：
  - **可扩展哈希（Extendable Hashing）**：使用**全局深度（Global Depth）** 和**局部深度（Local Depth）** 管理桶分裂，通过目录（Directory）进行弹性扩展
  - **线性哈希（Linear Hashing）**：**渐进式分裂**，通过分裂指针逐桶增长，无需全局目录，支持溢出链（Overflow Chains）

#### 重点公式与算法
- **Slotted Page 结构**：
  - 页头：`slot_count + free_space_pointer`
  - 槽数组：每个slot存`(record_id, offset, length)`，记录数据从页尾向前增长
- **可扩展哈希分裂条件**：
  - 当桶的**局部深度 < 全局深度**时，只需分裂该桶
  - 当**局部深度 == 全局深度**时，需要**目录翻倍** -> 全局深度+1
- **线性哈希分裂规则**：
  - 分裂指针 `split_ptr` 初始为0
  - 当负载因子超过阈值时，分裂 `split_ptr` 指向的桶，然后 `split_ptr++`
  - 查找时：先用 `hash(key) MOD (N * 2^level)`，若结果 < split_ptr，再用 `hash(key) MOD (N * 2^(level+1))`

## 二、查询执行：处理模型

> 主题：数据库系统如何遍历查询计划树并执行各个算子，包括迭代器、物化和向量化三种模型

### 查询执行（Query Execution）概述

#### 1. **课程上下文**

- **已学内容**：数据结构、磁盘↔内存数据搬运、排序（sort）、连接（join）
- **本讲重点**：如何**获取查询计划（query plan）**并**实际运行和执行它**

#### 2. **主讲内容框架**

- **处理模型（Processing Models）**：数据库系统可以采用的**不同执行方式**
- **访问方法（Access Methods）**：**顺序扫描（sequential scan）** vs **索引扫描（index scan）**

### 处理模型（Processing Models）

#### 1. **核心问题**

> 数据库系统如何遍历查询计划树（query plan tree）并执行各个算子（operators）？

#### 2. **三种主要模型**

| 模型 | 核心思想 | 优点 | 缺点 |
|------|----------|------|------|
| **迭代器模型（Iterator Model）** | 每个算子实现 `open()` → `next()` → `close()` 接口，**一次一行**（volcano/pipeline模型） | 简单通用，支持流水线 | 函数调用开销大 |
| **物化模型（Materialization Model）** | 每个算子**一次性输出所有结果**（materialized） | 适合OLTP，减少调度开销 | 内存压力大，不适合大结果集 |
| **向量化/批处理模型（Vectorized/Batch Model）** | 每次`next()`返回**一批元组**（batch/vector），如1024行 | 减少函数调用，利用SIMD指令，缓存友好 | 实现复杂度较高 |

- **关键术语**：**Iterator (volcano) model**、**Materialization model**、**Vectorization model**
- **实际应用**：
  - **OLTP**（事务型）：通常使用**物化模型**
  - **OLAP**（分析型）：通常使用**向量化模型**（如MonetDB、ClickHouse）

#### 3. **公式/算法要点**

```
迭代器模型执行流程：
Loop: result ← child.next()  →  process result  →  emit to parent

物化模型执行流程：
results ← child.output()  →  process entire result set  →  emit materialized output

向量化模型执行流程：
Loop: batch ← child.next_batch(N)  →  process batch (vectorized)  →  emit batch
```

### 迭代器模型（Iterator Model）与流水线处理

#### **核心概念：迭代器模型**
- **基本思想**：查询计划中的每个物理运算符（如Join实现、Scan操作符）都提供一个 **`next()` 函数**接口
- **每次调用** `next()` 时，操作符返回两种结果之一：
  - **返回一个元组**（Tuple）：可以是列的子集，也可以是记录ID（Record ID）
  - **返回空指针**（Null Pointer）：表示没有更多元组需要处理，告知调用者停止调用
- **关键机制**：父节点操作符调用子节点的 `next()`，从查询计划底部向上拉取数据，直到生成最终输出返回给客户端
- **别称**：火山模型（Volcano Model）、管道模型（Pipeline Model）
- **历史**：由B+树论文作者Graefe在80年代末90年代初的学术项目“Volcano”中系统化提出，虽非首创，但首次明确定义了该模型。

#### **流水线模型（Pipeline Model）解释**
- **核心目标**：理想情况下，希望**尽可能地将单个元组沿着流水线向上推送**，直到无法继续处理，然后再返回底部获取下一个元组
- **工作流程**：
  1. 从顶部根操作符开始，调用其 `next()` 函数
  2. 根操作符递归调用子操作的 `next()` 函数
  3. 数据从底部向上流式传输

#### **实际示例分析**
### 查询语句：
```
SELECT * FROM R, S WHERE R.id = S.sid AND S.value > 100
```

#### **物理操作符实现特点**
- **叶子节点**（Scan操作符）：直接遍历目标表中的元组
- **内部节点**（如Join、Filter）：通过 **`for` 循环** 调用子节点的 `next()` 获取数据

#### **执行顺序**
1. **根操作符**（Projection投影）调用 `next()`
2. 逐层向下递归调用子操作的 `next()` 函数
3. 数据从底部叶子节点开始，沿着查询计划树向上流动

#### **关键术语与公式**
- **`next()` 函数**：迭代器模型的核心接口，返回值类型可表示为：
  ```
  Tuple next() {
      if (hasMoreTuples) return tuple;
      else return NULL;
  }
  ```
- **元组（Tuple）**：可以是完整行、列子集或记录ID
- **空指针（Null Pointer）**：流结束信号，控制迭代终止

#### **重要说明**
- **发明者**：虽然没有完全发明此概念，但他是**首次系统化编码和解释**迭代器模型的实际含义
- **并行化技术**：发明了使迭代器模型能够并行化的技术（将在下节课讨论）

### 基于拉取模型（Pull Model）的查询执行与流水线

#### 1. 核心概念：**拉取模型（Pull Model）与迭代器（Iterator）模式**

- **基本机制**：每个查询操作符都实现一个 `next()` 函数，其父操作符通过调用子操作符的 `next()` 来“拉取”数据。
- **数据流**：根操作符（通常是投影 `Projection`）调用其子操作符的 `next()`，子操作符再调用其子操作符的 `next()`，形成**自上而下的调用链**。
- **关键性质**：每次 `next()` 调用只**返回一个元组（Tuple）**，即“一次一元组”的处理模式。

#### 2. 查询执行流程示例（以哈希连接 Hash Join 为例）

##### 执行顺序与状态机

- **阶段一：构建（Build Phase）**
  - 连接操作符（`Join`）被父节点调用 `next()`
  - 首先**迭代遍历左侧子节点**，反复调用其 `next()`，逐元组读取数据
  - 每读取一个元组，就将其**插入哈希表（Hash Table）**中
  - 直到左侧子节点所有元组读取完毕，哈希表构建完成

- **阶段二：探查（Probe Phase）**
  - 连接操作符开始**迭代遍历右侧子节点**，反复调用其 `next()`
  - 右侧数据流经过**过滤操作符（Filter）**：过滤操作符调用其子节点的 `next()`，对返回的元组进行**谓词评估（Predicate Evaluation）**，只将符合条件的元组向上传递
  - 过滤后的元组到达连接操作符，与已构建的哈希表进行**探查匹配**
  - 若匹配成功，则拼接并向上传递；若不匹配，则继续读取下一个元组

##### 操作符的层次调用关系

```
Projection (根操作符)
  └─ Join (Hash Join) —— 流水线中断点（Pipeline Breaker）
      ├─ 左子节点: Table Scan（构建阶段：读取所有数据构建哈希表）
      └─ 右子节点: Filter（探查阶段：逐元组读取并过滤）
          └─ 子节点: Table Scan（逐元组扫描）
```

#### 3. 流水线（Pipeline）与流水线中断点（Pipeline Breaker）

##### 核心概念：**操作符的高度可组合性（Composability）**

- 只要所有操作符都实现了 `next()` 接口，它们就可以**自由组合成任意查询计划**
- 一个操作符**不关心其子节点具体如何实现**——它可以是另一个连接、索引扫描或表扫描

##### **流水线中断点（Pipeline Breaker）**

- **定义**：一个操作符在其父节点调用 `next()` 时，**无法立即返回一个元组**，而是必须先处理完其子节点的全部数据
- **哈希连接（Hash Join）是典型的流水线中断点**：
  - 因为必须先**完全构建左侧哈希表**才能开始探查阶段
  - 这意味着在构建阶段，它会在一个循环中反复调用左侧子节点的 `next()`，直到处理完所有元组

##### 流水线中断的影响

- 中断点之前的所有操作符可以形成一个**流水线**：数据在操作符之间逐元组传递，无需物化中间结果
- 中断点之后的流水线只能从探查阶段开始形成
- **性能影响**：中断点会打破 CPU 缓存的局部性，并可能导致需要物化中间结果

#### 4. 关键算法与公式

##### 哈希连接构建阶段

```python
# 伪代码：构建阶段
while tuple = left_child.next():
    hash_table.insert(tuple.hash_key, tuple)
```

##### 哈希连接探查阶段

### 查询执行中的流水线与流水线中断

#### 核心概念：**流水线执行模型**

- **流水线的定义**：在数据库查询执行中，**流水线（Pipeline）** 是一种高效的处理方式，它允许一个元组（Tuple）从树的底层一路向上处理，**无需将中间结果写回磁盘**。
- **工作流程**：
    1.  **构建阶段**：首先，系统需要遍历左侧（通常是构建哈希表的一侧）的所有元组，以构建哈希表（例如，对于哈希连接）。这一步是**阻塞的**，必须看到全部数据。
    2.  **探测阶段**：一旦哈希表构建完成，当从树的另一侧调用 `next()` 时，就会形成一个流水线。处理流程是：从底部节点（例如，表扫描）获取一个元组。
        - 该元组通过**谓词评估**（Predicate Evaluation）。
        - 如果评估为真，则向上传递到下一个算子（例如，哈希探测）。
        - 哈希探测成功（找到匹配），则继续向上传递到投影（Projection）算子。
        - 最后输出结果。

- **流水线的优势**：
    - **避免磁盘I/O**：这是最大的优势。每个元组在被获取后，可以在内存中完成所有必要的计算，**一次性输出**，完全不需要将中间结果写回磁盘或临时文件。这对于磁盘密集型系统来说，性能提升巨大。
    - **工程实现简单**：流水线模型将复杂的查询分解为串行步骤。
    - **输出控制极其简单**：例如，如果查询带有 `LIMIT 10` 子句，则顶层的循环只需在计数达到10时停止调用 `next()`，查询立即结束。无需处理海量数据再截断。

#### 关键术语：**流水线中断符（Pipeline Breakers）**

- **定义**：有些运算符是**阻塞（Blocking）** 的，它们必须等待所有子节点发射完所有元组后才能开始工作。这些算子被称为**流水线中断符**。
- **常见的流水线中断符**：
    - **连接（Joins）**：特别是哈希连接，必须先构建完整个哈希表（处理所有左侧输入）。
    - **子查询（Sub-queries）**：某些子查询需要完全执行完毕。
    - **排序（Sorting）**：这是最典型的例子。**无法对数据进行全局排序，直到看到所有数据**。
- **影响**：流水线中断符会破坏流水线的连续性，迫使系统在这一点上物化（Materialize）中间结果。

#### 流水线的优缺点（讨论）

- **优点**：
    - 极致的内存效率。
    - 输出响应速度快（尤其是带 `LIMIT` 时）。
    - 降低了查询执行器的实现复杂度。
- **潜在缺点**：
    - **巨大的函数调用开销**：如果所有数据都在内存中，那么每个元组在流水线中经过每个算子时都会产生一次函数调用（`next()`）。这会导致大量的CPU开销，成为纯内存场景下的性能瓶颈。

#### 总结

流水线模型是现代数据库查询执行的核心。它的精髓在于**“一个元组完成所有工作”**，通过减少磁盘I/O来大幅提升性能。工程师需要识别并谨慎处理**流水线中断符**（如排序、连接），因为它们会打断高效的流水线流程。同时，也要注意在内存处理场景下，过度的函数调用可能成为新的瓶颈。

### 单次单元组处理模型的局限性与架构权衡

#### **核心概念：火山模型（Iterator Model）的“拉取”范式**
- **核心机制**：每个查询计划节点通过 `next()` 接口向上层返回**单个元组**，形成“拉取”数据流。
- **关键问题**：这种设计导致**高度碎片化**的函数调用链，每个元组都可能触发大量函数调用。
  - **代价**：如果数据驻留内存，**函数调用开销**仍不可忽视；若涉及磁盘I/O，虽然“跳转”成本相对可控，但**磁盘访问本身始终是瓶颈**。
- **优化局限**：无法对**下层节点**的执行行为进行全局优化。
  - **实例**：`ORDER BY ... LIMIT` 操作必须**拉取所有子节点数据**，进行完整排序后，才能应用限制。
  - **本质**：这是**执行模型固有特性**，与优化器的查询计划生成无关。

#### **执行计划与优化器的职责分离**
- **假设前提**：当前讨论基于**给定优化后的执行计划**，不涉及优化器如何生成该计划。
- **翻译流程**：SQL字符串 → (优化器) → 最优计划树 → (执行器) → 各节点函数实现（如PostgreSQL的Project节点）。
- **优化器核心任务**（期中后覆盖）：
  - 确定**连接顺序**（内表 vs 外表）
  - 选择**连接算法**
  - 下推谓词（如 `WHERE` 条件至Join之前）
  - 对比关系代数中谓词位于顶层的默认位置
- **教学挑战**：数据库系统各部分**紧密耦合**，必须先掌握连接算法才能讨论优化器，形成依赖关系。

#### **该模型的实际瓶颈**
- **单次元组处理**：每次获取一个元组后，可能立即执行昂贵操作（如聚合、函数计算）。
- **错过批量优化机会**：如果能在同一时刻处理**多个元组**，就可以通过**向量化**或**批处理**来“碰撞”数据，大幅降低指令级开销和函数调用成本。
- **结论**：尽管函数调用代价是可接受的代价，但**单次单元组**的细粒度处理方式才是火山模型性能扩展的**真正天花板**。

### 物化模型（Materialization Model）

#### **核心概念：自底向上，一次性产出全部结果**
- 与**迭代器模型**（自顶向下，`Next()` 逐条拉取）完全相反。
- **物化模型**不再依赖 `Next()` 函数，而是要求每个算子实现一个**一次性生成全部输出**的方法（如 `Output()` 或 `Materialize()`）。
- 执行流程是**自底向上（Bottom-Up）**：**子算子**完整执行，**将所有结果物化**（存为临时数据），然后**整体传递给父算子**。

#### **工作原理**
1.  **底层算子**（如扫描表）一次性读取所有数据，生成完整的结果集。
2.  将整个结果集**整体返回**（而非逐条返回）给上层算子。
3.  **上层算子**拿到全部数据后，完整执行自己的逻辑，再将其全部结果传递给更上层。
4.  重复直到根节点，最终得到完整查询结果。

#### **与迭代器模型的关键对比**

| 特性 | 迭代器模型（Iterator Model） | 物化模型（Materialization Model） |
| :--- | :--- | :--- |
| **执行方向** | 自顶向下（Top-Down） | 自底向上（Bottom-Up） |
| **数据传输** | 逐条/逐批（通过 `Next()`） | 一次性传输完整结果集 |
| **内存压力** | 较低（流水线化，边生成边消费） | **极高**（需要内存容纳完整中间结果） |
| **适用场景** | OLTP、内存不充裕、数据量大 | 数据量小、或需要批量优化 |

#### **关键挑战与讨论**
- **内存溢出问题**：如果某个算子的输出结果巨大（超过内存），物化模型会直接崩溃。此时必须采用 **混合散列连接** 或 **递归分区** 等**磁盘溢出策略**。
- **与查询计划的结合**：大多数数据库系统并不会在运行时动态判断是否需要溢出磁盘，而是**在查询规划阶段就确定好执行策略**，生成一个**固定的查询计划**。一旦确定采用物化模型，整个计划就“固化”下来。

#### **一句话总结**
> 物化模型是**直观但“笨重”** 的方式：**每个算子一次性算完所有结果，再整体传给上级**。它简化了编程逻辑（人类最容易理解），但内存消耗巨大，因此曾是20年前的主流实现，如今不再普遍。

### 物化执行模型 (Materialization Execution Model)

#### 核心概念

**物化模型** 的核心思想是 **“一次性输出所有结果”**。每个算子会完全阻塞，直到其子算子将**全部数据**作为一个完整的缓冲区或列表返回。这与流水线模型（逐行或逐批推送）截然相反。

#### 执行流程（以 Join 为例 Top-Down）

1.  **顶向下调用**：根节点调用其子节点的 `output` 函数。
2.  **阻塞等待**：该调用是**阻塞调用**。当前算子会完全挂起，直到子节点返回**所有元组**。
3.  **构建哈希表**：接收到左子节点的全部数据后，算子可以一次性构建完整的哈希表。
4.  **处理右子树**：然后调用右子节点的 `output` 函数，同样等待其返回全部元组。
5.  **结果汇总**：将右子节点的结果与哈希表匹配后，将完整结果向上传递。

#### 关键技术：内联 (Inlining) 与谓词下推

由于数据一次性集成，直接传递整个表（如十亿行）是极其低效的。因此，**物化模型强制进行内联优化**：

*   将树**顶层的算子（如过滤谓词）嵌入到底层扫描算子中**。
*   例如：在扫描时，**当数据页从内存调入后，立即应用过滤谓词**，只将符合条件的元组放入输出缓冲区。
*   这避免了将大量无关数据一路推送到上层后再丢弃。

#### 数据传递单元

*   **完整元组**：传递整行包含所有列的数据。
*   **单列 + 书签**：也可以仅传递单列数据，但必须配合额外的**记账机制**，用于追踪当前传递的是原始表的哪一部分（如行ID）。

#### 核心优缺点

*   **优点**：
    *   简化了算子的实现（无需管理复杂的中间状态）。
    *   便于进行全局优化（内联）。
*   **缺点**：
    *   **内存爆炸**：所有中间结果必须完全物化到内存中，对于大表极不友好。
    *   **延迟高**：必须等到整个子树完成后才能开始后续处理。

### 查询引擎中的迭代器模型与物化策略优化

#### 核心概念：流水线式物化 vs. 一次性物化
- **关键术语**：**内联谓词求值 (Inline Predicate Evaluation)**、**迭代器模型 (Iterator Model)**、**生成器模式 (Generator Pattern)**
- **核心思想**：在经典的迭代器模型（火山模型）中，查询执行是一行一行（或一批一批）拉取的。然而，早期某些系统（如VoltDB、RavenDB）采用了一种**“完全物化再过滤”**的策略：先将操作符所有输出结果生成并存储在内存中（类似Python的`yield`一次返回全部），然后再应用过滤谓词。
- **优缺点权衡**：
    - **优点**：简化了内存访问模式，所有数据都在缓冲区，方便后续批量操作。
    - **缺点**：当处理**大表**时，如果操作符的输出结果过大，则会导致**内存抖动 (Thrashing)**——生成所有输出时一次性占满内存，甚至触发OS交换（swap）。

#### 优化动机：为何特定系统选择完全物化？
- **场景**：**操作型工作负载 (Operational Workloads)**，如OLTP系统。
- **核心瓶颈**：对于索引扫描（Index Scan），每次只返回**一条记录**（如`WHERE id = Andy`），函数调用（`next()`）的开销远大于磁盘I/O。在**内存数据库**中（如VoltDB），无磁盘等待，因此**函数调用开销成为主要矛盾**。
- **解决方案**：跳过“按需拉取-过滤”的常规步骤（图中步骤4），直接**将数据拉入输出缓冲区再统一过滤**。这实质上是**用内存带宽换取函数调用开销**。

#### 具体系统案例分析

| 系统 | 类型 | 采用策略 | 核心原因 |
| :--- | :--- | :--- | :--- |
| **VoltDB / H-Store** | OLTP内存数据库 | 完全物化 + 内联谓词 | 索引扫描每次只返回一条记录，消除函数调用开销（内存中无磁盘延迟，函数调用成为瓶颈） |
| **RavenDB** | 文档数据库 | 完全物化 | 同样是操作型负载，每次获取少量文档，减少调度开销 |
| **MonetDB** | OLAP列式存储系统 (荷兰CWI) | 逆向优化（依赖MMAP + 尽量下推扫描） | **注意点**：MonetDB是OLAP系统，情况特殊。它依赖**内存映射文件 (MMAP)** 来管理数据。它并未完全采用“生成全部再过滤”，而是为了**减少物化开销**，尽量将谓词下推至扫描阶段，利用OS的换页机制（swap）处理大表溢出。其论文核心目标是“减少中间物化”，与VoltDB的动机相反。 |
| **DuckDB** | OLAP系统 | 源自MonetDB分支 (MonetDB Light)，但做了彻底重写 | 继承了MonetDB的列式思想和物化策略，但进行了现代化优化 |

#### 核心公式与算法暗示
- **开销模型对比**：
    - **传统迭代器**：总开销 ≈ **每行函数调用开销** × 行数 + 过滤开销
    - **完全物化方法**：总开销 ≈ **一次批量物化开销**（含内存拷贝）+ 过滤开销 + **可能的交换开销**
- **关键判断条件**：
    - 当 `每行函数调用开销 × 行数` > `批量物化开销 + P(交换) × 交换代价` 时，完全物化更优。
    - 大多数OLTP系统的行数（单次查询）极小，使得完全物化总是更优。

#### 总结性洞察
- **没有银弹**：查询执行策略的选择取决于**工作负载特性**（OLTP vs. OLAP）和
