# 将应用逻辑嵌入数据库系统：从对话式API到新范式

## 一、核心动机：为什么要把应用逻辑放入数据库？

### 传统应用与数据库交互模式：**对话式API（Conversational API）**

- **工作流程**：
    1. 应用连接到数据库服务器
    2. 发送一个查询（Query）
    3. **等待**服务器返回结果
    4. 应用处理结果
    5. **双方等待对方执行下一步**——形成**来回交替的等待链**

- **关键缺陷**：双方处于**单向等待、轮流执行**的状态，无法高效协同处理复杂逻辑

### 核心转变思路

**将应用（PHP/Python/JavaScript等）中的业务逻辑，嵌入到数据库系统内部执行**，而非仅仅将数据库当作“**愚蠢的数据存储（dumb data store）**”

---

## 二、每个数据库系统的底层通信基础：**Wire Protocol（线缆协议）**

- **定义**：数据库系统实现的自有通信协议，定义了客户端与服务器之间如何传输数据
- **课堂层级说明**：本科课程不深入讨论，属于高级课程内容
- **为什么重要**：所有查询、结果返回、状态同步都依赖此协议

---

### 核心教学目的
**带给你一种全新的思考方式**——关于如何构建应用程序，以及如何**更深度利用数据库系统**，而不仅仅是：
- 把数据库当成被动存储
- 每次查询都做成独立的往返对话

### 预期收获方向
- 理解**逻辑下沉**的概念：把业务逻辑从应用层迁移到数据库层
- 学习数据库**主动执行能力**，而非被动响应查询
- 为后续学习**存储过程、触发器、UDF（用户自定义函数）** 等高级特性打下基础

## 第 2 部分

## 将应用逻辑推入数据库：从“拉取数据”到“推送逻辑”

### 核心概念：减少通信开销，提升性能

传统的数据库交互模式是**客户端-服务器请求-响应循环**。应用服务器与数据服务器之间，通过“线协议”（Wire Protocol）进行多次往返通信。

- **传统模式工作流**：
    1.  应用发起事务（Begin Transaction）
    2.  发送SQL查询 → 数据服务器**解析、规划、优化、执行** → 返回结果
    3.  应用处理结果（包含if语句等逻辑），然后发送下一个查询（或中止/提交）
    4.  重复上述过程，直至提交事务
- **关键痛点**：大量的**往返延时**。每次通信，双方都必须等待对方响应，即使在同一台机器或同一机架内，这种等待依然存在。

### 解决问题的方向：将逻辑推送到数据附近

我们的目标不是将数据拉取到应用层执行逻辑，而是**将应用层的逻辑尽可能地推送到数据库服务器内部执行**。

- **核心思想**：**将查询逻辑推送到数据所在处，而不是把数据拉到查询逻辑所在处**。
- **实际意义**：让数据库系统能够**自主决定**下一步操作，而不必每次都与应用服务器协商。这能显著减少网络通信次数。

### 推入逻辑的潜在收益

将应用逻辑（如条件判断、事务控制）下放到数据库服务器，可以带来以下好处：

- **减少网络开销**：消除了大量等待通信的往返时间。
- **提升执行效率**：数据无需频繁穿越网络，直接在本地完成计算。
- **降低应用服务器负载**：应用服务器不再需要维护复杂的查询状态，只需下发“目标”，让数据库自行“执行”并返回最终结果。

## 第 3 部分

## 将逻辑推入数据库服务器：核心动机与收益

### 核心理念：减少客户端-服务端交互，提升效率

传统数据库交互模式中，每个查询后客户端等待指令，导致大量**网络往返**。将应用逻辑（如条件判断、事务控制、变更响应）直接放到**数据库服务器端执行**，能从根本上降低延迟、提升吞吐量。

**为什么要这么做？** 核心是让一次请求做尽可能多的工作，避免反复的“请求-等待-再请求”循环。

---

### 五大潜在收益

#### 1. **减少网络往返次数**
- **关键点：** 将多个逻辑步骤打包到**单一查询请求**中，让数据库服务器一次性完成所有工作。
- **为什么重要：** 每次网络往返都有固定延迟（尤其是远程数据库），减少次数直接减少等待时间。

#### 2. **更快的变更通知**
- **关键点：** 数据库服务器在数据变更的**当下**就能检测并响应，无需等待外部应用轮询。
- **典型场景：** 定义触发器或内置动作，当某行数据更新时，自动执行后续逻辑（如更新缓存、写日志）。

#### 3. **减少事务等待时间（对悲观锁尤为重要）**
- **核心机制：** 在**基于锁的事务模型**中，客户端持有锁的同时，若等待下一个查询指令，锁的持有时间会延长。
- **优化方式：** 通过减少往返次数，让数据库在**一次通信内**接收并执行“提交（commit）”命令，锁持有时间显著缩短，事务完成更快。
- **公式化理解：** 事务总时间 = 执行时间 + 等待客户端指令时间。减少等待时间 = 减少锁竞争。

#### 4. **事务整体执行更高效**
- **效果：** 减少每个事务中闲置等待的时间片段，让事务流程更紧凑，单位时间内处理更多事务。

#### 5. **软件工程层面的优势：统一逻辑，避免重复实现**

这是最“软”但也最具战略意义的收益：

- **问题：** 同一场景（如“用户订单限制逻辑”），桌面应用用C#实现、移动App用Swift/Java实现，两者必须保持行为一致。一旦bug修复或有新需求，所有版本都要同步修改。
- **解决方案：** 将此核心逻辑**嵌入数据库服务器**（例如通过存储过程、UDF），成为所有应用共享的**唯一实现**。
- **收益：**
  - 跨平台（C#、Java、Swift）无需重复编写相同逻辑。
  - 逻辑变更只在数据库端一次修改，所有客户端立即生效。
  - 消除因语言/环境差异导致的**移植性bug**。

> **警告（Potential Downsides）：** 并非所有逻辑都适合嵌入。例如，**用户自定义函数（UDF）** 如果设计不当，反而可能使查询性能**急剧恶化**（后续会分析原因）。

---

### 方法论概览：实现“嵌入逻辑”的技术手段

这些技术将在后续详细展开，此处先列举高维分类：

- **UDF（用户自定义函数）：** 自定义标量/表值函数
- **存储过程：** 预编译的SQL逻辑单元
- **触发器：** 数据变更时自动触发的逻辑
- **内联函数/视图 + 计算列：** 声明式逻辑嵌入

**核心权衡：** 逻辑嵌入越深，数据库服务器负担越重，但应用端简化程度越高。设计时需要根据查询模式、并发压力、维护成本做出平衡。

## 用户定义函数 (UDF) 深入解析

### 核心概念：嵌入数据库逻辑的多种方法

- **主要方法对比**：
  - **UDF（用户定义函数）**：最常用，直接嵌入自定义逻辑到SQL中
  - **存储过程**：相对较少使用
  - **UDF + 触发器**：组合使用，这是最常见的方案
  - **变更通知**：用于数据库状态变化的感知
  - **自定义类型**：扩展数据模型
  - **视图**：提供数据虚拟化层

### UDF 的动机与本质

**核心思想**：扩展数据库系统的**内置功能**，使其超越标准SQL函数（如字符串函数、日期函数）的能力限制。

- **为什么需要UDF**：
  - SQL标准定义了内置函数清单，但特定应用场景可能没有对应函数
  - 可以编写**自定义函数**，像调用 `STRLEN`、`CONCAT` 一样直接在SQL中使用
  - 解决“**用SQL难以清晰表达的逻辑**”问题

### UDF 的基本特征

- **输入参数**：通常为**标量值**
- **执行体**：支持**命令式编程结构**，包括：
  - `IF` 条件分支
  - `FOR` 循环
  - 异常抛出
  - 调用其他函数
- **返回值**：
  - **标量值**：如整数、字符串
  - **表/行集**：返回一个结果表或多行数据

### UDF 的定义语法结构

```sql
-- 伪代码结构示例
CREATE FUNCTION function_name (参数列表)
RETURNS return_type  -- 可以是标量或表类型
LANGUAGE lang_name   -- 如 SQL、PL/pgSQL、C等
AS $$
  -- 函数体：包含命令式逻辑
  DECLARE
    -- 变量声明
  BEGIN
    -- 循环、条件判断等逻辑
    RETURN result;
  END;
$$;
```

### 关键技术要点

- **嵌套调用**：UDF内部可以调用其他UDF或内置函数
- **执行环境**：函数在**数据库服务器端**执行，与查询处理器协同工作
- **事务一致性**：UDF执行在调用它的SQL事务上下文中

### 实际应用场景

| 场景 | 说明 |
|------|------|
| 复杂业务逻辑 | 用命令式语言实现SQL难以表达的计算 |
| 数据验证 | 在插入/更新前执行自定义校验规则 |
| 数据转换 | 将列值按特定规则转换，如加密、哈希计算 |
| 聚合计算 | 实现SQL标准未提供的自定义聚合函数 |

### 注意事项

- **跨平台兼容性**：将UDF移植到不同数据库或环境时可能遇到**bug**
- **性能考虑**：UDF的执行开销需纳入整体查询性能评估
- **调试难度**：相比应用层代码，数据库函数调试工具相对有限

## 第 5 部分

### 用户自定义函数 (UDF) 核心概念

SQL 在表达复杂逻辑时**不如命令式语言（如 C++、Python）灵活**。因此，数据库提供了用户自定义函数（UDF），允许将复杂逻辑封装在函数中。

**UDF 的定义包含两部分：**
*   **返回类型**：可以是**单个值**，或**单个结果表**（多行）。
*   **函数体（计算逻辑）**：
    *   **SQL 函数**：函数体由一系列 SQL 查询组成，**无深度逻辑**。
    *   **外部语言函数**：使用非 SQL 的编程语言（如 C、Python）编写。

---

### SQL 基础型 UDF (Single-Based UDF)

这是**最常见、支持度最广**的 UDF 形式。

*   **执行流**：数据库系统**按顺序执行**函数体中的一系列 SQL 查询。
*   **返回值**：**最后一个查询**的结果就是函数的返回值。

#### 创建语法 (CREATE FUNCTION)

```sql
CREATE FUNCTION get_foo(INT) RETURNS foo
AS $$
    -- 函数体：一系列 SQL 语句
    -- 最后一条语句的返回值，作为函数结果返回
    SELECT * FROM foo WHERE id = $1;
$$ LANGUAGE SQL;
```

*   **参数传递**：用 `$1`、`$2` 等序号代表传入的参数。
*   **函数体定义**：PostgreSQL 将其视为字符串，但会**解析 SQL 语法**。
    *   *注：其他语言（如 Python）的 UDF，函数体是**纯字符串**，直到运行时才解析。*

#### 调用位置

*   `SELECT` 语句的输出列表：`SELECT get_foo(5);`
*   `FROM` 子句：`SELECT * FROM get_foo(5);`
*   **不能**在 `WHERE` 子句中直接调用（因为返回的是表/行，而不是标量值）。

---

### 现场演示步骤

1.  **定义表 `foo`**：创建基础数据结构。
2.  **插入数据**：向表 `foo` 中填充数据。
3.  **调用 `get_foo` 函数**：验证函数是否正确返回满足条件的行。

## 第 6 部分

### SQL UDF（用户定义函数）高级特性：实时演示与依赖追踪

#### 1. 基础UDF定义与查询

- **核心概念**：在SQL中，**用户定义函数（UDF）** 可以像普通表一样在`FROM`子句中使用，返回表格结果。
- **关键术语**：
  - **表值函数**：返回多行多列数据的函数，可直接在`FROM`中调用。
  - **标量函数**：返回单个值的函数，可在`SELECT`中直接调用。
- **操作演示**：
  1. 定义表`Foo`并插入数据。
  2. 创建函数`possum`（可能是打字错误，实际应为函数名）。
  3. 在`SELECT`中直接调用函数：`SELECT possum() FROM Foo` → 返回单个元组列。
  4. 在`FROM`子句中调用函数：`SELECT * FROM possum()` → 展开为多个列，如同普通表。

> **公式/算法**：无特定公式，但核心思想是**函数调用位置决定返回格式**：
> - `SELECT func()` → 返回标量或单列元组。
> - `FROM func()` → 返回表格结构，可访问所有列。

#### 2. `ATOMIC`关键字：依赖追踪与引用完整性

- **核心概念**：`ATOMIC`关键字用于**声明函数的原子性**，并非事务相关，而是**让数据库系统追踪UDF之间的依赖关系**，类似于外键约束在表间的引用完整性。
- **关键术语**：
  - **原子性（Atomic）**：在此上下文中指函数依赖可被数据库系统识别并保护，防止删除被引用的函数。
  - **依赖追踪（Dependency Tracking）**：数据库系统记录函数间的调用链，当尝试删除被其他函数调用的UDF时，会阻止操作。
- **作用**：
  - 提供类似外键的**引用完整性保护**。
  - 确保开发者不会意外删除被其他函数依赖的UDF，减少错误。

#### 3. 依赖演示：三层函数调用链

- **操作演示**：
  1. 创建三个函数`A`、`B`、`C`，确保无重复定义（先`DROP`）。
  2. 定义：
     - `A(int)`：参数乘以3。
     - `B(int)`：调用`A`后乘以2。
     - `C(int)`：直接调用`B`。
  3. 调用`SELECT C(1)` → 返回结果（1 → A得3 → B得6）。
  4. **依赖关系**：`C` → `B` → `A`，形成三层依赖链。
  5. **关键测试**：尝试`DROP FUNCTION A`时，数据库系统会**拒绝操作**，因为`B`和`C`依赖于它，保护了引用完整性。

> **重要算法/规则**：依赖保护规则：
> - 若函数`F`被其他UDF引用，则`DROP FUNCTION F`将失败。
> - 类似地，修改函数签名（如参数类型）也可能受依赖限制。

#### 4. 实际意义与开发价值

- **核心概念**：`ATOMIC`和依赖追踪**提升UDF的可维护性**，避免因函数重构导致的连锁错误。
- **关键术语**：
  - **引用完整性（Referential Integrity）**：确保依赖关系不被破坏，类似外键约束。
  - **开发安全**：允许数据库系统在UDF层面提供比普通函数更强的保护机制。
- **总结**：这是SQL标准中**为UDF增强数据库逻辑一致性**的高级特性，尤其适合需要复杂函数调用的场景（如金融计算、数据管道）。

> **无开/结尾语**：以上内容直接为技术要点，无多余叙述。

## SQL 函数中的依赖关系与原子性

### **核心概念：函数间的依赖追踪**

在 SQL 中，函数可以相互调用，形成**依赖链**（Dependency Chain）。例如：
- 函数 C 调用函数 B
- 函数 B 调用函数 A

此时，C 依赖于 B，B 依赖于 A。

#### **非原子函数（无依赖追踪）**
- **行为特征**：使用 **普通 `FUNCTION`** 定义时，系统**不追踪函数间的依赖关系**
- **创建函数**：即使 A 已被删除，仍可创建调用 B 的 C，因为系统**不检查依赖是否存在**
- **执行风险**：运行时调用 C → 调用 B → 调用 A，但 A 不存在 → **运行时崩溃**（Dangling reference）

```sql
-- 创建非原子函数
CREATE FUNCTION C() AS $$ ... 调用 B ... $$ LANGUAGE SQL;
-- 即使 A 已删除，也能成功创建
-- 但调用时会报错
```

### **原子函数与依赖保护**

#### **`ATOMIC` 关键字**
- **核心机制**：在定义函数时使用 `ATOMIC` 关键字，系统会**在创建时验证并记录依赖关系**
- **依赖感知**：系统知道函数 A 被函数 B 调用，B 被函数 C 调用
- **删除保护**：尝试删除基函数 A 时，系统抛出错误：**"有对象依赖于 A"**

**语法示例**：
```sql
CREATE FUNCTION C() ATOMIC AS $$ 
  -- 函数体调用 B
$$ LANGUAGE SQL;
```

#### **级联删除（CASCADE）**
- **类比外键约束**：如同外键的 `ON DELETE CASCADE`
- **行为**：`DROP FUNCTION A CASCADE;` 会递归删除所有依赖 A 的函数（B 和 C）
- **目的**：**避免悬空引用**（dangling pointer），保证数据完整性

```sql
-- 级联删除：删除 A 及所有依赖它的函数
DROP FUNCTION A CASCADE;
```

### **为什么 `ATOMIC` 不是默认行为？**

- **历史原因**：`ATOMIC` 关键字是近 2-3 年才加入 PostgreSQL 的
- **技术原因**：传统 SQL 函数采用**惰性解析**（Lazy Parsing）策略
  - 函数体被当作**字符串**存储在系统目录中
  - **仅在运行时**（调用时）才进行解析、编译和依赖检查
  - 这种设计简化了函数创建，但牺牲了依赖保护

### **SQL 函数的本质：宏级别的代码复用**

- **类比**：SQL 函数就像**宏**——把一组 SQL 查询复制粘贴进一个黑盒
- **局限性**：虽然能复用代码，但**表达逻辑的能力有限**
  - 只能封装 SQL 查询
  - 不能实现复杂的编程逻辑（如循环、条件分支等原生 SQL 不支持的运算）

### **关键总结**

| 特性 | 非原子函数（无 `ATOMIC`） | 原子函数（带 `ATOMIC`） |
|------|--------------------------|------------------------|
| **依赖追踪** | ❌ 不追踪 | ✅ 创建时验证 |
| **创建时检查** | ❌ 不检查依赖是否存在 | ✅ 检查所有引用对象 |
| **删除基对象** | ✅ 允许删除，但运行时崩溃 | ❌ 阻止删除，或级联删除 |
| **使用场景** | 快速原型、复杂动态SQL | 生产环境、完整性要求高 |

### **实用建议**

- **开发阶段**：使用非原子函数，方便频繁修改和重建
- **生产环境**：**强烈建议使用 `ATOMIC`**，避免"依赖地狱"和运行时意外崩溃
- **维护操作**：使用 `CASCADE` 时要谨慎，确保了解完整依赖链再执行删除

## 第 8 部分

### 数据库中的用户定义函数（UDF）与存储过程语言

#### 核心概念：超越SQL的编程能力
- **用户定义函数（UDF）** 允许将 **一系列SQL查询打包到单个函数** 中，复用逻辑。
  - 但UDF仍然受限于 **SQL自身的表达能力**（只能描述SQL支持的操作）。
- 因此，部分数据库系统支持 **用其他语言（过程式/指令式）编写UDF**，突破SQL的局限性。

#### 关键术语：UDF语言与SQL标准
- **SQL标准定义了一个UDF编程语言**，名为 **SQL PSM（SQL Persistent Stored Module）**。
  - 这是一个20世纪90年代的企业级技术，名字比较古老。
  - **核心限制**：几乎没有数据库厂商真正实现该标准，而是各自推出自己的方言。

#### 主流数据库的UDF语言实现

##### Oracle：PL/SQL
- **最接近SQL标准的实现**，是数据库领域的老牌语言。

##### PostgreSQL：PL/pgSQL
- 基于Oracle的PL/SQL的**变体**，是PostgreSQL的默认过程式语言。

##### MySQL：不支持标准UDF语言
- 用户只能编写**C语言函数**或使用SQL函数，**没有**类似于PL/SQL的存储过程语言。

##### SQL Server / Sybase：Transact-SQL (T-SQL)
- **历史渊源**：
  - SQL Server最初于1990年代早期从Sybase**购买许可证**，用于移植到Windows NT。
  - 双方协议允许对源码进行操作，结果在1990年代中期**硬分叉（Hard Fork）**。
  - 微软重写了SQL Server（雇佣了之前研究火山模型（Volcano Model）的专家等人），而Sybase被SAP收购后进入维护模式。
  - 因此，SQL Server继承了Sybase在1980年代（早于SQL标准）的语言 **T-SQL**。
- **语言特点**：T-SQL语法 **类似Ada或Pascal语言**，与标准SQL存在方言差异。

#### 语言示例：类似Ada/Pascal的语法结构
- **重点**：这类UDF语言（如T-SQL、PL/SQL）的代码风格倾向 **块状结构（Block Structure）**，与C/Java等语言不同。
- **直观印象**：
  ```sql
  -- 典型的PL/SQL或T-SQL块结构示例（非精确语法）
  CREATE FUNCTION calculate_discount(price NUMERIC, rate NUMERIC)
  RETURNS NUMERIC AS
  BEGIN
      DECLARE discounted_price NUMERIC;
      SET discounted_price = price * (1 - rate);
      RETURN discounted_price;
  END;
  ```
  - 使用 `BEGIN...END` 定义代码块，`DECLARE` 声明局部变量，**强类型**风格明显。

#### 总结：核心记忆点
- **SQL PSM** 是标准但未被广泛采用；实际开发中接触到的UDF语言是 **PL/SQL（Oracle/PostgreSQL）** 或 **T-SQL（SQL Server）**。
- **历史原因**导致语言多样性（如SQL Server与Sybase的分叉）。
- **语法类似Pascal/Ada**，而非你熟悉的C系语法（如 `{}` 和 `;` 用法不同）。

## 第 9 部分

### PostgreSQL UDF 与 PL/pgSQL 语言特性

#### UDF 语言选择与安全性

- **核心概念**：数据库系统支持多种**编程语言**编写用户自定义函数（UDF），不同语言对**安全性**要求不同
- **PostgreSQL 默认语言**：**PL/pgSQL**（类似 Ada/Pascal 语法）
  - 标准制定者偏好 Ada 语言风格
  - 语法结构清晰，类似 Pascal
- **其他系统语言支持**：
  - Postgres：Python、Perl、Java、C
  - Amazon Trusted Language Extensions：支持 Python/Perl 的安全运行环境

#### C 语言 UDF 的安全性挑战

- **关键问题**：C 语言编写的 UDF 可以**执行任意内存操作**
  - 能使用 `malloc()` 分配内存
  - 可访问任意内存地址
- **解决方案对比**：
  - **Oracle**：在**沙箱隔离进程**中运行 C UDF
    - 使用 IPC（进程间通信）传递数据
    - 防止程序破坏数据库堆内存/崩溃服务器
  - **PostgreSQL**：直接**链接共享对象**，无安全限制
    - 编译成 `.so` 文件后直接加载
    - **风险**：可能破坏内存堆，导致数据损坏或服务器崩溃

#### PL/pgSQL 编程示例

- **函数定义结构**：
  ```sql
  CREATE FUNCTION get_foo() 
  RETURNS SETOF foo 
  LANGUAGE plpgsql
  ```
  - **RETURNS SETOF**：返回多行结果集
  - **LANGUAGE**：指定使用 PL/pgSQL 语言

- **关键语法特性**：
  - **强类型系统**：必须明确定义返回值类型
  - **声明式与过程式结合**：支持循环、条件判断等控制流
  - **SQL 集成**：可直接嵌入 SQL 语句

#### 实际应用建议

- **安全性原则**：
  - 优先使用 PL/pgSQL 编写 UDF（**安全**内存管理）
  - 避免在 PostgreSQL 中直接运行 C UDF（除非完全信任代码）
  - 使用沙箱环境运行不安全的 UDF 实现

- **性能权衡**：
  - C 语言 UDF 性能最佳，但安全性最差
  - PL/pgSQL 在安全性和性能间取得平衡
  - 复杂逻辑建议用外部语言实现，通过 API 调用

- **最佳实践**：
  - 生产环境使用**受信任的语言扩展**
  - 对第三方 UDF 进行**安全审计**
  - 考虑使用 AWS Trusted Language Extensions 等托管方案

## 第 10 部分

### PostgreSQL 中的 PL/pgSQL 函数深入解析

#### 1. **核心概念：PL/pgSQL 与 SQL 函数的区别**

- **SQL 函数**：函数体是单一 SQL 语句，隐式返回该语句的查询结果（最后一条查询自动成为返回值），解析器可以在函数**定义时**解析与优化。
- **PL/pgSQL 函数**：函数体是一个**过程化块**，支持变量声明、循环、条件分支、动态 SQL 等。
  - **必须显式指定返回子句**（`RETURNS SETOF foo` 或 `RETURNS INTEGER`）。
  - **函数体用双美元符号 `$$` 包裹**，表示一个字符串，可包含任意 PL/pgSQL 代码。
  - **函数在调用时执行**，解析器无法在函数定义时预知内部实际执行的 SQL，因为可能依赖运行时的状态（如条件分支、动态拼接的 SQL 字符串）。

#### 2. **关键词与返回值**

- **`RETURNS SETOF foo`**：表示函数返回一组元组，元组的模式与 `foo` 表相同（列名、类型一致）。
- **`RETURNS INTEGER`**：返回单个整数值。
- 在 PL/pgSQL 中，**不能隐式返回最后一条查询结果**，必须使用 `RETURN QUERY` 或 `RETURN` 显式返回。

#### 3. **动态 SQL（不推荐但存在于 PL/pgSQL 中）**

- **字符串拼接 + `EXECUTE`**：你可以像 .NET 的 `StringBuilder` 一样拼接 SQL 语句字符串，然后用 `EXECUTE` 执行。这会导致：
  - 函数定义时无法解析内部查询（无语法检测、无优化）。
  - 每次调用都可能生成不同的 SQL（例如依赖 if/else 条件或运行时数据）。
  - **这是 PL/pgSQL “棘手” 的特性**，因为它绕过了 PostgreSQL 编译期优化。

#### 4. **复杂示例：带循环的求和函数**

```plpgsql
CREATE OR REPLACE FUNCTION sum_foo_values()
RETURNS INTEGER AS $$
DECLARE
    foo_record RECORD;   -- 声明一个记录变量（类似游标的一行）
    out INTEGER := 0;    -- 声明输出变量并初始化
BEGIN
    -- 遍历 foo 表中每一行
    FOR foo_record IN (SELECT * FROM foo) LOOP
        -- 假设 foo 表有一个整型列，此处做累加
        out := out + foo_record.some_integer_column;
    END LOOP;
    RETURN out;          -- 返回累加结果
END;
$$ LANGUAGE plpgsql;
```

**执行方式**：`SELECT sum_foo_values();` 或 `SELECT * FROM sum_foo_values();`（取决于返回值是否为集合）。

- **变量声明前置**（受 Ada 语言影响，类似 80 年代风格）：所有变量必须在 `DECLARE` 块中定义。
- **`RECORD` 类型**：一种伪类型，表示任意行结构，通过 `FOR` 循环中的 `SELECT` 查询动态决定其字段。
- **`FOR ... IN ... LOOP`**：遍历查询结果，每次迭代填充 `foo_record` 变量。

#### 5. **函数调用示例**

```sql
-- 调用返回集合的函数
SELECT * FROM some_foo(1);

-- 调用返回标量的函数
SELECT sum_foo_values();
```

- 当函数定义为 `RETURNS SETOF foo` 时，在 `FROM` 子句中调用，结果如同查询一张表。
- 当函数定义为 `RETURNS INTEGER` 时，直接在 `SELECT` 后调用，返回单个值。

## 第 11 部分

## 用户定义函数 (UDF) 的调用、查询与性能分析

### 核心概念：UDF 作为“黑盒”嵌套查询

- **调用方式**：UDF 可以像普通函数一样在 `SELECT` 子句中使用，计算并返回一个标量值。调用时内部会嵌套执行另一个查询。
- **标量返回**：当 UDF 的**返回类型是整数/标量**时，它**可以直接用于 `WHERE` 子句**，例如：`WHERE id < someFoo(1)`。这使得复杂的逻辑可以在查询条件中复用。
- **本质行为**：UDF 内部调用的查询实际上是**一个嵌套的查询（nested query）**。函数执行时，会展开并运行其内部的查询逻辑。

### 核心问题：UDF 对优化器是“黑盒”

- **优化器盲区**：当你在查询中使用 `EXPLAIN` 分析执行计划时，数据库优化器只能看到 UDF 是一个黑盒调用。它无法“看穿”UDF 内部，去分析里面是否还有另一个查询，以及它的具体逻辑。
- **优化限制**：因为无法预知 UDF 内部行为，优化器**无法执行常规的优化技巧**（例如：将内部的子查询提前提取、重写为连接、或应用谓词下推等）。
- **性能影响**：这意味着即使是简单的逻辑，如果封装在 UDF 中，数据库也可能选择最朴素的执行方式（例如全表扫描 + 逐行调用函数过滤），而无法进行跨查询边界的全局优化。

### UDF 的执行路径不确定性

- **依赖因素**：UDF 的**实际执行路径**（即内部会执行哪些查询、走什么索引）不仅仅取决于输入参数，还**取决于数据库的当前状态**。因为 UDF 内部可以执行任意查询来读取数据库状态。
- **运行时决定**：与普通 SQL 查询在优化阶段就已确定执行计划不同，UDF 的具体行为要到**运行时才能真正确定**。

### 应用开发者视角 vs. 数据库系统视角

- **数据库系统视角**（劣势）：
    - UDF 是**性能更差的**选择，因为它是优化器的黑盒，导致无法进行全局优化。
    - 对于嵌套查询，优化器无法“看穿”UDF，无法将其内部查询与外层查询合并优化。

- **应用开发者视角**（潜在优势）：
    - UDF 允许开发者在**不同应用实现之间复用复杂逻辑**（例如，移动应用和 Web 应用共享同一个后端数据库）。
    - 某些逻辑用 UDF 实现**更容易、更直观**，避免了在不同编程语言中重写相同的复杂业务规则。

### 关键术语与总结

- **UDF (用户定义函数)**：在数据库内部定义的可调用函数，可以包含查询逻辑。
- **黑盒 (Black Box)**：优化器无法查看或分析 UDF 内部结构，这是其主要性能缺陷。
- **嵌套查询 (Nested Query)**：UDF 内部实际会执行一个查询，从行为上看类似于在 SQL 中写的子查询。
- **执行路径 (Execution Path)**：UDF 的具体执行逻辑在运行时才确定，依赖于参数和数据库状态。
- **优化器 (Optimizer)**：数据库的查询计划生成器，UDF 的存在使其优化能力大打折扣。

## 第 12 部分

### 用户自定义函数（UDFs）的利与弊

#### 优势：逻辑复用与性能优化

*   **核心概念：UDF 的核心价值在于“一次编写，到处运行” (Write Once, Run Anywhere)。**
    *   **逻辑复用 (Logic Reuse)：** 想象一下，你既需要开发手机 App，又需要做 Web 端。如果有一个非常复杂的业务逻辑（比如计算折扣、物流费用），你不需要在 Kotlin、Swift、JavaScript 等多种语言里分别实现一遍。
        *   **做法：** 你只需要把这个复杂逻辑写成一个 **UDF**，部署在 **数据库服务器** 上。
        *   **效果：** 不管是手机 App 还是 Web 端，都直接调用这个 UDF 来完成计算。**逻辑统一，维护方便**。
    *   **减少网络往返 (Reduce Round-Trips)：** 传统模式下，应用需要向数据库发送查询，拿到数据，然后在应用层用代码完成逻辑计算。如果逻辑还需要数据库里的其他数据，应用层又得再发一个查询。
        *   **UDF 方案：** 把整个逻辑“推”到数据库服务器内部执行。数据库可以直接操作内部数据完成所有计算，只将最终结果返回给应用。**这大幅减少了应用和数据库之间的“请求-回答”次数**。
    *   **表达能力强 (Expressiveness)：** 某些复杂的业务规则或数据处理逻辑（例如复杂的正则匹配、自定义聚合）用 **SQL 很难写**，或者写出来**晦涩难懂**。用 UDF（特别是 PL/SQL 或过程化语言）来表达，代码会**更清晰、更容易理解和维护**。

#### 劣势：查询优化器“黑盒”与并行性障碍

*   **核心概念：UDF 对查询优化器来说是一个“黑盒”，导致了严重的性能不确定性。**
    *   **黑盒优化器 (Black-Box Optimizer)：** 绝大多数数据库的查询优化器（这里的例外仅指少数几个，比如某特定数据库）都无法“看穿”UDF 的内部代码。
        *   **问题 1：无法评估选择性 (Selectivity Estimation)。** 当你把 UDF 用在 `WHERE` 或 `JOIN` 条件里时（例如 `WHERE udf\_check\_match(column) = true`），优化器完全不知道这个 UDF 会过滤掉多少行数据（即**选择率**）。
        *   **后果：** 优化器无法判断是先对这个表做过滤，还是先去做连接。这直接导致**优化器可能选择最差的连接顺序**（Join Order），把大量数据在早期就进行笛卡尔积，性能极差。理想情况是**尽早过滤掉不相关的数据** (Filter out sooner rather than later)，但黑盒让优化器无从下手。
        *   **问题 2：无法评估成本 (Cost Estimation)。** 无论是用 C 语言这种编译型语言编写的 UDF，还是用 PL/SQL 这种过程化语言编写的，系统都无法估计其执行时间。对于 C 语言，未知性更大；对于 PL/SQL，系统也仅仅是“不猜了”。
    <!---->
    *   **并行执行障碍 (Parallelism Challenges)：**
        *   **核心冲突：缺乏原子性保证。** 因为无法检查 UDF 内部逻辑，你**不知道它是否包含了更新操作、事务操作或全局变量修改**。如果有，这些操作在不同线程间并行执行会造成数据损坏。
        *   **历史/保守做法：** 为了绝对安全，许多数据库（包括早期版本的知名数据库，即使 PostgreSQL 已经对此做了很大改进）会**默认将包含 UDF 的查询降级为单线程执行**。这完全无法利用现代多核 CPU 的性能优势。
        *   **“诚实”提示：** 像 PostgreSQL 等系统允许用户通过 `HINT`（提示）来声明当前 UDF 是“可并行执行”的，前提是你作为开发者必须保证它确实是无副作用的。这是一种“我向你保证没问题，请放心并行”的约定，优化器会信任这个提示。

## 第 13 部分

### 用户自定义函数（UDF）的执行约束与挑战

#### 1. UDF的**可预测性**与**排序依赖**
- **核心概念**：数据库系统无法预先知道UDF内部执行的逻辑（如数据更新顺序、副作用），这导致优化器难以进行确定性分析。
- **关键差异**：
  - **SQL Server**：强制要求UDF必须是**不可变的**（immutable），即不能修改数据库状态或依赖外部可变数据。
  - **PostgreSQL**：允许UDF执行任意操作（包括更新数据）。但支持用户通过**提示（hints）** 手动声明UDF的**可并行化**（parallelizable）或**不可变性**（immutable）属性，这有助于优化器做出更优的执行计划。
- **存储过程**：这是另一种函数类型，**允许更新数据**，与不可变的UDF在语义上有本质区别。

#### 2. **动态查询生成**的编译难题
- **核心问题**：当UDF内部通过拼接字符串等方式动态构建并执行查询（即“注入式查询”）时，数据库在编译阶段**无法解析**该SQL语句。
- **后果**：无法为这些动态查询生成执行计划、进行索引选择或安全验证（如SQL注入防御），导致系统不得不采用保守策略，例如**逐行调用（row-by-row invocation）**，牺牲了批处理优化机会。

#### 3. **向量化执行**与**逐行调用**的性能鸿沟
- **核心对比**：
  - **传统处理**：对于普通SQL操作，数据库可以使用**向量化执行**（vectorized execution）或批处理（batching），一次处理一批元组（batch of tuples），利用CPU缓存和SIMD指令加速。
  - **UDF处理**：由于UDF通常定义为**单行操作**（operate on a single row），且其内部逻辑对优化器不透明，数据库被迫采用**逐行调用**（row-by-row invocation），即对每一行单独调用UDF。
- **更好的实践**：若能将UDF设计为**接受一批元组作为输入**，并一次性处理完，可以大幅减少函数调用的上下文切换开销，显著提升吞吐量。这要求UDF支持**批量接口**（batch interface）。

#### 4. **内存使用与分配**
- **UDF内部分配（如C语言`malloc`）**：分配的内存来自**操作系统**（OS）的进程堆，由UDF自身管理，与数据库的缓冲池（buffer pool）无关。
- **查询结果处理**：
  - 在执行查询期间，数据访问仍通过数据库的**标准缓冲池管理器**（如PostgreSQL的shared buffers）进行。
  - 但**最终结果**必须从数据库服务器的内存空间**复制**到用户进程（client process）中。此时，会使用标准内存分配函数（如`malloc`）在用户进程内部分配内存。
  - **结论**：所采用的编程语言（如C、PL/pgSQL）本身并不改变这一内存分配模型，关键在于数据库运行时环境的内存管理策略。

## 第 14 部分

### UDF 的“黑盒”困境与优化局限性

#### 1. 核心问题：UDF 作为“黑盒”对数据库优化的阻碍
- **核心概念**：当用户定义函数（UDF）使用**通用编程语言**（如 PL/pgSQL、C）时，数据库优化器无法“看穿”其内部逻辑。
- **关键术语**：**"Black Box Problem" (黑盒问题)**、**"Optimizer Blindness" (优化器盲区)**。
- 为什么是“黑盒”？
  - 数据库执行 UDF 时，是**逐元组**调用（tuple-at-a-time），即使 UDF 内部包含多个 SQL 查询，优化器也无法将两个查询**合并优化**为一个。
  - 例如：UDF 内先后调用了 `SELECT ...` 和 `UPDATE ...`，理论上可以合并成一条更优的语句，但优化器**不知道函数内部做了什么**，只能“照单执行”。

#### 2. 内存与资源管理差异
- **核心概念**：UDF 内的变量存储在**工作内存**（Work Memory）中，而非数据库的**缓冲池**（Buffer Pool）。
- **关键术语**：**"Work Memory"** vs **"Buffer Pool"**。
- 具体细节：
  - 当 Postgres 运行 UDF 时，内部的局部变量**不经过缓冲池**，直接放在进程的私有工作区。
  - 这意味着：UDF 内部的临时数据无法享受缓冲池的**缓存、预取、淘汰策略**，性能完全依赖函数本身的实现。

#### 3. 现有优化尝试：加速 UDF 执行，但未解决本质问题
- **核心概念**：已有多种方法尝试**加速 UDF 执行**，但都**没有解决“黑盒”问题**，即优化器依然无法对函数内部进行全局优化。
- **现有方案举例**：
  - **专用 UDF 语言（如 SingleStore/MemSQL 的 MPL）**：允许用户定义“并行函数”，但仅限于该语言生态。
  - **JIT 编译（SQL Server 等）**：将 UDF 编译为本地机器码（Native Code），使单次调用更快。**但它并未改变“逐元组调用”和“无法跨查询优化”的格局**。

#### 4. 为什么不能像宏（Macro）一样内联优化？
- **核心概念**：如果 UDF 内部只包含**纯 SQL 语句**，理论上可以像宏一样被“展开”（inline），让优化器全局优化。但现实是，数据库（如 Postgres）往往**不够智能**，无法自动检测并展开。
- **关键术语**：**"SQL Function Inlining"**。
- 具体示例（以 `getFoo(1)` 为例）：
  - 假设 `getFoo` 内部只执行一条简单查询（如 `SELECT * FROM t WHERE id = $1`），在理想世界里，执行 `SELECT getFoo(1)` 应该被优化器自动替换为 `SELECT * FROM t WHERE id = 1`。
  - **现实问题**：执行 `EXPLAIN SELECT getFoo(1)` 时，Postgres 通常**不会告诉你函数内部的具体执行计划**（“it doesn't tell me anything”）。
  - **原因**：优化器未能识别出该 UDF 只是一个“单查询”宏，仍然将其视为一个**不可分解的黑盒**。

#### 5. 总结：数据库 UDF 的优化瓶颈
- **核心矛盾**：通用性 vs 可优化性。
  - **通用语言 UDF**（如 C、PL/pgSQL）：功能强大，但牺牲了数据库的优化能力。
  - **纯 SQL 函数**：理论上可被内联优化，但多数数据库实现**未自动支持**，仍需人工重写或依赖特定扩展。
- **工程师应知**：
  - 当需要高性能时，**尽量避免在 UDF 中封装多条可合并的查询**。
  - 若 UDF 不可避免，考虑**手动将函数逻辑外提**到调用上下文中，让优化器掌控全局。
  - 关注数据库新特性（如 Postgres 13+ 对 SQL 函数自动内联的改进），但需验证实际效果。

### 函数扫描与运行时优化

*   **核心概念：函数扫描**：当您在查询中使用一个UDF（例如 `SELECT * FROM get_foo()`）时，PostgreSQL 不会将其视为一个普通的数据表，而是会生成一个**`Function Scan`（函数扫描）**节点在查询计划中。
*   **核心概念：运行时优化**：与普通查询（会在 `EXPLAIN` 时显示索引扫描等详细信息）不同，`Function Scan` 对应的函数体**不会被预先优化和计划**。
    *   **原因**：优化器在规划外部查询时，并**不了解函数内部的具体代码**（例如它包含哪些SQL语句），尤其是当函数体包含过程式语言（PL/pgSQL）的逻辑时。
    *   **结论**：函数内部的SQL语句（如 `SELECT`、`INSERT`）会**在函数被实际调用的运行时，才进行解析和优化**。因此，`EXPLAIN` 外部查询不会显示函数内部的索引扫描细节。

### UDF语言的多样性及其对执行的影响

*   **核心概念：语言类型决定执行方式**
    *   **纯SQL语言函数**：如果UDF是纯SQL语言，它本质上可以看作是一个“宏”。如果函数体内**只包含一条**SQL语句，优化器有可能（理论上）尝试将其内联展开到外部查询中进行统一优化。但这部分内容暗示，即使如此，这种优化也并非总是发生。
    *   **过程式语言函数（如PL/pgSQL）**：这是更强大的形式。它允许在函数体内使用：
        *   **流程控制**：`IF` 子句、循环。
        *   **多条SQL语句**：可以连续执行多个 `INSERT`、`UPDATE`、`SELECT` 等操作。
        *   **变量和逻辑**：提供远超普通SQL的编程能力。
*   **关键区别：宏 (Macro) vs. 子程序 (Subroutine)**
    *   **宏 (Macro - 单条SQL)**：优化器可能将其内联，但受限于优化器的能力。
    *   **子程序 (Subroutine - PL/pgSQL或多条SQL)**：**不可能**内联。它必须作为一个独立的执行上下文被调用。

### 事务性上下文与原子性

*   **核心概念：事务上下文**：当外部查询调用一个UDF时，**整个调用过程（包括外部查询）都处于同一个事务上下文中**。
    *   执行流程是：暂停外部查询 -> 执行UDF内部的SQL语句 -> 恢复外部查询。
    *   **原子性保证**：如果UDF内部的**任何一条SQL语句失败**，整个事务（包括外部查询和之前UDF执行成功的部分）**都会被回滚**。
*   **示例解析**：
    1.  定义了一个包含 `INSERT` 和 `SELECT` 的UDF。
    2.  外部调用该UDF，`INSERT` 操作成功执行（向 `foo` 表插入了一行）。
    3.  然而，该 `INSERT` 因为违反主键约束（`primary key`）而失败。
    4.  **结果**：不仅该 `SELECT` 操作失败，整个事务被回滚。`foo` 表中之前成功插入的行也会被撤消。
*   **公式/规则**：**`UDF调用失败` → `整个事务回滚`**。

### 总结对比

| 特性 | 普通SQL查询 | SQL语言UDF (单条) | PL/pgSQL或多条SQL UDF |
| :--- | :--- | :--- | :--- |
| **执行方式** | 直接由优化器规划 | 类似“宏”，可尝试内联 | 独立子程序，**无法**内联 |
| **优化时机** | 查询计划时 (Planning) | **运行时** (Runtime) | **运行时** (Runtime) |
| **逻辑能力** | 单条语句 | 单条语句 | IF/循环/多条语句，逻辑强 |
| **事务性** | 自身即是事务 | 在调用它的外部事务中执行 | 在调用它的外部事务中执行，**任何错误都回滚整个事务** |
| **`EXPLAIN`** | 显示索引扫描等节点 | 显示 `Function Scan` 节点 | 显示 `Function Scan` 节点，内部细节不可见 |

#### **UDF的性能陷阱与优化原理**

**核心概念**：
- **UDF（用户自定义函数）**：看似便捷，但在数据库系统中是“黑盒”，优化器无法理解其内部逻辑
- **性能对比**：纯SQL查询（秒级） vs 相同功能UDF（耗时13小时+），差距可达4-5个数量级

**为什么UDF如此缓慢**：
- **上下文切换惩罚**：每处理一行数据，数据库引擎需要切换执行上下文去调用UDF函数
- **优化器盲区**：优化器无法内联、并行化、或重排UDF内的操作（如JOIN无法并行执行）
- **嵌套调用开销**：若UDF内部再执行SQL查询，每行都会触发新的查询，形成O(n)甚至O(n²)的嵌套执行

**微软的突破性解决方案——Freud方法**：
- **核心技术**：将UDF解析为**关系代数表达式**，然后内联回外层查询计划中
- **关键步骤**：
  1. 解析UDF的SQL语句转换为关系代数树
  2. 将关系代数树注入外部查询的查询计划
  3. 应用所有标准优化器技巧（谓词下推、连接重排等）
- **重要性**：被业界称为“过去十年数据库领域最佳创意之一”，微软已申请专利

### 存储过程与函数：核心概念与关键区别

#### **1. 声明与权限控制**
- **存储过程 (Procedure)** : 可执行任意数据库操作，包括更新表、修改数据。
- **函数 (Function)** : 在 SQL Server 中默认是**只读**的，不能修改数据库状态。
- **核心要点**：声明时需明确类型，决定能否写入数据。

#### **2. 核心优势：消除网络往返**
- **传统模式**：客户端与数据库多次交互（SELECT→检查→UPDATE→UPDATE），每次交互产生网络往返（Round Trip）。
- **存储过程模式**：将 SQL 逻辑嵌入服务器端，应用只需一次调用（`CALL` 命令），**整个事务在服务器内完成**。
- **关键收益**：显著减少延迟，提升性能，尤其适合复杂事务。

#### **3. 函数与存储过程的语言支持**
- **函数实现语言**：
  - 常见：纯 SQL 命令
  - 扩展支持：PL/pgSQL、C、Python 及其他语言（取决于数据库实现）
- **存储过程实现语言**：同样支持多种语言，可包含条件判断、循环等任意逻辑。

---

### 存储过程实战示例：资金转账事务

#### **核心逻辑步骤**
```sql
-- 1. 声明变量
DECLARE @balance DECIMAL;
DECLARE @sender_id INT = 123;
DECLARE @receiver_id INT = 456;
DECLARE @amount DECIMAL = 100.00;

-- 2. 查询并存入变量
SELECT @balance = balance FROM accounts
WHERE account_id = @sender_id;

-- 3. 条件判断
IF @balance < @amount
    RAISE EXCEPTION 'Insufficient funds';  -- 抛出错误，中止事务

-- 4. 执行更新
UPDATE accounts
SET balance = balance - @amount
WHERE account_id = @sender_id;

UPDATE accounts
SET balance = balance + @amount
WHERE account_id = @receiver_id;

-- 5. 提交事务
COMMIT;
```

#### **关键语法与功能**
- **`INTO` 子句**：将 SELECT 结果直接赋给变量。
- **条件控制**：`IF...ELSE` 处理业务逻辑。
- **异常处理**：`RAISE EXCEPTION`（或类似语法）用于**主动回滚**并报错。
- **事务控制**：`COMMIT` 确认更改，确保**原子性**。

#### **与传统客户端代码对比**
| 实现方式 | 网络交互次数 | 风险与性能 |
|---------|------------|-----------|
| 客户端代码 | 3 次（SELECT→UPDATE1→UPDATE2） | 延迟高，中间状态易受干扰 |
| 存储过程 | 1 次调用 | 低延迟，隔离性好，事务完整 |

---

### 存储过程 vs. 函数：选择依据

| 特性 | 存储过程 (Procedure) | 函数 (Function) |
|------|---------------------|-----------------|
| 数据库修改 | 允许 | 不允许（纯计算/查询） |
| 事务控制 | 可包含 COMMIT/ROLLBACK | 不能包含事务语句 |
| 调用方式 | `CALL procedure_name` | `SELECT function_name()` |
| 返回值 | 多个结果集/输出参数 | 单个值（标量）或表 |

---

### 最佳实践总结
- **何时用存储过程**：需要**修改数据**、**复杂业务逻辑**、**多步事务**。
- **何时用函数**：**只读查询**、**计算逻辑**、**用于表达式或视图定义**。
- **性能关键点**：**将多次网络交互压缩为一次**是核心优化手段，尤其适用于游戏后端的高并发交易系统。

## 第 19 部分

### 存储过程与UDF（用户定义函数）的核心区别与应用

#### **1. 存储过程的本质：封装多步骤事务**
- **核心概念**：存储过程是**将多个数据库操作（如查询、更新、条件判断）打包成一个完整事务的机制**。它模拟了应用程序中的“高级业务操作”，例如“从账户A转账$50到账户B”。
- **关键特性**：
  - **事务性**：整个操作**要么全部成功，要么全部失败**（原子性）。例如：若账户A余额不足，则抛出异常，不执行任何修改。
  - **条件逻辑内置**：可在过程中添加`IF/THEN`等逻辑（如“如果余额<金额，则报错”），避免应用层重复检查。
  - **单次调用，多步执行**：应用只需调用一次`CALL transfer(...)`，数据库负责内部所有步骤。
- **示例流程**：
  ```sql
  -- 创建账户表
  CREATE TABLE accounts (name TEXT, balance INT);
  INSERT INTO accounts VALUES ('Mushu', 100), ('ThreeFingers', 50);
  
  -- 创建转账存储过程
  CREATE PROCEDURE transfer(from_acct TEXT, to_acct TEXT, amount INT)
  LANGUAGE plpgsql
  AS $$
  BEGIN
      -- 检查余额
      IF (SELECT balance FROM accounts WHERE name = from_acct) < amount THEN
          RAISE EXCEPTION '% 余额不足', from_acct;
      END IF;
      -- 先扣钱
      UPDATE accounts SET balance = balance - amount WHERE name = from_acct;
      -- 后加钱
      UPDATE accounts SET balance = balance + amount WHERE name = to_acct;
  END;
  $$;
  ```
  - **调用**：`CALL transfer('Mushu', 'ThreeFingers', 50);` → 成功后查询表，余额已更新。
  - **错误处理**：若调用`transfer('Mushu', 'ThreeFingers', 1000);`，则抛异常“Mushu 余额不足”，事务回滚。

#### **2. UDF vs 存储过程：设计意图不同**
- **UDF（用户定义函数）**：
  - **用途**：**作为查询（Query）的一部分，执行只读计算**。例如：计算税收、拼接字符串、或对表中每一行做转换。
  - **核心限制**：PostgreSQL 默认**只允许UDF为只读（READ ONLY）**，不允许写数据库或修改数据。
  - **典型用法**：在`SELECT`语句中调用，返回单值或表。
    ```sql
    -- 返回税后价格（只读）
    CREATE FUNCTION after_tax(price NUMERIC) RETURNS NUMERIC AS $$
        SELECT price * 1.10;
    $$ LANGUAGE SQL;
    -- 用法：SELECT after_tax(100); -- 返回110
    ```
- **存储过程**：
  - **用途**：**执行包含多个写操作（INSERT/UPDATE/DELETE）的完整事务**。它控制事务的生命周期（BEGIN/COMMIT/ROLLBACK）。
  - **典型场景**：转账、订单结算、数据迁移等需要**原子性多步骤修改**的操作。

#### **3. 为什么要用存储过程而不是全用应用层代码？**
- **现状与矛盾**：尽管存储过程能“减少应用层代码的调用次数”，但**实际使用较少**。原因如下：
  - **语言陈旧**：PL/SQL（PostgreSQL）、T-SQL（SQL Server）等语言语法**落后于现代编程语言**（如Python、C#），缺乏模块化、异常处理、类型推断等特性。
  - **调试困难**：无法像应用代码那样轻松打断点、打印日志。
  - **版本控制混乱**：存储过程通常存储在数据库中，难以像Git一样追踪变更。
  - **性能权衡**：虽然减少网络往返（单次调用代替多次REST API调用），但复杂数据库端计算可能增加数据库负载，降低扩展性。

#### **4. 关键总结：何时选择存储过程 vs UDF？**
| 场景 | 推荐使用 | 原因 |
|------|----------|------|
| 需修改数据（写操作） | **存储过程** | 事务控制、原子性、错误回滚 |
| 只读计算（如SELECT内） | **UDF** | 嵌入查询、轻量级、无副作用 |
| 跨多个表的高频业务逻辑 | **存储过程** | 减少网络开销，封装复杂规则 |
| 简单值转换（如格式化） | **UDF** | 可在查询中直接调用，无需独立事务 |

## 数据库触发器与 UDF (用户自定义函数)

### 触发器 (Trigger) 的核心概念

- **定义**: 触发器是一种**事件驱动的机制**，当数据库内的特定数据或表发生某个事件时，自动执行预定义的动作（通常是 UDF）。
- **三大要素**:
  - **事件类型**: 引起触发器执行的操作（如 INSERT、UPDATE、DELETE）。
  - **事件范围**: 操作影响的范围（例如是针对每一行还是整个 SQL 语句）。
  - **调用时机**: UDF 是在事件**之前 (BEFORE)** 还是**之后 (AFTER)** 触发，以及是**逐行**触发还是**逐语句**触发。

### 触发器实战案例：审计日志

**场景**：需要自动记录主表 `Foo` 中哪些旧值被更新覆盖，用于审计追踪。

**1. 主表 Foo**
- 结构：包含 `ID` 和 `value` 字段。

**2. 审计表 Audit**
- 结构：
  - `audit_id`：自增主键或唯一标识。
  - `foo_id`：外键，指向 `Foo.ID`（**引用完整性**）。
  - `old_value`：被覆盖前的原始值。
  - `change_timestamp`：修改发生的时间戳。

**3. UDF 函数定义**
- 当 `Foo` 表发生更新时，触发器调用此函数。
- **关键隐式变量**:
  - **`NEW`**: 代表即将插入或更新后的新值。
  - **`OLD`**: 代表更新或删除前的旧值。
- **函数逻辑核心**: 从更新操作中**提取 `OLD` 中的旧值**，并插入到审计表中。

### 触发器与 SQL 语句的执行顺序

- **BEFORE TRIGGER**: 在数据库实际修改数据**之前**运行，常用于验证或预处理数据。
- **AFTER TRIGGER**: 在数据修改**完成之后**运行，常用于日志记录或级联操作。
- **FOR EACH ROW vs. FOR EACH STATEMENT**:
  - **FOR EACH ROW**: 每修改一行就触发一次 UDF。
  - **FOR EACH STATEMENT**: 整个 SQL 语句执行完毕后只触发一次 UDF。

### 隐性规则与注意点

- 触发器中使用的 `NEW` 和 `OLD` 变量是**数据库系统隐式提供**的，无需人工声明。
- 不同的数据库实现（如 SQL Server vs. PostgreSQL）对只读约束、嵌套触发器行为的处理可能不同。
- **现代编程语言期望的优质特性**（如严格的只读模式）在数据库触发器中往往并非默认强制。

### 总结公式（逻辑结构）

```latex
\text{Trigger} = \text{Event (DML)} + \text{Scope (ROW | STATEMENT)} + \text{Timing (BEFORE | AFTER)} + \text{Action (UDF)}
```

## 第 21 部分

### 数据库触发器 (Triggers) 实现自动审计

#### 核心概念与关键术语
- **触发器 (Trigger)**：一种特殊的数据库对象，在指定事件（如 UPDATE）发生前/后自动执行。
- **审计 (Auditing)**：记录数据变更历史，常用于合规、调试、安全追踪。
- **隐式变量 `NEW` 与 `OLD`**：
  - `NEW`：包含即将插入或更新后的新行数据。
  - `OLD`：包含更新或删除前的旧行数据。
  - 这些变量**不是显式传入参数**，而是由触发器系统自动提供。

#### 触发器函数（UDF）定义要点
- **函数返回类型必须为 `TRIGGER`**，系统才能识别并自动传入 `NEW`/`OLD`。
- **典型审计逻辑**：
  1. 从 `OLD` 中提取变更前的数据（如旧值 `OLD.val`）。
  2. 从 `NEW` 中获取新值。
  3. 将时间戳、旧值、新值、主键引用等插入审计表。
  4. **返回 `NEW`**，表示使用更新后的新值继续执行（若返回 NULL 可中止操作）。

```sql
-- 示例：审计函数
CREATE OR REPLACE FUNCTION log_foo_updates()
RETURNS TRIGGER AS $$
BEGIN
    INSERT INTO foo_audit (changed_at, old_val, new_val, foo_id)
    VALUES (NOW(), OLD.val, NEW.val, OLD.id);
    RETURN NEW;  -- 返回 NEW 使更新生效
END;
$$ LANGUAGE plpgsql;
```

#### 触发器的安装与执行
- **创建触发器命令**：
  - `BEFORE UPDATE`：在更新操作实际执行前触发。
  - `FOR EACH ROW`：对每行受影响的数据执行一次。
- **调用已定义的 UDF**：`EXECUTE FUNCTION log_foo_updates()`。

```sql
CREATE TRIGGER audit_foo_update
BEFORE UPDATE ON foo
FOR EACH ROW
EXECUTE FUNCTION log_foo_updates();
```

#### 效果演示
- **原表 `foo`**：更新 `id=4` 的记录，`val` 从旧值改为 `'YY'`。
- **审计表 `foo_audit`**：自动生成记录，包含：
  - 时间戳
  - 旧值（`'Y'`）
  - 新值（`'YY'`）
  - 被修改行的主键引用（`foo_id=4`）

#### 事务一致性
- **触发器在事务内部执行**：触发器操作与触发它的查询（外层查询）**属于同一个事务**。
- **行为等价**：执行 `UPDATE` 并触发审计，在事务层面等同于依次执行：
  1. `UPDATE` 原表
  2. `INSERT` 审计表
- **可回滚**：如果外层的 `UPDATE` 所在事务回滚（`ROLLBACK`），触发器的审计插入也会**自动撤销**，保证数据一致性。

```sql
-- 示例：事务回滚时审计记录也会消失
BEGIN;
UPDATE foo SET val = 'ZZ' WHERE id = 4;
-- 此时审计表有记录
ROLLBACK;
-- 审计表记录也消失，事务原子性保证
```

#### 补充说明
- **SQL 标准兼容性**：`NEW`/`OLD` 是 SQL 标准的一部分，多数支持触发器的数据库（如 PostgreSQL、Oracle、MySQL）均遵循此模型。
- **常见应用场景**：除审计外，还可用于自动更新时间戳、级联操作、数据完整性校验等。

## 第 22 部分

### 事务中的触发器与用户定义函数（UDF）语义

#### 核心挑战：触发器内的操作如何在事务回滚中被正确处理？

- **问题本质**：当用户更新表 `Foo` 时，触发器自动向审计表 `FooAudit` 插入了记录。如果事务随后回滚，数据库系统如何知道要撤销触发器插入的这条记录？
- **关键洞察**：**触发器是数据库系统内部**的、绑定到特定事件（如 `UPDATE`）的用户定义函数。
    - 执行触发器时，它执行的每条SQL命令（如 `INSERT`）都被数据库引擎**全程追踪**。
    - 数据库引擎**知道自己干了什么**：因为它负责执行这些命令，所以它知道在哪个表插入了哪条数据。
- **结论**：回滚时，数据库系统**不需要区分**操作是应用代码直接发起的，还是由触发器间接发起的。它只需知道“在这个事务范围内，我向表 `Foo` 插入了某条记录”，然后统一回滚。回滚机制是基于**事务日志**，而不是基于操作来源。

#### 核心概念：应用代码零干预

- **用户受益**：开发者**无需**在应用代码中手动编写“由于我更新了 `Foo`，所以我也要删除 `FooAudit` 中对应的记录”这类逻辑。
- **数据库封装**：查询计划在执行更新操作前会检查：“这张表上是否有 `UPDATE` 触发器？”。如果有，则自动串联执行。
- **回滚的原子性**：无论操作链条有多长（更新 `Foo` → 触发器插入 `FooAudit`），整个链条被视为一个**原子事务**。一旦 `ABORT`，所有副作用都被撤销。

#### 边界案例：UDF 的“外部副作用”与事务的不一致性

- **关键问题**：用户定义函数能否产生**数据库无法控制的副作用**？例如：**发送电子邮件**。
- **技术现实**：可以，但存在**危险**。
    - **问题**：如果 UDF 内部通过 `SMTP` 调用了发送邮件的代码，电子邮件**已经发出**。之后事务回滚，数据库可以撤销所有内部操作（如插入表），但**无法召回已经发出的邮件**。
    - **后果**：这造成了**事务语义违反**——应用层认为事务失败了，但外界（用户）已经收到了邮件，导致状态不一致。
- **最佳实践**：**不要**在事务内的触发器或 UDF 中执行具有不可回滚外部副作用（发送邮件、调用外部API、写入文件系统）的操作。如果需要，应设计为异步、事务外执行。

#### 总结要点

- **原理**：数据库通过**事务级日志**追踪所有操作，包括触发器产生的操作，实现完整回滚。
- **架构**：触发器将应用逻辑下沉到服务端，减少了客户端代码的负担。
- **警告**：**UDF 的外部副作用**与事务的**原子性**天生冲突。必须在设计时明确隔离事务内部状态与外部世界。

## 触发器 (Triggers) 深度解析

### 用户自定义函数 (UDF) 的副作用与事务

- **核心概念**：UDF 可以产生副作用（如发送邮件），但存在**事务回滚风险**。
- **关键问题**：如果事务在 UDF 执行后回滚，副作用（如已发送的邮件）无法撤销。
- **关键术语**：**事务边界**、**副作用**、**补偿操作**（如撤回邮件，但通常不可行）。
- **结论**：一旦操作离开数据库系统范围（如邮件发出），就无法回滚，因此应避免在事务性 UDF 中产生关键副作用。

### 触发事件的类型

- **DML 事件**：
  - **INSERT**、**UPDATE**、**DELETE**
  - **TRUNCATE**（相当于不带 WHERE 的 DELETE，即清空表）
- **DDL 事件**：
  - **CREATE**、**ALTER**、**DROP**

### 触发器的作用域与触发时机

- **作用域**：
  - **表级触发器**：当对该表执行特定事件时触发（最常见）。
  - **数据库级触发器**：对整个数据库范围的事件响应。
  - **系统级触发器**：对系统级别的事件响应。
- **触发时机**：
  - **BEFORE**：在查询执行之前触发。
  - **AFTER**：在查询执行之后触发。
  - **BEFORE EACH ROW**：在每一行被修改之前触发。
  - **AFTER EACH ROW**：在每一行被修改之后触发。
- **特殊类型**：某些数据库系统支持**查询重写规则**，即当触发查询时，数据库会用触发函数替换整个查询。

### 无限循环陷阱

- **核心问题**：触发器可以互相调用，导致**无限递归**。
- **典型场景**：
  - 在表 `Foo` 上有一个触发器，它执行对 `Foo` 的插入或更新操作。
  - 该更新操作又会导致另一个触发器（如 `Foo_audit` 表上的触发器）被触发。
  - 这个新触发器又反过来影响 `Foo`，形成循环。
- **数据库防护机制**：
  - 大多数数据库系统会设置**最大嵌套深度计数器**（如 `max_recursive_triggers`）。
  - 当超过深度限制时，系统会自动终止执行，防止无限循环。
- **代码示例**（触发无限循环）：
  ```sql
  -- 创建一个在 Foo 表更新后触发的触发器
  CREATE TRIGGER bad_idea
  AFTER UPDATE ON Foo
  FOR EACH ROW
  BEGIN
      -- 在触发器内又更新 Foo 表
      UPDATE Foo
      SET vowel = 'a'
      WHERE id = 1;
      RETURN NEW;
  END;
  ```
  - 这个触发器会在更新 `Foo` 后再次尝试更新 `Foo`，导致自身被递归触发。

## 第 24 部分

### 数据库触发器与变更通知机制

#### 1. **触发器基础与常见陷阱**
- **核心概念**：触发器（Trigger）是数据库内嵌的自动化响应机制，当指定事件（INSERT、UPDATE、DELETE）发生时自动执行预定义函数。
- **关键术语**：`BEFORE` / `AFTER`（触发时机）、`FOR EACH ROW`（行级触发）、`OLD` / `NEW`（行数据引用）
- **常见陷阱示例**：
  - **事件不匹配**：在 `FooAudit` 表上建立触发器，但误写为监听 `Foo` 表的插入事件，导致触发器永远不会触发。
  - **无限循环风险**：触发器内部若再次执行相同表的相同操作（如 INSERT 又触发 INSERT），会导致无限递归。需谨慎设计逻辑（例如使用 `BEFORE` 阻止新插入或通过条件分支）。
  - **解决方案**：使用 `BEFORE` 替代 `AFTER`，或添加判断条件（如 `IF TG_OP = 'INSERT' THEN ...`）避免重复触发。

#### 2. **变更通知机制——向外部世界广播**
- **核心概念**：标准 SQL 的 **LISTEN / NOTIFY** 机制，允许数据库系统主动通知外部应用程序（如游戏服务器、微服务）发生了数据变更，避免应用层轮询数据库。
- **关键术语**：`LISTEN`（订阅频道）、`NOTIFY`（发送通知）、`pg_notify()`（PostgreSQL 专用函数）
- **工作流程**：
  1. 应用进程执行 `LISTEN channel_name` 订阅某个频道。
  2. 数据库中某张表的触发器内部调用 `NOTIFY channel_name, 'message payload'`。
  3. 订阅的应用进程异步接收到通知，从而触发后续逻辑（如更新缓存、触发业务流程）。
- **实用代码示例（PostgreSQL 风格）**：
  ```sql
  CREATE OR REPLACE FUNCTION notify_foo_change() RETURNS trigger AS $$
  BEGIN
      PERFORM pg_notify('foo_changes', row_to_json(NEW)::text);
      RETURN NEW;
  END;
  $$ LANGUAGE plpgsql;

  CREATE TRIGGER foo_audit_notify
      AFTER INSERT OR UPDATE OR DELETE ON foo
      FOR EACH ROW EXECUTE FUNCTION notify_foo_change();
  ```

#### 3. **与外部系统集成的注意事项**
- **核心概念**：触发器虽然强大，但原则上应保持轻量、无阻塞。**避免**在触发器中执行长耗时操作（如发送 HTTP 请求、发送邮件），否则会阻塞数据库事务。
- **类比延伸**：该机制类似**发布-订阅（Pub-Sub）模式**的数据库原生实现，可替代应用层“轮询表”的笨重方案，降低延迟和资源消耗。

#### 4. **SQL 标准 vs 数据库扩展**
- **关键术语**：标准 SQL 中的 `LISTEN` / `NOTIFY` 是通用规范，但不同数据库实现细节可能不同（如 PostgreSQL 使用 `pg_notify` 函数，MySQL 使用 `SIGNAL` / `RESIGNAL` 或用户自定义事件）。
- **实践建议**：若需跨数据库兼容，尽量使用标准 SQL 语法；若深度集成某一数据库特性（如 PostgreSQL），可直接使用其扩展函数获取更高效控制（如 `NOTIFY` 携带 JSON 格式数据）。

## 第 25 部分


### **核心概念：从“拉取”到“推送”**

传统数据库查询是**拉取（Pull）模式**——客户端需要反复轮询数据库以检查是否有更新。PostgreSQL 的 **LISTEN/NOTIFY** 机制实现了**推送（Push）模式**：当数据库发生变更时，主动向所有监听该事件的客户端发送通知。

- **NOTIFY**：在数据库触发器或函数中调用，向指定“频道”（Channel）发送消息。
- **LISTEN**：客户端连接至数据库后，监听指定频道，接收异步通知。
- **异步特性**：通知并非实时同步，需要一个“轮询”或“回调”机制来获取（例如 `SELECT 1` 模拟轮询）。

### **核心操作流程：触发通知**

1. **创建触发器函数**：这是一个 PostgreSQL 专属函数，用于在行级更新后生成通知。
2. **定义触发器**：指定在什么表、什么操作（如 INSERT）后，对每一行（FOR EACH ROW）执行该函数。
3. **调用通知**：函数内部使用 `pg_notify(channel, payload)` 向指定频道发送 JSON 格式的变更数据。

### **关键技术细节**

**关键函数与语法**：
- **`row_to_json(OLD/NEW)`**：将更新后的行（`NEW` 或 `OLD` 记录）转换为 JSON 对象。
- **`pg_notify(channel, payload)`**：发送通知，`payload` 通常是 JSON 序列化后的文本。
- **`LISTEN channel`**：客户端在连接上发出此命令，开始监听该频道的通知。

**触发器定义示例逻辑**：
```sql
-- 1. 创建触发器函数（处理更新后的行）
CREATE OR REPLACE FUNCTION notify_foo_updates() 
RETURNS trigger AS $$
BEGIN
  PERFORM pg_notify('foo_update', row_to_json(NEW)::text);
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- 2. 将触发器绑定到目标表的每次插入
CREATE TRIGGER foo_update_trigger 
AFTER INSERT ON audit_table 
FOR EACH ROW 
EXECUTE FUNCTION notify_foo_updates();
```

**客户端交互示意**：
- **上层终端**（生产者）：执行 INSERT/UPDATE，触发数据库自动发送通知。
- **下层终端**（消费者）：先 `LISTEN foo_update`，然后用 `SELECT 1` 模拟异步轮询，接收 JSON 格式的变更数据。

### **使用场景与限制**

- **推送外部系统**：将数据库变更实时推送给应用服务器、缓存层（如 Redis）、WebSocket 服务等，避免应用层轮询。
- **性能优势**：相比高频轮询，大幅降低数据库负载，适合高频率、低延迟的更新通知。
- **局限性**：
  - **PostgreSQL 专属**：不是 SQL 标准功能，不能跨数据库迁移。
  - **异步与可靠性**：通知是“尽力而为”的，客户端若断开连接则丢失通知；需配合客户端库实现自动重连和重试。
  - **复杂数据类型**：要避免在数据库中存储过于复杂的数据（如大型 JSON、二进制等），应只发送关键变更信息或 ID，完整数据由客户端通过常规查询获取。

> **总结**：通过 **LISTEN/NOTIFY** 结合触发器，你可以构建一个轻量级的实时数据管道，将数据库状态变更以 JSON 形式推送给外部系统，是游戏引擎后端、实时排行榜、状态同步等场景的有力工具。

## 第 26 部分

### 用户自定义类型 (User-Defined Types) 与扩展性

#### 核心动机：将复杂数据类型原子化存入数据库服务器
- 在游戏引擎或渲染管线中，经常有**非标量但需要原子性存储**的复杂数据类型（如坐标、颜色、变换矩阵等）。
- **目标**：避免在应用层编写解析JSON、Protobuf等序列化格式的自定义代码，让不同应用实现能直接复用数据库服务器中的类型。
- **核心术语**：**原子性 (Atomically)** 指的是将复合数据作为一个不可分割的整体存入服务器，而非拆散成多个属性。

#### 传统方案的痛点
- **方案一：简单拆分**  
  将复合数据拆成多个标量属性存储（如 `x, y, label`）。  
  → 优点：直接符合关系模型。  
  → 缺点：丧失原子性，需要额外逻辑保证一致性。
- **方案二：应用层序列化**  
  在应用层将数据序列化为JSON/XML或二进制格式（如Protocol Buffers、Thrift）后，作为BLOB存入。  
  → **警告：这是一个“糟糕的主意”**  
  - 数据库无法理解BLOB内部结构，无法进行索引、约束或类型检查。
  - 每个应用需要重复实现序列化/反序列化逻辑，维护成本高。

#### 历史关键突破：PostgreSQL 1980年代的“可扩展性”哲学
- PostgreSQL在1980年代提出**用户自定义类型 (User-Defined Types, UDTs)** 的概念，成为其核心设计思想之一。
- 该理念后来被纳入**SQL标准**，形成了**对象-关系模型 (Object-Relational Model)**，即**可扩展的关系模型**。
- 背景：当时受面向对象编程（OOP）兴起的影响，出现了独立的**面向对象数据库**（现已基本消失）。PostgreSQL的贡献是在关系模型内部优雅地融合了OOP思想。

### 实际应用：坐标类型的示例

#### 问题场景
需要定义一个包含 `x`, `y` 坐标和一个字符串标签的复合坐标类型。

```
原始数据：坐标 (x: float, y: float, label: string)
```

#### 三种存储方式的对比

| 方式 | 描述 | 问题 |
|---|---|---|
| **拆分存储** | 存为三个独立列：`x`, `y`, `label` | 失去原子性，操作需维护多列一致性 |
| **应用序列化** | 用JSON或Protobuf打包成BLOB | 数据库无法直接利用内部数据，索引/查询困难 |
| **用户自定义类型** | 创建新数据类型 `coordinate`，直接原子性存储 | 保留原子性，支持索引、约束、扩展操作 |

#### 推荐方案：用户自定义类型
- **定义语法**：  
  ```sql
  CREATE TYPE coordinate AS (
      x DOUBLE PRECISION,
      y DOUBLE PRECISION,
      label TEXT
  );
  ```
- **关键优势**：
  - **类型安全**：数据库会检查插入的数据是否符合定义。
  - **操作扩展**：可以添加自定义函数（如计算距离、坐标转换）作用于该类型，类似渲染管线中的数学库。
  - **免应用解析**：所有连接的应用都可以直接使用 `coordinate` 类型，无需额外序列化代码。

#### 与游戏引擎的类比
- 类似游戏引擎中定义 `Vector3`、`Matrix4` 等数学类型。
- `CREATE TYPE` 相当于在引擎的“数据类型系统”中注册新的复合类型，而非将其拆成三个独立变量或者压缩成裸字节流。

### 总结：关键概念与术语

- **核心概念**：**对象-关系数据库** = 在关系模型基础上增加**可扩展性**，允许用户定义复杂类型。
- **关键术语**：
  - **原子性 (Atomic)**：即使类型内部有多个字段，在数据库层面仍被视为一个不可分割的值。
  - **用户自定义类型 (UDT)**：SQL标准中通过 `CREATE TYPE` 定义的复合类型。
  - **应用序列化**：将数据在应用端转为JSON/BLOB后存入数据库的方法——**应避免**。
  - **关系模型的扩展 (Extensible Relational Model)**：传统关系模型只能处理标量类型，PostgreSQL的UDT打破了这一限制。
- **公式/算法相关**：无直接数学公式，但存在逻辑原则——**类型系统设计原则**：数据库能理解的数据结构越多，应用层代码就越精简。

## 第 27 部分

### 一、对象关系数据库与复合类型

*   **核心概念：从关系模型的“原子性”到“对象化”**
    *   传统关系数据库的列类型是**严格标量且原子**的（如 int, varchar）。
    *   **对象关系数据库**（Object-Oriented DBs, OODB）或类似 JSON/XML 的扩展，核心思想是让数据库能够存储**更复杂、非原子**的数据结构。
    *   **关键术语：复合类型 (Composite Type)**
        *   通过 `CREATE TYPE` 命令，将多个内置基本类型组合成一个新的、可复用的数据类型。
        *   示例：`CREATE TYPE Coordinate AS (x int, y int);` 这创建了一个包含 x 和 y 两个整数的复合类型。

*   **用户自定义类型 (UDT - User Defined Type)**
    *   **核心概念：扩展数据库的“类型系统”**
    *   对于比简单复合类型更复杂的需求（如自定义排序、特定运算逻辑），数据库允许用户用 **C/C++** 或 **C# (.NET)** 编写自定义类型。
    *   这意味着数据库服务器内部原本只有处理 `int`、`string` 的库和例程，现在**为你的自定义类型也加载了对应的处理逻辑**。这是一个非常强大的扩展机制。

### 二、复合类型的使用与限制

*   **使用示例：定义表和插入数据**
    ```sql
    -- 定义复合类型
    CREATE TYPE Coordinate AS (x int, y int);
    
    -- 创建使用复合类型的表
    CREATE TABLE ObjectTracker (
        id integer,
        location Coordinate
    );
    
    -- 插入数据
    INSERT INTO ObjectTracker VALUES (1, (10, 20));
    
    -- 查询：返回的是包含复合类型值的元组
    SELECT * FROM ObjectTracker;
    ```

*   **核心限制：数据库“不理解”你的类型**
    *   **“宏”的比喻**：复合类型对于数据库引擎来说，更像是一个**语法糖或宏**（macro）。引擎知道它由两个 int 组成，但不知道它的**语义**。
    *   **问题实例**：
        *   `SELECT SUM(location) FROM ObjectTracker;`
        *   **结果：报错 (Error)**。因为数据库引擎不知道如何将两个坐标“相加”。`SUM()` 是给标量数值准备的，引擎不知道你的 `(x + ?)` 逻辑是什么。

*   **解决方案：需要定义操作符**
    *   若要对复合类型执行 `+`、`=` 等操作，你必须像写 UDT 一样，用 C/C++ 实现这些操作符（`sum`、`plus`、`equals`）的函数。
    *   **难点**：即使 `=`（等于）比较容易定义，`+`（加）的定义完全是语义相关的。比如“两个坐标相加”是什么意思？是 `(x1+x2, y1+y2)` 还是别的？数据库**无法猜测你的意图**。

### 三、索引与存储

*   **关于索引的讨论**
    *   **核心问题**：能否在复合类型上创建索引？
    *   **答案**：**是的，可以**。
    *   **前提**：索引（如 B-Tree）通常依赖“可比较性”。对于复合类型，它默认可以利用内部的 `=` 操作，或者根据你为 UDT 定义的比较运算符来构建索引。
    *   **示例**：
        ```sql
        CREATE INDEX idx_location ON ObjectTracker(location);
        SELECT * FROM ObjectTracker WHERE location = (10, 20);
        ```

*   **存储与 JSON 的对比**
    *   **核心澄清**：这种原生复合类型**不是** JSON。在底层，它会被**原生存储 (stored natively)**，而不是作为文本字符串。
    *   与 JSON 的区别：JSON 是文本 blob/对象，需要解析；原生复合类型是数据库内部的、结构化的二进制格式，性能和类型检查更好。

### 数据库视图 (Views)：虚拟表与查询复用

#### 核心概念：什么是视图？

*   **视图是一个“虚拟表”**，其内容由一条 `SELECT` 查询语句定义。它本身不存储数据，只是对底层真实表数据的一种**逻辑映射**。
*   **工作原理**：当查询一个视图时，数据库系统会在内部自动将视图名替换为定义它的 `SELECT` 语句，然后执行该查询。这本质上像一个 **SQL 宏**。

#### 关键特性与优点

*   **查询复用与简化**
    *   对于**极其复杂的 `SELECT` 查询**，可以将其定义为一个视图。
    *   其他开发者（或应用程序）可以直接查询这个视图，**无需重复编写相同的复杂逻辑**。
    *   **性能提示**：使用视图不会让查询运行得更快。因为它只是宏替换，底层执行计划与直接写那个复杂 SQL 语句完全相同。

*   **数据安全与权限控制（列级安全）**
    *   可以定义一个视图，只暴露底层表的**部分列（属性）**，从而**隐藏敏感字段**。
    *   实现粗粒度的权限控制：给不同角色的用户分配不同的视图，而非直接给他们底层表的访问权限。
    *   **例子**：创建一个 `EmployeePublicView`，包含姓名和部门，但不包含工资列。然后只允许非HR人员访问这个视图。

#### 实际应用与极端案例

*   **典型应用**：构建报表、屏蔽不需要的列、为不同用户组提供定制化的数据视图。
*   **极端案例（SAP 系统）**：讲师提到其见过最复杂的数据库是某个 SAP 系统，其中存在**成千上万个视图**，并且这些视图是**层层嵌套**的（视图A基于视图B，视图B基于视图C……）。这通常是由于系统历史悠久（40-50年）且持续累积导致的。

#### 总结

| 特性 | 描述 |
| :--- | :--- |
| **本质** | **虚拟表**，存储的是查询定义，而非数据。 |
| **实现机制** | **宏替换**：使用时，数据库系统将视图名替换为对应的 `SELECT` 语句。 |
| **核心价值** | **简化复杂查询**（复用）、**实现列级安全**（隐藏敏感列）。 |
| **性能影响** | **无性能提升**（与直接执行底层 SQL 完全一样）。 |
| **风险** | 过度嵌套（视图套视图）会使数据库架构变得难以维护（如SAP的古老系统）。 |

## 第 29 部分

# 视图（Views）与物化视图（Select Into）的区别

## 视图（View）的核心概念

- **定义**：视图本质上是一个**命名的查询**，它被存储为数据库对象，使用时表现得像一张表
- **工作方式**：视图是**虚拟表**，每次被查询时都会**重新执行**其底层的SELECT语句
- **语法示例**：
  ```sql
  CREATE VIEW cs_students AS
  SELECT student_id, login_name
  FROM students
  WHERE department = 'CS';
  ```
- **数据时效性**：每次访问视图都会**获取最新的数据**（忽略事务隔离级别的影响）

## 视图的查询展开机制

- **重写原理**：当用户查询视图时，数据库系统会自动将视图名称**展开为原始查询**，形成一个**嵌套子查询**
- **优化器处理**：查询优化器会进行**去嵌套（unnesting）**操作，将视图查询与外部查询合并为**等价连接**（join）
- **宏类比**：视图本质上就像一个**数据库宏**，它在查询时被文本替换并重新优化执行

## 视图的安全应用

- **权限控制**：通过视图可以**隐藏敏感列**（如成绩grades），只暴露必要字段
- **数据隔离**：不同用户可以看到**不同的数据子集**，而无需修改底层表结构
- **示例场景**：管理员不需要查看学生成绩，可以通过视图只暴露学生ID和登录名

## 视图 vs. SELECT INTO（物化视图/临时表）

### 核心差异：计算时机

| 特性 | 视图（View） | SELECT INTO |
|------|--------------|-------------|
| **计算时机** | **按需计算**（on-demand） | **创建时计算** |
| **数据存储** | 不存储数据，仅存储查询定义 | 创建时生成**静态快照** |
| **数据更新** | 每次查询都**反映最新数据** | 底层数据变化后**不会自动更新** |
| **更新类型** | **虚拟视图**（virtual view） | 物化临时表 |

### SELECT INTO 的特性

- **一次性快照**：创建临时表时，数据是当时查询结果的**静态副本**
- **无自动传播**：如果原始表中的数据发生了变化，SELECT INTO创建的临时表**不会自动同步更新**这些变化
- **适用场景**：需要**固定数据集**进行后续操作，而不关心后续数据变化

## 视图的可修改性限制

- **标准规定**：SQL标准允许在某些**受限情况**下通过视图修改数据（UPDATE/INSERT/DELETE）
- **PostgreSQL限制**：PostgreSQL不支持直接通过视图修改数据
- **修改条件**：视图必须满足以下条件才可能允许修改：
  - 不包含**聚合函数**（Aggregation）
  - 基于**单个表**（single table）
  - 不涉及多个表的复杂连接

## 关键记忆点

- **视图 = 命名查询 + 按需执行 + 最新数据**
- **SELECT INTO = 静态快照 + 历史数据 + 无自动更新**
- **视图本质是查询宏，优化器负责去嵌套展开**
- **安全视图 = 数据脱敏 + 列级权限控制**
- **可修改性限制 = 无聚合 + 单表基础**

### 视图（Views）与物化视图（Materialized Views）

#### **1. 视图（Views）核心概念**
- **视图本质**：视图是一个**虚拟表**，其内容由查询定义。它不实际存储数据，而是每次查询时动态执行底层查询。
- **修改限制**：
  - **标准SQL**：在极少数情况下允许修改视图（如无聚合、单表），因为可以逆向映射回原始元组。
  - **PostgreSQL**：通常**不支持**直接通过视图修改数据（除非使用`INSTEAD OF`触发器或规则）。
  - **条件**：仅当视图无聚合、且基于单表时，可能重写修改操作回到原始表。

#### **2. 物化视图（Materialized Views）**
- **核心问题**：如果视图查询代价极高（如扫描PB级数据），每次访问都执行底层查询不现实。
- **解决方案**：**物化视图**将视图查询的结果**计算并存储**为临时表，像“快照”一样持久化。
- **维护策略**：
  - **简单实现**（如PostgreSQL）：`SELECT INTO` + `REFRESH`命令（全量重算）。
  - **高级实现**（如SQL Server）：支持**增量更新**，仅处理发生变化的底层数据行。
    - **示例**：在线游戏排行榜视图，某玩家得分变化时，只需更新该玩家对应的条目，无需重算全表。
  - **技术挑战**：逆向查询到具体影响的行并非易事，需要对查询进行反向分析。

#### **3. 增量维护的难度与系统对比**
- **难度来源**：需要从视图查询逻辑中**反向推导**出哪些数据变化会影响视图的哪些部分。
- **现有系统**：
  - **SQL Server**：业界领先，但并非完美，复杂查询仍可能退化为全量刷新。
  - **Materialize**：专门为维护物化视图设计的系统（基于流处理）。
  - **PostgreSQL**：仅支持`REFRESH MATERIALIZED VIEW`全量刷新。

### 数据库特性的实际使用频率总结

#### **1. 特性使用频率排序**
根据演讲者的经验，现实世界中各特性的使用频率由高到低：
1. **视图（Views）**：最常见，几乎所有应用都在用。

### 关键公式与算法（无直接数学公式，但有逻辑原则）

#### **视图修改原则**
```sql
-- 可修改视图的条件（简化版）
UPDATE view_name SET column = value WHERE condition;
-- 必须满足：视图基于单表，无聚合，无DISTINCT，无GROUP BY，无集合操作（UNION/INTERSECT/EXCEPT）
```

#### **物化视图增量维护的核心挑战**
```sql
-- 全量刷新 vs 增量更新
-- 全量刷新（简单但昂贵）：
REFRESH MATERIALIZED VIEW mv_name;

-- 增量更新（高效但复杂）：
-- 需系统自动识别变更行，并仅更新受影响的物化视图行
-- 例如：若视图为 SELECT count(*) FROM t GROUP BY category
-- 当t中某行category='A'被更新，只需更新mv中category='A'对应的count值
```

## 第 31 部分

## 存储过程与UDF的实际挑战与使用趋势

### 一、调试与开发困难

- **核心概念：存储过程/UDF的调试极其困难，是主要的开发阻力和风险来源**
- **关键术语：Trial-and-Error Debugging（试错式调试）**
  - **缺乏成熟的调试工具**：尤其是PostgreSQL，几乎没有专门的调试器，开发者只能通过“编译-报错-修改”的试错循环来调试代码
  - **Oracle/ SQL Server调试工具略好**：但整体上仍远不如应用层代码的调试体验
  - **效率极低**：复杂逻辑下，试错式调试会严重拖慢开发节奏，增加bug漏测的风险

### 二、数据库管理员（DBA）的保守倾向

- **核心概念：DBA对数据库变更极为保守，与敏捷开发的快速迭代节奏存在根本冲突**
- **关键术语：DBA Vetting Process（DBA审查流程）**
  - 每次新增或修改**存储过程（Stored Procedure）、UDF（用户自定义函数）**都需要DBA审核
  - DBA要确保：**无性能退化**、**不引发线上问题**
  - 审核周期通常很长，例如**需要一个月才能批准一个简单的UDF变更**
- **对敏捷开发的制约**：两周一个Sprint，但每次Sprint都需要DBA批准，实际上根本无法完成迭代
- **给出的最佳实践**：**优先将逻辑实现在应用代码中**，避免依赖DBA变成开发瓶颈

### 三、版本管理与部署同步的复杂性

- **核心概念：UDF/存储过程与应用代码各自独立维护版本，导致双轨版本管理，同步难度极大**
- **关键术语：Dual Version History（双轨版本历史）**
  - **滚动更新场景**：
    - 部分服务器/应用使用新版本UDF
    - 部分服务器/应用仍需旧版本UDF
    - 需要**同时维护多个版本的UDF**，并按调用方分发
  - **CI/CD集成挑战**：
    - 虽然有**Terraform**等基础设施即代码工具改善，但核心问题依然存在
    - 应用代码有独立的**版本历史（Version History）**
    - UDF/存储过程也有独立的**版本历史**
    - 必须保证两者**永远同步安装**，否则会出现接口不匹配的线上故障
    - 对于**小型团队（Small Shops）**，这种复杂度导致：**从开发效率角度出发，全部保留在应用代码中更快、更简单**

### 四、实际工业界的演进趋势

- **核心概念：从“全应用层”到“按需下推”的演进策略，是业界最务实的高性能路径**
- **关键术语：Incremental Push-down（增量下推策略）**
  - **初期阶段**：**所有业务逻辑都实现在应用代码中**，保证开发速度和灵活度
  - **瓶颈出现后**：**分析性能热点**，找出最慢的操作
  - **增量下推**：**仅将能显著加速的特定逻辑**，逐步迁移到数据库服务器（UDF/存储过程）
  - **原则**：不求一开始的完美架构，而是**基于实际的性能瓶颈，做出最小、最精确的数据库层优化**
  - **结论**：在大多数场景下，**全盘使用存储过程/UDF的做法已被业界弃用**，更合理的做法是“应用层为主，库层为辅，按需下推”

## 第 32 部分

### 性能优化与工程实践：从原型到成熟应用的演进策略

#### **1. 核心原则：增量式优化与瓶颈驱动**

*   **核心概念**：不要一开始就构建复杂的系统，而是采用**增量式优化**策略。
    *   首先找出当前系统中最慢、最瓶颈的环节（“最慢的东西”）。
    *   然后，逐步将计算或数据从“慢”的地方（如主线程、脚本层）**推送**到“快”的地方（如数据库服务器、GPU、专用计算层）。
    *   **关键术语**：**瓶颈驱动开发** (Bottleneck-Driven Development)。
*   **实践经验**：
    *   对于初创项目，不要试图构建分布式数据库、自定义查询优化器等高级基础设施。直接用现成的、简单的解决方案（如**UDS触发器**提到的实用触发机制）。
    *   随着应用成熟（更老的成熟代码），你会看到这些优化的痕迹越来越多。

#### **2. 对数据库与老系统的态度**

*   **核心概念**：不同生命周期的项目，技术选型和优化方法截然不同。
    *   **初创项目**：使用轻量级、直接的方案。不要过早引入复杂度。
    *   **成熟应用**：会看到大量的“微优化”和“特殊处理”（如你提到的增量推入数据库服务器）。
    *   **特别警告**：如果系统是**Oracle**，优化工作会非常棘手（“good luck”）。Oracle的优化器和锁机制复杂，往往限制了灵活的增量式优化策略。

#### **3. 技术栈的普适性**

*   **关键洞察**：提到的策略（增量优化、瓶颈识别、数据推送）**适用于所有主流数据库系统**。
    *   例如，在**Oracle**（OLAP系统）或**OTB**（可能指“开箱即用”系统）中，这些优化原理是**正交的**（orthogonal）。这意味着：
        *   方法本身与具体数据库底层实现无关。
        *   你可以在任何系统上应用这些思路：找出瓶颈，然后局部优化，逐步推进。

#### **4. 总结：一个成熟的工程师的视角**

*   **核心思维**：不是“我应该优化所有东西”，而是“我需要理解当前系统最慢的地方是什么，然后只优化那一个点”。
*   **工程操作**：
    *   **先确定慢在哪里**（CPU？I/O？网络？数据库查询？）。
    *   **再决定如何推入**（将计算推到GPU？将查询结果缓存到内存？将批量写入推入数据库的异步队列？）。
    *   **最后验证**：优化后，新的瓶颈会在别处出现，继续重复这个过程。
*   **工程格言**：“如果你能在一个全新系统中不构建这些东西（指复杂的优化基础设施），那就不要构建。等系统真正需要时再逐步加入。”
