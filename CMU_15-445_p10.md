## 第 1 部分

# 查询执行基础与查询计划

## 核心概念：从SQL到执行计划

**查询执行**是数据库系统中将**存储层**（磁盘数据管理）和**内存缓冲层**（数据进出内存）之上的**操作执行层**。核心任务是将SQL查询转化为实际可执行的操作序列。

### 查询转化流程
- **SQL查询** → **解析** → **抽象语法树（AST）** → **关系代数表达式** → **查询计划**
- 查询计划是数据库系统实际执行的**操作序列**，包含具体的算法选择

## 查询计划的核心组件

### 关系代数操作符
查询计划由一系列**关系代数操作符**组成：
- **扫描操作**：全表扫描、索引扫描
- **排序与聚合**：ORDER BY、GROUP BY、COUNT、SUM等
- **连接操作**：JOIN操作（后续章节详述）
- **其他关系操作符**：投影、选择、集合操作等

### 执行模型
数据库系统在执行查询计划时，需要考虑**数据传递方式**和**并发执行模型**：
- **单线程处理模型**
- **多线程处理模型**
- **多进程处理模型**

## 本期重点：排序与聚合算法

### 排序算法
- **核心目标**：支持ORDER BY操作，同时为其他操作（如合并连接）提供基础
- **算法选择**：根据数据量大小和内存限制选择不同的排序策略

### 聚合操作
- **常见聚合函数**：COUNT、SUM、AVG、MIN、MAX
- **分组聚合**：GROUP BY操作需要先排序或哈希分组

## 关键术语

- **查询计划（Query Plan）**：数据库执行SQL查询的具体操作序列和算法选择
- **抽象语法树（AST）**：SQL解析后的中间表示形式
- **关系代数**：查询优化的理论基础，将SQL转化为代数表达式
- **处理模型（Processing Model）**：决定数据如何在操作符之间传递和执行的架构

## 核心公式

查询计划生成过程的形式化表示：
$$SQL \xrightarrow{parse} AST \xrightarrow{translate} \text{关系代数} \xrightarrow{optimize} \text{查询计划}$$

其中查询计划包含一系列操作符：
$$\text{QueryPlan} = \{Op_1, Op_2, ..., Op_n\}$$

每个操作符对应具体的算法实现：
$$Op_i \in \{\text{排序}, \text{聚合}, \text{连接}, \text{扫描}, \text{投影}, ...\}$$

## 本周重点课程内容

1. **排序算法的实现**：如何在不同内存约束下高效排序
2. **聚合操作的实现**：分组和聚合函数的计算策略
3. **数据传递方式**：操作符之间的中间结果传递
4. **执行模型选择**：单线程 vs 多线程 vs 多进程

> **实践关联**：这些概念直接影响查询优化器的设计和执行引擎的实现，是构建高性能数据库系统的核心基础知识。

## 第 2 部分

### 查询执行的核心概念：从SQL到物理计划

#### 1. **SQL查询的转换流程**
- **核心概念**：SQL是一种**声明式语言**，只声明“需要什么结果”，不指定“如何获取结果”。数据库系统负责将SQL转换为可执行的物理操作。
- **关键步骤**：
  - **解析**：SQL查询首先被转换为**抽象语法树（AST）**。
  - **逻辑计划生成**：AST被翻译成**查询计划树**（或DAG，有向无环图）。这是一个**逻辑计划**，只描述数据的逻辑流向，不指定具体实现方法（如索引扫描还是全表扫描、使用哪种连接算法）。
  - **物理计划生成**：逻辑计划经过优化后，生成具体的**物理计划**，指定每个操作符的物理实现（例如：使用哈希连接还是嵌套循环连接、是否使用索引）。

#### 2. **查询计划树的结构**
- **叶子节点**：位于树的最底层，代表**数据访问方法**。常见方法包括：
  - **索引扫描**（Index Scan）
  - **全表扫描**（Table Scan）
- **内部节点**：代表**数据操作算子**，例如：
  - **连接（Join）**
  - **过滤（Filter）**
  - **投影（Projection）**
- **数据流动**：数据从叶子节点向上流动，经过各层算子处理后，最终返回给应用程序。

**示例**（连接表A和表B）：
```
      Projection
          |
        Join (A和B连接)
       /     \
   Filter    Access B
      |
   Access A
```
- **解释**：
  - `Access A`和`Access B`是叶子节点，负责从底层存储读取数据。
  - `Filter`对A的数据进行过滤，然后传递给`Join`。
  - `Join`将A和B的数据合并，输出给`Projection`。
  - `Projection`选择最终需要的列，返回结果。

#### 3. **逻辑计划 vs 物理计划**
- **逻辑计划**：只描述**做什么**（如“连接A和B”），不涉及**怎么做**（如“使用哪种连接算法”）。
- **物理计划**：指定**具体实现**，例如：
  - 使用**哈希连接**（Hash Join）还是**归并连接**（Merge Join）？
  - 是否使用**索引**来加速访问？
  - 数据是否需要在连接前**排序**？

#### 4. **查询优化的核心问题**
- **目标**：为给定的SQL查询找到**最优执行计划**（即代价最小的物理计划）。
- **优化方向**：
  1. **物理算法选择**：例如，哈希连接通常优于嵌套循环连接（尤其是大数据集时）。
  2. **逻辑重写**：改变树的结构以降低代价，例如：
     - **下推过滤**（Filter Pushdown）：尽早过滤数据，减少后续算子处理的数据量。
     - **连接顺序重排**：先连接小表，再连接大表，减少中间结果的大小。

#### 5. **重要公式与算法（后续深入）**
- **哈希连接伪代码**（简化版）：
  ```
  // 构建阶段：扫描小表，将其连接键和行存入哈希表
  for each row in small_table:
      hash_table[hash(join_key)] = row

  // 探测阶段：扫描大表，对每行检查哈希表
  for each row in large_table:
      if hash_table contains key:
          output join(row, hash_table[key])
  ```
- **归并连接**：要求两个表在连接键上已排序，然后像归并排序一样并行扫描。
- **索引扫描代价公式**（B+树）：
  - 等值查找代价：\( O(\log n) \)（树的高度）
  - 范围扫描代价：\( O(\log n + k) \)（k为结果行数）

#### 6. **总结：关键记忆点**
- **SQL → AST → 逻辑计划 → 物理计划**：这是所有数据库系统的通用执行路径。
- **叶子节点是数据源**，内部节点是操作算子。
- **逻辑计划是蓝图**，物理计划是具体施工图。
- **优化核心**：选择正确的物理算法 + 重写逻辑结构以降低代价。
- **哈希连接是当前工业界的主流选择**（大数据规模下效率高）。

## 第 3 部分

## 查询优化与存储感知的物理算法设计

#### 查询优化的两个层面
- **物理算法优化**：选择更优的底层算法实现。核心原则是 **哈希连接（Hash Join）通常优于排序-合并连接（Sort-Merge Join）**，这是经过实践验证的经验法则。
- **逻辑查询计划重写**：对关系代数表达式进行逻辑重组，与之前讨论的“关系代数逻辑重排”类似。典型的优化技术是 **过滤条件下推（Filter Push Down / Predicate Push Down）**，即在执行连接（Join）操作之前，先应用过滤条件筛掉无关数据，避免对后续不需要的数据进行昂贵的连接计算。

#### 核心转变：从“内存假设”到“磁盘意识”
- **传统算法分析的局限**：过去的算法课只关注**渐近复杂度**（比较次数、操作次数）。但在数据库/磁盘环境下，这种标准可能**并非正确**的评判依据。
- **真正的性能瓶颈**：算法的好坏往往取决于它**读写磁盘的频率和方式**。
    - 哈希表理论上是 $O(1)$，但如果产生大量**随机访问**（Random I/O），频繁跳跃到不同的磁盘页面，性能会很差。
    - 一个在算法理论上“更慢”的算法，如果能产生**顺序I/O（Sequential I/O）**，或者能将数据分块读入内存、一次性处理完毕且不再回访，反而可能更快。
- **核心目标**：选择对**磁盘I/O模式敏感**的算法，**最大化顺序I/O的比例**。

#### 为什么拒绝操作系统管理内存
- 操作系统不了解上层的**具体算法逻辑**，无法预知访问模式。
- 因此，系统（数据库/引擎）需要**自行管理缓冲池（Buffer Pool）**，以实现智能预取（Pre-fetching），控制何时、以何种顺序将数据换入换出磁盘，从而优化物理算子的性能。

---

### 关键术语与公式总结

| 概念 | 解释 |
| :--- | :--- |
| **查询优化** | 分物理优化（选算法）与逻辑优化（重写查询计划）。 |
| **过滤条件下推** | 逻辑优化的典型手法，先过滤再连接，减少连接数据量。 |
| **磁盘意识算法** | 以**磁盘I/O模式**（顺序 vs 随机）而非仅计算复杂度作为评价算法优劣的标准。 |
| **缓冲池（Buffer Pool）** | 数据库自行管理的内存区域，用于缓存磁盘页，实现算子间中间结果的管理与溢出。 |
| **顺序I/O vs 随机I/O** | 顺序I/O性能远优于随机I/O，是算法选择的关键权衡因素。 |

**核心原则公式（非数学，而是设计哲学）：**
$$
\text{算法真正的性能} \propto \frac{\text{顺序I/O量}}{\text{随机I/O量}} \times \frac{1}{\text{磁盘访问总次数}}
$$
**优先选择能产生大量顺序I/O、减少随机跳转的算法。**

## 第 4 部分

### 数据库为何需要排序？—— 摆脱OS束缚，拥抱自主控制

#### 核心动机：OS是“亦敌亦友”，我们需争取自主权

- **核心概念：OS是“Frenemy”（亦敌亦友）**。我们依赖OS启动硬件、处理底层交互，但它的通用设计（如内存管理、调度、文件系统）会干扰我们对查询和数据流的深度理解与优化。
- **关键术语：Kernel Bypass（内核旁路）**。这是一种避免通过OS API访问硬件（如磁盘、网络）的技术，让数据库直接与硬件通信，减少开销，提升性能。
- **目标：尽可能减少OS干预，争取以下控制权：**
    - **自主内存分配**：拒绝OS的通用内存管理（如页面置换），因为OS不知道数据库的算法模式（例如，预取策略、重复使用刚加载的块），无法做出最优的换入/换出决策。
    - **自主调度**：数据库更了解当前查询、数据特征和等待队列，能比OS的通用调度器做出更优的执行计划。
    - **直接访问硬件**：通过内核旁路或其他技术，绕过文件系统API，直接读写数据，减少上下文切换和拷贝开销。

---

### 排序的“必要性”：关系模型与查询优化的基石

#### 为什么数据库必须实现排序？

- **关系模型的天然需求**：关系数据库的核心操作（如合并连接、去重、分组、避免产生临时文件中的重复顺序）都依赖有序数据。
- **查询优化的基石**：
    - **高效连接**：**Sort-Merge Join（排序合并连接）** 是一种经典且高效的连接算法，其前提是两张表都已按连接键排序。
    - **去重与分组**：`DISTINCT` 和 `GROUP BY` 操作通常基于排序后扫描相邻重复项来实现。
    - **有序输出**：用户查询中的 `ORDER BY` 子句，是排序最直接的体现。
    - **预取优化**：排序后的数据访问模式更可预测（顺序扫描），便于数据库进行高效的预取（prefetching）和缓存，避免OS因未知算法而错误换出。
- **超越OS“通用”能力**：OS无法理解用户查询的语义（例如，下一批数据是哪张表、哪个键），因此它做的预取是盲目的。数据库排序能让自己预知未来访问模式，从而自定义内存和I/O策略。

---

### 历史插曲：80年代的“裸金属”数据库

- **早期尝试**：1980年代，有团队尝试直接在**裸金属**（bare metal）上运行数据库系统，即编写自己的固件，完全避开OS，直接与硬件对话。
- **现状**：如今已不再这样做，因为自行开发完整硬软件栈的**开销和复杂度得不偿失**。
- **现代方案**：采用**Kernel Bypass**（如利用 RDMA、DPDK、SPDK 等技术）这种“折中”方式，既保留OS的基础功能，又能在关键路径上绕过它，获得接近裸金属的性能。

**总结笔记**：数据库排序不只是为了满足 `ORDER BY`，它是关系模型内建的核心能力，用于驱动高效的连接、去重、分组，并让数据库成为自身内存和I/O的“主人”，从而在与OS的“亦敌亦友”关系中，掌握主动权。

## 第 5 部分

## 排序的必要性与基本场景

### **关系模型的无序本质**
- **核心概念**：关系模型和SQL**天生不保证顺序**，但许多查询（如`ORDER BY`）明确要求排序输出
- **典型用例**：游戏引擎中查询"前十名玩家"、作业1中的"top 10演员"等

### **内部隐式排序**
即使查询未显式要求排序，数据库系统内部也会主动排序以优化上层操作：
- **重复消除**：`DISTINCT`子句利用排序后一次线性扫描检测重复——**若当前记录与上一条键值相同，则为重复，直接跳过**
- **聚合操作**：`GROUP BY`子句在排序后可高效计算分组聚合
- **批量加载**：B+树的构建需要预排序数据，排序后能快速进行**批量加载**
- **查询计划优化**：上层算子假设输入已排序，可设计更高效的算法

### **排序策略的模块化选择**

#### **全内存排序**
- 如果所有数据**完全适合内存**，则使用标准排序算法
- 例如：PostgreSQL使用**快速排序**；其他可能的选择包括**样例排序**、**平滑排序**等

> **关键原则**：排序模块是数据库系统的"基础设施"，它为上层算法提供**有序数据流**，从而简化逻辑、提升性能。排序模块自身应具备**内存/磁盘自适应**能力，避免与操作系统直接交互。

## 第 6 部分

### 外部归并排序与 Top-K 堆排序：面向磁盘的排序策略

#### 核心问题：内存装不下怎么办？

- **简单情况：数据全部在内存**
  - 直接使用你喜欢的任何排序算法，比如**快速排序**。
  - 对于 PostgreSQL 或任何数据库而言，这都不是问题。
  - 但是，一旦**数据量超过内存容量**，快速排序就**完全不可用**。

- **为什么快速排序在磁盘上不行？**
  - 快速排序依赖“随机 pivot”，这会导致大量的**随机 I/O**。
  - 磁盘的**寻道时间**远高于顺序读写，随机 I/O 会带来灾难性的性能下降。
  - 因此，当需要“外排”（spill to disk）时，必须采用对磁盘访问模式友好的算法。

#### 核心内容：两种面向磁盘的排序策略

- **特殊情况：Top-K 查询（LIMIT / FETCH FIRST）**
  - **核心概念**：如果查询带有 `ORDER BY ... LIMIT N` 或 `ORDER BY ... FETCH FIRST N ROWS ONLY`，**不需要**对整个数据集进行多趟排序。
  - **算法**：**堆排序（Heap Sort）**。
  - **原理**：只需扫描一次数据，维护一个大小为 N 的最大堆（或最小堆），即可找到前 N 个结果。
  - **优点**：**单趟扫描即可完成**，无需多趟访问磁盘。
  - **SQL 语法说明**：`LIMIT ... OFFSET` 不是 SQL 标准，`FETCH FIRST N ROWS ONLY` 才是标准写法。

- **通用方案：外部归并排序**
  - **核心思想**：**分而治之 + 归并**。这是处理无法容纳在内存中的大数据集的标准策略。
  - **用途**：不仅是排序，也是后续数据库操作（如 **Join** 和 **Aggregation**）的基石。
  - **步骤（分治过程）**：
    1.  **划分（Divide）**：将大数据集切分成多个**小文件（Runs）**，每个小文件的大小刚好能够**完全读入内存**。
    2.  **内部排序（Sort）**：在内存中对每个小文件进行排序（例如使用快速排序），然后**写回磁盘**。这一步也称为 **“生成初始归并段”**。
    3.  **合并（Conquer）**：从已排序的小文件中，**以多路归并的方式**逐个读取最小的元素，写入新的输出文件中。这个输出文件就是全局有序的结果。
  - **关键洞察**：归并阶段可以**多层级**进行，因此处理的数据量远超可用内存容量。

#### 总结与延伸

- **何时用堆排序**：当查询是 **Top-K 类型**（`ORDER BY col LIMIT N`）时，优先使用。
- **何时用外部归并排序**：当需要**全局排序**且数据量大时，这是标准解法。
- **为什么要学习它**：外部归并排序的“分治”思想，不仅是排序本身，也是**数据库执行引擎中 Join 和 Aggregation 的核心实现手段**，即“**排序-聚合（Sort-Based Aggregation）**”与“**排序-连接（Sort-Merge Join）**”。
- **未来预告**：后续章节将讲解如何使用**排序**和**哈希**来实现这两种核心操作。

## 第 7 部分

## 使用堆（Heap）实现 Top-N 排序与 `WITH TIES` 子句

### 核心概念：`FETCH FIRST` 与 `WITH TIES`

- **标准 SQL 的 `FETCH FIRST` 子句**：这是 SQL 标准（ISO/IEC）中定义的官方语法，用于替代常见的 `LIMIT` 子句。`LIMIT` 实际上并非标准 SQL 的一部分。
  - 语法：`ORDER BY ... FETCH FIRST n ROWS ONLY`
  - 支持情况：SQL Server 及许多其他数据库系统支持此标准语法。

- **`WITH TIES` 扩展**：允许在结果集中包含与第 n 行**并列（tie）** 的所有行。
  - **核心行为**：即使取 `FETCH FIRST 4 ROWS`，如果第 4 名出现重复值，也会一并返回，实际返回行数可能 > 4。
  - **应用场景**：需要完整的分组排名结果（如“返回前三名，并列也要包含”）。

### 算法实现：基于堆（Priority Queue）的单次遍历

#### 核心数据结构

- **堆（Heap）**：一种优先队列，用于在**单次数据扫描**中维护当前已看到的前 N 个元素。
  - 默认行为：维护一个**最小堆（Min-Heap）**，堆顶是当前第 N 大的元素（即切割点）。
  - 内存假设：堆应当能完全放入内存；若不能，则需进入更复杂的查询优化阶段。

#### 算法步骤

1. **初始化**：创建一个空堆，容量设为 `N`（本例中 N=4）。
2. **逐行处理数据**：
   - **堆未满**：直接将当前元素插入堆。
   - **堆已满（已存 N 个元素）**：
     - 如果当前元素 **大于** 堆顶（堆中最小的元素），则：弹出堆顶，插入当前元素。
     - 否则：跳过当前元素。
3. **`WITH TIES` 特殊处理**：
   - 当遇到元素**等于**堆顶（即与第 N 名并列）时：
     - **不弹出堆顶**，而是直接额外存储该重复元素。
     - **结果**：堆的实际大小膨胀（可能超过 N），以容纳所有并列行。
     - **代价**：内存使用量成倍增加（最多可能存储所有重复值）。

#### 直观示例（原始数据：3, 4, 6, 2, 9, 4, 7, 8；N=4，带 `WITH TIES`）

| 步骤 | 当前元素 | 堆内容（小根堆） | 说明 |
|------|----------|------------------|------|
| 1 | 3 | [3] | 堆未满，插入 |
| 2 | 4 | [3, 4] | 堆未满，插入 |
| 3 | 6 | [3, 4, 6] | 堆未满，插入 |
| 4 | 2 | [2, 3, 4, 6] | 堆未满，插入 |
| 5 | 9 | [3, 4, 6, 9] | 堆满，9 > 堆顶2，弹出2，插入9 |
| 6 | 4 | [4, 4, 6, 9] | 堆满，4 ≥ 堆顶3，弹出3，插入4 |
| 7 | 7 | [4, 4, 6, 9] | 堆满，7 > 堆顶4，弹出4，插入7 → [4, 6, 7, 9] |
| 8 | 8 | [4, 6, 7, 9] | 堆满，8 > 堆顶4，弹出4，插入8 → [6, 7, 8, 9] |

- **最终堆内容**：`[6, 7, 8, 9]`（Top 4 无重复）
- **注意**：此示例未展示 `WITH TIES` 的膨胀场景。若步骤6出现第二个4且与堆顶相等，堆会存储 [3, 4, 4, 4, 6, 9]（膨胀）。

### 关键优化与注意事项

- **单次遍历（One-Pass）**：
  - 理想情况下，只需要扫描一次数据（O(n) 时间复杂度）。
  - 操作复杂度：每次堆的插入/弹出为 **O(log N)**，总复杂度 O(n log N)，N 远小于 n。
  
- **内存假设的重要性**：
  - **最佳情况**：堆完全在内存中，算法超高效。
  - **最坏情况**：堆无法放入内存，需回退到更复杂的**外部排序**或**多阶段归并**（后续查询优化课程会讨论）。

- **与 `LIMIT` 的关系**：
  - 优化器最佳实践：**一旦检测到 `LIMIT` 或 `FETCH FIRST` 关键字，立即尝试使用堆排序策略**，因这是大多数场景下的最优解。

- **重复值处理的代价**：
  - 带 `WITH TIES` 时，堆的内存可能远大于 N，极端情况下可能退化为存储所有行。
  - 需权衡：是否值得为处理并列行而消耗额外内存。

### 总结

- **堆 + 单次遍历**是实现 **Top-N 查询**的高效算法，时间复杂度 O(n log N)，内存开销 O(N)。
- **`WITH TIES`** 扩展会破坏堆的大小上界，导致内存膨胀，但保证了排名结果的完整性。
- 数据库优化器通常在解析阶段就选择此策略，是物理算子层面最关键的优化之一。

## 第 8 部分

# 外部归并排序：分治策略处理大规模数据排序

## 核心问题：数据量超出内存容量时的排序策略

当待排序数据无法全部装入内存时，**外部归并排序**是数据库系统采用的标准解决方案。该算法采用**分治策略**，将大规模数据排序分解为可管理的小任务。

### 算法两阶段流程

1. **排序阶段（Sorting Phase）**
   - 将数据分割成多个**独立的块（runs）**
   - 每个块的大小应能**完全装入内存**
   - 对每个块在内存中进行**内部排序**（如快速排序）
   - 将排序后的块写回磁盘

2. **合并阶段（Merging Phase）**
   - 将多个已排序的块**两两合并**（或N路合并）
   - 生成更大的已排序块
   - **重复合并过程**，直到所有数据合并为一个完整的有序数据集

### 算法复杂度

- 时间复杂度：**O(n log n)** - 与普通归并排序相同
- I/O开销：**关键性能瓶颈**，需要最小化磁盘读写次数

## 查询优化与统计信息

### 成本估算的挑战

数据库系统在查询优化时面临的核心困境：

- **统计信息维护**：系统通过内部统计信息预测各操作符的输出数据量
  - 使用**直方图**记录各属性的值分布
  - 估计**期望行数**、**重复率**、**唯一键数量**
- **预测准确性**：**统计信息预测往往不准**
  - 数据分布可能不均匀
  - 相关属性之间的依赖关系复杂
  - 统计信息可能过时

### 统计信息的关键指标

- **基数估计**：评估谓词过滤后剩余的数据量
- **选择率**：满足条件的元组占总元组的比例
- **唯一值数量**：某个属性上不同值的个数

## 索引对排序的优化

### B+树索引的直接扫描

- **问题**：如果ORDER BY属性上已存在**B+树索引**，扫描索引本身是否比扫描数据更快？
- **答案**：**是的，索引扫描可以避免排序**
  - B+树的叶子节点已经**天然有序**
  - 只需顺序遍历叶子节点链表即可得到有序结果
  - 需要权衡：**索引扫描的随机I/O** vs **完整排序的代价**

### 适用条件
- 查询需要全表扫描时优势明显
- 当有LIMIT限制时，可以**提前终止扫描**
- 需要考虑**聚簇索引** vs **非聚簇索引**的差异

## ORDER BY与LIMIT的交互语义

### 语义差异分析

- **标准语法**：`ORDER BY ... LIMIT N`
  - 先对全表排序，再取前N行
  - 排序操作可能代价巨大

- **嵌套查询优化**：
  - 内层查询：`LIMIT 4` 直接取前4个元组
  - 外层查询：`ORDER BY` 仅对4个元组进行快速排序
  - **语义确实改变**：内层LIMIT截断了数据集，外层ORDER BY只能排序截断后的结果

### PostgreSQL的特殊行为
- 支持**LIMIT的优化处理**：当只需"前几名"时
  - 可以使用**Top-N堆排序**替代全量排序
  - 维护大小为N的最大堆，单次遍历即可
  - **显著减少内存和CPU消耗**

## 关键术语总结

- **外部归并排序**：处理超内存数据排序的标准算法
- **分治策略**：将大问题分解为可管理的小块
- **统计信息**：查询优化器用于成本估算的数据分布信息
- **直方图**：记录属性值分布的统计工具
- **B+树索引扫描**：利用有序索引避免排序
- **Top-N排序**：LIMIT优化技术，使用堆结构高效获取前N行

## 第 9 部分

### 排序-合并算法与物化策略

#### 1. **算法核心：两阶段流程**
* **核心概念**：排序-合并算法的核心是将大规模数据集的排序过程分解为两个阶段：**排序阶段（Sorting Phase）** 和 **合并阶段（Merging Phase）**。
* **注意区分**：此处的“排序阶段”与后续将学到的“排序合并连接（Sort-Merge Join）”中的排序阶段不同，需留意上下文差异。
* **重复性**：这两个阶段会重复执行，直到整个数据集被完全排序。

#### 2. **具体执行步骤**
* **阶段一：排序与写入**
  * 将**能够放入内存的数据块**进行内部排序。
  * 将这些排序后的数据块（称为 **“有序运行（Runs）”**）写回磁盘。
* **阶段二：合并与扩展**
  * 从磁盘读取已排序的多个“运行”。
  * 将它们**合并成更大的有序运行**，其大小通常是之前单个运行的两倍。
  * 重复此过程，直到所有数据合并成一个完整的有序数据集。

#### 3. **关键数据结构：键值对（Key-Value Pairs）**
* **简化模型**：为了便于说明，假设被操作的数据对象是 **键值对**（Key-Value Pair）。在讨论中常忽略“值”部分，但必须理解其概念。
* **值的两种选择（物化策略）**：
  * **早期物化（Early Materialization）**：
    * **定义**：在“值”中**存储元组（Tuple）的完整拷贝**。
    * **代价**：在排序和磁盘间拷贝时，会**产生大量的数据复制**，代价高昂。
  * **晚期物化（Late Materialization）**：
    * **定义**：在“值”中只存储 **记录ID（Record ID，RID）**，通常为32位或64位整数。
    * **优势**：数据量小，**拷贝和传递成本极低**。
    * **补充**：可以根据上层系统实际需要，再通过投影（Projection）等手段获取元组的特定字段。相关内容将在下一周深入讲解。

#### 4. **权衡与总结**
| 策略 | **早期物化** | **晚期物化** |
| :--- | :--- | :--- |
| **核心思想** | 值包含完整数据 | 值包含记录ID |
| **优点** | 后续操作可直接使用数据 | 数据移动量小，效率高 |
| **缺点** | 排序过程中复制大量数据，开销大 | 需要额外的“物化”步骤来获取实际数据 |
| **适用场景** | 数据量小或后续需要频繁访问完整数据 | 数据量大，或后续仅需部分字段 |

## 第 10 部分

### 物化策略（Materialization Strategies）与排序算法

#### **1. 物化策略：早物化 vs 晚物化**

- **核心概念**：物化策略决定了在查询执行过程中，**何时**将元组的完整数据（所有列）从磁盘读入内存并传递到下一操作符。
- **关键术语**：**早物化（Early Materialization）**、**晚物化（Late Materialization）**、**记录ID（Record ID）**。

- **早物化**
    - **流程**：操作符在处理每一步时，直接将完整的元组（Tuple）从磁盘读入内存，并将包含所有列的数据传递到下一阶段。
    - **优点**：**数据只需读入一次**，后续所有操作都在内存中的完整元组上进行，无需二次回表（fetch）或随机I/O。
    - **缺点**：在流水线（Pipeline）的多轮次（如排序的多次Merge Pass）中，需要**反复拷贝大量完整数据**从一步到下一步，内存带宽和拷贝开销大。
    - **适用场景**：当查询结果集较小，或后续操作需要频繁访问多个列时。

- **晚物化**
    - **流程**：操作符先只处理**索引或键值**（如Record ID），直到最终必须输出结果或进行无索引的列访问时，才根据Record ID去获取完整的元组数据。
    - **优点**：**极大减少早期处理的数据量**，因为只传递轻量级的键/ID，尤其适合过滤效果好的查询（Filter早，丢弃大量元组）。
    - **缺点**：最终需要**缝合（Stitch）** 所有列的数据，可能引入大量随机I/O。
    - **常见场景**：**列式存储（Column Stores）** 的主流选择。因为列存储中，读取最小需要的列集，晚物化能有效避免读取无需使用的列。

- **决策权衡**
    - **早物化**：**I/O密集型**，一次大块读取，适合数据重用率高。
    - **晚物化**：**过滤优先型**，适合过滤能提前丢弃大量行的查询，避免浪费I/O在不需要的列上。

---

#### **2. 外部归并排序（External Merge Sort）**

- **核心概念**：当数据量远超可用内存（Buffer Pool大小）时，必须使用外部排序算法，将数据分块排序并分批合并。
- **关键术语**：**运行（Run）**、**K路归并（K-way Merge）**、**缓冲区池（Buffer Pool / B Buffers）**。

- **算法总览**
    - **第1阶段：生成初始运行（Initial Run）**
        - 将数据切分为多个**页（Page）**，每页大小由磁盘/内存页定义。
        - 利用有限的**B个缓冲区**，每次将一部分页读入内存，在内存中排序（如快排），生成一个**已排序的运行（Run）**，再写回磁盘。
        - 结果：磁盘上多个按页划分、各自有序的初始运行。
    - **第2阶段：多路归并（Multi-way Merge）**
        - **2路归并**：每次从磁盘读入 **2** 个运行的页，在内存中合并成一个新的、更大的运行，写回磁盘。
        - **K路归并（K-way Merge）**：使用 **B** 个缓冲区，其中 **K = B - 1** 个作为输入缓冲区（每个输入运行一个指针/游标Cursor），**1** 个作为输出缓冲区。遍历K个游标，取最小值写入输出缓冲区，直至所有输入耗尽。
        - 合并完成后，新的运行大小为 **K × 原运行大小**。
    - **重复**：持续进行多轮归并，直到整个表合并成一个单一的大型已排序运行。

- **关键约束与公式**
    - **缓冲区数量：** 算法必须**提前知道并分配好**缓冲区数量 `B`（这是有限内存）。
    - **归并轮次（Passes）计算：**
        - 初始运行数量：$N_0 = \lceil \frac{\text{总页数}}{\text{内存缓冲区页数}} \rceil$
        - 使用K路归并时，第 `i` 轮后的运行数量：$N_i = \lceil \frac{N_{i-1}}{K} \rceil$
        - 所需总归并轮次：$\lceil \log_{K} N_0 \rceil$
    - **I/O代价（典型公式）：**
        - **总I/O次数** = $2 \times \text{总页数} \times (\text{生成运行时读取+写入次数} + \text{每轮归并时读取+写入次数})$
        - 简化公式参考：**总数据量 × (1 + 归并轮次) × 2**（每次读写计一次I/O）。

- **图示理解（2路归并示例）**
    - **输入数据（无序）：** [5, 1, 9, 3, 7, 2, 8, 4]
    - **步骤1：生成运行**
        - 假设每次内存可排序2个页（简化），生成4个运行：`[1,5]`, `[3,9]`, `[2,7]`, `[4,8]`
    - **步骤2：第一轮归并（2路）**
        - 合并 `[1,5]` 与 `[3,9]` → `[1,3,5,9]`
        - 合并 `[2,7]` 与 `[4,8]` → `[2,4,7,8]`
    - **步骤3：第二轮归并（2路）**
        - 合并 `[1,3,5,9]` 与 `[2,4,7,8]` → `[1,2,3,4,5,7,8,9]`（最终有序结果）

## 第 11 部分

# 外部排序中的内存池与缓冲区管理

## 内存池（Buffer Pools）的核心概念

**内存池**是外部排序中**有限的内存资源**，用于同时处理：
- **输入数据**：从磁盘读取的数据页（runs）
- **输出数据**：排序后写回磁盘的新数据页

**关键特性**：
- 内存池大小是**有限且固定**的
- 由数据库系统通过**独立参数**控制（全局或查询级别）
- 不同的数据库系统有不同的命名：
  - **PostgreSQL**：`work_mem`
  - **MySQL**：`sort_memory`

**内存溢出机制**：
- 当排序所需内存**超过设置的阈值**时
- 查询会**自动溢出（spill）到磁盘**上继续处理
- 这是内存与磁盘之间的**自动平衡机制**

## 简单示例：两页数据的内存排序流程

### 基础条件设定
- **磁盘上的数据库**：仅有 **2 页数据**
- **内存限制**：每次只能加载 **1 页**
- **所需缓冲区**：
  - **输入缓冲区（Input Buffer）**：1 页
  - **输出缓冲区（Output Buffer）**：1 页

### 第 0 步（Pass 0）执行流程

**输入阶段**：
1. 从磁盘**读取所有 B 页**到内存中
2. 在内存中**对每一页进行排序**，形成初始的有序 runs
3. 每个 run 保持**原始页面大小**

**处理阶段**：
1. 加载第 1 页到输入缓冲区
2. 在内存中**对该页数据进行排序**
3. 将排序后的数据写入**输出缓冲区**
4. 将输出缓冲区**写回磁盘**

**为什么不进行原地排序（In-Place Sorting）？**
- 原始数据来自**表的持久化页面（Primary Storage Pages）**
- 原地排序会**覆盖原始数据**
- 如果同时有其他查询访问相同数据，会导致**数据混乱**
- 排序操作使用的是**临时缓冲区**：
  - 崩溃后丢失也无所谓
  - 不需要数据持久化保证

## 数据库排序参数对比

| 数据库系统 | 参数名称 | 作用 |
|-----------|---------|------|
| PostgreSQL | `work_mem` | 控制排序/哈希操作可用内存 |
| MySQL | `sort_memory` | 控制排序缓冲区大小 |

**参数作用范围**：
- **全局设置**：影响所有查询
- **查询级别**：仅影响单个查询
- **会话/客户端连接**：影响当前连接

## 核心公式与算法要点

### 缓冲区分配公式

设总内存池大小为 **M** 页：
- **输入缓冲区**：$\text{Input} = B_{in}$ 页
- **输出缓冲区**：$\text{Output} = B_{out}$ 页
- 总约束：$B_{in} + B_{out} \leq M$

### 第 0 步的排序扇入（Fan-in）为 1

因为每次只能处理 1 页，所以：
$$
\text{Fan-in}_{\text{pass }0} = 1
$$

这意味着：
- 每个 run 仅包含 1 页的数据
- 后续 pass 需要合并这些单页 runs

## 第 12 部分

### 外部归并排序 (External Merge Sort)：从理论到实践

#### 核心思想：分而治之，分阶段处理

- **核心概念**：当待排序的数据总量远超可用内存（RAM）时，无法将所有数据一次性加载到内存中进行排序。外部归并排序通过**分阶段**（Pass）的方式，将数据分成小块（页/块），先在内存中排序这些小块，再将它们逐步合并成更大的有序块，最终得到全排序的数据。
- **关键术语**：
    - **内存缓冲区 (Memory Buffer/Block)**：用于加载数据页进行排序和合并的固定大小内存单元. 例如，有3个可用缓冲区。
    - **数据页 (Page)**：磁盘上数据的基本存储单元，大小与内存缓冲区相同。
    - **归并顺序 (Degree of Merge, e.g. 2-way merge)**：指每一趟合并操作中，同时将多少个已排序的“游程”合并成一个更大的游程。
    - **游程 (Run)**：经过排序后，在磁盘上连续存储的一段有序数据。
    - **溢出 (Spill)**：当内存缓冲区满时，将数据写入磁盘的过程。

---

### 第一阶段：排序阶段 (Sort Phase)

- **核心概念**：将数据拆分成若干个能完全装入内存缓冲区的“块”（块即游程的雏形），并对其进行内部排序，然后将排序后的块写回磁盘。
- **操作流程**：
    1.  **读取**：将磁盘上的一个数据页（如 Page 1）加载到一个空闲的输入缓冲区。
    2.  **排序**：在内存中对这个缓冲区内的所有记录进行排序（使用快速排序、堆排序等内部排序算法）。
    3.  **写出**：将排序后的数据页写入磁盘的新位置，形成第一个“排序后的游程”的一部分。
    4.  **重复**：对磁盘上的所有数据页重复此过程，直到所有原始数据都被转换成若干个排序后的、独立的游程。

- **关键细节**：
    - **必须复制 (Must Copy)**：在排序阶段，**绝对不能**直接在原始数据页上进行排序！因为可能有其他并发查询正在访问这些页。所以，必须先**复制**数据页到内存缓冲区，再对副本进行排序，然后写出。这保证了数据的一致性和并发安全性。
    - **输出缓冲区**：通常需要一个输出缓冲区来暂存排序后的数据，并在填满后批量写出，以提高I/O效率。

---

### 第二阶段：归并阶段 (Merge Phase)  
本阶段以“2路归并”为例，即每一步合并两个已排序的游程。

- **核心概念**：将上一阶段产生的多个有序游程，通过反复合并，最终归并成一个全局有序的超大游程。

- **详细流程**：
    1.  **加载游程**：将两个已排序的游程（例如 Run 1 和 Run 2）的数据页分别加载到两个输入缓冲区中。每个输入缓冲区对应一个游程。
    2.  **双指针合并**：
        - 为每一个输入缓冲区维护一个**游标 (Cursor)**，指向当前待比较的元素。
        - 比较两个游标指向的元素（假设升序排列）。将**较小**的元素取出，放入输出缓冲区。
        - 将取出元素的游标后移一位。
    3.  **输出与消耗**：持续进行Step 2的比较和输出。
        - 当输出缓冲区写满时，将其写出到磁盘上，形成新的、更大游程的一部分（新游程的大小将是原两个游程大小之和）。
        - 当其中一个输入缓冲区的数据被全部消耗（指针超出范围）时，立即从磁盘加载该游程的下一个数据页到空闲的输入缓冲区中。
    4.  **循环结束**：重复Steps 1-3，直到两个输入游程中的所有数据都被合并到新的输出游程中。

- **关键图解与解释**：
    - **内存缓冲区配置**：使用3个缓冲区。2个用作输入（分别对应两个待合并的游程），1个用作输出（用于存放合并后结果）。
    - **“2路”的含义**：这里的`2`是指**合并的游程个数**，而不是数据页的数量。即使你的数据足够小，两个游程可以同时装入内存，但在流程上仍被认为是执行了一次“2路归并”。

---

### 示例澄清：一个更有代表性的场景

- **玩具示例的局限**：之前的例子只使用了2页的数据，可以完全装入内存，因此显得“归并”多此一举。这只是一个简化的流程演示。
- **真正的场景**：假设我们有**3个内存缓冲区**和**20页待排序数据**。过程如下：
    1.  **Pass 1 (排序)**：
        - 加载Page 1 -> 排序 -> 写出（形成Run A的一部分）。
        - 加载Page 2 -> 排序 -> 写出（形成Run B的一部分）。
        - ... 最终得到 **10个**已排序的2页游程（因为每次只能处理1页？不对，是每次排序1页，最终每个游程大小=1页？）。
        - *纠正*：通常排序阶段会将多个页读入内存排序后写成一个游程。有3个缓冲，可以读入2页，排序，写出一个2页的游程。对20页数据，经过Pass 1后会产生10个大小为2页的有序游程。
    2.  **Pass 2 (归并)**：
        - 合并Run 1和Run 2：使用2个输入缓冲（各读入1页）、1个输出缓冲。最终生成一个大小为4页的新游程。
        - 合并Run 3和Run 4：生成第二个大小为4页的新游程。
        - ... 最终，我们将10个2页游程归并为 **5个**大小为4页的新游程。
    3.  **Pass 3 (归并)**：
        - 重复步骤2，合并两个4页的游程，生成一个**8页**的游程。
        - ... 最终，我们将5个4页游程归并为 **3个**大小为...（注意：不能完全合并完，因为需要2个输入游程才能合并。最终可能产生3个游程，其中一个是8页，一个也是8页，一个是剩余的4页？）。
        - *核心*：这个过程会持续进行，每一趟Pass，游程的数量减半（2路归并），而单个游程的大小翻倍。

#### 总结与意义

外部归并排序是数据库系统和游戏引擎中处理大规模数据集排序（如构建渲染图、场景管理、资产索引预计算）的基础算法。

- **核心意义**：将**I/O成本**（磁盘读写）的优化置于**CPU计算**之前。算法设计的目标是**最小化磁盘访问次数**，因为磁盘I/O是系统的主要瓶颈。
- **工程实践**：通过调整归并顺序（`K-way merge`）和缓冲区大小，可以显著减少归并的趟数，从而降低I/O开销，提升系统吞吐量。

## 第 13 部分

### 外部排序中的多路归并：两阶段合并示例

本部分通过一个具体的**键值对排序**例子，详细演示了外部排序中**多路归并**的第二个关键阶段：**合并排序好的子文件（Run）**。这里限定为“两路归并”，即每次合并两个已排序的Run。

#### 核心概念：页面与游标

- **页面（Page）**：磁盘与内存之间数据传输的基本单位。每个页面可容纳固定数量的键（本例中为2个）。
- **运行（Run）**：一个已排序的键序列。一个Run可能由多个物理页面组成，逻辑上是一个整体。
- **游标（Cursor）**：在内存中指向当前Run内下一个待比较键的指针。归并时，通过移动游标逐键比较。

#### 第一阶段：内部排序（Pass 1）

- **操作过程**：
  - 按顺序将磁盘中的每个**单页**数据调入内存。
  - 对该页内的键进行**内部排序**。
  - 将排序后的页面写回磁盘。
- **关键点**：此阶段后，磁盘上的每一个物理页面都成为**独立且已排序**的Run（长度为1页）。

#### 第二阶段：多路归并（Pass 2）

- **核心思想**：利用**B-1个内存缓冲区**（B为可用内存页框数），在本例中B=3，因此可用2个输入缓冲区和1个输出缓冲区。
- **归并步骤**：
  1.  **加载**：将两个相邻的、已排序的页面（Run）分别加载到两个输入缓冲区。
  2.  **比较**：用游标分别指向两个Run的第一个键，比较它们的大小。
  3.  **输出**：将较小的键写入输出缓冲区。
  4.  **前进**：对应Run的游标后移，指向下一个键。
  5.  **重复**：重复步骤2-4，直到输出缓冲区写满。
  6.  **刷盘**：将满的输出缓冲区写入磁盘，清空后继续使用。
  7.  **切换**：当一个输入缓冲区被完全消耗（游标越界），立即从磁盘加载对应Run的下一页数据（若该Run还有剩余）。
- **示例推演**：
  - 初始状态：Run1 = `[3, 4]`，Run2 = `[6, 2]`（注意：此处的逗号仅代表键的分隔，非键值对）。
  - **输出缓冲区写满**：比较 `3` 和 `2` → `2` 写入，比较 `3` 和 `6` → `3` 写入，输出缓冲区 `[2, 3]` 满，刷盘。
  - **继续归并**：比较 `4` 和 `6` → `4` 写入，输出缓冲区 `[4]` 不满足刷盘条件，再比较... → 最终写入 `6`，输出缓冲区 `[4, 6]` 满，刷盘。
  - **结果**：两个单页的Run被合并成一个两页的逻辑Run，内容为 `[2, 3, 4, 6]`。

#### 重要数据结构：逻辑Run追踪

- **问题**：随着归并进行，一个逻辑Run可能跨越多个物理页面。
- **解决方案**：需要维护一个**额外的书签数据结构**来记录每个Run当前处理到了哪个物理页面，以及页面内的游标位置。这使得系统可以像遍历一个连续序列一样处理分布在多个页面上的逻辑Run。

## 第 14 部分

# 多路归并排序中的内存与页管理机制

## **核心概念：归并阶段的内存缓冲区需求**

归并排序在进行 **多路归并（K-way merge）** 时，**需要维护三个关键的内存页**：

- **两个输入页**：分别对应两个被归并的“运行（run）”
- **一个输出页**：用于存放归并结果，待写回磁盘

> **关键公式**：在 **B 路归并** 中，需要 **B-1 个输入缓冲页 + 1 个输出缓冲页**，总计 **B 个缓冲区页**

当输出缓冲区写满时，**立即刷写到磁盘（flush）**，释放内存空间以复用，从而持续接收新的归并结果。

---

## **增量归并与游标不回溯原理**

### **核心特性：游标永不回溯**

- 因为输入的两个“运行”已经是**内部有序的**
- 每次比较后，**只需移动较小值的游标**，**无需重新访问已处理过的页面**
- 这一性质保证了归并过程是**流式、线性的**，内存使用高效

---

## **归并是否“原地（in-place）”进行？**

### **简单情况：两路归并**

- **理论上可以原地交换**实现，但**复杂度过高**，不推荐
- 简化做法：**始终使用输出缓冲区**

### **复杂情况：多路归并**

- **无法直接原地交换**，必须依赖输出缓冲区
- **推荐做法**：统一采用输出缓冲模型，避免实现复杂性

---

## **磁盘页面复用与内存回收策略**

### **关键问题：输出缓冲区刷写后，能否覆盖原输入页面？**

| 策略 | 说明 |
|------|------|
| **覆盖写入** | 可以将输出结果写回磁盘，**复用已被读取完的输入页空间** |
| **重新分配** | 也可以写入**全新的磁盘区域**，保留原始运行（用于日志或恢复） |

> **核心权衡**：覆盖写入节省磁盘空间，但**丧失数据恢复能力**；重新分配提供更好的**容错性**。

---

## **元数据管理与崩溃恢复**

### **重要性质：归并过程中的元数据无需持久化**

- 记录“每个运行对应哪些物理页、顺序如何”的**辅助数据结构**
- **只保存在内存中**，**从不写入磁盘**
- **原因**：如果系统崩溃，**查询中间结果直接丢失，无需恢复**
- 元数据体积小，内存维护成本可忽略

> **本质**：归并排序的中间状态是**易失的（volatile）**，设计上允许崩溃后完全重新开始，无需持久化元数据。

---

## **总结：多路归并内存模型核心要点**

1. **最少缓冲页数** = 输入运行数 + 1（输出页）
2. **游标单向移动**，永不回溯，保证线性归并
3. **输出缓冲区写满即刷盘**，释放内存循环使用
4. **多路归并不做原地交换**，统一用输出缓冲
5. **元数据纯内存维护**，崩溃时丢弃，简化持久化逻辑
6. **磁盘页面可覆盖复用**，但需权衡空间与容错

## 第 15 部分

## 外部排序中的多路归并 (Multi-way Merge)

### 核心概念：**缓冲池与分页归并**
- 使用 **有限的缓冲池**（如示例中的3个缓冲页）完成大规模数据的排序
- 基本策略：**1个输入页 + 1个输入页 + 1个输出页** 进行归并
- 即使磁盘上有很多排好序的页面运行块 (page runs)，**只需要固定数量的缓冲页**即可完成归并

### 归并过程：**生成更大规模的已排序运行块**
- 从两个已排序页面开始，逐元素比较
  - 例如：左侧页头元素"2" < 右侧页头元素"4" → 输出"2"
  - 继续比较：左侧"3" < 右侧"4" → 输出"3"
- **页面耗尽处理**：当某侧页面读完，重新加载该侧的下一个页面
- **输出缓冲**：输出页填满后，**刷写到磁盘 (flush to disk)**，然后继续

### 关键公式：**归并趟数 (Number of Passes)**
- **总趟数** = \(1 + \lceil \log_2(N) \rceil\)
  - 第一趟（Pass 0）始终存在：**读取所有页面并逐页排序**（生成初始排序块）
  - 后续每一趟：**已排序块大小呈指数增长**，直到覆盖整个数据集
- \(N\) = 总页数

### **总I/O成本**
- **公式**：**总I/O成本** = \(2N \times \text{趟数}\)
  - \(2N\) 来源：**每一趟都必须读入所有数据，再写出所有数据**
  - 每一趟 = **1次输入 + 1次输出**

### 重要实践原则：**异步I/O以避免等待**
- 真实系统中缓冲池远大于3页
- **核心优化**：不要等输出缓冲填满再写，使用**异步I/O**
  - 当一个输出页写满，立即触发写入，同时**继续填充下一个输出页**
  - 避免CPU等待磁盘I/O完成

### 递进示例：从2页归并到4页
- 从初始的2页排序块出发，将它们归并为**4页的大排序块**
- 每次仅需读取两个输入页和一个输出页，**即使在处理更大规模数据时缓冲需求不变**

## 第 16 部分

### 外部排序中的双缓冲与多路归并

#### 异步 I/O 与双缓冲 (Double Buffering)

**核心概念：** 在真实系统中，**磁盘 I/O 是性能瓶颈**。不能让排序算法等待磁盘写入或读取完成（阻塞）。解决方案是**异步操作**，让 I/O 在后台进行，而 CPU 继续处理内存中的数据。

- **背景问题：** 输出缓冲区写满后，等待磁盘写入完成会阻塞排序进程。
- **解决方案：** 分配**两组内存页面**。
    - 一组用于当前正在处理（排序）的数据。
    - 另一组由**另一个线程或进程**在后台异步写入磁盘，或预取下一页数据。
- **操作流程示例：**
    1. 调入**页面1**，开始排序。
    2. **后台**异步调入**页面2**。
    3. 当页面1排序完成，页面2已准备就绪，可以无缝开始下一轮排序，无需等待 I/O。
- **核心公式/算法：** 双缓冲允许**预取 (pre-fetch)** 和**后台写入 (async write)**，消除 I/O 等待时间。

---

#### 通用外部排序模型 (General External Sort)

**核心概念：** 将双缓冲思想推广到多路归并，引入常量 **B**（可用缓冲页数）来决定每一趟需要多少个输出缓冲区。

##### 第0趟 (Pass 0): 生成初始顺串 (Initial Runs)
- **策略：** 使用**所有**可用缓冲区页面（B页）。
    - 因为此时不进行跨页面比较，只需对单个页面内的数据进行排序。
    - 每个页面独立排序并写出。
- **输出：** 生成 **\(\lceil N / B \rceil\)** 个已排序的顺串，每个顺串的大小为 **B** 页（最后一个可能不满，只有余数页）。

##### 后续趟数 (Subsequent Passes): 多路归并 (Merge)
- **策略：** 每一趟中，**保留1个页面作为输出缓冲区**。
    - 因此用于归并的输入缓冲区数量为 **\(B - 1\)**。
- **归并方式：** 每一趟将 \(B-1\) 个已排序的顺串归并成一个新的、更大的顺串。
- **趟数计算（通用公式）：**
    - **总趟数 = 1 (第0趟) + \(\lceil \log_{B-1} (\text{初始顺串数}) \rceil\)**
    - 公式可改写为： \(1 + \lfloor \log_{B-1} ( \lceil N/B \rceil ) \rfloor\)

---

#### 公式推导与实例计算

**核心公式（总 I/O 操作次数或趟数，简化版）：**
\[
\text{总趟数} \approx 1 + \lceil \log_{B-1} \left( \frac{N}{B} \right) \rceil
\]

**示例：** \(N = 108\) 页数据，\(B = 5\) 个缓冲页。

1.  **第0趟：**
    - 初始顺串数 = \(\lceil 108 / 5 \rceil = \lceil 21.6 \rceil = 22\) 个顺串。
    - 每个顺串大小为 5 页（最后一个顺串只有 3 页，因为 \(108 = 21 \times 5 + 3\)）。

2.  **后续趟数：**
    - 可用于归并的输入缓冲区 = \(B - 1 = 4\) 个。
    - 归并趟数 = \(\lceil \log_{4} (22) \rceil = \lceil \log_{4} 22 \rceil\)。
    - 计算：\(4^1 = 4\)，\(4^2 = 16\)，\(4^3 = 64\)。22 介于 16 和 64 之间，所以需要 **3趟** 归并（因为 \(4^2 = 16 < 22\)，需要再一趟才能归并完，实际上 \(4^3 = 64 > 22\)，但具体趟数计算为 \(\lceil \log_4 22 \rceil = 3\)）。

3.  **总趟数：** 第0趟 + 3趟归并 = **4趟**。

**公式细节说明（针对课件中的注释）：**
- 课件中提到的公式 \(1 + \log_2 \left( \frac{N - 3}{3} \right)\) 可能是针对特定情况（如 \(B=4\)，且考虑了最后一个不满缓冲区页的特殊处理），**但更通用的标准公式**是 \(1 + \lceil \log_{B-1} ( \lceil N/B \rceil ) \rceil\)。
- **重点理解**：常数 **B** 控制着归并的“路数”（\(B-1\)路归并），直接影响所需的趟数和总 I/O 成本。

## 第 17 部分

### 核心概念：归并排序中的比较优化

在外部归并排序中，**比较操作**是决定排序速度的关键因素之一。即使I/O效率已经通过多路归并（B-Way Merge）优化，**比较次数**依然会影响整体性能。

#### **1. 比较操作在归并阶段的重要性**
- **核心问题**：在归并阶段，需要从多个有序的“运行”（Run）文件中读取数据，并找出当前最小的元素。
- **瓶颈**：如果每次比较都遍历所有B个输入缓冲区（每个对应一个运行文件），复杂度为 **O(B)**，其中B是归并的路数。当B很大时（例如64或128），比较开销会显著增加。
- **优化思路**：**减少每次找出最小元素所需的比较次数**。

#### **2. 核心优化：使用“胜者树”或“败者树”**
- **核心概念**：**败者树（Loser Tree）** 是一种用于实现多路归并的高效数据结构，它能将每次选择最小（或最大）元素的比较次数从 **O(B)** 降低到 **O(log B)**。
- **原理**：
  - 构建一棵二叉树，叶子节点对应每个输入缓冲区的当前元素。
  - 内部节点记录树中两个子节点比较的**败者**，并让胜者继续向上传递。
  - 最终，根节点指向当前全局最小元素所在的缓冲区。
  - 当取出最小元素后，用该缓冲区的下一个元素替代，并沿着路径更新树，重新确定新的胜者。全程只需 **log B** 次比较。
- **公式与算法**：
  - 初始化：首次填充叶子节点，进行 **B次** 比较（构建完整树）。
  - 每次取出元素后的更新：比较次数 = **O(log B)**。
  - 总比较次数（对N个元素排序）：**N × O(log B)**（外加初始化的B次比较，占比可忽略）。

#### **3. 为什么这对游戏引擎和渲染引擎重要？**
- **背景关联**：虽然游戏引擎通常处理内存排序（如粒子排序、透明物体排序），但在需要**处理大量流式数据**（如地形页数据、超大型顶点缓冲）时，外部排序技术会被用到。
- **性能影响**：在渲染中，任何微小的I/O或CPU开销都可能被放大。使用败者树可以在**归并阶段显著减少CPU开销**，从而释放CPU周期用于渲染管线。
- **工程实践**：许多现代排序库（如标准库中的std::sort）在内存排序中使用内省排序，但在**外部排序场景（如数据库、大数据处理）**中，败者树是标准实现。

#### **4. 总结与记忆点**
- **核心要点**：从 **O(B)** 比较优化到 **O(log B)** 比较。
- **实现方式**：败者树（或胜者树），本质是**锦标赛排序思想**。
- **适用场景**：**任何需要高并发归并且路数很多（B较大）的外部排序算法**。
- **额外补充**：这是一种**计算机科学中“用空间换时间”的经典案例**——用少量额外的内存（树节点）换取CPU时间的指数级降低。

**一句话记忆**：归并排序时，别傻傻地每次遍历所有缓冲区，用败者树把比较次数从线性的B降到对数的log B。

## 第 18 部分

### 排序优化：减少IO与比较开销

#### 1. **比较函数的调用成本**
- **核心问题**：在通用数据库系统（如PostgreSQL, MySQL）中，排序算法需要支持**多种键类型**和**任意键组合**的比较。
- **传统实现**：通过**函数指针**传递比较函数，排序算法在每次比较时动态调用该函数。
  - 示例：STL（标准模板库）中传递函数指针或Lambda。
- **性能瓶颈**：
  - 函数指针间接调用（“跟随指针”）的开销在**内存内快速排序（QuickSort）** 中尤为显著。
  - **关键术语**：`间接调用开销`、`函数指针解引用`。

#### 2. **优化方案：代码特化（Code Specialization）**
- **核心思想**：为特定键类型**硬编码**比较逻辑到排序算法中，避免函数指针动态调用。
- **实现方式**：
  - **即时编译（JIT）**：部分系统（如Java、.NET）实时生成特化排序代码。
  - **静态代码生成**：如PostgreSQL通过**Perl脚本**自动生成多个版本的QuickSort（如`int32版本`、`浮点版本`），比较逻辑直接内嵌。
- **优点**：
  - 消除函数指针开销，排序速度显著提升。
  - **关键术语**：`内联比较`、`特化排序算法`。

#### 3. **优化方案：后缀截断（Suffix Truncation）**
- **核心问题**：对**可变长度字符串（VARCHAR）** 或任意长字符串进行排序时，整个字符串比较极其缓慢。
- **核心思想**：避免比较字符串的完整内容，只比较**足以区分顺序的前缀**。
- **类比**：类似于之前讨论的**数据压缩**技术——只处理必要的最小信息。
  - 示例：比较 `"apple"` 和 `"application"`，在匹配到 `"appl"` 后即可分出顺序，无需比较剩余字符。
- **优点**：
  - 大幅减少IO和CPU时间（尤其对长字符串）。
  - **关键术语**：`字符串前缀比较`、`截断比较`。

#### 4. **总结：优化优先级**
| 优化方向        | 适用场景                     | 核心手段                | 关键术语                       |
|----------------|------------------------------|------------------------|-------------------------------|
| 比较函数特化    | 内存内排序、多键类型         | 硬编码比较逻辑          | `内联比较`、`代码特化`         |
| 后缀截断        | 可变字符串排序、长键值       | 比较前缀而非全串        | `前缀截断`、`比较成本降低`     |

**公式/算法提示**：
- **比较开销模型**：
  - 传统比较：`Cost = IndirectCallOverhead + CompareLogic`
  - 特化后：`Cost = CompareLogic`（消除间接调用）
- **后缀截断收益**：假设字符串长度为 \(L\)，比较平均截断长度为 \(L_{cut} \ll L\)，则节省比例 ≈ \(1 - L_{cut}/L\)。

## 第 19 部分

### 后缀截断（Suffix Truncation）与字符串排序优化

#### 核心思路：**用整数比较替代字符串比较**
- 字符串比较（尤其是变长字符串）在排序时**极其缓慢**，因为需要逐字符对比直到发现差异。
- **后缀截断**的核心思想是：**只比较字符串的前缀**，并将这些前缀编码成**二进制整数**（如32位或64位整数）。
- **关键优势**：整数比较是CPU原生支持的单指令操作，比逐字符比较快几个数量级。

#### 操作流程
1. **提取前缀**：对每个字符串取出固定长度的前缀（如前4或8个字符）。
2. **二进制编码**：将前缀字符的ASCII码（或排序所需的编码）直接组合成一个整数。
   - 例如：字符串 "ABCD" 的前4个字符可以视为 0x41424344（A=0x41, B=0x42等）。
3. **执行整数排序**：比较这些整数即可快速完成排序。
4. **回退机制**：如果两个前缀整数**相等**，则说明真实字符串的前缀相同，此时才**回退到原始字符串**进行完整的逐字符比较。
   - 这是一种 **“乐观”策略**：**假设大多数情况下前缀不同**，因此大多数比较都极快。

#### 数学/算法表示
- 设字符串 \( S \)，选取前 \( k \) 个字符作为前缀 \( P = S[0:k] \)。
- 定义编码函数 \( E \)，将前缀映射为整数：
  \[
  E(P) = \sum_{i=0}^{k-1} \text{ord}(S[i]) \times 256^{k-1-i}
  \]
  （其中 \(\text{ord}\) 返回字符的编码值）
- 排序时的比较逻辑：
  \[
  \text{compare}(S_a, S_b) = 
  \begin{cases}
  E(P_a) < E(P_b) & \text{如果 } E(P_a) \neq E(P_b) \\
  \text{full\_string\_compare}(S_a, S_b) & \text{如果 } E(P_a) = E(P_b)
  \end{cases}
  \]

#### 重要前提：**编码顺序必须匹配字符串排序顺序**
- **核心约束**：必须保证整数比较的结果与字符串的字典序完全一致。
- 这意味着：
  - 如果使用ASCII编码，则没问题（因为ASCII的数值顺序与字典序相同）。
  - 如果使用**Unicode**或**本地化排序规则**，情况变得复杂。
    - 不同语言、不同区域设置下，字符的排序顺序可能不同。
    - 例如：某些语言中 "ä" 可能排在 "z" 之后，而ASCII中它的码点可能远小于 'z'。

#### 真实世界教训：**PostgreSQL的Unicode排序Bug**
- 曾经的PostgreSQL使用**libc**（C标准库）提供的字符串比较函数。
- 当**libc的新版本**更新了Unicode字符串的排序规则时，原有的基于前缀整数的排序假设被打破。
  - 因为之前使用的前缀整数编码是基于旧规则的顺序，而新规则下相同前缀可能对应不同的真实排序。
- 结果：数据库的B+树索引中的**有序性被破坏**，导致查询结果错误。
- **教训**：任何时候使用基于二进制截断的优化，**必须确保底层字符编码的排序规则是稳定且与比较函数一致的**。不能假设所有数据都是纯ASCII。

## 第 20 部分

### 数据库字符串排序与索引优化

#### 1. 字符串排序与归类（Collation）的陷阱
- **核心概念**：字符串的排序顺序依赖 **归类规则**，不同系统、不同版本的底层库（如 libc）可能改变排序行为。
- **关键术语**：
  - **Collation**：定义字符串比较和排序的规则（如大小写敏感、Unicode 权重等）。
  - **Libc 依赖**：早期 B+ 树索引在构建时基于特定版本的 libc 排序规则，升级后排序顺序可能改变，导致索引损坏或数据错乱。
- **潜在风险**：
  - **B+ 树索引的键顺序**一旦依赖系统库的排序，升级后索引可能失效，需要重建。
  - **企业级数据库**（如 PostgreSQL）通常依赖开源库实现排序，但规则变更可能影响已有数据。
- **示例场景**：  
  - 对 `varchar` 列建立 B+ 树索引 → 排序基于当前 libc。  
  - 升级 libc（如 Unicode 排序规则变化）→ 索引的键顺序与插入时不同 → 查询结果错误或索引损坏。

#### 2. 无索引时的查询行为：顺序扫描
- **核心概念**：如果表上 **没有索引**，查询计划的叶子节点（Leaf Nodes）**必须采用顺序扫描**（Sequential Scan）。
- **关键术语**：
  - **顺序扫描**：逐行读取整个表，无加速结构。
  - **回退选项**：顺序扫描是 **默认备选方案**，当索引缺失或无法使用时自动切换。
- **性能考量**：  
  - 小表：顺序扫描没问题。  
  - 大表：顺序扫描非常低效，需要索引支持。

#### 3. 数据库自动构建索引？
- **核心问题**：现代数据库系统是否会自动为表构建索引？
- **现状**：
  - **大多数数据库不会**自动构建索引，因为该问题属于 **AI 完备问题**（AI-Complete）—— 难以自动确定最优索引策略。
  - 少数系统（如某些商业数据库）有自动化索引建议功能，但非默认行为。
- **关键术语**：
  - **AI-Complete**：需要通用人工智能才能解决的问题，自动索引选择即是此类。

#### 4. 利用现有索引优化排序：B+ 树遍历 vs. 顺序扫描
- **核心概念**：如果表上已有 **B+ 树索引**，且排序键与索引键一致或部分匹配，数据库可以直接 **遍历 B+ 树** 获取排序后的数据，避免额外排序。
- **关键术语**：
  - **Order-Preserving Data Structure**：保持键顺序的数据结构，如 B+ 树。
  - **Clustered Index**（聚簇索引）：数据行物理顺序与索引顺序一致 → 遍历索引即可直接输出有序数据，**性能最佳**。
  - **Unclustered Index**（非聚簇索引）：数据行物理顺序与索引无关 → 遍历索引后需多次随机读取数据页，**效率不高**。
- **优化条件**：
  - **精确匹配**：`ORDER BY` 列与索引键完全相同 → 直接使用索引顺序。
  - **前缀匹配**：`ORDER BY` 列是索引键的 **前缀**（例如索引为 (A, B, C)，查询排序只用到 A）→ 仍可部分利用索引顺序。
- **数据库差异**：
  - **PostgreSQL**：非常保守，仅当 `ORDER BY` 键与索引键 **精确一致** 时才使用索引排序。  
  - **商业数据库**（如 Oracle、SQL Server）：更灵活，允许前缀匹配甚至部分匹配时使用索引。

#### 5. 总结：排序优化决策树
- **检查索引**：  
  - 如果存在与 `ORDER BY` 匹配的 **聚簇索引** → **直接遍历 B+ 树**，零额外排序成本。  
  - 如果存在 **非聚簇索引** + 覆盖足够多列 → 权衡随机 I/O 与排序开销，通常仍优。  
- **无索引**：  
  - 只能 **顺序扫描 + 显式排序**（`Sort` 操作）。  
- **数据库行为**：  
  - 保守派（如 PostgreSQL）：索引精确匹配才用。  
  - 激进派（如商业 DB）：支持更多匹配模式。

**公式/算法要点**：  
- 无索引排序代价：扫描全表 \( O(N) \) + 排序 \( O(N \log N) \)。  
- 聚簇索引遍历：直接输出有序数据，代价 \( O(\text{Leaf Count}) \)，远优于全表排序。

## 数据库排序与聚合操作实现

### 索引扫描策略与性能考量

**核心概念：索引类型决定I/O性能**
- **聚类索引（Clustered Index）**：最优情况
  - 叶子节点与数据页顺序一致
  - 只需**一次索引遍历** + **顺序扫描**数据页
  - I/O开销最小，因为数据按序连续存储

- **非聚类索引（Non-Clustered Index）**：潜在性能问题
  - 叶子节点顺序与数据页顺序**可能不同**
  - 每个记录访问可能产生**随机I/O**
  - 性能退化严重时，查询优化器可能放弃索引

**关键术语：查询优化器决策**
- 优化器需要在多个执行策略间估算I/O成本：
  - **直接索引扫描** vs **全表扫描+排序**
  - 估算基于统计信息（数据分布、页数等）
- **商业数据库**（Oracle, SQL Server）更激进地使用索引
- **PostgreSQL**更保守：只有在精确匹配场景键时才使用索引

### 排序算法选择策略

**核心概念：内存vs磁盘决定排序算法**

| 场景 | 算法 | 特点 |
|------|------|------|
| 全部在内存 | **快速排序（QuickSort）** | 平均O(n log n) |
| 需要Top-N结果 | **堆排序（HeapSort）** | 维护大小为N的堆 |
| 需要磁盘辅助 | **外排序归并（External Merge Sort）** | 分阶段处理海量数据 |

**高级优化：自适应查询处理**
- **策略**：先假设最优情况（快速排序），如果估算错误则丢弃结果，回退到归并排序
- **挑战**：需要在运行时持续检查假设是否正确，实现复杂度高
- **适用性**：仅商业数据库可实现，开源数据库难以支持

### 聚合操作实现

**核心定义**：聚合函数对单个属性的**多个值**进行计算，返回单一结果

**标准聚合函数**：
- COUNT, SUM, AVG, MIN, MAX
- 需要遍历数据集中指定属性的所有相关值

**实现方式与后续讨论**：
- 基于排序的聚合（利用已排序数据）
- 基于哈希的聚合（使用哈希表分组）
- 内存管理策略（何时溢出到磁盘）

**关键公式**：聚合操作的时间复杂度
- 排序基础聚合：$O(n \log n + \text{扫描输出})$
- 哈希基础聚合（理想）：$O(n + \text{哈希计算})$

## 第 22 部分

### 聚合实现基础：排序与哈希

实现**聚合操作**（Aggregation）的核心目标是对一个属性的多个值应用**聚合函数**（如 `MIN, MAX, SUM, COUNT, AVG`），最终将其转换为一个**单一标量值**。

**核心挑战：** 系统需要快速找到拥有相同属性值的元组，以便进行分组和计算。

**两种主流实现方法：**
*   **排序聚合 (Sort-based Aggregation)**
*   **哈希聚合 (Hash-based Aggregation)**

> **核心原则：** 尽量避免**随机I/O**，最大化**顺序访问**。
> **经验法则：** 哈希聚合在大多数情况下优于排序聚合，尤其是在磁盘速度较慢时。

---

### 1. 排序聚合 (Sort-based Aggregation)

#### 核心概念与工作原理

*   **核心思想：** 通过**排序**将所有相同值的元组聚集在相邻位置，然后通过一次**顺序扫描**合并结果。
*   **步骤：**
    1.  **扫描与过滤：** 扫描数据表，应用 `WHERE` 子句的过滤条件。
    2.  **早期投影 (Early Projection)：** 立即丢弃非必要的列。例如，只保留需要的 `course_id` 列（这就是**早期物化**策略）。
    3.  **排序：** 对剩余的单列进行排序。
    4.  **扫描并聚合：** 顺序扫描排序后的输出。**只需要追踪上一个输出值**，一旦发现新值，就输出上一个值的聚合结果。


2.  **投影**，丢弃所有列，仅保留 `course_id`。
3.  **排序** `course_id` 列。结果类似 `[101, 101, 102, 102, 102]`。
4.  **扫描**排序后的结果，记录最后一个值：
    - 看到 `101`，记录它为 `last_val`。
    - 看到第二个 `101`，与 `last_val` 相同，跳过。
    - 看到 `102`，输出 `101`（因为值发生了变化），然后将 `last_val` 更新为 `102`。
    - 扫描结束，输出最后一个值 `102`。

**优势：** 实现简单，输出有序（这在某些后续操作中可能有益）。
**劣势：** 排序本身成本高昂，尤其是数据量巨大时。

## 第 23 部分

### 查询优化的核心：尽早丢弃无用数据

#### 1. 去重操作的两种实现路径

- **排序去重法 (Sort-based Deduplication)**
  - **核心思想**：先对“单列”数据进行排序，然后**单次扫描**结果集，利用**游标（cursor）**追踪上一个值。
  - **关键逻辑**：如果当前值等于上一个值，判定为**重复项（duplicate）**，直接丢弃。
  - **适用场景**：当查询本身包含 `ORDER BY` 子句，且去重依据与排序列一致时（如按 `courseID` 排序并去重），可实现“**一石二鸟（kill two birds with one stone）**”——排序操作同时服务于排序需求和去重需求。

- **哈希去重法 (Hash-based Deduplication)**
  - **核心思想**：利用**哈希表（hash table）**或其它**内存数据结构（in-memory data structure）**，在构建哈希表的过程中**即时识别**重复项并丢弃。
  - **优势**：**无需全局排序**，只需维护一个哈希集合，效率更高。因为去重本质上只需要“是否见过”的判断，不需要顺序关系。
  - **关键术语**：**即时丢弃（throw away as you go）**——这是查询执行中**流水线化（pipelining）**思维的体现。

#### 2. 两种策略的取舍与内存瓶颈

- **哈希去重的致命弱点：随机I/O与磁盘抖动**
  - 当哈希表**无法完全装入内存**时，会发生**页面换入换出（page swapping）**，导致**磁盘抖动（disk thrashing）**。
  - **痛点**：刚刚写出的页面需要再次读入——“**为同一块数据支付两次代价**”（pay for the same real estate twice）。
  - **理想目标**：数据从磁盘读入内存后，**一次性完成所有必要处理**，处理后立即释放，**绝不回头**再读一次。

- **排序去重的内存友好性**
  - 排序算法（如外部归并排序）天然支持**流式处理（streaming）**和**分块处理**，对大数据的磁盘访问更可预测、更具顺序性，从而避免随机I/O的代价。

#### 3. 查询优化中的黄金法则：尽早丢弃

- **核心原则**：在查询执行计划中，**尽可能早地**（as early as possible）丢弃无用的数据。
  - **示例操作**：
    - **下推过滤（push down filter）**：在扫描阶段就过滤掉不符合条件的行。
    - **下推投影（push down projection）**：只保留查询需要的列，尽早去掉无关列。
  - **背后的哲学**：整个**查询优化（query optimization）** 的最终目标就是**最小化中间结果集的大小**，减少后续操作要处理的数据量。

#### 4. 现实对比：SQL vs. 手写逻辑

- SQL 的**声明式（declarative）** 特性：**数据库系统自动**决定使用排序去重还是哈希去重，并根据内存限制、数据分布、是否有 `ORDER BY` 等条件做出最优选择。
- 作为**引擎开发者**，你需要理解这些底层区别，才能在**自定义引擎**或**手工调优**时，做出正确的实现决策。

## 外部哈希聚合（External Hashing Aggregate）：一次性内存处理的优化策略

### 核心目标：**避免重复的磁盘I/O**

- **核心理念**：数据从磁盘加载到内存后，应**一次性完成所有必要的处理**，然后丢弃，**绝不再回读**。
- **关键类比**：与外部归并排序（External Merge Sort）类似，采用**多阶段处理**；本课程假设仅需**两阶段**即可完成，无需多趟归并。

---

### 第一阶段：**分区（Partitioning Phase）**

- **操作流程**：
  - 基于**哈希键（Hash Key）** 将数据划分到不同的**桶（Bucket）** 中。
  - 当某个桶在内存中**满了**，立即将其**写出到磁盘**，释放内存，继续填充新数据。
- **核心优势**：最大化**每页内存**的处理效率——每次读入一个页面，就完成所有重分组操作，然后立即写出，避免随机访问。

---

### 第二阶段：**聚合计算（Aggregation Phase）**

- **操作流程**：
  - 从磁盘**逐一读入**每个桶（分区）。
  - 对每个分区，在内存中**构建哈希表**，进行聚合计算。
- **具体逻辑**：
  - **去重（DISTINCT）**：若哈希表中已有该键，直接**丢弃**当前记录。
  - **分组聚合（GROUP BY）**：若哈希表中已有对应键，**增量更新**聚合值（如计数、求和、平均值等），维护**运行中的中间结果**。
- **最终输出**：所有分区处理完毕后，可能还需进行**额外计算**（如求平均需要总和与计数），得到最终正确结果。

---

### 内存假设与磁盘溢出处理

- **理想情况**：如果所有数据都能装入内存，上述过程非常简单直接。
- **现实挑战**：数据量超出内存时，哈希表的**随机访问**模式会**严重拖慢性能**。
- **核心策略**：**通过分区，将随机访问转化为顺序的磁盘读写**——每个桶在磁盘上连续存储，读入时也是顺序的，极大提升吞吐量。

---

### 关键公式与算法总结

- **哈希聚合的I/O成本估算**：
  - **读原始数据**：\( \text{Cost}_{\text{read}} = N \)（N为数据页数）
  - **写分区桶**：\( \text{Cost}_{\text{write}} = N \)（所有数据被写回磁盘一次）
  - **读回分区桶**：\( \text{Cost}_{\text{read\_back}} = N \)（每个桶读回一次）
  - **总成本**：约 \( 3N \)，相比排序聚合（约 \( 2N \log_2 M \)）在数据量大时更优（M为内存页数）。

- **核心约束**：每个**单个桶的大小**必须小于**可用内存大小**，否则无法在第二阶段直接聚合。若桶溢出，则需**多趟哈希**（后续课程讨论）。

## 第 25 部分

### 外部哈希聚合（External Hash Aggregation）

#### 核心目标：最大化顺序I/O
- 通过**将数据分区写入磁盘**，避免随机I/O，使后续读取每个分区时能进行**顺序扫描**。
- **核心思想**：一次只处理一个磁盘上的分区（bucket），为每个分区在内存中构建哈希表，完成聚合（如 `COUNT`、`SUM`、`DISTINCT`）后再释放，重复此过程直到所有分区处理完毕。

---

#### 第一阶段：分区（Partitioning Phase）

- **哈希函数** \( h \)：将输入元组根据分组键（group-by key）映射到指定分区。
- **分区定义**：一个或多个页面的**逻辑分组**，这些页面保证包含**相同分组键**的元组（即同一哈希值的元组会进入同一分区）。
- **内存缓冲区管理**：假设有 \( B \) 个缓冲区（buffer frames）。
    - **1个输入缓冲区**：用于读取待处理的元组（从磁盘或上游操作符的输出）。
    - **\( B-1 \) 个输出缓冲区**：每个分区对应一个输出缓冲区，专门用于写入该分区的数据。
    - 当一个输出缓冲区写满（装满一个页面），就**写入磁盘**，并维护一个内存中的数据结构，记录“哪个分区写到了磁盘的哪个位置”。

- **分区过程示例（以 `DISTINCT` 操作为例）**：
    1.  输入元组通过过滤（Filter）和投影（Projection）操作后，进入分区阶段。
    2.  对每个元组的分组键应用哈希函数，结果 `mod` 分区数，决定该元组进入哪个输出缓冲区（分区）。
    3.  输出缓冲区逐渐填充，写满的页面顺序地刷到磁盘。
    4.  **重要说明**：此阶段**不进行去重或聚合**，纯粹只为将数据分散到不同分区，确保同一分组键的元组落在同一分区内。

---

#### 关键区别：分区 vs. 哈希表冲突解决

- 在分区阶段，**哈希函数只用于决定元组属于哪个磁盘上的分区**，而不是用于在哈希表中查找冲突或重复项。
- 因此，**不存在“哈希表冲突”问题**：每个分区中的元组只是“物理上放在一起”，它本身不是一个哈希表。
- 真正的哈希表，以及冲突解决（如链地址法、开放寻址法），是在**第二阶段**（每个分区加载到内存后）才构建和使用的。
- **简单示例**：`hash(value) mod number_of_partitions` 仅输出一个分区编号（例如 0, 1, 2, ...），不涉及任何哈希表的探测或插入逻辑。

## 第 26 部分

### 哈希划分与临时数据管理

#### 1. 哈希函数在数据划分中的角色

- **核心概念**：将哈希函数**仅作为分区工具**，而非用于精确查找的索引结构。
  - 对每条记录的键进行哈希运算，然后 `mod` 分区数（例如 `hash(key) % N`），直接将记录追加到对应分区的页面末尾。
  - **不检查页面内是否已存在相同键**，即**不进行去重或冲突解决**。
- **关键区分**：此阶段的目标是**将数据集拆分为更小的子集**，以便后续对每个分区单独执行聚合操作（如 `COUNT`, `SUM`）。哈希的目的在于**均匀分散数据**，而非立即完成查询。
- **算法逻辑**：`PageID = hash(Key) mod NumberOfPartitions`，然后 `append(record)` 到该页面。

#### 2. 可能的优化与问题澄清

- **是否可在写入时去重**：理论上，可以在哈希划分过程中检查分区页面是否已存在相同键，若已存在则不写入。但**通常不这样做**，因为这违背了“简单划分”的初衷，会增加不必要的 I/O 和复杂度。
- **哈希冲突与数据倾斜**：所有哈希表面临的挑战（如不同键哈希到同一分区导致倾斜）在此同样存在。若数据分布不均，部分分区可能很快填满，而其他分区空闲。实际处理中需考虑**重新分区或使用更均匀的哈希函数**。

#### 3. 临时数据与崩溃恢复

- **临时数据存储位置**：这些中间分区页面应**存储在临时文件或独立 scratch 区域中**，而非直接写入数据库的主表堆（table heap）。这是为了避免与持久化数据混淆。
- **崩溃恢复策略**：对于查询执行过程中的临时数据，**查询失败即丢弃，不尝试恢复**。
  - **核心原则**：如果数据库正在计算查询时系统崩溃，重启后不会尝试恢复这些中间结果。客户端只需**重新提交查询请求**即可。
  - **设计哲学**：在传统数据库及分布式数据库中，这是常见做法。部分现代系统虽在改进，但大多数仍遵循“失败则重试”的原则，因为临时数据本身不具有持久化价值。
- **关键术语**：**Scratch data**、**Ephemeral partitions**。

#### 4. 总结要点

- **划分阶段**：哈希仅用于分割数据，不关心内容重复。
- **存储隔离**：临时分区的页面应存为独立临时文件。
- **崩溃容错**：查询崩溃后不恢复临时数据，由用户重新发起查询。

## 第 27 部分

### 双阶段哈希聚合：外内存哈希的优雅实现

#### 核心概念：为什么需要两阶段哈希？
**外内存哈希聚合**的核心挑战是：当数据量远超内存容量时，如何高效完成聚合操作（如 `GROUP BY`、`COUNT`、`SUM`）。

**两阶段哈希**通过**先分区、后聚合**的策略，将问题从“处理整个大数据集”变为“处理多个小分区”，每个小分区都能**独立放入内存**。

#### 第一阶段：分区（Partitioning）—— 构建磁盘上的桶

- **目标**：将原始数据**分散写入磁盘**上的多个独立分区（Partitions）。
- **机制**：
    - **不是哈希表**：第一阶段**不**进行去重或聚合。它只是简单的**哈希分桶**，将数据行分配到不同的磁盘分区中。
    - **哈希函数 \(h_1\)**：对分组键（Group Key）使用哈希函数 \(h_1\)，计算结果决定该行进入哪个分区（桶）。
    - **桶溢出处理**：第一阶段直接**忽略重复消除**，只关心“数据该去哪”。
- **容错特性**：
    - **崩溃即重启**：如果查询在运行期间崩溃，重启后**不会尝试恢复**。因为第一阶段是确定性操作，如果需要结果，由用户**重新提交查询**。这是典型的**非持久化、尽力而为**的设计哲学。

#### 第二阶段：重哈希与聚合（Rehashing & Aggregation）—— 在内存中完成计算

- **目标**：对每个分区，**独立构建内存哈希表**，完成最终的聚合操作。
- **关键假设**：每个分区的大小**必须能完全装入内存**。如果分区太大，可能需要进行**递归分区**（递归哈希），但这种情况比直接对整个数据集哈希要好得多。
- **执行流程**：
    1.  **顺序读取分区**：从磁盘读取一个分区的所有页（Pages），**顺序扫描**。
    2.  **新哈希函数 \(h_2\)**：使用**另一个哈希函数**（与第一阶段不同）在内存中构建哈希表。
    3.  **执行聚合**：在内存哈希表中，对每行数据执行所需的聚合函数（如 `SUM`、`COUNT`、`MAX`）。
    4.  **无需回溯**：因为**一个分区的数据之间没有交叉依赖**，处理完一个分区后，该分区数据可以丢弃，**永远不会回头**再查看其他页。
    5.  **输出结果**：当分区处理完毕，将内存哈希表的最终结果**追加**到结果缓冲区（Output Buffer），作为最终查询结果的一部分。

#### 图解流程：从分区到结果

```
[原始数据流] ---> 哈希函数 h1 ---> [磁盘分区 0] [磁盘分区 1] ... [磁盘分区 n]
                              (第一阶段：仅仅分桶，不聚合)
                                      
                                      
[磁盘分区 0] ---> 顺序读取 ---> [内存哈希表 (使用h2)] ---> 聚合运算 ---> [结果缓冲区]
[磁盘分区 1] ---> 顺序读取 ---> [内存哈希表 (使用h2)] ---> 聚合运算 ---> [结果缓冲区]
...
[磁盘分区 n] ---> 顺序读取 ---> [内存哈希表 (使用h2)] ---> 聚合运算 ---> [结果缓冲区]
                              (第二阶段：每个分区独立聚合)
```

#### 核心公式与算法特性

- **第一阶段**：\( \text{PartitionID} = h_1(\text{GroupKey}) \mod N \) （ \(N\) 是分区数）
- **第二阶段**：\( \text{BucketID} = h_2(\text{GroupKey}) \mod M \) （ \(M\) 是内存哈希表大小）
- **算法复杂度**：
    - **I/O代价**：读取一次原始数据（分区写入），再读取一次所有分区（聚合）。总计 **2次全数据I/O**。
    - **内存要求**：最大的分区必须小于可用内存。如果分区大小 \(S_{max} > M\)，则需要**递归哈希**（将分区再次分区）。
- **内存与I/O**：
    - **减少寻道时间**：第一阶段是**顺序写入**（写到磁盘的不同区域）；第二阶段是**顺序读取**。顺序I/O比随机I/O快几个数量级。

#### 内存与I/O的微妙平衡

- **为什么第一阶段要写磁盘？** 因为当内存装不下时，必须利用磁盘作为“扩展内存”。虽然慢，但比完全失败或全表扫描要高效。
- **为什么第二阶段不需要回溯？** 因为哈希函数保证了同一分组键的所有数据都落入了**同一个分区**。因此，不同分区之间**不存在数据依赖**，可以并行、独立处理。这是该算法**优雅的核心**。

#### 关键术语表

| 术语 | 解释 |
|---|---|
| **外内存哈希 (External Memory Hash)** | 数据量超过内存时，利用磁盘辅助完成哈希聚合的技术。 |
| **分区 (Partitioning)** | 第一阶段，将数据根据哈希值分散到多个独立文件/区域。 |
| **重哈希 (Rehashing)** | 第二阶段，对每个分区使用新哈希函数构建内存哈希表。 |
| **递归哈希 (Recursive Hashing)** | 当单个分区仍然过大时，对该分区再次应用分区算法，直到每个子分区能装入内存。 |

## 第 28 部分

## 两阶段哈希聚合算法详解

### 核心思想：分而治之 + 哈希确定性

- **核心概念**：利用**哈希函数的确定性**，将大规模数据分阶段处理，避免一次性将所有数据加载到内存中。
- **关键术语**：**两阶段哈希聚合 (Two-Phase Hash Aggregation)**、**分区 (Partition)**、**哈希表 (Hash Table)**。
- **核心流程**：
    1. **第一阶段**：扫描所有数据，使用**第一个哈希函数 (H1)** 将键值分散到不同的**分区 (partition)** 或**桶 (bucket)** 中。
    2. **第二阶段**：逐个将分区加载到内存中，为每个分区构建一个新的哈希表。
    3. **处理**：在新哈希表中，使用**第二个哈希函数 (H2)** 处理该分区内的数据（例如聚合、计数等）。
    4. **输出**：处理完一个分区后，将结果追加到最终输出缓冲区。

### 为什么需要第二个哈希表？—— 解决哈希冲突

- **问题**：在第一阶段分区后，**不同键可能哈希到同一个分区**（哈希冲突）。
- **解决方案**：第二阶段为每个分区构建一个**真正的哈希表**，使用线性探测 (linear probing) 等标准方法来处理冲突。
- **例子说明**：如果数据集中只有一个值 "445"，直接计数即可。但在**大规模数据集**中，存在大量不同的键可能落入同一分区，必须维护一个**运行计数 (running count)** 的数据结构来跟踪每个键的出现次数。

### 为什么丢弃第一个哈希表并构建第二个？—— 内存限制

- **关键决策**：第一阶段的分区表本质上只是一个**划分工具**，不是完整的哈希表。构建并丢弃它的目的是**避免假设内存能容纳整个巨型哈希表**。
- **设计理念**：通过分区，第二阶段使用的哈希表可以**更小、更节省内存**。
- **确定性保证**：由于哈希函数是确定性的，在扫描第二个分区时，可以确定键 "721" 不可能出现在其他分区中，因此可以安全地丢弃第一个哈希表，释放内存用于处理当前分区。

### 算法优势与应用场景

- **内存友好**：将大问题分解为内存能容纳的小问题，避免内存溢出。
- **适用场景**：数据量远超内存的数据库聚合操作、大规模流式数据处理。
- **关键公式**：
  - 不存在单一公式，但核心逻辑可表示为：
  $$ \text{输出结果} = \bigoplus_{p=1}^{N} \left( \text{哈希聚合}( \text{分区}_p, \text{H2}) \right) $$
  其中 $N$ 是分区数，$\bigoplus$ 表示结果合并（如追加到缓冲区）。

## 第 29 部分

### 分批哈希聚合：处理内存放不下的数据

当数据量超过可用内存时，一次性的哈希表构建是不可行的。**分批哈希 (Partitioned Hashing / Rehash)** 通过两阶段处理来解决这个问题，核心思想是：**先用一个较粗的哈希函数将数据分块，确保每个块都能装入内存，然后对每个块内的数据进行精确聚合**。

#### 第一阶段：分区 (Partitioning)

-   **目标**：将海量数据切分成多个可管理的小块（桶）。
-   **方法**：使用一个**一级哈希函数 (Phase 1 Hash)** 将数据分散到多个文件中。这一步不需要关心键是否重复，只负责粗粒度地分配。
-   **关键点**：这个哈希表的**大小必须足够大**，以确保每个分区内的数据量小于可用内存。这样，后续处理每个分区时，就不会出现内存不足的问题。

#### 第二阶段：重哈希与聚合 (Rehash & Aggregation)

-   **目标**：对每个分区独立进行精确的聚合计算。
-   **方法**：为当前处理的分区构建一个**更小的二级哈希表 (Phase 2 Hash Table)**。
-   **特点**：由于第一阶段保证了每个分区都能装入内存，这个二级哈希表可以正常使用，**不会产生因内存不足导致的假阴性（False Negative）**。
-   **操作**：对于分区内的每一行数据：
    -   如果**键不存在**于二级哈希表中，则添加一个新条目。
    -   如果**键已存在**，则执行 **Upsert** (更新插入)操作，更新该键对应的**聚合运行值**。

### 多聚合函数 (Multiple Aggregation Functions) 的处理

同一条查询可以包含多种聚合操作（如 `AVG`, `SUM`, `MIN`, `MAX`, `COUNT`）。哈希表的**值部分 (Value Portion)** 需要根据聚合类型来存储不同的状态信息。

-   **对于 `SUM` 和 `COUNT`**：直接存储一个**累加和 (Running Sum)** 或一个**计数器 (Running Count)**。每次遇到该键，就更新这个值。
-   **对于 `MIN` 和 `MAX`**：存储当前遇到的最小值或最大值，每次比较更新。
-   **对于 `AVG` (平均值)**：
    -   **需要存储两个值**：`COUNT` (计数) 和 `SUM` (总和)。
    -   **最终计算**：在扫描完整个分区后，不能直接得到结果。需要**第二遍遍历 (Second Pass)**，遍历这个分区最终的哈希表，将每个键的 `SUM` 除以 `COUNT`，得出真正的平均值。

### 算法选择：排序 vs. 哈希

在数据库或游戏引擎的数据处理中，当需要聚合时，有两种主流方法：排序 (Sort-based Aggregation) 和哈希 (Hash-based Aggregation)。

-   **选择依据**：**哈希通常是更好的选择** (Hashing has always been the better one to do)。
-   **原因总结**：
    -   **性能**：哈希的时间复杂度是近线性的（$O(n)$），而排序通常是 $O(n \log n)$。
    -   **局部性**：对于非均匀分布、存在大量重复键的数据，哈希能更早地得到结果，而排序必须先完成全部排序。
    -   **内存效率**：配合分批哈希策略，可以很好地处理溢出磁盘的数据，而排序在内存不足时（外部排序）也需要复杂的归并操作。

## 第 30 部分

# 聚合操作与数据库系统设计中的排序与哈希

## 排序 vs 哈希：聚合操作的核心选择

在数据库聚合操作中，**排序（Sorting）** 与 **哈希（Hashing）** 是两种核心策略。两者的选择直接影响系统性能。

### **核心原则：哈希通常优于排序**

- **实践结论**：在大多数场景下，哈希聚合比排序聚合性能更好。
- **验证方式**：如果数据库系统同时支持两种技术，选择哈希方式通常会获得更好的结果。
- **建设建议**：从零开始构建数据处理系统时：
  - 如果需要支持 **`ORDER BY`** 操作，应 **优先实现排序功能**
  - 利用排序功能可以辅助完成其他聚合操作
  - **有余力后再实现哈希功能**

### **为什么会这样？**

哈希通过键值直接映射，减少了**不必要的全局排序开销**。只有当结果本身需要有序输出时，排序才是必需的。

## 排序优化技术回顾

前面已经讨论过一系列排序优化策略，核心目标都是 **减少I/O开销、提升吞吐量**：

### **关键优化策略**

- **分块I/O（Chunking I/O）**  
  - 将数据分组，**摊还（Amortize）** 磁盘读取的固定开销  
  - 减少频繁的磁盘寻道，最大化顺序扫描的效率

- **最大化顺序扫描（Maximizing Sequential Scans）**  
  - 尽可能让磁盘读取连续，避免随机访问  
  - 顺序I/O比随机I/O快几个数量级

- **双缓冲（Double Buffering）**  
  - 当一线程等待I/O时，另一线程继续处理数据  
  - **避免线程阻塞**，保持CPU忙碌  
  - 实现流水线式的并行处理

## 内存数据库（In-Memory Database）

### **核心思想**

- 假设 **所有数据都存放在内存中**，包括：
  - 表数据（Tables）
  - 中间结果（Intermediate Results）
  - 索引（Indexes）
- **架构优化**：去掉缓冲池（Buffer Pool）等磁盘I/O管理层
- **性能收益**：系统运行速度大幅提升

### **历史与市场现状**

- **背景**：演讲者博士论文研究方向就是内存数据库
- **当初的预期**：
  - 内存越来越便宜、容量越来越大
  - 曾认为内存数据库会成为**未来的主导架构**
- **现实的发展**：
  - **闪存（Flash）** 降价速度更快、容量增长更快
  - 内存数据库市场 **并未如预期那样蓬勃发展**
  - 演讲者坦言：“我错了”

### **经验教训**

技术的演进往往不是单一变量决定的。尽管内存变便宜，但闪存的性价比优势改变了大容量数据存储的市场格局。系统设计需要**综合考量成本、性能和可扩展性**。

## 第 31 部分

### 内存优化与查询执行策略

#### 内存成本下降与传统数据库的演进
- **核心概念**：随着**Flash存储器成本大幅下降**，原本预测内存成本降低将主导市场的观点被修正，数据库系统依然依赖磁盘存储模型。
- **关键术语**：**In-Memory Database**（内存数据库）未如预期普及，传统**Disk-Oriented System**（磁盘导向系统）仍占主流。
- **实际影响**：即便内存足够大，PostgreSQL/MySQL等系统依然保留完整的**磁盘管理机制**（如页锁、缓冲池），不会因内存充裕而自动切换为纯内存执行。

#### 内存友好型优化的局限性
- **排序优化**：当数据**完全适合内存**时，可避免外部排序，直接使用**Quick Sort**（快速排序），但仍需保留**B+ Tree**（B+树索引）结构。
- **系统假设差异**：传统数据库**从底层假设数据不全部驻留内存**，即使增大**Sort Buffer**（排序缓冲区），查询优化器只会在特定查询中跳过磁盘溢出，但**Page Table Lookup**（页表查找）等底层操作无法省略。
- **性能启示**：内存充足可消除磁盘I/O，但**内存管理、并发控制**等开销依然存在，无法完全等同于纯内存系统。

#### 哈希 vs 排序：选择策略
- **低基数数据**：当数据**唯一值数量少**（如性别、状态码），**哈希聚合通常优于排序**。
- **哈希优势**：
  - 直接**一次扫描**维护哈希表，无需全局排序。
  - 即使产生**大分区**（冲突导致），可通过**Recursive Partitioning**（递归分区）进一步拆分，最终全部放入内存或磁盘。
- **特殊场景**：
  - **二元数据**（如0/1）：
    - 优化器可识别为**极低基数**，采用**Heap Sort**（堆排序）思想，**一次扫描**维护两值哈希表即可。
    - 但存在**Divide-and-Conquer**（分治）权衡，哈希仍不一定是绝对最优。
  - **聚合操作**：哈希表维护两个值的统计信息已足够，无需复杂排序。

#### 核心经验法则
- **哈希 vs 排序**：在**大多数聚合场景**，哈希性能更优，但需根据**数据分布**和**优化器智能程度**灵活选择。
- **系统设计启示**：传统数据库即使运行在现代大内存硬件上，其**核心架构仍为磁盘优化**，理解这一点有助于避免性能误判。

## 第 32 部分

### 数据库查询中的分治策略与连接算法优化

#### 1. 分治策略与哈希的权衡

- **核心概念：** 在数据库查询优化中，**分治（Divide and Conquer）** 与 **哈希（Hashing）** 是两种核心策略，它们之间存在性能权衡。
- **关键术语：** **分治策略**、**哈希方法**、**性能权衡**。
- **要点阐述：**
    - **分治策略**通过将大问题拆解为多个小问题来降低复杂度，但在递归或合并阶段可能引入额外开销。
    - **哈希方法**在理想情况下（无冲突、均匀分布）能提供接近 O(1) 的查找速度，但构建哈希表和维护哈希结构也有成本。
    - **权衡点：** 当数据分布不均匀或哈希冲突严重时，哈希的优势会下降，此时分治策略可能更稳定。**讲座者认为，在许多实际场景中，分治策略并不会输给哈希方法**，尤其是在需要处理复杂数据关系时。

#### 2. 连接算法分类与优化重点

- **核心概念：** 数据库查询的核心操作是 **连接（Join）**，有多种算法实现，优化目标是 **“让它跑得快”**。
- **关键术语：** **嵌套循环连接（Nest Loop Join）**、**索引循环连接（Index Loop Join）**、**排序合并连接（Sort Merge Join）**、**哈希连接（Hash Join）**。
- **算法对比与理解：**
    - **嵌套循环连接：** 最简单直观，对于小数据集或索引良好的情况有效。复杂度通常为 O(N * M)，其中 N 和 M 为两表行数。
    - **索引循环连接：** 利用索引加速内层表的查找，减少扫描次数。重点在于索引的选择与维护。
    - **排序合并连接：** 先对两表排序，再并行扫描合并。适用于大数据集且数据已排序或能高效排序的场景。复杂度为 O(N log N + M log M + N + M)。
    - **哈希连接：** 构建哈希表，然后用另一表探测。最适合 **等值连接（Equi-Join）**，在无索引时效率很高。复杂度期望为 O(N + M)。
- **优化核心思路：**
    - **讲座强调：** 连接算法是**最重要的算法**，因为数据库性能瓶颈常出现在连接操作上。
    - **优化方向：** 选择合适算法、利用索引、控制数据分布、减少 I/O 次数。**“花大量时间研究如何优化并让它跑得快”** 是核心工作。

#### 3. 重要公式与算法复杂度

- **嵌套循环连接（无索引）：**
  $$
  \text{复杂度} = O(|R| \times |S|)
  $$
  其中 |R| 和 |S| 分别代表两表行数。

- **哈希连接（等值连接）：**
  $$
  \text{期望复杂度} = O(|R| + |S|)
  $$
  前提是哈希函数均匀分布且桶大小合适。

- **排序合并连接：**
  $$
  \text{复杂度} = O(|R| \log |R| + |S| \log |S| + |R| + |S|)
  $$

#### 4. 总结与前瞻

- **核心要点：**
    1.  **分治与哈希**间的选择需根据数据特性（大小、分布、索引）权衡，没有绝对优劣。
    2.  **连接操作**是数据库查询优化的重中之重，掌握其底层算法是高性能引擎的基础。
    3.  **四种经典连接算法**各有适用场景，优化关键在**减少数据扫描次数**和**利用索引**。
- **后续学习：** 下一节课将深入讲解 **嵌套循环连接**、**索引循环连接**、**排序合并连接** 和 **哈希连接** 的具体实现与优化技巧。
