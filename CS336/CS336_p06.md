## 一、GPU 架构与执行模型

> 主题：GPU 的核心计算单元、线程组织方式与内存层次结构

### 1. 流式多处理器 (SM) 与线程组织

- **流式多处理器 (SM)**：GPU 的核心计算单元，如 A100/H100 包含大量 SM。每个 SM 内置众多 **FP32/FP64 计算单元**，并负责调度大量线程。
- **线程与线程块 (Thread Block)**：
  - **基本原子单元**：一个线程块会被调度到 **单个 SM** 上执行。块内包含大量线程，这些线程协同处理数据。
  - **核心优势**：块内线程可以通过 **共享内存 (Shared Memory)** 高效通信，速度接近 L1 缓存。这是实现高性能（如矩阵乘法）的关键。
  - **跨块通信成本极高**：不同线程块之间无法直接同步或高效通信，因此所有需要频繁交互的数据都应尽量放在同一个线程块内。
- **线程束 (Warp)**：GPU 上**同时执行的线程组**，是比线程块更小的执行单元。线程束机制的核心目的是**减少控制单元硬件开销**，多个线程共享一个指令流，从而将更多硅面积用于计算单元。
- **与 CPU 的区别**：
  - **CPU**：大量硅面积用于**控制单元、分支预测、缓存**等，以优化单线程性能。
  - **GPU**：**简化控制**，通过线程束的 SIMT（单指令多线程）模式并行执行大量线程。

### 2. 内存层次

- **寄存器文件 (Register File)**：每个线程私有的最快存储，是编写高性能代码时最需要“压榨”的资源。
- **共享内存 (Shared Memory)**：块内线程可访问的高速存储，速度接近 L1 缓存。
- **DRAM (全局内存)**：容量大但速度慢，是主要瓶颈之一。
- **缓存**：位于 DRAM 与寄存器之间，用于加速数据访问。

### 3. 波次 (Waves) 与负载均衡

- **核心概念**：在 GPU 上，你**无法精确控制**线程块的实际执行顺序。线程块被 GPU 调度器分组为连续的 **波次（Waves）**，这些波次内的线程块几乎同时执行。
- **均衡原则**：目标是确保所有波次的计算量负载均衡。理想情况下，线程块数量应**能整除 SM 数量**，且每个波次内的线程块工作量应尽可能平均。
- **实现策略**：为了充分利用计算资源，你需要**创建远多于 SM 数量的线程块**，从而让调度器有更多自由度来平衡波次负载。

### 4. 算术强度 (Arithmetic Intensity)

- **核心概念**：**算术强度**是 GPU 性能优化的**首要关注点**。它定义为**浮点运算次数与内存移动字节数的比率**。
- **公式表达**：
  $$ \text{算术强度} = \frac{\text{浮点操作数（FLOPs）}}{\text{内存访问字节数（Bytes）}} $$
- **背景原因**：GPU 的计算能力（FLOPs）增长速度远快于内存带宽（Bytes/s）。因此，许多操作实际上是**内存带宽瓶颈**，而非计算瓶颈。
- **一般规则**：
  - **计算密集型**：例如矩阵乘法（GEMM），如果实现得当，它是高算术强度的操作。
  - **内存受限型**：几乎所有其他操作（如元素级处理、softmax、激活函数等）都受限于内存带宽。
- **优化目标**：尽可能提高算术强度，即在每次内存访问中做更多计算。

### 5. 黄金法则

- **尽量将数据保留在 SM 内部的寄存器或共享内存中，避免频繁访问全局内存。**
- **线程块是性能优化的基本单元**：设计内核时，首先考虑如何划分线程块以及块内线程如何协作。

## 二、性能分析与基准测试方法

> 主题：如何正确测量 GPU 程序性能并定位瓶颈

### 1. 核心原则

- **编写高性能代码时，必须始终进行基准测试和性能分析（Profile）**。这是唯一能准确定位性能瓶颈的方法。
- **常见误区**：很多人会凭借直觉认为“某个部分是瓶颈”，并花大量时间优化它，结果发现那根本不是问题所在。
- **正确实践**：
  1. **使用高性能分析工具**：利用 Nsight、NVIDIA Nsight Systems、NVIDIA Nsight Compute、PyTorch Profiler 等工具进行精确的性能瓶颈分析。
  2. **识别瓶颈**：工具能清晰告诉你代码的哪一部分限制了整体性能。
  3. **集中优化**：一旦精确定位瓶颈，就集中精力优化代码的关键部分。

### 2. 基准测试 (Benchmarking)：测量“跑多快”

#### 核心概念：测量**端到端执行时间**

- **目的**：准确测量函数**稳定状态**下的运行时间，用于比较不同实现的性能。

#### 关键步骤：预热与计时

1. **预热迭代（Warm-up Iterations）**：**绝对必要**的步骤。
   - **原因**：首次运行代码时，后台有大量初始化：**机器代码编译**、**指令发送到GPU**、**CUDA内核即时编译**等。
   - **做法**：正式测试前，让函数运行若干次（如5-10次），确保后续计时测量的是**稳定状态**下的性能。

2. **多次试验（Multiple Trials）**：多次重复测试并统计（如取中位数或平均值），减少随机误差。

#### GPU同步问题

- **核心概念**：CPU和GPU是**独立计算单元**，两者可并行运行。CPU在将任务分发给GPU后**不会等待GPU完成**，而是继续执行后续代码。
- **问题**：若不在计时前后强制同步，则CPU计时器测量的可能是**调度时间而非GPU实际执行时间**。
- **解决方案**：调用 **`torch.cuda.synchronize()`**，阻塞CPU直到GPU上所有已排队的任务执行完毕。

#### 基准测试代码模板

```python
import torch
import time

def benchmark_cuda_op(op, repeats=10, warmup=3):
    # 预热步骤
    for _ in range(warmup):
        _ = op()
        torch.cuda.synchronize()
    
    # 正式计时
    timings = []
    for _ in range(repeats):
        torch.cuda.synchronize()  # 确保开始前GPU空闲
        start = time.time()
        _ = op()
        torch.cuda.synchronize()  # 等待GPU完成
        end = time.time()
        timings.append(end - start)
    
    # 返回平均时间
    return sum(timings) / len(timings)
```

### 3. 性能分析 (Profiling)：查找“时间花在哪”

- **核心概念**：进入函数内部，详细分析每个操作分别占用了多少时间。
- **基准测试 vs. 性能分析**：
  - **基准测试**：回答“**有多快？**”（端到端时间）
  - **性能分析**：回答“**为什么这么慢？**”（内部时间分布）

#### 性能分析器的失真与开销

- **分析器开销（Profiler Overhead）**：使用分析器会**引入额外开销**，导致测量的CPU/GPU时间比实际运行时间更长。
- **应对策略**：对于宏观性能（毫秒级），分析器影响不大；但若关注**微秒级**优化，结果可能失真。

### 4. 案例分析：MLP 基准测试

- **模型结构**：**16层**、**128维**的全连接MLP，激活函数为**GELU**。
- **计算流程**：
  ```python
  model = MLP(dim=128, depth=16)
  x = torch.randn(batch_size, 128)
  for _ in range(5):
      out = model(x)
      loss = out.mean()
      loss.backward()
  ```
- **关键发现**：
  - **运行时间与步骤数呈线性关系**：每增加一次前向/反向传播，总运行时间增加约固定值。
  - **运行时间与层数呈线性关系**：从2层扩展到5层，运行时间也按比例增加。
  - **矩阵乘法是主要瓶颈**：在前向和反向中，矩阵乘法操作始终是计算时间占比最高的部分。

## 三、GPU 运算的调用栈与性能分析实战

> 主题：深入理解 GPU 底层执行细节，从简单操作到复杂算子

### 1. 冰山之下：`A + B` 的调用链

一个看似简单的向量加法 (`torch.add(A, B)`)，实际上经历了一个完整的 **CPU -> GPU 调用链**：

1. **Python 接口**：你调用的 `A + B` 首先被 PyTorch 的 Python 接口捕获。
2. **调度器 (Dispatcher)**：根据张量的数据类型和设备分派到正确的后端实现。
3. **向量化逐元素内核**：调度器选择并调用一个为 CUDA 优化的原生函数。
4. **CUDA 内核启动 (Kernel Launch)**：CPU 通过 `cudaLaunchKernel` 函数将加法指令发送到 GPU。
5. **GPU 执行**：GPU 并行进行加法运算。
6. **CUDA 设备同步**：CPU 通过 `cudaDeviceSynchronize` 等待 GPU 完成操作。

**性能关键洞察**：对于简单的逐元素操作，**瓶颈在于 CPU 端的调度和启动开销**，而非 GPU 的并行执行能力。

### 2. 矩阵乘法 (`A @ B`) 的调用链

- **调用流程**：
  1. **张量接口**：调用底层接口。
  2. **CUTLASS 库**：NVIDIA 的高性能矩阵乘法 CUDA 库。
  3. **参数化内核**：根据矩阵尺寸和其他参数，实例化一个特定的、高度优化的内核。
  4. **内核启动 & 设备同步**：与加法相同。
- **性能关键洞察**：由于矩阵乘法计算量远大于逐元素加法，**瓶颈从 CPU 调度转移到了 GPU 计算**。

### 3. 复杂算子的内部解构：`torch.cdist`

- **核心概念**：像矩阵乘法这类“简单”算子，CPU操作与GPU内核几乎一一对应。**复杂算子**（如`torch.cdist`）则会展开为**多个底层GPU操作**。
- **计算逻辑拆解**：要计算两个矩阵的成对欧氏距离，底层需要：
  1. **计算点积**：需要矩阵乘法和逐元素操作。
  2. **开平方根**：对每个距离值进行 `sqrt()` 操作。
- **瓶颈识别**：**矩阵乘法 (GEMM)** 通常占据 **78%** 的 GPU 计算时间，是绝对的主导瓶颈。

### 4. 融合内核 (Fused Kernel) 的典范：GELU 与 Softmax

- **核心概念**：现代 GPU 计算的核心挑战之一是**内存墙**。**融合内核**是解决此问题的关键策略。
- **与传统非融合操作的对比**：
  - **非融合方式**：执行多个独立内核，产生多次内核启动开销和额外的显存读写。
  - **融合方式**：单个内核在一个 pass 内完成所有计算。
- **优势**：
  - **减少内核启动开销**：多次 GPU 内核启动合并为一次。
  - **提升内存带宽利用率**：数据驻留在寄存器中，无需反复读写全局内存。

### 5. 使用 NVIDIA Nsight Systems 进行时间线分析

- **核心工具**：NVIDIA Nsight Systems 提供 **CPU 与 GPU 活动的时间线视图**。
- **关键视图**：
  - **CUDA Hardware / GPU 活动**：显示 GPU 上实际执行的内核和内存操作。
  - **CPU Thread / 线程活动**：显示 CPU 端的活动。
  - **NVTX (NVIDIA Tools Extension)**：**核心标注手段**，将分析器时间线上的活动映射回具体的源代码片段。
- **MLP 执行流程分析**：
  1. **启动与初始化**：程序开始后，有一段 CPU 活动但 GPU 几乎空闲，这是 PyTorch 的初始化系统开销。
  2. **模型构建**：GPU 开始分配显存。
  3. **首次执行 / 动态编译**：首次调用算子时，PyTorch 会进行**运行时编译**，属于初始化开销。
  4. **稳定运行阶段**：时间线变得“规整”，CPU 快速提交任务，GPU 异步执行。

### 6. CPU/GPU 异步并行与打印损失的影响

- **核心概念**：CPU 和 GPU 是独立的执行设备，CPU 负责调度，GPU 异步执行。
- **CPU“超前运行”**：在 GPU 处理当前层时，CPU 可能已经调度了后续层。
- **打印损失的影响**：
  - **问题**：`print(loss)` 触发了一个**隐式的CUDA流同步**，强制CPU等待GPU完成对应任务。
  - **效果**：原本CPU可以超前GPU多个步骤，现在被限制为 **CPU与GPU基本同步**。
  - **工程设计启示**：在性能敏感的训练循环中，避免不必要的同步，可每隔N步打印一次损失。

## 四、CUDA 内核编写与性能优化

> 主题：从朴素实现到融合 CUDA 内核的优化路径

### 1. GELU 激活函数

- **精确公式**：`GELU(x) = x * Φ(x)`，其中 `Φ(x)` 是标准正态分布的累积分布函数。
- **近似公式**（常用实现）：
  $$ \text{GELU}(x) \approx 0.5x\left(1 + \tanh\left(\sqrt{\frac{2}{\pi}}(x + 0.044715x^3)\right)\right) $$

### 2. 朴素实现的性能瓶颈

- **错误示范**（低效的 PyTorch 代码）：
  ```python
  value = 0.5 * x * (1 + torch.tanh(math.sqrt(2 / math.pi) * (x + 0.044715 * x**3)))
  ```
- **性能开销来源**：
  1. **高延迟**：每次内核启动都有固定的开销。
  2. **带宽消耗**：中间结果需要在全局内存中读写。
  3. **无优化空间**：GPU 无法根据数据流的依赖关系进行跨指令优化。
- **基准测试结果**：
  - 朴素 PyTorch 实现：**~8.1 ms**
  - 融合 PyTorch 实现：**~1.1 ms**

### 3. CUDA 内核编写基础

#### 执行模型与层级结构

- **网格（Grid）** → **线程块（Block）** → **线程（Thread）**
- **坐标系统**：使用 **行坐标（row）** 和 **列坐标（col）** 定位元素。

#### 内核函数与包装器

- **内核**：实际运行在 GPU 上的函数，使用 `__global__` 关键字声明。
- **包装器**：运行在 CPU 上的函数，负责检查输入、分配输出、定义网格和块大小、启动内核。

#### 关键检查项

1. **设备检查**：确保输入张量位于 GPU 设备。
2. **连续性检查**：确保输入张量在内存中是连续的。
3. **输出张量分配**：使用 `torch.empty_like(x)` 而非 `torch.zeros_like`，节省不必要的清零操作。

#### 网格与块大小计算

- **元素总数**：`n_elements = x.numel()`
- **块大小**：通常选择 128、256 等。
- **块总数**：`num_blocks = ceil(n_elements / block_size)`
- **向上取整除法**：`cdiv(a, b) = (a + b - 1) // b`

#### 线程坐标计算

```cuda
int idx = blockIdx.x * blockDim.x + threadIdx.x;
if (idx < n) {
    // 执行实际计算
}
```

#### GELU 内核实现示例

```cuda
__global__ void gelu_kernel(float* x, float* y, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < n) {
        float xi = x[idx];
        y[idx] = 0.5f * xi * (1.0f + erff(xi / sqrtf(2.0f)));
    }
}
```

### 4. 性能对比结果

| 实现方式 | 执行时间 |
|---------|---------|
| 手动 Python | ~8.1ms |
| PyTorch 原生 | ~1.1ms |
| CUDA 手写内核 | ~1.8ms |
| Triton 手写内核 | ~1.85ms |
| Torch Compile | ~1.4ms |

**关键洞察**：
- CUDA 实现远超手动 Python，接近 PyTorch 优化实现。
- Triton 与 CUDA 性能几乎持平，但 **Triton 开发效率极高**。
- Torch Compile 性能优于手写 CUDA/Triton，对于简单逐元素运算，**开发者工作量几乎为零**。

### 5. 何时需要手写 CUDA/Triton？

- **不需要手动优化的场景**：对于**简单算子融合**或**标准矩阵乘法**，Torch Compile 已做到极致。
- **需要手写的场景**：当算法涉及**复杂的内存访问模式**或**非标准的计算结构**时，例如：
  - **Flash Attention**：通过 tile-based 分块和在线 softmax 重计算来避免大 attention 矩阵的显存读写。
  - **自定义的稀疏操作**：如结构化稀疏、动态稀疏。
  - **特殊的硬件特性利用**：如利用 Tensor Core 进行非标准的混合精度计算布局。
- **最终建议**：**先用 PyTorch + torch.compile 快速验证和优化，当它成为瓶颈或无法表达算法时，再下探到 Triton/CUDA。**

## 五、Triton 编程入门

> 主题：使用 Triton 编写高效 GPU 内核

### 1. Triton 核心概念

- **编程模型**：开发者只需在 **SM 级别（线程块）** 进行思考，无需关注单个线程的管理。
- **自动优化**：
  - **内存合并**：自动将内存读取操作分组为相邻的突发模式访问。
  - **共享内存**：自动管理 SM 内多线程写入内存时的同步。
- **关键优势**：
  - **性能强大**：可以显著超越许多 PyTorch 实现。
  - **调试便利**：全部使用 Python 编写，可以**逐步调试**。
  - **生态友好**：开发者仍处于熟悉的 **Python** 环境。

### 2. 从线程到块的视角转换

- **核心思想**：在 Triton 编程模型中，**不再以单个线程为单位思考**，而是将整个 **块（Block）** 视为一个整体进行向量化操作。
- **偏移量的本质**：不是单一数值，而是一个 **向量**，包含了当前块内所有坐标的集合。
- **边界处理**：位于数据末端时，偏移量可能超出边界，因此需要 **掩码（Mask）** 来过滤超出向量边界的元素。

### 3. Triton GELU 内核实现

```python
import triton
import triton.language as tl

@triton.jit
def gelu_kernel(x_ptr, y_ptr, n_elements, BLOCK_SIZE: tl.constexpr):
    pid = tl.program_id(0)
    offsets = pid * BLOCK_SIZE + tl.arange(0, BLOCK_SIZE)
    mask = offsets < n_elements
    
    x = tl.load(x_ptr + offsets, mask=mask)
    # GELU 近似计算
    y = 0.5 * x * (1 + tl.tanh(tl.sqrt(2 / math.pi) * (x + 0.044715 * x * x * x)))
    tl.store(y_ptr + offsets, y, mask=mask)
```

### 4. Triton Softmax 内核实现

- **核心设计**：每个块处理矩阵的一行，块大小设置为列数，网格大小设置为行数。

```python
@triton.jit
def softmax_kernel(x_ptr, y_ptr, stride, n_cols, BLOCK_SIZE: tl.constexpr):
    row_idx = tl.program_id(0)
    row_start_ptr = x_ptr + row_idx * stride
    col_offsets = tl.arange(0, BLOCK_SIZE)
    mask = col_offsets < n_cols
    
    row = tl.load(row_start_ptr + col_offsets, mask=mask, other=float('-inf'))
    
    row_max = tl.max(row, axis=0)
    row_exp = tl.exp(row - row_max)
    row_sum = tl.sum(row_exp, axis=0)
    softmax_row = row_exp / row_sum
    
    tl.store(y_ptr + row_idx * stride + col_offsets, softmax_row, mask=mask)
```

### 5. 性能对比结果（Softmax）

| 实现方式 | 运行时间 |
| :--- | :--- |
| 手动实现（Python 朴素） | **3.7 秒** |
| Torch compile | **1.3 秒** |
| PyTorch 原生实现 | **1.5 秒** |
| Triton 实现 | **1.9 秒** |

- **关键观察**：Torch compile 的实际表现优于原生 PyTorch 实现。
- **Triton 性能**：虽然比 PyTorch 原生略慢，但仍在可接受范围内。

### 6. 底层 PTX 代码分析

- **PTX (Parallel Thread Execution)** 是 NVIDIA GPU 的底层指令集架构，是 CUDA 代码编译后的中间表示。
- **PTX 关键特征**：
  - **内存加载指令**：`ld.global` — 全局加载指令。
  - **合并访问**：编译器自动优化，一次性加载多个数据。
  - **浮点运算指令**：`mul.f32`（单精度浮点乘法）。
- **结论**：尽管 PTX 看起来步骤繁琐，但由于是**寄存器级**的向量化操作，理论上**运行速度极快**。

## 六、关键公式与算法

### 1. 算术强度

$$ \text{算术强度} = \frac{\text{FLOPs}}{\text{Bytes}} $$

### 2. 向上取整除法

$$ \text{blocks} = \lceil \frac{\text{elements}}{\text{block\_size}} \rceil = \frac{\text{elements} + \text{block\_size} - 1}{\text{block\_size}} $$

### 3. GELU 激活函数

$$ \text{GELU}(x) = x \cdot \Phi(x) $$

近似公式：
$$ \text{GELU}(x) \approx 0.5x\left(1 + \tanh\left(\sqrt{\frac{2}{\pi}}(x + 0.044715x^3)\right)\right) $$

### 4. Softmax 函数

$$ \text{Softmax}(x_i) = \frac{e^{x_i}}{\sum_{j} e^{x_j}} $$

### 5. 欧氏距离

$$ d(\mathbf{p}, \mathbf{q}) = \sqrt{\sum_{i=1}^n (p_i - q_i)^2} $$

### 6. 线程坐标计算

$$ \text{全局线程ID} = \text{blockIdx.x} \times \text{blockDim.x} + \text{threadIdx.x} $$

### 7. Triton 块索引计算

$$ \text{线程块起始位置} = \text{block\_id} \times \text{block\_size} $$
$$ \text{数据元素索引} = \text{block\_id} \times \text{block\_size} + \text{thread\_offset} $$
