# CS149_p14

## 课程概述：多核架构与并行程序优化

> 主题：本讲系统性地介绍了多核架构、并行程序优化、GPU编程、内存一致性模型以及大规模数据处理框架的核心概念与技术。

## 一、多核架构与缓存一致性

### 核心概念
- **多核架构**：理解硬件如何组织多个处理器核心
- **缓存层次结构**：L1/L2/L3缓存的协作方式
- **缓存一致性协议（MSI/MESI协议）**：确保多核环境下数据一致性

### MSI协议（Modified-Shared-Invalid）
- **Modified（已修改）**：缓存行被当前核心独占且已修改，其他缓存中该副本为I
- **Shared（共享）**：多个核心拥有同一数据副本，与内存一致
- **Invalid（无效）**：缓存行中的数据已过时或不存在于缓存中

**关键原则**：
- **无需一致性通信**：当缓存行处于M或S状态，且无其他缓存持有M时，读取操作不需任何总线事务
- **写入独占**：仅在M状态下才能直接写入；若处于其他状态，需先发送请求独占信号

### MESI协议：从三态到四态的演进
- **E（Exclusive）状态**：缓存行是干净的（与内存一致），且仅被当前核心持有
- **核心优势**：从E状态写入时，可直接在本地将状态翻转（flip）为M，**无需任何总线事务**
- **S→E转换不存在**：无法从共享悄悄升级为独占，因为无法保证没有其他核心持有该数据

### 状态转换要点
- **E→M**：本地写入，无需总线事务（效率最高）
- **E→S**：监听到其他核心的读请求，降级为共享
- **S→I**：收到其他核心的写无效化请求
- **M→S**：监听到其他核心的读请求，需先写回内存，然后降级为共享
- **M→I**：收到其他核心的写无效化请求，需先写回内存

### 写入流程（从S→M）
- 当前缓存行处于S状态，需要写入
- **无需发送请求**：因为缓存行在S状态下，其他缓存可能有副本，但不会有M状态的竞争
- **直接升级**：缓存从S转为M，同时保证其他缓存中无M状态副本

### 读取流程（从I→S）
- 处理器一尝试读取缓存零持有的M状态数据
- 缓存零将M状态的缓存行写回内存（刷新操作），自身状态转为S
- 缓存一从内存获取更新后的值，状态设为S
- 缓存零转换为I

### 现代优化：五状态协议（MESIF/MOESI）
- **第五状态（F/E/O）**：引入Forwarding（转发）或Owned（拥有）状态
- **作用**：当缓存持有数据时，可直接转发给其他缓存，**无需刷新回内存**
- **优势**：减少内存访问延迟，提升效率

### 缓存行粒度与数据完整性
- 操作单元是完整的缓存行，而非单个变量
- 冲刷/刷新的数据量是整个缓存行（包含未被修改的其他数据）
- 每个地址拥有独立的状态机，不同地址的缓存行状态彼此不直接相互影响

### 驱逐（Eviction）行为
- 如果缓存行是脏的（Modified状态），驱逐时会将数据写回主存，然后标记为Invalid
- 如果缓存行是干净的（Shared/Exclusive状态），直接标记为Invalid，无需写回
- 驱逐与状态刷新是缓存容量管理的常规操作，不属于缓存一致性协议的核心逻辑

## 二、原子操作与无锁编程

### 互斥与原子操作
- **互斥（Mutual Exclusion）**：确保同一时刻只有一个线程可以执行某段代码（临界区）
- **原子操作（Atomic Operation）**：互斥的细粒度实现，保证读取-修改-写入整个序列不可中断

### 原子比较交换（Atomic Compare-and-Swap, CAS）
- **核心机制**：硬件层面保证的原子操作
  - 读取变量的当前值
  - 与参数compare的值比较
  - 若相等，则写入新值；否则，写入旧值（保持不变）
- **关键特性**：全过程中无其他线程/核心可以干扰
- **硬件原语**：平台提供CAS指令，编译器/运行时保证其原子性

**CAS伪代码**：
```c
bool compare_and_swap(int* ptr, int expected, int new_value) {
    if (*ptr == expected) {
        *ptr = new_value;
        return true;
    }
    return false;
}
```

### 从CAS构建原子最小值操作（Atomic Min）
- **核心原理**：读取-检查-尝试写入
  - 先读取当前值
  - 判断新值是否更小
  - 尝试用CAS原子地写入（如果当前值仍等于之前读取的值，则写入成功）
  - 若CAS失败（值已被其他线程修改），则重新读取并重试

```pseudo
function atomic_min(memory, new_val):
    while true:
        old_val = read(memory)
        if new_val >= old_val:
            break   // 新值不小，无需更新
        if CAS(memory, old_val, new_val):
            break   // 写入成功
        // 否则，loop 重试
```

### 无锁编程（Lock-Free Programming）
- **核心思想**：不强制互斥，而是通过原子操作和乐观假设来确保最终一致性
- **工作流程**：乐观读取 → 执行计算 → 原子尝试写回（CAS）→ 失败则重试
- **关键术语**：
  - **乐观假设（Optimistic Assumption）**：假设不会发生冲突，先干活再验证
  - **回退（Rollback/Retry）**：冲突发生时，所有计算可能作废，必须重来

### 无锁 vs 互斥

| 互斥（Mutex） | 无锁（Lock-Free） |
|--------------|------------------|
| 强制阻塞其他线程，直到自己完成 | 让所有线程同时运行 |
| 保证无冲突，但可能造成线程休眠 | 不阻塞，但冲突时重试 |
| 简单直接，但可能出现死锁、优先级反转 | 更精细，但可能浪费计算（重试） |

### 原子操作的开销与缓存一致性协议
- 原子操作在缓存一致性协议层面，必须被视为一个单一的"写"操作
- **原子操作会强制缓存行直接从I（无效）状态转换到M（修改）状态**，绕过了中间的S状态
- 即使CAS失败，也必须发起总线读X（Bus Read Exclusive）操作
- 总线读X会将数据载入缓存行，并锁定该缓存行，阻止其他处理器抢走

### 饥饿（Starvation）与公平性
- **饥饿定义**：系统中某些线程持续在进步，而指定线程永远无法成功完成操作
- **关键结论**：无锁（Lock-Free）与互斥（Mutex）在公平性上是正交的
- **缓解方法**：可以在失败后加入退避策略（如指数退避），让等待线程休眠

### 事务内存（Transactional Memory）
- **扩展思路**：将无锁思想推广到任意函数，该函数可能读写多个变量
- **工作方式**：
  1. 函数执行时，系统记录所有读写操作
  2. 提交时，系统自动检测与其它线程的读写是否冲突
  3. 冲突 → 回退一个线程（类似数据库事务）

## 三、并行程序优化策略

### 性能瓶颈识别
- **负载不均衡**：某些核心空闲，其他核心过载
- **缓存性能差**：频繁的缓存缺失导致性能下降

### 优化技术
1. **循环优化**：
   - 交换循环顺序以改善空间局部性
   - 循环分块（Tiling）减少缓存缺失
2. **工作负载均衡**：动态分配任务避免闲置核心
3. **数据并行思维**：将问题分解为可并行处理的独立数据块

### 典型问题模式
```cpp
// 缓存不友好示例
// 按列访问矩阵（空间局部性差）
for (int j = 0; j < N; j++)
    for (int i = 0; i < M; i++)
        sum += matrix[i][j];  // 每次访问不同行

// 优化版：按行访问
for (int i = 0; i < M; i++)
    for (int j = 0; j < N; j++)
        sum += matrix[i][j];  // 顺序访问相邻内存
```

## 四、GPU架构与CUDA编程

### 核心架构特征
- **SIMT（单指令多线程）**：GPU的并行执行模型
- **线程层次结构**：
  - **线程块（Thread Block）**：多个线程组成的工作组
  - **线程束（Warp）**：32个线程一起执行同一指令
- **Warp Divergence**：同一warp内不同分支路径导致性能下降

### CUDA编程模型
```cuda
// GPU内核启动示例
__global__ void kernel(float* data) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    // 线程块内的索引计算
    data[idx] = data[idx] * 2.0f;
}

// 主机端调用
kernel<<<numBlocks, blockSize>>>(d_data);
```

### CUDA线程块调度与资源限制
- **每个线程块需要的资源**：共享内存量、线程数量
- **驻留线程块数公式**：
  \[
  \text{最大驻留块数} = \min\left( \frac{\text{SM 最大线程数}}{\text{每块线程数}}, \frac{\text{SM 共享内存总量}}{\text{每块共享内存需求}} \right)
  \]
- **调度决策**：调度器优先考虑资源可用性，将线程块分散到不同SM以利用整个GPU的并行能力

### Warp执行模型
- GPU并不直接执行单个CUDA线程，而是将32个CUDA线程打包成一个Warp来执行
- **锁步执行（Lock-step）**：所有线程在同一时刻执行同一指令（但可能处理不同数据）
- **透明并行**：GPU的并行性（Warp）对开发者透明，你只需创建大量线程，硬件负责将其高效打包并调度到ALU上

### SIMD架构的实现差异：CPU与GPU

| 特性 | CPU SIMD | GPU SIMD |
| :--- | :--- | :--- |
| **指令生产者** | 编译器（显式生成宽指令） | 硬件调度器（隐式将标量指令分组） |
| **宽度决定** | 机器指令集架构（ISA）固定 | GPU架构的内部warp大小（如32） |
| **灵活性** | 需重新编译以适配新宽度 | 同一份二进制代码可适配不同宽度 |
| **分支处理** | 通常依赖编译器生成掩码/混合指令 | 硬件自动处理warp内分支发散 |

## 五、内存一致性模型

### 顺序一致性（Sequential Consistency, SC）
- 所有线程的指令可以被排列在一条单一的、全局的时间线上
- 每个线程内部的指令顺序（程序顺序）必须得到保留
- 所有线程之间的指令交叉必须有一个全局可见的、一致的顺序

### 松弛一致性（Relaxed Consistency）
- 当在不同线程之间观察时，单一全局时间线可能不存在
- 一个线程对不同地址的写入，在另一个线程看来，顺序可能与程序顺序不同
- **本质区别**：这与一致性（Coherence）是正交的概念
  - 一致性关注的是同一内存地址上的操作顺序
  - 松弛一致性关注的是不同地址上的操作顺序在不同线程间的可见性

### 关键区分：单线程 vs 多线程可见性
- **单线程内**：编译器（优化）和硬件绝对不能破坏程序顺序带来的结果
- **跨线程观察**：松弛一致性的规则只适用于一个线程如何观察另一个线程对不同地址的写入顺序

### 缓存一致性vs内存一致性
- **缓存一致性**：保证单变量在多核间可见
- **内存一致性**：定义多线程读写多个变量的全局顺序

### 解决方案：内存屏障（Memory Barrier）
```cpp
// 线程A
x = 1;    // 写x
y = 1;    // 写y

// 线程B
if (y == 1)       // 可能先看到y=1
    assert(x == 1); // 可能失败！x可能仍是0
```

## 六、MapReduce与大规模数据处理

### MapReduce编程模型
- **核心思想**：移动计算而非移动数据
- **仅有两个核心函数**：
  - **Map函数**：接受输入数据的一条记录，产生键值对作为输出
  - **Reduce函数**：接收Map阶段输出的同一Key的所有Value列表，聚合处理后输出结果

### 分布式执行流程
```
阶段 1: Map
├─ 数据分布在N台机器上
├─ 每台机器在其本地数据上运行Map函数
└─ 输出：大量中间键值对

阶段 2: Shuffle（自动发生）
├─ 系统将所有具有相同Key的键值对
├─ 通过网络传输到同一台机器
└─ 形成：(Key, [Value1, Value2, ...])

阶段 3: Reduce
├─ 每台机器对收到的(Key, Value列表)执行Reduce函数
└─ 输出：最终聚合结果
```

### 关键公式
**Map阶段**：
$$ \text{Map: } (k_{\text{in}}, v_{\text{in}}) \rightarrow \{(k_1, v_1), (k_2, v_2), \ldots, (k_n, v_n)\} $$

**Shuffle阶段**：
$$ \text{Shuffle: } \{(k_i, v_i)\} \rightarrow \{(k_i, [v_{i1}, v_{i2}, \ldots, v_{im}])\} $$

**Reduce阶段**：
$$ \text{Reduce: } (k_i, [v_{i1}, v_{i2}, \ldots, v_{im}]) \rightarrow \text{Final Output} $$

### MapReduce的痛点
- 所有数据通信通过写入文件系统 → 另一个节点读取完成，每次shuffle都涉及磁盘I/O
- 极度担心硬盘带宽：大量数据需要写入磁盘再读出，带宽成为瓶颈
- 这种设计低效，尤其不适合迭代计算或需要多次shuffle的场景

### Spark的革新：内存优先
- **核心思想**：在内存中完成所有操作，避免磁盘I/O的带宽开销
- **核心操作符**：使用正确的数据并行操作符，不仅仅是MapReduce的"Map → 按键分组 → Reduce"模式
- **关键不同**：Spark不依赖共享文件系统作为通信中介，而是通过内存中数据分区实现高效的shuffle

## 七、延迟 vs. 带宽：核心关注点

### 为何不担心延迟？—— 延迟可被并行性隐藏
- 在极度并行（Massive Parallelism）的环境中，延迟不再是瓶颈
- 所有操作都在多线程或大规模并行的框架下进行，延迟被完美隐藏

### 为何极度担心带宽？—— 吞吐量决定一切
- 真正的问题在于全数据集在计算阶段间的移动
- 即使计算逻辑非常简单，数据移动的带宽才是真正的瓶颈

### 案例警示：并行性迷思 vs. 局部性真理
- **场景**：处理约500GB的数据集
- **方案A**：使用5000个节点的Spark/MapReduce集群
- **方案B**：一台笔记本电脑（单核）
- **结果**：笔记本电脑的单核性能超越了5000节点的Spark集群
- **根本原因**：利用局部性（将数据留在内存中处理），单机可以跟上分布式集群的迭代速度

### 现代启示
- 2023年，在AWS上租用一台512GB内存的机器，成本仅为3美元/小时
- 与其花精力让你的代码对10000个节点的集群进行故障容忍和弹性恢复，不如考虑局部性，购买一台4TB内存的机器

## 八、微服务架构的成本权衡

### 核心矛盾：弹性扩展 vs. 通信开销
- **传统观点**：Amazon Lambda的优势在于弹性扩展和按需付费
- **现实教训（Prime Video案例）**：
  - Prime Video最初将整个系统构建为Lambda微服务（数千个小型函数）
  - 一年半后，他们回退到少数几个专用服务器上
  - **关键发现**：通信开销成为了瓶颈
  - **量化结果**：等性能下，成本降低10倍

### 何时才该"从第一天起就考虑扩展"？
- **决定性因素**：数据集大小
- 如果数据量巨大到单机无法容纳（例如TB级数据库），那么必须从第一天起就按分布式/可扩展系统来构建
- **核心原则**：扩展并非总是银弹。当数据可以放在单机上，或通信成本远高于计算成本时，过度扩展是反效率的
