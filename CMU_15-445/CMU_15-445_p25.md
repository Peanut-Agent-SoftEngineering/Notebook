## 一、云计算时代的数据库系统：OLAP与OLTP实践

> 主题：本讲探讨了数据库系统从传统本地部署向云端迁移的核心驱动力，并深入剖析了云原生OLAP系统BigQuery和OLTP系统Amazon Aurora的架构设计。

### 从传统本地部署到云端

- **传统模式**：Oracle、Teradata等公司向客户销售“数据库一体机”（database appliance），硬件存放在**客户本地数据中心**（on-premise data centers）
- **当前趋势**：企业和客户正将数据和负载迁移至云端，典型系统包括：
  - Google Cloud: **BigQuery**
  - AWS: **Redshift**
  - Azure: **Synapse**
  - 跨平台: **Snowflake**、**Databricks**

### 迁移至云端的核心驱动力

- **规模经济（Economies of Scale）**：超大规模云提供商能以**极低成本**提供硬件资源
- **硬件创新（Innovation）**：云厂商开始自研**定制硬件和ASIC**（专用集成电路）
  - AWS: **Graviton**（ARM架构服务器芯片）
  - Google: **TPU**（张量处理单元）
  - 核心价值：**软硬件协同定制**，针对云工作负载优化
- **运维简化（Operations）**：本地数据库的运维难度极高，云端可大幅降低运维负担
- **安全、治理与合规**：超大规模云厂商拥有长期深耕**安全、治理与合规**的优秀团队，将专业安全能力整合到单一服务中
- **丰富的云生态系统**：客户迁移数据库工作负载到云端，本质上是在迁移**所有工作负载**，包括AI部署服务、安全服务、OLTP/OLAP数据库等

### 架构假设的根本性变革

- **核心洞察**：当数据库迁移并重新架构到云端时，会引入**不同的假设前提**，直接导致**不同的系统架构**
- **未来展望**：在未来的5-10年内，**许多教科书基础将被重写**，我们对数据库系统架构的思维方式将发生根本性改变

### 关键技术术语速查表

| 术语 | 全称/含义 | 场景 |
|------|-----------|------|
| **OLAP** | 在线分析处理（Online Analytical Processing） | 大规模数据查询与分析 |
| **OLTP** | 在线事务处理（Online Transaction Processing） | 高并发事务（如银行交易） |
| **ASIC** | 专用集成电路 | Google TPU、AWS Graviton中的定制芯片 |
| **Hyperscale** | 超大规模云基础设施 | AWS、Google Cloud、Azure |
| **On-premise** | 本地部署（客户自行管理硬件） | 传统数据库应用模式 |

## 二、BigQuery：云原生OLAP系统

> 主题：BigQuery是Google Cloud Platform提供的无服务器、高可扩展、经济高效的云数据仓库，其核心架构基于计算与存储彻底分离的原则。

### 系统概览

- **核心定义**：BigQuery是**Google Cloud Platform (GCP)** 提供的一种**无服务器、高可伸缩、高成本效益**的云端数据仓库
- **关键特性**：
  - **完全托管、无服务器、无集群**：用户无需管理底层基础设施
  - **高可用性**：7×24小时运行，**四个九（99.99%）** 的正常运行时间
  - **可预测性能**：追求稳定、可预期的查询性能
  - **内置机器学习（Integrated ML）**：在BigQuery内部直接支持ML功能
  - **易用SQL + 提示（Hints）**：支持标准SQL并提供优化提示
  - **PB级存储和查询**：存储和查询能力可大规模扩展
  - **数据加密与持久化**：满足企业级客户需求
  - **实时流数据分析**：这是BigQuery的重要差异化特性

### 核心架构原则：计算与存储分离

#### 架构分层结构

1. **访问层（用户侧）**
   - **SQL 兼容**：支持标准 SQL
   - **REST API**：通过 REST 接口访问数据
   - **Web UI + CLI**：提供图形界面和命令行工具
   - **7种语言客户端库**：易于集成

2. **计算层（查询处理）**
   - **高可用计算集群**：负责查询处理的组件
   - **无服务器 + 多租户**：完全托管，自动伸缩

3. **网络互连层**
   - **Petabit级二分带宽网络**：连接计算和存储的高速网络

4. **存储层**
   - **复制持久化存储**：数据持久化且多副本冗余

5. **内存 Shuffle 层（中间层）**
   - **分布式内存 Shuffle 层**：位于计算和存储之间
   - 用途：处理大规模查询的 **Shuffle 操作**（数据重分布）

6. **数据接入方式**
   - **流式写入（Streaming Ingest）**：实时数据摄入
   - **流式读取**：实时数据读取
   - **批量加载（Bulk Load）**：离线导入
   - **导出（Export）**：数据输出

### 底层架构：Google 基础设施的深度整合

BigQuery 并非一个独立开发的系统，而是将多种 Google 核心基础设施组件**有机组合**成的一个**统一接口**。

| 组件名称 | 功能角色 | 类比/关键特性 |
| :--- | :--- | :--- |
| **Colossus** | **分布式存储层** | 这是 Google 的第二代**全局可扩展文件系统**。负责存储**所有静态数据**和查询的**中间/最终结果**。 |
| **Borg** | **资源管理与调度** | 这是 Google 的**集群管理系统**（对应开源界 Kubernetes 的前身）。负责**预配**所有计算资源。 |
| **Dremel** | **查询执行引擎** | 这是执行引擎的**核心技术**，负责将 SQL 查询转化为高效的**分布式执行图**。 |
| **Spanner** | **全局一致性与状态存储** | BigQuery 依赖 Spanner 来存储**元数据**、**状态信息**等关键数据。 |
| **Jupiter** | **Petabit级网络互联** | 提供了**Pb级 (Petabit) 的二分带宽**，解决了计算与存储分离后数据在网络中移动的瓶颈。 |

- **总结**：`BigQuery = Dremel（执行引擎） + Colossus（存储）+ Borg（资源）`

### 无服务器设计原则

#### 资源解耦：计算、存储、内存的三维分离

- **核心理念**：将计算、存储、内存视为独立的可伸缩维度
- **实际效果**：每个维度可按需独立扩展
  - 存储扩展不依赖计算节点
  - 计算扩展不涉及数据迁移
  - 内存分配与计算节点解耦

#### 按需共享资源

- **机制**：资源池化，多租户共享同一套基础设施
- **优势**：分摊基础设施成本，实现更低的使用价格

#### 容错与可重启性

- **核心假设**：在大规模部署中，**所有组件最终都会变得不可靠且缓慢**
- **应对策略**：
  - **查询子任务设计为确定性和可重复执行**
  - **多副本分发**：同一任务可同时分发多个副本
  - **解决"拖后腿者问题"**：同一个任务有多个副本并行执行，取最先完成的结果

#### 计算抽象：Slot 机制

- **定义**：**Slot** 是 BigQuery 中的**虚拟调度单元**，代表一个抽象的计算和内存资源组合
- **作用**：
  - 对用户隐藏底层硬件细节（CPU型号、内存大小、网络带宽）
  - 提供统一的资源配额管理接口
  - 实现"所见即所得"的资源模型

### 关键对比：与传统云数据仓库的差异

| 特性 | **传统云数据仓库** (如 Azure Data Warehouse) | **BigQuery (基于 Borg 等基础设施)** |
| :--- | :--- | :--- |
| **基础原语** | **虚拟机集群 (VM Cluster)**。工作负载被绑定在一个有限的、预定义的VM集群内。 | **容器 (Container)** + **资源声明**。计算资源由 Borg 在全球范围内动态、透明地调度。 |
| **资源边界** | 受限于VM集群的边界。你需要规划、创建和管理集群的规模。 | **没有集群边界**。资源池化，可以瞬间获取任意规模的计算资源。 |
| **存储关联** | 通常计算节点附带有本地存储。 | **计算与存储彻底分离**。两者独立弹性伸缩。 |
| **用户管理负担** | 需要关注集群的健康、缩容、扩容、配置优化。 | **零运维 (Serverless)**。用户只需提交查询。 |

## 三、BigQuery 查询执行机制

> 主题：BigQuery的查询执行采用分布式有向无环图模型，通过协调器、调度器和Worker的协同工作，实现高效、可预测的查询处理。

### 查询生命周期概览

```
用户提交查询
    ↓
Coordinator（协调器）← 第一个接收查询的组件
    ↓
设置分布式查询图（Distributed Query Graph）
    ↓
调度器（基于全局状态做决策）→ 从Borg管理的池化资源中分配Worker
    ↓
计算节点启动分布式处理
    ↓
计算 ↔ 存储 通过 Petabit 网络交互
    ↓
Shuffle 层处理大规模数据重分布
    ↓
返回查询结果
```

### 三大核心阶段

1. **阶段一：数据扫描（Stage 1 - Scan）**
   - **输入**：**分布式文件系统**中的**静态数据**
   - **处理**：由第一组 **Worker** 负责**扫描**数据，并行读取
   - **输出**：初始处理后的中间数据

2. **阶段二：数据混洗（Stage 2 - Shuffle）**
   - **功能**：这是一个**可选**但在数据需要**重新分配**时**至关重要**的中间层
   - **目的**：**数据重分发**，将阶段一输出的数据根据后续处理的需求进行**重新分区和排序**
   - **作用**：保证**需要在一起计算的数据被发送到同一个 Worker**

3. **阶段三：最终处理与输出（Stage 3 - Finalize）**
   - **输入**：从 Shuffle 层接收到的重分布数据
   - **处理**：由最后一组 Worker 进行**最终计算**（如聚合、排序）
   - **输出**：将最终结果**写回**分布式文件系统

### 性能与可预测性

- **为什么可预测性比速度重要？**
  - **客户需求（特别是大企业）**：宁愿每次运行**同样慢**，也不愿出现**有些快、有些慢**的不稳定现象
  - **典型例子**：一个工作负载包含10个查询，如果1个变快、9个变慢，客户会直接选择**每次都稳定（即使慢一点）** 的方案
- **如何实现可预测性？**
  - **假设基础**：依赖**PB级二分带宽网络**，确保节点间通信几乎无瓶颈
  - **存储系统**：能够从**Colossus**以稳定、可预测的速度读取数据
  - **结果**：相同查询反复执行，性能波动**极小**

## 四、内存级 Shuffle 架构

> 主题：BigQuery采用分离式内存Shuffle架构，将Shuffle从计算阶段的一部分变为独立的、有状态的、持久化的基础设施，实现更智能的资源管理。

### 传统架构的痛点

- **二次方伸缩特性（Quadratic Scaling）**：每个 worker 需与所有其他 worker 交换数据，数据量增加时网络/内存压力呈平方级增长
- **资源碎片化（Resource Fragmentation）**：部分 worker 内存爆满，部分空闲
- **隔离性差（Poor Isolation）**：一个慢 worker 拖垮整个查询

### BigQuery 的解决方案：分离式内存 Shuffle

#### 架构关键设计

- **三层分离架构**：
  - **存储层**（Google Cloud Storage / Colossus）
  - **计算层**（Worker 节点）
  - **Shuffle 层**：独立管理的 **RAM Disk**（基于内存的磁盘），与计算节点**物理分离**

#### 核心创新点

1. **性能提升**：**降低 Shuffle 延迟一个数量级**
2. **规模延伸**：**支持更大规模的 Shuffle**
3. **成本优化**：**资源成本降低 20%**

#### 关键机制：持久化 Shuffle 层

- **支持 Checkpoint（检查点）**：每个 Shuffle 阶段的状态可以被持久化保存
- **带来的调度灵活性**：
  - **弹性调度**：BigQuery 可以在 **Checkpoint 后主动抢占 Worker**，根据查询执行情况**动态调整 Worker 数量**
  - **重新优化**：基于前一个阶段的执行统计，动态调整后续阶段的执行计划

## 五、动态调度与动态查询执行

> 主题：BigQuery通过动态调度机制和动态查询执行技术，在运行时根据实际数据特征调整执行策略，保证可预测性和高性能。

### 动态调度机制

- **动态中心骨架调度器**负责**分配槽位（Slots）** 和**工作节点数量**
- **调度与执行流程**：BigQuery可以在**检查点**后抢占当前工作节点，基于查询执行情况**重新优化**并调整下一阶段的工作节点数量
- **核心目标**：
  - **处理机器故障**：节点失败后，通过检查点状态**快速恢复**
  - **资源公平分配**：工作负载管理器根据查询实际需求动态调整资源

### 为什么要动态查询执行？

- **优化器的局限性**：引用数据库权威专家David DeWitt的观点——**“优化器在连接树中向上5-6层时，仅仅是基于一厢情愿的假设。”**
  - 对于包含**大量连接**的复杂查询，无法可靠预测**6-7层之上**的连接结果
  - 静态优化会因**预估偏差**导致性能崩溃

### 动态重分区

- **问题**：数据写入分区时可能出现**倾斜**，导致下游处理瓶颈
- **核心机制**：
  - **协调器**实时监控每个分区的数据量
  - 当检测到某分区数据量过大时，会**在飞行中动态创建新的分区**
  - 工作节点停止向拥堵的分区写入，转而写入新分区
  - 已有数据也会被**重新分区并分发**到新分区中

### 连接操作重优化

BigQuery 可以在运行时根据实际数据特征，在**广播连接**和**洗牌连接**之间切换。

- **广播连接**：当**连接的一方表很小**时，将其广播给所有工作节点，速度快、数据传输量小
- **洗牌连接**：当**两张表都很大**时，必须对双方数据进行**洗牌**和重分区，数据移动量巨大
- **动态切换机制**：
  - **启动阶段**：BigQuery 可能先假设为**哈希连接**
  - **监控阶段**：引擎持续监控连接两侧数据的实际大小
  - **触发切换**：如果**某一侧数据**实际大小小于**广播阈值**，引擎会**取消该侧的洗牌**，并**立即切换为广播连接**

## 六、Capacitor：列式存储格式

> 主题：Capacitor是BigQuery的列式存储格式，专为读取优化设计，通过向量化评估、智能谓词重排序和行重排序等技术最大化压缩效率和查询性能。

### 核心设计理念

- **列式存储**：只读取查询所需的列，避免扫描整行，大幅提升 I/O 效率
- **向量化评估**：在数据从磁盘加载的过程中，CPU 就能以批量方式执行过滤、投影等操作
- **数据跳过**：通过 **Skip Indexes**，在读取文件前快速判断哪些数据块与查询无关，直接跳过

### 智能谓词重排序

对于多个 `WHERE` 条件，Capacitor 会根据以下**启发式信息**动态重排谓词评估顺序：
- 字典使用情况
- 唯一值基数：低基数列的过滤效率高，优先执行
- 空值密度：空值多的列，相关谓词可能更快排除大量数据
- 表达式复杂度：简单的比较运算优先于复杂的函数调用

### 行重排序与压缩优化

- **核心问题**：数据原始顺序可能不利于压缩
- **解决方案**：Capacitor 会在后台**重排行**，将相同或相似值的行聚集在一起，**最大化游程编码的压缩效果**
- **实际收益**：通过重排数据布局，可以节省**显著比例的字节数**

### 持续的在线优化

- **后台压缩**：Capacitor 在数据**写入后，仍能在后台持续进行压缩和重新排序**
- **自动聚簇**：BigQuery 支持用户指定表按哪些列聚类，Capacitor 有能力**自动地、渐进地**重新聚簇数据

## 七、数据聚类与计费模型

> 主题：BigQuery通过自动再聚类优化数据布局，并通过Slot机制实现无服务器架构下的资源细化和计费。

### 自动再聚类

- **核心概念**：BigQuery 提供**自动再聚类**能力，即使新数据不断插入，系统也能在后台逐渐将数据物理排序到正确的聚类块中
- **工作原理**：
  - 用户声明“按某列/多列聚类数据”后，BigQuery 会将存储块按聚类键排序
  - 新插入的数据起初是**半未聚类**状态，但会**随时间推移被自动再聚类**
- **关键优势**：
  - **数据跳过**：查询时无需扫描无关块
  - **共置**：计算聚合函数时，同一聚类键的数据集中在连续块中
  - **星型模式优化**：在事实表与维度表关联时可利用聚类减少随机 I/O

### 计费模型

- **两种收费模式**：
  - **按需付费**：按查询消耗的槽数量计费，但有上限
  - **预留槽**：客户预购固定数量的槽，适用于稳定负载
- **槽的定义**：**抽象资源**，槽数量决定了查询能调用的并行计算资源上限——**槽越多，查询越快**

## 八、大数据量元数据管理

> 主题：BigQuery将元数据管理视为数据管理问题，通过延迟元数据读取和系统表化技术，消除元数据加载瓶颈。

### 物理元数据 vs 逻辑元数据

**逻辑元数据**：规模小，定义数据的“骨架”。
- **内容**：表结构（Schema）、行列访问策略、权限控制等
- **特点**：体积小，变化不频繁

**物理元数据**：规模巨大，是查询执行的“导航仪”。
- **内容**：文件位置、多版本并发控制（MVCC）信息、列统计信息等
- **问题**：**规模惊人**，某些表的元数据可达**数十TB**，仅加载这些元数据就可能引入**数十分钟的延迟**

### 解决方案：将元数据视为数据管理问题

- **核心理念**：**元数据管理 = 数据管理问题**
- **做法**：将物理元数据也看作数据表，使用与处理用户数据相同的分布式数据处理技术来管理
- **目的**：**消除元数据加载瓶颈**

### 延迟元数据读取

- **核心流程**：
  1. **逻辑查询规划**：查询规划器首先**仅使用逻辑元数据**生成初始查询计划
  2. **查询重写**：将原始查询重写为一个**半连接**形式的查询，将原始查询与**元数据系统表**结合
  3. **生成文件列表**：重写后的查询在执行过程中生成实际需要读取的文件位置列表
  4. **分派执行**：文件列表被分派到**并行工作节点**上

### 直接读写 API

- **存储层**：BigQuery 提供了一套独立的、高性能的存储层，可通过 **读写 API** 直接访问
- **外部引擎集成**：其他计算引擎（如 **Spark、TensorFlow、Presto、Google Dataflow**）可以通过这些 API 直接读写 BigQuery 存储层的数据
- **读取 API 特性**：
  - **并行多流读取**：支持通过**多个流**并行读取数据
  - **列投影与过滤**：只读取必要的列和数据
  - **向量化查询处理**：在API层面尽可能靠近数据实现向量化查询
  - **快照一致性**：提供**时间点一致**的快照，支持**时间旅行**
- **写入 API 特性**：
  - **恰好一次语义**：确保每条流式数据只被处理一次
  - **事务支持**：支持**流级别事务**和**跨流事务**
  - **Schema演化检测**：自动检测并适应**Schema更新**

## 九、BigQuery 查询加速技术栈

> 主题：BigQuery通过分区与聚簇、物化视图和BI引擎内存缓存三层加速技术，满足不同数据热度的查询性能需求。

### 查询加速技术栈

#### 1. 基础层：数据分区与聚簇
- **分区**：将数据按时间等维度划分
- **聚簇**：按特定列对数据进行排序组织
- **核心机制**：利用**分区跳过**和**聚簇跳过**，**只读取必要的数据分片**

#### 2. 中间层：物化视图
- **物化视图**：预计算并存储**子查询或完整查询**的结果
- **关键特性**：支持**增量更新**，当底层数据变化时，只计算和更新**有变动的部分**
- **用途**：用于**查询优化**，加速重复出现的查询模式

#### 3. 顶层：BI引擎内存缓存
- **BI引擎**：提供**内存级**数据缓存
- **核心架构**：
  - 数据缓存在内存中
  - **向量化查询引擎**紧邻内存数据部署
- **性能目标**：实现**亚秒级延迟**和**超高并发**

### 不同数据类型的加速选择

| 数据类型 | 推荐加速方式 | 特点 |
|---------|------------|------|
| 底层基础数据 | 分区+聚簇 | 减少扫描量 |
| 中层数据 | 物化视图 | 预计算、增量维护 |
| 顶层热数据 | BI引擎内存缓存 | 亚秒级响应、高并发 |

### BI引擎架构深度解析

- **有状态工作节点**：数据在内存中缓存的载体，保持状态，避免重复加载
- **向量化查询引擎**：紧邻缓存数据，利用**SIMD指令集**进行批量数据处理
- **与BigQuery存储的集成**：BI Engine 建立在现有 BigQuery 存储之上，不改变底层存储模型

### 流式摄取与实时处理

- **传统架构的痛点**：需要**独立的BI服务器**、维护**ETL管道**、管理**OLAP立方体**
- **BigQuery 的一体化方案**：
  1. **直接流式摄取**：数据直接从数据源**流式写入 BigQuery**
  2. **自动物化视图更新**：流式数据**实时触发物化视图的增量更新**
  3. **BI Engine 缓存**：热点查询结果**自动缓存到内存**
  4. **消除层级**：无需独立 BI 服务器，无需管理传统 OLAP 立方体

## 十一、Amazon Aurora：云原生OLTP系统

> 主题：Amazon Aurora通过计算与存储分离的架构设计，将日志视为数据库，从根本上解决了传统云数据库的I/O瓶颈问题。

### 传统关系型数据库的局限

- **紧耦合架构**：传统关系型数据库将 **SQL事务、缓存、日志** 等功能**紧密耦合**在同一个节点上，并依赖**本地附加存储**
- **核心假设**：这一架构基于30年前的设计假设，即计算和存储是**共置**的
- **单体架构的问题**：
  - **爆炸半径大**：一旦数据库崩溃，影响范围是整个系统
  - **高可用恢复成本高**：故障转移和恢复需要大量时间和资源
  - **不适应云环境**：在虚拟机上直接部署这种单体架构，无法充分利用云的灵活性

### 核心架构原则：计算与存储分离

- **核心概念**：必须将**计算**与**存储**解耦
- **分离的优势**：
  - **生命周期不同**：存储是**持久化**的，计算是**按需**的
  - **弹性伸缩**：计算实例可以随时**启动、关闭、扩缩容**
  - **高可用与持久性**：当计算节点失效时，存储层的数据不受影响

### I/O瓶颈：写入双倍数据的代价

- **写入流程**：数据库通常会将数据**写入两次**：
  1. **写入日志（WAL）**：首先写入**预写式日志**，以保证事务的**持久性**
  2. **写入数据文件**：当数据页被**检查点**刷新时，再次将数据写回存储
- **主备同步的I/O压力**：主库写入WAL后，还需要将WAL同步到备库，导致巨大的I/O开销

### Aurora 核心设计理念：日志即是数据库

**核心原则：** `The log is the database`

#### 计算与存储分离

- **计算层**：仅运行 MySQL 或 PostgreSQL 实例的前端逻辑
  - **唯一写入操作：将 Redo Log 通过网络发送到共享存储**
  - **彻底取消检查点：** 不再需要将整页数据写回磁盘
- **存储层**：本质是一个 **"Redo Log Machine"**
  - **高度并行化**：每个存储节点只接收和处理与自己相关的日志片段
  - **日志应用下推**：日志在存储层就地应用，直接更新页面
  - **数据冗余**：数据跨 **3 个可用区** 进行 **6 副本复制**

#### 读写副本架构

- **主节点**：接收写请求，将 Redo Log 写入共享存储，同时复制给所有只读副本
- **只读副本**：
  - **缓存感知**：副本仅将日志中与自己**缓冲池**相关的部分应用到本地缓存
  - **懒惰丢弃**：不相关的日志更新直接丢弃
  - **实现读扩展**：通过增加只读副本来分担读负载

### 存储节点核心机制

#### 存储分段与黄金尺寸

- **Segment（段）**：数据分区的基本单元，**10GB** 大小
- **Protection Group（保护组）**：由 **6 个副本**组成的复制集合
- **分段尺寸的权衡**：
  - **段太小**：故障频繁时，增加网络开销
  - **段太大**：修复卷时，传输数据耗时过长
  - **Aurora 策略**：选择“**能保证最快修复速度的最大尺寸**”
  - **修复时间估算**：在 **10 Gbps 网络** 连接下，修复一个段大约需要 **10 秒**

#### 数据冗余与高可用机制

- **目标**：容忍一个**完整可用区 + 一个额外副本**的故障
- **实现**：在 **3 个可用区**内，数据被复制 **6 份**，每个 AZ 内包含 **2 份** 数据副本
- **仲裁机制**：
  - **写仲裁**：需要 **4/6** 的副本确认写入成功
  - **读仲裁**：读取时只需要获取 **3/6** 的副本响应的最新版本

#### 存储节点的工作职责

- **核心角色**：存储节点本质是一个 **重放机器**
- **主要工作流**：
  1. **接收日志**：从主实例接收**日志记录**
  2. **持久化确认**：将日志写入**输入队列**，并确保持久化，**立即向主实例发送确认**
  3. **后台处理**：日志合并、页面重放、持久化回写、备份到 S3
  4. **节点间通信**：通过 **Gossip 协议** 在副本节点之间通信

### 灾难恢复与提交协议

#### 多级 LSN 与恢复点

- **Segment Complete LSN (SCL)**：每个存储节点维护的低水位标记，表示已接收并确认的所有日志记录
- **Protection Group Complete LSN (PG CL)**：数据库层观察到所有 4/6 个 Segment 的 SCL 都推进后，PG CL 才能推进
- **Volume Complete LSN (VCL)**：数据库层观察到所有保护组的 PG CL 都推进后，VCL 才能推进
- **层次关系**：SCL → PG CL → VCL，逐级推进

#### 事务提交协议

- **提交条件**：当事务的 Commit LSN 小于或等于 **Volume Complete LSN (VCL)** 时，数据库才能确认事务提交
- **异步确认机制**：所有事务的确认是 **异步的**，多个事务可以 **"搭便车"** 共享同一个 LSN 推进

#### 快速恢复

- **传统数据库**：需要 **Checkpoint** 检查点，将页面持久化到某个 LSN，然后截断日志
- **Aurora 的优化**：
  - **无需传统 Checkpoint**，因为日志本身就是持久化的数据
  - 恢复时只需从 VCL 开始回放，避免了扫描整个日志
  - **Gossip 协议** 和反向链接确保节点间数据一致

### 快速故障恢复

#### 恢复流程

1. 数据库层崩溃后重新启动
2. 执行恢复协议，建立**一致性点（Consistency Point, CP）**
3. **CPL之后的所有数据将被视为已删除**
4. 数据库即可**立即开放业务**

#### 设计优势

- **消除跨存储节点的两阶段提交**
- **无需执行重做或回滚日志处理**
- 数据库层只需：
  1. **附加到存储层**
  2. **建立一致性点**
  3. 数据库即可**立即开放业务**

## 十二、云原生数据库架构总结与展望

> 主题：云原生数据库通过计算与存储分离实现弹性伸缩和高可用，HTAP的未来可能从OLAP系统演变而来，而CDC是连接事务处理与分析处理的桥梁。

### 核心架构特征

- **核心概念**：云原生数据库将计算与存储分离，摒弃了传统数据库的复杂度
- **关键技术**：
  - **两阶段提交**：在跨存储节点的事务中，需要两阶段提交来保证原子性
  - **无 Redo/Undo 日志**：因为架构简化，无需处理复杂的重做或撤销日志处理
- **启动流程简化**：
  1. **附加到存储**：数据库进程直接挂载到分布式存储层
  2. **建立一致性点**：快速对齐到全局一致性状态
  3. **即开即用**：无需漫长恢复，立即服务请求
- **类似架构系统**：
  - **Azure SQL Hyperscale**：微软的云原生数据库
  - **阿里云 PolarDB**：同样基于分离架构

### HTAP 的未来

- **核心观点**：未来是否会有一个系统能**高效处理 OLTP（事务）和 OLAP（分析）**？
- **行业猜测**：
  - **可能路径**：HTAP 更可能从 **OLAP 系统演变而来**，而非改造 OLTP
  - **原因**：OLTP 系统基于固定页设计，工作负载“粘性”高，不适合直接做分析
  - **成功案例**：如 **SingleStore**（从零构建的混合架构）
- **替代方案（两系统 + CDC）**：
  - **核心思路**：保持**两个独立系统**（一个 OLTP，一个 OLAP）
  - **技术桥接**：使用 **CDC（变更数据捕获）** 将 OLTP 的变更日志实时同步到 OLAP 系统

### 性能、成本与网络开销的权衡

#### 性能与成本的博弈

- **核心模型**：云端服务商普遍采用基于**资源消耗**的计费模型
- **核心权衡**：
  - **性能提升**：分配更多资源会**加速**查询执行
  - **成本增加**：性能的提升直接导致了**更高的资源消耗**和客户的**账单**
- **解决方案**：客户可以设置一个**资源消耗上限**，系统在此预算内尽力优化

#### 网络传输的开销

| 特性 | OLAP (分析型负载) | OLTP (操作型/事务负载) |
| :--- | :--- | :--- |
| **主要瓶颈** | **吞吐量** | **延迟** |
| **数据访问模式** | 大规模、顺序扫描大量数据 | 点查询、小规模、分页读取 |
| **网络开销影响** | **相对较小**，瓶颈更可能出现在**存储端的吞吐量**上 | **影响显著**，每次跨网络读取小数据块的额外延迟成本会放大 |
