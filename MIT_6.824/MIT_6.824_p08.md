## 一、线性一致性（Linearizability）核心概念

> 主题：线性一致性的定义、判定方法及其在分布式系统中的重要性

### 定义与重要性

- **可线性化**是存储系统中**强一致性**的标准定义
- Lab 3 实现必须满足可线性化要求
- 用途：
  - 判断强一致系统行为是否可接受
  - 识别非可线性化系统与其偏离程度

### 操作历史（Operation History）

- **核心概念**：可线性化定义在特定的**操作历史**上
  - 观察客户端请求序列
  - 每个请求有**发送时间**和**响应时间**
  ［请求开始——执行——响应完成］

#### 操作历史示例图示

```
客户端 A:  │←──── Write(X,0) ────→│
                                       时间→
客户端 B:                             │←─ Read(X) ─→│
```

- **竖线标记**：请求发送/响应时间点
- 表示法：`Write(X,0)` 表示设置变量 X 值为 0（对应 Lab 3 的 Put 操作）

#### 关键观察点

- 第二个请求（读）在第一个请求（写）**完成之后才启动**
- 这种**时间顺序性**对一致性判断至关重要

### 检验可线性化的方法

1. 记录所有请求的时间区间（从发送到响应）
2. 检查是否可以**重新排列**操作顺序
3. 重新排列后必须满足：
   - **保持真实世界的先后顺序**（不重叠的操作不能调换顺序）
   - **每个读操作返回的值必须符合逻辑**（最后写入的值）

### 核心公式/规则

> **可线性化条件**：
> 对于任意操作历史，存在一种操作顺序，使得：
> 1. 每个操作都能被“压缩”到时间区间内的**某一点**
> 2. 压缩后的顺序与**真实时间顺序一致**（不重叠操作保持原有顺序）
> 3. 每个读操作都能返回**最近一次写操作**写入的值

### 线性一致性的直观定义

- **核心概念**：线性一致性要求**所有客户端**看到的操作历史必须遵循**同一个全局顺序**
- **关键特性**：系统表现得就像只有**一份数据副本**，所有操作按**单一线性序列**执行

### 关键术语总结

| 术语 | 定义 |
|------|------|
| **线性一致性** | 所有操作具有单一全局顺序的系统特性 |
| **全局总序** | 所有操作必须遵循的唯一顺序 |
| **实时序（Real-time order）** | 基于真实时钟确定的操作先后关系 |
| **线性化历史（Linearizable history）** | 存在一个合法总顺序的历史 |
| **副本延迟** | 多副本系统中，不同副本更新不同步导致的时间差 |
| **观察冲突** | 不同客户端对同一操作的观察结果不一致 |

## 二、线性一致性的判定方法

> 主题：通过构造全序和检测有向环来判定历史是否可线性化

### 判定策略一：构造合法总顺序（证明可线性化）

如果你能为所有操作找到一个**总顺序**，满足以下两个条件，则证明该历史是线性一致的：

1. **遵守实时约束**：如果操作 `Op1` 在真实时间上完全结束于 `Op2` 开始之前，那么总顺序中 `Op1` 必须在 `Op2` 之前。
2. **读操作返回最近写入值**：在总顺序中，每个读操作必须返回它**前面最近的那个写操作**所写入的值。

**公式化表述**：  
若存在一个顺序 $\prec$，使得：
- 对任意操作 $a, b$，若 $a$ 的真实结束时间 < $b$ 的真实开始时间，则 $a \prec b$。
- 对任意读操作 $r$ 读取键 $k$，若 $r$ 在 $\prec$ 中返回的值 $v$，则存在一个写操作 $w$ 写入 $k=v$，且 $w \prec r$，并且不存在另一个写操作 $w'$ 写入 $k$ 使得 $w \prec w' \prec r$。

即：**存在合法总顺序 → 历史可线性化**。

### 判定策略二：寻找有向环（证明不可线性化）

观察操作之间的**必须排序关系**，构建一个有向图：

- **边规则**：
  - 实时序边：若 $Op1$ 结束时间 < $Op2$ 开始时间，则 $Op1 \rightarrow Op2$。
  - 写-读边：若读操作看到某个写操作的值，则该写操作必须在读之前。
  - 读-写边：若读操作未看到某个后续写操作的值，则该写必须在读之后（或读必须在写之前）。

如果在图中**发现了有向环**（例如：$A \rightarrow B \rightarrow C \rightarrow A$），则证明**不存在任何合法的总顺序**。

**公式化表述**：  
若存在操作序列 $Op_1, Op_2, \dots, Op_n$ 使得：
- 对每个 $i$，$Op_i$ 必须在 $Op_{i+1}$ 之前（由规则推导），且
- $Op_n$ 必须在 $Op_1$ 之前，

则：**存在有向环 → 历史不可线性化**。

### 线性化点（Linearization Point）

- **线性化点**：每个操作（特别是写操作）的**生效瞬间**。对于读操作，其返回值必须对应于在该点之后、最新写入的值。
- **判定方法**：
  1. 为每个操作在其时间区间内**选择一个线性化点**。
  2. 检查所有线性化点构成的顺序是否是一个**全序（$<$）**。
  3. 验证该全序是否与历史中的**实际返回值一致**（即读操作必须看到在该点之前最近一次写入的值）。

### 历史性（History-by-History）验证

- **非静态属性**：线性一致性不是对系统设计的永久断言，而是**逐段执行历史（history）** 的判断。
- **观察者模型**：
  - 我们只能观察系统对外暴露的**操作序列和响应值**，不关心内部实现。
  - 在未观察到矛盾前，**假设系统是线性一致的**；随着更多操作被记录，检查是否仍能保持无环性。

## 三、线性一致性的证明实例分析

> 主题：通过具体操作历史演示如何构造全序并验证线性一致性

### 操作历史回顾

- **操作1**: `Write(X, 0)` — 在时间线上最早完成
- **操作2**: `Write(X, 2)` — 在操作1之后开始并完成
- **操作3**: `Read(X, 2)` — 与操作2并发执行
- **操作4**: `Write(X, 1)` — 在操作3之后开始
- **操作5**: `Read(X, 1)` — 在操作4之后开始

### 构造全序序列

> **全序**: `Write(X,0) → Write(X,2) → Read(X,2) → Write(X,1) → Read(X,1)`

### 实时一致性验证

对每个操作在其**时间区间内选择一个逻辑时间点**，使其符合全序顺序：

- **操作1**：选择最早的时间点，在其实际完成时间之前已放置
- **操作2**：选择在操作1之后的某个时间点
- **操作3**：选择与操作2**并发区间内**的某个点，但逻辑上排在操作2之后
- **操作4**：选择在操作3之后的某个时间点
- **操作5**：选择在操作4之后的某个时间点

> **关键观察**：这种时间点选择是**可行的**，因为所有操作都在其实际时间区间内被分配了一个逻辑时间，且该逻辑时间顺序没有违反任何实时约束。

### 读取正确性验证

**最终检查条件**：每个读取操作是否看到了**同一变量上最接近的前一个写入操作**所写入的值？

- **Read(X, 2)** → 其最接近的前一个写入是 `Write(X, 2)` → **正确匹配** ✓
- **Read(X, 1)** → 其最接近的前一个写入是 `Write(X, 1)` → **正确匹配** ✓

### 结论

该历史记录是**可线性化的**，因为：
1. ✅ 存在一个**全序操作序列**
2. ✅ 该序列**匹配实时顺序**（通过逻辑时间点选择证明）
3. ✅ 每个读取操作**读到了正确的值**（匹配最接近的前一个写入）

> **重要启示**：线性化并不总是直观明显的，有时需要**系统性地构造和验证**全序序列才能确认。

## 四、线性一致性的深入理解与常见误区

> 主题：线性一致性的时间模型、反例分析及实践启示

### 关键示例分析：读操作与写操作的时序关系

#### 场景一：可线性化的历史

- **写操作顺序**：
  - 写0（X=0）
  - 两个并发的写操作：写1和写2
- **读操作结果**：
  - 读X得到2
  - 读X得到1
- **为什么可线性化？**
  - 虽然读操作看到的值不是按严格的时间顺序（先看到2，后看到1），但**只要存在一种合法的全局顺序**即可
  - 合法顺序可以是：写0 → 写2 → 读2 → 写1 → 读1
  - 注意：**第一次写入的值不一定是1**，初始值0也可以是合理的

#### 场景二：不可线性化的反例

**修改条件**：如果读X得到2的操作**完成后**，客户端才发起写2的操作

- **违反实时顺序**：
  - 读操作（得到2）必须在写操作（写2）之前完成
  - 任何线性化顺序都必须保持这个实时顺序
  - 读发生时，只有写0和写1可能存在
  - 因此读操作**只能看到0或1**，不可能看到2

### 违反线性一致性的案例

#### 场景设置
- 两个客户端对同一个变量 `X` 进行操作
- 共有两个写操作：写 `X = 2` 和写 `X = 1`
- 两个客户端分别进行两次读取

#### 矛盾的产生

**客户端1的观察**：
- 第一次读到 `X = 2`
- 第二次读到 `X = 1`
- **隐含顺序**：写2 → 写1

**客户端2的观察**：
- 第一次读到 `X = 1`
- 第二次读到 `X = 2`
- **隐含顺序**：写1 → 写2

#### 核心问题
- **线性一致性规则**：必须存在**单一全局操作顺序**，所有客户端看到的操作必须与这个顺序一致
- **矛盾点**：客户端1和客户端2的观察结果**相互矛盾**，无法推导出统一的顺序
- **结论**：这种情况**不满足线性一致性**

### 时间模型的精确定义

#### 客户端视角的时间模型

- **请求发起时刻**：客户端发送请求的瞬间（图中第一条竖线）
- **响应接收时刻**：客户端收到响应的瞬间（图中第二条竖线）
- **内部实现不可见**：副本网络、复制协议等中间过程对客户端是黑盒

#### 灰色地带：需要考虑的情况

- **重传请求**：如果客户端需要重新发送请求，会影响时间线的判断
- **网络延迟**：请求和响应之间的时间间隙由网络和系统决定

### 重要公式与原则

**实时顺序约束**：
$$
\text{如果 } finish(A) < start(B) \text{，则 } A \rightarrow_{linearizable} B
$$

**可线性化的判定准则**：
- 找到一种合法的**全局操作顺序**
- 该顺序必须**尊重所有操作的真实时间顺序**
- 每个读操作必须返回**该顺序中最后一次写操作的值**

### 实践启示

1. **不要被表面顺序迷惑**：并发操作可以有任意的线性化顺序
2. **初始值总是候选**：初始值（如0）始终是合法的首次写入值
3. **实时顺序不可违背**：这是最硬性的约束条件
4. **客户端视角优先**：关注客户端看到的行为，而非内部实现细节

## 五、线性一致性与强一致性的关系

> 主题：强一致性等价于线性一致性，以及脏读被禁止的原因

### 核心概念：实时顺序与新鲜数据

- **线性一致性**：系统必须保证**所有操作**（读/写）在全局实时顺序（Real-time Order）下是合法的。即：
  - 如果操作A在操作B开始之前完成，那么A的效果必须在B之前可见。
  - **不允许任何客户端读到过期数据（Stale Data）**。
- **关键术语**：**实时顺序**、**操作间隔**、**合法排序**、**新鲜数据（Fresh Data）**。

### 关键公式与规则

- **实时顺序约束**：设操作$O_1$完成的时间点为$t_1$，操作$O_2$开始的时间点为$t_2$。如果$t_1 < t_2$，则所有客户端必须看到$O_1$的完成先于$O_2$的开始。
- **间隔重叠时的自由**：如果两个操作的**时间间隔（Interval）发生重叠**，系统可以**任意选择执行顺序**（即可以交换顺序），只要读操作返回的是"在该时刻合法的值"。

### 实例分析：为什么"脏读"被禁止？

1. **场景**：客户端A写$X=1$后立即完成；客户端B读$X$，但回复延迟或副本滞后。
   - **非线性一致行为**：读请求返回了旧值（如$X=0$）。
   - **原因**：写操作已完成（实时顺序上先于读请求），但读操作未看到最新提交的值。
   - **结论**：**线性一致性系统不允许任何形式的"陈旧读"**，即使读请求的发起时间在写完成之后，也必须返回最新提交的值。
2. **顺序依赖消除**：若两个写操作重叠，读操作返回的结果可能出现在任意顺序下，但一旦一个写操作完成，后续所有读必须看到该写的结果。

### 强一致性与线性一致性的关系

- **强一致性**：通常被理解为**单副本行为**，即系统表现得就像只有一份数据副本一样。
- **等价性**：在大多数场景下（包括本文的语境），**强一致性等价于线性一致性**。两者都要求：所有操作按全局实时顺序排列，且读操作必须看到最新的写结果。

### 系统实现中的挑战

#### 根本原因
- 构建的是**复制系统**（如 Raft 复制或多缓存系统）
- 多个服务器保存了 `X` 的**多个副本**
- 不同副本可能在**不同时间点**持有**不同的值**

#### 线性一致性的要求
- 尽管存在多个副本和延迟，系统必须**表现得就像只有一份数据**
- 所有副本的最终行为必须**符合单一线性序列**
- **公式化表示**：设系统状态为 $S$，操作序列为 $O = [o_1, o_2, ..., o_n]$，则任何客户端观察到的状态变化必须满足：
  \[
  \forall \text{client } c: S_c(t) = S_{global}(t)
  \]
  其中 $S_c(t)$ 是客户端 $c$ 在时间 $t$ 观察到的系统状态，$S_{global}(t)$ 是全局线性序列在该时间点的状态

## 六、客户端请求重试与去重机制

> 主题：客户端重试对线性一致性的影响及服务器端去重机制

### 核心挑战：客户端请求超时与状态不确定性

- **客户端发起读取请求后未收到响应**，原因可能多种多样：
  - **领导者崩溃**：处理请求过程中宕机
  - **网络丢包**：请求未到达领导者，或响应在返回途中丢失
  - **领导者部分处理**：收到请求但处理半途崩溃，未完成执行也未发送响应

- **关键问题**：从**客户端视角**，它只知道发出去一个请求，没有收到回复，**无法区分具体失败原因**

### 客户端重试策略

- 大多数系统实现中，客户端会：
  - **重新发送请求**（可能发给同一领导者，也可能发给不同领导者）
  - **设置超时机制**：第一个请求无响应 → 超时 → 发送第二个请求 → 最终获得响应

### 服务器端：重复请求过滤机制

- **核心方案**：服务器维护一张**基于客户端唯一请求编号索引的表格**
  - **关键术语**：**请求去重（Duplicate Request Filtering）**
  - 服务器**记录已处理过的请求及其原始响应**
  - 当收到重复请求时，**直接返回之前存储的响应**，而不是重新执行

- **为什么必须去重**：
  - **写请求绝对不能执行两次**，否则会导致状态不一致（例如两次相同的写入操作）
  - 需要**保证幂等性（Idempotency）**

### 重传请求的时间范围界定

- **关键概念：客户端重传（Client Retransmissions）**
  - 当客户端发送请求后未及时收到回复，可能会自动重发请求
  - 从客户端应用层的视角看，请求的实际时间跨度是从**首次发送**到**最终收到响应**的整个时间段
  - **重要结论：** 即使有多次重传，整个请求的时间范围应被视为从第一次发送到最后一次收到响应的完整区间

### 具体场景分析：读请求与写请求的并发

假设场景：
- 写请求在时间点 $T_2$ 完成（写入值 4）
- 读请求在时间点 $T_1$ 发送，但由于网络问题，经过多次重传后，在 $T_3$ 才收到响应
- 读请求的时间范围是 $[T_1, T_3]$，完全覆盖了写请求的完成时间 $T_2$

- **线性一致性的正确理解：**
  - 如果读请求的时间范围完全包含了写请求，那么读请求返回**值 3** 或 **值 4** 都是合法的
  - **核心原则：** 线性一致性要求的是**请求的时间区间**与**操作的实时顺序**之间的关系，而不是网络传输的实时时间点
  - 从应用层角度看，请求的开始时间是**第一次发送**，结束时间是**最后一次收到响应**

### 关键结论：应用必须容忍这些规则

- **使用线性一致性系统的硬性要求：**
  - 应用开发者必须编写**容忍这些规则**的代码
  - **绝对禁止**的假设：不能假设"收到响应时，系统在该时刻的值就是响应中的值"
  - 例如：不能这样写代码——"我收到响应值为 3，所以在我收到响应时系统的值就是 3"
  - 因为响应可能是在请求发送时（或者更早的时刻）的值，经过重传后才到达

> **线性一致性的时间范围定义：**
> $$
> \text{请求的时间范围} = [\text{首次发送时间}, \text{最终收到响应时间}]
> $$
> 在这个范围内的任何写操作，都可能与读请求并发，从而影响返回值的合法性

> **应用层的核心约束：**
> $$
> \text{不允许假设：收到响应时的系统状态} = \text{响应中包含的值}
> $$
> 必须接受：响应值可能是请求发送时（而非收到响应时）的系统状态快照

## 七、ZooKeeper：通用协调服务的设计与 API

> 主题：ZooKeeper 作为通用协调服务的核心设计理念、复制架构与性能权衡

### 为什么学习 ZooKeeper？

- **成功的真实世界系统**：ZooKeeper 是一个开源、被广泛集成到实际软件中的服务，其设计的合理性和实用性已经过大规模验证。
- **核心技术点**：
  1. **超越库的通用服务**：与 Raft 这类作为**库**存在的复制协议不同，Raft 需要你的应用显式地调用它，而 ZooKeeper 是一个**独立的、通用的服务**。它提出的核心问题是：**能否构建一个通用协调服务，封装掉分布式系统构建中的大部分痛点，让任何人开箱即用？**
  2. **复制系统的 API 设计**：ZooKeeper 本身就是一个通过复制实现容错的系统。

### 核心概念：通用协调服务

- **目标**：提供一个 **通用、独立、容错的协调服务**，让上层应用可以专注于业务逻辑，而不是重复实现分布式一致性、选主、配置管理等复杂难题。
- **如何实现容错**：和大多数系统一样，ZooKeeper 通过**复制**实现容错。它由多个服务器组成（例如 3、5 或 7 台），即使部分服务器失效，服务仍然可用。

### 复制带来的性能瓶颈

- **ZooKeeper 本质**：一个具备**容错性**的通用协同服务
  - 容错性通过**复制**实现：通常部署 3、5 或 7 台服务器
  - **关键矛盾**：购买 7 台服务器的成本是 1 台的 7 倍，但能否获得 **7 倍的性能提升**？

- **性能问题的本质追问**：当服务器数量变为原来的 *n* 倍，系统性能能否也变为原来的 *n* 倍？

### ZooKeeper 的复制机制与 Raft 的关系

- **ZooKeeper 底层协议**：**Zab (ZooKeeper Atomic Broadcast)**
  - 从复制性能角度看，**Zab 与 Raft 几乎一致**
  - 分析重点放在**复制系统的性能**上，而非 ZooKeeper 的具体功能

- **系统架构模型**：
  - **客户端**：可能有数百个
  - **领导者 (Leader)**：接收所有客户端请求
  - **ZooKeeper 层**：客户端与之通信
  - **Zab 层**：位于 ZooKeeper 下层，管理**日志复制**
  - **日志内容**：客户端提交的操作序列（与 Raft 高度相似）

- **请求处理流程**：
  1. 客户端发送请求
  2. Zab 层将请求副本发送给**所有副本节点**
  3. 副本节点将请求追加到**内存日志**中
  4. （通常也会**持久化到磁盘**，以防崩溃后恢复）

### 关键结论：更多服务器 ≠ 更高性能

- **领导者的瓶颈效应**：
  - **领导者是无可争议的性能瓶颈**
  - 所有请求都必须经过领导者处理
  - 增加副本服务器数量，**并不会减轻领导者的负载**

- **数学化表述**：
  - 设服务器数量为 *N*
  - 总计算能力（CPU 周期）与 *N* 成正比
  - 但**有效吞吐量** ≈ **单个领导者**的处理能力
  - **性能缩放比例**：**O(1)**，而非 O(*N*)

- **核心认知**：
  > **增加副本数，获得的是容错性，而非计算性能。** 性能受限于单点瓶颈——领导者。

### 单一领导者复制的性能瓶颈与读扩展

- **反直觉的现实**：在标准的领导-跟随者复制架构中，**增加服务器数量反而会降低系统整体性能**。
- **根本原因**：领导者是**单点瓶颈**。
  - 每一个写请求都必须经过领导者处理。
  - 领导者必须将每个请求的副本发送给**所有**跟随者。
  - **随着服务器数量（N）增加，领导者的工作量线性增长**：它需要发送 N-1 份副本。
- **性能公式**：系统性能 ≈ **与服务器数量成反比**。
  - 公式化表达：总吞吐量 ≈ `(领导者处理能力) / (1 + 发送给N-1个跟随者的开销)`

### 关键术语：读重工作负载（Read-Heavy Workload）

- **定义**：在现实世界中（如网页服务），绝大多数请求是**读取（Read）** 操作，写入（Write）操作相对较少。
- **简单直觉**：既然写必须经过领导者，那么**能否让跟随者来处理读请求？** 这就是"读扩展（Read Scaling）"的核心思想。
- **潜在优势**：通过让跟随者分担读请求，可以**利用大量空闲的服务器资源**，从而显著提升系统的整体读吞吐量，同时不增加领导者的写负载。

## 八、读优化：从领导者到副本的读取请求分发

> 主题：通过将读请求分发到副本实现读取性能的线性扩展，以及由此带来的数据一致性问题

### 核心挑战：Raft 系统的读写不对称性

- **写入请求** 必须全部经过领导者（Leader）
- **实际负载特性**：绝大多数系统是 **读密集型（read-heavy）**
  - 例如网页浏览：主要是读取数据来渲染页面
  - 写入操作相对较少
- **核心洞察**：可以利用读多写少的特性来优化系统

### 基本优化策略：分离读写路径

#### 写入路径不变
- 所有写入请求（如 Lab 3 中的 Put）仍发送给领导者
- 这一点对写入性能没有直接帮助，但减轻了领导者负担

#### 读取路径优化
- **读请求**（如 Lab 3 中的 Get）可直接发送给 **任意一个副本（Replica）**
- 工作机制：
  - 随机选择一个副本
  - 副本拥有日志副本和键值表
  - 副本可以直接回复客户端的读取请求
- **核心优势**：
  - 读取负载被分散到所有服务器
  - 服务器越多 → 支持的客户端越多
  - **线性可扩展性**：读取性能随服务器数量增长

### 关键问题：数据一致性（Staleness）

#### 为什么副本可能不是最新的？

**根本原因**：Raft 的多数派机制

- 领导者只等待 **多数派（Majority）** 的 AppendEntries 响应
- 一旦多数派确认，领导者就提交操作并继续下一个
- **非多数派副本可能缺失某些日志条目**

#### 具体场景分析

假设：
- 领导者 + 多数派服务器已看到前 3 个请求
- 某个副本只看到前 2 个请求（缺少请求 B）

```
场景：读取键 X
请求序列：操作1 → 操作2(A) → 操作3(B)
多数派：已执行操作1,2,3
非多数副本：只执行了操作1,2

读取结果：
  - 多数派副本 → 返回正确值（包含B）
  - 非多数副本 → 返回**过时值**（缺少B）
```

#### 过时读取（Stale Read）的定义
- 副本虽然拥有所有必需的功能组件（日志、键值表）
- 但由于未接收或执行最新的写入操作
- 返回的结果不是系统当前的最新状态

### 总结：读取优化的权衡

| 优势 | 风险 |
|-----|------|
| 显著提升读取吞吐量 | 可能返回过时数据 |
| 线性可扩展读取能力 | 不保证线性一致性 |
| 减轻领导者负载 | 需要应用层处理一致性需求 |

**关键术语**：
- **读密集型负载（Read-heavy Workload）**
- **过时读取（Stale Read）**
- **多数派（Majority）** vs **非多数派副本**
- **读写路径分离（Read/Write Path Separation）**

## 九、ZooKeeper 的一致性保证

> 主题：ZooKeeper 放弃线性一致性以换取高性能，但提供写操作线性化和 FIFO 客户端顺序保证

### 核心问题：为什么副本不能直接服务读请求？

**线性一致性系统**要求：**任何读操作必须返回最近一次写入的最新值**。

#### 三个关键障碍

- **日志同步延迟**：Leader 发送日志条目给 Follower，但 Follower 尚未收到或应用该日志
  - ZooKeeper 与 Raft 机制相同：Leader 先发送日志，获得多数派确认后，再发送提交通知
  - Follower 可能**只看到了日志，但没看到提交命令**
- **网络分区风险**：从客户端视角看，某个副本可能已与 Leader **完全断开连接**（Partitioned）
  - Follower 自身**无法感知自己是否刚被切断**，因为它还在运行

> **关键洞察**：如果没有额外的聪明机制，**线性一致性系统不允许副本直接服务读请求**——这会返回**脏读（Stale Read）**。

### Lab 3 的注意点

- **Lab 3 要求线性一致性**，所以**不可以**让副本直接服务读请求
- 性能优化的"把读请求发送给副本"策略在线性一致性下**被禁止**

### ZooKeeper 的解法：放弃线性一致性

#### 核心策略

- **ZooKeeper 不是线性一致的系统**
- 从 Table 2 可以看到：ZooKeeper 的**读性能随服务器数量增加而大幅提升**
- 这意味着 ZooKeeper **允许副本服务读请求**

#### 如何保证安全？

- **答案很简单：不提供线性一致性的读**
- ZooKeeper 的**一致性规则**（不是线性一致性）允许读操作返回**过时数据**
- 系统不再被线性一致性的规则束缚，因此**不承担必须返回最新数据的义务**

#### 权衡总结

| 维度 | 线性一致性系统 | ZooKeeper 方式 |
|------|---------------|----------------|
| 读取位置 | Only Leader | 所有副本 |
| 性能扩展 | 受限 | **随节点数线性提升** |
| 数据新鲜度 | **保证最新** | 可能返回过时数据 |
| 适用场景 | 强一致性需求 | **读多写少、可容忍短暂过期** |

### ZooKeeper 的一致性保证

尽管放弃了读取的线性化，ZooKeeper 仍然提供了一组**一致性保证**，帮助基于其构建的应用进行推理。

#### 保证一：写操作的线性化

- **核心术语**：**写操作的线性化** (Linearizable Writes)
- **含义**：
  - 即使多个客户端并发提交写操作，系统**表现出的行为**如同所有写操作是**按某种顺序一个一个执行**的。
  - 并且，系统**遵守写操作的实时顺序**：如果一个写操作在另一个写操作开始之前就已经完成，那么系统一定会让第一个写操作先于第二个写操作生效。
- **关键点**：**只有写操作**（以及任何**修改状态的操作**，如"带读的写"或"条件更新"）是线性化的。**纯读取操作不在此列**。

#### 保证二：客户端操作的 FIFO 顺序

- **核心术语**：**客户端操作的 FIFO 顺序** (FIFO Order of Client Operations)
- **含义**：单个客户端发起的操作，会按照其发起顺序被系统执行。即，如果客户端先执行了写，再执行读，那么读至少能看到该写操作的结果（或者更后面的结果），但不会看到该写操作之前的状态。

### 总结表格：ZooKeeper 一致性保证

| 保证类型 | 适用范围 | 核心行为 | 关键影响 |
| :--- | :--- | :--- | :--- |
| **写操作线性化** | 所有修改状态的操作 | 所有写操作在全局有一个严格的全序 | 确保写操作不会并发地产生歧义，是系统一致性的基础 |
| **FIFO 客户端顺序** | 单个客户端的操作 | 写请求通过客户端序列号保证顺序；读请求必须不后退 | 允许异步写操作；保证读操作不会看到不一致的时间线 |

## 十、FIFO 客户端顺序保证与 ZXID 机制

> 主题：通过 ZXID 追踪实现 FIFO 客户端顺序保证，以及读操作的单调性

### 核心概念：读取操作的顺序约束

- **FIFO 顺序保证**的核心是：**同一个客户端发出的多个读取操作，必须按照它们发出的顺序，看到**至少一样新**（或更新）的系统状态**。
- 如果客户端先执行了 `Read A`，然后又执行了 `Read B`，那么 `Read B` 看到的数据状态必须**不早于** `Read A` 看到的状态。
  - `Read A` 在日志中的执行点记为 `Point X`。
  - `Read B` 必须在 `Point X` **或之后** 的日志点执行。

### 实现机制：ZXID 追踪

- **关键术语：** **ZXID (ZooKeeper Transaction ID)**。这是领导者（Leader）为每个日志条目分配的唯一、单调递增的标识符（本质上是一个条目编号）。
- **工作流程：**
  1. **响应携带 ZXID：** 当副本（Replica）响应客户端的读取请求时，它会在响应中附带**紧接在该读取执行点之前**的日志条目的 `ZXID`。这个 `ZXID` 代表了客户端"最近看到"的数据状态。
  2. **客户端记忆 ZXID：** 客户端会记住它收到的**最高 ZXID**（即 `highest ZXID seen`）。
  3. **请求携带 ZXID：** 当客户端后续向**任何一个副本**（无论是同一个还是因为故障切换到的另一个）发送新的读取请求时，它会在请求中附带上这个 `highest ZXID`。
  4. **副本约束与延迟：** 接收请求的副本收到 `ZXID` 后，必须确保自己**已经应用了所有直到该 ZXID 的日志条目**，才能响应这个读取请求。
     - **如果副本的日志落后于客户端的 ZXID：** 它必须等待，直到它追上这个进度，然后才能安全地响应。

### 故障切换下的保证

- 即使客户端因原副本故障而切换到新副本，**FIFO 顺序保证依然有效**。
- 客户端切换后发送的新读取请求，其附带的 `highest ZXID` 强制要求新副本必须提供**至少与旧副本上次读取时一样新**的数据。

> **总结公式/算法要点：**
> 1. **客户端状态：** `local_highest_zxid`。
> 2. **读取请求：** `Read(key, local_highest_zxid)`。
> 3. **副本响应条件：** `Replica_Log_Length >= local_highest_zxid`。
>    * 若满足，则执行读取并返回结果和新 ZXID。
>    * 若不满足，则**阻塞**或**拒绝**。

### 读操作的 FIFO 客户端顺序

- **FIFO Client Order（先进先出客户端顺序）** 适用于**单个客户端的所有请求**（包括读和写）
- 机制：
  1. 客户端先向 **Leader** 发送一个写请求
  2. 写请求尚未被 Leader 处理/提交时，客户端又向某个 **副本（Replica）** 发送读请求
  3. 副本必须**阻塞（stall）该读请求**，直到客户端**之前的写操作**已被该副本执行
  4. 读操作才能继续进行
- **目的**：保证同一个客户端的操作按**发送顺序被系统观察到**

### 关键场景：写后读（Read-Your-Writes）

- 这是 FIFO 客户端顺序最直观的体现：
  - 客户端向 Leader 写入一个值（例如 `value = 17`）
  - 然后立即向某个副本读取同一个数据
  - **必须读到 `17`**，否则系统就违反了客户端请求的执行顺序
- 如果读不到刚写入的值，说明：
  - 副本可能尚未应用该写入
  - 系统需要**自动阻塞读请求**，直到写操作完成

### 重要约束：读操作不保证看到最新数据

- **ZooKeeper 不保证副本读取到的数据是全局最新的**
- 原因：
  - Leader 可能将数据**提交并发送给多数副本**，但当前读取的副本**不在这个多数集合中**
  - 该副本尚未收到最新的写入
- 结论：**副本读可能返回过时数据**，这是 ZooKeeper 的设计妥协，以换取更高的读取性能

### 算法/协议细节：读请求的阻塞与恢复

**读请求处理逻辑（副本视角）：**
```
函数 handleReadRequest(client, dataKey):
    lastWriteZXID = client.getLastWriteZXID()
    
    while (not self.hasApplied(lastWriteZXID)):
        wait()  // 阻塞，等待与Leader同步完成
    
    localValue = self.localStore.get(dataKey)
    return localValue
```

**写后读一致性保证的条件：**
$$
\text{clientRead}(key) = 
\begin{cases}
\text{stall\_until}(\text{writeCommitted}(key, \text{ZXID}_{write})), & \text{if } \text{ZX
