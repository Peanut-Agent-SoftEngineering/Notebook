# CS149_p17

## 一、事务性内存：核心概念与设计空间

> 主题：事务性内存的基本概念、设计空间与实现策略综述

### 核心回顾：事务性内存设计空间

- **数据版本控制策略**：如何跟踪旧数据和新数据
  - **急切 (Eager)**：原地更新，需要 undo log
  - **懒惰 (Lazy)**：写缓冲，提交时一次性写入

- **冲突检测策略**：何时检测事务间的冲突
  - **悲观 (Pessimistic)**：在每次内存访问时检测冲突
  - **乐观 (Optimistic)**：假设无冲突，在提交时检测冲突

- **关键发现**：
  - 悲观策略可能导致**死锁**，系统需具备死锁检测与恢复能力
  - 乐观策略中，**提交的事务拥有最高优先级**，冲突时未提交事务被终止

### 实现策略组合与可行性分析

- **理论上的四种组合**：
  - 懒惰 + 乐观
  - 懒惰 + 悲观
  - 急切 + 乐观
  - 急切 + 悲观

- **急切+乐观实现的困境**：
  - 当冲突被检测到时，事务**可能已对内存进行了修改**
  - 恢复过程复杂，因为修改已部分生效
  - 因此这种组合在实践中**并不成功**

### 关键公式与抽象模型

- **性能开销模型**：
  \( \text{Total Overhead} = \text{STM Overhead} - \text{HTM Acceleration} \)

- **冲突检测公式**（乐观策略提交时）：
  \( \text{Transaction Commits} \iff \forall r \in R_{tx}, \text{version}(r) \text{ is unchanged since read} \)
  - 其中 \( R_{tx} \) 为事务的读集
  - 如果读集中任一数据版本被修改，则事务必须终止

## 二、软件事务内存 (STM) 的实现机制

> 主题：STM 的软件实现方式、核心数据结构与算法

### 核心实现思路：通过软件屏障（Software Barriers）插入事务管理

- **核心概念**：将程序中的**普通读写操作**（`load`/`store`）替换为对事务管理系统的**显式调用**，即**软件屏障**。这些屏障本质上是对事务管理器的函数调用，用于告知系统需要进行哪些记账操作（记录读集、写集等）。
- **实施方式**：程序员编写的代码必须被**重写**，在每个读取和写入操作的位置，插入对应的事务管理函数调用（例如 `transaction_read`、`transaction_write`）。
- **关键区别**：这里的 **“屏障”** 是**软件事务内存的术语**，**不要与同步屏障（memory barrier/fence）混淆**。
- **代码路径分化**：
  - 如果同一段代码可能**既在事务内执行，又在事务外执行**，则需要维护**两个函数副本**：
    1. **无插桩版本**：用于事务外执行。
    2. **有插桩版本**：插入了软件屏障，用于事务内执行。
  - **高级优化**：在支持**即时编译（JIT）** 的运行时中（如 Java、C#），可以自动执行**函数克隆**，即自动为每个函数生成两个版本（事务内/事务外），并自动插入软件屏障。

### 核心数据结构：事务描述符与事务记录

#### 1. 事务描述符（Transaction Descriptor）

- **定位**：**每个线程**（或每个事务）的私有数据结构，包含该线程当前事务的**全部状态**。
- **核心成员**：
  - **回滚日志（Rollback Log）**：记录事务执行前的原始数据，用于事务**中断时恢复**。
  - **写缓冲区（Write Buffer）**：暂存事务写入的新值，直到事务**提交时才对外可见**。体现了**提交时写（commit-time write）** 或**缓冲更新**策略。
  - **冲突检测信息**：用于判断其他事务是否修改了本事务读取或写入的数据。
  - **读集（Read Set）**：事务执行过程中**读取过的所有数据位置**的列表。
  - **写集（Write Set）**：事务执行过程中**写入过的所有数据位置**的列表。
- **作用域**：该结构可能是**线程局部**的（事务嵌套或事务重用时），也可能是**事务局部**的。

#### 2. 事务记录（Transaction Record / Meta-data）

- **定位**：与**每个数据元素**（或对象、或对象内的更细粒度字段）**关联**的元数据。
- **内容**：记录**该数据元素当前的访问状态**，例如：
  - 数据是否被其他事务锁定？
  - 当前数据值的版本号是多少？
  - 数据是否处于某种中间状态（如被某个未提交事务写入）？
- **粒度**：
  - **粗粒度**：基于对象（Object）。
  - **细粒度**：基于对象内的某个字段（Field）或字（Word）。

### 关键类比与框架

| 概念领域 | 缓存一致性协议（硬件） | 软件事务内存（软件） |
| :--- | :--- | :--- |
| **状态跟踪** | 缓存块的 **状态位**（MESI等） | 数据对象的 **事务记录** |
| **操作通知** | 自动的**总线嗅探** | 显式的**软件屏障**（函数调用） |
| **回滚与恢复** | 硬件无法回滚（需依赖外部机制） | 基于 **回滚日志** 完成 |
| **写可见性** | 写直达/写回 | 基于 **写缓冲区** 的缓冲式提交 |

### 数据粒度与事务处理：缓存一致性的类比与权衡

#### 核心概念：事务粒度与假共享

- **核心思想**：事务处理中数据映射的**粒度**（Granularity）直接类比于**缓存一致性**（Cache Coherence）中缓存行的粒度。你需要决定是**按对象**还是**按单个数据元素**来管理事务记录。
- **关键术语**：
  - **事务粒度**：事务操作的最小数据单元（对象 vs. 字段）。
  - **假共享**：不同事务操作**不同数据**但**属于同一对象**时，产生的非必要冲突。
  - **负担（Overhead）**：管理事务记录所需的额外存储和计算资源。

#### 两种粒度方案及其对比

| 粒度类型 | 核心做法 | 优点 | 缺点 |
| :--- | :--- | :--- | :--- |
| **对象粒度** | 每个对象对应一个事务记录。通过**对象ID**或**全局哈希表**映射。 | - **管理负担低**：只需跟踪一个对象。 <br> - **复用优化**：同一对象的多次操作可**摊销**（Amortize）记录开销。 | - **假共享严重**：不同事务若操作同一对象的不同字段（如X与Y），会被误判为冲突，导致**不必要的序列化**。 |
| **字段/元素粒度** | 按单个数据元素（如32/64位字段、结构体成员）管理事务记录。通常基于**地址**映射（C/C++）。 | - **假共享少**：可真正独立并发操作不同字段，**提高并行性**。 | - **管理负担高**：需要跟踪大量细粒度元素。 <br> - **理解困难**：代码层（字段）与硬件层（缓存行）不对应，抽象层次低。 |

#### 典型示例：假共享的后果

- **假设**：
  - 对象A包含字段`x`、`y`、`z`（例如三维坐标）。
  - **事务1**：只读/写 `x` 和 `y`。
  - **事务2**：只读/写 `z`。
- **对象粒度下**：两个事务因访问同一对象而被判定为**冲突**，即使它们操作的是不同字段。这导致了**假共享**。
- **字段粒度下**：两者无冲突，可安全并发执行，**并行性提升**。

#### 权衡总结与建议

- **核心权衡**：**管理负担** vs. **并行性收益**。
- **工程实践**：
  - 通常从**对象粒度**开始，因为其**实现简单、负担低**，且可利用对象的局部性进行优化（如重复操作同一对象时摊销开销）。
  - 当对象**假共享成为性能瓶颈**时，可**选择性**地将关键对象降级为**字段粒度**（即“混合粒度”策略）。这种折中往往能获得“相当不错”的性能收益。
  - **硬件对齐**：虽然字段粒度在软件中可自由定义，但最终硬件操作仍以**缓存行**（Cache Line）为最小单位。因此，即使采用字段粒度，也应考虑**对齐和填充**以避免跨缓存行的假共享。

## 三、STM 算法详解：McRT 与并发控制

> 主题：McRT 算法的核心机制、乐观读取与悲观写入策略

### 核心数据结构：事务与记录的分离

- **全局时间戳（Global Timestamp）**：
  - 概念：一个不断递增的计数器。
  - 作用：**每笔事务成功提交时，该计数器+1**。它作为全局的“版本时钟”，用于判断数据冲突。
  - 类比：可以想象成全局的“版本号”，每次成功提交都会产生一个新版本。

- **本地时间戳（Local Timestamp）**：
  - 概念：一笔事务**开始执行时**，记录下当前的全局时间戳值。
  - 作用：作为事务的“出生时间”，用于后续验证读取的数据是否被其他事务更新过。

- **事务记录（Transaction Record）**：
  - 结构：一个 **32位值**。
  - 编码规则：
    - **最低有效位（LSB）为 0**：表示该数据**被锁定**，当前正被某笔事务写入（独占模式）。
    - **最低有效位（LSB）为 1**：表示该数据是一个**版本号**（即全局时间戳的副本），处于共享模式。
  - 核心思想：通过检查最低位，快速判断数据是“正在被独占修改”还是“处于共享可读状态”。就像是一个 **“锁/版本”二合一标识**。当为0时，相当于独占锁（正在写入）；当为1时，相当于共享版本号（可安全读取）。

### 算法核心：乐观读取 + 悲观写入

该算法名为 **McRT**（Intel 开发），采用 **乐观并发控制（Optimistic Concurrency Control）** 读取和 **悲观并发控制（Pessimistic Concurrency Control）** 写入。

- **执行读取操作 (`stm_read`)**：
  1. **直接读取**：直接从内存中读取目标数据。
  2. **验证读取**：在返回数据前，**检查该数据的事务记录**。
     - **如果事务记录最低位为 0（被锁定）**：说明有另一笔事务正在写入该数据，**不能保证读取到的数据是正确的**，通常需要回滚或重试。
     - **如果事务记录最低位为 1（版本号）**：说明数据未被锁定。接着，**验证该版本号是否小于或等于当前事务的本地时间戳**。
       - 如果**版本号 <= 本地时间戳**：数据在事务开始后未被更新过，**读取是有效的**。
       - 如果**版本号 > 本地时间戳**：数据在事务开始后被其他事务提交更新过了，**数据可能不一致，读取无效**。
  - **特点**：读取是**乐观**的，因为它假定数据没有被修改，并直接读取；但随后必须通过版本来**验证**这一假设。这是一种 **“先读取，后验证”** 的策略。

- **执行写入操作**：
  1. **获取独占权**：写入前，必须**获取目标数据上的锁**。
  2. **修改数据**：在锁的保护下直接修改内存。
  3. **写事务记录**：将目标数据的事务记录的最低有效位置为 0，表明该数据正被当前事务独占。
  - **特点**：写入是**悲观**的，因为它在执行前就通过锁**阻塞**了其他事务对同一数据的访问，防止并发写入冲突。

### 算法总结与类比

- **McRT 算法** 通过 **全局版本号** 和 **本地版本号** 来实现乐观读取。
- 读操作的核心验证逻辑是：`当前数据版本号 <= 事务开始时的版本号`？ 如果是，则安全；否则，数据已被更新。
- 写操作通过**加锁**实现互斥，保证数据修改的原子性。
- 你可以把它看作一种 **“读者不阻塞写者，写者阻塞写者”** 的机制，但在读取后需要通过版本号确认读操作没有被写者影响。

### 事务内存的并发控制机制：锁与乐观读

#### 1. 核心问题：事务间的数据争用
- **背景**：当多个事务同时访问同一数据，可能出现读写冲突。
- **关键挑战**：如何保证事务读取到的数据是“一致的、未过时的”，同时避免频繁的锁等待。

#### 2. 事务内存（STM）的两种检查条件
- **锁检查**：**“数据未被锁定”**  
  - 如果数据被其他事务锁定（正在写），当前事务必须等待。
  - **作用**：防止读取到“正在被修改的半成品数据”。
- **版本检查**：**“数据版本 < 本地时间戳”**  
  - **核心概念**：每个数据项都带有一个全局时间戳（记录最后一次更新的事务时间）。  
  - **解释**：本地时间戳（事务开始时记录）必须大于数据的最新更新时间，否则说明该事务开始时，数据已经被其他事务修改了（当前事务读到了过时数据）。  
  - **条件**：`数据版本 > 本地时间戳` → 数据已过时 → 事务必须终止。

#### 3. STM 的读写策略：**悲观写 + 乐观读**
- **写入操作（悲观）**：  
  - 在写入前就尝试获取锁（Eager Locking）。  
  - 检查是否满足：数据未被锁定 **且** 数据版本 < 本地时间戳。  
  - 如果通过检查，则：  
    - 获得锁  
    - 将数据插入到“写集”（Write Set）  
    - 创建撤销日志（Undo Log）以备回滚  
    - 立即写入数据（急切写入）。  
  - **关键术语**：**Eager Write**、**Pessimistic Locking**。

- **读取操作（乐观）**：  
  - 读取时不加锁，不检查是否有其他读/写冲突。  
  - 冲突仅在**提交阶段**检测，而不是在读取时刻。  
  - **风险**：后续可能发现读到的数据已被其他事务更新（版本冲突），导致事务需要回滚。  
  - **关键术语**：**Optimistic Read**、**Lazy Conflict Detection**。

#### 4. 验证阶段（Validation）：**读集（Read Set）的重置检查**
- **流程**：  
  1. 获取**全局最新时间戳**。  
  2. 遍历读集中的每一项，检查：  
     - **版本检查**：该项的时间戳是否大于本地时间戳？  
       - 若 `数据版本 > 本地时间戳` → 读取了老数据 → **终止**事务。  
     - **锁检查**：该项是否被锁定？  
       - 若被锁定 → 说明其他事务正在修改该数据，当前事务读取了错误的值 → **终止**事务。  
- **解锁后的处理**：若所有读集项都通过检查，则：  
  - 更新**本地时间戳**为当前全局时间戳（相当于事务从现在开始执行）。  
  - 原因：事务的“眼睛”被更新到最新快照，后续操作基于更准确的时间点。

#### 5. 关键区分：**锁 vs. 版本**
| 场景 | 含义 | 处理方式 |
|------|------|----------|
| **数据被锁定** | 其他事务正在**当前**写入该数据 | 必须等待（或放弃），因为正在修改中 |
| **数据版本过旧** | 其他事务**已经**修改了该数据，且当前本地时间戳低于数据版本 | 必须放弃，因为读到了旧快照 |

- **总结**：锁表示**正在发生**的修改，版本表示**已发生**的修改。两者都意味着读取结果不可靠，需要终止事务。

#### 6. 公式与算法逻辑（伪代码风格）
```
STM_Write(addr, value):
    if not (addr.isLocked() and addr.version < local_ts):
        wait()  // 等待锁释放 或 更新版本
    lock(addr)
    write_set.add(addr)
    undo_log.create(addr, old_value)
    addr.data = value

STM_Read(addr):
    // 不检查锁，直接读取
    read_set.add(addr)
    return addr.data

STM_Validate():
    global_ts = get_global_timestamp()
    for each addr in read_set:
        if addr.isLocked() or addr.version > local_ts:
            abort()  // 终止事务
    local_ts = global_ts  // 更新本地时间戳到最新
```

### 基于乐观并发控制的原子性提交与事务隔离

#### 核心机制：时间戳与锁的协同

- **时间戳结构与用途**
  - 每个对象维护一个**时间戳**（例如初始值：Foo=3, Bar=5）。
  - **最低有效位**（LSB）充当**锁标志**（0表示未锁定，1表示已锁定）。
  - **时间戳递增规则**：每次提交时，时间戳**增加2**（而非1），以确保高位变化与锁状态无关。

- **提交流程**
  1. **验证读取集（Read Set）**：确认所有读取的对象在事务期间未被修改，通过检查其时间戳是否与开始读取时一致。
  2. **写入集（Write Set）操作**：对每个要修改的对象执行：
     - **获取锁**：利用时间戳的LSB标记锁定状态。
     - **记录回滚日志（Undo Log）**：保存旧值（用于急迫模式下可能的回滚）。
     - **执行写入**：将新值写入对象。
  3. **提交完成**：对写入集中的每个对象，**释放锁**并将时间戳设置为**全局时间戳**（当前递增后的值）。

#### 关键区别：验证顺序与原子性

- **验证发生在提交前**：在递增时间戳之前，先验证所有读取的对象是否仍然有效（时间戳未变）。
- **提交后时间戳的更新**：使用**预递增的旧时间戳**进行验证，然后通过加2生成新时间戳，确保锁位被清除，版本号递增。

#### 示例分析：跨对象原子复制与读取隔离

##### 场景设置
- **交易T1**：原子性地将`Object Foo`的值复制到`Object Bar`。
- **交易T2**：原子性地读取`Object Bar`，期望结果要么是完全旧值（0），要么是完全新值（97），**不允许看到部分更新**。

##### 执行过程与冲突处理

1. **T1执行写入**
   - 读取`Foo.x`（时间戳3）加入**读取集**。
   - 读取`Bar`上的锁（初始时间戳5），获取写入锁，并记录旧值到**回滚日志**。
   - 写入`Bar.y`（如新值97），此时`Bar`时间戳变为5（LSB=1，标记锁定）。

2. **T2尝试读取Bar**
   - 遇到`Bar`已锁定（时间戳LSB=1）。
   - **等待策略**：T2等待锁释放，或根据实现选择**终止并重试**（此处采用等待）。

3. **T1提交过程**
   - **验证读取集**：检查`Foo.x`的时间戳仍为3（确认未被其他事务修改）。
   - **递增时间戳**：将`Bar`的时间戳从5增加2变为7（同时清除锁位）。
   - **释放锁**：完成提交。

4. **T2恢复读取**
   - `Bar`时间戳变为7（表示已提交且未锁定）。
   - T2读取到的`Bar.y`值为97（完整的新值），确保**原子性**。

#### 原子性保证原理
- 锁的持有阻止了其他事务在T1提交**中途**读取或修改`Bar`。
- 时间戳递增2的机制保证了**版本号的连续性**，同时锁位的清除标记了提交完成。
- 读取集验证确保T1读取的`Foo`值在事务期间未被改变，从而保证复制操作的**一致性**。

#### 重要算法与公式

- **时间戳递增**：  
  ```text
  新时间戳 = 旧时间戳 + 2
  ```
  - 目的：锁位（LSB）清零，版本号（高位）正确递增。
  - 验证条件：读取集内每个对象的时间戳仍然等于事务开始时读取的值。

- **写入集操作规程**：
  1. 获取锁（设置LSB=1）。
  2. 记录旧值到撤销日志（Undo Log）。
  3. 执行新值写入。
  4. 提交时：释放锁并设置时间戳为`全局时间戳`。

- **验证顺序**：**先验证读取集，后递增写入集时间戳**。  
  - 错误顺序（先递增后验证）可能导致**脏读**或**不可重复读**。

#### 总结：乐观并发控制的关键点

- **乐观锁本质**：通过时间戳验证替代传统锁冲突检测，适用于**低冲突**环境。
- **时间戳的双重角色**：既是版本号，又是锁标志（LSB）。
- **原子性实现**：先锁定写入对象，再验证读取集，确保操作要么全部完成，要么全不执行。
- **隔离性级别**：提供**快照隔离**（Snapshot Isolation）级别的保证，防止部分更新被其他事务看到。

### 双重读取与提交机制

#### 核心概念：读取集（Read Set）验证
- 当存在**两个独立的读取操作**时，需要仔细管理读取集条目
- **关键操作**：每次提交会将**全局时间戳（Timestamp）增加2**（例如：从5变为7）
- 对**同一个对象的两次读取**，如果没有正确记录，可能导致验证失败

#### 悲观检测与放弃机制
- 在**悲观冲突检测**方式下，权利集（Rights Set）的具体内容**不重要**，因为：
  - 使用**撤销日志（Undo Log）** 进行回滚
  - 冲突时通过撤销日志恢复状态
- **正确性要求**：为每个权利集添加独立的条目，确保一致性

#### 提交与重新执行流程

- **提交步骤**
  1. **释放停滞（Stall）**：解除对事务`X2`的等待
  2. **重新读取**：从事务`X2`获取新的读取值（带有**新的日期戳**）
  3. **到达提交点**：执行最终提交操作

- **验证与放弃**
  - **验证读集（Validate Read Set）**：检查第一个读取值是否仍有效
  - **验证失败**：若发现读取值已过期（变为无效），则：
    1. **放弃（Abort）** 当前事务
    2. **重新执行（Re-execute）** 事务
    3. 获取正确的值给条目对象

#### 全局时间戳更新策略

- **关键问题**：全局日期戳的更新是否**原子性**进行？
- **标准做法**：
  - 先**原子性地增加**全局日期戳
  - 使用**旧的全局日期戳**作为**重置基础（Reset Basis）** 进行验证
  - 保证验证的一致性

#### 提交验证算法
```
验证：
    if (读取值时间戳 已过期):
        放弃事务
        重新执行
否则:
        提交事务（全局时间戳 += 2）
```

#### 原子更新公式
```
新全局时间戳 = 旧全局时间戳 + 2
验证基准 = 旧全局时间戳  // 用于验证读取集一致性
```

### 懒乐观并发控制 (Lazy Optimistic Concurrency Control) 的提交与冲突处理

#### 核心概念：交易提交时的冲突检测

- **关键规则**：一个交易提交时，其 **写集 (Write Set)** 必须与 **其他正在运行或即将开始的交易** 的 **读集 (Read Set)** 进行检查。
- **冲突定义**：如果提交交易的写集与另一个交易的读集有交集（即**写-读冲突**），则该另一个交易必须 **回滚并重启**。
- **重叠允许**：一个交易的提交可以与另一个交易的**开始**在时间上重叠，但冲突检查会决定后者的命运。

#### 示例场景解析：多处理器交易系统

##### 1. 状态概览
- **处理器**: P1, P2, P3
- **交易**: T1 (在P1), T2 (在P2), T3, T4
- **关键操作**: 填充表格，确定每个交易的 **读集** 和 **写集**。

##### 2. 具体冲突案例

| 处理器 | 交易 | 读集 | 写集 | 状态 |
|-------|------|------|------|------|
| P1 | T1 | {C, A, C} | ? | 执行中 |
| P2 | T2 | {A} | ? | 执行中 |
| ？ | T4 | {E} | ? | 待开始 |

- **事件**: 交易 **C (T1)** 在处理器 P1 上 **提交**。
- **冲突检查**:
  - 提交交易 (T1) 的 **写集** 与其它交易的 **读集** 进行比对。
  - **结果**: 提交事件发生时，**没有其他交易的读集包含冲突项**。因此，**无需回滚任何交易**，提交成功。

##### 3. 另一个冲突场景

| 处理器 | 交易 | 读集 | 写集 |
|-------|------|------|------|
| P2 | T2 | 提交中 | 包含 `e` |
| - | T4 | 包含 `e` | 待开始 |

- **事件**: 交易 **T2** 提交，其写集包含 `e`。
- **冲突**: 交易 **T4** 的读集也包含 `e`（**写-读冲突**）。
- **结论**: 交易 **T4** 必须 **撤销并重启**。

#### 硬件与总线机制

- **全局可见性**：提交事件必须是 **全局已知的**，所有处理器能立即看到提交的写集。
- **总线 (Bus) 的作用**：
  - 总线提供了 **序列化点 (Serialization Point)**。
  - 所有交易通过总线进行通信，使得“谁先到达”的顺序明确。
  - **优点**：序列化简化了冲突检测和推理。
- **多总线系统**：
  - 可以存在多辆总线，但这会 **使冲突检测问题变得复杂**，因为失去了单一的序列化顺序。
- **覆盖写 (Overwrite)**：
  - 如果一个交易从未读取某个值，而只是覆盖它，则从数据一致性角度看，**覆盖不会引发冲突**（因为没有读-写依赖），结果依然是正确的。

## 四、STM 性能分析与优化

> 主题：STM 的性能开销、屏障优化与可扩展性权衡

### 核心性能开销：从单核到可扩展性的权衡

- **关键概念**：STM的**性能开销**由**额外同步操作**（如日志记录、冲突检测、回滚机制）引入，体现在**运行时间增加**的百分比上。
- **性能数据来源**：McCarty 论文中的实验数据，对比了 **哈希映射** 和 **树映射** 两种数据结构，测试操作以 **80% 读取 + 20% 插入** 为主。
- **性能对比维度**：
  - **横轴**：不同的同步策略。
  - **纵轴**：**性能开销百分比**（相对于无线程安全保护的基线代码）。
  - **基线**（**0%**）：**非线程安全代码**，无法在并行下正确运行。
  - **深蓝色条**：使用**单一粗粒度锁**（如 Java `synchronized` 装饰整个方法）。
  - **紫色/红色条**：使用**无编译优化的 STM**。
  - **粉色/橙色条**：使用**经过编译优化的 STM**。

#### 单处理器环境下的开销对比（无扩展性）

- **粗粒度锁开销**：约 30% 开销。虽然开销较低，但**无扩展性**（之前已证明）。
- **无优化 STM 开销**：高达 **70-80%** 开销，性能较差。
- **优化后 STM 开销**：显著降低，约 **40%** 开销。
- **核心结论**：在单核场景下，粗粒度锁开销最低，但无法利用多核；STM（特别是优化后）则提供了**合理的并发能力**。

#### 可扩展性：STM 的竞争优势

- **粗粒度锁的局限**：虽然开销相对较低，但在多线程环境下**无法正确扩展**，成为性能瓶颈。
- **细粒度锁 vs. STM**：虽然图中未显示细粒度锁的数据，但 STM 提供了更简单的编程模型和自动的冲突解决。
- **STM 的价值**：结合编译器优化，STM 提供了一个**既可扩展**（能利用多核）且**开销合理**（约 30-40%）的方案，避免了粗粒度锁的瓶颈和细粒度锁的复杂性。
- **重要提醒**：不能与“无同步”代码直接比较，因为那是**线程不安全**的。应将其与“细粒度锁”的开销与复杂性进行对比。

### 原子区域操作的实现步骤（基于急切版本控制）

#### 示例场景
原子区域：`atomic { obj.f1 = 42; }`

#### 实现步骤（基于乐观读取、悲观写入、急切版本控制策略）

1. **获取事务描述符**：获取当前线程的**事务上下文**。
2. **检查读取集**：**乐观读取**，对字段 `f1` 执行读取操作，并记录到**读取日志**中，用于后续冲突检测。
3. **检查写入集/获取锁**：**悲观写入**，尝试获取对字段 `f1` 所属对象的**写入锁**（或标记为已锁定）。如果无法获取锁，则事务可能**中止并重试**。
4. **记录到撤销日志**：在真正写入之前，先将 **`f1` 的旧值**记录到**撤销日志**中，以便在事务**回滚（abort）**时能恢复到原始状态。
5. **执行写入**：**急切执行**，直接对 `obj.f1` 写入新值 `42`。
6. **提交阶段**：检查**读取集**中的每个项是否在事务执行期间被其他事务修改过。如果发现冲突，则**回滚**，利用撤销日志恢复原值，释放锁，并重新开始事务。如果无冲突，则**提交**，释放所有锁，清理日志。

### 屏障优化与性能

#### 冗余屏障问题
- 屏障操作会引入**显著的开销**
- **发现**：屏障中的许多操作实际上是**冗余的**

#### 优化方案
- **分解屏障**：将复杂的屏障拆分为**单个组件**
- **消除冗余**：
  - 去除重复的对象日志记录
  - 简化权利集管理
- **结果**：额外开销**显著降低**

#### 性能指标
- 引用**McCurdy的论文数据**：
  - 优化后的事务内存系统性能提升显著
  - 具体数据需要参考原论文（此处理论描述）

## 五、硬件事务内存 (HTM) 的核心动机与设计

> 主题：HTM 的硬件优化思路、缓存一致性协议扩展与冲突检测机制

### 软件事务内存 (STM) 的性能瓶颈：屏障开销

- **核心问题**：STM 的性能开销主要来自 **软件事务内存屏障**。
  - 每个线程的每次读写操作都必须插入显式的屏障指令（如 `read_barrier`, `write_barrier`）。
  - 这导致每个线程的操作成本高昂，数据表明每个操作可达 **2 到 8 个额外周期**。
  - **关键术语**：**显式屏障**、**读写检测**。
- **性能分解**：在典型应用（如 `vacation` 三层服务器、`k-means` 聚类）中，大量执行时间被消耗在：
  - **屏障指令**（图中白色/黑色部分）：用于检测“我是否在做一次读取/写入”。
  - **提交成本**：事务结束时，需要验证并提交修改。

### 硬件优化的思路：消除显式屏障

- **核心目标**：**“做加载和存储，让硬件事务管理系统做正确的事。”**
  - 用户代码只需执行普通的 `load` 和 `store`。
  - **不再需要**显式的 `read_barrier()` 或 `write_barrier()` 调用。
- **实现方式**：**“骑在（leverage）现有的一致性协议和缓存架构上”**。
  - **利用缓存**：使用缓存行存储事务的读/写集版本信息。
  - **利用一致性协议**：通过监听缓存一致性消息（如共享请求、独占请求）来自动检测冲突。

### HTM 的核心硬件机制

#### 1. 检查点与回滚支持

- **需求**：当事务冲突或失败时，处理器必须能回滚到事务开始前的状态。
- **实现**：**检查点寄存器状态**。
  - 在事务开始时，保存处理器寄存器内容（`PC`, `GPRs`, `Flags` 等）的快照。
  - 回滚时，恢复该快照，丢弃事务内所做的所有修改。

#### 2. 缓存行元数据的增强 (R-bit & W-bit)

- **架构改动**：在每个缓存行中，除了已有的**一致性位**（MESI状态），增加两个额外的**事务状态位**。
  - **R-bit (Read bit)**：**读状态位**。表示该缓存行已被当前处理器**读取**过（是事务读集的一部分）。
  - **W-bit (Write bit)**：**写状态位**。表示该缓存行已被当前处理器**写入**过（是事务写集的一部分）。
- **核心作用**：利用这些位进行**自动冲突检测**，无需软件指令。

#### 3. 基于缓存一致性协议的冲突检测

- **核心逻辑**：通过监听总线上的**缓存一致性请求**，结合 R/W-bit 来判断是否发生冲突。
- **冲突场景 1：写-写/写-读冲突**
  - **条件**：当前缓存行设置了 **W-bit**（表明我写了它）。
  - **事件**：其他处理器发送了对该缓存行的 **共享请求**（想读）或 **独占请求**（想写）。
  - **判定**：**这是冲突**。因为你已修改，而他人想读/写。
- **冲突场景 2：读-写冲突**
  - **条件**：当前缓存行设置了 **R-bit**（表明我读了它）。
  - **事件**：其他处理器发送了对该缓存行的 **独占请求**（想写）。
  - **判定**：**这是冲突**。因为你已读取，而他人想写入（违反可串行化）。
- **冲突场景 3：无冲突**
  - 如果当前缓存行**既没有 R-bit 也没有 W-bit**，或他人请求的访问模式与之不冲突（如读-读），则事务可继续。

#### 总结：HTM 如何通过硬件“魔法”工作

1. **事务开始**：硬件保存寄存器检查点。
2. **事务执行**：
   - **读操作**：从缓存加载数据，并自动设置该缓存行的 **R-bit**。
   - **写操作**：写入缓存，并自动设置该缓存行的 **W-bit**（通常数据直接写入本地缓存，尚未提交）。
3. **隐式冲突检测**：硬件通过缓存一致性协议（如 MESI）自动监听总线请求。一旦检测到前述冲突，**硬件自动触发回滚**，恢复寄存器检查点并清除缓存中的 R/W-bit。
4. **事务提交**：如果无冲突，事务成功提交。清空所有 R/W-bit，使脏数据对全局可见（无需显式屏障指令）。

**一句话总结：HTM 利用缓存行上新增的 R/W 位和现有的一致性协议，将 STM 软件屏障的开销消除，让硬件自动完成冲突检测与回滚，实现了“无感”的事务执行。**

### 事务内存中的缓存一致性协议扩展与冲突处理

#### 核心概念
