# 分布式训练：多GPU与多节点并行

## 核心概念：硬件层级与通信成本

### 硬件层次结构（从快到慢）

- **单GPU内部**：
  - **L1缓存**：速度极快，容量极小，在流式多处理器（SM）内部
  - **高带宽内存（HBM）**：容量更大，但速度远慢于L1

- **单节点内部**（多GPU）：
  - **NVLink**：同一节点内GPU之间的高速互联
  - **NVSwitch**：节点内多个GPU的交换连接

- **跨节点**：
  - **网络通信**：速度最慢，成为主要瓶颈

### 核心原则：保持高算术强度

> **算术强度** = 计算量 / 数据传输量

**关键思想**：计算必须在SM内的ALU单元完成，但输入输出数据可能位于：
- **最佳情况**：数据已在L1缓存中
- **最差情况**：数据在HBM或其他GPU上

**上周**（单GPU优化）：
- **融合（Fusion）**：合并多个操作减少中间读写
- **分块（Tiling）**：将数据分块加载到共享内存/L1，本地计算后写回HBM

**本周**（多GPU/多节点）：
- **复制和分片**模型参数
- **优化通信状态**，因为数据传输速度远慢于计算速度

## 集体操作：分布式编程的基础原语

### 基本概念

- **集体操作（Collective Operations）**：涉及多个参与者的通信模式
- **世界规模（World Size）**：参与通信的设备总数
- **历史背景**：源自1980年代并行编程，比手动点对点通信提供更好的抽象

### 常用集体操作

| 操作 | 描述 | 关键特性 |
|------|------|----------|
| **AllReduce** | 所有设备上的数据求和/求平均后，结果分发给每个设备 | 最常用于梯度同步 |
| **Broadcast** | 一个设备发送数据给所有其他设备 | 单向传播 |
| **Scatter** | 一个设备将数据分片发送给不同设备 | 数据分布 |
| **Gather** | 所有设备将数据发送给一个设备 | 数据收集 |
| **AllGather** | 所有设备收集所有其他设备的数据 | 完全交换 |
| **ReduceScatter** | 先在本地reduce，然后scatter结果 | 高效组合 |

### 在PyTorch中的实现

```python
import torch.distributed as dist

# 初始化分布式环境
dist.init_process_group(backend='nccl')

# AllReduce操作
dist.all_reduce(tensor, op=dist.ReduceOp.SUM)

# Broadcast操作
dist.broadcast(tensor, src=0)

# 更多操作...
dist.all_gather(tensor_list, tensor)
dist.reduce_scatter(tensor, tensor_list)
```

**性能基准测试**：了解不同集体操作在不同数据量下的延迟和带宽特性至关重要

## 分布式训练范式

### 数据并行（Data Parallelism）

**核心思路**：
- 每个设备持有完整的模型副本
- 训练数据被分片到各个设备
- 每个设备独立前向/反向传播
- 通过**AllReduce**同步梯度

**关键指标**：
- **Batch Size** = 每个设备的本地Batch × 设备数量
- **梯度同步**是主要通信开销

### 张量并行（Tensor Parallelism）

**核心思路**：
- 单个张量/矩阵操作被拆分到多个设备
- 每个设备负责计算一部分
- 需要设备间的**点对点通信**

**典型场景**：
- 超大模型单GPU放不下
- 矩阵乘法拆分（行/列分割）

### 流水线并行（Pipeline Parallelism）

**核心思路**：
- 模型按层/阶段分割到不同设备
- 数据以微批次（micro-batch）形式流经流水线
- 前向和反向传播交错执行

**关键挑战**：
- **气泡（Bubble）**：流水线启动和排空时的空闲时间
- **负载均衡**：各阶段计算时间需相近

## 总结与关键公式

### 通信成本模型

$$
\text{通信时间} = \alpha + \beta \times \text{数据量}
$$

- $\alpha$：**延迟**（每次通信的固定开销）
- $\beta$：**带宽倒数**（每字节传输时间）

### 并行效率

$$
\text{效率} = \frac{\text{单GPU训练时间}}{\text{多GPU训练时间} \times \text{GPU数}}
$$

### 设计原则

1. **数据本地性**：尽量在L1/共享内存中完成计算
2. **通信隐藏**：让计算和通信重叠
3. **操作融合**：合并多个小通信为一个大通信
4. **选择正确的并行策略**：根据模型大小和硬件拓扑

### NVIDIA 集体通信原语详解与硬件实现（第二部分）

#### 一、 核心集体通信原语（复习与深化）

在本节中，我们将回顾并精炼一组**经过验证、经得起时间考验**的并行计算基础操作。这些操作是构建大规模分布式渲染和计算管线的基石。

> **术语澄清**：在并行计算语境中，“世界规模”本质上指参与计算的**设备总数**。例如，若有4个设备，则编号分别为 0, 1, 2, 3。这里的“节点编号”直接对应设备ID，请勿与线性代数中的节点概念混淆。

以下为四种最关键的集体操作：

- **广播（Broadcast）**：**核心概念**是将一个**根节点**（如节点0）的数据（`T0`）**复制并发送**到所有其他节点。结果是所有节点都拥有相同的完整数据块。

- **散射（Scatter）**：**核心概念**是将一个**根节点**上持有的**一组值**（如4个值），**分割并分发**给每个节点。**每个节点获得唯一且不同的数据片段**。这是数据并行的起点。

- **收集（Gather）**：**核心概念**是**散射的逆操作**。将散布在各个节点上**不同的数据值**，**汇集**到一个**目标根节点**处。

- **归约（Reduce）**：**核心概念**是执行一个**结合且可交换**的操作（如 **加法、求最小值、最大值、平均值**）到来自不同节点的数值上，并将**最终结果**放在一个**根节点**上。

#### 二、 快速记忆与区分变种操作

“全（All-）”前缀与组合操作是常见的易混淆点，请通过以下逻辑来区分：

- **全收集（All-Gather）**：等同于 **Gather + Broadcast**。结果不限于单个根节点，而是**所有节点**都拥有最终的完整数据集合。

- **全归约（All-Reduce）**：等同于 **Reduce + All-Gather**。结果不限于单个根节点，而是**所有节点**都拥有最终的归约结果。

- **归约散射（Reduce-Scatter）**：一个更复杂的组合。先将各节点的值进行**归约**，然后将结果**散射**到各个节点上。可以理解为 **Reduce + Scatter**，但操作是**原子化**的，中途不产生完整数据副本。

| 操作 | 根节点示例 (节点0) | 其他节点 (节点1,2,3) | 数据流向 |
| :--- | :--- | :--- | :--- |
| **Broadcast** | 发送数据 | 接收数据 | 0 -> 所有 |
| **Scatter** | 发送**片段** | 接收**片段** | 0 -> 1,2,3 |
| **Gather** | 接收数据 | 发送数据 | 1,2,3 -> 0 |
| **Reduce** | 接收并计算 | 发送数据 | 所有 -> 0 (计算) |
| **All-Gather** | 接收并**共享** | 接收并**共享** | 所有 -> 所有 |
| **All-Reduce** | 接收、计算并**共享** | 发送、接收并**共享** | 所有 -> 计算 -> 所有 |
| **Reduce-Scatter** | 接收、计算并**分发** | 发送、接收并**分发** | 所有 -> 计算 -> 分散 |

> **助记技巧**：
> - “All-” 代表操作结果**分发给所有设备**。
> - “-Scatter” 代表结果**被分割**后分发。

#### 三、 硬件实现的演进：从传统瓶颈到专用互联

理解这些原语的硬件实现方式，是优化性能的关键。

##### 3.1 传统架构：PCIe 与 以太网瓶颈

- **场景**：单节点内插多GPU，或跨节点连接。
- **路径**：GPU A 数据 → GPU显存 → **PCIe总线** → **CPU内存**（内核缓冲区） → **以太网** → 另一节点 → PCIe → GPU B。
- **问题**：路径漫长，**数据拷贝开销巨大**。数据必须经过CPU内存，且以太网协议并非为高频、低延迟的GPU间通信设计，导致**延迟高、带宽利用率低**。

##### 3.2 现代高性能架构：NVIDIA NVLink 与 NVSwitch

这是深度学习、大规模科学计算和高端渲染（如集群渲染）的核心技术。

- **NVLink（机内/节点内）**：
    - **核心概念**：**绕过CPU**，直接在GPU之间建立**高速、低延迟**的点对点或全互联通路。
    - **优势**：数据无需经过PCIe和CPU内存，实现**GPU-to-GPU直接通信**，极大降低了延迟并提高了有效带宽。

- **NVSwitch（跨节点/机架间）**：
    - **核心概念**：一种**专用的高带宽交换机**，通过NVLink协议连接多个节点上的GPU，构建一个**无阻塞、全互联**的GPU池。
    - **优势**：**替代了以太网**。NVSwitch专为这类集体通信负载优化，避免了传统网络协议的开销和握手延迟，实现了跨机架的**线性带宽扩展**。

> **关键洞察**：对于H100这样的顶级计算卡，每个GPU通过NVLink拥有**高达18MB（或更高）** 的专用显存带宽用于互联，这使得All-Reduce等操作可以在极短时间内完成，是训练大模型和运行大规模渲染农场的基础。

**总结**：传统架构（PCIe + 以太网）是性能的瓶颈，而现代架构（NVLink + NVSwitch）通过**直连GPU**和**专用交换机**，实现了低延迟、高带宽的集体通信，是为大规模并行计算量身定制的解决方案。

### 一、NVIDIA NVLink 与硬件通信优化

*   **核心概念：** 为了规避传统PCIe总线的瓶颈，NVIDIA设计了 **NVLink**，这是一种**高带宽、低延迟的直接GPU间互联协议**。
*   **关键术语：**
    *   **NVLink Switch：** 模拟并跳过传统连接步骤，专门针对GPU间的数据负载进行优化，实现**直接点对点通信**。
    *   **H100 GPU的NVLink规格：**
        *   **容量：** 每个GPU拥有 **18MB** 专用缓存（用于NVLink）。
        *   **总带宽：** 提供 **900 GB/s** 的聚合带宽，远快于PCIe和以太网。
*   **性能对比：**
    *   **与PCIe/以太网比：** **明显更快**（数量级差异）。
    *   **与HBM（高带宽内存）比：** 从SM单元通过HBM读取数据，速度比NVLink快大约 **3-4倍**（具体数据因Blackwell等新架构而异，约提升2-3倍）。
*   **关键区别：**
    *   **PCIe：** 是**通用总线**，设计用于多种设备（显卡、声卡、SSD），且数据通常**必须经过CPU**，导致延迟增加。
    *   **NVLink：** 是**专用GPU互联**，数据可直接在GPU间传输，**绕过CPU**，但**与CPU的通信仍需要通过PCIe**（例如调度任务、初始化）。

### 二、硬件拓扑查看与诊断

*   **实用工具：** 使用命令行工具（如 `nvidia-smi topo -m`）可以查看GPU之间的**实际物理连接关系**。
*   **输出示例：**
    *   每对GPU之间都通过 **NV18**（即NVLink 18条链路）互联。
    *   显示GPU与网卡（NIC）、CPU等其他设备的连接拓扑。
*   **解读：** 网卡通常通过PCIe连接到CPU，再通过PCIe交换机与GPU通信，形成完整的 **GPU <-> CPU <-> 网卡** 的通信路径。

### 三、NCCL：NVIDIA 集合通信库

*   **核心概念：** **NCCL (NVIDIA Collective Communications Library)** 是一个专门优化过的**集合通信库**，负责在GPU集群上高效执行`AllReduce`、`AllGather`等操作。
*   **工作机制：**
    1.  **初始化阶段：** 探测硬件拓扑（GPU、NVLink、网络等），**自动优化**数据在GPU间的传输路径。
    2.  **执行阶段：** 用户调用集合操作，NCCL在底层启动**CUDA内核**直接发送/接收数据。
*   **核心优势：**
    *   **抽象层次：** 程序员只需说“让所有GPU拥有相同的张量”，无需手动管理点对点传输和同步。
    *   **硬件优化：** 深度针对NVLink和高端网络进行优化，自动选择最佳算法（如Ring、Tree算法）。

### 四、PyTorch Distributed：高级抽象层

*   **核心概念：** **PyTorch分布式库（`torch.distributed`）** 为NCCL提供了**更简洁的Python接口**，让用户能通过`all_gather(tensor)`等高级API调用，而无需直接操作NCCL的C++ API。
*   **关键特性：**
    *   **硬件后端支持：**
        *   **`nccl` 后端：** 专门用于 **GPU**，性能最优。
        *   **`gloo` 后端：** 用于 **CPU**，可在笔记本上**调试**作业时使用，无需GPU。
    *   **代码示例：** 在PyTorch中，`dist.all_gather(tensor_list, tensor)` 自动在所有进程中分发数据。
*   **优势总结：** 这体现了高级原语的价值：将意图（“我需要这个张量在所有机器上”）与实现（复杂的PCIe/NVLink通信、数据拷贝）分离，极大简化分布式训练开发。

## 分布式集体通信基础操作

### 核心概念：高级原语（High-Level Primitives）

**分布式集体通信原语**是构建分布式训练和计算的基础组件，它们提供了**高级抽象**，使得开发者无需直接处理底层网络通信细节。

- **无需手动管理GPU间数据传递**：高级原语封装了复杂的同步和数据交换逻辑
- **硬件无关性**：逻辑上确保代码能在不同硬件配置下运行，实际性能取决于GPU算力和互联带宽
- **支持高级功能**：如 `SDP`（序列数据并行）等现代分布式训练技术

### 进程组管理与协调机制

#### 关键函数与术语
- **`world size`**：参与并行计算的**总进程数**
- **`rank`**：每个进程的**唯一标识符**，范围从 `0` 到 `world size - 1`
- **进程组初始化**：所有进程启动后需完成自我初始化，并通过协调机制**互相发现**

#### 进程发现与同步
```python
# 典型的多进程协调流程
# 1. 初始化分布式环境
# 2. 使用协调工具确保进程组成员间可通信
# 3. 建立数据交换通道
```

- **GPU环境协调**：使用 `NCCL`（NVIDIA Collective Communications Library）作为后端
- **非GPU环境协调**：回退到其他协议，如 `GLOO`

### 同步屏障（Barrier）

**`torch.distributed.barrier()`**

> **核心机制**：创建一个**同步点**，等待进程组中**所有进程都到达此处**后才继续执行

**应用价值**：
- 保证**打印语句输出集中**，避免异步执行导致的输出混乱
- 提供**同步保障**，确保后续操作基于**所有进程已完成前一阶段**的假设

**异步特性提醒**：
- 所有分布式操作本质上是**异步执行的**
- `Barrier` 提供**显式同步时刻**，用于需要确定执行顺序的场景

### 张量初始化实例

在每个进程中构造本地张量：
- 基础序列：`[0, 1, 2, 3]`
- 偏移量：`+ rank`（进程编号）
- **示例输出**（未排序，反映异步并行特性）：
    - 进程0（rank=0）：`[0, 1, 2, 3]`
    - 进程1（rank=1）：`[1, 2, 3, 4]`
    - 以此类推

输出顺序**无序**，反映了**异步执行和输出缓冲**的自然行为。

### 全局归约（All-Reduce）操作

**`torch.distributed.all_reduce(tensor, op=ReduceOp.SUM)`**

#### 核心操作流程
1. **每个进程拥有不同的张量**（基于偏移量构造）
2. **所有进程张量对应位置求和**，结果同步到**每个进程**
3. **原地操作**：原张量被覆盖为归约结果

#### 同步模式
```python
# 默认同步模式（本示例使用）
all_reduce(tensor, op=SUM)  # 阻塞直到完成
# 异步模式（可选）
all_reduce_async(tensor, op=SUM)  # 可与其他计算重叠
```

#### 计算示例（4个进程，向量长度4）
| 进程 | 归约前张量 | 归约后张量 |
|------|------------|------------|
| 0 | [0, 1, 2, 3] | [6, 10, 14, 18] |
| 1 | [1, 2, 3, 4] | [6, 10, 14, 18] |
| 2 | [2, 3, 4, 5] | [6, 10, 14, 18] |
| 3 | [3, 4, 5, 6] | [6, 10, 14, 18] |

**解析**：
- 第1列：$0+1+2+3 = 6$
- 第2列：$1+2+3+4 = 10$
- 第3列：$2+3+4+5 = 14$
- 第4列：$3+4+5+6 = 18$

**异步操作的价值**：
- 通信与计算**重叠**
- 发送数据的同时可以执行其他计算，**隐藏通信延迟**

### 分段归约散射（Reduce Scatter）

**`torch.distributed.reduce_scatter(output, input_list, op=ReduceOp.SUM)`**

#### 核心特性
- **非原地操作**：需要**显式分配输出张量**
- **输入维度要求**：输入列表长度需等于 `world size`
- **输出本质**：每个进程只获得**归约结果的特定片段**

#### 操作流程
1. 创建输入张量列表，长度 = 进程数
2. 分配输出张量（初始为任意值）
3. 执行 `reduce_scatter`：
   - **先归约**：对应位置的张量求和
   - **后散射**：将归约结果的每个片段分配到对应 `rank` 的进程

#### 示例（4个进程）
- **归约阶段**：输入列表对应位置元素全局求和
- **散射阶段**：
  - `rank 0` 获得第一个求和结果
  - `rank 1` 获得第二个求和结果
  - 以此类推

**对比理解**：
- `All-Reduce`：**每个进程都得到完整结果**
- `Reduce Scatter`：**每个进程只得到结果的一部分**（自己的那一段）

### 集体通信操作详解：Reduce Scatter 与 All Gather

#### 1. **Reduce Scatter 操作**

- **核心概念**：**分布式归约 + 分散结果**
  - 功能：对所有设备上的输入张量进行**逐元素求和**（或其他归约操作，如求平均值、最大值），然后将**归约后的结果切片**分散到各个设备上。
  - **关键流程**：
    1. 输入：每个设备有一个形状为 `(world_size, ...)` 的张量（假设第一维大小为 `world_size`）。
    2. 系统通过**约定**知道：维度 `0` 必须等于 `world_size`，从而推断出每个设备对应的输出切片（例如 `rank 0` 得到第 `0` 个元素的归约结果，`rank 1` 得到第 `1` 个元素的归约结果，依此类推）。
    3. 输出：每个设备得到一个**部分归约结果**（如 `rank i` 得到全局归约结果中第 `i` 个元素的值）。
  - **类比**：Reduce Scatter = 先做全局归约（求和），再将结果**分散**到各设备，每个设备仅保留一个切片。

- **代码示例流程**：
  ```python
  # 每个rank上的输入张量，形状为 (world_size,)
  input = [a0, a1, a2, a3]  # 假设 world_size = 4
  # 执行 reduce_scatter，operation = 求和
  output = dist.reduce_scatter(input, op=dist.ReduceOp.SUM)
  # 结果：rank 0 得到 sum(all_ranks的a0)，rank 1 得到 sum(all_ranks的a1)，等等
  ```

- **重要公式**：
  - 设 `data_rank_i[j]` 为 `rank i` 上输入张量的第 `j` 个元素。
  - Reduce Scatter 结果对 `rank k`：
    \[
    \text{output}_k = \sum_{i=0}^{\text{world\_size}-1} \text{data}_i[k]
    \]

- **备注**：必须确保输入张量的**某一维度等于世界大小**，系统才能自动推断输出切片归属。

#### 2. **All Gather 操作**

- **核心概念**：**收集所有设备的张量并拼接**
  - 功能：将每个设备上的输入张量（通常是部分数据）**收集起来**，并在所有设备上**复制完整的结果**。
  - **关键流程**：
    1. 输入：每个设备有一个张量（例如 Reduce Scatter 的输出）。
    2. 所有设备的输入被收集并沿着一个维度拼接。
    3. 输出：每个设备都得到完整的拼接后张量。
  - **目的**：All Gather 是 **Reduce Scatter 的逆操作**，两者结合可构成 **All Reduce**。

- **代码示例流程**：
  ```python
  # 每个rank上的输入张量（例如Reduce Scatter后的部分结果）
  input = [partial_result]  # 形状为 (1,)
  # 执行 all_gather
  output = dist.all_gather([buffer], input)
  # 结果：所有rank都得到完整张量 [sum_0, sum_1, sum_2, sum_3]
  ```

- **结合 Reduce Scatter 与 All Gather 等价于 All Reduce**：
  - **证明**：
    1. Reduce Scatter 将全局归约结果分散到各设备（每个设备有一个子结果）。
    2. All Gather 收集这些子结果并在所有设备上复制完整结果。
    3. 最终，每个设备都拥有全局归约后的完整张量，这正是 All Reduce 的效果。
  - **公式化**：
    \[
    \text{AllReduce} = \text{ReduceScatter} + \text{AllGather}
    \]

### 3. **集体通信带宽基准测试（单节点示例）**

#### 3.1 测试流程与注意事项

- **核心步骤**：
  1. **初始化**：设置 `world_size = 4`，创建包含一亿个元素的张量（数据类型：`float32`，每个元素 4 字节）。
  2. **预热**：先执行一次操作（如空循环或小规模 All Reduce），确保所有内核已加载，避免首次运行因初始化导致的计时偏差。
  3. **清理环境**：使用 `dist.barrier()` 确保所有设备就绪；同步 CUDA 流（`torch.cuda.synchronize()`）。
  4. **计时**：
     - 启动计时器 → 执行 All Reduce → 再次同步 CUDA 流（确保异步操作完成） → 停止计时。
     - 打印耗时（建议以微秒或毫秒为单位）。
  5. **计算带宽**：带宽（实际传输量）的计算需要考虑**实际发送/接收的字节数**。

#### 3.2 带宽计算中的“两倍因素”

- **问题**：为什么带宽计算时需要乘以 2？
- **详细解释**：
  - **直观误区**：假设每个设备发送大小为 `S` 的张量给其他设备，总发送量似乎为 `S * (world_size - 1)`。
  - **实际原因**：
    1. **All Reduce 是双向过程**：每个设备既要**发送**自己的数据给其他设备，也要**接收**其他设备的数据。
    2. **典型实现（Ring All Reduce）**：
       - 数据在设备间以环状传递，每个设备在每一轮中会**发送**一个数据块，同时**接收**一个数据块。
       - 整个过程中，每个设备**收发总量**为 `2 * N`（其中 N 是张量字节数），但这属于**节点内部**的通信量。
    3. **带宽计算目标**：衡量**实际从每个设备流出的数据量**或**整体链路吞吐量**。
       - 若仅考虑“发送”方向，每个设备发送量为 `S * (world_size - 1)`。
       - 但更准确的**有效带宽**公式为：
         \[
         \text{有效带宽} = \frac{\text{总数据量}}{\text{耗时}} = \frac{2 \times \text{张量字节数}}{\text{耗时}}
         \]
         - 因为一次 All Reduce 操作，每个设备需要**发送一次数据**并**接收一次数据**（即数据的完整广播和归约）。
       - 例如：若张量大小为 400MB，`world_size=4`，总传输量按 `2 × 400MB = 800MB` 计算更符合实际通信模式。

- **总结**：两倍因子反映了**发送和接收的双向性**，而非单纯的发送或接收量。这对于与硬件带宽（如 PCIe 或 NVLink 的理论带宽）进行比较至关重要。

---

### 4. **关键术语与公式汇总**

| 术语 | 定义 | 公式/要点 |
|------|------|-----------|
| **Reduce Scatter** | 全局归约后，将结果切片分散到各设备 | `output_k = Σ data_i[k]` |
| **All Gather** | 收集所有设备的张量，并在每个设备上复制完整结果 | 拼接分散的切片 |
| **All Reduce** | Reduce Scatter + All Gather 的组合 | 最终每个设备有全局归约结果 |
| **有效带宽** | 实际传输量（含双向）除以耗时 | `BW = (2 × 张量字节数) / 耗时` |

### 5. **代码示例：All Reduce 基准测试模板**

```python
import torch
import torch.distributed as dist
import time

def bench_all_reduce(world_size=4, numel=100_000_000, dtype=torch.float32):
    dist.init_process_group("nccl", rank=...)
    tensor = torch.randn(numel, dtype=dtype).cuda()

    # 预热
    for _ in range(10):
        dist.all_reduce(tensor)
    torch.cuda.synchronize()
    dist.barrier()

    # 计时
    dist.barrier()
    start = time.time()
    dist.all_reduce(tensor)
    torch.cuda.synchronize()
    end = time.time()

    elapsed_us = (end - start) * 1e6
    byte_size = tensor.numel() * tensor.element_size()
    # 有效带宽（双向）
    bandwidth = (2 * byte_size) / (elapsed_us * 1e-6)  # GB/s
    print(f"耗时: {elapsed_us:.2f} us, 带宽: {bandwidth:.2f} GB/s")
```

- **注意**：实际通信模式会受硬件拓扑（如环形、树形）影响，两倍因子是理论下限，真实带宽需结合具体实现测量。

## 集合通信性能分析：AllReduce 与 ReduceScatter

### 1. AllReduce 操作的带宽与时间计算

**核心概念：** AllReduce 操作的时间消耗与**世界大小（进程数）** 直接相关，存在 **“两倍因子”** 现象。

- **两倍因子的来源：**
  - 每个节点需要 **发送** 自己的输入数据
  - 同时需要 **接收** 聚合后的结果
  - 这意味着一次 AllReduce 操作的实际通信量是数据量的 **2倍**（发送 + 接收）
  
- **总时间公式：**
  \[
  \text{总时长} = \text{世界大小} \times \text{实际经过的墙钟时间}
  \]
  - 例如：4个处理器 → 实际时间是单次操作的 **4倍**

- **带宽计算公式：**
  \[
  \text{带宽} = \frac{\text{字节数}}{\text{时长}}
  \]
  
- **实测结果示例：**
  - 实际带宽约为 **277 GB/s**
  - 理论值可达约 **900 GB/s**
  - **差异原因：** 取决于张量大小、设备数量、网络状况等多种因素
  - **关键建议：** 必须进行实际基准测试，不能依赖理论值

### 2. ReduceScatter 操作分析

### 核心概念：Reduce Scatter 的本质

### 数据并行（Data Parallelism）深度解析

#### 1. 核心思想：沿批次维度切割

- **模型结构**：假设一个4层MLP，每层是一个矩阵乘法，隐藏维度为 `1024`。

  - `世界大小 = 4`（假设有4个进程）

##### 训练循环（核心步骤）

1. **前向传播**：
   - 逐层执行：**矩阵乘法** -> **非线性激活** -> **矩阵乘法** -> **非线性处理**。
   - 共4层操作。

2. **损失计算**：损失值仅为虚构，用于演示梯度计算流程。

3. **反向传播**：实现标准的梯度下降反向传播。

#### 3. 关键操作：同步梯度（DDP）

- **核心代码**：在反向传播后，对每一层的参数梯度调用 `all_reduce` 操作。
  - `all_reduce` 的对象是 **参数梯度**。
  - 默认操作是 **求和（SUM）**，然后根据进程数求**平均（MEAN）**。

- **流程说明**：
  - 你接管了标准的前向/反向传播流程。
  - 在反向传播后，**主动调用集合通信**来混合所有进程的梯度。
  - 这一步完成后，所有进程的梯度变得**一致**。

- **参数更新**：
  - 混合梯度后，按常规的梯度下降方式更新参数。

### 数据并行 (Data Parallelism, DP) 与 分布式数据并行 (Distributed Data Parallel, DDP) 核心机制

*   **核心概念**: **分布式数据并行** 的核心思想是**数据切分，模型复制**。每个进程（Rank）持有**完整的模型副本**，但只处理**数据的一个子集**。
*   **核心操作**: **All-Reduce (全规约)**。
    *   **负责流程**: 每个 Rank 独立计算其本地数据的损失和梯度。然后，通过 All-Reduce 操作，将所有 Rank 计算出的梯度进行**求和** 或 **平均**，并将结果广播回每个 Rank。
    *   **关键特性**: **All-Reduce 是一个同步点**。它会强制所有参与的进程在进入下一步前都完成各自的梯度计算和规约操作。这保证了所有进程最终获得**完全一致的梯度**，从而同步更新出**完全一致的模型参数**。
    *   **异常处理**: 如果有一个 Rank 由于故障等原因**错过** All-Reduce 调用，所有其他 Rank 会在该同步点**永久挂起**，等待该掉队进程。
*   **梯度视角**: 虽然不同 Rank 因为处理的数据不同，其**损失值**不同，但经过 All-Reduce 操作后，所有 Rank 的**梯度**已经归一化为**完全相同**。因此，后续的参数更新步骤（`optimizer.step()`）在所有 Rank 上的行为一致。
*   **通信开销**: 所有 Rank 都需要互相传递梯度数据。当模型参数总量很大时，All-Reduce 的通信开销会成为主要瓶颈。

### 全分片数据并行 (Fully Sharded Data Parallel, FSDP) 挑战

*   **核心思想**: **数据和模型状态都分片**。
*   **优化目标**: 以牺牲通信量（增加通信步骤）来换取**显存节省**。
*   **挑战**: 相比 DDP，原理和实现都**更加复杂** (提到“像一团乱麻”)。它需要在计算过程中动态地“收集” (all-gather) 和“释放” (reduce-scatter) 模型参数、梯度和优化器状态。
*   **选择**: 在某些场景下，用户可能跳过 FSDP，选择**张量并行**来降低显存占用，因为实现相对更直接。

### 张量并行 (Tensor Parallelism, TP) 原理

*   **核心概念**: **模型切分，数据复制**。将**模型各层**（特别是线性层）的权重矩阵沿着**特征维度** (有时是隐藏维度 `Hidden Dim`) 切分到多个 GPU 上。
*   **目标**: 解决**单卡显存无法容纳巨大模型参数**的问题。
*   **核心操作**:
    *   **切分**: 假设模型的一层线性层权重为 `[N_hidden, N_hidden]`。
        *   DDP: 每个 Rank 拥有完整 `[N_hidden, N_hidden]` 的权重副本。
        *   **TP**: 每个 Rank 只拥有权重的一部分，例如 `[N_hidden, N_hidden / World_Size]` (按列切分) 或 `[N_hidden / World_Size, N_hidden]` (按行切分)。
    *   **前向传播**:
        1.  每个 Rank 接收**相同**的数据输入 `[Batch_Size, N_hidden]`。
        2.  每个 Rank 使用其本地的部分权重矩阵进行矩阵乘法，输出一个 **部分激活值** `[Batch_Size, N_hidden / World_Size]`。
        3.  **通信**: 需要进行 **All-Gather** 操作，将所有 Rank 产生的部分激活值收集起来，拼接成完整的激活值 `[Batch_Size, N_hidden]`。
*   **视角转变**:
    *   **DDP**: 切分 `Batch Size` 维度。
    *   **TP**: 切分 `Hidden Dimension` (模型容量) 维度。
    *   **公式举例**: 若模型隐藏维度 `N_hidden = 1024`，世界大小 `World_Size = 4`，则本地隐藏维度 `Local_N_hidden = 1024 / 4 = 256`。
*   **实现细节**: 通常需要手动为每个层分配通信内存，并显式插入通信操作（如 All-Reduce 或 All-Gather）来同步中间结果。

### 张量并行（Tensor Parallelism）的深入通信机制

- **核心概念**：在张量并行中，每个rank（进程）持有模型权重的垂直切片（如矩阵的列），因此每个rank只拥有**部分激活值**。
- **关键操作**：为了获得完整的激活值，必须执行 **全收集（All-Gather）** 通信。
    - **内存分配**：首先为**全局激活值**分配内存，其形状为 `batch size × world size × local dim`（即完整维度）。
    - **通信原语**：使用**集体通信（Collective Communication）** 中的 `All-Gather`，而不是点对点通信。
    - **数据流**：每个rank的 `x`（形状 `batch size × local dim`）不同，全收集后每个rank都获得**相同的完整激活值**，其形状为 `batch size × full dim`。
- **性能影响**：由于每层前向传播都需要进行一次大规模通信，**张量并行对互联网络带宽要求极高**。频繁的激活值传递是性能瓶颈。

### 反向传播的复杂性

- **跳过说明**：演讲者表示反向传播实现较繁琐，但并非理论上的困难，而是**编程实现上的复杂性**（需要处理梯度计算和通信）。
- **本质**：反向传播涉及同样的 `All-Gather` 反向操作（如 `Reduce-Scatter`），逻辑对称但代码量更大。

### 流水线并行（Pipeline Parallelism）

- **不同思路**：不再分割权重，而是**按层分割模型**。每个rank获得**连续的一组层**。
- **数据流**：所有rank处理**相同完整的数据**，但数据顺序流过不同rank（节点）。
    - 例：4层网络，2个节点 → 每个节点分配2层。
- **参数分配**：每个rank只加载和存储其分配的层参数。

#### 流水线气泡（Pipeline Bubble）问题

- **直接执行的缺陷**：若不优化，前一个rank必须完成所有层计算后才能向下一rank发送数据，导致大量 **空闲时间（气泡）**。

#### 缓解方案：微批次（Micro-batches）

- **核心思想**：将原始批次拆分为多个**微批次**（如 `batch size=128` 拆成4个 `batch size=32`）。
- **工作流**：
    1. rank 0 处理微批次1的层1 → 发送给 rank 1。
    2. rank 0 立即开始处理微批次2的层1，同时 rank 1 处理微批次1的层2。
    3. 如此流水线化，减少气泡。
- **通信变化**：
    - **不再使用集体通信**，改为 **点对点通信（Peer-to-Peer）**。
    - **原语**：`Send` / `Recv`（或异步的 `Isend` / `Irecv`）。
    - **具体流程**：
        1. rank i 通过 `Recv` 获取上一层传来的 `tensor x`。
        2. 计算分配给该rank的层（如2层网络）。
        3. 通过 `Send` 将结果传递给下一个rank。

### 流水线并行：从朴素实现到生产级优化

#### 1. 朴素实现的核心机制

- **基本流程**：将一个输入张量 `x` 分配给当前计算层级（rank），执行完计算后，通过**点对点通信**将结果发送给下一个 rank，再处理下一个微批次。
- **同步阻塞**：当前版本中，发送和接收操作是**同步的**，即必须等待数据完全传输或接收完成，才能继续后续流程。
- **仅覆盖前向传播**：该实现只展示了前向传播阶段，**未涉及反向传播**，而实际训练中需要**交错前向和反向步骤**来优化内存使用与计算效率。

---

#### 2. 关键缺失：通信与计算未重叠

- **瓶颈**：同步通信导致 GPU 在等待数据传输时处于**空闲状态**，无法利用计算与通信的并行性。
- **改进方向**：应采用**异步通信**（如 `MPI_Isend` / `MPI_Irecv`），让 GPU 在数据仍传输时继续执行其他微批次的计算或反向传播。

---

#### 3. 异步通信的代码级实现

- **关键函数**：使用 `i_send`（异步发送）和 `i_recv`（异步接收），它们会**立即返回一个句柄**，代表未完成的通信操作。
- **操作流程**：
  1. 启动异步发送/接收（非阻塞）。
  2. 继续执行其他计算任务（如下一个微批次的前向或反向）。
  3. 在需要使用通信结果前，调用 `wait` 或 `get` 方法等待所有操作完成。
- **区分多个通信**：当同一个 rank 向同一目的地发送多个消息时，需要通过**消息标签**或**通信上下文**来区分不同数据流，避免混淆。

---

![截图](CS336_p08_frames/chunk_10_49m00s-55m01s/frame_main_045_005146.jpg)

#### 4. 同步范式 vs. 事件驱动范式

| 特性 | 同步范式（当前主流） | 事件驱动范式（异步训练） |
|------|----------------------|--------------------------|
| **典型应用** | 超大规模同步训练（如 GPT-4） | 早期分布式训练（如 2010s 异步 SGD） |
| **通信模式** | 严格同步步调，所有 rank 对齐后执行下一步 | 梯度就绪即发送，服务器累积梯度 |
| **容错性** | 低：单个 rank 失败需整体重启 | 高：可处理工作者崩溃，梯度累积不受影响 |
| **开发者心智模型** | 类 `barrier` 同步：等待前一步完成再前进 | 回调驱动：注册事件响应函数 |

- **现代工业**：尽管规模巨大，但**同步训练仍是主流**，因为其收敛稳定性和实现简洁性优于异步方案。

---

#### 5. 生产级流水线优化要点

- **前向与反向交错**：将微批次分割，先执行部分前向，再插入反向步骤，使通信与计算时间重叠。
- **显存管理与调度**：通过事件驱动调度器，避免数据未就绪时启动计算。
- **区分消息源与消息标签**：使用标签 `tag` 或 `color` 识别同一 rank 发出的多个数据流，确保接收端正确匹配。

> **总结**：朴素流水线并行是概念上的起点，但真正高效实现需要**异步通信**、**计算/通信重叠**以及**交错前反向**，同时保留同步训练的整体框架以保障收敛。

## 并行计算中的通信与同步策略

### 核心概念：点对点通信的顺序保证

- **发送顺序保留**：当从同一进程向同一目的地发送两条消息时，这两条消息会被放入同一个**数据流**中，因此**发送顺序是严格保留的**。这确保了消息的序列化到达，不会乱序。

- **异步并发问题**：虽然单个目标的消息顺序有保证，但**向不同等级（rank）或进程发送的数据**则**没有顺序保证**，这些操作可以自由交错执行，随时可能发生。

### 关键风险：未匹配的收发导致死锁

- **阻塞行为**：如果一个进程执行了发送操作，但**没有对应的接收者在另一侧准备接收**，该进程会**永久卡住**。这是因为发送操作可能在等待接收方确认，从而陷入无限等待。

- **调试困难**：在实际代码中，很难区分这种卡住是因为**代码逻辑错误**（永远到不了接收点）还是**网络延迟**（只是时间问题）。这是一个需要谨慎处理的典型并行陷阱。

### 前向传播与反向传播的流程

- **前向传播终点**：在分布式场景下，最后一个等级（rank）最终**拥有所有激活值**，完成了完整的前向传播计算。

- **反向传播起点**：如果有人实现反向传播，将从**损失梯度的计算**开始，然后**从当前等级传回上一等级**，依此类推。原始内容中提到时间紧张，建议将反向传播实现留到次年课程。

### 从简单MLP到复杂模型的扩展

- **核心思路演示**：通过简单的MLP模型演示了三个数据张量管道的并行方案。关键是要理解**核心思想**，而不是只盯着特定模型。

- **实际挑战**：要训练Transformer等复杂模型，需要处理：
  - **通信与计算重叠**：简单实现中往往没有精细处理，需要更复杂的代码和记录机制。
  - **参数确定与记录**：对于任意架构（如Transformer层），必须手动确定各层结构并做大量记录，而MLP只是简单切分模型的演示。
  - **参考资源**：建议查看 **Megatron-LM** 或 **PyTorch的SDP**（张量并行实现），这些是工业级复杂度的参考。

### JAX生态：更优雅的并行声明式方案

- **核心思想**：JAX允许你定义模型后，**单独定义分片策略**，由**JAX编译器自动处理**其余所有细节。这比PyTorch的手动记录方式更简洁。

- **示例（基于Levanter工具包）**：
  ```python
  # 伪代码示意：只需声明要分片的维度
  model.shard(along='attention_heads')  # 按注意力头部分片
  model.shard(along='embedding_dim')    # 按嵌入维度分片
  ```

- **概念简洁性**：你只需拥有一个基本计算图，然后指定哪些维度（如**注意力头部维度**、**模型嵌入维度**）需要分片，编译器自动生成并行代码。

- **重要公式**（张量并行分片声明）：
  - 对于注意力头的分片：沿 **头维度（head_dim）** 切分
  - 对于MLP的分片：沿 **嵌入维度（embedding_dim）** 切分
  - 分片策略表示为：**`model = shard(model, along=dimension)`**

### 核心并行策略与概念总结

本节内容总结了多种深度学习模型并行化的核心思想，并通过一个JAX的简短插曲对比了不同框架的哲学。最后，通过问答环节澄清了一些关键概念。

#### 一、 并行化策略的统一视角：沿着维度分割

无论是数据并行、张量并行还是流水线并行，其核心思想都是 **“将模型或数据沿着某个维度进行切分，并分配到不同设备上”** 。每一种并行策略都对应于切分一个特定的维度。

*   **数据并行**: 沿着 **批次维度** 切分数据。
*   **张量并行**: 沿着 **宽度维度**（嵌入维度）切分模型参数。
*   **流水线并行**: 沿着 **深度维度**（层）切分模型。
*  **序列并行**: 沿着 **上下文长度维度** 切分。

#### 二、 JAX 插曲：更高级的并行抽象

JAX提供了一种比PyTorch下手动管理集体通信（如 `all-reduce`）更高级的并行化方法。

*   **核心概念**: JAX 允许你 **声明式地指定** 哪个维度需要被切割（例如，是批次维度还是嵌入维度），以及如何将这些切割后的维度映射到实际的 **TPU 硬件** 上。
*   **编译器的作用**: 一旦你声明了这种“分片映射”，**JAX编译器** 会自动将其编译成底层的、高效的原语（如 `all-reduce`、`all-gather`），并自动处理复杂的数据重排。
*   **与PyTorch的对比**: 这种高级抽象让开发者不必手动处理底层的通信细节，从而简化了开发。但坚持使用PyTorch（手动处理）能让你更清晰地看到 **底层实际发生了什么**，这对于深入理解系统至关重要。
*   **结论**: 在实际工程中，你通常不需要从零开始实现这些高级抽象。

#### 三、 权衡取舍：重计算 vs. 存储 vs. 通信

在并行训练中，一个反复出现的主题是如何在 **重计算**、**存储** 和 **通信** 之间进行权衡。

*   **方案1: 重新计算**
    *   **优点**: 节省内存（显存）。
    *   **缺点**: 消耗计算时间。
    *   **适用场景**: 当计算开销远低于通信开销或内存访问开销时，**重计算通常是更好的选择**。
*   **方案2: 存储在内存中，并承担数据传输成本**
    *   **优点**: 无需重算。
    *   **缺点**: 消耗内存，并可能带来额外的数据搬运（例如，从CPU内存到GPU显存）成本。
*   **方案3: 存储在另一块GPU内存中，并通过通信获取**
    *   **优点**: 利用了分布式环境中的空闲显存。
    *   **缺点**: **通信速度通常比本地内存访问慢得多**，因此这可能是三者中最慢的方案。
*   **关键判断**: 你通常处于 **通信受限** 或 **内存受限** 的状态。重算是一种在特定条件下（计算密集、通信稀疏）可行的优化。

#### 四、 关于未来与物理极限的思考

虽然硬件（如HBM高带宽内存）在不断进步，但这并不意味着并行化技术将变得无足轻重。

*   **硬件趋势**: GPU的HBM容量和带宽正在快速增长。
*   **模型趋势**: 模型规模的增长速度可能更快，会不断逼近硬件的物理极限。
*   **结论**: 系统层面的优化（即本节讨论的并行化策略）将始终伴随我们，因为总有更大的模型需要被适应。

#### 五、 Q&A 关键澄清

*   **问题：数据并行中，即使参数大致同步，是否仍可能存在数据依赖问题（例如，批量归一化）？**
    *   **答案**:
        *   在 **语言模型** 中，这个问题通常不显著，因为 **层归一化** 被广泛使用，而不是批量归一化。
        *   只要所有参数使用 **相同的随机种子初始化** 并同步更新，它们应该保持一致。
        *   可能存在 **GPU上的非确定性问题**，但影响通常很小。
*   **问题：PyTorch是否有类似JAX的高级并行化库？**
    *   **答案**:
        *   **是的**，PyTorch 有 `FSDP` （全分片数据并行） 库。在实际工作中，你应该使用它，而不是手动实现。
        *   `FSDP` 基本是一个自动化包装器，可以自动处理大多数模型的分片。
        *   **与JAX的对比**:
            *   PyTorch 的 `FSDP` 虽然自动化，但在 **灵活性** 和 **自定义分片** 方面可能不如JAX的声明式系统成熟。
            *   当前的 **PyTorch基础设施**（例如，对于自定义分片）仍在开发中，不如 **JAX + TPU 生态系统**（在该特定生态内）那样完善。

### 计算生态系统的两极分化与硬件利用

*   **核心对比：声明式 vs. 命令式硬件利用**
    *   **JAX / TPU 生态**：代表了**声明式**的编程范式。用户从“高处”定义模型，一切由框架自动处理。这种模式依赖于Google成熟的底层基础设施。
    *   **GPU 生态（如针对深度探索的）**：代表了**命令式**的极致。开发者需要深入到“镍粒级别”（即硬件最微观的细节），手动执行大量干预操作以榨取性能。这种模式下，硬件本身（如连接性）可能表现较差，需要大量手动的、精细的优化。

*   **核心论断**：**如何利用硬件完全取决于你所在的生态系统**。JAX让你“自然发生”，而GPU导向的生态则要求深度干预。

### 激活检查点（Activation Checkpointing）的最佳API

*   **核心问题**：在PyTorch和JAX中，理想的API应如何允许用户指定**需要重新计算**的部分？
*   **现有局限**：“要么全部，要么不”（Either All or Nothing）的粗粒度控制是不理想的。
*   **理想策略**：选择性地进行**隔层检查点**。
    *   **场景**：假设一个网络由“大型模块”（如哺乳动物/主干网络）和“逐点非线性变换”交替组成。
    *   **优化**：对于逐点变换，其输入可以直接从上一层的输出中轻松地“重新计算”出来。因此，**没必要存储两份中间结果**。只需在关键的大型模块后存储一个版本，后续的轻量计算直接重算即可。

### 专用硬件 vs. 通用GPU

*   **趋势**：深度学习的训练/推理正在从通用GPU转向**专用硬件**。
    *   **推理领域**：已经广泛采用（如Groq、Cerebras等公司的专用推理芯片）。
    *   **训练领域**：新的硬件设计正在尝试提供更多的**板载内存**（避免数据移动）。
*   **核心痛点**：**数据移动是游戏规则的改变者**。如果数据不用移动，操作会被极大简化。GPU存在大量**遗留问题**（如为分支操作和临时计算设计的架构），这些在深度学习（以矩阵乘法为主）中并不必要。因此，硬件优化空间巨大。

### 持续训练（Continuous Training）的可能性

*   **关键问题**：在模型训练中，是否可以避免完全重头训练，而是利用**持续训练**（例如，从检查点继续）？
*   **直接回答**：**绝对可以**。当前训练过程本质上是在进行“梯度计算步骤”。只要你有一个**半训练好的检查点**，就可以直接在此基础上继续训练，无需从随机初始化开始。

### 模型规模与硬件的物理限制

*   **问题**：是否存在物理或技术上的原因，导致无法无限增大模型（例如，不能制造无限大的GPU并互联成无限大的集群）？
*   **核心限制**：确实存在**物理限制**。
    1.  **功耗与散热**：制造超高密度或无限大的GPU，会带来巨大的功耗和散热问题，无法有效处理。
    2.  **带宽限制**：节点间的连接带宽、内存带宽等存在物理极限。你不可能无限制地往一块芯片上塞东西。这些物理约束决定了实际进展只能专注于特定部分的优化，而不是无限地扩大规模。

### 核心架构演进：从以计算为中心到以数据为中心

*   **核心概念：** 传统的CPU时代是**以控制为中心**的，代码是“一等公民”，数据需要被移动到执行单元。
*   **深度学习的变革：** 深度学习任务（如PyTorch中的计算图）**本质上是以数据流为中心**。
    *   **特性：** 计算图是**静态的**（即从训练开始到结束，所有要执行的计算在开始时就已经确定）。
    *   **优势：** 利用这种**预先确定性**，可以**更智能地调度计算**，避免了传统架构中处理“临时不确定性计算”的灵活性开销。
*   **通俗理解：** 传统编程像临时指挥交通，而深度学习像铺设好铁轨的火车系统，提前规划好所有路线。

### 计算图与硬件架构的关键问题

*   **计算图的存储位置？**
    *   计算图是一个**概念性**的结构，它**不是**一个被实际物理“放置”到GPU上的图示。
    *   代码在CPU上运行，当调用PyTorch等函数需要GPU执行时，CPU会**调度**底层的GPU内核进行运算。计算图的逻辑流程由CPU驱动。

*   **通信原语的执行主体？**
    *   **概念：** 通信原语（如AllReduce等集合操作）在PyTorch分布式中定义了操作类型，它是抽象层的定义。
    *   **执行者：** 这些操作**可以在CPU或GPU上执行**。
    *   **调度者：** **CPU扮演主控角色（Master）**。CPU调用NCCL（NVIDIA Collective Communications Library）等库，这些库在CPU上运行，并**启动内核**来在GPU之间移动数据。

### 对未来的思考与行业趋势

*   **GPU的演进方向：**
    *   **硬件极限：** 无法制造无限大或超高密度的GPU，存在**功耗**和**带宽**的物理限制。
    *   **权衡：** 某些方案（如将内存集成在芯片上）牺牲了灵活性（如通用性），换取了更高的密度和性能。
*   **未来方向：** 从“CPU时代”（重控制）向“GPU时代”（重数据流）演进，利用计算图的**静态确定性**来优化硬件和调度策略，是未来的发展重点。
